    Getters/Setters. Evil. Period.

    The following text is a translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    有一个旧辩论始于2003年，由Allen Holub在他著名的文章《为什么getter和setter方法是邪恶的》中引发，关于getter/setter方法是否是一种反模式，并且应该避免使用，还是它们是面向对象编程中必不可少的东西。我将尝试在这个讨论中提出我的观点。

    以下文本的要点是：getter和setter是一种糟糕的做法，使用它们的人不能被原谅。再次强调，我并不是说在可能的情况下应该避免使用get/set。不。我是说你的代码附近绝对不应该有它们。

    足够傲慢以引起你的注意了吗？你已经使用那个get/set模式15年了，还是一位备受尊敬的Java架构师？你不想听一个陌生人说这些胡言乱语？好吧，我理解你的感受。当我偶然发现David West的《对象思考》时，我也几乎有同样的感觉，这是我迄今为止读过的关于面向对象编程最好的书。所以，请冷静下来，试着理解我尝试解释的内容。

    Existing Arguments

    在面向对象的世界中，有一些反对“访问器”（另一个称呼为获取器）的论点。我认为，所有这些论点都不够有力。让我们简要地浏览一下它们。

    讲述，不要询问 Allen Holub说：“不要询问你做工作所需的信息；而是要求拥有信息的对象为你完成工作。”

    违反封装原则 一个对象可以被其他对象撕裂开来，因为它们能够通过设置器将任何新数据注入其中。该对象无法足够安全地封装其自身状态，因为任何人都可以修改它。

    暴露实现细节 如果我们可以从另一个对象中获取一个对象，那么我们过于依赖于第一个对象的实现细节。如果明天它发生变化，比如说结果的类型改变了，我们也必须修改我们的代码。

    All these justifications are reasonable, but they are missing the main point.

    Fundamental Disbelief

    大多数程序员认为对象是带有方法的数据结构。我引用了Bozhidar Bozhanov的文章《Getters and Setters Are Not Evil》（http://java.dzone.com/articles/getters-and-setters-are-not）。

    But the majority of objects for which people generate getters and setters are simple data holders.

    这个误解是一个巨大误解的结果！对象不是“简单的数据容器”。对象不是附带方法的数据结构。这个“数据容器”的概念来自于过程式编程语言，尤其是C和COBOL。我再说一遍：对象不是一组数据元素和操纵它们的函数。对象不是一个数据实体。

    What is it then?

    A Ball and A Dog

    在真正的面向对象编程中，对象就像你和我一样是[生物]（），它们是有生命的有机体，拥有自己的行为、属性和生命周期。

    一个生物体能有一个setter吗？你能把球“设置”给一只狗吗？实际上不行。但下面这段软件正在做的正是这个。

    Dog dog = new Dog();
dog.setBall(new Ball());

    How does that sound?

    你能从狗那里取出一个球吗？嗯，如果它吞下了这个球并且你要进行手术的话，可能是可以的。在这种情况下，是的，我们可以从狗那里取出一个球。这就是我所说的。

    Dog dog = new Dog();
Ball ball = dog.getBall();

    Or an even more ridiculous example:

    Dog dog = new Dog();
dog.setWeight("23kg");

    Can you imagine this transaction in the real world?

    它看起来和你每天写的东西相似吗？如果是的，那么你是一个过程式程序员。承认吧。这就是大卫·韦斯特在他的书第30页上对此的看法。

    Step one in the transformation of a successful procedural developer into a successful object developer is a lobotomy.

    你需要进行脑叶切除术吗？嗯，我绝对需要并且在阅读韦斯特的《对象思维》时得到了它。

    Object Thinking

    Start thinking like an object and you will immediately rename those methods. This is what you will probably get:

    Dog dog = new Dog();
dog.take(new Ball());
Ball ball = dog.give();

    现在，我们将狗视为一只真正的动物，它可以从我们这里接过一个球，并在我们要求时将球还给我们。值得一提的是，狗无法还回“NULL”。狗根本不知道“NULL”是什么。通过对象思维，可以立即从您的代码中消除NULL引用。

    除此之外，对象思维将导致对象的不可变性，就像“狗的体重”这个例子一样。你会这样重新编写它：

    Dog dog = new Dog("23kg");
int weight = dog.weight();

    狗是一个不可变的生物，不允许任何外部人改变她的体重、大小、名字等等。她可以在要求时告诉她的体重或名字。公共方法展示了对象内部的某些“内部”是没有问题的。但这些方法并不是“获取器”，并且它们永远不应该有“获取”前缀。我们并不是在从狗“获取”任何东西。我们并不是在获取她的名字。我们只是在询问她告诉我们她的名字。看到区别了吗？

    我们这里并不是在讨论语义学的问题。我们是在区分过程式编程思维和面向对象思维。在过程式编程中，我们操作数据，根据需要进行获取、设置和删除。我们掌控一切，数据只是一个被动的组成部分。对我们来说，这只狗什么都不是——它只是一个”数据容器”。它没有自己的生命。我们可以自由地从中获取所需的任何内容，并将任何数据设置到其中。这就是C、COBOL、Pascal和许多其他过程式语言的工作方式。

    相反地，在真正的面向对象的世界中，我们将对象视为生物体，具有自己的出生日期和死亡时刻，有自己的身份和习惯，如果你愿意的话。我们可以要求狗提供一些数据（例如她的体重），她可能会返回给我们那些信息。但我们始终记住狗是一个主动的组成部分。她决定我们请求后会发生什么。

    这就是为什么在一个对象中以set或get开头的任何方法在概念上是错误的。这与破坏封装无关，尽管很多人会争论这一点。这是关于你是否像一个对象思考，还是仍然用Java语法写COBOL。

    PS. 是的，你可能会问，—JavaBeans，JPA，JAXB和许多其他依赖于get/set表示法的Java API呢？Ruby的内置特性简化了访问器的创建，这些都是我们的不幸。与真正理解和欣赏真实对象的美丽世界相比，停留在原始的过程化COBOL世界中要容易得多。

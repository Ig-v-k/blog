    Best Hosted Continuous Integration Services for a Private Repository

    我所参与的每个项目都始于一套连续集成流水线的设置。我是云服务的忠实爱好者，所以一直在使用Travis。最近，我的几个客户对这个选择提出了质疑，主要是因为价格的原因。因此，我决定对市场进行简要分析。

    我在我能找到的每个CI服务中都配置了开源项目Rultor。它们都是免费提供给开源项目使用的。它们都是托管的，不需要任何服务器安装。以下是按照我的个人偏好顺序排列的这些服务（前四个是最好的，强烈推荐）：

    顺便说一句，这个列表中的一些平台联系我，并要求我再次对它们进行评估。有些甚至提供给我金钱，希望我将它们排名提高（开个玩笑）。不管怎样，是否这是一个好迹象由你决定（我认为这是好的，它们关心自己的公共关系）。我在列表中用 ☺ 表情符号标记了它们。

    Shippable很容易配置，因为它能够直接理解.travis.yml。用户界面易于导航，因为根本没有“设置”页面（或者我没有找到）。一切都通过存储库中的shippable.yml文件进行配置。该服务看起来稳定且健壮，目前还没有任何抱怨。关于它们特别酷的是它们允许您在Docker容器中构建。

    Travis是我迄今为止看到的最好的平台。主要是因为它是最受欢迎的。它与GitHub完美地集成在一起，并且有适当的文档。一个重要的缺点是每月129美元的价格。”用这些钱，你可以获得一个专用的EC2实例并在那里安装Jenkins”——我的一些客户说。我强烈不同意，因为Jenkins将需要24x7的管理，这比129美元要贵得多，但解释总是很困难的。

    Wercker是一款来自阿姆斯特丹的欧洲产品，目前仍处于beta测试阶段，因此对所有项目免费。该平台看起来非常有前景。对于私有存储库仍然免费，并得到投资支持。他们还有一个有趣的构建“盒子”概念，可以类似于Docker容器进行预配置。在过去几个月中，它的稳定性相当高，至今没有任何投诉。

    AppVeyor是唯一可以运行Windows构建的服务。尽管我主要使用Java和Ruby，这两种语言被认为是平台无关的，但它们往往恰好相反。当你的构建在Linux上成功时，并不能保证它也能在Windows或Mac上通过。我计划在每个项目中使用AppVeyor，并与其他CI服务结合使用。下面是我如何将Maven构建与AppVeyor集成。

    SemaphoreApp易于配置和使用。它给人一种轻量级系统的印象，这一点我非常欣赏。不足之处是，他们没有预安装任何Maven，但这很容易通过一个简短的自定义脚本来解决，该脚本可以下载和解压最新的Maven。另一个不足之处是他们不能通过文件（如.travis.yml）进行配置，您需要通过用户界面执行所有操作。他们还支持构建之间的缓存。

    Snap-CI是ThoughtWorks的产品，ThoughtWorks是Go的创作者，它是一个开源的持续集成服务器。它看起来比其他产品更加复杂，可以定义”阶段”并将它们组合成流水线。我还不确定这些机制如何帮助我们主要从事的小型和中型项目，但它们看起来很”酷”。另外，每个构建的内存限制只有2GB，这对我一些Java项目来说是非常不幸的，因为会导致构建失败。除此之外，他们不提供对构建服务器的完全访问权限，例如我们无法在/etc目录中进行任何修改，这对我们来说是一个终止条件。

    Codeship的功能很好，但是他们的网页界面看起来有点过时。除此之外，他们承诺能够与pull requests一起工作，但我没能成功配置。即使他们能够构建pull requests，却没有通知我们在GitHub上的pull requests。也许我会找到解决办法，但目前还不清楚。

    CircleCI我还不知道为什么我的构建失败了。真的很难配置和理解发生了什么。正在努力弄清楚…

    SolanoLabs看起来相当不成熟且难以配置。他们甚至不支持在添加新存储库时自动配置GitHub挂钩。然而，他们的销售人员对我进行了相当激进的垃圾邮件推销。

    Hosted-CI仅适用于iOS/OSX。即使对于开源项目，他们也不提供任何免费的服务。我还没有机会测试它们。

    CloudBees基本上是一个托管的Jenkins。我真的不喜欢Jenkins，所以不能推荐这个平台。

    DeployBot甚至不允许我通过GitHub登录，是吗？他们似乎更注重”部署”而不仅仅是持续集成。

    Vexor看起来很不错，提供了一种非常独特的计费模式——按构建次数收费，而不是按月收费。我肯定会推荐你试一试。不过，我自己没能让它运行起来…

    GreenHouseCI是一个适用于移动应用（iOS，Android等）的CI平台。听起来很有趣，只是我没有一个完整的移动应用来测试它。

    顺便说一句，如果你不喜欢将持续集成保留在云端的想法，可以考虑以下本地软件包（按照优先顺序排列）：Jenkins，TeamCity，Go，Strider，BuildBot。

    请记住，无论您的持续集成服务有多好且多么昂贵，除非您将主分支设置为只读，否则它都无法帮助您。

    1 This means that the platform can build your repo in Linux environment. Almost all of them do that by default, unless you configure them otherwise.
    2 Some of them can build on Windows platform.
    3 MacOS support means that an Objective-C/Swift product can be built there.
    4 I mean GitHub pull request support here. Some of them can be integrated with GitHub and will build pull requests before they are merged. Build status will be visible in GitHub. A pretty convenient feature.
    5 Logs compression is a critical feature, at least for me. Most of my logs are from Maven and without col -b they look too long and unreadable.
    6 Looks like Docker containers must be supported by all of them, but unfortunately it’s not the case. Ideally, all builds should run in containers.

    Strict Control of Java Code Quality

    The following text is a translation of the original article, performed by ChatGPT (gpt-3.5-turbo):

    Il existe de nombreux outils qui contrôlent la qualité du code Java, notamment Checkstyle, PMD, FindBugs, Cobertura, etc. Tous sont généralement utilisés pour analyser la qualité et générer des rapports élaborés. Très souvent, ces rapports sont publiés par des serveurs d’intégration continue, tels que Jenkins.

    Qulice takes things one step further. It aggregates a few quality checkers, configures them to a maximum strict mode, and breaks your build if any of them fail.

    Sérieusement. Il y a plus de 130 vérifications dans Checkstyle, plus de 400 règles dans PMD et plus de 400 bugs dans FindBugs. Tous devraient dire : “Oui, nous aimons votre code.” Sinon, votre construction ne devrait pas réussir.

    Que pensez-vous ? Serait-il pratique pour vous que votre code soit rejeté chaque fois qu’il échoue à l’une des 900 vérifications ? Serait-il productif pour l’équipe de contraindre les développeurs à se concentrer autant sur la qualité du code ?

    If you join one of our teams as a Java developer, you will develop your code in branches and, then, Rultor will merge your changes into master. Before actually merging, though, Rultor will run an automated build script to make sure that your branch doesn’t break it.

    En tant qu’outil d’analyse statique, Qulice n’est qu’une des étapes du script de construction automatisée. Il s’agit en réalité d’un plugin Maven et nous automatisons les constructions Java avec Maven 3x. Ainsi, si vos modifications enfreignent l’une des règles de Qulice, votre branche entière est rejetée.

    Votre première réaction - je l’ai vue des centaines de fois - sera négative. Vous pourriez même être assez frustré pour quitter le projet immédiatement. Vous pourriez dire quelque chose comme ceci (je cite des histoires de la vie réelle):

    “These quality rules entirely ruin my creativity!”

    Au lieu de perdre du temps sur ces virgules et accolades mal placées, nous ferions mieux de développer de nouvelles fonctionnalités !

    “I’ve done many successful projects in my life, never heard about this ridiculous quality checking…”

    Cette première réaction est simplement logique. J’ai vu beaucoup de gens dire des choses comme cela, aussi bien dans des projets open source que commerciaux. Non seulement en Java, mais aussi en PHP (avec phpcs et phpmd) et en Ruby (avec rubocop et simplecov).

    How do I answer? Read on.

    Mon expérience me dit que plus tôt quelqu’un peut s’habituer au contrôle strict de qualité de Qulice, plus vite il/elle peut apprendre et grandir ; meilleur programmeur il/elle est ; et plus loin il/elle peut aller avec nous et nos projets.

    Ayant cette expérience à l’esprit, je recommande que tous les nouveaux membres du projet fassent preuve de patience et essaient de s’habituer à cette nouvelle approche de la qualité. En quelques semaines, ceux qui s’y tiennent commencent à comprendre pourquoi cette approche est bénéfique pour le projet et pour eux, en tant qu’ingénieurs Java.

    Why is it good? Read on.

    Let’s take one simple rule as an example. Here is a piece of Java code that Qulice would complain about (due to the DesignForExtension rule from Checkstyle):

    public class Employee { public String name() { return “Jeff”; } }

    Qu’est-ce qui ne va pas avec ce code ? Le nom de la méthode() n’est pas final et peut être remplacé par une classe qui étend Employee. Du point de vue de la conception, c’est incorrect, car une classe enfant est autorisée à modifier une classe parente, en remplaçant sa méthode.

    Quel est le bon design ? Celui-ci :

    java public class Employee { public final String name() { return “Jeff”; } }

    Maintenant, la méthode est finale et ne peut pas être substituée par des classes filles. C’est un design beaucoup plus sûr (selon Checkstyle, et je suis d’accord).

    Alors, disons que nous rendons cette règle obligatoire pour toutes les classes du projet. Qu’est-ce que le projet en retire ? Il peut promettre à ses membres (programmeurs) une meilleure qualité de travail, par rapport aux autres projets qui n’ont pas cette restriction, principalement en raison de :

    Prévisibilité de la conception — Je n’ai pas à faire défiler toute la classe pour m’assurer qu’elle ne contient pas de méthodes pouvant être accidentellement remplacées. Je sais avec certitude que cela ne peut pas se produire dans ce projet. En d’autres termes, je sais à quoi m’attendre.

    Moins de tours cachés - Une plus grande prévisibilité de conception conduit à une meilleure visibilité des erreurs et des astuces. La normalisation du code source le rend uniforme. Cela signifie qu’il est plus facile à lire et à repérer les problèmes.

    Normes de l’industrie - La décision d’utiliser cette conception est prise par Checkstyle, et non par un architecte de projet. Pour moi, en tant que développeur de projet, cela signifie que je suis conforme aux normes de l’industrie. Cela rend le projet (et ses dirigeants) plus respectables.

    Apprentissage—Je parie que la plupart d’entre vous qui lisent ce message ne connaissaient pas la règle de conception expliquée ci-dessus. Rien qu’en lisant cet article, vous avez appris quelque chose de nouveau. Imaginez tout ce que vous pourriez apprendre après avoir rendu votre code conforme aux 900 règles de Qulice (Checkstyle + PMD + FindBugs).

    Le point concernant l’apprentissage nous amène à la dernière réflexion à discuter, qui est aussi la plus importante.

    En tant que programmeur, j’espère que vous réalisez déjà ce que vous obtenez en travaillant sur un projet qui élève sa barre de qualité aussi haut que Qulice le demande. Oui, vous apprendrez beaucoup de nouvelles choses sur l’écriture de code Java de qualité.

    En plus de cela, je dirais en fait que vous obtenez des leçons gratuites avec chaque nouvelle ligne de code que vous écrivez. Et le professeur est un logiciel, écrit par des centaines de développeurs Java, depuis les dix dernières années. Qulice ne fait que regrouper ces outils logiciels ensemble. En vérité, ce sont les développeurs qui sont les véritables auteurs des vérifications de qualité et des règles.

    Alors, que dois-je dire à ceux qui se plaignent que les règles de qualité sont trop strictes ? Je dis ceci : “Voulez-vous apprendre et vous améliorer, ou voulez-vous simplement être payé et vous en sortir ?”

    ps. Vous pouvez utiliser mon settings.jar pour IntelliJ, ils sont plutôt stricts et vous aideront à nettoyer votre code avant même que Qulice ne commence à se plaindre.

    所有列出的错误都与面向对象编程一般以及特别与Java相关。

    你的类应该是一个对真实生活实体的抽象，不包括“validators”、“controllers”、“managers”等。如果你的类名以“-er”结尾——那就是一个糟糕的设计。顺便提一下，以下是我对一个好对象的七个特点。此外，这篇文章对这个观点做了更详细的解释：不要创建以-ER结尾的对象。

    当然，实用工具类是反模式，比如Apache的StringUtils、FileUtils和IOUtils。以上都是糟糕设计的完美例子。请阅读这篇后续文章：面向对象编程的实用工具类替代方案。

    当然，永远不要添加后缀或前缀来区分接口和类。例如，所有这些名称都是非常错误的：IRecord，IfaceEmployee或RecordInterface。通常，接口名称是现实生活实体的名称，而类名称应该说明其实现细节。如果对实现没有特别要说的地方，将其命名为Default，Simple或类似的名称。例如：

    方法可以返回某个值或者返回空。如果一个方法返回某个值，那么它的名称应该解释返回的内容，例如（不要使用 get 前缀）。

    java boolean isValid(String name); String content(); int ageOf(File file);

    java boolean isValid(String name); String content(); int ageOf(File file);

    如果它返回为空，则它的名称应该解释它的作用。例如：

    java void save(File file); void process(Work work); void append(File file, String line);

    保存(File file)函数；处理(Work work)函数；追加(File file, String line)函数。

    您可以在《优雅的对象》这本书的第2.4节中了解更多关于这个想法的内容。刚才提到的规则只有一个例外—JUnit的测试方法。下面将对其进行解释。

    在JUnit测试中，方法名应该使用没有空格的英文句子来创建。通过例子来解释会更容易理解：

    重要的是在你的Javadoc的第一句话中以你正在测试的类的名称开头，后面跟上can（或cannot）。因此，你的第一句话应该总是类似于“某人可以做某事。”

    方法名称将完全相同，只是没有主语。如果我在方法名称的开头添加一个主语，我应该得到一个完整的英文句子，就像上面的例子一样：“HttpRequest以Unicode格式返回其内容。”

    请注意，测试方法不以“can”开始。只有Javadoc注释使用“can”。

    将测试方法始终声明为抛出异常是一个好的实践。

    避免使用复合变量名，比如timeOfDay、firstItem或httpRequest。我指的是类变量和方法内的变量都要遵循这个原则。变量名应该足够长，以避免在其可见范围内产生歧义，但如果可能的话不要太长。变量名应该是单数或复数形式的名词，或者是适当的缩写。关于这个问题，可以参考这篇文章：《复合名称是一种代码异味》。例如：

    尽量避免使用一次性变量。所谓的一次性变量是指只被使用一次的变量。就像在这个例子中一样：

    java String name = “data.txt”; return new File(name);

    不用说，你绝对不应该吞咽异常，而是应该尽可能地让它们上升。私有方法应该始终让已检查异常传递出去。

    永远不要使用异常来控制流程。例如，这段代码是错误的：

    java int size; try { size = this.fileSize(); } catch (IOException ex) { size = 0; } Java int size; 尝试 { size = this.fileSize(); } 捕获 (IOException ex) { size = 0; }

    对于缩进，主要规则是括号要么在同一行结束，要么在同一行关闭（反向规则适用于闭合括号）。例如，以下是不正确的，因为第一个括号没有在同一行关闭，且后面有符号。第二个括号也有问题，因为前面有符号，并且没有在同一行打开：

    类常量应该在你想要在类方法之间共享信息，并且这些信息是你的类的特征时使用。不要将常量用作字符串或数字字面值的替代品——这是非常糟糕的做法，会导致代码污染。常量（就像面向对象编程中的任何对象一样）应该在现实世界中有意义。这些常量在现实世界中有什么意义：

    Java类文件 Document { private static final String D_LETTER = “D”; // 不良实践 private static final String EXTENSION = “.doc”; // 良好实践 }

    另一个典型的错误是在单元测试中使用常量，以避免在测试方法中重复使用字符串/数字字面值。请不要这样做！每个测试方法都应该使用自己的一组输入值。

    在每个新的测试方法中使用新的文本和数字。它们是独立的。那么，为什么它们必须共享相同的输入常量呢？

    这是一个测试方法中数据耦合的示例。

    java User user = new User(“Jeff”); // 可能还有其他代码 MatcherAssert.assertThat(user.name(), Matchers.equalTo(“Jeff”));

    在最后一行，我们将“Jeff”与第一行的相同字符串文字结合在一起。如果几个月后有人想要更改第三行的值，他/她必须额外花费时间找到在同一方法中还使用了“Jeff”的地方。

    为了避免这种数据耦合，你应该引入一个变量。更多相关信息请参阅：《关于单元测试脚手架的几点思考》。

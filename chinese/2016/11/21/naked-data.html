    Encapsulation Covers Up Naked Data

    The following text is a translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    封装是面向对象编程的核心原则，使对象具有坚实、内聚和可信等特性。但究竟什么是封装呢？它是否只是保护对象外部对私有属性的访问？我认为它远不止如此。封装导致所有层级和形式上的数据都不会裸露。

    这就是裸数据是什么（C代码）：

    C int t; t = 85; printf(“The temperature is %d F”, t);

    C int t; t = 85; printf(“温度为 %d 华氏度”, t);

    在这里，t是数据，可以通过周围的代码公开访问。任何人都可以修改或阅读它。

    为什么这样不好？有一个原因：紧密而隐蔽的耦合。

    代码中关于 t 的部分不可避免地对数据做出了很多假设。例如，在 int t 之后的两行决定了温度是以华氏度表示的。在编写这段代码的时候，这可能是正确的，但这种假设将代码与数据耦合在一起。如果明天我们将 t 改为摄氏度，代码将不知道这个变化。这就是为什么我称之为隐藏的耦合。

    如果我们将t的类型从int更改为double，printf行将不会在小数点后打印任何内容。再次，耦合是存在的，但它是隐藏的。后来，我们将无法找到我们在代码中做出这些或其他关于t的假设的所有位置。

    这将严重影响可维护性。

    这并不是一个解决方案，正如你所能想象的（现在的Java）。

    java类 温度 { private int t; public int getT() { return this.t; } public void setT(int t) { this.t = t; } }

    它看起来像一个物体，但数据仍然是裸露的。任何人都可以从该物体中检索数据，并决定它是华氏还是摄氏，是否有小数点后的数字等等。这还不是封装！

    唯一的封装t的方法是确保任何人都无法直接触碰它，也无法通过从对象中检索它来获取。我们该如何做到呢？只需停止暴露数据，开始暴露功能。例如，以下是做法：

    java类 Temperature { private int t; public String toString() { return String.format(“%d F”, this.t); } }

    我们不允许任何人再检索 t 值了。他们所能做的只是将温度转化为文本。如果我们决定将 t 值改为摄氏度，我们将只在一个地方进行一次修改：在类 Temperature 中。

    如果我们在将来需要其他功能，比如数学运算或转换为摄氏度，我们会在Temperature类中添加更多方法。但是我们绝不允许任何人触碰或了解t。

    这个想法与我们之前讨论过的“打印机取代获取器”的概念相似，只是从更广泛的角度来看。在这里，我想说的是任何逃逸对象的数据元素都是裸露的，并导致可维护性问题。

    问题是我们如何完全不使用裸数据工作，对吧？最终我们必须让对象交换数据，不是吗？是的，这是正确的。但并不完全如此。我将在下一篇文章中解释这一点。

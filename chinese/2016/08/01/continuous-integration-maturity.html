    8+2 Maturity Levels of Continuous Integration

    The following text is a translation of the original article, performed by ChatGPT (gpt-3.5-turbo):

    Il y a plusieurs niveaux que vous devez franchir avant que votre pipeline d’intégration continue ne devienne parfait. J’en ai trouvé huit et j’ai présenté mes découvertes lors des DevOpsDays à Salt Lake City il y a quelques semaines (regardez la vidéo). Maintenant, il est temps de les écrire et de vous demander : À quel niveau êtes-vous ? Postez votre réponse ci-dessous.

    Code source. Ici, vous écrivez simplement du code source sur vos ordinateurs et peut-être quelque part sur le serveur. Le mieux que vous puissiez faire ici est de le construire manuellement, disons, chaque jour. Est-ce une intégration continue ? Eh bien, dans une certaine mesure, à condition de ne pas oublier de compiler et de regrouper régulièrement tout.

    Build automatisé. À ce niveau, votre build est automatisé, ce qui signifie que vous pouvez compiler, tester et empaqueter l’ensemble du produit en une seule ligne de commande. Faites attention, une seule ligne. Vous devez être capable d’appuyer sur un seul bouton et obtenir soit une erreur, soit un build réussi.

    Git. À ce niveau, vous gardez votre code source dans Git. Vous pouvez le garder dans un autre système de contrôle de source, mais cela serait étrange - Git est le statu quo pour le moment. Vous devriez être en mesure d’obtenir un nouvel ordinateur, sans rien dedans, récupérer le code source à partir d’un dépôt Git et exécuter une compilation complète.

    Pull Requests. Chaque modification apportée à votre code source doit être soumise via une demande de pull, ce qui signifie que vous hébergez votre dépôt sur GitHub. Vous pouvez l’héberger ailleurs, mais cela serait étrange car GitHub est l’état actuel. Encore une fois, personne ne devrait pouvoir effectuer des commits directement sur la branche principale, à moins de passer par des forks et des demandes de pull.

    Code Reviews. Chaque demande d’extraction doit passer par une revue de code obligatoire avant d’être fusionnée. Vous devez avoir une politique de revue de code qui explique qui effectue les revues, ce qui se passe si l’auteur n’est pas d’accord avec le relecteur, etc. Mais aucune demande d’extraction ne peut être fusionnée sans avoir été revue.

    Tests. À ce niveau, votre code est couvert par des tests unitaires (et des tests d’intégration), et chaque changement est accompagné d’un nouveau test. Votre construction automatisée exécute tous les tests ensemble et échoue si l’un d’entre eux échoue.

    Analyse statique. Vérifier la qualité de votre code sans l’exécuter, c’est de quoi il s’agit dans l’analyse statique. À ce niveau, la qualité de votre code est vérifiée par la construction automatisée. Si la qualité est inférieure au seuil, la construction échoue.

    Pre-Flight Builds. Cette idée est expliquée ici.

    Simulation de production. La construction est exécutée dans un conteneur, qui simule l’environnement et les données de production.

    Tests de stress. Les tests de performance et de stress sont automatisés et exécutés sur chaque version.

    Au fait, lors de la présentation, J’ai également mentionné les problèmes auxquels vous pourriez être confrontés à chaque niveau de maturité.

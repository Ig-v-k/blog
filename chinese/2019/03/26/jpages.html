    这还不是一个网页应用程序，只是一个框架，可以将简单的HTTP请求派发为HTTP响应。尽管其中没有严谨的面向对象编程（OOP），但它仍然能够正常工作。现在我们应该关注一个更重要的问题：如何为网页应用程序添加更多功能，使其能够处理不同的页面、渲染更大的内容并处理错误？上述代码片段中的请求变量应该以某种方式转换为响应。

    最简单的方法是：1）将请求转换为包含所有细节的DTO，然后2）将其发送给一个“控制器”，该控制器知道如何处理DTO中的数据，然后3）从控制器接收响应的DTO，将数据取出并呈现响应。这就是Spring和大多数其他框架的做法。然而，我们不会走这条路，而是尝试不使用DTO，纯粹面向对象地实现。

    我必须说，可能有多种设计方案，都采用面向对象的风格。现在我只会展示其中一种选择。你肯定知道我们的Takes框架，它诞生于几年前，具有自己的设计，同样是面向对象的。但是我现在要提议的这个似乎更好。你也可能想出其他的东西，所以请随时在下面的评论中发表你的想法，甚至创建一个GitHub仓库，在那里分享你的想法。

    我建议我们引入两个接口：Resource（资源）和Output（输出）。Resource是服务器端的实体，它根据传入的请求参数进行改变。例如，当我们只知道请求是GET /时，它是一个资源。但是，如果我们还知道请求有例如Accept: text/plain，我们可以改变请求并创建一个新的请求，以提供纯文本。这是接口：

    java接口资源 { Resource refine(String name, String value); }

    这是我们创建和突变的方式：

    java 资源 r = new DefaultResource() .refine(“X-Method”, “GET”) .refine(“X-Query”, “/”) .refine(“Accept”, “text/plain”);

    请注意：每次调用.refine()都会返回一个接口Resource的新实例。所有这些实例都是不可变的，就像对象一样。由于这种设计，我们不会将数据与其处理器分开。资源即是数据也是处理器。每个资源都知道如何处理数据，并且只接收它应该接收的数据。从技术上讲，我们只是以面向对象的方式实现了请求分派。

    然后，我们需要将资源转换为响应。我们赋予资源自己将自己渲染到响应的能力。我们不希望数据以某种DTO的形式逃离资源。我们希望资源打印响应。给资源添加一个额外的print()方法如何？

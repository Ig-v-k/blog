    Elegant READMEs

    Il y a quelque temps, j’ai écrit un billet de blog intitulé “An Open Code Base Is Not Yet an Open Source Project” où j’ai suggéré quelques qualités importantes d’un bon référentiel/projet open source. L’une d’entre elles était un fichier README bien rédigé. Ici, je vais essayer de donner quelques conseils sur la manière de créer un bon fichier README et sur les erreurs à éviter. J’espère que vous le trouverez utile.

    Je vais énumérer tout ce dont vous avez besoin dans le fichier README, dans l’ordre dans lequel ces éléments doivent apparaître :

    Pourquoi les inclure ? Le titre est déjà dans l’URL de votre dépôt et la description du projet se trouve dans le sous-titre GitHub. Pourquoi vous répéter ? À la place, commencez par un logo et une liste de badges juste après.

    Faites attention à la ligne vide après le logo. N’oubliez pas de spécifier la taille de l’image dans l’attribut de hauteur. Vous pouvez omettre la largeur, car une seule dimension suffit pour l’HTML. Ne la rendez pas plus grande que 100px en hauteur.

    Vous en avez besoin, comme je l’ai dit plus tôt. Mais la question est de savoir comment les positionner à l’intérieur du README. Vous devriez ne pas mettre plus de cinq badges dans une ligne. Ensuite, séparez les lignes avec une ligne vide. Voyez comment cela est fait dans notre dépôt zold-io/zold.

    Vous devriez les regrouper d’une certaine manière dans les lignes, en utilisant une certaine logique. Je les regroupe par niveau de détails techniques. La première ligne concerne CI, la couverture de code, la qualité du code. La deuxième ligne concerne davantage la gestion, etc. Cela dépend de vous, mais assurez-vous que tous les badges qui restent dans la même ligne ont la même hauteur ! Si un badge a une hauteur différente de tout le reste, donnez-lui sa propre ligne, mais ne mettez jamais deux badges de hauteur différente sur la même ligne !

    Aussi, rappelle-toi que seules dans quelques cas très spécifiques (comme les badges, par exemple), tu es autorisé à avoir des lignes plus longues que 80 caractères. Traite ton document README comme un morceau de code source. Formate-le correctement et élégamment. La largeur de la ligne est l’une de ces règles de formatage qui rendra ton document plus agréable à regarder. Quatre-vingts caractères. C’est tout.

    Votre premier paragraphe après les badges devrait expliquer de quoi parle le produit. Faites attention : un paragraphe, pas une page de texte. Vous devriez mettre la description du produit dans un seul paragraphe. Pas de puces ici, pas de nouvelles lignes, pas d’indentations. Juste un simple morceau de texte :

    Voici le début du corps de votre README. Le contenu exact dépend de votre cas d’affaires spécifique et de la nature de votre produit. Cependant, peu importe le produit, il y a quelques recommandations.

    Premièrement, ne remplacez pas automatiquement les Javadocs générés (ou autres documents similaires dans votre langue) par le README. Le README n’est pas une documentation utilisateur pour tout ce que vous avez dans votre dépôt. Il existe d’autres endroits pour cela. Ce que vous pouvez faire ici, c’est expliquer quelques-uns des cas d’utilisation les plus intéressants. Voyez comment nous l’avons fait dans le projet yegor256/takes.

    Deuxièmement, commencez chaque cas d’utilisation avec un en-tête de deuxième niveau (## en Markdown) et essayez d’éviter les en-têtes de troisième niveau (###). Est-ce que j’ai besoin de dire que le quatrième niveau est absolument interdit ? Et bien sûr, encore une fois, pas d’indentation. Commencez toujours vos lignes de texte à partir de la position tout à gauche, que ce soit un paragraphe de texte ou un en-tête de section.

    Troisièmement, gardez à l’esprit que vous allez bientôt changer votre produit et vous ne voulez pas toujours vous souvenir de l’endroit exact dans le fichier README où vous devez effectuer les modifications. Vous voulez que votre README reste aussi court que possible et aussi général que possible. C’est pourquoi, si possible, évitez les détails spécifiques et redirigez plutôt le lecteur vers les parties pertinentes de la documentation générée automatiquement.

    Quatrièmement, utilisez le moins de mots possible. Personne ne s’intéresse à lire votre prose pendant plus de quelques secondes, et cela uniquement pour comprendre comment utiliser votre produit ou comment effectuer des modifications. Ne vous concentrez pas sur vous-même, nous ne nous intéressons pas à vous. Concentrez-vous sur nous et nos besoins. Expliquez-nous comment cela fonctionne et appelez cela un README. Pas de philosophie, pas de prose. Utilisez votre blog et Twitter pour cela.

    Commencez par un en-tête de section “Comment contribuer” et expliquez brièvement ce qu’il faut faire pour créer une demande d’extraction sur votre dépôt. Imaginez que vous parlez à un développeur junior qui ne sait même pas ce que sont Java et Maven (si votre projet les utilise). Vous devez expliquer comment installer les bons outils, comment construire le projet, comment apporter des modifications, comment l’exécuter en mode de rechargement à chaud (lorsque je fais des modifications et les vois immédiatement à l’écran), comment créer la bifurcation (fork) et à quoi s’attendre lorsque la bifurcation est soumise.

    Ne sois pas trop verbeux. En fait, sois aussi concis que possible. Toujours, lorsque cela est possible, redirige le lecteur vers la documentation de ces outils ou vers des articles de blog qui expliquent mieux. Regarde comment cela est fait dans le README de zold-io/wts.zold.io, une application web écrite en Ruby. Le court texte en bas de la page explique ce qu’il faut installer (en fournissant des liens vers les tutoriels d’installation), comment exécuter l’application en local, comment lancer la construction, comment exécuter un test unitaire unique et que faire si cela ne fonctionne pas. C’est assez concis et, je crois, facile à comprendre.

    Vous ne voulez pas que votre contributeur potentiel s’en aille. C’est pourquoi cette partie de l’ensemble du README est la plus importante. Assurez-vous de vous adresser à un jeune programmeur, pas à vous-même. Comme on dit, votre grand-mère devrait pouvoir vous comprendre ici.

    Et ne nous enseignez pas. Nous ne sommes pas intéressés à devenir des experts dans les frameworks que vous utilisez ou dans Docker, qui est nécessaire pour exécuter vos trucs. Nous voulons simplement exécuter vos trucs, apporter des modifications, obtenir une nouvelle version et partir. Alors, s’il vous plaît, ne me dites pas “d’abord, vous devez apprendre Docker”. Non, je ne le veux pas. Si c’était le cas, je l’aurais déjà fait moi-même. Dites-moi comment l’utiliser dans ce cas spécifique et épargnez-moi le reste.

    GitHub dispose d’un onglet spécial “releases” pour cela. Pas besoin de le répéter dans le README. Assurez-vous simplement que votre onglet “releases” contient suffisamment d’informations et d’artefacts binaires à télécharger. Ne dites pas un mot à leur sujet dans le README.

    GitHub trouve automatiquement votre fichier LICENSE.txt dans le répertoire principal de votre référentiel et comprend la licence. Il vous suffit de créer ce fichier et de ne rien mentionner dans le README à propos de la licence, c’est simplement du bruit. Si je veux savoir quelle est la licence, je sais où cliquer.

    Ceci est quelque chose de l’époque pré-GitHub. Je vous recommanderais de vous appuyer sur l’onglet “releases” et de tout ce que vous voulez nous dire là. Certains anciens projets maintiennent des journaux des modifications, comme celui-ci dans rubocop-hq/rubocop, par exemple. Je ne pense pas que ce soit une bonne idée.

    GitHub a un onglet spécial dans chaque dépôt, appelé “contributors”. Il n’y a absolument aucune raison de reproduire la liste dans le fichier README. Cependant, il y a une raison : aider les contributeurs à se promouvoir. Dans ce cas, je recommanderais de créer une section (avec un titre) appelée “Acknowledgements”, qui devrait répertorier les contributeurs les plus actifs, avec leurs URL de blog, leurs comptes Twitter, et ainsi de suite.

    Si vous n’avez personne à remercier, ne faites pas de bruit en nous disant qui sont les contributeurs. Nous les connaissons, GitHub nous le dit.

    How Much Cohesion Is Enough?

    The following text is a translation of the original article, performed by ChatGPT (gpt-3.5-turbo):

    Lequel est meilleur : books.del(42) ou books.book(42).del() ? Je fais les deux et je peux rarement dire lequel est meilleur. La première option est plus courte, tandis que la deuxième est plus orientée objet. La première option est plus difficile à étendre, tandis que la deuxième est plus verbeuse et nécessite plus de lignes de code, ce qui augmente les chances de faire des erreurs. Laquelle préférez-vous ?

    L’un ou l’autre fonctionnera, bien sûr, mais la question est de savoir quel est le design le plus orienté objet ? Cela semble dépendre de la taille des objets books. S’il est petit, il n’est pas nécessaire de récupérer le livre d’abord, nous pouvons simplement le supprimer là-bas.

    java interface Livres { void create(String title); void delete(int id); void rename(int id, String title); }

    Cependant, s’il est plus grand, il vaut mieux obtenir d’abord le livre.

    java interface Livre { void supprimer(); // Here! void renommer(String titre); } interface Livres { void creer(String titre); int total(); Iterable tous(); Livre tous(String requete); Livre livre(int id); // Here! }

    Y a-t-il une ligne de démarcation claire entre ces deux éléments ? Y a-t-il une règle stricte ou peut-être la question ambiguë “Cette classe est-elle déjà assez grande ?” doit-elle être répondue à chaque fois par un vote ?

    Essayons de donner deux réponses extrêmes : 1) jamais assez grand, et 2) toujours assez grand.

    Si une classe n’est jamais prête à extraire des parties de celle-ci et à les transformer en de nouveaux objets, nous finirons avec une classe très volumineuse, avec de nombreux attributs, méthodes et de longues listes d’arguments dans presque tous les cas.

    Si une classe est toujours suffisamment bonne pour les extractions, nous finirons par avoir de nombreuses petites classes, des méthodes avec presque aucun argument, et … une meilleure conception orientée objet.

    Le dénominateur commun est la cohésion.

    Les classes hautement cohésives comprennent des attributs et des méthodes qui sont liés les uns aux autres, tandis que les classes non cohésives incluent tout ce que leurs développeurs ont décidé d’ajouter, même si certains éléments n’appartiennent pas vraiment ensemble. La première réponse nous donnera une classe avec une très faible cohésion, tandis que la deuxième produira une grande quantité de petites classes hautement cohésives.

    Ainsi, la deuxième option est meilleure ? Oui, c’est le cas. Des classes plus petites, une cohésion plus élevée, … mais plus d’opportunités de perdre le focus et de disperser la fonctionnalité dans trop d’endroits. “Toutes les méthodes dans un objet” est un design beaucoup plus populaire, même s’il est moins cohérent, précisément parce qu’il est plus facile à créer : il suffit de tout mettre au même endroit et d’en finir avec ça. Plus tard, bien sûr, des problèmes de maintenabilité se manifesteront.

    Le point essentiel est qu’il n’y a pas de distinction exacte entre le design correct et incorrect dans ce cas. Nous devons simplement faire de notre mieux pour maintenir des classes hautement cohésives en réduisant le nombre de méthodes dans chacune d’entre elles. S’il y a seulement quelques méthodes, il n’est pas nécessaire d’extraire le Book, mais une fois que le nombre de méthodes devient plus important, le Book est un candidat parfait pour une nouvelle entité à définir.

    How many methods is OK?

    Personne ne sait, mais je vous suggérerais de remettre en question la cohésion de votre classe dès que vous voyez plus de sept méthodes ou plus de quatre attributs. De plus, commencez à réfléchir à la refonte lorsque l’une de vos méthodes (à l’exception des constructeurs) accepte plus de deux arguments.

    How Cookie-Based Authentication Works in the Takes Framework

    The following text is a translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    当您在Facebook登录页面输入您的电子邮件和密码时，您会进入您的帐户。然后，无论您在网站上的任何地方去，您总是会在页面的右上角看到您的照片。Facebook会记住您，不会再次反复要求输入密码。这是通过HTTP cookie实现的，被称为基于cookie的身份验证。尽管这种机制经常引起一些安全问题，但它非常受欢迎和简单。以下是Takes如何通过几行代码实现此功能的方法。

    First, let’s see how it works. Moreover, let’s see how I believe it should work.

    第一步：用户输入电子邮件和密码，然后点击“提交”。服务器接收到包含此信息的POST请求。

    POST / HTTP/1.1
Host: www.facebook.com
Content-Type: application/x-www-form-urlencoded

email=me@yegor256.com&password=itisasecret ```

服务器将提供的信息与其记录进行匹配，并决定如何处理。如果信息无效，服务器将返回相同的登录页面，要求您重新输入。如果信息有效，服务器将返回类似以下内容的信息：

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=me@yegor256.com

    
由于响应状态码为303，浏览器会跳转到Location头部指定的页面，并打开站点的首页。这是它发送给服务器的内容：

    GET / HTTP/1.1 Host: www.facebook.com Cookie: user=me@yegor256.com

    
服务器从 Cookie 头部获取我的电子邮件，并理解这是我！无需再次要求密码。服务器信任来自 Cookie 的信息。就是这样。这就是基于 Cookie 的身份验证的全部内容。

## Wait ... What About Security?

好的，那安全性如何呢？如果服务器信任任何带有用户电子邮件的Cookie头的浏览器请求，任何人都可以从其他地方发送我的电子邮件并获取对我的账户的访问权限。

防止这种情况发生的第一步是使用一个只有服务器知道的秘密加密密钥对电子邮件进行加密。除了服务器本身，没有人能够以与服务器解密所需的方式进行加密。以使用竹子作为秘密密钥的异或密码加密为例，响应将是这样的：

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=b1ccafd92c568515100f5c4d104671003cfa39

    
This is not the best encryption mechanism, though; for proper encryption, it's better to use something stronger like DES.

这听起来都很好，但如果有人劫持服务器和浏览器之间的流量，并获得一个经过适当加密的电子邮件cookie会怎样呢？在这种情况下，即使不知道其内容，窃贼也可以使用同一个cookie进行身份验证。服务器会相信这些信息并允许该人进入我的账户。这种类型的攻击被称为中间人攻击（MITM）。为了防止这种情况发生，我们应该使用HTTPS，并告知浏览器该cookie是敏感的，不应在没有SSL加密的情况下返回给服务器。这是通过Set-Cookie头部中的一个额外标志来实现的：

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=me@yegor256.com; Secure

    
还有一种与基于cookie的身份验证相关的攻击类型，这种攻击基于浏览器的能力，即将与网页关联的所有cookie暴露给其中执行的JavaScript。攻击者可以将一些恶意的JavaScript代码注入到页面中（不要问我如何...只有在您的整个HTML渲染出现错误时才会发生），这段代码将获得对cookie的访问权限。然后，该代码将把cookie发送到其他地方，以便攻击者可以收集它。这种攻击类型称为跨站脚本攻击（XSS）。为了防止这种攻击，还有一个名为HttpOnly的Set-Cookie头的标志。

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=me@yegor256.com; Secure; HttpOnly ```

    这个标志的存在将告诉浏览器，这个特定的 cookie 只能通过 HTTP 请求传输回服务器。JavaScript 无法访问它。

    How It’s Done in Takes

    这是Takes框架中设计的基于cookie的身份验证机制。整个框架由takes组成，它们接收请求并生成响应（本文详细解释了该框架）。当请求进来时，我们应该在Cookie头中找到身份验证cookie并将其翻译为用户凭据。当响应发送出去时，我们应该在其中添加带有加密用户凭据的Set-Cookie头。就是这样，只需要这两个步骤。

    Let’s say we have an account page that is supposed to show the current user’s balance:

    最终类TkAccount实现Take接口{ private final Balances balances; @Override public Response act(final Request request) { final Identity user = // 从请求中获取 return RsHTML( String.format( “<html>您的余额为 %s</html>”, this.balances.retrieve(user) ) ); } }

    请求进来后，我们应该检索用户的身份，该身份已编码在一个身份验证 cookie 中。为了使该机制可重复使用，我们有一个名为 TkAuth 的装饰器，它包装了一个现有的 take（获取用户身份的函数），解码传入的 cookie，并在请求中添加一个新的 TkAuth 头部，其中包含用户的身份信息。

    最终的编解码器为 codec = new CcHex(new CcXOR(new CcPlain()))； 最终的密码为 pass = new PsCookie(codec)； 创建一个 TkAuth(new TkAccount(), pass)。

    当TkAuth再次接收到带有验证cookie的请求时，它会要求传递解码cookie并返回有效的Identity或者Identity.ANONYMOUS。

    然后，当响应返回给浏览器时，TkAuth要求通过对身份进行编码并将Set-Cookie添加到响应中。

    PsCookie uses an instance of Codec in order to do these backward and forward encoding operations.

    当我们的TkAccount希望从请求中获取当前已验证的用户身份时，它可以使用RqAuth，这是Request的一个实用修饰器。

    最终类TkAccount实现Take接口 { @Override public Response act(final Request request) { final Identity user = new RqAuth(request).identity(); // 对用户进行其他操作 } }

    The RqAuth decorator uses the header, added by PsCookie, in order to authenticate the user and create an Identity object.

    How Is It Composable?

    这个机制确实非常可扩展和“可组合”。假设我们想在集成测试期间跳过身份验证。以下是方法：

    新的TkAuth( take, //原始应用程序”take” new PsChain( new PsFake(/如果运行集成测试/), new PsCookie( new CcHex(new CcXOR(new CcPlain())) ) ) );

    PsChain实现Pass并尝试通过逐个询问封装的通行证来验证用户。链中的第一个是PsFake。它在其构造函数中使用一个布尔参数，决定是返回一个假身份还是什么也不返回。只需一个布尔触发器，我们就可以在应用程序中关闭整个身份验证机制。

    Let’s say you want to authenticate users through Facebook OAuth. Here is how:

    新的TkAuth( take, // 原始应用程序 “take” new PsChain( new PsByFlag( new PsByFlag.Pair( PsFacebook.class.getSimpleName(), new PsFacebook( “… Facebook API key …”, “… Facebook API secret …” ) ) ), new PsCookie( new CcHex(new CcXOR(new CcPlain())) ) ) );

    当用户在您的网站上点击登录链接时，浏览器会跳转到facebook.com，验证他/她的身份。然后，Facebook返回一个302重定向响应，并在Location头部中设置为登录链接中提供的URL。该链接必须包含像这样的内容：?PsByFlag=PsFacebook。这将告诉PsByFlag这个请求是用于验证用户身份的。

    PsByFlag将遍历所有封装的“pair”并尝试找到正确的一个。PsFacebook将是第一个也是正确的一个。它将使用提供的凭据连接到Facebook API，并检索有关用户的所有可能信息。

    Here is how we can implement a logout mechanism:

    新的TkAuth( take, //原始应用程序”take” new PsChain( new PsByFlag( new PsByFlag.Pair( PsFacebook.class.getSimpleName(), new PsFacebook( “… Facebook API key …”, “… Facebook API secret …” ) ), new PsByFlag.Pair( PsLogout.class.getSimpleName(), new PsLogout() ) ), new PsCookie( new CcHex(new CcXOR(new CcPlain())) ) ) );

    Now, we can add ?PsByFlag=PsLogout to any link on the site and it will log the current user out.

    You can see how all this works in a real application by checking out the TkAppAuth class in Rultor.

    XSL转换（XSLT）是将一个XML文档转换成另一个的强大机制。然而，在Java中，XML操作非常冗长和复杂。即使是对于一个简单的XSL转换，您也需要编写几十行代码，如果需要适当的异常处理和日志记录，甚至可能更多。jcabi-xml是一个小型的开源库，通过几个简单的方法实现了XML解析和XPath遍历，使生活变得更加轻松。让我们看看这个库如何帮助进行XSL转换。

    首先，让我们看一个实际的例子——rultor.com——一个托管的DevOps助手，可以自动化发布、合并和部署操作。Rultor将与最终用户（也称为“对话”）的每个对话会话保存在DynamoDB记录中。每个对话中有多种情况需要处理，这就是为什么在记录的多个列中使用多个列并不真正可行的原因。相反，我们只保留每个对话的几个参数（如ID和名称）在记录列中，并将其余所有参数放在一个单独的XML列中。

    一旦用户在GitHub的工单中发布了 @rultor 的状态，Rultor就必须用一个完整的状态报告来回应当前的对话。为了创建这样的文本回答（普通用户不会喜欢XML响应），我们需要从必要的DynamoDB记录中获取那个XML列并将其转换为简单的英文文本。

    这是我们如何通过jcabi-xml和它的类XSLDocument来实现的。

    创建每个应用程序运行时只有一次XSL文档是一个好的实践。我们有一个静态实用方法XSLDocument.make()来实现这一点：

    java final class Foo { private static final XSL STYLESHEET = XSLDocument.make( Foo.class.getResourceAsStream(“stylesheet.xsl”) ); public XML style(final XML xml) { return Foo.STYLESHEET.transform(xml); } }

    请注意我们正在使用XSLT 2.0。内置的Java XSLT实现不支持2.0版本，为了使其正常工作，我们使用了这两个Maven Saxon依赖项：

    xml net.sourceforge.saxon saxon 9.1.0.8 runtime net.sourceforge.saxon saxon 9.1.0.8 xpath runtime

    xml net.sourceforge.saxon saxon 9.1.0.8 运行时 net.sourceforge.saxon saxon 9.1.0.8 xpath 运行时

    开始使用jcabi-xml进行XSL转换，您只需要将此依赖项添加到您的pom.xml文件中即可。

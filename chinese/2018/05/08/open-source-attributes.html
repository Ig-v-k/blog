    An Open Code Base Is Not Yet an Open Source Project

    The following text is a translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    几周前有人建议我尝试将IntelliJ IDEA的静态分析规则整合到我们的聚合器Qulice中，其中包括Checkstyle、PMD、FindBugs和其他一些分析器。我确实喜欢IDEA的规则，其中一些是独特且非常有用的。我问是否可以在Maven Central找到它们（它们是用Java编写的），答案是：“你必须自己找出如何使用它们，但它们是开源的。”接下来是我对这种情况的看法：我相信开源不仅仅意味着代码可读，不需要授权。它意味着更重要的东西。

    只是将一段代码公开并不足以称其为开源软件。实际上，如果开放但还不具备重用条件（这正是开源世界的全部意义），这只会损害产品和其作者的声誉。正如埃里克·雷蒙德在他著名的作品《大教堂与集市》中所说：“优秀的程序员知道该写什么。伟大的程序员知道该重写（和重用）什么。”

    软件产品的作者有责任帮助那些“优秀”的程序员重复使用代码。编码、测试、调试以及确保“在我的笔记本上能运行”是一回事。而使代码可读性好且可重用则是完全不同的工作，可能需要更多的时间。

    正如Karl Fogel在《开源软件的生产》中所说：“大多数自由软件项目都会失败。”它们之所以失败（除了其他很多因素），是因为没有足够关注以下基本事项（无特定顺序）：

    README

    我确定你将你的产品托管在GitHub上。（如果没有，你怎么了？）在根目录下肯定有一个README文件，解释产品的内容以及我们如何使用它。一些好的例子包括：leejarvis/slop, mongobee/mongobee, ronmamo/reflections和yegor256/takes（这是我的）。一些不好的例子包括：qos-ch/slf4j, rzwitserloot/lombok和junit4/blob（不要像这些人那样）。

    无论你的网站、Javadoc、Wiki、邮件列表和Twitter有多么丰富，README 是我们期望看到所有内容的地方。只有在我们产生兴趣时，我们才会进一步深入调查。阅读其他项目中的 README 文件，并复制它们最好的想法。README 是你的展示窗口，必须发光发亮。

    License

    我们大多数人都不关注这个官僚主义。直到最近，我也没有关注过。我以为一旦我的代码开放，我就可以忘记任何权益和版税了。他们只会使用我的代码，我永远不会看到任何利润。我附加的许可证毫无意义——反正没有人会读它。这正是在大多数情况下发生的事情。但只有在那些用户还很菜鸟的时候。

    几年前，我在一个软件项目中担任建筑师的角色，我们需要创建一个硬件组件分析器，如CPU、内存、硬盘等等。我们需要确保在运行相当复杂和定制化的测试后，所有这些组件都能正常工作。我显而易见的建议是使用开源工具，它们可以为我们完成繁重的工作。我们只需要进行集成。这是一个非常棒的想法，直到我们中的一些人决定检查那些工具的许可证。

    那是我意识到我犯了一个错误，没有关注许可证所说的内容的时候。例如，我们在一些工具中发现了GPL，如果我们的产品不是开源的，那么它就不允许我们重新使用代码。由于我们正在创建的是专有软件，我们明白我们不能使用类似于Copyleft的模块，只能使用MIT、BSD或类似的许可证。

    I’m suggesting you think about the license before publishing the product. I’ve used MIT in all my products since 2016.

    Distribution

    一堆.rb文件只是无法重复使用的Ruby代码。或许对于那些我非常鄙视的黑客来说，它可能是可以的。但对于那些懒得读自己代码，更别提别人代码的专业开发人员来说，它绝对行不通。

    “从GitHub获取”已经不再是对待我们——你的程序员同行——的一种礼貌方式了。这在二十年前或许还可以，但是现在我们有了代码仓库。你必须通过其中一个公共代码仓库将你的产品作为一个”构件”分发出去，并让我们能够从那里获取到它，跳过测试和打包的过程，直接将其作为一个产品使用（例如一个Ruby gem或Java JAR）。

    我正在谈论像Maven Central、npmjs或RubyGems这样的仓库。你必须找到一种方法将你的产品部署到这些仓库中。尽管这些仓库尽力简化过程，但这并不是一项容易的任务。我们在所有的项目中都使用Rultor来帮助我们优化部署流程。

    yegor256/takes to Maven Central (details);

    yegor256/xcop to RubyGems (details);

    yegor256/tacit to npmjs.

    像Maven、NPM、Rake、Grunt、Gradle等包管理器是重用开源软件（包括专有软件）的标准和传统方式。如果您的产品不在公共存储库中，那它就不是一个产品，只是一个代码库而已。

    Javadoc

    我们都讨厌写文档。而且我们讨厌没有文档的库。我通常觉得为我的类写Javadoc块很无聊，但我明白如果没有它们，我在这些类中编写的代码将不会引起任何人的兴趣。

    对于那些Javadoc块来说，最好的格式是“通过示例”。我建议您展示如何使用类，尤其是与其邻居类的组合。此外，我建议您不要在Javadoc块之外的任何地方编写文档。（其他语言中也有类似的块，但名称不同。）

    Javadoc的问题在于很难以使文本格式看起来视觉上有吸引力。也许这就是为什么许多程序员仍然依赖于维基或项目网站的原因。我建议您保持在Javadoc块内，并学习它们的格式语法。

    Badges

    正如你所看到的，我喜欢徽章。首先，它们让一个代码库看起来像是在“积极维护”，尤其是那些绿色的徽章。它们并没有提供任何有价值的信息。它们主要传达的是：“我们的作者非常有品味，看看我们的颜色是多么完美搭配！”

    开玩笑的话，要添加所有这些徽章并不容易。每个徽章都需要一些时间，来整合第三方系统，确保数字足够令人自豪，并且保持其可控性。如果代码库没有受到监管，徽章最终将开始失效。

    Continuous Integration

    为了使用你的代码，我们必须相信它，也就是说我们必须确保它能够正常工作，或者至少通过自动化测试。（我需要说你必须有测试吗？）我们如何确保它能正常工作？持续集成就是答案。我们必须能够查看最近的持续集成构建日志，并确保其干净无误。

    这是一种信任问题。你可能从未使用那些Travis构建，也可能忽略了它们的红色和绿色信号，但它们对我们来说很重要——对你的客户来说也是如此。我在创建新仓库后立即为我的所有项目添加Travis徽章。

    Contribution Guidelines

    对于一个经常使用GitHub的上瘾者来说，弄清如何向你发送一个拉取请求并不是个问题。然而，我们大多数人，至少起初，都是活跃用户而不是贡献者。我们会尝试使用你的产品，并试图根据我们的需求进行定制。如果我们迷失了方向，我们会感到沮丧并离开。

    为了防止这种情况发生，您必须解释一个有纪律的贡献者在更改您的代码库时应该做些什么。以下是我建议在您的 CONTRIBUTING.md 中回答的问题：

    How do I run an automated build?

    How big/small does a pull request have to be in order to be accepted?

    What are your style guidelines?

    How do bugs have to be reported, tagged, explained?

    What makes a good bug report?

    Here is the text I use in all my projects: ISSUE_TEMPLATE.md and PULL_REQUEST_TEMPLATE.md.

    Quality Wall

    最后，如果你运气好的话，我们会使用你的产品并对其做出贡献感兴趣。你会开始收到我们的拉取请求。问题是如果你不保护自己，我们会多快毁掉你的代码库。

    如果您严格审查每个拉取请求，并拒绝任何看起来不像“优秀”代码的东西，您将失去我们，也就是您的贡献者。我们不想编写优秀的代码，我们想要对您的产品进行改变，使其更适合我们的需求。代码的优秀程度是您的问题，而不是我们的问题。

    另一方面，如果你接受任何东西的话，这个架构将失去其强大性（如果它曾经有的话），你将再次失去我们，即你的贡献者。这一次你将失去我们，因为产品将变得糟糕且难以维护和贡献。

    保持平衡的最佳方法是“雇佣”一个工具来帮助你：建立自动化、静态分析、自动化测试和覆盖控制。当某人引入的更改违反其内部质量期望时，您必须配置产品以失败。我也使用Rultor来实现这一点。

    Did I forget anything?

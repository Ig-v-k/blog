    Builders and Manipulators

    The following text is a translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    以下是一个简单的面向对象编程（OOP）方法命名原则，我正在努力在我的代码中遵循它：如果方法是操作性的，命名应为动词；如果方法是构建性的，命名应为名词。就是这样，没有中间地带。像saveFile()或getTitle()这样的方法不适用，必须重命名和重构。此外，“操作性”方法必须始终返回void，例如print()或save()。让我解释一下。

    首先，我必须说这个想法与Bertrand Meyer在他的书《面向对象软件构造》中提出的建议非常相似，他建议我们将一个对象的方法划分为两个明显分离的类别：查询和命令。

    这个原则背后的思想相当哲学化。让我们从建造者开始，他们被认为是要创造或找到一个物体然后归还它。假设我有一家书店，我要求它按照书名给我一本书：

    java接口图书架 { Book find(String title); }

    这显然是一个“构造器”（或者按梅耶的术语，是一个“查询”）。我要求一本书，然后就被给了。然而，问题在于方法的名称。它被称为“find”，这意味着我知道这本书将如何处理。它将被找到。

    然而，这不是我们应该对待我们的对象的方式。我们不能告诉它们如何完成我们想要它们完成的工作。相反，我们必须让它们决定书籍是被找到、构建，还是从内存缓存中获取。当我们查询时，我们必须说明我们要寻找的结果，并让对象决定如何构建这个结果。这种方法更合适的名称应该是book()。

    请将以下段落从英文翻译成中文，不要更改专有名词：

    java接口书架 { 书 书(String 标题); }

    准则是：建造者总是一个名词。如果方法返回内容，那它必须是一个名词。最好的情况是，方法的名称应该解释方法返回的内容。如果它是一本书，就称其为book()。如果是一个文件，就称其为file()，等等。以下是一些好的建造者示例：

    java接口Foo { float speed(Actor actor); Money salary(User user); File database(); Date deadline(Project project, User user); }

    在这里，相反地，有几个不恰当命名的建筑商的例子：

    Java接口Foo{ float calculateSpeed(Actor actor); Money getSalary(User user); File openDatabase(); Date readDeadline(Project project, User user); }

    在建筑师的名字中没有动词的位置！

    顺便提一下，并不仅仅是关于名称的问题。一个构造器，由于其名称不包含动词，不应对封装的实体进行任何修改。它只能创建或查找某些东西并返回。就像纯函数一样，它不能有任何副作用。

    接下来，有“操纵器”（或者在迈耶的术语中称为“命令”）。它们为我们做一些工作，修改由对象封装的实体。它们与建造者相反，因为它们实际上对对象所抽象的世界进行改变。例如，我们要求书架将一本新书添加到自身中：

    java接口 Bookshelf { void add(Book book); }

    该方法将书籍添加到存储中。存储将如何被修改我们不清楚。但我们知道由于该方法的名称是一个动词，将会进行修改。

    此外，操纵者不能返回任何东西。我们通常将其响应视为空值类型。这主要是为了将代码中的命令式部分与陈述式部分分离开来。我们要么接收对象，要么告诉它们该做什么。我们不能在一个方法中混合这些活动。

    这些规则的目的是使代码更简单。如果你遵循这些规则，且你的所有构建者只返回对象，你的操作者只修改世界，整个设计将变得更容易理解。方法会更小，名称也会更短。

    当然，很多时候你会很难找到那些名字。不时地，你会想要从一个操作者那里拿回一些东西，或者让你的构建者做一些改变，比如说缓存。尽量抵制这种诱惑，坚持原则：一个方法要么是构建者，要么是操作者，中间没有什么。上面的例子相对简单，实际生活中的代码要复杂得多。但这就是原则将帮助我们的地方——让代码变得更简单。

    我也了解名词/动词原则，该原则建议将类的命名总是以名词形式，将它们的方法以动词形式命名。我认为这是一个错误的观点，因为它无法区分建造者和操作者，并鼓励我们始终以命令式指令思考。我相信面向对象编程(OOP)应该更多地涉及对象的声明性组合，即使我们有时需要从其他对象获取它们，而不是通过构造函数进行实例化。这就是为什么在大多数情况下我们需要建造者，同时我们也必须明显区分它们和其他方法——操作者之间的差别。

    Objectionary: Dictionary and Factory for EO Objects

    The following text is a translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    自从Kernighan和Ritchie的时代，我们在库中共享二进制代码。你需要使用C++中的printf()打印一些文本吗？你可以使用包含700多个其他函数的libc库。你需要复制一个Java流吗？你可以使用Apache Commons IO中的copy()和其他140多个方法和类。在我所了解的所有语言中都是如此，比如Ruby、Python、JavaScript、PHP：你需要一个对象、一个类、一个函数或一个方法—你必须将整个库添加到你的构建中。与之相比，处理单个对象不是更加优雅吗？

    这个想法并不是新的，也不是我的。我是从《对象思考》这本书中得到的，作者是大卫·韦斯特（David West）。他建议创建一个“Objectionary”（第306页），是一个“字典和对象工厂的结合”，具有以下特性：

    总物体数目少于2000个；

    每个对象都是一个自主的可执行实体；

    每个物体都有一个独特的ID和一个独特的“地址”。

    物体只不过是物体的集合。

    物体需要专门的硬件虚拟机来执行。

    十七年后（该书出版于2004年），我们在我们的新编程语言EO之上实施了这个想法。这种语言故意比Java或C++简单得多。您可以在这里阅读它更多或更少正式的描述。

    将EO程序转化为可执行实体并发布到Objectionary，需要按照以下强制步骤进行，假设JVM被用作目标平台（标有🌵的步骤是由我们的eo-maven-plugin实现的）：

    Assemble🌵:

    Parse🌵: .eo ➜ .xmir

    Optimize🌵: .xmir ➜ better .xmir

    发现🌵：找到所有的外国化名

    拉🌵：下载外国.eo对象

    解决方案🌵：下载并解压缩.jar文件。

    位置🌵: 将artifact .class文件移动到target/classes/目录下。

    马克🌵：将在.jar文件中找到的mark .eo资源标注为外来的。

    ↑ Go back to Parse if some .eo files are still not parsed

    Transpile🌵: .xmir ➜ .java

    Assemble🌵：与上述相同，但用于测试。

    Compile: .java ➜ .class

    Test: run all unit tests

    Unplace🌵：移除 artifact .class 文件

    解开🌵：删除自动生成的 .java 文件

    复制🌵：将.eo文件复制到.jar内的EO-SOURCES/文件夹中。

    部署：将.jar文件打包并放入Maven中央仓库。

    推动：向yegor256/objectionary发送拉取请求

    合并：我们测试并合并拉取请求。

    这是一个迭代过程，一次又一次地循环，直到所有需要的.eo对象被解析并且它们的原子以.class文件的形式存在。然后，所有的.xmir文件被转译为.java，然后编译为.class二进制文件。然后，进行测试，打包，并部署到Maven中央仓库。然后，通过拉取请求合并到Objectionary的主分支。

    算法的第一部分可以通过我们的Maven插件自动化完成，只需将.eo源文件放置在src/main/eo/目录下，并将其添加到pom.xml中即可。

    xml org.eolang eo-maven-plugin register assemble transpile copy unplace unspile [...] [...]

    注册目标将扫描src/main/eo/目录，找到所有.eo源文件，并将它们“注册”到target/eo-foreigns.csv特殊CSV目录中。接下来，组装目标将调用以下目标：解析、优化、发现、拉取和解决。当这些目标进行解析、优化、拉取等操作时，都使用CSV目录。

    当所有文件都完成后，assemble检查目录：是否仍然需要解析任何.eo文件？如果需要，将开始另一个循环，再次进行解析。当所有.eo文件都被解析后，会执行目标转换(transpile)操作，将.xmir文件转换为.java文件，并将它们放置在target/generated-sources目录下。剩下的工作由标准的maven-compiler-plugin完成。

    让我们详细讨论每个步骤。

    请看，这是位于src/main/eo/hello.eo的.eo源代码。

    ```text +alias org.eolang.io.stdout

    [] > hello “Jeff” > user stdout > @ “Hello, %s!” user ```

    它将被解析为XMIR（XML中间表示）：

    xml Jeff 你好，%s！

    如果你想知道这个XML是什么意思，阅读这份文档：有一个关于XMIR的部分。

    在这一步骤中，由解析器生成的XMIR经过多次XSL转换，有时会获得额外的元素和属性。我们的示例XMIR可能会获得一个新属性@ref，将引用指向定义对象的行。

    xml Jeff 你好，%s！

    一些XSL转换可能会检查语法或语义错误，并且如果发现错误，会添加一个新元素。因此，如果解析没有发现任何语法错误，所有其他错误将在XMIR文档中可见，例如像这样：

    xml <error line=>该程序没有包</error> Jeff 你好，%s！

    顺便说一下，这不是一个真正的错误，我只是编造出来的。

    在这一步中，我们找出哪些对象是“外部的”。在我们的例子中，对象“user”不是外部的，因为它在我们面前的代码中定义了，而对象“stdout”在这里没有定义，所以它是一个外部对象。

    浏览所有的.xmir文件，我们可以通过查看它们的名称轻松判断哪个对象是外部的。一旦我们看到对org.eolang.io.stdout的引用，我们就会检查包含所有.eo源文件的目录中是否存在org/eolang/io/stdout.eo文件。如果该文件不存在，我们将该对象名称放入CSV目录，并宣称它是外部的。

    在这里，我们只是试图通过查看其GitHub存储库，找到Objectionary中所有外部对象的源代码.eo文件。例如，这是我们找到stdout.eo的地方。我们在那里找到它们并拉取到本地磁盘上。

    请注意，我们提取的是源代码。不是二进制文件或编译后的XMIR文档，而是以.eo格式的源代码。

    这是在拉取之后stdout.eo可能看起来像的样子。

    ```xml +package org.eolang.io +rt jvm org.eolang:eo-runtime:0.10.2

    [text] > stdout /bool ```

    这个对象是一个原子。这意味着即使我们有它的源代码，没有特定平台的二进制代码，它也是不完整的。原子是由运行平台实现的对象，EO程序在其中执行（也称为FFI机制）。以+rt（运行时）开头的那一行解释了在哪里获取运行时代码。jvm部分是运行时的名称。

    我们前往Maven中央库，在那里找到了org.eolang:eo-runtime:0.10.2的构件，并解压它（毕竟它是一个带有.class文件的zip归档文件）。

    顺便说一下，一个程序可能包含多个+rt元指令，例如：

    ```text +package org.eolang.io +rt jvm org.eolang:eo-runtime:0.10.2 +rt ruby eo-core:0.5.8 +rt python eo-basics:0.0.3

    [text] > stdout /bool ```

    在这里，三个运行时平台将知道如何获取stdout原子的缺失代码：EO➝Java将前往Maven Central获取JAR构件，EO➝Ruby将前往RubyGems尝试通过名称eo-core和版本0.5.8找到宝石，而EO➝Python将前往PyPi尝试找到版本为0.0.3的eo-basics软件包。

    接下来，我们将在解压的JAR中找到的所有.class文件放入目标/classes目录中。我们这样做是为了帮助Maven编译器插件在类路径中找到它们。

    每个到达的JAR文件中都可以找到.eo源文件。它们是在构建该JAR文件时存在于类路径中的程序。我们也将它们视为外部对象，并添加到CSV目录中。

    当目录中注册的所有外部对象都被下载、编译和优化后，我们就准备开始转译。我们不直接将XMIR编译成字节码，而是将其转译为.java文件，并让Java编译器来生成字节码。

    输出代码更容易阅读和调试。

    在这一步骤中，标准的Maven编译器插件会在target/generated-sources目录中找到自动生成的.java文件，并将其转换为.class文件。

    这里，我们删除从依赖项中解压出的所有.class文件。这是必要的，以避免将它们打包到最终的JAR文件中。

    我们之所以要进行放置和取消放置，只是因为Maven编译器插件不允许我们在运行时扩展类路径。如果可能的话，我们只需从Maven中央仓库下载依赖项并将它们添加到类路径中，而无需解压、放置和取消放置。

    在这里，我们从target/classes/目录中删除了所有由.eo自动生成的.class文件。我们不想发布由.eo源代码生成的二进制文件。我们只想发布由原始的.java文件生成的原子文件。

    在这一步中，我们将从src/main/eo/目录中获取所有的.eo源文件，并将它们复制到target/classes/EO-SOURCES/目录中。稍后，它们将与.class文件一起打包成一个.jar文件，并部署到Maven中央仓库。在复制过程中，我们将运行时版本中的0.0.0替换为当前部署的版本。请看一下stdout.eo文件，在它的源代码库中。

    ```text +package org.eolang.io +rt jvm org.eolang:eo-runtime:0.0.0

    [text] > stdout /bool ```

    +rt行的版本为0.0.0。当源代码被复制到JAR文件时，此文本将被替换。

    将源代码与二进制文件一起发布的动机如下。当原子二进制文件从Java编译为字节码时，它们会与转译后的源代码放在一起。它们一起进行编译。此外，单元测试还依赖于原子源代码和自动生成/转译的源代码。我们希望JAR的未来用户知道在编译过程中我们放置的源代码，也许让他们能够重现它，或者至少知道他们获取的二进制文件的环境。

    从更实际的角度来看，我们需要在JAR中有这些资源，以便让Mark步骤理解哪些对象值得与已解决的原子一起提取。

    在这里，我们将目标/类文件夹中的所有内容打包成一个JAR归档文件，并部署到Maven Central。

    我建议将资源部署到GitHub Pages上，让用户可以在网页上看到它们。此外，当我们向Objectionary发起pull request时，这也将会很有帮助。请检查我EO库中的.rultor.yml脚本，它会正确地将.eo资源部署到GitHub Pages上，并替换其中的0.0.0版本标记。

    当部署完毕并且Maven Central更新其CDN服务器后，就是提交拉取请求到yegor256/objectionary的时候了。对象的.eo源代码放在objects/目录中，它们的单元测试放在tests/目录中。基本上，我们只需将src/main/eo/和src/test/eo/复制到那里。但是，停下…有一个重要的细节。在源代码中，正如前面所说，我们将+rt版本设置为0.0.0。在这里，当我们复制到Objectionary时，版本必须设置为实际的数字。

    当拉取请求到达时，yegor256/objectionary存储库中预先配置的GitHub Action会将所有.eo源代码转译为所有已知平台，并运行所有单元测试。如果一切正常，我们会审查该拉取请求，并决定所建议的对象是否与Objectionary中已有的其他对象相符。

    一旦合并拉取请求，这些对象将成为EO所有对象的集中字典的一部分。请查看此拉取请求，其中一个新对象被提交到Objectionary，在其原子部署到Maven Central之后。

    Single Statement Unit Tests

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    已经有很多关于单元测试模式和反模式的文章和书籍被写出来了。我想再添加一个建议，我相信这个建议可以帮助我们使我们的测试和生产代码更加面向对象。这就是：一个测试方法必须只包含一个单一的assert语句。

    请看来自OpenJDK 8的RandomStreamTest的这个测试方法，创建者是Brian Goetz。

    @Test
public void testIntStream() {
  final long seed = System.currentTimeMillis();
  final Random r1 = new Random(seed);
  final int[] a = new int[SIZE];
  for (int i=0; i < SIZE; i++) {
    a[i] = r1.nextInt();
  }
  final Random r2 = new Random(seed);
  final int[] b = r2.ints().limit(SIZE).toArray();
  assertEquals(a, b);
}

    这个方法分为两个部分：算法和断言。算法准备了两个整数数组，而断言则比较它们，如果它们不相等就会抛出AssertionError。

    我是说，我们应该尽量避免第一部分，即算法。我们必须拥有的唯一东西是断言。以下是我如何重新设计这个测试方法：

    @Test
public void testIntStream() {
  final long seed = System.currentTimeMillis();
  assertEquals(
    new ArrayFromRandom(
      new Random(seed)
    ).toArray(SIZE),
    new Random(seed).ints().limit(SIZE).toArray()
  );
}
private static class ArrayFromRandom {
  private final Random random;
  ArrayFromRandom(Random r) {
    this.random = r;
  }
  int[] toArray(int s) {
    final int[] a = new int[s];
    for (int i=0; i < s; i++) {
      a[i] = this.random.nextInt();
    }
    return a;
  }
}

    如果Java有标识符，这段代码会显得更加优雅。

    @Test
public void testIntStream() {
  assertEquals(
    new ArrayFromRandom(
      new Random(System.currentTimeMillis() as seed)
    ).toArray(SIZE),
    new Random(seed).ints().limit(SIZE).toArray()
  );
}

    正如你所见，这个方法中只有一个“statement”：assertEquals()。

    Hamcrest 与其assertThat() 以及其 collection 的基本匹配器，是使我们的单语句测试方法更加内聚和可读的完美工具。

    如果我们同意遵循这个原则，将会有许多实际的好处。

    • 可重用性。我们将需要创建的用于测试断言的类将在其他测试方法和测试用例中可重用。就像上面的例子中，ArrayFromRandom 可以在其他地方使用一样。类似地，Hamcrest 匹配器可能会成为一个可重用测试组件的库。

    • 简洁性。由于只有一个assert时很难创建一个长的测试方法，你和其他程序员都会不可避免地编写更短、更易读的代码。

    • 易读性。通过一个单独的 assert，测试方法的意图将始终明显。它将以意图声明开始，而所有其他较低级别的细节将被缩进。

    • 不变性。如果测试方法没有算法代码的位置，那么在生产代码中几乎不可能有setter。您不可避免地会创建不可变对象，以便通过单个assert来进行测试。

    当将这个原则应用于我们的测试时，我们获得的最大好处是它们变得声明性和面向对象，而不是算法化、命令式和过程化。

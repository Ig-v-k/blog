    Constructors or Static Factory Methods?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我相信Joshua Bloch在他非常好的书籍《Effective Java》中首先提到：与构造函数相比，静态工厂方法是实例化对象的首选方式。我不同意。不仅因为我认为静态方法是纯粹的邪恶，而且主要是因为在这种特殊情况下，它们假装成好的，并让我们认为我们必须喜欢它们。

    让我们从面向对象的角度来分析推理并看看为什么它是错误的。

    这是一个具有一个主构造函数和两个次构造函数的类：

    class Color {
  private final int hex;
  Color(String rgb) {
    this(Integer.parseInt(rgb, 16));
  }
  Color(int red, int green, int blue) {
    this(red << 16 + green << 8 + blue);
  }
  Color(int h) {
    this.hex = h;
  }
}

    这是一个类似的类，有三个静态工厂方法：

    class Color {
  private final int hex;
  static Color makeFromRGB(String rgb) {
    return new Color(Integer.parseInt(rgb, 16));
  }
  static Color makeFromPalette(int red, int green, int blue) {
    return new Color(red << 16 + green << 8 + blue);
  }
  static Color makeFromHex(int h) {
    return new Color(h);
  }
  private Color(int h) {
    this.hex = h;
  }
}

    你更喜欢哪一个？

    根据Joshua Bloch的说法，使用静态工厂方法而不是构造函数有三个基本优点（实际上有四个，但第四个在Java中不适用了）。

    • They have names.

    • They can cache.

    • They can subtype.

    我相信这三个都是合理的……如果设计有问题的话。它们是绕过问题的好借口。让我们逐个来看。

    They Have Names

    这是使用构造函数创建一个 红色番茄 颜色对象的方式。

    Color tomato = new Color(255, 99, 71);

    这是使用静态工厂方法的做法。

    Color tomato = Color.makeFromPalette(255, 99, 71);

    似乎makeFromPalette()比仅使用new Color()更具语义丰富性，对吗？是的。如果我们只是将这三个数字传递给构造函数，谁知道它们代表什么意思呢？但是，”palette”这个词立即帮助我们理解一切。

    True.

    然而，正确的解决方案是使用多态性和封装，将问题分解成几个语义丰富的类：

    interface Color {
}
class HexColor implements Color {
  private final int hex;
  HexColor(int h) {
    this.hex = h;
  }
}
class RGBColor implements Color {
  private final Color origin;
  RGBColor(int red, int green, int blue) {
    this.origin = new HexColor(
      red << 16 + green << 8 + blue
    );
  }
}

    现在，我们使用正确的类的正确构造函数：

    Color tomato = new RGBColor(255, 99, 71);

    See, Joshua?

    They Can Cache

    假设我需要在应用程序的多个地方使用红色番茄颜色。

    Color tomato = new Color(255, 99, 71);
// ... sometime later
Color red = new Color(255, 99, 71);

    将创建两个对象，这显然是低效的，因为它们是相同的。最好将第一个实例保存在内存中，并在第二次调用时返回它。静态工厂方法可以解决这个问题：

    Color tomato = Color.makeFromPalette(255, 99, 71);
// ... sometime later
Color red = Color.makeFromPalette(255, 99, 71);

    然后，在Color内部的某个地方，我们保留一个私有静态Map，其中包含了所有已经实例化的对象。

    class Color {
  private static final Map<Integer, Color> CACHE =
    new HashMap<>();
  private final int hex;
  static Color makeFromPalette(int red, int green, int blue) {
    final int hex = red << 16 + green << 8 + blue;
    return Color.CACHE.computeIfAbsent(
      hex, h -> new Color(h)
    );
  }
  private Color(int h) {
    return new Color(h);
  }
}

    它在性能方面非常有效。对于像我们的Color这样的小对象，问题可能不太明显，但是当对象更大时，它们的实例化和垃圾回收可能会浪费很多时间。

    True.

    然而，有一种面向对象的方法可以解决这个问题。我们只需要引入一个新的类Palette，它成为了颜色的存储库：

    class Palette {
  private final Map<Integer, Color> colors =
    new HashMap<>();
  Color take(int red, int green, int blue) {
    final int hex = red << 16 + green << 8 + blue;
    return this.computerIfAbsent(
      hex, h -> new Color(h)
    );
  }
}

    现在，我们实例化Palette一次，并在每次需要时向其请求返回一种颜色。

    Color tomato = palette.take(255, 99, 71);
// Later we will get the same instance:
Color red = palette.take(255, 99, 71);

    看，Joshua，没有静态方法，也没有静态属性。

    They Can Subtype

    假设我们的类Color有一个方法lighter()，它应该将颜色转变为下一个可用的较浅颜色：

    class Color {
  protected final int hex;
  Color(int h) {
    this.hex = h;
  }
  public Color lighter() {
    return new Color(hex + 0x111);
  }
}

    然而，有时更希望通过一组可用的Pantone颜色选择下一个较浅的颜色。

    class PantoneColor extends Color {
  private final PantoneName pantone;
  PantoneColor(String name) {
    this(new PantoneName(name));
  }
  PantoneColor(PantoneName name) {
    this.pantone = name;
  }
  @Override
  public Color lighter() {
    return new PantoneColor(this.pantone.up());
  }
}

    然后，我们创建一个静态工厂方法，该方法将决定哪种 Color 实现对我们来说最合适。

    class Color {
  private final String code;
  static Color make(int h) {
    if (h == 0xBF1932) {
      return new PantoneColor("19-1664 TPX");
    }
    return new RGBColor(h);
  }
}

    如果请求的是真正的红色，我们会返回一个PantoneColor的实例。在其他所有情况下，它只是一个标准的RGBColor。这个决定是由静态工厂方法来做出的。我们将这样调用它：

    Color color = Color.make(0xBF1932);

    不可能使用构造函数来执行同样的“分叉”，因为它只能返回它所声明的类。静态方法具有返回Color的任何子类型的必要自由。

    True.

    然而，在面向对象的世界中，我们可以并且必须以完全不同的方式进行。首先，我们将使Color成为一个接口：

    interface Color {
  Color lighter();
}

    接下来，我们将把这个决策过程移动到它自己的类Colors中，就像我们在之前的例子中所做的一样。

    class Colors {
  Color make(int h) {
    if (h == 0xBF1932) {
      return new PantoneColor("19-1664-TPX");
    }
    return new RGBColor(h);
  }
}

    并且我们将使用Colors类的一个实例，而不是在Color内部使用静态工厂方法。

    colors.make(0xBF1932);

    然而，这仍然不是真正的面向对象的思维方式，因为我们将决策从它所属的对象中分离出来。无论是通过静态工厂方法make()还是新的类Colors—不管怎样—我们将对象分成了两部分。第一部分是对象本身，第二部分是决策算法，它留在其他地方。

    一个更加面向对象的设计是将逻辑放入一个名为PantoneColor的类的对象中，这个对象会装饰原始的RGBColor。

    class PantoneColor {
  private final Color origin;
  PantoneColor(Color color) {
    this.origin = color;
  }
  @Override
  public Color lighter() {
    final Color next;
    if (this.origin.hex() == 0xBF1932) {
      next = new RGBColor(0xD12631);
    } else {
      next = this.origin.lighter();
    }
    return new PantoneColor(next);
  }
)

    然后，我们创建一个 RGBColor 的实例，并使用 PantoneColor 进行装饰。

    Color red = new PantoneColor(
  new RGBColor(0xBF1932)
);

    我们要求red返回一个较浅的颜色，它返回的是Pantone调色板中的颜色，而不是仅在RGB坐标上较浅的颜色。

    Color lighter = red.lighter(); // 0xD12631

    当然，这个例子相当原始，如果我们真的希望它适用于所有的 Pantone 颜色，还需要进一步改进，但我希望你明白我的意思。逻辑必须保持在类的内部，而不是在外部的某个地方，也不是在静态工厂方法中，甚至不是在其他辅助类中。当然，我说的是属于这个特定类的逻辑。如果涉及到类实例的管理，那么可以有容器和存储，就像上面的例子一样。

    总结一下，我强烈建议您永远不要使用静态方法，特别是当它们要替代对象构造函数时。通过构造函数来创建一个对象是任何面向对象软件中最“神圣”的时刻，不要错过这种美感。

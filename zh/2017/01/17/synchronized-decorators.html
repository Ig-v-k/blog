    Synchronized Decorators to Replace Thread-Safe Classes

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    你知道线程安全是什么吗？如果不知道，下面有一个简单的例子。所有类都必须是线程安全的，对吗？并不完全是这样。其中一些类必须是线程安全的吗？再次错了。我认为没有任何一个类必须是线程安全的，但是它们都必须提供同步装饰器。

    让我们从一个例子开始（顺便说一句，它是可变的）：

    class Position {
  private int number = 0;
  @Override
  public void increment() {
    int before = this.number;
    int after = before + 1;
    this.number = after;
  }
}

    你觉得这个对象是否是线程安全的？该术语是指当多个线程同时使用该类的对象时，该对象是否能够无错误地运行。假设我们有两个线程同时使用同一个对象position，并在同一时刻调用其方法increment()。

    我们期望当两个线程都完成时，number整数应该等于2，因为它们各自会增加一次，对吗？然而，很可能这不会发生。

    让我们看看会发生什么。在两个线程中，当它们开始时，before的值都将为0。然后after将被设置为1。接下来，两个线程都会执行this.number = 1，结果我们得到的是number中的1，而不是预期的2。看到问题了吗？具有这种设计缺陷的类不是线程安全的。

    最简单且最明显的解决方案是将我们的方法 synchronized（https://docs.oracle.com/javase/tutorial/essential/concurrency/syncmeth.html）。这将确保无论有多少个线程同时调用它，它们都会按顺序执行，而不是并行执行：一个线程接着一个线程。当然，这会花费更长的时间，但它将防止发生那个错误。

    class Position {
  private int number = 0;
  @Override
  public synchronized void increment() {
    int before = this.number;
    int after = before + 1;
    this.number = after;
  }
}

    一个类保证无论有多少个线程与其一起工作，它都不会出错，这就被称为线程安全。

    现在的问题是：我们是需要使所有的类都具有线程安全性，还是只有其中一部分类需要呢？看起来让所有的类都没有错误似乎更好，对吗？为什么有人会想要一个可能在某个时刻出错的对象呢？嗯，并非完全如此。请记住，这涉及到性能方面的考虑；我们通常不会有多个线程，并且我们总是希望我们的对象尽可能地运行得快。线程之间的同步机制肯定会减慢我们的速度。

    我认为正确的方法是拥有两个类。第一个类不是线程安全的，而另一个类是一个同步装饰器，它会像这样：

    class SyncPosition implements Position {
  private final Position origin;
  SyncPosition(Position pos) {
    this.origin = pos;
  }
  @Override
  public synchronized void increment() {
    this.origin.increment();
  }
}

    现在，当我们需要我们的position对象是线程安全的时候，我们使用SyncPosition修饰它。

    Position position = new SyncPosition(
  new SimplePosition()
);

    当我们需要一个简单的位置，没有任何线程安全性时，我们会这样做：

    Position position = new SimplePosition();

    在我看来，使类的功能既丰富又线程安全，违反了著名的单一职责原则。

    顺便提一句，这个问题与防御性编程和验证器非常相似。

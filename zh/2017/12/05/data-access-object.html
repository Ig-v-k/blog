    DAO is Yet Another OOP Shame

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    有人问我对DAO的看法，我意识到，尽管我写过关于ORM、DTO和getter的东西，但我还没有机会提到DAO。下面是我的看法：与它的朋友ORM、DTO和getter一样，这是一个遗憾。简而言之，数据访问对象是一个对象，它”为某种类型的数据库或其他持久化机制提供了一个抽象接口”。目的是崇高的，但实现很糟糕。

    这是它的示例：

    class BookDAO {
  Book find(int id);
  void update(Book book);
  // Other methods here ...
}

    这个想法很简单——方法find()创建了一个DTO Book，其他人将新数据注入其中并调用update()。

    BookDAO dao = BookDAOFactory.getBookDAO();
Book book = dao.find(123);
book.setTitle("Don Quixote");
dao.update(book);

    你问什么有问题？所有ORM存在的问题都有，只是我们用一个DAO代替了“session”。问题依旧存在：book并不是一个对象，而是一个数据容器。我引用了自己在ORM文章中的三年前的陈述，只是稍微改了一下名字：“DAO并没有将数据库交互封装在对象内部，而是将其分离开来，实质上撕裂了一个完整和有凝聚力的有机体。”要获取更多详细信息，请查看那篇文章。

    然而，我必须说，在我大部分的宠物项目中，我有一些类似于DAO的东西，但它们不返回或接受DTO。相反，它们返回对象并有时接受对它们的操作。以下是一些示例。请看这个来自Wring.io的Pipes接口。

    interface Pipes {
  void add(String json);
  Pipe pipe(long number);
}

    它的方法add()在“collection”中创建一个新项，方法pipe()从集合中返回一个单一对象。Pipe不是DTO，它是一个普通对象，完全能够执行所有必要的数据库操作，不需要DAO的帮助。例如，有一个Pipe.status(String)方法来更新其状态。我不会使用Pipes来做这个，我只会使用pipe.status("Hello, world!")。

    这是来自Jare.io的另一个例子：接口Base，它返回一个类型为Domain的对象列表。然后，当我们想要删除一个域名时，我们只需调用domain.delete()。该域名完全能够执行所有必要的数据库操作。

    DAO的问题就在它的名字中，我相信。它说我们正在访问“数据”，并确实如此：去数据库，检索一些数据，并返回数据。不是一个对象，而是数据，也被称为“数据传输对象”。正如我们之前讨论过的那样，直接对数据进行操作会破坏封装性，并使面向对象的代码变得过程化和难以维护。

    class BooksDAO {
    Book find(int id);
    void save(Book book);
    }

    SRP is a Hoax

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    根据Robert Martin的《Clean Code》，单一职责原则意味着“一个类应该只有一个变化的原因。”让我们试图解密这个相当模糊的陈述，并看看它如何帮助我们设计更好的面向对象软件。如果它确实有帮助的话。

    我在关于SOLID的帖子中提到过SRP，说它并不能真正帮助程序员理解拉里·康斯坦丁（Larry Constantine）在1974年引入的古老的“高内聚”概念。现在让我们通过示例来看一下，并分析我们如何在考虑SRP的情况下改进一个类，以及它是否会变得更具面向对象的特性。

    让我们尝试使用jcabi-s3中的AwsOcket类（我简化了代码）。

    class AwsOcket {
  boolean exists() { /* ... */ }
  void read(final OutputStream output) { /* ... */ }
  void write(final InputStream input) { /* ... */ }
}

    如果我说错了，请纠正我，但根据SRP，这个类负责太多的事情：1）检查AWS S3中对象的存在性，2）读取其内容，以及3）修改其内容。对吗？这不是一个好的设计，必须进行更改。

    为了改变它并使其只负责一件事，我们必须引入一个getter，它将返回一个AWS客户端，然后创建三个新类：ExistenceChecker，ContentReader和ContentWriter。它们将进行检查、读取和写入操作。现在，为了读取内容并将其打印到控制台，我当前正在执行以下操作：

    if (ocket.exists()) {
  ocket.read(System.out);
}

    明天，如果我重构这个类，我将会这样做：

    if (new ExistenceChecker(ocket.aws()).exists()) {
  new ContentReader(ocket.aws()).read(System.out);
}

    除了这些检查器、读者和写者实际上不是类，而是纯粹的过程持有者这一事实之外，使用这个ocket变成了一场噩梦。我们无法再确定当我们将其传递到某个地方时会发生什么。例如，我们无法保证从中获取的内容是否会被即时解密或解码。我们简单地无法对其进行修饰。它不再是一个对象，而是一个AWS客户端的持有者，由其他某些类在某处使用。

    是的，现在它只负责一件事：封装对AWS客户端的引用。就单一职责原则而言，它是一个完美的类。但它不再是一个对象了。

    如果您将SRP原则充分应用于任何类，那么相同的情况也会发生：它将成为数据或其他对象的持有者，同时还具有一系列setter和getter。也许还会有一个额外的方法。

    我的观点是，SRP是一个错误的理念。

    将类定义得小而紧密是个好主意，但将它们仅责任化为“一件事”是对“高内聚性”概念的一种误导性简化。这只会将它们变成其他事物的愚蠢承载者，而不是更小实体的封装者和装饰者，以构建更大的实体。

    在我们为这个虚假的SRP（单一职责原则）理念进行斗争的过程中，我们失去了一个更为重要的原则，这个原则实际上是关于真正的面向对象编程和思维的封装。一个对象负责多少事情并不那么重要，而更重要的是它对所封装实体的保护程度有多高。一个拥有一百个方法的庞大对象远不及一个只有五对getter和setter的DTO（数据传输对象）问题严重！这是因为DTO将问题分散到整个代码中，我们甚至无法找到它，而庞大对象总是在我们面前，我们总是可以将其重构为较小的部分。

    封装首先考虑，大小其次（若适用）。

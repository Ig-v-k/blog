    Lazy Loading and Caching via Sticky Cactoos Primitives

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    你显然知道什么是懒加载，对吧？而且你无疑也知道缓存。据我所知，在Java中没有一种优雅的方式来实现它们。下面是我通过使用Cactoos原语自己发现的方法。

    假设我们需要一个对象来加密一些文本。用更面向对象的方式说，它将封装文本并且变成其加密形式。下面是我们将如何使用它（让我们首先创建测试）：

    interface Encrypted {
  String asString() throws IOException;
}
Encrypted enc = new EncryptedX("Hello, world!");
System.out.println(enc.asString());

    现在让我们用一种非常原始的方式来实现它，只使用一个主要的构造函数。加密机制只会对传入数据中的每个字节加上+1，并假设加密不会破坏任何东西（这是一个非常愚蠢的假设，但为了这个例子的完整性，它会起作用）。

    class Encrypted1 implements Encrypted {
  private final String text;
  Encrypted1(String txt) {
    this.text = txt;
  }
  @Override
  public String asString() {
    final byte[] in = this.text.getBytes();
    final byte[] out = new byte[in.length];
    for (int i = 0; i < in.length; ++i) {
      out[i] = (byte) (in[i] + 1);
    }
    return new String(out);
  }
}

    看起来正确吗？我进行了测试，它可以工作。如果输入是"Hello, world!"，输出将是"Ifmmp-!xpsme\""。

    接下来，假设我们希望我们的类接受一个InputStream和一个String。我们想要像这样调用它，例如：

    Encrypted enc = new Encrypted2(
  new FileInputStream("/tmp/hello.txt")
);
System.out.println(enc.toString());

    这是最明显的实现方式，有两个主要的构造函数（再次强调，该实现是简单粗糙的，但能够工作）。

    class Encrypted2 implements Encrypted {
  private final String text;
  Encrypted2(InputStream input) throws IOException {
    ByteArrayOutputStream baos =
      new ByteArrayOutputStream();
    while (true) {
      int one = input.read();
      if (one < 0) {
        break;
      }
      baos.write(one);
    }
    this.text = new String(baos.toByteArray());
  }
  Encrypted2(String txt) {
    this.text = txt;
  }
  // asString() is exactly the same as in Encrypted1
}

    从技术上讲，它可以工作，但流读取正好在构造函数内部进行，这是一种不好的做法。主构造函数除了属性赋值之外不应该做任何事情，而次要构造函数只能创建新对象。

    让我们尝试重构并引入延迟加载：

    class Encrypted3 implements Encrypted {
  private String text;
  private final InputStream input;
  Encrypted3(InputStream stream) {
    this.text = null;
    this.input = stream;
  }
  Encrypted3(String txt) {
    this.text = txt;
    this.input = null;
  }
  @Override
  public String asString() throws IOException {
    if (this.text == null) {
      ByteArrayOutputStream baos =
        new ByteArrayOutputStream();
      while (true) {
        int one = input.read();
        if (one < 0) {
          break;
        }
        baos.write(one);
      }
      this.text = new String(baos.toByteArray());
    }
    final byte[] in = this.text.getBytes();
    final byte[] out = new byte[in.length];
    for (int i = 0; i < in.length; ++i) {
      out[i] = (byte) (in[i] + 1);
    }
    return new String(out);
  }
}

    工作得很好，但看起来很丑。最丑的部分当然是这两行。

    this.text = null;
this.input = null;

    它们使对象可变，并且使用NULL。相信我，这很丑陋。不幸的是，懒加载和NULL引用总是一起出现在经典示例中。然而，有一种更好的实现方法。让我们重新构造我们的类，这次使用Cactoos中的Scalar。

    class Encrypted4 implements Encrypted {
  private final IoCheckedScalar<String> text;
  Encrypted4(InputStream stream) {
    this(
      () -> {
        ByteArrayOutputStream baos =
          new ByteArrayOutputStream();
        while (true) {
          int one = stream.read();
          if (one < 0) {
            break;
          }
          baos.write(one);
        }
        return new String(baos.toByteArray());
      }
    );
  }
  Encrypted4(String txt) {
    this(() -> txt);
  }
  Encrypted4(Scalar<String> source) {
    this.text = new IoCheckedScalar<>(source);
  }
  @Override
  public String asString() throws IOException {
    final byte[] in = this.text.value().getBytes();
    final byte[] out = new byte[in.length];
    for (int i = 0; i < in.length; ++i) {
      out[i] = (byte) (in[i] + 1);
    }
    return new String(out);
  }

    现在看起来好多了。首先，只有一个主构造函数和两个次要构造函数。其次，该对象是不可变的。第三，还有很大的改进空间：我们可以添加更多的构造函数，接受其他数据源，例如File或字节数组。

    简而言之，应该以“延迟”方式加载的属性在对象内部表示为“函数”（Java 8中的lambda表达式）。在我们不触及该属性之前，它不会被加载。一旦我们需要处理它，该函数将被执行，我们就会得到结果。

    然而，这段代码有一个问题。每次调用asString()时，它都会读取输入流，这显然行不通，因为只有第一次流中才有数据。在后续的每次调用中，流将为空。因此，我们需要确保this.text.value()只执行封装的Scalar一次。后续的所有调用必须返回先前计算的值。因此，我们需要对其进行“缓存”。下面是如何实现的：

    class Encrypted5 implements Encrypted {
  private final IoCheckedScalar<String> text;
  // same as above in Encrypted4
  Encrypted5(Scalar<String> source) {
    this.text = new IoCheckedScalar<>(
      new StickyScalar<>(source)
    );
  }
  // same as above in Encrypted4

    这个StickyScalar确保只有第一次调用其value()方法会传递给封装的Scalar。所有其他调用都会接收到第一次调用的结果。

    最后要解决的问题是并发性。我们上面的代码不是线程安全的。如果我创建一个Encrypted5的实例并将其传递给两个线程，同时调用asString()，结果将是不可预测的，因为StickyScalar不是线程安全的。然而，还有另一个原始类型可以帮助我们解决这个问题，叫做SyncScalar：

    class Encrypted5 implements Encrypted {
  private final IoCheckedScalar<String> text;
  // same as above in Encrypted4
  Encrypted5(Scalar<String> source) {
    this.text = new IoCheckedScalar<>(
      new SyncScalar<>(
        new StickyScalar<>(source)
      )
    );
  }
  // same as above in Encrypted4

    现在我们是安全的，设计也很优雅。它包括了延迟加载和缓存。

    我现在在许多项目中都采用这种方法，它看起来方便、清晰且面向对象。

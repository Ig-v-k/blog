    Java 9: The Good, The Bad, and Private Interface Methods

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Java 9 几周前发布。请查看发布说明，其中包括许多有趣的功能。然而，我认为并不是所有的东西都像 Oracle 和 Java 粉丝所描述的那样好。我在 Java 世界中看到了三种趋势，分别是好的、坏的和丑陋的。让我们从好的一方面开始。

    The Platform

    第一个趋势是编译Java、打包JAR文件和运行字节码的平台明显改进。它在每个新的Java发布版本中都有明显的进步。以下是Java 9带来的改进列表，毫无疑问非常有用：

    • “JSR 376: Module System a.k.a. Jigsaw” translated to Chinese is: “JSR 376：模块系统，又称为Jigsaw”

    • JEP 222: jshell

    • “JEP 238: Multi-release JARs” should be translated to Chinese as follows: “JEP 238：多版本JAR文件”

    • JEP 282: jlink

    • “JEP 158: Unified logging” should be translated to Chinese as: “JEP 158：统一日志记录”

    这个平台显然越来越成熟。这是一个好的趋势。

    The JDK

    自从Java 6以来，我观察到的第二个趋势表明，JDK实际上是由Oracle设计、开发和维护的一套类和接口的集合，随着每个新版本的发布，它变得越来越庞大。在Java 9中，除其他内容外，他们还新增和扩展了以下功能：

    • JEP 221, 224 225, 261: Javadoc处理（扩展）

    • “JEP 268: XML Catalogs (new)” should be translated to Chinese as: “JEP 268: XML目录（新）”.

    • JEP 262：TIFF 图像 I/O（新）

    • “JEP 251: 多分辨率图像（新）”

    • “JEP 110: HTTP 2.0 client (new)” should be translated to Chinese as: “JEP 110：HTTP 2.0 客户端（新）”.

    • JEP 236：Nashorn的解析器（扩展）。

    当然，一些功能必须在JDK本身中实现，如Unicode支持（JEP 267），平台特定的桌面功能（JEP 272），自旋等待提示（JEP 285），紧凑字符串（JEP 254）和进程API（JEP 102）。它们的实现取决于底层平台，并且必须与JVM一起提供。

    但是，HTTP 2.0客户端与JAX-RS，JPA，JAX-WS，JDBC以及其他许多东西一起出现在JDK中，而依我看来，它们应该尽可能远离Oracle。它们不是平台特定的，可以更好地由开源社区以独立包的方式设计。我认为将它们集合在一个巨大的品牌下是一个错误。

    我认为大公司只是在杀死软件市场，而不是使其变得更好，因为它们暴露给了金融和政治动机。这正是JDK所发生的。由于Oracle的垄断，它在增长方面缺乏灵活性和动力。换句话说，我们被困在Oracle及其重要合作伙伴认为正确的事情上。

    因此，使JDK变得更大是一个不好的趋势。相反，我相信，Oracle只会从使其变得更小中受益，将一切非平台特定的事务委派给开源社区，在某种程度上支持程序员，并推动市场上的开放和有效的标准化过程。

    The Language

    Java在1995年由James Gosling在Sun Microsystems开发，作为一种面向对象的语言。对于这种面向对象的说法存在许多争议，我也不确定Java是更多面向对象还是过程化的。然而，它在官方上被认定为面向对象的。

    自从其首个版本以来，Java从C/C++继承了许多过程性特征，包括静态方法、NULL、实现继承等等。我理解，它并不是一门完美的面向对象语言，也不打算成为一门完美的语言。其核心理念是创建一种可以“编写一次，随处运行”的东西。然而，这门语言本身也非常重要，而不仅仅是JVM。它简洁而有吸引力。

    Java 5在2004年迈出了重要的一步，通过添加泛型、for-each循环、可变参数和静态导入等功能来改进语言。然而，引入了注释和枚举，这使得语言从面向对象范式转向了完全不同的过程式范式。

    Java 7在2011年增加了try-with-resource，这是一个很好的举措，符合面向对象编程范例。

    Java 8在2014年添加了lambda表达式，这是一个很棒的特性，但与面向对象编程毫无关系。Lambda和Streams API使Java成为了面向对象、过程式和函数式范式的混合体。默认方法也被添加到了接口中，这使得类型变成了代码的库。类型变成了库！如果你问我，这甚至比实现继承更糟糕。

    现在Java 9对接口进行了下一个“改进”，允许它们拥有私有方法。类型中的私有静态方法！你能相信吗？下一步会是什么呢？我猜是Java 10中的属性。

    此外，让我们来看看JDK中一些核心类的变化情况，以了解语言的发展方向。仅提供两个例子。

    集合的工厂方法（JEP 269）。而不是引入新的构造函数并允许我们这样做：

    List<Integer> list = new ArrayList<>(1, 2, 3);

    …in Java 9 they created more static factory methods and made us do this:

    List<Integer> list = List.of(1, 2, 3);

    “少用构造函数，多用静态方法！”似乎是引入这个JEP的人的哲学。毋庸置疑，这完全违背了面向对象编程的精神。对象必须由构造函数创建，而不是静态方法，无论Joshua Bloch说什么。静态方法使我们无法看到new运算符的使用时机，这就是为什么代码可维护性降低的原因——我们根本不知道实例化的是哪个类以及其构造函数的真实参数是什么。

    顺便提一下，使用Cactoos，您可以用正确的方式完成它：

    List<Integer> list = new ListOf(1, 2, 3);

    This is OOP.

    InputStream中的新方法。在已经臃肿的类InputStream中添加了三个新方法：transferTo()、readNBytes()和readAllBytes()。现在，当我们想要将输入流复制到输出流时，我们应该这样做：

    input.transferTo(output);

    这是年轻的面向对象编程程序员经常犯的最典型的错误之一：他们使他们的接口变得很庞大，仅仅因为他们需要更多的功能。我猜测“接口隔离原则”是著名的SOLID原则的一部分，已经存在多年了。Oracle，你怎么了？下一步是什么？在Java 10中，我们也会有saveToFile()和printToConsole()吗？emailToAFriend()呢？

    这是您可以使用 IOUtils 实用类从 commons-io 实现相同效果的方式。

    IOUtils.copy(input, output);

    它并非完美，但更好。最面向对象的方式是使用对象，而非实用类和静态方法。这就是 Cactoos 的工作原理。

    new LengthOf(new TeeInput(input, output)).length();

    This is OOP.

    In my opinion, Java is getting uglier, and this is a trend. Does it mean that it’s time to quit? No! No matter how ugly you are, we will always love you Java!

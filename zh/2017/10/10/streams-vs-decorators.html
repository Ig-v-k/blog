    Streams vs. Decorators

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Streams API是在几年前的Java 8中引入的，与lambda表达式一起。作为一个守规矩的Java专家，我尝试在我的一些项目中使用这个新特性，例如这里和这里。我并不是很喜欢它，回到了好的老装饰器上。此外，我创建了Cactoos，一个装饰器库，来替代在很多地方都不太好用的Guava。

    这是一个简单的例子。假设我们有一组来自某个数据源的测量数据，它们都是介于零和一之间的数字。

    Iterable<Double> probes;

    现在，我们需要仅显示其中的前10个，忽略零和一，并将它们重新缩放到（0..100）。听起来很简单，对吗？有三种方法可以做到：过程式、面向对象和Java 8的方式。让我们从过程式方式开始：

    int pos = 0;
for (Double probe : probes) {
  if (probe == 0.0d || probe == 1.0d) {
    continue;
  }
  if (++pos > 10) {
    break;
  }
  System.out.printf(
    "Probe #%d: %f", pos, probe * 100.0d
  );
}

    为什么这是一种过程化方式？因为它是命令式的。为什么它是命令式的？因为它是过程化的。开个玩笑。

    这是非常重要的，因为我们正在向计算机提供关于将数据放在何处以及如何迭代访问的指令。我们并没有声明结果，而是以命令式的方式构建它。虽然它可行，但并不具备可扩展性。我们无法将此算法的部分应用于其他用例。我们无法轻松修改它，例如从两个源中获取数字而不是一个等。这是过程性的。足够说了。不要以这种方式进行操作。

    现在，Java 8 给我们带来了 Streams API，它被认为是提供了一种函数式的方式来做相同的事情。让我们试着使用它。

    首先，我们需要创建一个 Stream 的实例，Iterable 并不直接允许我们直接获取。然后，我们使用流 API 来完成这项工作：

    StreamSupport.stream(probes.spliterator(), false)
  .filter(p -> p == 0.0d || p == 1.0d)
  .limit(10L)
  .forEach(
    probe -> System.out.printf(
      "Probe #%d: %f", 0, probe * 100.0d
    )
  );

    这将起作用，但对于所有探测器都将显示为“Probe #0”，因为forEach()不能与索引一起使用。截至Java 8（以及Java 9），在Stream 接口中没有forEachWithIndex()。这里有一个使用原子计数器的解决方法。

    AtomicInteger index = new AtomicInteger();
StreamSupport.stream(probes.spliterator(), false)
  .filter(probe -> probe != 0.0d && probe != 1.0d)
  .limit(10L)
  .forEach(
    probe -> System.out.printf(
      "Probe #%d: %f",
      index.getAndIncrement(),
      probe * 100.0d
    )
  );

    “那有什么问题？”你可能会问。首先，看看当我们在Stream接口中找不到正确的方法时，我们是如何轻易陷入麻烦的。我们立即摆脱了“流式”范式，回到了老旧的过程化全局变量（计数器）。其次，我们并不真正了解filter()、limit()和forEach()方法内部发生了什么。它们到底是如何工作的？文档称这种方法是“声明式”的，Stream接口中的每个方法都返回某个类的实例。它们是哪些类？仅凭这段代码我们无从得知。

    这两个问题是相互关联的。这个流式 API 最大的问题是非常庞大的接口 Stream——它有很多方法。在写这篇文章的时候，有 43 个方法。四十三个方法，一个接口里面！这违背了面向对象编程的每一个原则，从 SOLID 开始，一直到更严肃的原则。

    如何以面向对象的方式实现相同的算法？以下是我在Cactoos中的实现方式，它只是一组简单的Java类的集合：

    new And(
  new Mapped<Double, Scalar<Boolean>>(
    new Limited<Double>(
      new Filtered<Double>(
        probes,
        probe -> probe != 0.0d && probe != 1.0d
      ),
      10
    ),
    probe -> () -> {
      System.out.printf(
        "Probe #%d: %f", 0, probe * 100.0d
      );
      return true;
    }
  )
).value();

    让我们看看这里发生了什么。首先，Filtered修饰了我们的可迭代对象probes，将其中的某些项取出。注意，Filtered实现了Iterable。然后Limited，也是一个Iterable，只取出了前十个项。然后Mapped将每个探测器转换为Scalar<Boolean>的实例，它负责打印行。

    最后，And的实例遍历“标量”列表，并要求每个标量返回boolean。它们打印出该行并返回true。由于结果是true，And会尝试下一个标量。最终，它的value()方法返回true。

    但是等等，这里没有索引。让我们添加它们。为了做到这一点，我们只需使用另一个称为 AndWithIndex 的类。

    new AndWithIndex(
  new Mapped<Double, Func<Integer, Boolean>>(
    new Limited<Double>(
      new Filtered<Double>(
        probes,
        probe -> probe != 0.0d && probe != 1.0d
      ),
      10
    ),
    probe -> index -> {
      System.out.printf(
        "Probe #%d: %f", index, probe * 100.0d
      );
      return true;
    }
  )
).value();

    我们现在将我们的探针映射到Func<Integer, Boolean>，而不是Scalar<Boolean>，以便让它们接受索引。

    这种方法的美妙之处在于所有的类和接口都很小，这就是它们为什么非常可组合的原因。为了使探测器的迭代有限，我们使用Limited进行修饰；为了进行过滤，我们使用Filtered进行修饰；为了做其他事情，我们创建一个新的修饰器并使用它。我们不仅限于像Stream这样的单一接口。

    底线是装饰器是一种面向对象的工具，用于修改集合的行为，而流则是另一种我甚至找不到名称的东西。

    顺便说一下，以下是如何使用Guava的Iterables实现相同算法的方法。

    Iterable<Double> ready = Iterables.limit(
  Iterables.filter(
    probes,
    probe -> probe != 0.0d && probe != 1.0d
  ),
  10
);
int pos = 0;
for (Double probe : probes) {
  System.out.printf(
    "Probe #%d: %f", pos++, probe * 100.0d
  );
}

    这是一种奇怪的面向对象和函数式风格的组合。

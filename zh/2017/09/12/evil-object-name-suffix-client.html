    Yet Another Evil Suffix For Object Names: Client

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    不久之前，我们讨论了关于对象和类名中的“-ER”后缀。我们一致认为它们是邪恶的，如果我们希望我们的代码真正面向对象，希望我们的对象成为对象而不是过程的集合，就必须避免使用它们。现在我准备引入一个新的邪恶后缀：“Client”。

    让我先举个例子。以下是具有这种后缀的对象的示例（它是来自AWS Java SDK的AmazonS3Client的伪代码版本）：

    class AmazonS3Client {
  createBucket(String name);
  deleteBucket(String name);
  doesBucketExist(String name);
  getBucketAcl(String name)
  getBucketPolicy(String name);
  listBuckets();
  // 160+ more methods here
}

    所有的“客户端”看起来相似：它们通过一些访问凭证封装了目标URL，并公开了一些方法，用于在“服务器”之间传输数据。尽管这种设计看起来像一个[合适的对象]（），但它实际上并没有遵循真正的面向对象的精神。这就是为什么它不像应该那样易于维护的两个原因。

    其范围太广。由于客户端是服务器的抽象，它不可避免地必须代表服务器的全部功能。当功能相当有限时，这并没有问题。以Apache HttpComponents的HttpClient为例。然而，当服务器更加复杂时，客户端的大小也会增加。在撰写本文时，AmazonS3Client中有超过160个(!)方法，而几百个版本之前只有几十个。

    它是以数据为中心的。客户端-服务器关系的核心概念是数据传输。以AWS S3服务的HTTP RESTful API为例。在AWS一侧有实体：存储桶、对象、版本、访问控制策略等，服务器将它们转化为JSON/XML数据。然后这些数据传输到我们这里，我们的客户端处理JSON或XML数据。对我们来说，它始终是数据，而不是真正的存储桶、对象或版本。

    后果取决于情况，但以下是最有可能的：

    过程性代码。由于客户端返回数据，与该数据相关的代码很可能是过程性的。看一下AWS SDK方法返回的结果，它们看起来都像是对象，但实际上它们只是数据结构：S3Object，ObjectMetadata，BucketPolicy，PutObjectResult等。它们都是只有获取器和设置器的数据传输对象。

    重复的代码。如果我们真的决定保持面向对象的方式，我们将不得不将客户端返回给我们的数据转换为对象。很可能这会导致多个项目中出现代码重复的情况。当我开始使用S3 SDK时，我也遇到了这个问题。很快我意识到为了避免重复，最好创建一个将S3 SDK数据转换为对象的库：jcabi-s3。

    测试中的困难。由于客户端在大多数情况下是一个相当大的类/接口，对其进行单元测试或创建其测试替身/伪造对象是一项相当复杂的任务。

    静态问题。尽管客户端类的方法并非静态的，但它们的外观与实用类非常相似，后者因为反面面向对象而闻名。我们对实用类的问题与对“客户端”类的问题几乎相同。

    可扩展性问题。不用说，当客户端对象具有160多个方法并且不断增长时，几乎不可能对其进行修饰。唯一可以向其添加新功能的方法是创建新的方法。最终我们得到了一个巨型类，无法在不修改的情况下重新使用。

    替代方案是什么？

    正确的设计是用代表服务器端实体的客户端对象替换“clients”，而不是整个服务器。例如，使用S3 SDK，可以是“Bucket”、“Object”、“Version”、“Policy”等。它们每一个都提供了真实存储桶、对象和版本的功能，这些功能可以由AWS S3提供。

    当然，我们需要一个高级对象，它以某种方式代表整个API/服务器，但它应该很小。例如，在S3 SDK示例中，它可以被称为“Region”，意思是整个AWS region中的存储桶。然后我们可以从中检索一个存储桶，就不再需要一个区域了。然后，要列出存储桶中的对象，我们要求存储桶为我们执行此操作。每次都不需要与整个“服务器对象”进行通信，尽管从技术上讲，这种通信当然是发生的。

    总结一下，问题并不完全在于名称后缀，而在于在客户端上代表整个服务器而不是其“实体”的这个概念本身。这样的抽象有两个问题：1）太庞大；2）非常数据驱动。

    顺便说一句，查看一些 JCabi libraries（Java）的例子，这些例子展示了没有 “client” 对象的面向对象客户端：jcabi-github，jcabi-dynamo，jcabi-s3，或者 jcabi-simpledb。

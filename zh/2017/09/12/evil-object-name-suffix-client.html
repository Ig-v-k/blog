    Yet Another Evil Suffix For Object Names: Client

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    前一段时间我们讨论了对象和类名中的“-ER”后缀。我们一致认为它们是邪恶的，如果我们希望我们的代码真正面向对象，我们的对象是对象而不是过程的集合，就必须避免使用它们。现在我要介绍一个新的邪恶后缀：Client。

    让我先举一个例子。这是一个带有这种后缀的对象可能是什么样子的（它是 AmazonS3Client 的伪代码版本，来自于 AWS Java SDK）：

    class AmazonS3Client {
  createBucket(String name);
  deleteBucket(String name);
  doesBucketExist(String name);
  getBucketAcl(String name)
  getBucketPolicy(String name);
  listBuckets();
  // 160+ more methods here
}

    所有的“客户端”看起来都很相似：它们使用一些访问凭证封装目标URL，并提供一些方法，用于传输数据到/从“服务器”。尽管这种设计看起来像一个合适的对象，但它并没有真正遵循面向对象的真正精神。这就是为什么它不像应该那样易于维护的两个原因。

    • 它的范围太广泛。由于客户端是服务器的抽象，它不可避免地必须代表服务器的全部功能。当功能相当有限时没有问题。以Apache HttpComponents的HttpClient为例。然而，当服务器更复杂时，客户端的大小也会增长。在撰写本文时，AmazonS3Client中有超过160个(!)方法，而在几百个版本之前，它只有几十个方法。

    • 它以数据为中心。客户端-服务器关系的核心概念是数据传输。以AWS S3服务的HTTP RESTful API为例。在AWS一侧有一些实体：存储桶、对象、版本、访问控制策略等，服务器将它们转换为JSON/XML数据。然后数据传递到我们这一侧，我们的客户端处理JSON或XML。对我们来说，它始终是数据，从未真正变成存储桶、对象或版本。

    后果取决于具体情况，但以下是最有可能的：

    • 过程性代码。由于客户端返回数据，与该数据交互的代码很可能是过程性的。看一下AWS SDK方法返回的结果，它们都像是对象，但实际上它们只是数据结构：S3Object，ObjectMetadata，BucketPolicy，PutObjectResult，等等。它们都是只包含getter和setter的数据传输对象。

    • 重复的代码。如果我们决定保持面向对象的方式，我们就需要将客户端返回给我们的数据转换为对象。很可能这会导致多个项目中出现代码重复。当我开始使用S3 SDK时，我也遇到了这个问题。很快我意识到为了避免重复，最好创建一个库来将S3 SDK的数据转换为对象：jcabi-s3。

    • 测试的困难。由于客户端在大多数情况下是一个相当庞大的类/接口，对其进行单元测试或创建其测试替身/伪装是一项相当复杂的任务。

    • 静态问题。客户端类，即使它们的方法不是静态的，看起来与实用类非常相似，而实用类因其反面面向对象的特性而广为人知。我们对实用类的问题几乎与我们对“客户端”类的问题相同。

    • 可拓展性问题。不用说，当客户端对象拥有160多个方法并且不断增长时，几乎不可能对其进行装饰。唯一能够为其添加新功能的方式就是创建新的方法。最终，我们得到了一个巨兽类，无法在不修改的情况下进行重用。

    “替代方案是什么？”

    正确的设计是用客户端对象替换“客户端”，这些对象代表服务器端的“实体”，而不是整个服务器。例如，使用S3 SDK，可以使用Bucket、Object、Version、Policy等。每个对象都暴露了真实的存储桶、对象和版本的功能，这些功能可以由AWS S3提供。

    当然，我们需要一个高级对象来以某种方式代表整个API/服务器，但它应该很小。例如，在S3 SDK示例中，它可以被称为“Region”，表示整个AWS区域和存储桶。然后，我们可以从中检索一个存储桶，就不再需要区域了。然后，要列出存储桶中的对象，我们要求存储桶为我们执行此操作。每次都无需与整个“服务器对象”进行通信，尽管从技术上讲，这样的通信当然是发生的。

    总结一下，问题不完全在于名称后缀，而是在于将整个服务器表示在客户端而不是其“实体”的这个概念上。这样的抽象是1）太庞大，2）非常数据驱动。

    顺便提一下，查看一下一些 JCabi 库（Java）的例子，它们是没有 “client” 对象的面向对象客户端库：jcabi-github、jcabi-dynamo、jcabi-s3 或者 jcabi-simpledb。

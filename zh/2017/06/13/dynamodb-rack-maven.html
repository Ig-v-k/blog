    DynamoDB + Rake + Maven + Rack::Test

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在我的Ruby宠物网页应用SixNines.io中，我使用DynamoDB，这是一种由AWS提供的NoSQL云数据库。它运行得很好，但问题是很难创建一个集成测试，以确保我的代码可以与“真实”的DynamoDB服务器和表一起工作。让我向你展示如何解决这个问题。该代码是开源的，你可以在yegor256/sixnines GitHub存储库中查看。

    How to bootstrap DynamoDB Local

    首先，您需要使用DynamoDB Local，这是由AWS专门为测试目的创建的命令行工具。在进行集成测试之前，您需要启动它，并在测试完成后停止它。

    为了简化事情，我建议您使用jcabi-dynamodb-maven-plugin，这是我几年前开发的一个Maven插件。您需要将pom.xml添加到您的存储库中，并从Rakefile中启动/停止Maven，就像我在这里所做的那样。

    task :dynamo do
  FileUtils.rm_rf('dynamodb-local/target')
  pid = Process.spawn('mvn', 'install', chdir: 'dynamodb-local')
  at_exit do
    `kill -TERM #{pid}`
  end
  begin
    Dynamo.new.aws.describe_table(table_name: 'sn-endpoints')
  rescue Exception => e
    sleep(5)
    retry
  end
end

    首先，我会删除dynamodb-local/target目录，这是Maven存放临时文件的目录，以确保我们总是从零开始。

    然后，我使用Process.spawn作为后台进程启动mvn install，并将pid作为其进程ID（这在Windows上不起作用，只适用于Linux/Mac）。然后，我立即注册一个at_exit的Ruby钩子，如果Ruby出现任何原因而死亡，该钩子将被执行。我相信很明显为什么我必须这样做——为了避免在Rake完成或终止后后台运行垃圾。

    请注意，我使用的是 kill -TERM 而不是 kill -KILL，以便给 Maven 一个机会来完成所有工作，正确终止 DynamoDB 本地实例，关闭其 TCP 端口并退出。

    How to check that it’s running

    接下来，我正在检查 DynamoDB Local 中的 sn-endpoints 表的状态。如果服务器正常运行，它必须存在。根据 sn-endpoints.json 中的 JSON 配置，它将由 jcabi-dynamodb-maven-plugin 创建。

    很可能表格不会立即准备好，因为启动Maven、启动服务器和在其中创建表格需要时间。这就是为什么，如果出现异常，我会捕获它，等待5秒钟然后再尝试。我会一直尝试很多次，直到服务器准备好为止。最终它会准备好。在我的MacBook上大约需要12-15秒，也就是2-3次尝试/异常。

    How to connect to DynamoDB Local

    我的课程需要知道如何在集成测试期间连接到服务器。在生产中，他们需要连接到AWS，在测试中，他们需要知道我刚刚启动的DynamoDB本地实例。这就是我在我的类Dynamo中的内容，它负责与DynamoDB进行连接。它决定连接到哪里是基于环境变量RACK_ENV的，该变量在test__helper.rb中设置为"test"，这个文件在rake/testtask之前包含在所有其他测试之前，感谢它名字中的双下划线。

    如果环境变量设置为"test"，Dynamo将从dynamodb-local/target/dynamo.yml文件中获取连接参数，该文件是由maven-resources-plugin:copy-resources创建的。DynamoDB Local数据库的TCP端口将在此处显示，以及DynamoDB的认证密钥和密码。

    How to run the integration tests

    这是最简单的部分。我只需按照生产中预期的方式使用我的对象，并将它们连接到 DynamoDB 本地而不是 AWS。

    我正在使用Rack::Test来测试整个应用程序，通过一组HTTP调用。例如，这里我正在尝试渲染Jeff的用户账户页面。它的HTTP响应代码应该是200。

    require 'test/unit'
require 'rack/test'
class AppTest < Test::Unit::TestCase
  include Rack::Test::Methods
  def app
    Sinatra::Application
  end
  def test_user_account
    header('Cookie', 'sixnines=jeff')
    get('/a')
    assert_equal(200, last_response.status)
  end

    现在你可以从命令行运行整个测试。你可以在发布新版本时查看Rultor的运行情况：完整日志。另外，可以看看它在Travis中的运行情况：Travis中的运行情况。简而言之：

    • 您在命令行中调用rake，Rake会启动；

    • Rake试图运行default任务，该任务依赖于test。

    • Rake尝试运行test，它依赖于dynamo。

    • Rake，在test任务中，使用此pom.xml在后台运行mvn install。

    • Maven unpacks DynamoDB Local 安装包；

    • Maven 保留 一个随机的TCP端口，并将其值存储在${dynamo.port}中。

    • Maven saves ${dynamo.port} and key/secret pair info;

    Maven会保存${dynamo.port}和键/密钥对信息；

    • Maven 启动 DynamoDB 本地版本，将其绑定到保留的 TCP 端口；

    • Rake在保留端口上等待DynamoDB Local的可用性。

    • 从test__helper.rb开始，Rake 导入所有的测试类。

    • 环境变量 RACK_ENV 被设置为 "test"；

    • Rack::Test 尝试 分发一个网页；

    • Dynamo 从 dynamo.yml 加载 YAML 配置，并连接到 DynamoDB 本地。

    • Rake stops;

    • Ruby终止Maven并停止DynamoDB本地。

    RAII in Java

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    资源获取即初始化（RAII）是由 Bjarne Stroustrup 在C++中引入的一个设计思想，用于实现异常安全的资源管理。由于Java具有垃圾回收机制，因此没有这个特性，但是我们可以使用try-with-resources来实现类似的功能。

    RAII解决的问题是显而易见的，请看下面的代码（我相信您知道Java中的Semaphore是什么以及它在Java中的工作原理）：

    class Foo {
  private Semaphore sem = new Semaphore(5);
  void print(int x) throws Exception {
    this.sem.acquire();
    if (x > 1000) {
      throw new Exception("Too large!");
    }
    System.out.printf("x = %d", x);
    this.sem.release();
  }
}

    代码相当原始，没有做任何有用的事情，但你应该能够理解：如果从多个并行线程中调用方法print()，只会允许其中五个并行打印。有时它不会允许其中一些打印，并且如果x大于1000，会抛出异常。

    这段代码的问题是——资源泄漏。每次使用x大于1000调用print()都会从信号量中获取一个许可，并且不会归还它。在五次带有异常的调用中，信号量将为空，所有其他线程都不会打印任何内容。

    解决方案是什么？在这里：

    class Foo {
  private Semaphore sem = new Semaphore(5);
  void print(int x) throws Exception {
    this.sem.acquire();
    if (x > 1000) {
      this.sem.release();
      throw new Exception("Too large!");
    }
    System.out.printf("x = %d", x);
    this.sem.release();
  }
}

    在我们抛出异常之前，我们必须释放许可。

    然而，还存在另一个问题：代码重复。我们在两个地方释放许可。如果我们添加更多的 throw 指令，我们还必须添加更多的 sem.release() 调用。

    在 C++ 中引入了一种非常优雅的解决方案，称为 RAII。以下是在 Java 中的示例：

    class Permit {
  private Semaphore sem;
  Permit(Semaphore s) {
    this.sem = s;
    this.sem.acquire();
  }
  @Override
  public void finalize() {
    this.sem.release();
  }
}
class Foo {
  private Semaphore sem = new Semaphore(5);
  void print(int x) throws Exception {
    new Permit(this.sem);
    if (x > 1000) {
      throw new Exception("Too large!");
    }
    System.out.printf("x = %d", x);
  }
}

    看看方法Foo.print()内部的代码是多么美丽。我们只需创建一个Permit类的实例，它就会立即在信号量上获得一个新的许可。然后我们通过异常或正常方式退出print()方法，Permit.finalize()方法释放许可。

    优雅，不是吗？是的，确实如此，但在Java中不起作用。

    这不起作用的原因是，与C++不同，Java不会在对象的可见范围关闭时销毁它们。当我们退出print()方法时，Permit类的对象不会被销毁。它最终会被销毁，但我们不知道具体时间。很可能在所有许可在信号量中被获取并被阻塞之后才会被销毁。

    在Java中也有解决方案。它不像C++中的解决方案那样优雅，但确实可行。下面是解决方案：

    class Permit implements Closeable {
  private Semaphore sem;
  Permit(Semaphore s) {
    this.sem = s;
  }
  @Override
  public void close() {
    this.sem.release();
  }
  public Permit acquire() {
    this.sem.acquire();
    return this;
  }
}
class Foo {
  private Semaphore sem = new Semaphore(5);
  void print(int x) throws Exception {
    try (Permit p = new Permit(this.sem).acquire()) {
      if (x > 1000) {
        throw new Exception("Too large!");
      }
      System.out.printf("x = %d", x);
    }
  }
}

    请注意try块和类Permit现在实现的Closeable接口。当try块退出时，对象p将被“关闭”。它可以在最后退出，也可以通过return或throw语句退出。无论哪种情况，都会调用Permit.close()：这就是Java中try-with-resources的工作原理。

    我引入了acquire()方法，并将sem.acquire()移到了Permit构造函数之外，因为我认为构造函数必须是无代码的。

    总结一下，当你处理可能泄漏的资源时，RAII是一种完美的设计方法。尽管Java没有内置它，但我们可以通过try-with-resources和Closeable来实现它。

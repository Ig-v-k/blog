    The TDD That Works for Me

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Test-driven development（即TDD）是由Kent Beck在2002年重新发现并在他的著名书籍中解释。在2014年，Ruby on Rails的创始人David Heinemeier Hansson表示，TDD已经死亡并且只会损害架构。Robert Martin（SOLID原则的发明者）不同意，并且解释说TDD可能只在某些情况下无法奏效。几天后，他甚至将TDD的重要性与医学上的洗手重要性进行比较，并补充说“如果有一天，TDD成为法律的力量，我也不会感到惊讶。”两年后，就在几个月前，他写了更多关于这个问题的文章，以及更多，还有更多。这个话题似乎非常热门。当然，我有自己的看法，让我分享一下。

    理论上，TDD 意味着“先编写测试，然后编写代码”。实际上，根据我在过去四年中与250多名开发人员一起工作的经验，它意味着在心情好且没有其他事情要做时编写测试。这是合乎逻辑的，如果我们从字面上理解 TDD，就像 Kent Beck 在书中所描述的一样。

    在没有面前的课程的情况下，编写一份测试是困难的。如果我们谈论的是真正的代码，而不是计算器示例，我甚至会说这是不可能的。这也非常低效，因为测试本质上比它们验证的代码更加严格—首先创建它们将导致许多重新循环，直到设计稳定下来。

    在过去的四年里，我个人用Java、Ruby、PHP和JavaScript写了将近30万行代码，并且我从未按照书本上的方式进行过测试驱动开发（TDD）：”编写一个测试，使其运行，使其正确”。

    Code, Deploy, Break, Test, Fix

    虽然我是自动化测试（单元测试或集成测试）的超级粉丝，并且完全同意Uncle Bob的观点：那些不写测试的人应该被关进监狱，但我对TDD有自己的理解。这就是我的理解：

    • 首先，我编写没有任何测试的代码。大量的代码。我实现功能并创建设计。几十个类。当然，构建是自动化的，部署管道已配置好，我可以在沙盒中自行测试产品。我确保“它在我的机器上正常运行”。

    • 然后，我将其部署到生产环境中。是的，它会直接提供给我的“用户”，因为我觉得它对我来说是有效的，所以没有经过任何测试。如果是一个开源项目或者是我自己的宠物项目，那么这些用户可能是真实用户；如果是一个盈利项目，那么这些用户可能是手动测试人员。

    • 然后，他们会对其进行打破。无论是测试还是使用它，都没有关系。他们只是发现问题并报告错误。尽可能多地报告。

    • 在一些错误报告后，我选择其中最严重的错误…嘿！…我创建了一个自动化的测试。错误对我来说是我的测试不够强大的信号；我必须先修复它们。一个新的测试将证明代码是有问题的。或者我修复一个已有的测试。这就是我所谓的“先测试”。在我成功破坏构建并用新的测试证明问题存在之前，我不会触碰生产代码。然后，我进行git commit。

    • 最后，是时候修复这个问题了。我对生产代码进行了修改，以确保构建再次变为绿色。然后，我执行 git commit 和 git push。然后，我回到”部署”步骤；更新后的产品将发送给我的用户。

    偶尔，我不得不对产品进行重大修改，比如引入新功能或进行大规模重构。在这种情况下，我会回到第一步，并且不进行测试。

    The Reasoning Behind

    这种“不进行测试前期”的做法背后的理由很简单：我们不需要在出现问题之前进行测试，主要是因为我们明白测试所有东西或修复所有错误在技术上是不可能的。我们只需要修复对业务可见且无法忍受的问题。如果业务不关心或者我们的用户/测试人员看不到我们的错误，我们就不应该“浪费”项目资源来修复它们。

    另一方面，当业务或我们的用户/测试人员抱怨时，我们必须对自己非常严格；我们的测试系统很薄弱，必须首先修复。我们不能只修复生产代码并部署，因为在这种情况下，经过一些重构后我们可能会再次犯同样的错误，而我们的测试无法捕捉到它。用户会再次发现这个错误，业务会再次支付我们来修复。那将是资源的浪费。

    正如你所见，一切都是以金钱为驱动的。首先，如果没有人为此付费，就不要修复任何问题。其次，如果他们真的付了款，那就一劳永逸地修复它。就是这样简单。

    The Dynamics

    由于这种只在出现问题时进行测试和修复的方法，项目的整个生命周期中生产代码和测试代码的平衡并不相同。当项目开始时，几乎没有测试。然后，随着错误数量的增加，测试数量也增加。最终，情况稳定下来，我们可以将产品从测试版移至首个发布版本。

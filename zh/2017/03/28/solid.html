    SOLID Is OOP for Dummies

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    你肯定了解SOLID这个首字母缩略词。它代表了五个面向对象编程的原则，如果遵循，可以使你的代码既易读又易扩展。这些原则几乎是30年前引入的，但在此期间它们真的使我们成为更好的程序员了吗？我们真的因此更好地理解面向对象编程吗？我们写出了更多“易读和易扩展”的代码吗？我不这样认为。

    让我们逐一看看它们是如何“帮助”的。

    S

    “S” 指的是 单一职责原则，根据 Clean Code 一书中的 Robert Martin 所述，意味着 “一个类应该只有一个变化的原因”。

    这个陈述听起来对我来说非常模糊，但是书中解释说，对象必须以问题为中心，并且负责“一件事情”。当然，决定这个“一件事情”是由我们来决定的。

    这就是我们所知的“高内聚”（high cohesion），自从1974年Larry Constantine在IBM Systems Journal中写到它以来。为什么需要在15年后创建一个带有模糊名称和非常有争议的定义的新原则呢？

    O

    这封信是关于开闭原则，该原则由Bertrand Meyer在1988年的《面向对象软件构造》（http://amzn.to/2lNxy44）中提出。简单来说，它意味着一个对象不应该可修改。我非常赞同这一观点。

    但是，它还说它应该是可扩展的，字面上通过实现继承来实现，这被称为一种反面的面向对象编程技术。因此，这个原则并不适用于对象和面向对象编程。它可能适用于模块和服务，但不适用于对象。

    L

    第三个字母是关于Liskov替换原则，该原则由Barbara Liskov在1987年提出。这是SOLID五原则中最无害的一部分。简单来说，它表明如果你的方法需要一个Collection，那么ArrayList也能够工作。

    它也被称为子类型化，是任何面向对象语言的基础组成部分。为什么我们需要将其称为原则并“遵循”它呢？是否可能在没有子类型化的情况下创建任何面向对象的软件？如果这是一个原则，让我们也在这里添加“变量”和“方法调用”。

    说实话，我怀疑这个原则被添加到SOLID中主要是为了在”SO”和”ID”之间填补某种空白。

    I and D

    我猜他们两个都是在罗伯特·马丁在施乐公司工作时介绍的。

    接口隔离原则指出，如果你只需要Collection x甚至Iterable x，就不应该声明List x。我非常同意。让我们看下一个。

    依赖反转原则意味着你必须声明List x，而不是ArrayList x，并让对象的提供者决定它是ArrayList还是LinkedList.对我来说，这个原则听起来也很合理。

    然而，所有这些与Constantine在1974年引入的“松散耦合”有什么不同呢？我们真的需要简化和模糊以便更好地学习吗？不，不是为了更好地学习，而是为了更好地销售。这就是我的观点。

    My point is…

    这些原则的要点是它们只是以非常基本、模糊和市场化的方式解释了“内聚性和耦合性”给新手。新手会购买书籍、研讨会和培训课程，但真的无法理解背后的逻辑。他们真的需要吗？他们只是码农，对吧？

    “但是一个对象必须负责一件事！”这是我经常在会议上听到的。人们会在不了解内聚性是什么，也不理解他们所祈祷的“一件事”是什么的情况下学习这个口号。其实，并不存在所谓的“一件事”，朋友们！内聚性存在不同的层次。

    谁有罪？Uncle Bob & Co.

    他们不比Ridley Scott和其他好莱坞的赚钱高手更好，他们只是制作原始和容易引人落泪的电影以赚取利润。人们通过观看这些电影变得更加愚蠢，但这并不是他们所关心的。面向对象编程的魔法原理也是如此，程序员依赖于它们，认为真理就在那里，而真正的真理甚至连这个“魔法”的创造者都没有理解。

    SOLID 是一种赚钱的工具，而不是用于改善代码的工具。

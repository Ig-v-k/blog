    Traits and Mixins Are Not OOP

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    让我一开始就说清楚，我们将在这里讨论的特性是纯粹由那些急需进行脑叶切除手术的人给面向对象编程带来的“毒药”，正如David West在他的《对象思考》一书中所建议的那样。这些特性有不同的名称，但最常见的是“traits”（特质）和“mixins”（混入）。我真的搞不懂当我们拥有这些特性时，我们还如何称之为面向对象编程。（参考链接已省略技术术语和专有名词翻译）

    首先，简单来说它们的工作原理如下。让我们以Ruby模块作为示例实现。假设我们有一个名为Book的类：

    class Book
  def initialize(title)
    @title = title
  end
end

    现在，我们希望类Book使用一个静态方法（一个过程），它能够执行一些有用的操作。我们可以将其定义在一个实用类中，然后让Book来调用它：

    class TextUtils
  def self.caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  def print
    puts "My title is #{TextUtils.caps(@title)}"
  end
end

    或者我们可以使其更加“方便”并且扩展我们的模块，以便直接访问其方法。

    module TextModule
  def caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  extend TextModule
  def print
    puts "My title is #{caps(@title)}"
  end
end

    看起来不错——如果你不理解面向对象编程和静态方法之间的区别的话。而且，就算我们暂时忘记面向对象纯洁性，这种方法对我来说实际上看起来更不易读，尽管它的字符更少；当它被调用为#{caps(@title)}而不是#{TextUtils.caps(@title)}时，很难理解caps()方法是从哪里来的。你不这么认为吗？

    当我们使用include命令时，Mixin的作用开始更好地发挥。我们可以组合它们来构建我们想要的类的行为。让我们创建两个Mixin。第一个称为PlainMixin，将以原样打印书的标题，而第二个称为CapsMixin，将对已经打印的内容进行大写处理。

    module CapsMixin
  def to_s
    super.to_s.split.map(&:capitalize).join(' ')
  end
end
module PlainMixin
  def to_s
    @title
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    调用不包含混合的Book将按原样打印其标题。一旦我们添加了include语句，to_s的行为将被覆盖，print方法将产生不同的结果。我们可以组合混合以生成所需的功能。例如，我们可以再添加一个混合，将标题缩短为16个字符：

    module AbbrMixin
  def to_s
    super.to_s.gsub(/^(.{16,}?).*$/m,'\1...')
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include AbbrMixin, CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    我相信你已经明白它们都可以访问类Book的私有属性@title。实际上，它们对类中的一切都有完全访问权限。它们实际上是我们注入到类中的”代码片段”，使其更强大和复杂。这种方法有什么问题呢？

    这与注解、数据传输对象（DTO）、getter方法和实用类的问题是一样的——它们将对象拆散并将功能的各个部分放在对象无法看到的地方。

    在混合模式的情况下，功能位于 Ruby 的 modules 中，它们对 Book 的内部结构做出了假设，并进一步假设程序员在内部结构发生变化后仍然能理解 Book 中的内容。这样的假设完全违反了封装的核心理念。

    这种混合通用性和对象私有结构之间的紧密耦合只会导致难以维护和理解的代码。

    非常明显的替代品是可组合的装饰器。请参考文章中给出的示例：

    Text text = new AllCapsText(
  new TrimmedText(
    new PrintableText(
      new TextInFile(new File("/tmp/a.txt"))
    )
  )
);

    难道它看起来不是与我们之前用Ruby mixins做的非常相似吗？

    然而，与混入不同，装饰器使对象小而紧凑，在其上层添加额外的功能。混入则相反——它们使对象更复杂，因此不易阅读和维护。

    我真诚地认为它们只是有害物质。无论是谁发明它们的，都离理解面向对象设计的哲学相差甚远。

    Traits and Mixins Are Not OOP

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    让我一开始就说清楚，我们将在这里讨论的功能是由那些迫切需要进行脑叶切除手术的人带给面向对象编程的纯粹的”毒药”，就像大卫·韦斯特在他的《对象思维》一书中建议的那样。这些功能有不同的名称，但最常见的是特征和混入。当这些特征存在时，我真的无法理解我们如何仍然称之为面向对象编程。

    首先，以下是它们的工作原理。让我们使用Ruby模块作为一个示例实现。假设我们有一个类Book：

    class Book
  def initialize(title)
    @title = title
  end
end

    现在，我们希望类Book使用一个有用的静态方法（一个过程）。我们可以在一个实用类中定义它，并让Book调用它：

    class TextUtils
  def self.caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  def print
    puts "My title is #{TextUtils.caps(@title)}"
  end
end

    或者我们可以使其变得更加”方便”，并通过”扩展”我们的模块以直接访问其方法：

    module TextModule
  def caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  extend TextModule
  def print
    puts "My title is #{caps(@title)}"
  end
end

    这看起来很好——如果你不理解面向对象编程和静态方法之间的区别的话。而且，如果我们暂时忘记面向对象编程的纯洁性，这种方法实际上对我来说看起来更不可读，即使它的字符较少；当它被像#{caps(@title)}这样调用时，很难理解caps()方法是从哪里来的，而不是#{TextUtils.caps(@title)}。你不觉得吗？

    当我们include它们时，混合起来的效果更好。我们可以将它们组合起来构建我们正在寻找的类的行为。让我们创建两个混合。第一个将被称为PlainMixin，并以原样打印书的标题，第二个将被称为CapsMixin，并将已打印的内容大写：

    module CapsMixin
  def to_s
    super.to_s.split.map(&:capitalize).join(' ')
  end
end
module PlainMixin
  def to_s
    @title
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    如果在不包括混合的情况下调用 Book，它的标题将会按原样打印出来。一旦我们添加了 include 语句，to_s 方法的行为就会被覆盖，print 方法会产生不同的结果。我们可以组合混合来实现所需的功能。例如，我们可以添加一个额外的混合，将标题缩短为16个字符：

    module AbbrMixin
  def to_s
    super.to_s.gsub(/^(.{16,}?).*$/m,'\1...')
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include AbbrMixin, CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    我相信你已经明白它们都可以访问类Book的私有属性@title。实际上，它们对类中的所有东西都有完全的访问权限。它们实际上就是我们注入到类中的“代码片段”，使其更强大和复杂。这种方法有什么问题？

    这与批注、数据传输对象（DTO）、获取器和实用类是同样的问题–它们将对象拆分，并将功能的各个部分放置在对象看不见的地方。

    在 mixins 的情况下，功能位于 Ruby 中的 modules，它对 Book 的内部结构做出了假设，并进一步假设程序员在内部结构改变后仍然能理解 Book 中的内容。这种假设完全违反了封装的基本思想。

    混合和对象私有结构之间如此紧密的耦合只会导致难以维护和理解的代码。

    非常明显的替代方法是可组合的装饰器。请参考文章中提供的示例。

    Text text = new AllCapsText(
  new TrimmedText(
    new PrintableText(
      new TextInFile(new File("/tmp/a.txt"))
    )
  )
);

    它看起来不是很类似于我们之前在Ruby中使用的mixin吗？

    然而，与混合(mixins)不同，装饰器(decorators)使对象更小且更具内聚性，为其添加额外的功能。混合(mixins)则相反，它们使对象更复杂，因此更难以阅读和维护。

    我真诚地认为它们只是有害的。无论是谁发明它们的人都离真正理解面向对象设计的哲学还有很长的路要走。

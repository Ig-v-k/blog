    Traits and Mixins Are Not OOP

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    让我首先明确一点，我们在这里讨论的特性是纯粹的“毒药”，是那些迫切需要脑叶切除手术的人为面向对象编程带来的，就像大卫·韦斯特在他的《对象思维》一书中所建议的那样。这些特性有不同的名称，但最常见的是traits和mixins。当我们拥有这些特性时，我真的无法理解我们如何仍然称之为面向对象编程。

    首先，简单说一下它们的工作原理。让我们以Ruby模块作为一个示例实现。假设我们有一个类Book：

    class Book
  def initialize(title)
    @title = title
  end
end

    现在，我们希望类Book使用一个有用的静态方法（一个过程）。我们可以在一个实用类中定义它，然后让Book调用它：

    class TextUtils
  def self.caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  def print
    puts "My title is #{TextUtils.caps(@title)}"
  end
end

    或者，我们可以使它变得更加“方便”，并“扩展”我们的模块以便直接访问其方法：

    module TextModule
  def caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  extend TextModule
  def print
    puts "My title is #{caps(@title)}"
  end
end

    看起来不错——如果你不了解面向对象编程和静态方法之间的区别的话。而且，如果我们暂时忘记OOP的纯粹性，这种方法对我来说实际上看起来更不易读，尽管它的字符数更少；当它被调用为#{caps(@title)}而不是#{TextUtils.caps(@title)}时，很难理解方法caps()是从哪里来的。你不这么认为吗？

    当我们 include（包含）它们时，Mixins开始发挥更好的作用。我们可以将它们组合起来构建我们正在寻找的类的行为。让我们创建两个mixins。第一个称为PlainMixin，将以原样打印书的标题，第二个称为CapsMixin，将将已经打印的内容转为大写：

    module CapsMixin
  def to_s
    super.to_s.split.map(&:capitalize).join(' ')
  end
end
module PlainMixin
  def to_s
    @title
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    调用未包含mixin的Book将以原样打印其标题。一旦我们添加了include语句，to_s方法的行为将被覆盖，print方法将产生不同的结果。我们可以组合mixins来产生所需的功能。例如，我们可以再添加一个mixins，它将会将标题缩写为16个字符。

    module AbbrMixin
  def to_s
    super.to_s.gsub(/^(.{16,}?).*$/m,'\1...')
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include AbbrMixin, CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    我相信你已经理解他们都可以访问类Book的私有属性@title。实际上，他们对类中的所有内容都有完全访问权限。它们实际上是我们注入到类中的”代码片段”，以使其更强大和复杂。这种方法有什么问题？

    这个问题与注解、数据传输对象(DTO)、getter方法和实用类一样——它们会将对象拆分，并将功能的不同部分放置在对象无法看到的地方。

    对于混合，功能在Ruby“模块”中，它对Book的内部结构做出假设，并进一步假设程序员在内部结构更改后仍然能理解Book中的内容。这样的假设完全违反了封装的核心理念。

    这种混合物和对象私有结构之间的紧密耦合只会导致难以维护和理解的代码。

    非常明显的替代选择是可组合的装饰器。请看文章中给出的例子：

    Text text = new AllCapsText(
  new TrimmedText(
    new PrintableText(
      new TextInFile(new File("/tmp/a.txt"))
    )
  )
);

    它看起来不是很类似于我们之前在Ruby混入中所做的吗？

    然而，与混入不同，装饰器使对象小而紧密，将额外的功能层叠在其上。混入则相反——它们使对象更加复杂，因此更难阅读和维护。

    我真诚地认为它们只是一种有害物质。无论是谁发明它们的人，都离理解面向对象设计的哲学相去甚远。

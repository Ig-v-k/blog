    Fluent Interfaces Are Bad for Maintainability

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    流畅接口，最早由马丁·福勒（Martin Fowler）提出的术语，是一种非常方便的面向对象编程中与对象通信的方式。它使它们的外观更易于使用和理解。然而，它破坏了它们的内部设计，使它们更难以维护。Marco Pivetta在他的博文流畅接口是邪恶的中谈到了这个问题；现在我也要加入我的几分意见。

    让我们来看看我自己的库jcabi-http，几年前我创建了它，当时我认为流畅的接口是一件好事。下面是使用该库进行HTTP请求并验证其输出的方法：

    String html = new JdkRequest("https://www.google.com")
  .method("GET")
  .fetch()
  .as(RestResponse.class)
  .assertStatus(200)
  .body();

    这种方便的方法链式调用让代码变得简短明了，对吧？表面上是这样的。但是库的类的内部设计，包括JdkRequest，就是你看到的那个类，并不优雅。最大的问题是它们相当大，而且要扩展它们几乎是不可能的，因为这会让它们变得更大。

    例如，现在JdkRequest有method()、fetch()和其他几个方法。当需要新功能时会发生什么？唯一的方法就是通过添加新方法来扩大类的规模，这样就会危及其可维护性。例如，在这里我们添加了multipartBody()，在这里我们添加了timeout()。

    每当我在jcabi-http中接收到一个新的功能请求时，我总是感到害怕。我明白这很可能意味着要在Request、Response和其他已经臃肿的接口和类中添加新的方法。

    我实际上尝试在图书馆里做一些事情来解决这个问题，但这并不容易。看看这个.as(RestResponse.class)方法调用。它的作用是使用RestResponse装饰一个Response对象，使其拥有更多方法。我只是不想让Response包含50多个方法，就像其他许多库一样。以下是它的功能（这是伪代码）：

    class Response {
  RestResponse as() {
    return new RestResponse(this);
  }
  // Seven methods
}
class RestResponse implements Response {
  private final Response origin;
  // Original seven methods from Response
  // Additional 14 methods
}

    正如您所见，我将所有可能的方法都放在补充装饰器RestResponse、JsonResponse、XmlResponse和其他装饰器中（其他）。这样做很有帮助，但是为了使用类型为Response的中心对象编写这些装饰器，我们必须使用那个”丑陋”的方法as()，它严重依赖于反射和类型转换。

    换句话说，流利接口意味着大型类或一些丑陋的变通方法。我之前提到过这个问题，当我写了关于Streams API和接口Stream时，它是完全流畅的。总共有43个方法！

    这是流畅接口的最大问题——它们强制对象变得庞大。

    流畅接口非常适合其用户，因为所有方法都集中在一个地方，类的数量非常少。使用它们非常容易，尤其是在大多数集成开发环境中使用代码自动完成。它们还使得客户端代码更易读，因为“流畅”构造看起来类似于普通英语（也称为领域特定语言）。

    这完全是真的！然而，它们对对象设计造成的损害是代价太高。

    什么是替代选择？

    我建议你使用装饰器和智能对象。如果我现在可以重新设计jcabi-http，我会这样做：

    String html = new BodyOfResponse(
  new ResponseAssertStatus(
    new RequestWithMethod(
      new JdkRequest("https://www.google.com"),
      "GET"
    ),
    200
  )
).toString();

    这段代码与上面的第一个片段相同，但它更加面向对象。当然，这段代码的明显问题是IDE几乎无法自动完成任何操作。此外，我们将不得不记住许多类的名称。而对于习惯流畅接口的人来说，这种构造方式看起来相当复杂难懂。此外，它与DSL的理念相去甚远。

    但是这里列出了一些好处。首先，每个对象都很小，非常凝聚，并且它们都是松耦合的——这在面向对象编程中是明显的优点。其次，向库中添加新功能就像创建一个新的类一样简单；无需修改现有的类。第三，由于类很小，单元测试更加简化。第四，所有类都可以是不可变的，这也是面向对象编程中的明显优点。

    因此，似乎存在着实用性和可维护性之间的冲突。流畅的接口对用户来说很好，但对库开发人员来说不好。小型对象对开发人员来说很好，但难以理解和使用。

    似乎是这样，但只有当你习惯了庞大的类和过程式编程时才会这样认为。对我来说，大量的小类似乎是一个优势，而不是一个缺点。内部明晰、简单、易读的库更容易使用，即使我不确定哪些类最适合我。即使没有代码自动补全，我也能自己找到答案，因为代码整洁清晰。

    此外，我经常发现自己对扩展现有功能非常感兴趣，无论是在我的代码库中还是通过向库发送拉取请求。如果我知道我引入的更改是隔离的且易于测试，那么我会更有兴趣这样做。

    因此，我不再提供流畅接口，只提供对象和装饰器。

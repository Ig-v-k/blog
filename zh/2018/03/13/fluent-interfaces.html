    Fluent Interfaces Are Bad for Maintainability

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Fluent interface, first coined as a term by Martin Fowler, is a very convenient way of communicating with objects in OOP. It makes their facades easier to use and understand. However, it ruins their internal design, making them more difficult to maintain. A few words were said about that by Marco Pivetta in his blog post Fluent Interfaces are Evil; now I will add my few cents.

    让我们以我自己的库jcabi-http为例，这是我几年前创建的，当时我认为流畅接口是一件好事。下面是使用该库进行HTTP请求并验证其输出的方法：

    String html = new JdkRequest("https://www.google.com")
  .method("GET")
  .fetch()
  .as(RestResponse.class)
  .assertStatus(200)
  .body();

    这种方便的方法链式使得代码简短明了，对吧？是的，在表面上是这样的。但是，库的类的内部设计，包括JdkRequest，也就是你看到的那个类，离优雅还有很大的距离。最大的问题是它们相当庞大，而且很难甚至不可能在不让它们变得更庞大的情况下进行扩展。

    例如，现在JdkRequest有method()，fetch()和其他一些方法。当需要新功能时会发生什么？唯一的方法是通过添加新方法来使类变得更大，这就危及了其可维护性。例如，这里我们添加了multipartBody()，而这里我们添加了timeout()。

    当我在jcabi-http中收到一个新的功能请求时，我总是感到害怕。我明白这很可能意味着要在Request、Response和其他已经臃肿的接口和类中添加新的方法。

    我实际上尝试在图书馆里做一些事情来解决这个问题，但并不容易。看看这个.as(RestResponse.class)方法调用。它的作用是用RestResponse装饰一个Response，以使其具有更多方法。我只是不想让Response包含50多个方法，就像其他许多库一样。这是它的功能（这是伪代码）：

    class Response {
  RestResponse as() {
    return new RestResponse(this);
  }
  // Seven methods
}
class RestResponse implements Response {
  private final Response origin;
  // Original seven methods from Response
  // Additional 14 methods
}

    正如您所见，我没有将所有可能的方法添加到Response中，而是将它们放在补充装饰器RestResponse，JsonResponse，XmlResponse和其他中。这样做很有帮助，但为了使用类型为Response的中央对象编写这些装饰器，我们必须使用那个”丑陋”的方法as()，它严重依赖于反射和类型转换。

    换句话说，流畅的接口意味着大型类或一些丑陋的解决方法。我之前提到过这个问题，当我写到Streams API和接口Stream时，它是完全流畅的。它有43个方法！

    这是流畅接口的最大问题——它们迫使对象变得庞大。

    流利接口非常适合其用户，因为所有方法都集中在一个地方，而且类的数量非常少。使用它们很容易，尤其在大多数IDE中配合代码自动完成功能。它们还使客户端代码更易读，因为“流利”结构看起来类似于普通英语（也称为DSL）。

    这一切都是真的！然而，它们对对象设计造成的破坏是代价，而这个代价太高了。

    什么是替代选择？

    我建议您使用装饰器和智能对象代替。如果我现在有机会重新设计jcabi-http，我会这样设计：

    String html = new BodyOfResponse(
  new ResponseAssertStatus(
    new RequestWithMethod(
      new JdkRequest("https://www.google.com"),
      "GET"
    ),
    200
  )
).toString();

    这段代码与上面的第一个代码片段相同，但它更加面向对象。当然，这段代码的明显问题是，IDE 几乎无法自动完成任何事情。此外，我们将不得不记住许多类的名称。对于习惯流畅接口的人来说，这个结构看起来相当难以阅读。此外，它与 DSL 的想法相去甚远。

    但以下是好处清单。首先，每个对象都很小，非常内聚，并且它们都是松散耦合的——这在面向对象编程中是明显的优点。其次，向库中添加新功能就像创建一个新类一样简单；无需修改现有类。第三，由于类很小，单元测试变得简单。第四，所有类都可以是不可变的，这也是面向对象编程中的明显优点。

    因此，似乎存在着有用性和可维护性之间的冲突。流畅接口对用户来说很好，但对库开发者来说不好。小对象对开发者来说很好，但难以理解和使用。

    似乎是这样的，但只有当你习惯了大型课程和过程式编程时才会这样。对我来说，大量的小类似乎是一种优势，而不是一个缺点。那些清晰、简单、易读的库要容易使用得多，即使我不确定哪些类对我来说最合适。即使没有代码自动补全，我也可以自己解决问题，因为代码是干净的。

    另外，我经常发现自己对扩展现有功能非常感兴趣，无论是在我的代码库中还是通过向库提交拉取请求的方式。如果我知道我所引入的更改是独立的且易于测试，我会更加有兴趣去做这件事。

    因此，我不再使用流畅接口，只使用对象和装饰器。

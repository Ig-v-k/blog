    Operator new() is Toxic

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    为了实例化对象，在大多数面向对象的语言中，包括Java、Ruby和C++，我们使用new()运算符。嗯，除非我们使用静态工厂方法，但我们不使用它们，因为它们是邪恶的。即使看起来很容易随时创建一个新对象，我仍建议对这个有点有害的运算符更加小心。

    我相信你明白这个运算符的问题在于它将对象耦合在一起，使得测试和重用变得非常困难甚至不可能。假设有一个在文件中的故事需要以UTF-8文本形式读取（我在使用Cactoos的TextOf）。

    class Story {
  String text() {
    return new TextOf(
      new File("/tmp/story.txt")
    ).asString();
  }
}

    看起来非常简单，但问题很明显：Story 类无法重复使用。它只能读取一个特定的文件。而且，由于它只能从一个固定位置读取内容，所以测试将变得相当困难，无法进行任何更改。更正式地说，这个问题被称为“不可打破的依赖”—我们无法打破 Story 和 /tmp/story.txt 之间的链接—它们将永远在一起。

    为了解决这个问题，我们需要引入一个构造函数，并让 Story 接受内容的位置作为参数。

    class Story {
  private final File file;
  Story(File f) {
    this.file = f;
  }
  String text() {
    return new TextOf(this.file).asString();
  }
}

    现在，Story的每个用户都必须知道文件的名称：

    new Story(new File("/tmp/story.txt"));

    这并不是特别方便，尤其是对于之前使用“Story”的用户来说，他们对文件路径一无所知。为了帮助他们，我们引入了一个辅助构造函数：

    class Story {
  private final File file;
  Story() { // Here!
    this(new File("/tmp/story.txt"));
  }
  Story(File f) {
    this.file = f;
  }
  String text() {
    return new TextOf(this.file).asString();
  }
}

    现在我们只需通过一个无参数的构造函数来创建一个实例，就像之前一样。

    new Story();

    我相信你很清楚这个技术，也被称为依赖注入。我实际上没有说任何新的东西。我希望你注意这里的位置和三个代码片段中new运算符的数量。

    在第一个片段中，两个new操作符都在text()方法中。在第二个片段中，我们失去了其中一个。在第三个片段中，一个操作符在方法中，而第二个操作符移动到了构造函数中。

    记住这个事实，让我们继续前进。

    如果文件不是UTF-8编码，而是KOI8-R编码，那么TextOf类和Story.text()方法将会抛出异常。然而，TextOf类能够读取任何编码，只需要在构造函数中添加一个附加参数。

    new TextOf(this.file, "KOI8_R").asString();

    为了使Story能够使用不同的编码方式，我们需要引入一些额外的次要构造函数并修改其主要构造函数。

    class Story {
  private final Text text;
  Story() {
    this(new File("/tmp/story.txt"));
  }
  Story(File f) {
    this(f, StandardEncodings.UTF_8);
  }
  Story(File f, Encoding e) {
    this(new TextOf(f, e));
  }
  Story(Text t) {
    this.text = t;
  }
  String text() {
    return this.text.asString();
  }
}

    这只是依赖注入，但要注意new运算符的位置。它们现在都在构造函数中，而不再留在text()方法中。

    这里的趋势对我来说很明显：new 操作符在方法中的使用越多，类的可重用性和可测试性就越低。

    换句话说，new 运算符是一个相当有毒的东西，所以尽量在方法中将其使用降至最低。确保在次要构造函数中实例化每个或几乎每个东西。

    The Educational Aspect of Static Analysis

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    经常，加入我们项目的新程序员会问我们是否有自动格式化工具，以使Java代码看起来与Qulice期望的完全一样。（Qulice是我们使用的静态分析工具。）我总是回答说，拥有这样一个自动化的代码整理工具只会有害，不能帮助项目及其成员改进和成长。下面是我这样认为的原因。

    静态分析，我们与只读主分支的结合方式，是对您的拉取请求进行全自动严格审查，主要用于发现代码格式错误。假设我们希望整个存储库的Java代码看起来像这样：

    final class Doc {
  private final File file;
  public void remove() {
    if (this.file.exists()) {
      this.file.delete();
    }
  }
}

    然而，您将其作为更大任务的一部分进行了重构，并以此提交了一个拉取请求。

    class Doc {
  private File f;
  public void remove()
  {
    if (f.exists())
      f.delete();
  }
}

    对于你们中的一些人来说，这可能并不像一个很大的区别，因为这两个代码片段都可以编译并且以完全相同的方式工作。然而，对于我们这些仓库的维护者来说，这是一件大事。我们确实希望我们的类始终是final的，我们确实希望它们是不可变的（因此所有属性也应该是final的），我们希望在所有属性引用前加上this.，并且我们希望代码的格式保持一致，因为我们相信代码的统一性会极大地增加其可维护性。

    当然，我们可以创建一个工具，您可以使用它来重新格式化代码，使其看起来符合我们的要求。但在那种情况下，您将永远不会了解项目对您的要求以及原因。

    你不会了解我们规则背后的原因。你永远不会去考虑它们。你不会真的在意它们。但它们不仅仅涉及空格和括号的格式。在 Qulice 中有超过 900 条规则，其中一些是专门为我们正在宣扬的面向对象的理念而设计的。

    因此，简单来说，我们不希望您轻松地通过静态分析阶段。我们希望您付出努力以便学习。

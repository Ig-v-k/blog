    Nine Steps of Learning by Refactoring

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    最近在Twitter上有人问我，如果不了解代码的工作原理，如何进行重构。我回答说这是一种“通过重构学习”的方式。然后我尝试在Google上搜索相关内容，却什么也没找到。我感到很惊讶。对我来说，重构似乎是研究源代码最有效、最明显的方法。以下是我通常进行的九个面向对象的步骤。

    根据维基百科的定义，代码重构是”在不改变代码外部行为的情况下，重组现有计算机代码的过程。” 重构的目标是使代码更易读并适合进行修改。

    在他著名的書籍《重構：改善現有程式設計》中，Martin Fowler 提出了一些重構技巧，有助於使程式碼更簡單、更抽象、更易讀等。其中一些從面向物件的角度來看是有爭議的，比如「封裝字段」，但大部分是有效的。

    这是我在不了解代码但需要修改它时通常采用的做法。这些技巧按照复杂性的顺序进行排序，从最简单的开始。

    See the difference?

    当我看到别人的代码时，我首先要做的是让我的IDE上没有“红色标记”。大多数这些红色标记很容易去除，但还有一些需要花些时间进行重构。在这个过程中，我会对我所要处理的“垃圾”程序有很多了解。

    Remove Empty Lines

    我之前曾写过，方法体内的空行是不好的。它们是冗余复杂性的明显指标。程序员倾向于在方法中添加空行以简化代码。

    这是来自Apache Maven代码库的一种方法（随机选择的类 RepositoryUtils，但几乎所有其他类的格式都相同）。

    除了“全红”之外，他们的代码中还有很多空行。删除这些空行将使代码更易读，并且还能帮助我理解其工作原理。较大的方法需要重构，因为没有空行，它们几乎变得难以阅读。因此，我通过将它们分解成更小的方法来压缩、理解和缩小它们。

    Make Names Shorter

    我通常倾向于为变量选择简短的单名词名称，为方法选择简短的单动词名称。我认为更长的“复合”名称是不必要的代码复杂性的指标。

    例如，我在Spring Boot的o.s.b.w.e.u.UndertowServletWebServerFactory类中发现了这个方法registerServletContainerInitializerToDriveServletContextInitializers（69个字符！）。我想知道作者为什么省略了couldYouPlease前缀和otherwiseThrowAnException后缀。

    开玩笑的话，这样长的方法名清楚地表明代码太复杂了，无法用简单的“register”或者“registerContainer”来解释。似乎有许多不同的容器、初始化器、servlet和其他需要以某种方式注册的对象。当我加入一个项目并看到一个带有这个名字的方法时，我就准备好迎接大麻烦了。

    当我开始处理外国或遗留代码时，将名称缩短是我必须进行的重构步骤。

    Add Unit Tests

    大多数类（和方法）没有任何文档，特别是当我们谈论闭源商业代码时。如果这些类具有相对描述性的名称并且规模较小且内聚性强，那我们就算是幸运的。

    然而，我更喜欢处理单元测试而不是文档。它们能更好地解释代码，并证明代码的运行正常。当我不理解一个类的工作原理时，我会尝试为其编写一个单元测试。在很多情况下，这是不可能的，由于很多原因。在这种情况下，我会尝试应用我从Michael Feathers的《与遗留代码高效合作》和Steve Freeman与Nat Pryce的《面向对象软件的演化式设计》这两本书中学到的一切。这两本书主要关注的是当你不知道如何进行测试时该怎么做。

    Remove Multiple Returns

    我之前提到，在一个方法中存在多个return语句并不是面向对象编程应该鼓励的事情。相反，一个方法必须始终有一个单一的退出点，就像函数式编程中的那些函数一样。

    看一下Spring Boot的o.s.b.c.p.b.Binder类中的这个方法（那里有很多类似的例子，我随机选择了这个）。

    在这个小的方法中有五个return语句。对于面向对象的代码来说，这太多了。对于过程式的代码来说还可以，有时我也写这样的代码。例如，我们的这个 Groovy 脚本也有五个return关键字。

    但这是Groovy，它不是一个类。它只是一个过程，一个脚本。

    重构和删除多个return语句无疑有助于让代码更加清晰。主要是因为如果没有它们，就必须使用更深层次的if/then/else语句嵌套，然后代码开始变得丑陋，除非将其分解为更小的部分。

    Get Rid of NULLs

    NULLs（为空）是邪恶的，这是众所周知的事实。然而，它们仍然无处不在。例如，在Spring Boot v2.0.0.RELEASE中有4,100个Java文件和243K行代码，其中包括7,055次使用null关键字。这意味着大约每35行就有一个null。

    相反地，我几年前创立的Takes Framework有771个Java文件，154K行代码和58个null关键字。这大约是每2700行中有一个null。看到区别了吗？

    当你移除 NULL 值时，代码会变得更加清晰，但这并不容易。有时甚至是不可能的。这就是为什么在 Takes 中仍然有那些 58 个 null 的情况。我们无法移除它们，因为它们来自 JDK。

    Make Objects Immutable

    如我之前所示，不可变性有助于保持对象更小。我处理的外部代码中，大多数类都是可变的。而且很大。

    如果您查看由 jpeek 分析的任何工件，您会发现其中大多数约有80%的类是可变的。从可变性到不可变性的转变是面向对象编程中的一个巨大挑战，如果解决了这个问题，将会得到更好的代码。

    这个将事物变为不可变的重构步骤是纯粹有益的。

    Remove Static

    静态方法和属性对于过程式编程人员来说非常方便。如果你的代码是面向对象的，它们就需要消失。在Spring Boot中有7482个static关键字，这意味着每32行代码就有一个。相反，在Takes中我们有310个static，这是每496行出现一个。

    将这些数字与关于NULL的统计数据进行比较，您会发现摆脱”static”是一项更复杂的任务。

    Apply Static Analysis

    这是最后一步，也是最复杂的一步。它之所以复杂是因为我将静态分析器配置到了它们的最大潜力甚至更高。我正在使用 Qulice，它是 Checkstyle、PMD 和 FindBugs 的聚合器。这些工具本身就很强大，但 Qulice 使它们变得更加强大，添加了几十个定制检查。

    我在静态分析中使用的原则是0/100。这意味着要么整个代码库是干净的，没有任何Qulice的投诉，要么就是脏的。没有中间地带。这不是一种非常典型的静态分析观点。大多数程序员只是用这些工具来收集关于他们的代码的“意见”。而我则将它们用作重构的指南。

    How to Be Lazy and Stay Calm

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在软件开发这个行业中，我最沮丧的是在修复小错误之前需要理解大问题范围，尤其是当代码是遗留代码，而且不是我的时候。实际上，当代码是我的时候，更加令人沮丧。在解决一个小问题之前总是需要进行所谓的“深思熟虑”，这让我对编程产生了厌恶情绪。直到我开始以不同的方式思考并鼓励自己变得懒惰。下面是我的做法。

    我几年前在这篇博文中谈到过这个问题：如何偷懒又保持冷静。然而，在我们的Telegram群组中，我们讨论Zerocracy时，一些程序员一遍又一遍地问我相同的问题：当项目对我来说完全陌生，我只有30分钟的时间，而且错误非常复杂时，我应该怎么办？

    Zerocracy的一个核心原则是#NoAltruism。这意味着你应该始终只考虑自己和个人利益。除非你为此付费，否则你不应该试图改进项目、提高其质量、修复代码或重构任何东西。

    首先，当你面对即将获得报酬的任务时，而且你不知道如何解决它时，不要责怪自己。你不应该成为刚刚接触到的遗留代码的专家。严格来说，你不应该成为任何领域的专家。项目不像你的妈妈一样，不指望你聪明或精通技术。它需要你关闭问题票。

    如果错误严重，代码混乱，你不知道花多少时间去理解它，更别提修复它了，你该怪谁呢？嗯，你可以责怪你周围的每个人，但首先你应该责怪代码库本身。你如何责怪它呢？你通过创建新的问题票报告它的低质量，问题票可能会是这样的：

    • 类X没有足够的文档，我不理解它的工作原理。

    • 方法X太复杂了，我不知道它具体是做什么的。

    • 算法X很混乱，我弄不清它在做什么。

    • 这里使用了库X，但我不明白为什么你不使用库Y。

    • 类命名规则并不明确，请进行文档记录。

    • 数据组织的原则并不明显，请记录下来。

    然而，不要犯许多程序员犯的错误，当我们告诉他们票务是解决问题的唯一正确方式时。他们会开始在票务中提问并寻求帮助，就像这样：

    • 请解释一下如何对X类进行单元测试。

    • 请帮我创建X类。

    • 我应该把X类放在哪个包中？

    • 我应该使用哪个库来执行X操作？

    这个项目不是一个学校，它不关心让你变得更聪明或者对它的代码更加熟悉。没人会向你解释任何事情，因为那浪费金钱和时间。相反，这个项目将会修复它的代码库，使之更加清晰明了，对你和其他人来说都更容易理解。因此，不要询问解释或者帮助，而是要求文档和源代码修复。

    接下来你该做什么？你坐下来等待，直到那些问题解决。谁会解决它们？你不需要关心，那是项目经理的问题。也许他/她甚至会把那些问题再分配给你，让你来解决。但如果那样的话，你的工作范围将会有所不同。你不再需要修复错误，而是需要记录一些功能或者重构一些模块。

    在这个新的、更小的范围内，你将会遇到其他问题。你会创建新的问题，指责周围的每个人，然后这些问题可能再次回到你身上。如此循环往复。最终，一个问题的范围将会尽可能小，以便在30分钟内解决。

    看到这个算法了吗？我相信你看到了，但将其应用于现实生活和真实的软件项目是非常困难的，原因有几个明显的心理因素。

    • 你感到羞愧。在你不够聪明的时候，你被训练得会感到内疚。我能说什么呢？就停止吧！

    • 你是一个完美主义者。你希望完成整个工单，解决整个问题，并理解整个范围。我能说什么呢？只要项目继续按小时/月支付你，这个问题就无法解决。一旦他们开始按结果支付报酬，这种病就会得到治愈。

    • 你没有激情。你根本不关心代码质量。你不想让它看起来整洁，甚至分不清什么是整洁，什么是混乱。你只想在月底拿到薪水。在这种情况下，你甚至都不知道该报告哪些问题。我能说什么呢？我猜你得试试找另一份工作。也许做个经理？

    • 你感到害怕。责怪项目并报告问题可能会让人觉得你对代码库和创建它的人有负面态度，但这并不是真实情况。相反，你的态度是积极的，因为你关心它并希望它变得更好。我能说什么呢？让你的问题报告听起来非常客气和温和。但是请继续报告问题。

    • 你没有时间。你现在必须解决问题，没有时间等待你报告的投诉问题的解决。我能说什么呢？责怪管理层，要求更多时间。更多的时间。但永远不要责怪自己。

    软件开发是一个完美的领域来采取捷径、偷懒和保持冷静，因为我们的工作通常是离散的，并且可以非常逐步地进行。很少有可能无法归咎于项目并将工单暂停，直到解决了您所有的抱怨。不过，我无法想象出这样的情况。如果您能想象得出来，请告诉我。

    The source code is too complex, you have to modify it, and you can't understand how it works. Whose fault is it? #codeahead

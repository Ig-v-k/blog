    Object Validation: to Defer or Not?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我之前说过构造函数必须是无代码的，除了属性初始化之外不应该执行其他操作。此后，最经常被问到的问题是：那参数的验证呢？如果参数是“损坏”的，创建一个“无效”的对象有什么意义呢？这样的对象会在后面的一个意外时刻失败。在实例化的那一刻抛出一个异常岂不是更好？所谓的快速失败，也就是这个意思吧？这是我的看法。

    让我们从这段 Ruby 代码开始：

    class Users {
  def initialize(file)
    @file = file
  end
  def names
    File.readlines(@file).reject(&:empty?)
  end
}

    我们可以使用它从文件中读取用户列表。

    Users.new('all-users.txt').names

    有许多滥用此类的方法：

    • 将nil传递给构造函数而不是文件名；

    • 请传递另一个不是String的东西；

    • 传递一个不存在的文件；

    • 传递一个目录而不是一个文件。

    你看到了我们可以犯的这四个错误之间的区别吗？让我们看看我们的班级如何保护自己免受每个错误的影响：

    class Users {
  def initialize(file)
    raise "File name can't be nil" if file.nil?
    raise 'Name must be a String' unless file.is_a?(String)
    @file = file
  end
  def names
    raise "#{@file} is absent" unless File.exist?(@file)
    raise "#{@file} is not a file" unless File.file?(@file)
    File.readlines(@file).reject(&:empty?)
  end
}

    第一和第二个潜在错误在构造函数中被过滤掉了，而其他两个则在方法中被过滤掉了。为什么我要这样做呢？为什么不把它们全部放在构造函数中呢？

    因为前两个妥协了对象的状态，而另外两个妥协了它的运行时行为。你记得对象是一组封装的其他对象的代表，称为属性。Users 类的对象不能代表 nil 或数字。它只能代表一个文件，其名称为 String 类型。另一方面，文件的内容和它是否真的是一个文件并不会使状态无效。它只会给行为带来麻烦。

    尽管差异看起来微妙，但显而易见。与封装对象的交互有两个阶段：连接和通信。

    首先，我们封装了 file 并确保它确实是一个文件。我们还没有开始与其交互，我们只是想确保它真的是一个我们将来能够交互的对象。如果它是 nil 或一个 float，那么我们将来肯定会遇到问题。这就是为什么我们在构造函数中引发一个异常。

    然后第二阶段是对话，我们将控制权委托给对象，并期望它能正确地行为。在这个阶段，我们可能会有其他验证程序，以确保我们的交互顺利进行。需要注意的是，这些验证是非常情境化的。我们可能会多次调用names()，每次都会有与磁盘上的文件不同的情况。一开始可能不存在该文件，而几秒钟后它将准备好并可供读取。

    理想情况下，编程语言应该为第一类验证提供工具，例如严格类型检查。例如，在Java中，我们不需要检查file的类型，编译器会更早地捕获该错误。在Kotlin中，由于他们的空安全功能，我们可以摆脱NULL检查。相比这些语言，Ruby的功能较弱，因此我们必须进行”手动”验证。

    因此，总结起来，在构造函数中进行验证并不是一个坏主意，只要这些验证不涉及到对象本身，而只是确认它们是否足够好以供以后使用。

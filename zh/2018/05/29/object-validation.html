    Object Validation: to Defer or Not?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我之前说过，构造函数必须是无代码的，除了属性初始化之外不应该做任何其他事情。从那时起，最常被问到的问题是：参数验证怎么办？如果它们是“不正确的”，那么创建一个处于“无效”状态的对象有什么意义呢？这样的对象将在以后的某个意外时刻失败。难道不是在实例化的那一刻就抛出异常更好吗？所谓的快速失败，就是这个意思。以下是我的看法。

    让我们从这个 Ruby 代码开始：

    class Users {
  def initialize(file)
    @file = file
  end
  def names
    File.readlines(@file).reject(&:empty?)
  end
}

    我们可以使用它从文件中读取用户列表。

    Users.new('all-users.txt').names

    有许多方法可以滥用这个类：

    将nil传递给构造函数，而不是文件名；

    传递其他类型的东西，而不是 String；

    传递一个不存在的文件；

    传递一个目录而不是一个文件。

    你能看出我们可以犯的这四个错误之间的区别吗？让我们看看我们的班级如何保护自己免受这些错误的影响：

    class Users {
  def initialize(file)
    raise "File name can't be nil" if file.nil?
    raise 'Name must be a String' unless file.is_a?(String)
    @file = file
  end
  def names
    raise "#{@file} is absent" unless File.exist?(@file)
    raise "#{@file} is not a file" unless File.file?(@file)
    File.readlines(@file).reject(&:empty?)
  end
}

    第一和第二个潜在的错误在构造函数中被过滤掉了，而另外两个错误则稍后在方法中被过滤掉。为什么我要这样做呢？为什么不把它们全部放到构造函数里呢？

    因为前两个妥协对象状态，而另外两个妥协的是它的运行时行为。你记得对象是代表它封装的一组其他对象，称为属性。类Users的对象不能代表nil或数字。它只能代表一个具有String类型名称的文件。另一方面，文件包含的内容以及它是否真的是一个文件并不会使状态无效。这只会给行为带来麻烦。

    即使差异看起来微小，但很明显。与封装对象的交互有两个阶段：连接和对话。

    首先，我们封装了 file 并希望确保它确实是一个文件。我们目前还没有与它交流，我们只是想确保它确实是一个我们将来能够交流的对象。如果它是 nil 或者是一个 float，那么我们将来肯定会遇到问题。这就是为什么我们在构造函数中引发异常的原因。

    然后第二阶段是交流，我们将控制权委托给对象，并期望它能正确地表现。在这个阶段，我们可能会有其他验证程序，以确保我们的交互顺利进行。需要提到的是，这些验证非常情境化。我们可能会多次调用 names()，每次都会在磁盘上有不同的情况。首先，它可能不存在，而在几秒钟后，它将准备好并可供读取。

    理想情况下，编程语言应该提供第一类型的验证工具，例如严格类型检查。例如，在Java中，我们不需要检查file的类型，编译器会更早地捕捉到该错误。在Kotlin中，我们可以通过他们的空值安全功能来摆脱NULL检查。Ruby不如这些语言强大，这就是为什么我们必须进行”手动”验证的原因。

    因此，总结来说，在构造函数中进行验证并不是一个坏主意，只要验证过程不涉及对象，而只是确认它们足够好以供以后使用。

    Object Validation: to Defer or Not?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我之前说过构造函数必须无代码，除了属性初始化之外不做其他事情。从那以后，最常被问到的问题是：关于参数的验证怎么办？如果它们是“错误的”，在创建一个“无效”状态的对象有什么意义呢？这样的对象将在后面的某个意想不到的时刻失败。在实例化的那一刻抛出异常不是更好吗？可以说是为了快速失败。这是我认为的。

    让我们从这段 Ruby 代码开始：

    class Users {
  def initialize(file)
    @file = file
  end
  def names
    File.readlines(@file).reject(&:empty?)
  end
}

    我们可以使用它从文件中读取用户列表。

    Users.new('all-users.txt').names

    有许多方法可以滥用这个类：

    • 将 nil 传递给构造函数，而不是文件名；

    • 传递其他东西，这些东西不是String类型；

    • 传递一个不存在的文件；

    • 传递一个目录而不是一个文件。

    你看到我们可以犯的这四个错误之间的区别了吗？让我们看看我们的班级如何保护自己免受每个错误的影响：

    class Users {
  def initialize(file)
    raise "File name can't be nil" if file.nil?
    raise 'Name must be a String' unless file.is_a?(String)
    @file = file
  end
  def names
    raise "#{@file} is absent" unless File.exist?(@file)
    raise "#{@file} is not a file" unless File.file?(@file)
    File.readlines(@file).reject(&:empty?)
  end
}

    第一个和第二个潜在错误在构造函数中被过滤出去了，而另外两个则是在方法中被过滤的。为什么我要这样做呢？为什么不把它们都放到构造函数里呢？

    因为前两个会影响对象的状态，而后两个则会影响对象的运行行为。你还记得一个对象是其封装的其他对象的代表，称为属性。Users类的对象不能代表nil或者一个数字，它只能代表一个文件，文件名的类型为String。另一方面，文件的内容以及它是否真的是一个文件并不会使状态无效，只会对行为造成麻烦。

    尽管这种差异看起来微妙，但显而易见。与封装对象的交互有两个阶段：连接和交流。

    首先，我们封装这个file，并希望确保它确实是一个文件。我们还没有开始与它交流，我们不希望它为我们工作，我们只想确保它是一个我们将来能够与之交流的对象。如果它是nil或者一个浮点数，我们将来肯定会遇到问题。这就是为什么我们在构造函数中引发异常的原因。

    然后第二阶段是交流，我们将控制权委托给对象，并期望它能正确地行为。在这个阶段，我们可能有其他的验证过程，以确保我们的交互能够顺利进行。重要的是要提到这些验证是非常情境化的。我们可能会多次调用names()，每次都会在磁盘上有不同的文件情况。一开始它可能不存在，而几秒钟后它就准备好并可供读取了。

    理想情况下，编程语言应该为第一类型的验证提供工具，例如通过强类型。例如在Java中，我们不需要检查file的类型，编译器会更早地捕捉到这个错误。在Kotlin中，由于它们的Null Safety功能，我们可以摆脱NULL检查。Ruby比这些语言弱大一些，所以我们必须手动进行验证。

    因此，总结起来，在构造函数中进行验证并不是一个坏主意，前提是这些验证不接触对象，只是确认它们是否足够好以供后续使用。

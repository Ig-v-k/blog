    Monolithic Repos Are Evil

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我们都将我们的代码保存在Git版本控制库中。问题是我们是否应该为每个新模块创建一个新的库，还是尽可能将所有内容保存在一个所谓的“单体”库（或简称为“monorepo”）中。市场领导者，如Facebook和Google，倡导第二种方法。我认为他们是错误的。

    让我们以以下JavaScript函数作为示例。它从Zold节点（使用jQuery）下载一个JSON文档，并将其内容的一部分放置在HTML页面上。然后根据数据的值对其进行着色。

    // main.js
function main() {
  $.getJSON('http://b1.zold.io/', function(json) {
    var $body = $('body');
    $body.text(json.nscore);
    var color = 'red';
    if (json.nscore > 500) {
      color = 'green';
    }
    $body.css('color', color);
  });
}

    很明显，不是吗？只有一个 main.js 文件，它能完成我们所需的一切。我们只需将它添加到 HTML 中，它就能工作。

    <html>
  <head>
    <script src="https://code.jquery.com/jquery-3.3.1.min.js"/>
    <script src="main.js"/>
  </head>
  <body onload="main();">loading...</body>
</html>

    现在让我来重构它。让我将其分成两部分。第一部分将加载数据，第二部分将是一个jQuery插件，根据所包含的数据对HTML内容进行着色。这就是插件的样子。

    // colorize.js
$.fn.colorize = function() {
  var data = parseFloat(this.text());
  var keys = Object.keys(colors)
    .map(function (k) { return parseInt(k); })
    .sort(function (a,b) { return a - b; })
    .reverse();
  for (i = 0; i < keys.length; ++i) {
    var max = keys[i];
    if (data >= max) {
      this.addClass(colors[max]);
      return;
    }
    this.removeClass(colors[max]);
  }
  return this;
}

    main.js 将会是这样的：

    // main.js
function main() {
  $.getJSON('http://b1.zold.io/', function(json) {
    $('body')
      .text(json.nscore)
      .colorize({ 500: 'green', 0: 'red' });
  });
}

    现在，我们不再有一个庞大的代码块，而是有两个较小的代码块，它们必须一起加载到目标HTML中。

    <html>
  <head>
    <script src="https://code.jquery.com/jquery-3.3.1.min.js"/>
    <script src="colorize.js"/>
    <script src="main.js"/>
  </head>
  <body onload="main();">loading...</body>
</html>

    两个部分比一个部分更好吗？看起来Google，Digital Ocean和Mozilla并不这么认为。

    I disagree.

    为了说明我的观点，我将JavaScript函数提取到一个新的独立的jQuery插件中。以下是我所做的：

    • 创建了一个新的 GitHub 仓库 yegor256/colorizejs；

    • 阅读说明；

    • 研究了一些jQuery插件，并学习了几个示例；

    • 发现大多数人使用了Gulp，我从未听说过。

    • 决定使用npm进行JavaScript打包（还有其他选择吗，对吧？）；

    • 为npm创建了package.json文件；

    • 当我发现npm包colorize已经存在时，将GitHub仓库更名为colorizejs。

    • 配置了.travis.yml，用于Travis；

    • 创建了一个 README.md 文件，并解释了如何使用和安装它；

    • 决定使用MIT许可证并创建LICENSE.txt文件；

    • 配置了PDD，以实现自动收集谜题；

    • 配置了 .rultor.yml 用于 Rultor；

    • 尝试创建一个单元测试，结果惨败（经过整整一个小时的研究），因为我几乎没有JS单元测试的经验。

    • 在Stack Overflow上发布了一个问题；

    • 只有在我提供悬赏之后，几个人才回答了这个问题。

    • @brian-lives-outdoors的答案是最好的，他甚至提交了一个拉取请求，其中包含一个单元测试，我已经合并了；

    • 发布了第一个版本0.0.1到npmjs.com;

    • 修改了代码，使其适用于类和颜色。

    • 实现并发布了下一个版本0.1.0；

    • 添加了它 到 Zold 前端，进行了测试，并发布了它—在这里查看。

    花费了将近三周的等待和四个小时的工作，只是为了将一小段JavaScript代码转移到一个新的代码库并分别发布。这样值得吗？嗯，我认为是的。但是我找到的大多数其他博客作者认为，最好将所有内容保留在一个单一的巨大代码库中，主要是因为这样对于“生产力”更有益。例如，单一代码库的优势（Dan Luu撰写）, 单一巨大代码库的优势和劣势（Google的案例研究，Ciera Jaspan等人撰写）以及Tomas Votruba的《开源中单一巨大代码库如何拯救我的懒散》。

    还有一些对这两种方法的很好的分析，例如Fabien Potencier在dotScale 2016的演讲Monolithic repositories vs. Many repositories和Peter Seibel的Repo Style Wars: Mono vs Multi。

    简而言之，他们都声称使用单体库可以提高生产力，因为进行更改所需的操作数量较少。确实，在单体库中，将只有一个分支、一组提交、一个拉取请求、一个合并、部署和发布。同时，测试会更容易，无论是手动测试还是单元测试。持续集成更容易配置，等等。

    所有这些“合理”的论点让我想起了当我宣讲对象分解并建议多个对象比一个大的对象更好时听到的内容。想象一下一个有3000行代码的大类，它做了很多事情，并且它们之间非常紧密耦合。测试它，进行更改，部署，审查等都是“容易”的。因为一切都保留在一个文件中，对吧？我们不需要在不同的类之间跳转来理解设计。我们只需要看一个屏幕，上下滚动，就可以了。对吗？完全错误！

    我猜我不需要解释为什么它是错的。我们不再以那种方式设计我们的软件。我们知道紧耦合是一个坏主意。我们知道一组更小的组件比一个更大的坚固部件更好。

    为什么我们不能将同样的逻辑应用于代码库呢？我相信我们可以。当然，就像面向对象编程一样，精细化的设计需要更多的技能和时间。看看我为这个小的jQuery插件所做的工作吧。我花了几个小时来编码和思考。我甚至不得不学习Gulp和Jasmine，而我很可能不再使用它们了。但是我们从中获得的好处是巨大的。这是我对它们的简要列表：

    • 封装。 每个存储库封装了一个单独的问题，将其细节隐藏起来，不让其他人知道。幸运的是，这样一来，每个存储库需要处理的范围就变得更小了。范围越小，就像面向对象编程一样，维护和修改就越容易。维护越容易，开发成本就越低。我猜谷歌的人们并不真的担心成本。相反，他们希望自己的薪水增长。一个大而难以维护的单体存储库是实现这一目标的完美工具。

    • 快速构建。当存储库较小时，其自动化构建所需的时间较短。看看 Travis 在我的 jQuery 插件上花费的时间。它只需要 51 秒，速度很快。我们都知道，构建速度越快，对于生产力来说就越好，因为构建更容易用作开发工具。

    • 准确的度量。我不知道您在项目中是否依赖度量指标，但我们在Zerocracy确实关注数字，比如代码行数、代码击中数、提交次数、类、方法、内聚性、耦合度等。度量指标的准确性一直是个问题。计算一个大型代码库的代码行数没有任何意义，因为这个数字将包含应用程序完全不同部分的许多文件。此外，还会涉及不同的语言和文件格式。假设一个代码库有20万行Java代码，15万行XML代码，5万行JavaScript代码和4万行Ruby代码。您能对这个代码库说一些具体的东西吗？它算大吗？它是一个Java代码库吗？更重要的是，它能与其他代码库进行比较吗？实际上不能。它只是一个杂乱无章的文件存储。

    • 同质任务。较小的代码库往往具有较小的技术堆栈，这意味着每个代码库只使用少量语言和框架，或者（这是首选情况）—每个代码库只使用一种语言或技术。由于这一点，程序员的管理变得更容易，因为任何工单/问题都可以分配给任何人。这样可以更容易地使任务的大小和复杂度相似。这显然意味着项目更易于管理。

    • 单一编码标准。如果代码仓库较小，更容易统一编码风格。当仓库较大时，代码库的不同部分将会有不同的风格，几乎不可能使每个人达成一致。换句话说，较小的代码仓库比较大的仓库看起来更美观。

    • 短名称。每个存储库都不可避免地拥有其自己的命名空间。例如，在我刚创建的 JS 存储库中，我只有两个文件：colorizejs.js 和 test-colorizejs.js。我对其中的命名并不在意，因为命名空间非常小。短名称和较小的命名空间意味着更好的可维护性。

    • 简单测试。代码库越大，其依赖关系就越多，这些依赖关系很难进行模拟和测试。非常大的代码库变得基本无法测试，因为它们需要大量的集成测试，而这些测试很难维护。较小的库、框架和模块更容易保持在简单和快速单元测试的水平上。

    因此，我认为代码库和模块越小越好。理想情况下，我会说，代码库的最大可接受大小是50,000行代码。超过这个限制的部分都很适合进行拆分。

    What do you think is better, a bigger code repository with everything inside, or many smaller ones with their own builds, dependencies, issues, and pull requests?

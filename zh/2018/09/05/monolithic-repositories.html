    Monolithic Repos Are Evil

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我们都将我们的代码保存在Git版本控制仓库中。问题是我们是否应该为每个新模块创建一个新的仓库，还是尽可能保留在一个所谓的“单体”仓库中（或简称为“单体仓库”）。市场领导者，如Facebook和Google，支持第二种方法。我认为他们是错的。

    让我们以以下 JavaScript 函数作为示例。它从Zold节点（使用jQuery）下载一个 JSON 文档，并将其内容的一部分放置在 HTML 页面上。然后根据其值着色数据。

    // main.js
function main() {
  $.getJSON('http://b1.zold.io/', function(json) {
    var $body = $('body');
    $body.text(json.nscore);
    var color = 'red';
    if (json.nscore > 500) {
      color = 'green';
    }
    $body.css('color', color);
  });
}

    很明显，对吧？只需一个 main.js 文件就能满足我们的所有需求。我们只需将其添加到 HTML 中，它就可以工作了。

    <html>
  <head>
    <script src="https://code.jquery.com/jquery-3.3.1.min.js"/>
    <script src="main.js"/>
  </head>
  <body onload="main();">loading...</body>
</html>

    现在，让我来重构它。让我把它分成两个部分。第一部分将加载数据，第二部分将是一个jQuery插件，根据包含的数据来给HTML内容上色。这就是插件的外观。

    // colorize.js
$.fn.colorize = function() {
  var data = parseFloat(this.text());
  var keys = Object.keys(colors)
    .map(function (k) { return parseInt(k); })
    .sort(function (a,b) { return a - b; })
    .reverse();
  for (i = 0; i < keys.length; ++i) {
    var max = keys[i];
    if (data >= max) {
      this.addClass(colors[max]);
      return;
    }
    this.removeClass(colors[max]);
  }
  return this;
}

    main.js 会是这样的：

    // main.js
function main() {
  $.getJSON('http://b1.zold.io/', function(json) {
    $('body')
      .text(json.nscore)
      .colorize({ 500: 'green', 0: 'red' });
  });
}

    现在，我们不再有一个庞大的代码块，而是有两个较小的代码块，这两个代码块必须一起加载到目标HTML中。

    <html>
  <head>
    <script src="https://code.jquery.com/jquery-3.3.1.min.js"/>
    <script src="colorize.js"/>
    <script src="main.js"/>
  </head>
  <body onload="main();">loading...</body>
</html>

    两件事情胜过一件？看起来Google，Digital Ocean和Mozilla并不这么认为。

    I disagree.

    为了说明我的观点，我将JavaScript函数提取到一个新的独立的jQuery插件中。下面是我所做的事情：

    当我发现npm包colorize已经存在时，我将GitHub仓库改名为colorizejs。

    为 Travis 配置了 .travis.yml；

    花了将近三个星期的等待和四个小时的工作，只是为了将一小段JavaScript代码移到一个新的存储库并单独发布。这值得吗？嗯，我认为是的。但是，我找到的大多数其他博客作者都认为，将所有内容保留在单个庞大的存储库中会更好，主要是因为这样更有利于生产力。例如，单体存储库的优势 由Dan Luu撰写，单体存储库的优缺点（Google的案例研究）由Ciera Jaspan等人撰写，以及 开源中的单体存储库如何拯救了我的懒惰 由Tomas Votruba撰写。

    还有一些对这两种方法的很好的分析，例如Fabien Potencier在dotScale 2016上的演讲Monolithic repositories vs. Many repositories和Peter Seibel的Repo Style Wars: Mono vs Multi。

    简而言之，他们都声称单体存储库的生产力更高，因为为了进行更改所需的操作较少。确实，在单一存储库中将只有一个分支、一个提交集、一个拉取请求、一个合并、一个部署和发布。同时，进行手动测试和单元测试也更容易。连续集成更易于配置，等等。

    所有这些“合理”的论点让我想起了当我宣讲对象分解并建议多个对象比一个大对象更好时听到的内容。想象一下一个有3000行代码的大类，它做了很多事情并且它们都非常紧密耦合。测试它、进行更改、部署、审查等都很“容易”。因为一切都在一个文件中，对吧？我们不需要跳来跳去去理解设计。我们只需要看一个屏幕，上下滚动，就这样。对吗？完全错误！

    我猜我不需要解释为什么这是错误的。我们不再以这种方式设计我们的软件。我们知道紧耦合是一个坏主意。我们知道一组较小的组件比一个较大的固定部件更好。

    为什么我们不能将相同的逻辑应用到仓库中呢？我相信我们可以。当然，就像面向对象编程一样，精细化的设计需要更多的技能和时间。看看我用这个小的jQuery插件做了什么。我花了几个小时来编码和思考。我甚至不得不学习Gulp和Jasmine，而我可能再也不会使用它们了。但是我们从中获得的好处是巨大的。这是我对它们的简短列表：

    封装。每个存储库封装了一个单独的问题，将其详细信息隐藏起来，不让其他人知道。由于这样，每个存储库需要处理的范围变得更小。范围越小，就像面向对象编程（OOP）一样，维护和修改就越容易。维护越容易，开发成本就越低。我猜谷歌的人可能不太担心成本。相反，他们希望自己的薪水增长。一个巨大且难以维护的单体存储库就是实现这一目标的完美工具。

    快速构建。当一个仓库很小时，自动构建所需的时间也很短。看一下 Travis 对我的 jQuery 插件的构建所花费的时间。它只需要 51 秒。速度很快。我们都知道，构建越快，对于生产力来说越好，因为它更容易将构建作为开发工具使用。

    准确的度量指标。我不知道您在项目中是否依赖度量指标，但我们在Zerocracy确实关注数字，比如代码行数，代码点击数，提交次数，类，方法，内聚性，耦合性等等。度量指标是否准确一直是一个问题。对于一个庞大的代码库来说，计算代码行数毫无意义，因为这个数字会包括应用程序完全不同部分的很多文件。而且还会有不同的语言和文件格式。比如一个仓库有20万行Java代码，15万行XML代码，5万行JavaScript代码和4万行Ruby代码。您能对这个仓库说出具体的内容吗？它是否很大？它是一个Java仓库吗？更重要的是，它能与其他仓库进行比较吗？实际上不能。它只是一个大杂乱的文件存储。

    同质任务。规模较小的代码库往往具有较小的技术堆栈，这意味着每个代码库只使用少量的编程语言和框架，或者（这是首选情况）—每个代码库只使用一种编程语言或技术。由于这一点，程序员的管理变得更加容易，因为任何工单/问题可以分配给任何人。任务的大小和复杂性更容易保持一致。这显然意味着项目的管理更加可控。

    单一编码规范。如果代码库很小，标准化编码风格就更容易。当代码库很大时，不同部分的代码将采用不同的风格，要让所有人达成一致几乎是不可能的。换句话说，较小的代码库看起来比较美观。

    简短名称。每个仓库都会有自己的命名空间。例如，在我刚创建的 JS 仓库中，我只有两个文件：colorizejs.js 和 test-colorizejs.js。我对它们内部的命名并不在意，因为命名空间非常小。较短的名称和较小的命名空间意味着更好的可维护性。

    简单测试。代码库越大，它所依赖的内容也越多，这些内容很难进行模拟和测试。非常庞大的代码库基本上无法进行测试，因为它们需要大量的集成测试，而这些测试很难维护。相比之下，较小的库、框架和模块更容易保持在简单和快速单元测试的水平上。

    因此，我相信存储库和模块越小越好。理想情况下，我会说，代码库的最大可接受大小是50,000行代码。超过这个限制的部分都是完美的分解候选项。

    What do you think is better, a bigger code repository with everything inside, or many smaller ones with their own builds, dependencies, issues, and pull requests?

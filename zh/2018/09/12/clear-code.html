    Code Must Be Clean. And Clear.

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    有一本由罗伯特·马丁所写的著名书籍叫做《代码整洁之道》。这个书名对我们所有人来说是一个明显的呼吁：代码必须要“整洁”。我想，整洁就像厨房一样——没有脏盘子，地板上没有垃圾，毛巾也不发臭。根据马丁的观点，代码中需要清理的包括大型方法、命名不具描述性的变量、紧耦合、缺乏SOLID和SRP的遵循，以及其他许多问题。读这本书是值得的。然而，源代码还有另一个方面需要考虑，那就是它有多么“清晰”。

    当烤箱没有污垢时，厨房是干净的。但是如果它的电控面板是法语的，我就无法使用厨房。即使它非常干净。如何使用它并不清楚—这就是为什么它没用。

    这个隐喻适用于源代码。使其干净是第一步，也是非常重要的一步，它将消除所有那些书籍中谈到的编码反模式，包括我最喜欢的 Code Complete（作者：Steve McConnell）、Working Effectively With Legacy Code（作者：Michael Feathers）和 Clean Code。这是一个非常重要的步骤，但不是最重要的。一个有用的脏厨房比我无法使用的干净厨房更好，不是吗？

    让代码变得简洁，但却让其他人难以理解是我们大多数人都会陷入的陷阱。所谓的“其他人”包括我们在同一张桌子上坐着的项目合作开发人员，以及在我们都被谷歌录用后五年后加入项目的富有想象力的初级贡献者。在这个非常长的时间范围内，所有这些人都必须能够在不需要任何额外帮助的情况下使用这段源代码。烤箱必须使用他们的语言，而不是设计者的语言。

    你如何做到这一点？你如何确保代码是清晰的，而不仅仅是整洁的？

    测试一下吧。请一个不参与项目的人来查看你的代码，并告诉你它的清晰程度。不要问他们你的类和代码结构有多美丽——那是使其清晰的原因。相反，让他们在短短30分钟内修复一个错误，看看他们的反应。你将会意识到代码有多么清晰，以及它是否能够以一个陌生人能够理解的语言进行交流。

    这是可维护性的定义。如果一个陌生人能够在不到一个小时内修改你的代码并修复一个错误，那么这个代码是可维护的。显然，代码的整洁会有所帮助。但这还不够。还需要其他一些东西，我不太知道如何描述。唯一实现这一目标的方法是让陌生人定期查看你的代码，尝试做出贡献，并在某些地方不明确时报告错误。

    让你的代码开放，并鼓励程序员在某些内容不仅有问题而且不清楚的情况下报告错误，是实现高可维护性的两种最佳方式。

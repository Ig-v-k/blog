    Fear of Decoupling

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    对象通过它们的方法进行交流。在主流编程语言（如Java或C#）中，一个对象可能有一组独特的方法，还有一些它被迫拥有的方法，因为它“实现”了某些类型，也被称为“接口”。与许多程序员交谈的经验告诉我，我们大多数人对实现过多接口方法的对象感到非常害怕。我们不想处理它们，因为它们是“多态”的，因此不可靠。这是一个合理的恐惧。让我们试着分析它的来源。

    通常情况下，让我们从一个简单的Java示例开始。这是我要通过PayPal API发送给用户的一笔金额。

    interface Money {
  double cents();
}

    现在我在这里，发送资金的方法是：

    void send(Money m) {
  double c = m.cents();
  // Send them over via the API...
}

    这两段代码被我们称为松耦合。方法send()并不知道提供了哪个类以及方法cents()的具体实现。也许它是一个简单的一美元的常量对象。

    class OneDollar implements Money {
  @Override
  double cents() {
    return 100.0d;
  }
}

    或者它可能是一个更复杂的实体，首先建立网络连接，以获取当前美元兑欧元的汇率，更新数据库，然后返回某些计算的结果。

    class EmployeeHourlyRate implements Money {
  @Override
  double cents() {
    // Fetch the exchange rate;
    // Update the database;
    // Calculate the hourly rate;
    // Return the value.
  }
}

    方法send()并不知道其第一个参数具体是什么。它只能希望方法cents()会正确完成工作。如果失败了怎么办？

    如果我是send()方法的开发者，并且我完全准备好为我方法所造成的错误承担责任，我确实想知道我的合作者是谁。而且，我希望确切地知道他们的工作情况。不仅仅是工作，而是按照我的期望进行工作。最好的情况是我自己编写它们。理想情况下，我想确保在我实现它们后没有人会修改它们。你明白我的讽刺意味，对吧？

    这可能听起来像个笑话，但我听过很多次这种论调。他们说，“与其依赖可恶的多态性，然后花数小时调试我没有编写的东西，还不如完全确保两个部分能够很好地协同工作。” 他们是对的，你知道的。多态性——当一个看似原始的Money类型的对象可以做任何它想做的事情，包括HTTP请求和SQLUPDATE查询——并没有为整个应用程序增加可靠性，对吧？

    No, it doesn’t.

    显然，多态使得开发人员在处理Money类型及其”祖先”时变得更加简单，因为他们不必太关心用户。他们唯一需要担心的是在调用cents()时如何返回double。他们不需要担心速度、潜在异常、内存使用等许多其他事情，因为接口并不要求这些。接口只告诉他们返回double并算了事。让其他人来担心其他一切。很简单，是吧？但这种思维方式可能被称为幼稚和自私。

    Yes, it is.

    However…

    您肯定听说过”快速失败”的理念，简而言之，它认为为了使应用程序强大稳定，我们必须确保其组件在面对任何潜在的异常情况时都尽可能脆弱和易受攻击。它们必须在任何可能的情况下都会崩溃，并让用户处理失败。根据这样的理念，任何对象都不会对其对等对象有任何好的假设，并且总是试图将问题升级到更高的层级，最终会影响到最终用户，而最终用户会将这些问题反馈给团队。团队会修复所有问题，整个产品将稳定下来。

    如果哲学相反，每个对象都在其个体微观层面上努力解决问题，那么大部分异常情况将永远不会被用户、测试人员、架构师和程序员看到，而他们本应该处理这些异常情况并找到解决方案。由于个体对象的这种“谨慎”心态，整个应用程序的稳定性和健壮性将受到影响。

    我们可以将相同的逻辑应用于「松耦合的恐惧」。

    当我们担心Money.cents()的工作方式并希望控制其行为时，我们对自己和整个项目都造成了巨大的伤害。从长远来看，我们不是让产品更加稳定，而是使其不稳定。有些人甚至希望通过以这种方式声明方法send()来禁止多态性。

    void send(EmployeeHourlyRate m) {
  // Now I know that it's not some abstract Money,
  // but a very specific class EmployeeHourlyRate, which
  // was implemented by Bobby, a good friend of mine.
}

    在这里，我们限制了我们的代码可能存在的错误数量，因为我们了解Bobby，我们已经看过他的代码，我们知道它是如何工作的以及可以预期的异常情况。我们是安全的。是的，我们是的。就目前而言是的。但从战略上讲，通过不允许我们的软件在所有不寻常的情况下产生所有可能的错误和抛出所有可能的异常，我们严重限制了它进行适当测试的能力，这就是为什么它不稳定。

    正如我之前提到的，提高软件质量的唯一方法就是找出并修复其中的错误。我们修复的错误越多，剩下未发现和未修复的错误就会越少。对错误的恐惧以及我们预防错误的意图只会给自己带来麻烦。

    相反，我们应该让每个人都来实现Money，而不仅仅是Bobby，并将这些实现传递给send()函数。是的，其中一些实现可能会引发问题，甚至可能导致UI可见的失败。但是，如果我们的管理层正确理解了软件质量的概念，他们不会因为错误而责怪我们。相反，他们会鼓励我们尽可能多地发现这些问题，使用自动化测试进行复现，修复并重新部署。

    因此，对于脱钩的恐惧无非就是故障保护。

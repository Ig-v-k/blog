    Fear of Decoupling

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    对象通过它们的方法进行交流。在主流编程语言中，比如Java或C#，一个对象可能具有一组独特的方法，还有一些它被强制要求拥有的方法，因为它实现了某些类型，也被称为接口。我与许多程序员交流的经验告诉我，我们大多数人对实现了太多接口方法的对象感到害怕。我们不想处理它们，因为它们是多态的，因此不可靠。这是一个合理的恐惧。让我们试着分析它的来源。

    像往常一样，让我们从一个简单的Java示例开始。这里是一笔我要通过PayPal API发送给用户的金额。

    interface Money {
  double cents();
}

    现在我在这里，发送资金的方法是：

    void send(Money m) {
  double c = m.cents();
  // Send them over via the API...
}

    这两段代码是我们所说的松耦合。send() 方法并不知道是哪个类提供的，也不知道 cents() 方法是如何实现的。也许它是一个简单的一美元的常量对象：

    class OneDollar implements Money {
  @Override
  double cents() {
    return 100.0d;
  }
}

    或者它是一个更复杂的实体，首先建立一个网络连接，以获取当前美元对欧元的汇率，更新数据库，然后返回某些计算的结果。

    class EmployeeHourlyRate implements Money {
  @Override
  double cents() {
    // Fetch the exchange rate;
    // Update the database;
    // Calculate the hourly rate;
    // Return the value.
  }
}

    No, it doesn’t.

    显而易见，多态性使得开发这种类型的Money及其”祖先”的开发者的生活变得更简单，因为他们不需要太过关心用户。他们只需要考虑在调用cents()时如何返回double。他们不需要关心速度、潜在异常、内存使用等许多其他事情，因为接口并不要求这些。接口只告诉他们返回double就可以了，其他事情就交给别人来操心。很简单，对吧？但这种思维方式可能被认为是幼稚和自私的。

    Yes, it is.

    However…

    你肯定听过“快速失败”这个概念，简单地说，它声称为了使应用程序健壮稳定，我们必须确保其组件尽可能脆弱、易受任何潜在异常情况的影响。它们必须尽可能地经常出错，并让用户处理这些故障。根据这样的理念，任何对象都不会对其对应对象有任何好的假设，而总是试图将问题升级到更高的级别，最终会影响到最终用户，他们将把这些问题反馈给团队。团队将解决所有问题，整个产品将稳定下来。

    如果哲学观相反，每个对象都试图在其个体微观层面处理问题，那么绝大多数特殊情况将对用户、测试人员、架构师和程序员等应该处理并找到解决方案的人来说，是看不见的。由于个体对象的这种“细心”思维方式，整个应用程序的稳定性和健壮性将会受到影响。

    我们可以将相同的逻辑应用于“害怕松耦合”的情况。

    当我们担心Money.cents()的工作方式并希望控制其行为时，我们对自己和整个项目造成了很大的不利影响。从长远来看，我们会削弱该产品的稳定性，而不是使其更加稳定。有些人甚至想通过以这种方式声明方法send()来禁止多态性。

    void send(EmployeeHourlyRate m) {
  // Now I know that it's not some abstract Money,
  // but a very specific class EmployeeHourlyRate, which
  // was implemented by Bobby, a good friend of mine.
}

    在这里，我们限制了我们的代码可能出现的错误数量，因为我们了解Bobby，我们看过他的代码，我们知道它是如何工作的，以及有哪些异常情况。我们是安全的。是的，我们是的。就目前而言是这样。但从战略角度来看，通过不允许我们的软件在所有不寻常的情况下产生所有可能的错误和抛出所有可能的异常，我们严重限制了它进行适当测试的能力，这就是为什么它变得不稳定。

    正如我先前提到的，提高软件质量的唯一途径是发现并修复其中的错误。我们修复的错误越多，隐藏且尚未修复的错误就会越少。对错误的恐惧和我们预防错误的意图只会自食其果。

    相反，我们应该让每个人，不仅仅是Bobby，来实现Money并将这些实现传递给send()。是的，其中一些可能会引起麻烦，甚至可能导致用户界面可见的故障。但是，如果我们的管理层正确理解软件质量的概念，他们不会因为错误而责怪我们。相反，他们会鼓励我们尽可能地发现这些错误，用自动化测试进行复现，修复并重新部署。

    因此，对于脱钩的恐惧无非就是故障保护。

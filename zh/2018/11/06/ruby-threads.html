    Do You Test Ruby Code for Thread Safety?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    你是一个Ruby开发者吗？如果是的话，我相信你对并发性和线程安全性有一个非常模糊的概念。无意冒犯，但这是我在过去半年处理Ruby代码和与Ruby程序员交流后得出的结论。最近我一直在积极地使用Ruby进行编程，并且我喜欢这门语言以及其周围的生态系统。我们正在创建的实验性加密货币Zold几乎完全是用Ruby编写的。这告诉你什么？我喜欢Ruby。但是当涉及到并发性时，还存在一些空白的地方。非常关键。

    看看这个Ruby类：

    require 'sinatra'
class Front < Sinatra::Base
  configure do
    IO.write('idx.txt', '0')
  end
  get '/' do
    idx = IO.read('idx.txt').to_i + 1
    IO.write('idx.txt', idx.to_s)
    idx.to_s
  end
end
Front.run!

    这是一个简单的网络服务器。它可以工作——尝试像这样运行它（您需要安装Ruby 2.3+）。

    $ gem install sinatra
$ ruby server.rb

    然后，打开http://localhost:4567，你将会看到计数器。刷新页面，计数器会递增。再试一次，它能正常工作。计数器位于idx.txt文件中，它本质上是一个全局变量，在每个HTTP请求中我们都会递增它。

    让我们为其创建一个单元测试，以确保它被自动测试到。

    require 'minitest/autorun'
require 'net/http'
require 'uri'
class FrontTest < Minitest::Test
  def test_works
    front = Thread.start do
      Front.run!
    end
    sleep 1
    count = Net::HTTP.get(URI('http://localhost:4567/')).to_i
    assert_equal(1, count)
    Front.stop!
  end
end

    好的，这不是一个单元测试，更像是一个集成测试。首先，我们在一个后台线程中启动一个Web服务器。然后，我们等待一秒钟，以给该线程足够的时间来启动服务器。我知道，这是一个非常丑陋的方法，但对于这个小例子来说，我没有更好的办法。接下来，我们发送一个HTTP请求并将其与预期的数字1进行比较。最后，我们停止Web服务器。

    到目前为止一切都很好。现在，问题是，当许多请求发送到服务器时会发生什么？它是否仍会返回正确的连续数字？让我们试试看：

    def test_works
  front = Thread.start do
    Front.run!
  end
  sleep 1
  numbers = []
  1000.times do
    numbers << Net::HTTP.get(URI('http://localhost:4567/')).to_i
  end
  assert_equal(1000, numbers.uniq.count)
  Front.stop!
end

    在这里，我们发送一千个请求并将所有返回的数字放入一个数组中。然后，我们对数组进行uniq操作并count其元素。如果有一千个元素，说明一切正常，我们收到了一个正确的连续且唯一数字的列表。我刚刚测试过，它可以正常工作。

    但是我们是一个一个地发送请求，这就是为什么我们的服务器没有任何问题。我们不是并发地发送请求，而是严格按顺序一个接一个地发送。让我们尝试使用一些额外的线程来模拟并行执行HTTP请求。

    require 'concurrent/set'
def test_works
  front = Thread.start do
    Front.run!
  end
  sleep 1
  numbers = Concurrent::Set.new
  threads = []
  5.times do
    threads << Thread.start do
      200.times do
        numbers << Net::HTTP.get(URI('http://localhost:4567/')).to_i
      end
    end
  end
  threads.each { |t| t.join }
  assert_equal(1000, numbers.to_a.count)
  Front.stop!
end

    首先，我们将数字列表保存在 Concurrent::Set 中，它是 Ruby Set 的线程安全版本。其次，我们启动五个后台线程，每个线程进行200个HTTP请求。它们并行运行，并通过在每个线程上调用 join 等待它们完成。最后，我们从 Set 中取出数字，并验证列表的正确性。

    毫无疑问，它失败了。

    当然，你知道原因。因为实现是不线程安全的。当一个线程正在读取文件时，另一个线程正在写入文件。最终，它们会很快冲突，文件的内容会被破坏。我们在测试中放入的线程越多，结果就越不准确。

    为了使这种类型的测试更容易，我创建了一个简单的 Ruby gem，叫做 threads。下面是它的工作原理：

    require 'threads'
def test_works
  front = Thread.start do
    Front.run!
  end
  sleep 1
  numbers = Concurrent::Set.new
  Threads.new(5).assert(1000) do
    numbers << Net::HTTP.get(URI('http://localhost:4567/')).to_i
  end
  assert_equal(1000, numbers.to_a.count)
  Front.stop!
end

    就是这样。这一行代码 Threads.new() 代替了所有其他创建线程的代码行，确保它们同时启动，然后收集它们的结果，并确保它们的堆栈跟踪在控制台中可见，以便在发生崩溃时（默认情况下，后台线程的错误日志是不可见的）。

    在你的项目中尝试这个宝石吧，它已经经过了很好的测试，我在所有并发测试中都使用它。

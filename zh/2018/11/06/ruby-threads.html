    Do You Test Ruby Code for Thread Safety?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    你是一名Ruby开发者吗？如果是的话，我很确定你对并发性和线程安全性有一个非常模糊的概念。不冒犯，但这是我在过去半年与Ruby代码打交道和与Ruby程序员交流后得出的结论。最近我一直在积极地使用Ruby进行编程，我确实喜欢这门语言以及与之相关的生态系统。我们正在创建的实验性加密货币Zold几乎完全使用Ruby编写。这告诉你什么？我喜欢Ruby。但是当涉及到并发性时，存在一些空白点。非常显著。

    看看这个Ruby类：

    require 'sinatra'
class Front < Sinatra::Base
  configure do
    IO.write('idx.txt', '0')
  end
  get '/' do
    idx = IO.read('idx.txt').to_i + 1
    IO.write('idx.txt', idx.to_s)
    idx.to_s
  end
end
Front.run!

    这是一个简单的网络服务器。它能正常工作—尝试像这样运行它（您需要安装Ruby 2.3+）：

    $ gem install sinatra
$ ruby server.rb

    然后，打开 http://localhost:4567，你将会看到计数器。刷新页面，计数器会递增。再试一次。它有效。计数器在文件 idx.txt 中，它本质上是一个全局变量，我们在每个HTTP请求中递增它。

    让我们为此创建一个单元测试，以确保它能够自动进行测试。

    require 'minitest/autorun'
require 'net/http'
require 'uri'
class FrontTest < Minitest::Test
  def test_works
    front = Thread.start do
      Front.run!
    end
    sleep 1
    count = Net::HTTP.get(URI('http://localhost:4567/')).to_i
    assert_equal(1, count)
    Front.stop!
  end
end

    好的，这不是一个单元测试，更像是一个集成测试。首先，我们在后台线程中启动一个Web服务器。然后，我们等待一秒钟，给该线程足够的时间来引导服务器。我知道，这是一个非常丑陋的方法，但对于这个小例子，我没有更好的办法。接下来，我们发送一个HTTP请求并将其与期望的数字1进行比较。最后，我们停止Web服务器。

    到目前为止，一切都很好。现在的问题是，当许多请求发送到服务器时会发生什么？它是否仍然会返回正确的连续数字？让我们试一试：

    def test_works
  front = Thread.start do
    Front.run!
  end
  sleep 1
  numbers = []
  1000.times do
    numbers << Net::HTTP.get(URI('http://localhost:4567/')).to_i
  end
  assert_equal(1000, numbers.uniq.count)
  Front.stop!
end

    在这里，我们发起了一千个请求，并将所有返回的数字放入一个数组中。然后我们对数组进行uniq操作，并计算其中元素的数量。如果有一千个元素，说明一切正常，我们收到了一个正确的连续唯一数字列表。我刚刚测试过，它可以正常工作。

    但我们逐个制作它们，这就是为什么我们的服务器没有任何问题。我们不是同时制作它们。它们严格按顺序进行。让我们尝试使用一些额外的线程来模拟HTTP请求的并行执行。

    require 'concurrent/set'
def test_works
  front = Thread.start do
    Front.run!
  end
  sleep 1
  numbers = Concurrent::Set.new
  threads = []
  5.times do
    threads << Thread.start do
      200.times do
        numbers << Net::HTTP.get(URI('http://localhost:4567/')).to_i
      end
    end
  end
  threads.each { |t| t.join }
  assert_equal(1000, numbers.to_a.count)
  Front.stop!
end

    首先，我们将数字列表保存在 Concurrent::Set 中，它是 Ruby Set 的线程安全版本。其次，我们启动五个后台线程，每个线程都会发起 200 次 HTTP 请求。它们都并行运行，我们通过对每个线程调用 join 来等待它们完成。最后，我们从 Set 中取出数字并验证列表的正确性。

    毫不意外，它失败了。

    当然，你知道原因。因为这个实现不是“线程安全”的。当一个线程在读取文件时，另一个线程正在写入。最终，很快它们会发生冲突，导致文件内容损坏。我们在测试中使用的线程越多，结果就会越不准确。

    为了使这种类型的测试更加容易，我创建了一个简单的Ruby gem，叫做「threads」。它的工作原理如下：

    require 'threads'
def test_works
  front = Thread.start do
    Front.run!
  end
  sleep 1
  numbers = Concurrent::Set.new
  Threads.new(5).assert(1000) do
    numbers << Net::HTTP.get(URI('http://localhost:4567/')).to_i
  end
  assert_equal(1000, numbers.to_a.count)
  Front.stop!
end

    就是这样了。这一行代码 Threads.new() 取代了其它所有的代码行，我们不再需要创建线程、确保它们同时开始运行，然后收集它们的结果，还要确保在控制台中能够看到它们的堆栈跟踪，以防它们崩溃（默认情况下，后台线程的错误日志是不可见的）。

    尝试在你的项目中使用这个宝石，它已经经过了相当完善的测试，而且我在所有并发测试中都使用它。

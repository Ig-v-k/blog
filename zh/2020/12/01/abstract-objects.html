    Abstract Objects

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在面向对象的语言中，你如何创建对象？让我们以一些经典的语言为例，比如C++、Java或C#。首先你要定义一个类，然后创建一个实例。第一步被称为抽象，而第二步被称为实例化。在函数式编程中也存在一对类似的操作：声明一个函数是抽象，而使用具体参数调用它则是应用。问题是：为什么面向对象编程需要类和对象，而函数式编程仅通过函数就能实现？

    这是一个在EO中的抽象对象。

    [id db] > book
  db.query > title
    "SELECT title FROM book WHERE id=?"
    id

    对象的名称是book。它有三个属性：id、db和title。前两个属性是“free”：它们尚未绑定到任何对象。第三个属性title已经绑定到db.query对象的副本。book对象是抽象的，因为它有一些自由属性——它不足以指代现实世界中的特定实体。

    构造函数 db.query 从绑定到属性 db 的对象中获取一个抽象对象 query 并对其进行复制，同时指定两个参数：SQL 字符串和 id。复制一个抽象对象的过程被称为“应用”，因为它与函数式编程中将参数应用于函数非常相似。

    使用抽象的book的正确方式是复制它，并指定参数。

    book 42 mysql > b

    在这里，42是id，mysql是db，而b是一个新对象—book的副本。在𝜑-演算中，这将被写成以下公式：

    b ↤ book(42, mysql)

    为了获取 b 的 title 并将其命名为 t，我们只需要执行：

    b.title > t

    我们也可以将所有内容放在一行中：

    (book 42 mysql).title > t

    可以对抽象对象应用一些参数，同时保留其他属性的自由。例如，这将创建另一个抽象对象，仍然需要指定id以成为一个闭合对象。

    book mysql:db > x

    后缀:db帮助编译器理解应将哪个自由属性绑定到对象mysql。随后，x可以再次复制，创建一个闭合对象b。

    x 42 > b

    尽管EO的语法允许使用book.title这样的表达方式，但这将导致编译时错误：禁止访问抽象对象的自由属性或依赖于其他自由属性的属性。然而，假设我们给book添加一个新属性print，为了将其id打印到控制台，我们将执行以下操作：

    [id db] > book
  stdout > print
    sprintf
      "The book ID is %d"
      id
  db.query > title
    "SELECT title FROM book WHERE id=?"
    id

    print属性不需要绑定db属性，因为它不使用它。可以仅使用id复制book，然后调用print（假设这是在EO交互式控制台中进行的）。

    $ book 42:id > x
x
$ x.print
The book ID is 42

    顺便问一下，你知道还有其他以抽象对象为特点的语言吗？还是EO是第一个引入这种设计的？

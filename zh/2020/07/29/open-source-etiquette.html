    Open Source Etiquette

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    这是一份关于开源软件开发的常规礼仪规则简要清单。实际上，这些规则也适用于其他地方，但在进行基于GitHub的编码时最为明显。我坚信，不管是现在还是将来，所有编程都将是开源的，并且这些规则将适用于每个人。因此，现在就开始遵循它们是很有意义的，无论你是活跃的Apache贡献者还是《Java入门指南》的快乐拥有者。

    没有特定的顺序：

    进行小的拉取请求。谷歌和苏黎世大学的Caitlin Sadowski等人最近进行的一项研究表明，变更的大小与审查质量之间存在着很强的相关性：较大的变更（拉取请求）对质量产生负面影响。根据这篇文章，谷歌开发人员被强烈鼓励进行小而递增的变更。除了Google，还有许多其他人明确表示了相同观点：Microsoft，Zalando，Atlassian，和OpenSource.com。

    不要合并更改。将所有想要进行的更改放入一个拉取请求中，以便更快地合并它们，并只进行一次审查，这可能很诱人。但请不要这样做。这样只会使整个过程更加复杂，而且会让审查者感到恼火。原则很简单：一次更改=一个拉取请求。

    在您的文档中使用美观的Markdown。我没有找到任何关于这个的科学研究，可能是因为这是显而易见的：文本”why is f nil?”比”why is f nil?”更容易阅读。富文本格式不仅使文本看起来更漂亮，而且还帮助读者更快地消化内容并获得更多的乐趣。在学习Markdown之后，我建议阅读PetaBridge的Aaron Stannard的博客文章：如何专业地使用Github。

    回应评论。无论何时，都请以对方的GitHub昵称作为开头，撰写您的评论（无论是在拉取请求还是问题中）。如果您不这样做，消息将无法到达对方的收件箱，并很可能会丢失。在GitHub上活跃的人每天会收到数百封来自GitHub的电子邮件：他们不会阅读它们。他们只会阅读到达他们的“通知”中的内容。

    说“请”，“谢谢”和“对不起”。根据教皇方济各的说法，成功的秘诀就在于说出这三个简单的词语。他并不是指开源开发者，但这个建议完全适用于我们程序员。有很多关于在线礼仪的文章，它们基本上都是相同的：礼貌地提问，心怀感激，准备承认错误。我推荐GitHub高级产品经理Ben Balter的GitHub沟通规则。

    Ping他们。当你对分支进行一些更改并希望再次审查你的拉取请求时：发布一条明确要求审查者再次查看的消息。当你提交拉取请求时，发布一条消息，将其发送给项目的架构师：要求他们审查该PR。等等。不要期望他们会关注你的活动。他们还有其他事情要做。你有责任ping他们。

    制作描述性的提交。Git提交消息的格式通常在每个项目中都非常具体（我确定您正在使用Git）。然而，有一些相似之处和共同的规则。我建议阅读以下博文：Chris Beams的如何编写Git提交消息，Jeremy Gunter的关于提交礼仪的一些技巧，以及Tim Pope的关于Git提交消息的注意事项。另外，还可以参考这些网站：conventionalcommits.org 和 50/72格式化。

    拥有头像。康涅狄格大学的Kristine L. Nowak等人进行的一项研究表明，拥有头像的用户，尤其是拥有女性化和拟人化头像的用户，比那些没有个人资料图片（或使用GitHub提供的默认图片）的用户更容易引起注意。当然，仅仅拥有头像还不足以引起关注；您的GitHub个人资料还必须有其他许多东西：描述、电子邮件、固定存储库等。使用此个人资料作为示例：@m0nica。

    保持在线。离线沟通比在线票务要便宜得多：只需在办公室里大声提出问题，您就能立即得到答案。无需编写那些冗长的票务，用英语阐述问题，等待被指派者发布答案等等。然而，离线交流会对项目造成伤害，原因很多。每当您从GitHub问题转移到Slack聊天以讨论问题时，您都在对项目及其所有参与者不利。请记住这一点。

    好好报告。就像Git的提交一样，不同项目的错误报告规则也有所不同，但基本原则保持不变。只要谷歌一下”如何编写错误报告“，听听那些博主们说什么。你的错误报告比你的代码更能展示你是谁。你可以通过向Stack Overflow提交问题来练习：那里的社区会惩罚你的所有错误，快速训练你的报告技巧。

    编写优雅的 README。我之前写过这个问题：在开源项目中，完美编写和格式化的 README 文件的重要性难以言喻。编写良好的代码很重要，但是展示它也是在线礼仪的一部分：高质量的文档意味着对您产品的用户的尊重。

    这个看起来就是这样。如果你做到了这些，你将向其他开发者展现出尊重，他们也会回报你以尊重。我有没有漏掉重要的东西吗？

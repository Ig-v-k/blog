    Open Source Etiquette

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    这里是开源软件开发常见礼仪规则的简短列表。实际上，这些规则也适用于其他地方，但在进行GitHub编码时最为明显。我坚信，迟早所有的编程都将成为开源，并且这些规则将适用于每个人。因此，现在开始遵循这些规则是有意义的，无论你是积极的Apache贡献者还是《Java入门》一书的快乐拥有者。

    没有特定顺序：

    制作小的拉取请求。来自Google和苏黎世大学的Caitlin Sadowski等人最近的研究表明，变更的大小与代码审查质量之间存在着强烈的相关性：较大的变更（拉取请求）会对质量产生负面影响。根据这篇文章，Google鼓励开发者进行小的、渐进式的变更。除了Google之外，还有许多其他人明确表示了相同的观点：Microsoft、Zalando、Atlassian和OpenSource.com。

    不要合并更改。将你想要进行的所有更改放入一个单独的拉取请求中，以便更快地合并它们并只需进行一次审查可能很诱人。但请不要这样做。这只会使流程变得更加复杂，并且会让审查者感到恼火。原则很简单：一个更改 = 一个拉取请求。

    在您的文档中使用优雅的Markdown。我没有找到任何科学研究，也许是因为这是显而易见的：文本“为什么f是nil？”比“为什么f nil？”更容易阅读。富文本格式不仅使文本看起来更漂亮，而且还帮助读者更快速、更愉悦地理解内容。在学习Markdown之后，我建议阅读PetaBridge的Aaron Stannard的博客文章：如何专业地使用Github。

    回应评论。始终如一地，无论是在拉取请求还是问题上，都要以对方的 GitHub 昵称开头回应评论。如果不这样做，信息将无法到达对方的收件箱，很可能会被忽略。在 GitHub 上活跃的人每天都会收到来自 GitHub 的数百封电子邮件：他们不会阅读它们。他们只会阅读出现在他们的“通知”中的内容。

    说“请”，“谢谢”和“对不起”。根据教皇方济各的说法，成功的秘诀就在于说出这三个简单的词语。他并不是指开源开发人员，但这个建议完全适用于我们程序员。关于在线礼仪有很多文章，它们基本上都是一样的：礼貌地提问，心怀感激，并愿意承认错误。我推荐由GitHub的高级产品经理Ben Balter所写的GitHub沟通规则。

    Ping Them. 当你对分支进行了一些更改并希望再次审核你的拉取请求时，请明确地向你的审核者发送一条消息，要求他们再次查看。当你提交一个拉取请求时，向其中发布一条消息，将其发送给项目的架构师：请求他们审查这个PR。等等。不要期望他们观察你的活动。他们还有其他事情要做。你有责任提醒他们。

    进行详细的提交。Git提交消息的格式（我相信您正在使用Git）通常在每个项目中都非常具体。然而，也有一些相似之处和共同的规则。我建议阅读以下博文：Chris Beams的如何编写Git提交消息，Jeremy Gunter的关于提交规范的一些建议，以及Tim Pope的关于Git提交消息的注意事项。此外，还请参考以下内容：conventionalcommits.org和50/72格式。

    拥有 头像。康涅狄格大学的Kristine L. Nowak等人的一项研究表明，拥有头像的用户，尤其是女性和拟人化的头像，比那些没有个人照片（或使用GitHub提供的默认照片）的用户更容易引起注意。当然，仅有头像是不够的；你的GitHub个人资料还必须包含其他许多内容：描述、电子邮件、固定仓库等。以该个人资料作为例子：@m0nica。

    保持在线。线下沟通比在线票务便宜得多：只需在办公室大声提问，立刻就能得到答案。无需书写那些冗长的票务，用英语表达问题，等待指派人回答等等。然而，线下交谈对项目造成了很多伤害，原因很多。每次从GitHub问题转到Slack聊天讨论问题，都对项目及其所有参与者不利。请记住这一点。

    友好报告。就像Git提交一样，不同项目的错误报告规则不同，但基本原则保持不变。只需在Google上搜索“如何编写错误报告”并听取那些博主的建议。你的错误报告能更好地展示你的个性，甚至比你的代码还要好。你可以通过向Stack Overflow提交问题来练习：那里的社区会对你的所有错误进行惩罚，快速培养你的报告技巧。

    制作优雅的README文件。我之前写过这个：在开源项目中，一个完美编写和格式化的README文件的重要性无法过分强调。编写优秀的代码很重要，但是展示它也是在线礼仪的一部分：高质量的文档意味着对您产品的用户的尊重。

    似乎就是这样了。如果你做到了这些，你将向其他开发者表达了尊重，并且他们也会回报你同样的尊重。我有没有忘记什么重要的东西？

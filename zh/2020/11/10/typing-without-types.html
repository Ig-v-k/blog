    Strong Typing without Types

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    1974年，Liskov和Zilles 定义一种强类型语言为”当一个对象从调用函数传递到被调用函数时，其类型必须与被调用函数中声明的类型兼容。”确实，强类型检查无疑会减少类型错误的数量，从而提高质量。然而，问题是：我们真的需要类型来强制执行类型吗？

    例如，这是一个我们期望接收到一个 Java interface Book 实例的地方：

    void print(Book b) {
  System.out.printf(
    "The ISBN is: %s%n", b.isbn()
  );
}

    类型Book可能是这样的：

    interface Book {
  String isbn();
}

    如果将一个不“实现”接口Book的对象传递给方法print()，编译器将会报告“类型不匹配”错误。对于程序员来说，很难犯一个错误并将Car类型的对象传递给方法print()。然而，通过动态类型转换仍然是可能的。

    Car car = new Car("Mercedes-Benz G63");
print(Book.class.cast(car)); // Here!

    这段代码可以编译通过，但在运行时会抛出 ClassCastException，因为无法将 Car 强制转换为 Book。

    强类型的美妙之处在于它可以防止错误。然而，它增加了代码的复杂性：你需要首先创建类型，你需要在所有函数中声明它们，你需要进行难以调试的类型转换等等。弱类型的支持者对此进行了大量抱怨，并创造出像Ruby这样完全没有类型的语言。

    def print(b)
  puts(format("This is ISBN: %s", b.isbn))
end

    在这里，函数print()不期望b具有特定的类型。无论输入什么类型都可以。稍后，在调用.isbn时，运行时会检查传入的b是否具有该方法。如果有，一切都正常，如果没有，则会引发运行时错误NoMethodError。

    So far so good.

    然而，这里的想法是：如果我们将动态类型的简洁性和简洁性与强类型的安全性相结合，通过完全摒弃类型，让编译器从与对象一起使用的代码中推断类型信息，会怎么样？这是我们的代码：

    void print(Book b) {
  System.out.printf(
    "The ISBN is: %s%n", b.isbn()
  );
}

    想一想这个：在编译时，很明显b必须至少有一个isbn()方法。没有必要强迫程序员明确定义类型Book并在print()方法的签名中提到只有书籍才受欢迎：这些知识可以轻松地从print()方法的主体中推断出来！编译器可以查看print()方法中的所有语句，并清楚地理解将使用对象b来做什么。这些信息足以可视化传入对象的“类型”。没有必要要求程序员明确地执行此操作，并在新文件中花费另外五行代码来声明类型Book。编译器可以为我们完成这项工作。

    当然，要使这个工作起来，我们必须禁止任何类型的转换，这在Java、C++、C#和其他伪面向对象的语言中是不可能的。但是在EO中是可能的！

    WDYT?

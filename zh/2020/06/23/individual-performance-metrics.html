    To Measure or Not to Measure

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    问题是在StackExchange上提出的，九年前（就在该网站刚刚推出的时候）：”如果不是代码行数，那么通过什么指标来衡量远程程序员的有效性呢？” 答案并不令人惊讶，都是这样的观点：程序员不应该被衡量！我敢打赌那些回答问题的人自己就是程序员。确实，为什么一个程序员会对被衡量和仅仅被当作一个数字感兴趣呢？

    首先，让我们看看为什么给程序员设置绩效指标可能被认为是不好的做法（我个人认为：这只是过度高薪的程序员/经理为了保住自己的工作而找的借口，他们只是尽情地做他们想做的事情，浪费雇主的钱而已）。

    • “个人绩效指标违背了Scrum的精神，” Selçuk Özdoğan 说道。

    • “如果不加以控制，英雄主义思维会破坏团队精神。”来自Quovantis的产品经理Bhuwan Jain如此说道。

    • “绩效评估会破坏士气并破坏团队合作。” Samuel A. Culbert，加州大学洛杉矶分校安德森管理学院的管理学教授，如此说道。

    • “在合作是成功所必需的环境中，个人奖励促进了竞争。”，敏捷教练Avienaash Shiralige在AgileBuddha上说道。

    • “个人指标会阻碍团队协作，” Sean McHugh 在Axosoft担任Scrum Master的人表示。

    • “绩效评估是有害的且完全不必要的，” 来自Scrumology的联合创始人和首席顾问Kane Mar说道。

    • “绩效指标会抑制主动性、创新性和冒险精神，” 《度量的暴政》的作者 Jerry Z. Muller 表示。

    • “人类本质上不能被简单归结为一个数字，” 一位认证Scrum培训师 Jimi Fosdick 表示。

    你觉得怎么样？“继续付钱给我，但不要指望我给你任何回报！你也敢检查我的表现！”——我听到了这些，也不感到惊讶。所发生的事情被称为绩效管理革命，其要点是：现代管理如此薄弱，我们迫切需要一个官方名称来定义这种混乱，以避免困惑。敏捷就是这个名称。

    经理们不再负责，程序员们掌控着一切。这真是令人悲伤。

    然而，并非每个人都相信这种无政府主义。一些专家认为指标实际上是有帮助的。Bradley Kirkman等人声称“认可一个团队成员似乎对团队中的其他成员有积极且具有传染性的影响”，而2001年由Jack Welch提出的著名的Vitality Curve仍然存在。

    我相信对度量标准的所有负面评价都是由于其不正确的使用所引起的。确实，如果例如软件工程师的唯一绩效指标是他们在办公室的工作时间，那么这只会对所有度量标准产生不好的影响。这样的度量标准确实会造成伤害，毫无疑问。但是缺乏好的度量标准会造成更大的伤害。如何选择它们——这才是真正的问题。让我建议一些，而不是著名而不正确的代码行数（LoC）。理想情况下，您会从这个列表中选择一个组合，甚至全部使用它们：

    • 交付的功能。给一个工单（或 GitHub 问题）添加一个“功能”标签，然后计算程序员关闭带有此标签的工单的数量。当然，每个工单必须由产品经理关闭，而不是由程序员关闭。

    • 已合并的拉取请求。每个拉取请求必须经过同行评审，以确保其有意义并符合项目的质量标准。太小或太大的拉取请求必须被拒绝。

    • 修复的错误。错误与功能的工作方式相同，但通常较小。程序员关闭的错误越多，越好。当然，关闭错误的工作必须由产品经理或其他报告错误的程序员来完成。

    • 已报告的漏洞。一旦项目接受了报告的漏洞，报告者将获得额外的积分。这就是项目质量提升的方式：鼓励每个人主动报告漏洞。

    • 发布版本。在一个有纪律的项目中，每天都会发布新版本；而在其他项目中，可能是每周或每几个月（或者从不）发布一次。每次发布都是一项充满压力的任务，因此对程序员进行奖励似乎是合理的。

    • 正常运行时间。有一组DevOps指标可以展示生产环境中的服务质量，包括MTBF、MTTF、故障率等等。正常运行时间越长，程序员及其产品的质量就越好。

    • 拉取请求的成本。我相信，程序员提交的拉取请求合并所需的时间越短，程序员的能力就越好。他们的拉取请求越快通过同行评审和质量控制，就越好。初级程序员通常提交过大或过于复杂的拉取请求，在同行评审中导致很多麻烦。他们还会引起合并冲突，有时甚至会导致过期的分支和不适合合并的拉取请求。

    • 已发布的文档页面。FAQ页面、Javadoc块、Wiki页面、博客文章等等——它们通过增加可维护性来帮助项目更接近用户和未来的开发人员。当然，在发布之前，每一段文字都必须经过验证。

    • 受 mentee 影响的结果。资深程序员可能是初级程序员的导师，并可能因其 mentee 所获得的奖励而获得回报。上述所有指标都可以通过这种方式工作，当他们的学生表现更好或更差时，奖励或惩罚导师。

    我再怎么强调也不为过：每个度量指标都必须有一个质量控制机制。仅仅测量报告的错误而不检查它们的质量会导致作弊：程序员会报告他们喜欢的任何错误，只是为了增加数字。每个错误都必须由一位架构师进行验证：重复或质量低劣的错误报告将被拒绝。对于每个度量指标同样如此：没有控制的信任会导致作弊。

    值得一提的是，功能、错误、拉取请求和文档页面可能具有不同的“复杂性”、“紧急程度”和“严重程度”，这也应该考虑在内，增加或减少每个指标中的数字。

    大多数这些指标可以自动收集，无需任何人工干预，例如通过GitHub API。

    在理想的管理环境中，项目根据收集的指标来补偿程序员的工作。程序员不是通过工资获取报酬，而是通过功能、错误、文档页面等来获得报酬。你的项目离这个乌托邦有多远，决定了你作为项目经理的专业水平。糟糕的经理不会进行任何衡量，只是通过保持薪水高和控制低来让每个人都“开心”……直到项目用尽资金。另一方面，非常优秀的经理让指标控制每个人，让最优秀的人开心，而让最差劲的人……迅速找到出路。

    Which one are you?

    Is there a metric to measure programmers' performance objectively?

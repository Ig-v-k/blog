    New Metric: the Distance of Coupling

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    封装，正如您所知，是面向对象编程中的四个关键原则之一。根据Grady Booch等人的说法，封装是“隐藏对象中不贡献其基本特征的所有秘密的过程”。从实际角度来说，它涉及到我们在Java和C++中使用的那些private属性：它们对我们对象的用户是不可见的，因此它们无法被修改或读取。Booch等人认为封装的目的是“在不同的抽象之间提供明确的屏障”，从而实现“关注点的明确分离”。然而，它是否真的按计划工作呢？我们是否真的在对象之间有明确的屏障？让我们来看一下。

    首先，我不是第一个也不是唯一一个提出这个问题的人。David West 在很早之前就说过，“在大多数情况下，封装更多地是一种纪律而不是一个真正的障碍”，并且“很少有对象的完整性在任何绝对意义上得到保护”。在实践中，“尊重对象的封装是用户的责任。”确实，让我们来看一下我关于裸数据的博文中的 Temperature 类。

    class Temperature {
  private int t;
  public int getT() { return this.t; }
  public void setT(int t) { this.t = t; }
}

    我们可以说 t 属性是真正的封装吗？从技术上讲，是的：无法通过点符号直接修改它。简而言之，我们无法这样做：

    Temperature x = new Temperature();
x.t = 10;

    我们甚至无法做到这一点：

    int y = x.t;

    然而，我们可以通过getter getT()和setter setT()完全做同样的事情。因此，类Temperature的设计者给予了我们访问其属性的能力，但是间接地，通过getter和setter。我会说这里违反了封装的原则，我确信Allen Holub会同意我的观点。解决方案是什么呢？关于裸数据的文章提出了使用TellDontAsk原则，并且我们应该摒弃getter。

    class Temperature {
  private int t;
  public String toString() {
    return String.format("%d F", this.t);
  }
}

    现在，Temperature类不允许我们读取它的属性t。相反，我们只能告诉它准备温度的字符串表示，并将其返回给我们。也许这并不完全符合“告诉”范式的经典例子，因为有一些数据会返回，但是现在它看起来比以前好多了。这种重构的美妙之处在于客户端和对象之间的耦合更少。通过getter（或通过点表示法直接访问属性），客户端能够获取温度的数值，并在假设它是摄氏度的情况下重新计算成华氏度。而通过返回的字符串，客户端将不会这样做。字符串只会被用作最终产品，不可修改。或者也许不是这样？

    如果客户这样做：

    Temperature x = new Temperature();
String txt = x.toString();
String[] parts = txt.split(" ");
int t = Integer.parseInt(parts[0]);

    现在看起来怎么样？这不是一种违反封装的行为吗？toString()的结果没有按照预期的方式处理。它不是作为一个固定的字符串，而是作为具有一些内部结构的数据，而这些结构已知于客户端。客户端对输出的了解是关键问题。客户端了解得太多，并且利用这些知识为自己的利益对数据进行解构和操作。

    我们真的可以禁止客户端这样做吗？据我所知，任何编程语言中都没有这样的功能。当方法的输出传递给客户端时，客户端被允许根据需要进行任何操作。如果我正确理解了韦斯特博士的意思，这就是对封装性的不尊重。而且我们甚至没有讨论过反射API，它可以让我们在不调用任何方法的情况下将“t”从“Temperature”中拿出来。

    因此，封装不是一种明确的屏障。只要我们有尊重封闭性的愿望，它就存在。如果我们不尊重它，没有什么能阻止我们任意滥用一个对象。即使是private属性修饰符也无济于事。此外，它们只会产生封装的幻觉，而事实上每个人都能根据自己的业务需求做任何他们认为合适的事情。

    我有一个提议，或许可以帮助我们更明确地实现封装。

    如果我们能够控制将数据和对象返回给客户端后发生的情况，会怎么样呢？如果我们可以阻止客户端对toString()方法的输出进行split()操作，会怎么样呢？我认为我们可以在编译时通过检查代码并确定与对象交互的时刻与它们被返回的位置之间的距离来实现这一点。在上面的示例中，距离为两个：1）首先，我们执行split操作，2）其次，我们执行parseInt()操作。当然，较大的应用程序会有更大的数字。

    似乎我们可以将这个距离数值作为整个应用程序中对象之间耦合的度量标准。数值越大（或者所有数值的平均值越大），设计就越糟糕：在良好的设计中，我们不应该从一个方法中取出某个东西然后进行一些复杂的处理。根据上面提到的TellDontAsk原则，我们应该让我们的对象完成工作，并且只返回一个快速的概要。这个距离度量标准将会告诉我们准确的信息：我们违反了松耦合原则的次数以及程度。

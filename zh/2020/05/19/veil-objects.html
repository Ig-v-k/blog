    Veil Objects to Replace DTOs

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    这是我在几天前在使用 Codexia（一个 Ruby Web 应用程序）时发现的一个新想法。我需要从 PostgreSQL 中提取数据行并将对象返回给客户端。对我来说，这一直是一个问题，如何在不将对象转换为 DTO 的情况下完成此操作。下面是我找到并命名的解决方案：Veil Objects。

    假设我从 PostgreSQL 中提取了项目列表：

    class Projects
  def fetch
    @pgsql.exec('SELECT * FROM project')
  end
end

    @pgsql上的exec()方法（我正在使用pgtk宝石）返回一个由Hashes组成的数组，如果我们将它们转换为JSON格式，它们看起来像这样：

    [
  {"id": 1, "name": "foo", "author": "yegor256"},
  {"id": 2, "name": "bar", "author": "yegor256"},
  {"id": 3, "name": "zoo", "author": "yegor256"}
]

    很好的话，让fetch()方法返回一个对象数组而不是哈希数组会很好。所以我的Project类看起来是这样的：

    class Project
  def initialize(pgsql, id)
    @pgsql = pgsql
    @id = id
  end
  def name
    @pgsql.exec(
      'SELECT name FROM project WHERE id=$1',
      [@id]
    )[0]['name']
  end
  def author
    @pgsql.exec(
      'SELECT author FROM project WHERE id=$1',
      [@id]
    )[0]['author']
  end
end

    这是为单个项目操作而设计的完美工具：

    p = Project.new(pgsql, 123)
name = p.name
author = p.author

    这里有两个SQL请求并不是什么大问题。然而，如果我将散列列表转换为像这样的项目，我将面临严重的性能问题。

    class Projects
  def fetch
    @pgsql.exec('SELECT * FROM project').map do |r|
      Project.new(@pgsql, r['id'].to_i)
    end
  end
end

    这就是会让我性能受损的地方：

    projects.fetch do |p|
  puts "#{p.name} is created by #{p.author}"
end

    这段代码会生成太多冗余的SQL请求。我们将来回到PostgreSQL获取我们几毫秒前的数据，而当时我们正在执行SELECT * FROM project。

    最简单和最明显的解决方案是将检索到的哈希封装到Project对象中。换句话说，将Project转换为一个数据传输对象（DTO），一个数据的持有者。嗯，在这种情况下，我们甚至可能不需要一个对象，而是可以返回带有数据的哈希。但这不是我们想要设计的面向对象软件的方式。我们想要处理对象，而不是数据结构。同时，我们也不希望对象愚蠢到去数据库获取我们一秒钟前的相同数据。这里是我提出的解决方案：

    require 'veils'
class Projects
  def fetch
    @pgsql.exec('SELECT * FROM project').map do |r|
      Veil.new(
        Project.new(@pgsql, r['id'].to_i),
        name: r['name'],
        author: r['author']
      )
    end
  end
end

    这个来自 veils gem 的新 Veil 对象是 Project 的一个装饰器。它的行为类似于一个 Project，但它的一些方法被重新定义了：name() 和 author()。当它们被调用时，调用不会传达到封装的 Project，而是返回存储在 Veil 中的数据。

    它被称为“veil”是因为它像面纱一样：预设的数据仅在调用其他未预设的方法之前返回。如果发生这种情况，面纱被揭开，Veil 对象变得完全透明，通过发送所有方法调用。

    因此，DTO 的效率与 OOP 的优雅相结合。

    我在 yegor256/codexia 中使用这些新的 veil 对象，所以你可以看到它们是如何工作的。

    附言：我还创建了一个 Unpiercable 类，它的行为与 Veil 完全相同，但永远不会被揭开。当你不希望发生任何数据修改交互并且只想要一些方法被预先计算时，它非常有用。

    再附言：这是 Kotlin 的实现。

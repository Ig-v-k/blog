    Fat vs. Skinny Design

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在面向对象编程中，类型/类层次结构似乎可以以两种极端的方式设计：一种是考虑到完全封装数据；另一种是只有少数接口使原始数据可见，并让类来处理、解析并将其转化为较小的数据元素。你可能会感到惊讶，但我认为第二种选择更加优雅。在我看来，我们并没有失去面向对象的特性，相反，我们获得了很多灵活性、可重用性、可测试性等等。

    请看这个（我们称之为fat，稍后我会解释为什么）：

    interface Article {
  Head head();
}
interface Head {
  Author author();
  String title();
}
interface Author {
  String name();
  String email();
}

    为了获得作者的姓名，我们执行以下操作：

    // It is stored in PostgreSQL (that's why the Pg
// prefix) and retrieves everything using SQL
Article a = new PgArticle();
String name = a.head().author().name();

    从视觉上看，这个设计可能看起来像这样（在UML中）：

    允许混合接口文章接口头接口作者文章 o-> 头：”head()”头 o-> 作者：”author()”作者：字符串名称() 作者：字符串邮箱() 文章 <– PgArticle 头 <– PgHead 作者 <– PgAuthor PgArticle o-> PgHead PgHead o-> PgAuthor 组件 PostgreSQL PgArticle ..> PostgreSQL PgHead ..> PostgreSQL PgAuthor ..> PostgreSQL

    现在，让我们将其与另一种设计进行比较（这个设计比之前的设计要瘦得多，我甚至可以称之为苗条）。

    interface Article {
  String head();
}
class TxtHead {
  private final Article article;
  String author();
  String title();
}
class TxtAuthor {
  private final Head head;
  String name();
  String email();
}

    在这里，为了获取作者的姓名，我们必须提取头部作为一个String，提取作者作为一个String，然后提取姓名作为一个String：

    Article a = new PgArticle();
String head = a.head();
String author = new TxtHead(head).author();
String name = new TxtAuthor(author).name();

    在UML中，它的可视化效果如下：

    允许混合接口文章 TxtAuthor *- TxtHead TxtHead *- Article Article : String head() TxtHead : String author() TxtAuthor : String name() TxtAuthor : String email() Article <– SqlArticle 组件 PostgreSQL SqlArticle ..> PostgreSQL

    在第一版设计中有三个接口，而第二版只有一个接口和两个类。我称第一版为“胖”是因为它返回接口，这些接口已经实现了我们所需要的功能，我们不需要再用额外的装饰器或适配器来覆盖它们。它的三个接口层次结构已经足够丰富，可以提供我们所需的一切。这就是为什么它是“胖”的原因。另一方面，第二版则相对“瘦”，只有一个接口，返回的是纯文本数据，我们需要自己解析它。我们需要给它“打扮”起来。

    看起来苗条的设计更好，原因有很多：

    • 可扩展性。苗条的设计肯定更容易扩展。为了从作者那里提取一些新信息，我们只需向TxtAuthor类添加一个新方法即可。我们不需要重新设计整个接口层级结构以及修改它们的所有实现。我们处理的是纯数据，稍后在装饰器、适配器和其他辅助智能类中进行管理和解析。

    • 内聚性。瘦设计明显更具内聚性，因为与管理PostgreSQL数据相关的所有内容都集中在一个类SqlArticle中。相反，胖设计将功能分散在许多类中，正因如此，使得整个类集更难以维护。

    • 可重用性。TxtAuthor 类绝对可以在任何需要解析作者信息的地方使用，而 PgAuthor 类只适用于一种特定情况：获取和解析与 PostgreSQL 相关的数据。

    • 可测试性。显然，瘦设计更容易进行测试，因为模拟单个接口比模拟整个层级结构要简单得多。为了测试TxtAuthor类，我们只需将一些虚假文本传递给它的构造函数并检查其运行情况。而要测试PgAuthor类，我们需要做更多的工作，包括运行一个虚假的PostgreSQL服务器实例。

    以上所有内容都适用于1) 从PostgreSQL数据检索和2) 向PostgreSQL数据操作。当然，操作可能需要许多方法存在于SqlArticle中，这将使精简设计看起来不太美观，并且显而易见的是，其中一些方法必须移动到较低级别的类/接口中。这仅仅证明了并不总是可能通过单一接口来实现精简设计，就像上面的例子一样。有时我们只是不得不让它更加臃肿。

    然而，与纤细设计相关的一个严重问题是，它使得原始裸露数据从 SqlArticle 中跳出，这与我们所知的面向对象编程的核心思想背道而驰。的确，如果我们让 TxtHead 进行解析，我们可能会失去一些仅在 SqlArticle 中可用的与 PostgreSQL 相关的有趣上下文。我们不希望复杂的数据解析发生在离数据生成地很远的地方。我们希望所有与数据相关的事情都发生在数据所在的地方：SqlArticle 内部。

    这是一个合理的担忧，但是让与PostgreSQL相关的信息（如连接设置）从PgArticle传递到PgHead，再传递到PgAuthor，这会更加违反数据封装原则。

    在现实生活中，当然无法想象纯粹的单一界面瘦身设计。它们都会在一定程度上臃肿。然而，我的建议是尽量使设计变得更加精简，让界面用户可以根据自己的喜好进行装饰。这个建议与我之前提到的智能类别非常接近，但这次的原则更加广泛。

    Fat vs. Skinny Design

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    似乎面向对象编程(OOP)中的类型/类层次结构可以以两种极端的方式设计：一种是完全封装数据；另一种是只提供少量接口，将原始数据可见，并让类处理、解析并将其转化为较小的数据元素。也许你会感到惊讶，但我认为第二种选择更加优雅。在我看来，我们并没有失去面向对象的特性，反而获得了更多的灵活性、可重用性、可测试性等等。

    来看看这个（我们称之为“fat”，稍后我会解释为什么）：

    interface Article {
  Head head();
}
interface Head {
  Author author();
  String title();
}
interface Author {
  String name();
  String email();
}

    要获取作者的姓名，我们需要执行以下操作：

    // It is stored in PostgreSQL (that's why the Pg
// prefix) and retrieves everything using SQL
Article a = new PgArticle();
String name = a.head().author().name();

    从视觉上看，这个设计可能是这样的（在UML中）：

    允许混合 接口 文章 接口 头 接口 作者 文章 o-> 头：“head()” 头 o-> 作者：“author()” 作者：字符串 名字() 作者：字符串 电子邮件() 文章<– PgArticle 头<– PgHead 作者<– PgAuthor PgArticle o-> PgHead PgHead o-> PgAuthor 组件 PostgreSQL PgArticle ..> PostgreSQL PgHead ..> PostgreSQL PgAuthor ..> PostgreSQL

    现在，让我们将其与另一种设计进行比较（这种设计比之前的设计要“瘦”得多，我甚至会称其为“纤细”）。

    interface Article {
  String head();
}
class TxtHead {
  private final Article article;
  String author();
  String title();
}
class TxtAuthor {
  private final Head head;
  String name();
  String email();
}

    在这里，为了获取作者的姓名，我们需要提取头部作为String，提取作者作为String，然后提取姓名作为String。

    Article a = new PgArticle();
String head = a.head();
String author = new TxtHead(head).author();
String name = new TxtAuthor(author).name();

    从视觉上看，UML的表示如下：

    allowmixing 接口 Article TxtAuthor *- TxtHead TxtHead *- Article Article : String head() TxtHead : String author() TxtAuthor : String name() TxtAuthor : String email() Article <– SqlArticle 组件 PostgreSQL SqlArticle ..> PostgreSQL

    第一个设计中有三个接口，而第二个设计只有一个接口和两个类。我将第一个设计称为“胖”设计，因为它返回的接口已经实现了我们所需要的功能，我们不需要再使用额外的装饰器或适配器来覆盖它们。它的三个接口层次结构足够丰富，可以提供我们所需的一切。这就是为什么它是“胖”的原因。另一方面，第二个设计相对来说很“瘦”，只有一个接口，它返回了我们需要自己解析的纯文本数据。我们需要对它进行“打扮”。

    从很多方面来看，瘦设计似乎更好：

    • 可扩展性。瘦身设计肯定更容易扩展。为了从作者那里提取一些新信息，我们只需要在TxtAuthor类中添加一个新的方法即可。我们不需要重新设计整个接口层级并修改所有实现。我们处理的是纯数据，稍后在装饰器、适配器和其他辅助智能类中进行管理和解析。

    • 内聚性。瘦设计明显更具内聚性，因为所有与 PostgreSQL 数据管理有关的内容都集中在一个类 SqlArticle 中。相反地，胖设计将功能分散在多个类中，因此使整个类集更难以维护。

    • 可重用性。TxtAuthor 类绝对可以在任何需要解析作者信息的地方使用，而 PgAuthor 类只适用于一个特定情况：获取和解析与 PostgreSQL 相关的数据。

    • 可测试性。显然，精简的设计更容易进行测试，因为模拟单个接口比模拟整个层次结构要简单得多。要测试TxtAuthor类，我们只需向其构造函数传递一些假文本并检查其工作情况。要测试PgAuthor类，我们需要做更多的工作，包括运行一个假的PostgreSQL服务器实例。

    以上所说的一切对于从PostgreSQL数据检索和对PostgreSQL数据操作都是适用的。当然，操作可能需要许多方法存在于“SqlArticle”中，这将使得精简设计看起来很丑陋，并且很明显其中一些方法必须移动到较低级别的类/接口中。这只是证明了并不总是可能通过单个接口实现精简设计，就像上面的例子一样。有时候我们必须让它更加臃肿。

    然而，精简设计存在一个严重的问题：它让裸露的原始数据从“SqlArticle”中脱离出来，这与我们所知的面向对象编程的理念相违背。确实，如果让“TxtHead”进行解析，我们可能会失去一些仅在“SqlArticle”内部才有的有趣的与PostgreSQL相关的上下文信息。我们不希望复杂的数据解析发生在数据产生的地方很远的地方。我们希望所有与数据相关的事情都发生在数据所在的地方：在“SqlArticle”内部。

    这是一个合理的担忧，但是让与PostgreSQL相关的信息（如连接设置）从“PgArticle”移动到“PgHead”，然后再移动到“PgAuthor”是对数据封装原则的更大违反。

    在现实生活中，当然不可能想象纯粹的单一接口的精简设计。它们都会在某种程度上变得臃肿。不过，我的建议是尽量让设计变得更加精简，让接口使用者可以根据自己的喜好进行装饰。这个建议与我之前关于智能类的说法非常接近，但这次原则更加广泛。

    Fat vs. Skinny Design

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在面向对象编程中，类型/类层次结构似乎可以以两种极端的方式设计：一种是完全封装数据；另一种是只提供少量接口，将原始数据暴露出来，让类来处理、解析并将其转化为更小的数据元素。也许你会感到惊讶，但我认为第二种选择更加优雅。在我看来，我们并没有丧失面向对象的特性，反而获得了很多灵活性、可重用性和可测试性等等。

    请看这个（我们称之为fat，稍后我会解释为什么）：

    interface Article {
  Head head();
}
interface Head {
  Author author();
  String title();
}
interface Author {
  String name();
  String email();
}

    为了获得作者的姓名，我们需要执行以下操作：

    // It is stored in PostgreSQL (that's why the Pg
// prefix) and retrieves everything using SQL
Article a = new PgArticle();
String name = a.head().author().name();

    从视觉上看，此设计可能是这样的（在UML中）：

    allowmixing 接口 Article 接口 Head 接口 Author Article o-> Head : “head()” Head o-> Author : “author()” Author : String name() Author : String email() Article <– PgArticle Head <– PgHead Author <– PgAuthor PgArticle o-> PgHead PgHead o-> PgAuthor 组件 PostgreSQL PgArticle ..> PostgreSQL PgHead ..> PostgreSQL PgAuthor ..> PostgreSQL

    现在，让我们将其与另一种设计进行比较（这种设计比之前的那种要精简得多，我甚至可以称之为苗条）。

    interface Article {
  String head();
}
class TxtHead {
  private final Article article;
  String author();
  String title();
}
class TxtAuthor {
  private final Head head;
  String name();
  String email();
}

    在这里，为了获取作者的姓名，我们需要提取头部作为String，提取作者作为String，然后提取名字作为String。

    Article a = new PgArticle();
String head = a.head();
String author = new TxtHead(head).author();
String name = new TxtAuthor(author).name();

    在UML中，它的可视化效果如下：

    允许混合接口Article TxtAuthor *- TxtHead TxtHead *- Article Article : String head() TxtHead : String author() TxtAuthor : String name() TxtAuthor : String email() Article <– SqlArticle 组件PostgreSQL SqlArticle ..> PostgreSQL

    第一种设计有三个接口，而第二种设计只有一个接口和两个类。我称第一种设计为“肥胖”，因为它返回的是已经实现了我们所需功能的接口，我们不需要再用其他装饰器或适配器来覆盖它们。它的三个接口层次结构足够丰富，可以给我们提供所需的一切。这就是它为什么“肥胖”。另一方面，第二种设计则相对“苗条”，只有一个接口，返回的是我们需要自行解析的纯文本数据。我们需要将其打扮得漂亮一些。

    看起来瘦身设计更好，有很多原因支持这一观点：

    可扩展性。瘦身的设计肯定更容易进行扩展。为了从作者那里提取一些新信息，我们只需要向TxtAuthor类添加一个新方法即可。我们不需要重新设计整个接口层级结构或修改所有的实现。我们处理的是纯数据，稍后在装饰器、适配器和其他辅助智能类中进行管理和解析。

    内聚性。瘦设计肯定更具内聚性，因为与管理 PostgreSQL 数据相关的所有内容都集中在一个类 SqlArticle 中。相反，胖设计将功能分散在许多类中，因此使整个类集更难维护。

    可重用性。TxtAuthor 类肯定可以在任何需要解析作者信息的地方使用，而 PgAuthor 类只适用于一个特定情况：获取和解析与 PostgreSQL 相关的数据。

    可测试性。显然，纤细的设计更容易进行测试，因为模拟单个接口比模拟整个层次结构要简单得多。要测试TxtAuthor类，我们只需向其构造函数传递一些虚假的文本，并检查其工作情况。要测试PgAuthor类，我们需要进行更多的工作，包括运行一个虚假的PostgreSQL服务器实例。

    以上所说的对于 1) 从 PostgreSQL 数据检索和 2) 到 PostgreSQL 数据操作都是正确的。当然，操作可能需要在 SqlArticle 中存在许多方法，这会使得瘦设计看起来很丑陋，并且很明显有些方法必须移动到较低级别的类/接口中。这只是证明并不总是可能使用单个接口来实现瘦设计，就像上面的例子一样。有时我们只是必须使它变得更厚。

    然而，与瘦设计相关的一个严重问题是：它使得原始的裸数据从SqlArticle中跳出，这违背了面向对象编程的核心理念，正如我们所知道的。确实，如果让TxtHead进行解析，我们可能会丢失一些仅在SqlArticle内部可用的有趣的与PostgreSQL相关的上下文。我们不希望复杂的数据解析发生在数据产生的地方之外。我们希望所有与数据相关的事情都发生在数据所在的地方：在SqlArticle内部。

    这是一个合理的担忧，但是让与PostgreSQL相关的信息（如连接设置）从PgArticle移动到PgHead，然后再移动到PgAuthor，更大程度上违反了数据封装原则。

    在现实生活中，当然不可能想象出纯粹的单接口瘦设计。它们都会在一定程度上是臃肿的。然而，我的建议是尽量让设计更加精简，让接口用户可以根据自己的喜好进行装饰。这个建议与我之前提到的“智能类别”非常相似，但这次的原则更为广泛。

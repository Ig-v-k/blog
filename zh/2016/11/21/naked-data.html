    Encapsulation Covers Up Naked Data

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    封装是面向对象编程的核心原则，使对象具有坚固、内聚、可信等特性。但是，封装到底是什么呢？它只是保护对象外部对私有属性的访问吗？我认为它远不止于此。封装导致所有层次和形式上都没有“裸露”的数据。

    这就是裸数据是什么（C代码）：

    int t;
t = 85;
printf("The temperature is %d F", t);

    在这里，t是数据，可以通过周围的代码公开访问。任何人都可以修改或阅读它。

    为什么这是不好的呢？有一个原因：紧密和隐藏的耦合。

    关于t的代码不可避免地对数据做出很多假设。例如，在int t之后的这两行代码决定了温度是以华氏度表示的。在编写代码的时候，这可能是正确的，但这个假设将代码与数据耦合在一起。如果明天我们将t更改为摄氏度，代码将不知道这个变化。这就是我所说的隐藏耦合。

    如果我们将t的类型从int更改为double，printf行将不会在小数点后打印任何内容。再次强调，耦合是存在的，但它是隐藏的。后来，我们将无法简单地找到在代码中我们对t进行了这些或其他假设的所有位置。

    这将严重影响可维护性。

    这并不是一个解决方案，正如你所能想象的（现在的Java）。

    class Temperature {
  private int t;
  public int getT() { return this.t; }
  public void setT(int t) { this.t = t; }
}

    看起来像一个对象，但数据仍然裸露。任何人都可以从对象中检索 t 并决定它是华氏度还是摄氏度，是否有小数点后的数字等等。这还不是封装！

    唯一封装t的方法是确保无论是直接访问它还是从对象中检索它，都无法触及它。我们该如何做到这一点呢？只需停止暴露数据并开始暴露功能。以下是一个示例：

    class Temperature {
  private int t;
  public String toString() {
    return String.format("%d F", this.t);
  }
}

    我们不再允许任何人取回t。他们能做的只是将温度转换为文本。如果我们决定将t更改为摄氏度，我们将只在一个地方的Temperature类中进行一次更改。

    如果我们将来需要其他功能，比如数学运算或者转换为摄氏度，我们可以在 Temperature 类中添加更多的方法。但是我们绝不让任何人接触或者了解 t。

    这个想法与我们之前讨论的“打印机而不是获取器”非常接近，尽管视角更加广泛。在这里，我是在说任何逃逸对象的数据元素都是“裸露”的，会导致可维护性问题。

    问题是我们如何完全不使用裸数据，对吗？最终我们必须让对象交换数据，对吗？是的，这是正确的。但并非完全如此。我会在下一篇文章中解释。

    class Book {
    private int price;
    public int getPrice() {
    return this.price;
    }
    }

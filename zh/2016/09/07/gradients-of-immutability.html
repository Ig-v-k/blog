    Gradients of Immutability

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    好的对象是不可变的，但不一定是常量。我在这里、这里和这里都试图解释过，但现在是时候再试一次了。实际上，我越想越明白，不可变性并不是非黑即白的——还有一些更多的渐变；让我们来看看。

    如我们在这里达成的协议，一个对象是某个实体（或实体集、其他对象、数据、内存、文件等）的代表。让我们来研究一些在我们看来完全相同但代表不同事物的对象，然后分析它们的不可变性及原因。

    Constant

    这是常数；它不允许对封装实体进行任何修改，并始终返回相同的文本（为了简洁起见，我跳过了构造函数）。

    class Book {
  private final String ttl;
  Book rename(String title) {
    return new Book(title);
  }
  String title() {
    return this.ttl;
  }
}

    这是我们通常在谈论不可变对象时心中所想的。这样的类非常接近于一个纯函数，这意味着无论我们用相同的初始值实例化多少次，title() 的结果都将是相同的。

    Not a Constant

    Check out this one:

    class Book {
  private final String ttl;
  Book rename(String title) {
    return new Book(title);
  }
  String title() {
    return String.format(
      "%s (as of %tR)", this.ttl, new Date()
    );
  }
}

    该对象仍然是不可变的，但由于title()方法的存在，它不再是一个纯函数—如果我们以至少一分钟的间隔多次调用它，它将返回不同的值。该对象是不可变的，只是不再是一个常量。

    Represented Mutability

    How about this one:

    class Book {
  private final Path path;
  Book rename(String title) {
    Files.write(
      this.path,
      title.getBytes(),
      StandardOpenOption.CREATE
    );
    return this;
  }
  String title() {
    return new String(
      Files.readAllBytes(this.path)
    );
  }
}

    这个不可变对象将书名保存在一个文件中。它不是一个常量，因为它的方法title()可能在每次调用时返回不同的值。此外，所表示的实体（文件）也不是一个常量。我们无法确定它是可变的还是不可变的，因为我们不知道Files.write()是如何实现的。但我们确定它不是一个常量，因为它接受变更请求。

    Encapsulated Mutability

    一个不可变对象不仅可以表示，甚至可以封装一个可变对象。就像在前面的例子中，一个可变的文件被封装起来了。尽管它被不可变类Path所表示，但真正的磁盘文件是可变的。我们可以在内存中做同样的事情。

    class Book {
  private final StringBuffer buffer;
  Book rename(String title) {
    this.buffer.setLength(0);
    this.buffer.append(title);
    return this;
  }
  String title() {
    return this.buffer.toString();
  }
}

    这个对象仍然是immutable（不可变的）。它是线程安全的吗？不是。它是一个常量吗？不是。它是不可变的吗？是的。感到困惑了吗？当然。

    My point is that immutability is not binary; there are many forms of it. The most simple one is, of course, a constant. Constants are almost the same as pure functions in functional programming. But object-oriented programming allows us to take a few steps forward and give immutable objects more permissions and flexibility. In OOP, we may have many more forms of immutability.

    所有这些示例共同之处在于我们的对象对其封装的实体是忠诚的。没有任何能够修改它们的设置器。所有封装的对象都是final的。

    这是唯一可以区分可变对象和不可变对象的特征。后者始终忠于其封装和代表的实体。至于其他的…那就看情况而定。

    Inheritance Is a Procedural Technique for Code Reuse

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我们都知道继承是不好的，而组合优于继承是一个好主意，但我们真的明白为什么吗？在所有介绍这个主题的文章中，作者都说继承可能对你的代码有害，所以最好不要使用它。这个”最好”部分让我感到困惑；这是否意味着有时候继承是有意义的？我几周前采访了David West（他是我最喜欢的关于面向对象编程的书《Object Thinking》的作者），他说继承在面向对象编程中根本不应该存在（完整视频）。也许韦斯特博士是对的，我们应该完全忘记Java中的extends关键字？

    我认为我们应该。而且我认为我知道原因。

    这并不是因为我们引入了不必要的耦合，正如Allen Holub在他的为什么extends是邪恶的文章中所说的那样。他绝对是正确的，但我认为这并不是问题的根本原因。

    “继承”作为一个英语动词，有许多含义。我猜，这个链接中所指的是Simula中的继承发明者们的意思：”从父母或祖先那里遗传（品质、特征或倾向）”。

    从另一个对象中派生出特征是一个很好的想法，它被称为子类型化。它完美地适用于面向对象编程，并实际上实现了多态性：Article类的对象继承了Manuscript类中对象的所有特征并添加了自己的特征。例如，它继承了自身打印的能力并添加了自身提交到会议的能力。

    interface Manuscript {
  void print(Console console);
}
interface Article extends Manuscript {
  void submit(Conference cnf);
}

    这是子类型化，它是一种完美的技术；每当需要一份手稿时，我们可以提供一篇文章，而没有人会注意到任何问题，因为类型Article是类型Manuscript的子类型（Liskov替换原则）。

    但是从父类到子类的方法和属性的复制与“派生特性”有什么关系呢？实现继承就是这样—复制—它与我上面引用的“继承”一词的含义无关。

    实现继承更接近于不同的含义: “在前任持有人死亡时作为继承人接受（金钱、财产或头衔）”。你问谁已经死了？如果一个对象允许其他对象继承它封装的代码和数据，那么它就死了。这就是实现继承。

    class Manuscript {
  protected String body;
  void print(Console console) {
    console.println(this.body);
  }
}
class Article extends Manuscript {
  void submit(Conference cnf) {
    cnf.send(this.body);
  }
}

    类 Article 从类 Manuscript 中复制了方法 print() 和属性 body，就像它不是一个活生生的生物，而是一个我们可以继承其部分的死物，”金钱、财产或头衔”。

    实现继承是作为一种代码重用机制创建的，它根本不适用于面向对象编程。是的，一开始它可能看起来很方便，但从对象思维的角度来看，这是完全错误的。就像获取器和设置器一样，实现继承将对象转化为带有数据和过程的容器。当然，将其中一些数据和过程复制到新对象中以避免代码重复是很方便的。但这不是对象的本质。它们不是死的，它们是活的！

    不要用继承“杀死”它们！

    因此，我认为继承是不好的，因为它是一种用于代码重用的过程化技术。毫不奇怪，它引入了多年来人们一直在讨论的所有问题。因为它是过程化的！这就是为什么它不适用于面向对象编程的原因。

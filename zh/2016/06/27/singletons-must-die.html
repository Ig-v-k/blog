    Singletons Must Die

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我认为说单例模式是一种反模式太明显了，因为有很多关于这个的文章。然而，更常见的问题是如何在没有单例的情况下定义全局事物；对于我们中的许多人来说，这个答案并不明显。有几个例子：数据库连接池、存储库、配置映射等。它们都自然地似乎是”全局”的；但是我们该怎么处理它们呢？

    我假设你已经知道什么是单例模式以及为什么它是一个反模式。如果不知道的话，我建议你阅读这个 Stack Overflow 的讨论帖子：单例模式有什么问题？

    既然我们都同意这是一个糟糕的交易，那么如果我们需要在应用程序的多个不同位置访问数据库连接池，我们该怎么办呢？我们只需要类似下面这样的东西：

    class Database {
  public static Database INSTANCE = new Database();
  private Database() {
    // Create a connection pool
  }
  public java.sql.Connection connect() {
    // Get a new connection from the pool
    // and return it
  }
}

    稍后在JAX-RS REST方法中，我们需要从数据库中检索某些内容。

    @Path("/")
class Index {
  @GET
  public String text() {
    java.sql.Connection connection =
      Database.INSTANCE.connect();
    return new JdbcSession(connection)
      .sql("SELECT text FROM table")
      .fetch(new SingleOutcome(String.class))
  }
}

    如果您对JAX-RS不熟悉，它是一个简单的MVC架构，而这个text()方法是一个”控制器”。另外，我正在使用JdbcSession，这是一个简单的JDBC包装器，来自jcabi-jdbc。

    我们需要Database.INSTANCE是一个单例吧？我们需要它在全局范围内可用，以便任何 MVC 控制器都能直接访问它。由于我们都明白并同意单例是一种不好的设计，我们用什么来替代它呢？

    一个依赖注入就是答案。

    我们需要将这个数据库连接池的依赖项作为控制器的一部分，并确保通过构造函数提供。然而，在这种特殊情况下，对于JAX-RS，我们无法通过构造函数实现，因为它的架构很丑陋。但是我们可以创建一个ServletContextListener，在其contextInitialized()方法中实例化一个Database，并将该实例作为servletContext的属性添加。然后，在控制器内部，我们通过将javax.ws.rs.core.Context注解添加到setter方法上，并使用getAttribute()来检索servlet上下文。这绝对是可怕且过程化的，但比单例模式要好。

    一个合适的面向对象设计会通过构造函数将Database的实例传递给所有可能需要它的对象。

    然而，如果有很多依赖关系，我们该怎么办呢？我们会创建一个有10个参数的构造函数吗？不，我们不会这样做。如果我们的对象确实需要10个依赖关系来完成工作，我们需要将它们拆分成更小的依赖关系。

    就是这样。忘记单例；不要使用它们。将它们转换为依赖项，并通过new操作符从对象传递给对象。

    Test Methods Must Share Nothing

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    常量…我之前写过一些关于它们的文章，大多是说如果公开使用它们是件坏事。它们可以减少重复，但也会引入耦合。摆脱重复的更好方法是创建新的类或方法—这是一种传统的面向对象编程方法。这似乎是有道理的，在我们的项目中，我看到越来越少的公开常量。在某些项目中，我们根本没有它们。但有一件事仍然困扰着我：单元测试。大多数程序员似乎认为，当静态分析指出同一文件中有太多相似的文字时，摆脱它们的最佳方法是使用私有的静态文字。这是错误的。

    单元测试自然而然地会重复很多代码。测试方法包含相似或几乎相同的功能，这几乎是不可避免的。嗯，我们可以使用更多的@Before和@BeforeClass功能，但有时候这是不可能的。我们可能在一个FooTest.java文件中有20个测试方法。不可能在一个“before”中准备所有对象。所以我们不得不在我们的测试方法中一遍又一遍地做某些事情。

    让我们来看看我们的Takes Framework中的一个类：VerboseListTest。这是一个单元测试，它有一个问题，我想告诉你这个问题。看看那个MSG私有字面值。它首次在setUp()方法中作为对象构造函数的参数使用，然后在几个测试方法中使用来检查对象的行为。让我简化一下这段代码：

    class FooTest {
  private static final String MSG = "something";
  @Before
  public final void setUp() throws Exception {
    this.foo = new Foo(FooTest.MSG);
  }
  @Test
  public void simplyWorks() throws IOException {
    assertThat(
      foo.doSomething(),
      containsString(FooTest.MSG)
    );
  }
  @Test
  public void simplyWorksAgain() throws IOException {
    assertThat(
      foo.doSomethingElse(),
      containsString(FooTest.MSG)
    );
  }
}

    这基本上是VerboseListTest中正在发生的事情，而且这是非常错误的。为什么呢？因为这个共享字面量MSG在这两个测试方法之间引入了一种不自然的耦合。它们没有任何共同之处，因为它们测试Foo类的不同行为。但是这个私有常量将它们联系在一起。现在它们某种程度上有了关联。

    如果有一天我想修改其中一个测试方法，可能也需要修改另一个测试方法。比如说，我想要查看doSomethingElse()在封装的消息为空字符串时的行为。我该怎么办？我改变常量FooTest.MSG的值，而这个常量又被另一个测试方法使用。这被称为耦合，而这是一件坏事。

    我们要做什么？嗯，我们可以在两个测试方法中使用那个"something"字符串字面值。

    class FooTest {
  @Test
  public void simplyWorks() throws IOException {
    assertThat(
      new Foo("something").doSomething(),
      containsString("something")
    );
  }
  @Test
  public void simplyWorksAgain() throws IOException {
    assertThat(
      new Foo("something").doSomethingElse(),
      containsString("something")
    );
  }
}

    如你所见，我删除了setUp()方法和私有静态常量MSG。现在我们有了什么？代码重复。字符串"something"在测试类中出现了四次。没有静态分析器会容忍这种情况。此外，VerboseListTest中有七个测试方法，它们都在使用MSG。因此，我们将有14个"something"的出现，对吗？是的，没错，这很可能是这个测试用例的作者之一引入常量的原因——为了消除重复。顺便说一下，@Happy-Neko在拉取请求#513中做了这件事，@carlosmiranda审查了代码，我批准了这些更改。所以，三个人都犯了这个错误，包括我自己。

    那么，有没有一种正确的方法既可以避免代码重复，又不会引入耦合？答案就在这里：

    class FooTest {
  @Test
  public void simplyWorks() throws IOException {
    final String msg = "something";
    assertThat(
      new Foo(msg).doSomething(),
      containsString(msg)
    );
  }
  @Test
  public void simplyWorksAgain() throws IOException {
    final String msg = "something else";
    assertThat(
      new Foo(msg).doSomethingElse(),
      containsString(msg)
    );
  }
}

    这些文字必须是不同的。当静态分析器在许多地方看到"something"时，它就会这样说。它质疑我们——为什么它们是相同的？到底在任何地方都使用"something"是如此重要吗？为什么不能使用不同的文字？当然，我们可以。我们也应该这样做。

    底线是每个测试方法都必须有自己的数据和对象。它们绝不能在测试方法之间共享。测试方法必须始终独立，没有任何共同之处。

    考虑到这一点，我们可以很容易地得出结论，像setUp()这样的方法或者测试类中的任何共享变量是邪恶的。它们不应该被使用，也不应该存在。我认为它们在JUnit中的发明给Java代码带来了很多伤害。

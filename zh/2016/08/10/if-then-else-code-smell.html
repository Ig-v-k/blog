    If-Then-Else Is a Code Smell

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在大多数情况下（甚至可能是所有情况），if-then-else 可以且必须被一个装饰器或者仅仅是另一个对象取代。我已经计划写这个问题将近一年了，但直到今天才在我的代码中找到了一个完美地展示这个问题的真实案例。所以现在是时候来演示并解释了。

    请看一下来自yegor256/rultor的类DyTalk以及它的方法modify()。简而言之，它阻止你在XML文档未进行任何修改时向DynamoDB保存任何数据。这是一个有效的情况，需要进行验证，但它的实现方式是错误的。以下是它的工作方式（一个过度简化的例子）：

    class DyTalk implements Talk {
  void modify(Collection<Directive> dirs) {
    if (!dirs.isEmpty()) {
      // Apply the modification
      // and save the new XML document
      // to the DynamoDB table.
    }
  }
}

    你想知道什么出了问题？这个 if-then-else 分支功能实际上并不属于这个对象——这就是问题所在。修改 XML 文档并将其保存到数据库是它的功能，而如果修改指令集为空，则不保存任何内容（这与防御性编程非常相似）。相反，应该有一个装饰器，它的形式应该是这样的：

    class QuickTalk implements Talk {
  private final Talk origin;
  void modify(Collection<Directive> dirs) {
    if (!dirs.isEmpty()) {
      this.origin.modify(dirs);
    }
  }
}

    现在，如果我们需要在指令列表为空的情况下使我们的对话更加智能化，我们会使用QuickTalk进行修饰。好处是显而易见的：DyTalk类更小，因此更具凝聚力。

    但问题不仅仅是如此。我们能否将其规范化？我们能否说“每一次分叉”都是不好的，并且应该将其移出类？那么在方法内发生的无法转换为装饰器的分叉又该如何处理呢？

    我建议使用这个简单的规则：如果可以将 if-then-else 分支转换为装饰器，就必须这样做。如果没有这样做，那就是代码异味。明白吗？

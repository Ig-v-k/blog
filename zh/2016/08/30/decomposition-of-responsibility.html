    Vertical vs. Horizontal Decomposition of Responsibility

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    负责太多事情的对象是一个问题。因为它们的复杂性很高，很难进行维护和扩展。”责任分解”是我们为了将这些过于复杂的对象拆分成较小的对象而采取的措施。我看到了两种类型的重构操作：纵向和横向。我相信前者比后者更好。

    假设这是我们的代码（使用的是Ruby语言）：

    class Log
  def initialize(path)
    @file = IO.new(path, 'a')
  end
  def put(text)
    line = Time.now.strftime("%d/%m/%Y %H:%M ") + text
    @file.puts line
  end
end

    显然，这个类的对象做得太多了。它们将日志行保存到文件中，并对其进行格式化，这明显违反了著名的单一职责原则。这个类的对象将负责太多事情。我们需要将一些功能从中提取出来，并放到另一个对象（或多个对象）中。我们需要分解它的责任。无论我们把它放在哪里，提取后的Log类将如下所示：

    class Log
  def initialize(path)
    @file = IO.new(path, 'a')
  end
  def put(line)
    @file.puts line
  end
end

    现在它只将行保存到文件中，这很完美。这个类是内聚且小巧的。让我们创建一个实例：

    log = Log.new('/tmp/log.txt')

    接下来，我们把刚刚提取出来的具有格式化功能的代码放在哪里呢？分解责任有两种方法：水平和垂直。这种方法是水平的：

    class Line
  def initialize(text)
    @line = text
  end
  def to_s
    Time.now.strftime("%d/%m/%Y %H:%M ") + text
  end
end

    为了同时使用 Log 和 Line，我们必须这样做：

    log.put(Line.new("Hello, world"))

    看到为什么是水平的吗？因为这个脚本同时看到了它们。它们都处于相同的可见级别。当我们想要记录一行时，我们将总是需要与它们两个进行通信。Log和Line的两个对象都在我们面前。为了记录一行，我们必须处理这两个类。

    “[script] -down-> [log] [script] -down-> [Line]” translated to Chinese:

    “[脚本] -down-> [日志] [脚本] -down-> [行]”

    相反，这种责任的分解是“垂直的”：

    class TimedLog
  def initialize(log)
    @origin = log
  end
  def put(text)
    @origin.put(Time.now.strftime("%d/%m/%Y %H:%M ") + text)
  end
end

    TimedLog类是一个装饰器，以下是我们如何一起使用它们：

    log = TimedLog.new(log)

    现在，我们只是在日志中添加了一行。

    log.put("Hello, world")

    责任被垂直分解。我们仍然有一个进入log对象的入口点，但该对象“包含”两个对象，一个包裹在另一个对象中：

    “[脚本] -down-> [定时日志] [定时日志] -down-> [日志]”

    总的来说，我认为责任的水平分解是一个不好的想法，而垂直分解则更好。这是因为垂直分解的对象减少了复杂性，而水平分解的对象实际上增加了复杂性，因为它的客户端必须处理更多的依赖和接触点。

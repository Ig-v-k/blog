    The Law of Demeter Doesn't Mean One Dot

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    你可能听说过30年前的Demeter法则（LoD）。最近有人问我对它的看法。不仅仅是我的看法，还有如何保持对象小并遵守LoD的可能性。根据这个法则，我们不允许像book.pages().last().text()这样做。而是应该使用book.textOfLastPage()。这让我感到困惑，因为我强烈不同意。我认为第一种构造在面向对象编程中是完全有效的。因此，我进行了一些研究，以找出这个法则是否真的是一个法则。我发现这个法则是完美的，但在面向对象编程世界中对它的普遍理解是错误的（并不令人惊讶）。

    面向对象编程：客观的风格感，K.Lieberherr, I.Holland 和 A.Riel，OOPSLA’88 会议论文集，1988年。

    这是它被介绍的地方。让我们看看它字面上说的是什么（在那个PDF文件中查找第3节）。

    对于所有类C，对于所有附加到C的方法M，M发送消息的所有对象必须是与以下类相关联的类的实例：1）M的参数类（包括C），2）C的实例变量类。

    假设这是一个Java类：

    class C {
  private B b;
  void m(A a) {
    b.hello();
    a.hello();
    Singleton.INSTANCE.hello();
    new Z().hello();
  }
}

    根据迪米特法则（LoD），对四个不同的hello()方法的四次调用都是合法的。那么什么是非法的呢？我问自己。毫不奇怪，答案是：a.x.hello()。这将是非法的。直接从另一个对象访问属性并与其交互是违法的。

    但是我们无论如何都不会这样做。我们正在谈论 book.pages().last().text()。在这一系列的方法调用中，我们没有访问任何属性。我们是在请求我们的对象为我们构建新的对象。法律对此有什么说法？让我读一下并引用：

    由M创建的对象，或者由M调用的函数或方法创建的对象，被视为M的参数。

    换句话说，方法调用book.pages()返回的对象Pages是一个完全有效可用的对象。然后，我们可以在其上调用方法last()并获得一个对象Page，然后调用方法text()等。这是一个完全有效的情景，完全不违反规定，正如我所预期的。

    那么这种对法律的普遍理解是从哪里来的？为什么Wikipedia将其称为“一个点”的规则，并说“一个对象应该避免调用另一个方法返回的成员对象的方法”？这与原始论文完全相反！发生了什么事？

    答案很简单：getters。

    大多数面向对象编程（OOP）开发者认为大部分返回任何内容的对象方法都是 getters。而事实上，getters 和直接访问对象属性没有区别。这就是为什么维基百科实际上说“不要直接访问属性，因为大部分方法都是 getters，也不要碰它们，傻瓜”。

    看到这真是悲哀。

    所以底线是，德米特法则并不完全反对方法链。当然，它反对的是获取器和直接属性访问。但是，谁不反对呢，对吧？

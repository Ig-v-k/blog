    Data Transfer Object Is a Shame

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    DTO ，据我所了解，是ORM设计模式的一个基石，我“非常喜爱”。但让我们直入主题：DTO只是个耻辱，而发明它的人是错的。他所做的没有任何借口。

    顺便提一下，据我所知，他的名字是Martin Fowler。也许他不是DTO的唯一发明者，但他使其合法并推荐了使用。恕我直言，他是错的。

    面向对象编程的关键思想是将数据隐藏在对象后面。这个思想有一个名字：封装。在面向对象编程中，数据不应该是可见的。对象只能访问它们封装的数据，而不能访问其他对象封装的数据。对于这个原则无法争辩——这就是面向对象编程的全部内容。

    然而，DTO与该原则完全相悖。

    让我们看一个实际的例子。假设这是一个从某个RESTful API获取JSON文档并返回DTO的服务，我们可以将其存储在数据库中。

    Book book = api.loadBookById(123);
database.saveNewBook(book);

    我猜这就是loadBookById()方法内部会发生的事情。

    Book loadBookById(int id) {
  JsonObject json = /* Load it from RESTful API */
  Book book = new Book();
  book.setISBN(json.getString("isbn"));
  book.setTitle(json.getString("title"));
  book.setAuthor(json.getString("author"));
  return book;
}

    我是对的吗？我敢打赌我是对的。这已经让我感到恶心了。无论如何，让我们继续。这很可能会发生在saveNewBook()方法中（我正在使用纯JDBC）。

    void saveNewBook(Book book) {
  Statement stmt = connection.prepareStatement(
    "INSERT INTO book VALUES (?, ?, ?)"
  );
  stmt.setString(1, book.getISBN());
  stmt.setString(2, book.getTitle());
  stmt.setString(3, book.getAuthor());
  stmt.execute();
}

    这个 Book 是数据传输对象设计模式的一个典型例子。它只是在两段代码、两个过程之间传输数据而已。这个对象 book 很蠢。它只知道…什么都不做。它实际上并不是一个对象，而是一个被动且贫血的数据结构。

    什么是正确的设计？有几种选择。例如，对我来说，这个看起来不错。

    Book book = api.bookById(123);
book.save(database);

    这是bookById()函数中发生的事情。

    Book bookById(int id) {
  return new JsonBook(
    /* RESTful API access point */
  );
}

    这是在Book.save()中发生的事情。

    void save(Database db) {
  JsonObject json = /* Load it from RESTful API */
  db.createBook(
    json.getString("isbn"),
    json.getString("title"),
    json.getString("author")
  );
}

    如果有许多书籍的 JSON 参数无法很好地作为参数放入一个 createBook() 方法中，会发生什么情况呢？这样怎么样呢：

    void save(Database db) {
  db.create()
    .withISBN(json.getString("isbn"))
    .withTitle(json.getString("title"))
    .withAuthor(json.getString("author"))
    .deploy();
}

    有很多其他选择。但主要的一点是数据永远不会从对象book中逸出。一旦对象被实例化，数据对其他人来说是不可见或不可访问的。我们只能要求我们的对象保存自己或将自己打印到某种媒体上，但我们将永远不会从中获取任何数据。

    DTO的理念本身是错误的，因为它将面向对象的代码变成了过程式的代码。我们有用于处理数据的过程，而DTO只是数据的容器。不要以那种方式思考，也不要这样做。

    PS. 还有一些 DTO 的其他名称：business objects，domain objects（不在 DDD 中），entity objects，JavaBeans。

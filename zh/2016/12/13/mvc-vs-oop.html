    MVC vs. OOP

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    模型-视图-控制器 (MVC) 是一种我们都非常熟悉的架构模式。它是几乎所有UI和Web框架的事实标准。它方便易用，简单高效。对于过程化的程序员来说，这是一个伟大的概念…但如果你的软件是面向对象的，你应该像我一样不喜欢MVC。以下是原因。

    这就是MVC架构的样子：

    [控制器] -down- [模型] [控制器] -down- [视图]

    控制器负责处理从模型接收到的数据，并将其注入视图中——这正是问题所在。数据逃离了模型，并变得“裸露”，这是一个大问题，正如我们之前所同意的那样。面向对象编程的核心在于封装——数据隐藏。

    MVC 架构则恰恰相反，它通过“暴露”数据和隐藏行为来运作。控制器直接处理数据，对其目的和属性做出决策，而对象则应该了解数据的一切并将其隐藏起来，但实际上却显得贫乏无力。这正是任何过程式架构所基于的原则；代码负责数据。以这段 C++ 代码为例：

    void print_speed() { // controller
  int s = load_from_engine(); // model
  printf("The speed is %d mph", s); // view
}

    函数print_speed()是控制器。它从模型load_from_engine()获取数据s，并通过视图printf()进行渲染。只有控制器知道数据以英里/小时为单位。引擎返回没有任何属性的int。控制器只是简单地假设数据是以mph为单位的。如果我们想在其他地方创建类似的控制器，我们将不得不再次做出类似的假设。这就是所谓的“裸数据”问题，它会导致严重的可维护性问题。

    这是上述代码（伪C++）的面向对象替代方案。

    printf(
  new PrintedSpeed( // view
    new FormattedSpeed( // controller
      new SpeedFromEngine() // model
    )
  )
);

    在这里，SpeedFromEngine.speed() 返回以英里每小时为单位的速度，为整数；FormattedSpeed.speed() 返回 "%d mph"；最后，PrintedSpeed.to_str() 返回消息的完整文本。我们可以称它们为 “模型、视图和控制器”，但实际上它们只是相互装饰的对象。它们仍然是同一个实体—速度。但通过装饰，它变得更加复杂和智能。

    我们不会把速度这个概念分解开来。速度就是速度，不论是谁在使用它，它在哪里展示出来。它只是从装饰器中获得新的行为。它会增长，但永远不会分崩离析。

    总结一下，控制器是MVC三元组中的一个纯粹的过程化组件，它将模型转化为被动的数据持有者，并将视图转化为被动的数据渲染器。控制器、持有器、渲染器… 这真的是面向对象编程吗？

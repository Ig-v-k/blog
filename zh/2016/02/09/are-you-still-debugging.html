    Are You Still Debugging?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    【调试】(https://en.wikipedia.org/wiki/Debugging)是“一种交互式运行程序/方法的过程，在每个语句之后中断执行流并显示…” 简而言之，它是一种非常有用的技术… 对于一个糟糕的程序员来说。或者一个仍然在用C编写过程化代码的老程序员。面向对象的程序员从不调试他们的代码—他们编写单元测试。我在这里的观点是，单元测试是一种完全取代调试的技术。如果需要调试，那么设计是糟糕的。

    假设我是一个糟糕的命令式过程式编程者，这是我的Java代码：

    class FileUtils {
  public static Iterable<String> readWords(File f) {
    String text = new String(
      Files.readAllBytes(Paths.get(f)),
      "UTF-8"
    );
    Set<String> words = new HashSet<>();
    for (String word : text.split(" ")) {
      words.add(word);
    }
    return words;
  }
}

    这个静态实用方法读取文件内容，然后找出其中所有唯一的单词。非常简单。然而，如果它不起作用，我们该怎么办呢？假设这是文件内容：

    We know what we are,
but know not what we may be.

    从中，我们得到以下单词列表：

    "We"
"know"
"what"
"we"
"are,\n"
"but"
"not"
"may"
"be\n"

    现在对我来说看起来不对劲…那么下一步是什么呢？要么文件读取不正确，要么分割有问题。让我们进行调试，对吧？让我们通过输入给它一个文件，逐步进行追踪和观察变量。我们会找到错误并修复它。但是当类似的问题出现时，我们就必须再次进行调试！而这就是单元测试应该预防的。

    我们应该创建一个单元测试，其中重现了问题。然后我们修复问题并确保测试通过。这是我们在问题解决中节省投资的方法。我们不会再次修复它，因为它不会再次发生。我们的测试将防止它再次发生。

    然而，所有这一切只有在创建单元测试变得容易时才能起作用。如果很困难，我会懒得去做。我只会调试和修复问题。在这个特定的例子中，创建一个测试是一个相当昂贵的过程。我的意思是单元测试的复杂度将会很高。我们需要创建一个临时文件，填充数据，运行方法并检查结果。为了弄清楚发生了什么以及错误在哪里，我将不得不创建一些测试。为了避免代码重复，我还必须创建一些辅助工具来帮助我创建临时文件并填充数据。这是一项很大的工作。嗯，也许不是“很多”，但比几分钟的调试要多得多。

    因此，如果你认为调试更快更容易，那就考虑一下你代码的质量吧。我敢打赌它有很多重构的机会，就像上面的例子中的代码一样。以下是我会如何修改它。首先，我会将其转换为一个类，因为实用的静态方法是一种不好的实践：

    class Words implements Iterable<String> {
  private final File file;
  Words(File src) {
    this.file = src;
  }
  @Override
  public Iterator<String> iterator() {
    String text = new String(
      Files.readAllBytes(Paths.get(this.file)),
      "UTF-8"
    );
    Set<String> words = new HashSet<>();
    for (String word : text.split(" ")) {
      words.add(word);
    }
    return words.iterator();
  }
}

    看起来已经好多了，但复杂性仍然存在。接下来，我会将其拆分成较小的类：

    class Text {
  private final File file;
  Text(File src) {
    this.file = src;
  }
  @Override
  public String toString() {
    return new String(
      Files.readAllBytes(Paths.get(this.file)),
      "UTF-8"
    );
  }
}
class Words implements Iterable<String> {
  private final String text;
  Words(String txt) {
    this.text = txt;
  }
  @Override
  public Iterator<String> iterator() {
    Set<String> words = new HashSet<>();
    for (String word : this.text.split(" ")) {
      words.add(word);
    }
    return words.iterator();
  }
}

    现在你的想法是什么？为Words类编写一个测试是一项相当简单的任务：

    import org.junit.Test;
import static org.hamcrest.MatcherAssert.*;
import static org.hamcrest.Matchers.*;
public class WordsTest {
  @Test
  public void parsesSimpleText() {
    assertThat(
      new Words("How are you?"),
      hasItems("How", "are", "you")
    );
  }
}

    那花了多少时间？不到一分钟。我们不需要创建临时文件并加载数据，因为Words类不会处理文件。它只是解析传入的字符串并找出其中的唯一单词。现在很容易修复，因为测试很小，我们可以轻松地创建更多的测试；例如：

    import org.junit.Test;
import static org.hamcrest.MatcherAssert.*;
import static org.hamcrest.Matchers.*;
public class WordsTest {
  @Test
  public void parsesSimpleText() {
    assertThat(
      new Words("How are you?"),
      hasItems("How", "are", "you")
    );
  }
  @Test
  public void parsesMultipleLines() {
    assertThat(
      new Words("first line\nsecond line\n"),
      hasItems("first", "second", "line")
    );
  }
}

    我的观点是，当编写单元测试所需的时间显著超过点击跟踪输入/跟踪输出按钮所需的时间时，调试是必要的。这是合乎逻辑的。我们都是懒惰的，想要快速简单的解决方案。但调试会耗费时间和浪费能量。它帮助我们找到问题，但不能防止它们再次出现。

    调试在我们的代码是过程化和算法化的时候是必需的—当代码完全关注于怎样达到目标而不是目标是什么时。再次参考上面的例子。第一个静态方法完全关注于我们如何读取文件、解析文件和查找单词。甚至它的命名为readWords()（一个动词）。相反地，第二个例子关注于将会被实现的是什么。它可以是文件的Text或文本的Words（两者都是名词）。

    我相信在纯粹的面向对象编程中没有调试的地方，只有单元测试！

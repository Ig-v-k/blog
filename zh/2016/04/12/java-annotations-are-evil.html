    Java Annotations Are a Big Mistake

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    注解在Java 5中被引入，我们都感到很兴奋。这是一个很好的工具，可以使代码更简洁！不再需要Hibernate/Spring XML配置文件！就在我们需要它们的代码中，只需要注解。不再需要标记接口，只需要一个保留运行时、反射可发现的注解！我也感到很兴奋。此外，我开发了一些大量使用注解的开源库。以jcabi-aspects为例。然而，我不再感到兴奋。此外，我认为注解是Java设计中的一个大错误。

    长话短说，注解存在一个大问题—它们鼓励我们在一个对象之外实现object功能，这与封装的原则背道而驰。对象不再是完整的，因为其行为不完全由自身的方法定义—一些功能留在其他地方。为什么这是不好的呢？让我们看几个例子。

    @Inject

    假设我们用 @Inject 注解一个属性：

    import javax.inject.Inject;
public class Books {
  @Inject
  private final DB db;
  // some methods here, which use this.db
}

    然后我们有一个知道要注入什么的注入器

    Injector injector = Guice.createInjector(
  new AbstractModule() {
    @Override
    public void configure() {
      this.bind(DB.class).toInstance(
        new Postgres("jdbc:postgresql:5740/main")
      );
    }
  }
);

    现在我们通过容器创建了Books类的一个实例。

    Books books = injector.getInstance(Books.class);

    类Books不知道如何以及谁会将DB类的实例注入其中。这将在幕后并在其控制之外发生。注入将完成此操作。这可能看起来很方便，但这种态度会给整个代码库带来很多损害。控制丧失了（不是倒置，而是丧失了！）。对象不再负责。它无法对所发生的事情负责。

    相反，以下是应该如何完成此操作的方式:

    class Books {
  private final DB db;
  Books(final DB base) {
    this.db = base;
  }
  // some methods here, which use this.db
}

    这篇文章解释了为什么依赖注入容器从一开始就是一个错误的想法：依赖注入容器是代码污染者。注解基本上促使我们创建和使用容器。我们将功能从我们的对象中移出并放入容器或其他地方。这是因为我们不想一遍又一遍地重复相同的代码，对吗？这是正确的，重复是不好的，但是将一个对象拆分开来更糟糕。要糟糕得多。这也适用于ORM（JPA/Hibernate），注解在其中被积极使用。查看这篇文章，它解释了ORM的问题：ORM是冒犯性的反模式。注解本身并不是关键的动因，但它们通过将对象拆分并将部分放在不同的地方来帮助我们和鼓励我们。它们是容器、会话、管理器、控制器等。

    @XmlElement

    这是JAXB的工作方式，当您想要将您的POJO转换为XML时。首先，您将@XmlElement注解附加到getter方法上。

    import javax.xml.bind.annotation.XmlElement;
import javax.xml.bind.annotation.XmlRootElement;
@XmlRootElement
public class Book {
  private final String title;
  public Book(final String title) {
    this.title = title;
  }
  @XmlElement
  public String getTitle() {
    return this.title;
  }
}

    然后，您需要创建一个编组器，并要求它将一个Book类的实例转换为XML。

    final Book book = new Book("0132350882", "Clean Code");
final JAXBContext ctx = JAXBContext.newInstance(Book.class);
final Marshaller marshaller = ctx.createMarshaller();
marshaller.marshal(book, System.out);

    谁在创建XML？不是book。别的人，在Book类之外。这样做是非常错误的。相反，应该这样做。首先，那个对XML一无所知的类：

    class DefaultBook implements Book {
  private final String title;
  DefaultBook(final String title) {
    this.title = title;
  }
  @Override
  public String getTitle() {
    return this.title;
  }
}

    然后，将其打印成XML的装饰器。

    class XmlBook implements Book{
  private final Book origin;
  XmlBook(final Book book) {
    this.origin = book;
  }
  @Override
  public String getTitle() {
    return this.origin.getTitle();
  }
  public String toXML() {
    return String.format(
      "<book><title>%s</title></book>",
      this.getTitle()
    );
  }
}

    现在，为了将书籍打印成XML格式，我们要执行以下操作：

    String xml = new XmlBook(
  new DefaultBook("Elegant Objects")
).toXML();

    XML打印功能位于XmlBook中。如果您不喜欢装饰器的想法，您可以将toXML()方法移动到DefaultBook类中。这不重要。重要的是功能始终保持在其所属的对象内部。只有对象自己知道如何将自己打印成XML。没有其他人能知道！

    @RetryOnFailure

    这是一个例子（来自[我的自己的库]（））：

    import com.jcabi.aspects.RetryOnFailure;
class Foo {
  @RetryOnFailure
  public String load(URL url) {
    return url.openConnection().getContent();
  }
}

    编译完成后，我们会运行一个所谓的AOP weaver，它在技术上将我们的代码转换成如下形式：

    class Foo {
  public String load(URL url) {
    while (true) {
      try {
        return _Foo.load(url);
      } catch (Exception ex) {
        // ignore it
      }
    }
  }
  class _Foo {
    public String load(URL url) {
      return url.openConnection().getContent();
    }
  }
}

    我简化了在方法调用失败时重试的实际算法，但我相信你能理解。AspectJ，AOP引擎，使用@RetryOnFailure注解作为一个信号，告诉我们这个类必须被包装到另一个类中。这是在幕后进行的。我们看不到那个实现重试算法的辅助类。但是由AspectJ编织器生成的字节码包含了一个修改过的Foo类的版本。

    这正是这种方法的问题所在——我们看不到，也无法控制那个补充对象的实例化。对象组合，是对象设计中最重要的过程，却被隐藏在幕后。你可能会说，既然它是补充性的，我们就不需要看到它。但我不同意。我们必须看到我们的对象是如何组合的。我们可能不关心它们的工作方式，但我们必须看到整个组合过程。

    一个更好的设计应该是这样的（而不是批注）：

    Foo foo = new FooThatRetries(new Foo());

    然后，FooThatRetries的实现。

    class FooThatRetries implements Foo {
  private final Foo origin;
  FooThatRetries(Foo foo) {
    this.origin = foo;
  }
  public String load(URL url) {
    return new Retry().eval(
      new Retry.Algorithm<String>() {
        @Override
        public String eval() {
          return FooThatRetries.this.load(url);
        }
      }
    );
  }
}

    现在，Retry的实现。

    class Retry {
  public <T> T eval(Retry.Algorithm<T> algo) {
    while (true) {
      try {
        return algo.eval();
      } catch (Exception ex) {
        // ignore it
      }
    }
  }
  interface Algorithm<T> {
    T eval();
  }
}

    代码变长了吗？是的。变得更清晰了吗？更多了。我很后悔两年前开始使用jcabi-aspects时没有理解它。

    The bottom line is that annotations are bad. Don’t use them. What should be used instead? Object composition.

    有什么比注释更糟糕的呢？配置。例如，XML 配置。Spring XML 配置机制是一个糟糕设计的典型例子。我之前已经说过很多次了。让我再重复一遍——Spring 框架是 Java 世界中最糟糕的软件产品之一。如果你能远离它，你将对自己有很大的帮助。

    在OOP中不应该有任何“配置”。如果我们的对象是真实的对象，我们无法对它们进行配置。我们只能实例化它们。而最好的实例化方法是使用new运算符。这个运算符是面向对象编程开发者的关键工具。把它从我们手中拿走并给予我们“配置机制”是不可饶恕的罪行。

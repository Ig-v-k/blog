    Printers Instead of Getters

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    获取器和设置器是邪恶的。无需对此争论，这是确定的。你不同意吗？我们以后再讨论。现在，假设我们想要摆脱获取器。关键问题是怎么可能？我们确实需要从对象中获取数据，对吗？不对。错了。

    我建议使用“打印机”代替。对象不再通过获取器来公开数据，而是具备将自身“打印”到某种媒体的功能。

    假设这是我们的类：

    public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
}

    我们需要将其转换为XML格式。一个更或多或少传统的方法是通过getters和JAXB进行转换。

    import javax.xml.bind.annotation.XmlElement;
import javax.xml.bind.annotation.XmlRootElement;
@XmlRootElement
public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
  @XmlElement
  public String getIsbn() {
    return this.isbn;
  }
  @XmlElement
  public String getTitle() {
    return this.title;
  }
}

    这是一种非常冒犯物体的方式。我们基本上将内部的一切都暴露给公众。它原本是一个不错的自给自足的实体物体，而我们却将其变成了一袋数据，任何人都可以通过多种方式访问。当然，我们可以访问它以进行阅读。

    你可能会说，拥有这些getter方法很方便。我们都习惯了它们。如果我们想将其转换为JSON，它们会非常有帮助。如果我们想在JSP中将这个可怜的对象用作数据对象，getter方法将帮助我们。在Java中有很多例子，getter方法被广泛使用。

    这不是因为它们非常有效。这是因为我们在思考方式上非常”过程化”。我们不相信我们的对象。我们只相信它们存储的数据。我们不希望这个Book对象生成XML。我们希望它给我们数据。生成XML的任务由我们来完成。Book对象太愚蠢了，不能胜任这项工作。我们聪明多了！

    我建议停止这种思考方式。相反，让我们试着给这个可怜的Book一个机会，并为它配备一个”打印机”：

    public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
  public String toXML() {
    return String.format(
      "<book><isbn>%s</isbn><title>%s</title></book>",
      this.isbn, this.title
    );
  }
}

    这不是最好的实现方式，但你已经有了想法。对象不再暴露其内部细节。我们无法获取它的ISBN和标题。我们只能要求它以XML格式打印自己。

    如果需要另一种格式，我们可以添加一个额外的打印机。

    public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
  public String toJSON() {
    return String.format(
      "{\"isbn\":\"%s\", \"title\":\"%s\"}",
      this.isbn, this.title
    );
  }
}

    再次強調，這不是最佳的實現方式，但您可以看到我試圖展示的內容。每次我們需要一種新的格式時，我們都會創建一個新的打印機。

    也許您會說，如果有很多格式，對象會變得相當大。這是真的，但一個大的對象本來就是一個糟糕的設計。我會說，如果有不止一個打印機，這是一個問題。

    那麼，如果我們需要多種格式該怎麼辦？使用”媒體”，其中打印機將能夠打印到媒體上。例如，假設我們有一個代表MySQL中一條記錄的對象。我們希望它能夠打印成XML、HTML、JSON、某種二進制格式等等。我們可以將許多打印機添加到該對象上，但對象會變得臃腫且難看。為了避免這種情況，引入一個新的對象，代表數據將要打印到的媒體：

    public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
  public Media print(Media media) {
    return media
      .with("isbn", this.isbn)
      .with("title", this.title);
  }
}

    再次强调，这是一个非常原始的不可变Media类的设计，但是你已经理解了——media接受数据。现在，我们想要将我们的对象打印为JSON（这个设计并不完美，因为JsonObjectBuilder并不是不可变的，尽管看起来是一个…）：

    class JsonMedia implements Media {
  private final JsonObjectBuilder builder;
  JsonMedia() {
    this("book");
  }
  JsonMedia(String head) {
    this(Json.createObjectBuilder().add(head));
  }
  JsonMedia(JsonObjectBuilder bdr) {
    this.builder = bdr;
  }
  @Override
  public Media with(String name, String value) {
    return new JsonMedia(
      this.builder.add(name, value)
    );
  }
  public JsonObject json() {
    return this.builder.build();
  }
}

    现在，我们创建一个JsonMedia的实例，并要求我们的书在那里打印自己。

    JsonMedia media = new JsonMedia("book");
media = book.print(media);
JsonObject json = media.json();

    看！JSON 对象已经准备好了，而书本对于刚刚打印出来的内容一无所知。我们需要将书本打印成 XML 吗？我们创建了 XmlMedia，它将把书本打印成 XML。Book 类保持简洁，而“媒体”对象的复杂性则没有限制。

    我在这里的观点很简单——没有 getter，只有打印机！

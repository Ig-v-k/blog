    SQL as a Service

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我从2007年开始思考这个问题，大约是在亚马逊发布S3的时候。我甚至尝试过几次实施，但都在设计阶段失败了。我听说过一家初创公司也尝试过，但也失败了。我仍然不确定是否有可能实现，但它肯定可以成为云数据管理市场的畅销产品。等等，你可能会说，Google Cloud SQL、AWS RDS、Microsoft Azure、Heroku PostgreSQL和其他很多产品呢？它们与我所指的完全不同。

    让我给你举个类比。假设你想要将一段二进制数据存储在云端。我有两个解决方案供你选择。第一个是一个拥有 FTP 的托管服务器。你每月支付5美元给我，我会给你提供服务器的FTP访问权限，该服务器有100Gb的磁盘空间。你可以将文件上传到那里并进行下载。运行得非常顺利。而我还有第二个选择，那就是 AWS S3。你也可以通过他们的 API 来上传和下载你的数据。而你需要支付的是每个API请求、每个托管的字节和每个传输的字节，而不是固定的月费。你会选择哪个方案呢？

    显然，你会选择S3。为什么呢？这两者之间的根本区别是什么？关键区别在于它们的服务级别协议（SLAs）：第一个带有FTP的是一个服务器，而第二个是一个服务。

    一个FTP服务器提供商保证您计算资源（CPU、磁盘、带宽等）的可用性，而S3保证数据的可用性。如果FTP服务器上的磁盘崩溃，将及时更换，但数据将会丢失。如果磁盘已满，您可以订购额外的服务器，但您有责任不要忘记。如果磁盘空间未被使用，您仍需支付每月5美元。等等。

    AWS S3在市场上是一个突破，十多年前，正是因为这个差异。他们在我们都习以为常的web hosting之上添加了一个新的service layer。这个想法仍然是相同的—数据仍然在云端，我们上传和下载—但是服务等级协议不同了。我们不再需要担心磁盘溢出、为未使用空间支付过多费用、定期备份、SSH终端等等。他们只是给我们一个简单的API和承诺数据在那里且是安全的。

    现在已经是2019年了，我们对于关系数据仍然没有相同的解决方案。无论选择哪个提供商，他们所做的只是给你一台（或一组）安装了MySQL或PostgreSQL（或他们自己版本）的机器，并按小时计费。他们仍然只提供“好旧的FTP”，没有额外的服务层。

    这就是我所期望的真正的云上关系型数据服务级别协议的声音。

    • 自动缩放。不用让我们担心托管数据所需的资源量。只需针对较大的数据集收取更多费用，并确保我们的请求在可预测的时间内返回。

    • 按数据付费。让我们为每个SQL请求、每个存储的字节和每个传输的字节付费。需要多少服务器和磁盘来托管所有这些内容——这不应该是我们关心的事情。

    • 受限制的 SQL。在大多数项目中，不需要使用 MySQL 或 PostgreSQL 方言中的大多数命令。只需提供 INSERT、SELECT、UPDATE 和 DELETE 即可。

    • 索引。使用我们进行的SQL查询的统计数据自动创建索引。

    • Schema版本控制。通过类似于Liquibase的方式，使得可以通过创建新的ALTER TABLE或CREATE TABLE脚本来更新schema，并将其应用于现有的数据库。

    • 快照和回滚。可以对数据进行快照，应用新的模式版本，如果出现问题，则可以回滚到先前创建的某个快照。

    SQL as a Service

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    自从2007年以来，我就一直在思考这个问题，大约在那个时候，亚马逊推出了S3。我甚至尝试过几次去实现它，但总是在设计阶段之后失败了。我听说过一家初创公司也试图做同样的事情，但也失败了。我仍然不确定是否可能实现，但它肯定有望成为云数据管理市场的畅销产品。等等，你可能会说，Google Cloud SQL、AWS RDS、Microsoft Azure、Heroku PostgreSQL以及其他许多产品呢？它们甚至离我所说的还差得远。

    让我给你举个类比。假设你想在云端存储一段二进制数据。我有两个解决方案供你选择。第一个是一个托管的服务器，使用FTP。你每月支付我5美元，我会给你提供服务器的FTP访问权限，该服务器的磁盘容量为100Gb。你可以将文件上传到那里，然后再下载回来。效果非常好。而第二个选择是亚马逊S3。你也可以通过他们的API上传和下载数据。不同的是，你需要支付每次API请求、每个字节的存储和传输费用，而不是按月收费。你会选择哪个呢？

    显然，你会选择S3。为什么呢？这两者之间的根本区别是什么？关键区别在于它们的服务级别协议(SLAs)：第一个是一个服务器，第二个是一个服务。

    一个FTP服务器提供商保证计算资源（CPU、磁盘、带宽等）的可用性，而S3保证数据的可用性。如果FTP服务器上的磁盘崩溃，将会及时更换，但数据将会丢失。如果磁盘空间满了，您可以订购额外的服务器，但这是您自己的责任不要忘记。如果磁盘空间未被使用，您仍需支付每月5美元。等等。

    AWS S3在市场上是一个突破，十多年前，正是因为这个差异。他们在我们都习惯的网络托管之上添加了一个新的服务层。这个想法仍然相同—数据仍然在云中上传和下载—但是SLA是不同的。我们不再需要担心磁盘溢出、支付过多未使用空间、定期备份、SSH终端等等。他们只是给了我们一个简单的API，并承诺数据在那里且安全。

    现在是2019年，我们在关系数据方面仍然没有同样的解决方案。无论您选择哪个供应商，他们所做的只是提供一个安装了MySQL或PostgreSQL（或他们自己版本）的机器（或集群），并按照运行时间收费。他们仍然提供的只是”好老的FTP”，没有额外的服务层在其之上。

    这就是我期望一个真正的云中关系数据服务级别协议听起来的样子：

    • 自动扩展。不要让我们担心托管数据所需的资源量。只需对较大的数据集收取更多费用，并确保我们的请求在可预测的时间内返回。

    • 按数据付费。让我们为每个SQL请求、每个存储的字节和每个传输的字节付费。需要多少服务器和硬盘来托管所有这些内容——那不应该是我们关心的。

    • 限制的SQL。MySQL或PostgreSQL方言中的大部分命令对大多数项目来说并非必需。只需提供INSERT、SELECT、UPDATE和DELETE，就可以了结。

    • 索引。利用我们所进行的SQL查询的统计数据自动创建索引。

    • 模式版本控制。通过类似于Liquibase的方式，使得可以通过创建新的ALTER TABLE或CREATE TABLE脚本来更新模式，并将其应用于现有数据库。

    • 快照和回滚。使得可以对数据进行快照，应用新的模式版本，如果出现问题，则可以回滚到之前创建的某个快照。

    真的很难实施吗？

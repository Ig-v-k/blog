    SQL as a Service

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    自从2007年亚马逊推出S3以来，我一直在思考这个问题。我甚至尝试过几次实施，但总是在设计阶段就失败了。我听说过一家初创公司也尝试过，但也失败了。我还不确定是否有可能实现，但它肯定可以成为云数据管理市场的畅销产品。等等，你可能会说，Google Cloud SQL、AWS RDS、Microsoft Azure、Heroku PostgreSQL和其他很多产品呢？它们甚至离我所说的还差得远。

    让我举个例子。假设你想在云中存储一段二进制数据。我有两个解决方案供你选择。第一个是带有FTP的托管服务器。你每月支付5美元，我为你提供服务器的FTP访问权限，服务器的磁盘容量为100GB。你可以将文件上传到其中并下载回来。完全正常。我还有第二个选项，即AWS S3。你也可以上传和下载数据，但是通过它们的API。并且你需要为每个API请求、每个托管的字节和每个传输的字节支付费用，而不是每月费用。你会选择哪一个？

    显然，你会选择S3。为什么？这两者之间的根本区别是什么？关键区别在于它们的SLA：第一个带有FTP的是一个服务器，而第二个是一个服务。

    FTP服务器提供商保证计算资源（CPU、磁盘、带宽等）的可用性，而S3保证数据的可用性。如果FTP服务器上的磁盘崩溃，它将及时更换，但数据将丢失。如果磁盘已满，你将能够订购一个额外的服务器，但这是你的责任不要忘记。如果磁盘空间没有被使用，你仍然需要支付每月5美元。等等。

    AWS S3在市场上取得了巨大的突破，十多年前，正是因为这个区别。他们在我们已经习惯的老式网络托管的基础上添加了一个新的服务层。这个想法仍然是一样的——依然是将数据上传和下载到云中——但SLA不同了。我们不再需要担心磁盘溢出、为未使用的空间支付过多费用、定期备份、SSH终端等等。他们只是给了我们一个简单的API，并承诺数据在那里并且是安全的。

    现在是2019年，我们在关系型数据方面仍然没有同样的问题。不管你选择哪个提供商，他们所做的只是给你一台（或一个集群）安装有MySQL或PostgreSQL（或它们自己的版本）的机器，并按小时计费。他们仍然给你提供了没有额外服务层的“老式FTP”。

    这就是我对于在云中真正的关系型数据SLA的期望。

    • 自动扩展。不要让我们担心托管数据所需的资源量。只需针对更大的数据集收费更多，并确保我们的请求能够在可预测的时间内返回。

    • 按数据计费。让我们为每个 SQL 请求、每个存储的字节和每个传输的字节付费。需要多少台服务器和硬盘来托管所有这些——这不是我们要考虑的事情。

    • 受限的SQL。MySQL或PostgreSQL方言中的大部分命令对大多数项目来说并非必需。只需提供INSERT、SELECT、UPDATE和DELETE就可以了，就这样吧。

    • 索引。使用我们所进行的SQL查询的统计数据自动生成它们。

    • 架构版本控制。使得通过类似于Liquibase的方式更新架构成为可能：我们创建一个新的ALTER TABLE或CREATE TABLE脚本，然后将其应用到现有的数据库中。

    • 快照和回滚。使得可以对数据进行快照，应用新的模式版本，如果出现问题，可以回滚到之前的某个快照。

    真的很难实施吗？

    The Joy of Programming

    昨天我正在为关于Java和面向对象编程的一个未来演讲制作幻灯片，却在寻找关于“逻辑的透明性”方面的有力论据时遇到了困难。我本打算说，对于程序员来说，理解他们所做的一切是如何工作的非常重要，即使他们看不到它或从不想看到它。但后来我意识到，也许不是每个人都这样想。也许有些程序员更喜欢对大多数事情保持一片黑暗，只要他们面前的代码“正常工作”就行。因此，我写了这篇博文，想问问你站在哪一边。

    很多年前，我参加了一次工作面试，作为一个应聘者。他们正在寻找一位网络架构师，而我正在和他们的首席技术官交谈。他要求我解释当我们在网络浏览器中输入新的URL并点击“回车”时背后发生了什么。我给他画了一个小图，图中有一个域名注册商、几个DNS服务器、一个负载均衡器、几个HTTP服务器、几个数据库和几个IP中继。我相信他对我的回答印象深刻（尽管当时他们没有录用我），并告诉我大多数网站开发人员并不理解这个图中的大部分内容。根据他的说法，他们只知道HTTP服务器的工作原理，对其他部分几乎不关心。他们中的大多数人甚至不知道HTTP是什么，只要PHP代码能够按照预期工作即可。

    我记得那次面试，并开始向我后来面试的人提出类似的问题，因为我是自己公司的首席技术官，也是其他几个项目的架构师。他的结论得到了证实。事实上，大多数程序员不理解例如DNS的工作原理，也不知道它的作用。而且，他们没有这些信息也感到很好。这意味着他们是糟糕的程序员吗？

    让我给你讲另一个故事，就在几天前发生的事情。我正在解决一个相当复杂的技术问题，试图将我以前没有机会使用过的软件与另一款我之前也没有见过的软件进行集成。这花了我整整三天的时间。现在我回顾这段短暂的旅程，意识到当我开始使用新东西时，我经历了几个特定的阶段，而这些阶段对我来说总是一样的。

    • 热情。在这里，我通常会从新的库的一个简短入门页面开始迅速上手。我会浏览一下文档，忽略其中大部分，因为它们没有任何意义，然后快速地复制粘贴推荐的代码。一切似乎很简单，我期望几分钟之内代码就能运行起来。而通常情况下，它确实能够运行。

    • 猜测。我开始对刚刚复制粘贴的简单代码进行更改，并对其背后的逻辑进行一些假设。我不知道我正在使用的产品是如何设计的，但我需要依靠某些东西。所以，我依靠我能够猜测的东西。

    • 挫折。显然，我大部分的假设都是错误的。我开始谷歌和查看Stack Overflow。我得到的答案（如果有的话）并没有太大帮助，因为整体情况仍然不明确，我能做的就是根据来自各个随机来源的建议来修补我的代码使其工作。但是我仍然处于黑暗之中，整体设计概念仍然不清楚。我仍然希望在不阅读完整的开发者手册手稿的情况下解决所有问题。

    • 抑郁。很快我就意识到我只是一只试图启动飞机的猴子。也许它会飞起来，也许我甚至能成功降落。但我仍然只是一只猴子，这让我感到非常沮丧。我对此毫无乐趣可言。我讨厌自己的愚蠢。我讨厌那些创建这个库并不明显使用的人。我也讨厌我的工作。

    我相信这四个阶段对于我们软件工程师来说都非常普遍。然而，之后发生的事情区分了优秀的程序员和糟糕的程序员。老实说，根据情况不同，我既属于优秀的程序员，也属于糟糕的程序员。当我成为一个糟糕的程序员时，我只是让它工作起来，然后就算了，仍然不知道内部发生了什么，也不知道为什么它以这种方式工作而不是我之前尝试过的那种方式。我的沮丧并没有消失，只会升级。我仍然讨厌自己，但我的软件工作了。我继续解决下一个问题，等待周末的到来。

    当我成为一名优秀的程序员时，我会怎么做？我会更深入地研究问题，学习我所使用的软件，下载其源代码，阅读其文档，直到我理解它的运作原理。然后，我会回到我代码的那一部分，凭借充分的理解来修复它，并结束一天的工作。有时候，我甚至会写一篇博客文章，比如这篇关于Nutch的，这篇关于Liquibase的，或者这篇关于CasperJS的。我的抑郁症完全消失了。我不再讨厌自己，不再讨厌我的工作，也不再讨厌那些“愚蠢”的库的开发者。我甚至通过我的博客文章帮助他们的项目。

    我该如何决定选择哪种方式：像一个糟糕的程序员一样行事，还是像一个优秀的程序员一样行事？你可能认为这取决于我为手头工作拥有的时间/预算，但你会错的。它完全取决于我是否”准备好”生活在沮丧中。

    一段时间以前，我告诉自己我不会再这样做了。我不想成为一个驾驶飞机的“猴子”，我想成为一名飞行员。当然，学习曲线会更长，但结果是…快乐。当我清晰透明地看到我编写的每一行代码的逻辑时，我真的很享受我的工作。我清楚地知道为什么写它们，它们的作用以及如果我改变它们会发生什么。当然，我并不知道所有的细节，但我知道在哪里点击以找到这些细节。这就是使我工作有趣的原因：我的编码逻辑的透明性。

    我真的喜欢负责我代码所做的一切。我喜欢感觉它完全按照我的意愿运行。你可以说我是一个控制狂——也许是吧。但这正是在编程的过去30年中我获得最多乐趣的地方：看到计算机按照我的意愿执行任务。如果某些库试图夺走这种控制权，让我坐在副驾驶座上，我会尽一切努力重新掌控主导权。我想要知道发生了什么，我希望事情按照我的意愿发生。我想要享受我的工作。

    现在，回到我的会议演讲。我打算在我的幻灯片中展示这段代码（这就是使用Spring Framework的正确方式）。

    @Controller
public class HelloController {
    @GetMapping("/hello")
    public String handle(Model model) {
        model.addAttribute("message", "Hello World!");
        return "index";
    }
}

    然后我会问一个问题：你知道谁在创建HelloController类的实例以及该实例如何被使用吗？是谁在调用它的方法，为什么以及何时调用？此外，你知道如何修改这种行为吗？我在会议上的观点是，一个好的框架必须使其用户更容易理解其内部。因为好的程序员想要知道内部情况。另一方面，糟糕的程序员只会满足于像飞行猴子一样驾驶飞机。

    What about you?

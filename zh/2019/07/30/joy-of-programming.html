    The Joy of Programming

    昨天我正在为我未来关于Java和面向对象编程的演讲制作幻灯片，卡在了寻找有说服力的关于“逻辑透明性”的论点上。我本来要说，程序员能够理解他们所做的一切是很重要的，即使他们看不见它或者从不想看见它。但后来我意识到，也许并不是每个人都这样想。也许有些程序员更喜欢对大部分事情保持一无所知的状态，只要他们面前的代码“正常工作”就可以了。因此，我写了这篇博客文章，想问问你是站在哪一边的。

    很多年前，我参加了一次求职面试，应聘一个网络架构师的职位，我当时正在和他们的首席技术官交谈。他让我解释一下，当我们在网页浏览器中输入一个新的URL并点击“Enter”时，幕后发生了什么。我给他画了一个小图，上面有一个域名注册商，几个DNS服务器，一个负载均衡器，几个HTTP服务器，几个数据库，以及一些IP中继设备。我相信他对我的回答印象深刻（尽管他们当时没有雇佣我），并告诉我大多数网络开发人员并不理解这幅图的大部分内容。据他所说，他们只知道HTTP服务器是如何工作的，对其他内容几乎不关心。他们中的大多数人甚至不知道HTTP是什么，只要PHP代码能够按照预期工作就行。

    我记得那次面试，并且后来在我作为自己公司的首席技术官和其他项目的架构师时，开始对我面试的人提出类似的问题。他的结论得到了证实。确实，大多数程序员不理解例如DNS是如何工作以及它的作用。而且，他们对于没有这些信息也感到很满意。这是否意味着他们是糟糕的程序员呢？

    让我给你讲另一个故事，就在几天前发生在我身上。我正在解决一个非常复杂的技术问题，尝试将我以前没有机会使用过的软件与另一款我之前也没有见过的软件进行整合。这花费了我整整三天的时间。现在我回顾这段短暂的旅程，意识到当我开始使用新的东西时，我总是经历几个特定的阶段，这些阶段对我来说总是相同的：

    • 热情。我通常从新库的Quick Start一页开始快速入手。我会浏览文档，忽略其中大部分内容，因为它们毫无意义，然后迅速复制粘贴他们推荐的内容。一切都看起来很简单，我期望代码能在几分钟内运行。通常情况下，它确实如此。

    • 猜测。我开始对刚刚复制粘贴的简单代码进行更改，并对其背后的逻辑进行一些假设。我不知道我使用的产品是如何设计的，但我需要依靠某些东西。因此，我依赖于我可以猜测的东西。

    • 挫败。显然，我大部分的假设都是错误的。我开始谷歌和在Stack Overflow上查找。我得到的答案（如果有的话）并没有帮助太多，因为整体的情况仍然不明确，我能做的最好只是根据我从随机来源得到的建议来修补我的代码，让它能够工作。但我仍然处于黑暗中，整体的设计理念仍然不清晰。我仍然希望能够解决所有问题，而不必阅读完整的开发人员手册。

    • 沮丧。很快我意识到自己只是一只试图驾驶飞机的猴子。也许它会飞起来，也许我甚至能成功降落。但我仍然是一只猴子，这让我感到非常沮丧。我对此毫无乐趣。我讨厌自己愚蠢的样子。我讨厌那些创建这个库并且让它使用起来显得不明显的人。我也讨厌我的工作。

    我相信这四个阶段对于我们所有的软件工程师来说都非常常见。然而，之后发生的事情区分了好的程序员和坏的程序员。说实话，我属于这两个类别，取决于情况。当我成为一个糟糕的程序员时，我会让所有的东西都能运转起来，然后就结束了，却仍然不知道内部发生了什么，也不知道为什么它能以这种方式工作，而之前尝试过的方式不起作用。我的沮丧没有消失，只会升级。我仍然讨厌自己，但是我的软件能运行。我继续解决下一个问题，等待周末的到来。

    当我成为一个好的程序员时，我会更深入地研究问题，学习我使用的软件，下载其源代码，阅读其文档，直到理解发生了什么。然后，我回到我自己的代码部分，基于充分的理解进行修复，然后就结束了。有时候，我甚至会写一篇博文，比如关于Nutch的一篇，关于Liquibase的一篇，或者关于CasperJS的一篇。我的沮丧完全消失了。我不再讨厌自己，不再讨厌我的工作，也不再讨厌那些“愚蠢”的库的开发人员。我甚至通过我的博文帮助他们的项目。

    我如何决定选择哪种方式：像一个糟糕的程序员一样还是像一个好的程序员一样行事？你可能会认为这取决于我为手头的工作拥有的时间/预算，但你是错的。它完全取决于我对陷入沮丧的准备程度。

    一段时间以前，我告诉自己我不再这样做了。我不想再成为一个驾驶飞机的猴子，我想成为一个飞行员。当然，学习曲线会更长，但结果是…快乐。当我清晰、透明地看到我编写的每一行代码的逻辑时，我真的享受我的工作。我确切地知道为什么我要写它们，它们做什么，如果改变它们会发生什么。当然，我不知道所有的细节，我知道在哪里点击以找到这些细节。这就是让我的工作对我来说有趣的地方：我的编码逻辑的透明性。

    我真正享受掌控我代码所做的一切。我喜欢感觉它完全按照我的意愿工作。你可能会说我是一个控制狂——也许是这样。但这是我在过去30年编程中获得最多乐趣的地方：看到计算机按照我的意愿工作。如果一些库试图夺走这一点，把我放在乘客座位上，我会尽一切可能重新回到驾驶座位上。我想知道发生了什么，我希望这按照我的意愿发生。我想享受我的工作。

    现在，回到我的会议演讲。我将在我的幻灯片中展示这段代码（这是使用 Spring Framework 的方式）。

    @Controller
public class HelloController {
    @GetMapping("/hello")
    public String handle(Model model) {
        model.addAttribute("message", "Hello World!");
        return "index";
    }
}

    然后我会问一个问题：你知道谁在创建HelloController类的实例以及该实例如何被使用吗？是谁在调用它的方法，为什么以及何时进行调用？此外，你知道如何修改这个行为吗？我在会议上的观点是，一个好的框架必须使其用户更容易理解其内部。因为优秀的程序员想要知道内部情况。而对于糟糕的程序员来说，他们可以接受成为飞行猴子。

    What about you?

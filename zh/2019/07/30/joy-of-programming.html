    The Joy of Programming

    昨天我在为我未来的有关Java和面向对象编程的演讲准备幻灯片时，遇到了逻辑的透明性这一点找不出令人信服的论据。我原本打算说，对程序员来说，能够理解他们所做的一切工作方式是很重要的，即使他们看不到它或者从不想看到它。但是后来我意识到，也许不是每个人都这样认为。也许有些程序员更愿意对大部分事情保持不了解，只要他们面前的代码“正常工作”即可。因此，我写下了这篇博文，想问问你站在哪一边。

    多年前，我作为应聘者参加了一次工作面试。他们正在寻找一位网络架构师，我与他们的首席技术官进行了交谈。他要求我解释当我们在网络浏览器中输入一个新的URL并点击“Enter”时，在背后会发生什么。我为他画了一个简单的图表，其中包括一个域名注册商、几个DNS服务器、一个负载均衡器、几个HTTP服务器、几个数据库以及几个IP中继。我相信他对我的回答印象深刻（尽管他们当时没有雇佣我），他告诉我大多数网络开发人员对这个图像的大部分内容都不了解。据他所说，他们只知道HTTP服务器的工作原理，对其他方面关心甚少。他们中的大多数人甚至不知道HTTP是什么，只要PHP代码能够实现预期的功能即可。

    我记得那次采访，并且后来在我采访其他人时开始提出类似的问题，因为我是自己公司的首席技术官，还是其他几个项目的架构师。他的结论得到了证实。事实上，大多数程序员不理解例如DNS是如何工作的，也不知道它的作用。而且，他们对于没有这些信息也感到很满意。这是否意味着他们是糟糕的程序员？

    让我给你讲另一个故事，就在几天前发生在我身上的。我正在解决一个相当复杂的技术问题，试图将我之前没有机会使用过的软件与我之前也没有见过的另一款软件进行集成。这花了我整整三天的时间。现在我回顾这段短旅程，意识到当我开始使用新东西时，我总是经历几个特定的阶段：

    热情。在这里，我通常从一个新库的“快速开始”一览开始迅速入门。我浏览文档，忽略大部分内容，因为它们没有任何意义，并迅速复制粘贴他们推荐的内容。一切似乎很简单，我期望代码几分钟内就能运行。通常也确实如此。

    猜测。我开始对刚刚复制粘贴的简单代码进行修改，并对其背后的逻辑进行一些假设。我不知道我正在使用的产品是如何设计的，但我需要依靠一些东西。因此，我依靠我可以猜测的内容。

    挫折。显然，我的大部分假设都是错误的。我开始搜索和查找stackoverflow。我得到的答案（如果有的话）并没有太大帮助，因为整体情况还是不清楚的，我所能做的就是根据来自随机来源的建议来修补我的代码，使其能够工作。但我仍然一直处于黑暗之中，整体设计概念仍然不清晰。而且我仍然希望在不阅读完整的开发者手册的情况下解决所有问题。

    抑郁。很快我意识到，我只是一只试图启动飞机的猴子。也许它会飞起来，也许我甚至能成功着陆。但我仍然是一只猴子，这让人非常沮丧。我对此毫无乐趣。我讨厌自己的愚蠢。我讨厌那些创建这个库的人，让使用变得不明显。我也讨厌我的工作。

    我相信这四个阶段对于我们所有的软件工程师来说都是非常常见的。然而，之后发生的事情区分了优秀的程序员和糟糕的程序员。说实话，我属于两种情况，具体取决于情况。当我成为一个糟糕的程序员时，我只是让一切都能运行起来，然后结束这一天，但是我对内部发生的事情以及为什么它以这种方式工作而不是之前尝试的方式不了解。我的沮丧没有消失。它只会升级。我仍然讨厌自己，但我的软件能够工作。我继续解决下一个问题，等待周末的到来。

    当我是一个优秀的程序员时，我会做些什么呢？我会深入研究问题，学习我使用的软件，下载其源代码，阅读其文档，直到我理解其中的原理。然后，我会回到我的代码部分，以充分理解为基础进行修复，并且结束当天的工作。有时候，我甚至会写一篇关于此的博客文章，比如关于Nutch的一篇，关于Liquibase的一篇，或者关于CasperJS的一篇。我的抑郁感完全消失了。我不再讨厌自己，不再讨厌我的工作，也不再讨厌那些“愚蠢”的库的开发者。我甚至通过我的博客文章来帮助他们的项目。

    我该如何决定选择哪一种方式：像一个糟糕的程序员还是像一个优秀的程序员？你可能会认为这取决于我为手头的工作所拥有的时间/预算，但你会错的。这完全取决于我对生活在沮丧中的准备程度。

    一段时间前，我告诉自己我不会再这样做了。我不想成为一名驾驶飞机的猴子，我想成为一名飞行员。当然，学习曲线会更长，但结果是…快乐。当我清楚透明地看到我编写的每一行代码的逻辑时，我真的很喜欢我的工作。我确切地知道为什么我要写它们，它们做什么，以及如果我改变它们会发生什么。当然，我并不知道所有的细节，但我知道在哪里点击以找到这些细节。这就是使我的工作对我来说有趣的地方：我的编码逻辑的透明度。

    我真的喜欢负责我代码的一切。我喜欢感觉它完全按照我的意愿工作。你可以说我是一个控制狂——也许是这样。但这正是在过去30年的编程中带给我最多乐趣的地方：看到计算机按照我的意愿工作。如果某些库试图夺走这种乐趣，让我坐在副驾驶座上，我会尽一切努力重新回到驾驶座上。我想知道发生了什么，我希望事情按照我的意愿发生。我想享受我的工作。

    现在，回到我的会议演讲。我将在我的幻灯片中展示这段代码（这是使用 Spring Framework 的正确方式）。

    @Controller
public class HelloController {
    @GetMapping("/hello")
    public String handle(Model model) {
        model.addAttribute("message", "Hello World!");
        return "index";
    }
}

    然后我会问一个问题：你知道谁在创建 HelloController 类的实例以及如何使用该实例吗？谁在调用它的方法，为什么以及何时调用？而且，你知道如何修改这种行为吗？我在会议上的观点是，一个好的框架必须使其用户更容易理解其内部。因为优秀的程序员确实想要知道内部是什么样的。另一方面，糟糕的程序员对于成为飞机驾驶猴子是可以接受的。

    What about you?

    How Much Cohesion Is Enough?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    哪一个更好：books.del(42) 还是 books.book(42).del()？我两种方法都用过，很少能说哪一个更好。第一个选项更短，而第二个选项更面向对象。第一个选项更难扩展，而第二个选项更冗长，需要更多的代码行数，这意味着出错的机会更大。你更喜欢哪一个？

    当然，两个都可以使用，但问题是哪种设计更面向对象？这似乎取决于对象books的大小。如果它很小，就没有必要先获取这本书，我们可以直接在那里删除它。

    interface Books {
  void create(String title);
  void delete(int id);
  void rename(int id, String title);
}

    然而，如果它更大，最好先获取book。

    interface Book {
  void delete(); // Here!
  void rename(String title);
}
interface Books {
  void create(String title);
  int total();
  Iterable<Book> all();
  Book all(String query);
  Book book(int id); // Here!
}

    这两者之间有明确的硬性界限吗？有严格的规定吗？或者这个模棱两可的问题“这个班级已经足够大了吗？”是否应该每次都通过投票来回答？

    让我们尝试给出两个极端的答案：1）永远不够大，和2）永远足够大。

    • 如果一个类永远不准备将其任何部分提取出来并转换为新对象，那么我们最终会得到一个非常庞大的类，其中包含许多属性、方法以及几乎所有方法中的长参数列表。

    • 如果一个类总是足够好以进行提取，我们将得到许多小的类，几乎没有参数的方法，以及… 更好的面向对象设计。

    共同的共同点是凝聚性。

    高度凝聚的类包括彼此相关的属性和方法，而非凝聚的类包括其开发人员决定添加的任何元素，即使有些元素可能并不真正属于一起。第一个答案将给我们一个凝聚度非常低的类，而第二个答案将产生大量高度凝聚的小类。

    因此，第二个选项更好？是的，是的。更小的类，更高的凝聚度，…但也更容易分散注意力和功能到太多的地方。”一个对象中的所有方法”是一种更受欢迎的设计，即使它的凝聚度较低，正是因为它更容易创建：只需把一切放在一个地方并称之为完成。当然，以后会出现可维护性问题。

    底线是，在这种情况下，没有确切的正确和错误设计之间的区别。我们只需尽力保持类高度凝聚，通过减少每个类中的方法数量。如果只有几个方法，就不需要提取Book，但一旦方法数量增加，Book就是定义新实体的理想候选。

    多少方法是可以接受的？

    没有人知道，但我建议一旦你看到超过七个方法或超过四个属性，就要质疑你的类的凝聚度。此外，当你的任何方法（构造函数除外）接受超过两个参数时，开始考虑重构。

    Which design is better? #elegantobjects

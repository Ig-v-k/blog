    How Much Cohesion Is Enough?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    哪个更好：books.del(42) 还是 books.book(42).del()？我两种都用过，很少能分辨出哪个更好。第一种选项更短，而第二种更面向对象。第一种选项更难扩展，而第二种更冗长，需要更多的代码行数，这意味着出错的可能性更高。你更喜欢哪个？

    当然，两种方式都可以，但问题是哪种设计更符合面向对象的原则？这似乎取决于books对象的大小。如果它很小，就没有必要先获取这本书，我们可以直接在那里删除它。

    interface Books {
  void create(String title);
  void delete(int id);
  void rename(int id, String title);
}

    然而，如果它更大，最好先获取book。

    interface Book {
  void delete(); // Here!
  void rename(String title);
}
interface Books {
  void create(String title);
  int total();
  Iterable<Book> all();
  Book all(String query);
  Book book(int id); // Here!
}

    这两者之间有明确的硬性界线吗？是否有一条严格的规定，或者也许这个模糊的问题“这个班级已经足够大了吗？”应该每次通过投票来回答？

    让我们试着给出两个极端的答案：1）永远不够大，和2）永远足够大。

    如果一个类永远没有准备好将其任何部分提取出来并转换为新对象，我们最终会得到一个非常庞大的类，其中包含许多属性、方法，几乎每个方法都有长长的参数列表。

    如果一个类总是足够好用于提取，我们最终将得到许多小类、几乎没有参数的方法，以及…更好的面向对象设计。

    共同点是凝聚力。

    高度内聚的类包括彼此相关的属性和方法，而非内聚的类则包括其开发人员决定添加的任何元素，尽管有些元素可能并不真正属于一起。第一个答案将给我们一个具有极低内聚度的类，而第二个答案将产生大量高度内聚的小类。

    那么，第二个选项更好吗？是的，确实如此。较小的班级，更高的凝聚力……但也会有更多失去焦点和将功能分散到太多地方的机会。”一个对象中的所有方法”是一种更受欢迎的设计，尽管它的内聚性较差，但由于更容易创建：只需将所有东西放在一个地方就可以了。当然，以后会出现可维护性问题。

    总之，在这种情况下，没有确切的区分正确设计和错误设计。我们只需尽力保持类的高内聚性，通过减少每个类中的方法数量来实现。如果只有几个方法，就不需要提取“Book”，但一旦方法的数量增加，提取“Book”就是定义新实体的完美选择。

    有多少种方法可以？

    没人知道，但我建议你在看到超过七个方法或超过四个属性时，质疑你的类的内聚性。另外，当你的任何方法（除了构造函数）接受超过两个参数时，开始考虑重构。

    Which design is better? #elegantobjects

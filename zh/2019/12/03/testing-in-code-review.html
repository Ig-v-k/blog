    Does Code Review Involve Testing?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    当你审查来自他人的拉取/合并请求时，你会检出分支并运行构建吗？我通常不会，但有些人会。他们明显的理由是：运行构建，甚至手动测试产品，有助于发现更重要的错误。仅仅查看源代码可能无法揭示最近在HTML/CSS中引入的所有视觉缺陷，例如。最好检出分支，启动Apache，在Chrome中打开站点，看看有什么问题。然后，截屏，附加到拉取请求中，并将其退回给作者。但我不同意这个观点，下面是我的理由。

    讨论并不新鲜，请查看这个和这个在SO上的帖子。然而，似乎那里的所有答案都忽略了关键点。

    众所周知，在任何软件项目中都存在两个相互冲突的角色：构建者和析构者，也被称为程序员和测试人员。程序员添加新功能并修复错误。他们的成果是创造的功能量：越多越好。而测试人员则会损坏产品并报告错误，尽其所能证明产品尚未准备好交付给客户。在某个时间点上，团队（或管理层）决定战斗结束，产品可以交付。

    由于这种基本的冲突，可以达到“质量”。

    程序员在代码通过合并流水线后完成他们的任务：更改在他们的笔记本电脑上完成，单元测试在本地通过，静态分析没有任何问题，构建是干净的，并且分支被合并到主干。这是程序员停下来领取奖金的地方。

    测试人员在成功在部署到暂存或生产环境的产品中发现一个新的缺陷后，便完成了他们的任务：缺陷被找到、报告并且被项目方接受。这是测试人员停止工作并获得奖金的时候。

    这是显而易见的。如果不是这样的话，你可能想要阅读 Glenford Myers 的《软件测试艺术》或 Yegor Bugayenko 的《Code Ahead》。你也可以观看这个视频。

    现在，在这个冲突中，代码评审员的位置在哪里呢？

    我认为代码审查是合并流水线的一部分，与单元测试、静态分析器、代码检查工具、覆盖控制工具、变异测试工具和项目中可能需要的其他工具一起，目的是使程序员的生活更加困难，提高源代码的质量。合并流水线的目标是保护代码仓库免受程序员的影响。

    代码审查员与代码检查工具或静态分析器没有什么不同：它们会阻止有问题的代码分支进入主干。当他们阻止时，他们会给出理由。他们的工作（与代码检查工具类似）是在找到分支中的问题并解释清楚时完成的。当代码检查工具拒绝拉取请求时，它会明确指出问题所在的行以及如何修复。这就是一个优秀的代码审查员必须做的：找到有问题的行，解释问题，并提出解决方案。

    这是代码审查人员必须得到报酬的原因：完成的审查。

    什么是一个完整的审查？“一切都好”听起来像是一个完整的审查吗？对于一个代码检查工具来说是；对于一个代码审查人员来说不是。这个更好一些：“我发现了三个问题，解释了它们，并且它们要么被讨论过要么被修复了。”一个代码审查人员的工作描述可能会是这样的：找到三个最关键的问题，解释它们，并确保它们要么被修复，要么正确地进行了讨论。

    评审员如何发现这三个问题-这由他们决定。他们可以对代码进行视觉检查或运行构建。然而，只要他们成功发现问题并确保代码作者理解这些问题并修复它们或解释为何无法修复（或不是问题），他们仍然会得到报酬。对代码进行视觉检查很快，而检出分支并运行构建则是一个更耗时的操作。此外，在本地运行构建后发现的错误很难在代码审查格式中解释。与分支作者的讨论将需要更长的时间，这意味着完成代码审查需要更多时间，这也意味着代码评审人员的效率较低。

    我的观点是，聪明的代码审阅者不会这样做，因为这样做是低效的。请注意，他们不是因为不在乎，而是因为他们知道更好的为项目做贡献的方式。面对现实吧，当面前的分支通过了所有像代码检查和单元测试这样的自动化检查，但仍然存在我们只能通过执行代码才能复现的一些错误时，我们的自动化测试就有问题了。一个高效、负责任且贪婪的代码审阅者不会向代码作者解释出了什么问题。相反，他们会创建一个新的错误，责怪合并流程太弱。当然，这个新错误会得到奖励。

    因此，作为代码审查人员，您可以在本地与该分支一起工作，对其进行测试，并将您的发现报告给作者。但这将违背您的个人利益，并对项目没有利益。相反，您应该向项目投诉自动化测试的质量低，并将审查暂停。当投诉得到解决，测试变得更强大时，您可以重新开始审查，而审查会被合并流程拒绝，而不是您拒绝。

    在这种情况下，每个人都是赢家：流程变得更强大，您因报告的错误获得额外的奖励，并且审核被以非常具体的“可复现”原因拒绝。

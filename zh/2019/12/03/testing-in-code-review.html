    Does Code Review Involve Testing?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    当你审查别人的拉取/合并请求时，你会检出分支并运行构建吗？我通常不这样做，但有些人会这样做。他们的显而易见的理由是：运行构建，甚至手动测试产品，有助于找到更重要的错误。仅仅查看源代码可能无法揭示最近引入的HTML/CSS的所有可见缺陷，例如。最好是检出分支，启动Apache，在Chrome中打开站点，看看有什么问题。然后，截图，将其附加到拉取请求中，并将其退回给作者。但是我不同意这一点，以下是我的原因。

    这个讨论并不新鲜，在SO上看看这个和这个。然而，似乎所有的答案都没有抓住关键点。

    众所周知，任何软件项目中都存在两个相互冲突的角色：构造者和破坏者，也称为程序员和测试人员。程序员添加新功能和修复错误。他们的成果是创建的功能数量：越多越好。而测试人员会破坏产品并报告错误，尽一切可能证明产品还不准备好交付给客户。在某个时间点，团队（或管理层）决定战斗结束，产品可以交付。

    由于这种本质上的冲突，质量得以实现。

    当代码通过合并流水线时，程序员完成他们的工作：更改在他们的笔记本电脑上完成，单元测试在本地通过，静态分析没有投诉，构建干净，分支合并到主干。这是程序员停下来并获得奖金的地方。

    当测试人员在部署到暂存或生产环境的产品中找到一个新的缺陷时，他们完成了他们的工作：找到错误，报告，并被项目接受。这是测试人员停下来并获得奖金的地方。

    这是显而易见的。如果不是，请阅读Glenford Myers的《软件测试的艺术》或Yegor Bugayenko的《Code Ahead》。您还可以观看这个视频。

    现在，一个代码审阅者在这个冲突中的位置在哪里？

    我认为代码审查是合并流水线的一部分，与单元测试、静态分析器、代码检查工具、突变测试器以及项目可能想要放入其中的其他所有内容一起，以使程序员的生活更加困难，源代码的质量更高。合并流水线的目标是保护代码仓库免受程序员的影响。

    代码审阅者与代码检查器或静态分析器没有区别：它们阻止有错误代码的分支进入主干。当他们阻止时，他们给出理由。他们的工作完成（类似于代码检查器）是在他们成功找到问题并对其进行解释时。当代码检查器拒绝拉取请求时，会准确指出问题所在，哪一行，以及如何修复。这就是一个优秀的代码审阅者必须做的：找到有问题的行，解释问题，并提出解决方案。

    这就是代码审阅者应该得到报酬的原因：完成的审查。

    什么是完成的审查？”一切正常”听起来像是完成的审查吗？对于代码检查器来说是的，对于代码审阅者来说不是。更好的说法是：”我找到了三个问题，解释了它们，它们要么得到了讨论，要么被修复了。”这就是代码审阅者的工作描述：找到三个最重要的问题，解释它们，并确保它们要么被修复，要么得到正确的讨论。

    代码审阅者如何找到这三个问题，这取决于他们。他们可以对代码进行可视检查或运行构建。然而，当他们成功找到问题并确保代码的作者理解这些问题并修复它们或解释为什么无法修复（或不是问题）时，他们仍然会得到报酬。对代码进行可视检查速度较快，而检出分支并运行构建则需要更长时间。此外，运行构建后发现的错误在代码审查格式中很难解释。与分支的作者进行的讨论将更长，这意味着完成代码审阅所需的时间更长，这意味着代码审阅者的效率较低。

    我的观点是聪明的代码审阅者不会这样做，因为这样做效率低下。请注意，他们这样做不是因为他们不在乎，而是因为他们知道更好的为项目做贡献的方法。面对事实吧，当面前的分支通过像代码检查器和单元测试这样的自动化检查时，仍然存在一些我们只能通过执行代码才能重现的错误，我们的自动化测试就有问题了。一个有效、负责任而贪婪的代码审阅者不会向代码的作者解释出了什么问题。相反，将创建一个新的错误，责怪合并流水线太脆弱。当然，这个新的错误会得到奖励。

    因此，作为代码审阅者，你可以在本地处理分支，对其进行测试，并向作者报告你的发现。但这对你个人的利益来说不是有利的，对项目也没有好处。相反，你应该向项目抱怨自动化测试的质量低，并将审阅放在等待状态。当投诉得到解决时，测试变得更强大，你可以继续审阅，但该审阅将被合并流水线拒绝，而不是你拒绝。

    在这种情况下，每个人都会受益：流水线变得更强大，你因报告的错误而获得额外的奖金，审阅被拒绝，并有一个非常具体的可重现的原因。

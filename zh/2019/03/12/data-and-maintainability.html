    How Data Visibility Hurts Maintainability

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我一直在写关于面向对象编程及其陷阱的大量内容，声称我们习惯于的大多数设计模式和“良好实践”实际上是[错误和有害的]，以至于我完全忘记了解释更大的问题。有人在博客文章中不久前问我关于“裸露”数据的问题：我们要解决的问题是什么，如果我们不足够封装我们的数据，为什么可维护性会受到影响？这就是答案。

    我重新阅读了优雅对象，第1卷的前几页，这本书完全致力于现代面向对象编程的问题缺陷，发现可维护性直接在其中被提及：“我写作的主要目标是提高您的代码的可维护性。”然后它还解释了可维护性是“我理解您的代码所需的时间。”你可能对此表示异议，但问题仍然存在：缺乏“真正”的对象导向和适当的封装如何影响可读性？

    我不知道确切的答案，但这是我的个人版本，源自我多年的编码和痛苦经历：更小的作用域 是更好的可维护性的关键成功因素，缺乏封装会导致更大的作用域。你知道什么是可见性作用域吗？让我们以这段C代码为例：

    void print() {
  for (int i = 0; i < 10; ++i) {
    printf("%d * 2 = %d", i, i * 2);
  }
  for (int i = 0; i < 10; ++i) {
    printf("%d * 3 = %d", i, i * 3);
  }
}

    有两个变量i，在两个不同的for循环中可见，在两个不同的可见范围内。每个范围的大小为三行代码。这段代码如何，完全相同：

    void print() {
  int i = 0;
  while (++i < 10) {
    printf("%d * 2 = %d", i, i * 2);
  }
  i -= 10;
  while (++i < 10) {
    printf("%d * 3 = %d", i, i * 3);
  }
}

    现在，变量i的可见范围是十行代码。这段代码的功能和第一个片段一样好，但可维护性较低，因为为了理解正在发生的事情以及如何修改它，我需要更多的时间。我需要阅读一个包含10行代码的代码块，而不是两个包含3行代码的块。在开始进行任何修改之前，我需要理解整个print()方法。我需要理解这个可怜的i的生命周期算法，以及为什么它会减少10而不是重新赋值为零——这是前一位程序员留给我的惊喜。也许他们不知道存在for循环？

    很明显，第一个片段比第二个更好。问题是我们如何“促使”程序员编写类似第一个片段那样的代码，并使每个变量和函数的范围更小？我们可以教导他们，为他们编写书籍，说服他们，培训他们，甚至可能因为范围过大和可读性较差的代码而惩罚他们，但如果编程语言本身不能防止这些大范围的发生，那么没有什么真正有用。最好是“发明”一种编程语言或整个编程范式，使其更难或不可能扩大范围。

    例如，在C/C++、Python、Ruby和许多其他现代编程语言中，你可以想象的最大范围是全局（global）范围，例如在这里：

    int i = 0;
void print() {
  while (++i < 10) {
    printf("%d * 2 = %d", i, i * 2);
  }
}

    现在变量 i 不仅在函数 print() 内可见，而且在我们开发的应用程序的所有其他地方都可见。i 的可见范围是整个代码库的大小。毋庸置疑，这使函数 print() 的代码非常难读。当 print() 开始执行时，我根本不知道可以期望 i 的值—我必须浏览整个代码库才能找到它。如果是一个小应用程序，也许我可以应付，但如果是一个大型软件，我会遇到大麻烦。那么，我们创建一种编程语言，它不允许全局变量怎么样？这将解决这个问题。程序员将没有技术能力来定义它们，它们的作用域必然会更小。

    我相信物体是为了这个目的而发明的：强制程序员将其可见范围保持较小。实际上，函数和子例程也是为此而发明的，但对于“强制”部分的强调较弱，因为它们可以与尚未分解的代码部分共存。相反，对象应该是对象世界的一流公民，只与对象进行通信。

    They were.

    但是接着C++出现了，并毁了一切。

    让我们尝试向我们的 C 代码片段中引入一个对象，以大多数一些 C++ 程序员的方式来做。

    class Idx {
public:
  int get() { return v; }
  void add(int a) { v += a; }
private:
  int v = 0;
}

    And then:

    void print() {
  Idx i();
  while (i.get() < 10) {
    printf("%d * 2 = %d", i.get(), i.get() * 2);
    i.add(1);
  }
  i.add(-10);
  while (i.get() < 10) {
    printf("%d * 3 = %d", i.get(), i.get() * 3);
    i.add(1);
  }
}

    发生了什么变化？没什么变化。我们不再有一个简单的“标量”变量i，而是有了一个“对象”，它内部存储了一个整数值，并提供了一些方法来访问和修改它。这有助于我们最小化范围吗？一点也不。而且，print()的长度现在甚至更长几行。但现在我们有了一个对象，可以称我们的代码为面向对象的！

    这是大多数“对象”如今的用法，主要得益于C++：它们只是数据持有者，而数据的真正使用者仍然在它们的外部。对象Idx对其持有的数据的真正用途一无所知。它不知道它的v被用作步长计数器，并且在打印文本之前会被乘以某个值。Idx只是一个数据持有者，真正的逻辑在它的外部。

    可维护性问题未解决，范围未减小，代码复杂性也未减少。相反，它增加了，因为现在为了理解print()的工作原理，我必须知道Idx的内部情况。在这个特定的例子中，对象范式承诺解决一部分问题，让我不再担心它，但实际上它只让问题变得更大，给我带来了两个问题：print()和Idx。

    为什么感谢C++？因为C++在C过程式编程思想的基础上增加了面向对象的特性，从不想要禁止其中的一些特性，而是强制程序员按照预期的方式编写对象：将所有需要的内容封装在黑盒中，从外部甚至无法知道内部是什么！C++甚至没有试图将编程范式从过程和变量转变为对象和方法。C++的创建者Bjarne Stroustrup只是为程序员提供了方法和类，并说：“使用它们，它们比变量更方便，…或者有时不是，…我不知道”（这可能不是他的原话，但我相信这是他想表达的意思）。阅读他的书，你会看到有多少页专门讲述了面向对象的哲学，有多少页专门讲述了运算符和语句的技术细节。

    一个合适的面向对象的解决方案看起来会有所不同，并且会涉及真正的封装，其中数据从不“逃离”其所有者的边界。首先，这是我如何设计Idx… 好吧，我首先会给它改个名字，叫做Line。

    class Line {
public:
  Line(int m): mul(m) {};
  void print() {
    printf("%d * %d = %d", v, m, v * m);
  }
  bool next() {
    bool n = true;
    if (v < 10) {
      ++v;
    } else {
      n = false;
    }
    return n;
  }
private:
  int mul;
  int v = 0;
}

    现在这里是print()的代码。

    void print() {
  Line a(2);
  while (a.next()) { a.print(); }
  Line b(3);
  while (b.next()) { b.print(); }
}

    正如你所看到的，print()无法访问Line的内部数据。print()所能做的就是要求Line向前移动并打印自己。Line内部实现这种逻辑的方式无人知晓，也无人关心。由于Line中没有任何getters，我们无法从中获取数据。

    由于我们无法取得数据，所以无法在 print() 中构建任何逻辑。我们根本没有任何可用的东西，没有数据，没有整数，没有数字。我们只能处理对象，这些对象不信任我们的内部。我们只能礼貌地要求它们为我们做些事情。print() 的作用范围现在非常小，并且与 Line 的内部非常好地隔离开来。适当的封装帮助我们实现了这一点：通过不暴露 Line 的内部，我们使任何人进入其自己的范围变得不可能。print() 无法对 Line 封装的数据进行任何操作。

    因此，数据越可见和可访问，可维护性越低。

    面向对象编程的基本概念是基于封装的，这并不仅仅意味着限制从外部修改对象属性的能力，也包括禁止任何人读取这些属性。

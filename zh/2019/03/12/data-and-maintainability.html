    How Data Visibility Hurts Maintainability

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我一直在写关于面向对象编程及其缺陷的文章，声称我们习惯于的大多数设计模式和“良好实践”实际上是错误和有害的，以至于完全忘记了解释更大的问题。有人在博客文章“裸露”数据中不久前问我：我们正在解决什么问题，如果我们不足够“封装”数据，为什么可维护性会受到影响？以下是答案。

    我重新阅读了《优雅对象，卷1》的前几页，这本书完全致力于解决现代面向对象编程缺陷的问题，并发现可维护性直接在其中提到：“我在写作时试图实现的主要目标是提高您的代码的可维护性”，然后还解释了可维护性是“理解您的代码所需的时间”。你可以对此进行争论，但问题依然存在：缺乏“真正”的面向对象和适当的封装如何影响可读性？

    我不知道确切的答案，但这是我自己的版本，源自我多年的编码和痛苦经历：较小的范围是更好可维护性的关键成功因素，而缺乏封装会导致更大的范围。你知道可见性范围是什么意思，对吗？让我们以这段C代码为例：

    void print() {
  for (int i = 0; i < 10; ++i) {
    printf("%d * 2 = %d", i, i * 2);
  }
  for (int i = 0; i < 10; ++i) {
    printf("%d * 3 = %d", i, i * 3);
  }
}

    有两个变量i，在两个不同的for循环中可见，两个不同的可见范围。每个范围的大小为三行代码。这段代码如何，完全相同：

    void print() {
  int i = 0;
  while (++i < 10) {
    printf("%d * 2 = %d", i, i * 2);
  }
  i -= 10;
  while (++i < 10) {
    printf("%d * 3 = %d", i, i * 3);
  }
}

    现在，i 的可见范围是十行代码。代码与第一个片段一样有效，但可维护性较低，因为为了理解发生的事情以及如何修改它，我需要更多的时间。我需要阅读一个10行的代码块，而不是两个3行的代码块。在开始进行任何修改之前，我需要理解整个 print() 方法。我需要理解那个可怜的 i 的生命周期算法，以及为什么它被递减了10而不是被重新赋值为零—这是之前的程序员留给我的惊喜。也许他们不知道 for 循环的存在？

    很明显，第一个代码段比第二个要好。问题是我们如何让程序员编写像第一个代码段那样的代码，并缩小每个变量和函数的作用域？我们可以教导他们，为他们撰写书籍，说服他们，培训他们，甚至可能因为作用域较大和代码可读性较差而对他们进行惩罚，但如果编程语言本身无法防止这些大范围的发生，那么什么都不会真正有所帮助。最好发明一种编程语言或整个编程范式，使其更难或不可能扩大作用域。

    例如，在C/C++、Python、Ruby和许多其他现代语言中，你可以想象的最大作用域是全局作用域，例如在这里：

    int i = 0;
void print() {
  while (++i < 10) {
    printf("%d * 2 = %d", i, i * 2);
  }
}

    现在变量i不仅在函数print()内可见，而且在我们开发的应用程序的许多其他地方都可见。i的可见范围是整个代码库的大小。不用说，这使得函数print()的代码非常难读。当执行print()时，我根本无法知道i的预期值—我必须查看整个代码库才能找到它。如果是一个小应用程序，也许我能应付，但如果是一个大型软件，我将遇到很大的麻烦。那么，我们来创建一种编程语言，它将不允许全局变量怎么样？这将解决这个问题。程序员将无法定义全局变量，并且它们的作用域将不可避免地更小。

    我相信对象的发明正是为了做到这一点：强制程序员将其可见范围保持较小。实际上，函数和子程序也是为此而发明的，但对于”强制”这一部分强调较弱，因为它们可以与尚未分解的代码部分共存。相反，对象被认为是对象世界的一等公民，只与对象进行通信。

    They were.

    但是接下来出现了C++，搞砸了一切。

    让我们尝试向我们的 C 代码片段引入一个对象，就像一些 C++ 程序员所做的那样：

    class Idx {
public:
  int get() { return v; }
  void add(int a) { v += a; }
private:
  int v = 0;
}

    And then:

    void print() {
  Idx i();
  while (i.get() < 10) {
    printf("%d * 2 = %d", i.get(), i.get() * 2);
    i.add(1);
  }
  i.add(-10);
  while (i.get() < 10) {
    printf("%d * 3 = %d", i.get(), i.get() * 3);
    i.add(1);
  }
}

    发生了什么变化？并不多。我们不再使用简单的“标量”变量i，而是使用一个“对象”，它内部存储一个整数值，并提供一些方法来访问和修改它。这有助于我们减少范围吗？完全没有。此外，print()函数的长度现在甚至长了几行。但是现在我们有了一个对象，可以称我们的代码为面向对象的！

    这就是大多数“对象”如今的使用方式，主要得益于C++：它们只是数据容器，而数据的真正使用者仍然在它们的外部。对象Idx对其持有的数据的真实用途一无所知。它不知道自己的v被用作步长计数器，并在打印一些文本之前被乘以某个值。Idx只是一个数据容器，而真正的逻辑则在它的外部。

    可维护性问题未得到解决，范围未缩小，代码复杂性未减少。相反，它增加了，因为现在为了理解 print() 如何工作，我必须知道 Idx 内部是什么。在这个特定的例子中，面向对象范式承诺解决一部分问题，让我不再担心，但实际上它只是让问题变得更大，给我带来了两个问题：print() 和 Idx。

    为什么要感谢C++？因为C++在C过程式编程思想的基础上增加了面向对象的特性，并且从未考虑禁止其中的一些特性，以强制程序员按照预期的方式编写对象：作为封装了所需一切的黑箱，从外部甚至不允许任何人了解内部是什么！C++甚至没有试图从过程和变量范式转变为对象和方法。C++的创造者Bjarne Stroustrup只是为程序员提供了方法和类，并说：“使用它们，它们比变量更方便，…或者有时候不是，…我不知道”（虽然我不确定这是他的原话，但我相信它非常接近他的想法）。阅读他的书，你会看到有多少页面专门讲述对象导向的哲学，以及有多少页面讲述运算符和语句的技术细节。

    一个合适的面向对象的解决方案看起来会有所不同，并且会涉及真正的封装，其中数据永远不会“逃离”其所有者的边界。首先，我将展示如何设计 Idx… 嗯，我首先会给它改个名字，叫做 Line。

    class Line {
public:
  Line(int m): mul(m) {};
  void print() {
    printf("%d * %d = %d", v, m, v * m);
  }
  bool next() {
    bool n = true;
    if (v < 10) {
      ++v;
    } else {
      n = false;
    }
    return n;
  }
private:
  int mul;
  int v = 0;
}

    现在这里是print()的代码。

    void print() {
  Line a(2);
  while (a.next()) { a.print(); }
  Line b(3);
  while (b.next()) { b.print(); }
}

    正如你所看到的，print() 无法访问 Line 的内部数据。print() 只能请求 Line 前进并打印自己。关于 Line 内部数据的实现逻辑，没有人知道，也没有人想知道。由于 Line 没有任何 getter 方法，我们无法从中获取数据。

    由于我们无法获取数据，无法在print()中构建任何逻辑。我们根本没有任何可用的数据、整数或数字。我们只能处理对象，而这些对象并不信任我们的内部。我们只能礼貌地要求它们为我们做些什么。print()的作用域现在非常小，并且与Line的内部非常隔离。适当的封装帮助我们实现了这一点：通过不暴露Line的内部，我们使任何人都无法进入其自身范围。print()无法对Line封装的数据做任何操作。

    因此，数据的可见性和可访问性越高，维护性就越低。

    面向对象编程的核心理念基于封装，这并不仅仅意味着限制外部修改对象属性的能力，还禁止任何人读取这些属性。

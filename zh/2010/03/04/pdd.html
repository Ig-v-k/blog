    Puzzle Driven Development

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    PDD，即拼图驱动开发，是一种将编程任务细分并使其能够并行实现的方法。PDD方法广泛应用于XDSD方法论中。该方法正在等待美国专利商标局（USPTO）的专利申请（申请号12/840,306）。

    让我们通过一个例子来回顾一下这个方法。比如说，你是一个程序员，被要求设计和实现一个Java类。这是正式的任务描述：“DBWriter类必须扩展java.io.Writer抽象类，并将所有传入的数据保存到数据库中。”

    你有一个小时来完成这个任务。显然，对你来说一个小时是不够的，因为问题要比规定的时间更复杂，需要更多的工作来完成。此外，还有很多未知因素。

    • 我们需要保存哪些信息，以及以什么格式保存？

    • DB schema 是什么？它是一个 SQL 还是 NoSQL 数据库？

    • 如何连接到数据库？JDBC？JPA？DAO？

    • 如何处理异常？

    让我们在试图以最高抽象层次解决问题时牢记所有这些未知因素。当然，我们从一项测试开始。

    import org.junit.*;
import static org.mockito.Mockito.*;
public class DBWriterTest {
  @Test
  void testSavesDataIntoDatabase() throws Exception {
    DataBridge mapper = mock(DataBridge.class);
    Writer writer = new DBWriter(mapper);
    try {
      writer.write("hello, world!");
    } finally {
      writer.close();
    }
    verify(mapper).insert("hello, world!");
  }
}

    在上面的测试中，我们定义了类的预期行为。测试无法编译，因为缺少两个类：DataBridge和DBWriter。让我们先实现这个桥接。

    import java.io.IOException;
public interface DataBridge {
  void insert(String text) throws IOException;
}

    接下来，写作本身：

    import java.io.IOException;
import java.io.Writer;
import java.utils.Arrays;
public class DBWriter implements Writer {
  private DataBridge bridge;
  public DBWriter(DataBridge brdg) {
    this.bridge = brdg;
  }
  @Override
  void flush() throws IOException {
  }
  @Override
  void close() throws IOException {
  }
  @Override
  void write(char[] cbuf, int off, int len) throws IOException {
    String data = new String(Arrays.copyOfRange(cbuf, off, off + len));
    this.bridge.insert(data);
  }
}

    使用上述代码，我们解决了这个问题。在例子中，我们成功地设计、实现并测试了所需的DBWriter类。随后，这个类现在可以立即被其他类“按原样”使用。

    当然，实施尚未完成，因为我们没有向数据库写入任何内容。此外，我们仍无法回答样例场景中提出的大多数问题。例如，我们仍然不知道数据库需要如何连接，它的类型（SQL还是NoSQL），正确的数据格式等等。然而，我们已经做出了一些重要的架构假设，这使得我们能够实现该类并使其可供其他类使用。

    现在是时候在我们的代码中识别出未知项，并用谜题标记它们了。每个谜题都是对改进的请求。我们希望向别人提问，帮助我们完善和纠正我们的假设。这是我们需要添加的第一个谜题：

    public interface DataBridge {
  /**

* @todo #123 I assumed that a simple insert() method will be

* enough to insert data into the database. Maybe it's

* not true, and some sort of transaction support may be

* required. We should implement this interface and create

* an integration test with a database. */ void insert(String text) throws IOException; } ```

这个谜题有三个元素：`@todo`标签，`#123` 定位器和一条评论。定位器显示如下内容：“这个谜题是在处理票号为 #123 的工单时创建的。”

让我们再添加一个谜题：

    void write(char[] cbuf, int off, int len) throws IOException { // @todo #123 I assumed that the data should be sent to the database // as its received by the writer. Maybe this assumption // is wrong and we should aggregate data into blocks/chunks // and then send them to the data bridge. String data = new String(Arrays.copyOfRange(cbuf, off, off + len)); this.bridge.insert(data); } ```

    这个谜题说明了我们的担忧之一，因为我们不确定这个架构决策是否正确。实际上，目前的设计非常原始，很可能是错误的。为了完善并重构它，我们需要从任务规范者那里获得更多信息。

    任务已经完成。现在，你可以将你的分支重新合并到 master 分支，并将票据返还给指派给你的人。他现在的任务是找到其他能够解决我们刚刚创建的谜题的人。

    每个现在创建的谜题都会产生其他的谜题，这些谜题将由其他人解决。因此，我们的简单一小时任务有可能生成数百个其他任务，可能需要数天甚至数年才能完成。然而，你的目标是尽快完成你的具体任务，并将你的分支重新集成到“master”分支中。

    Best Practices

    有一些简单的规则可以帮助你正确地放置拼图。

    首先，您应该将@todo注释放在您的代码遇到存根的地方。例如，在单元测试中。您正在实现一个测试，但由于类尚未实现，测试失败。您可以使用@Ignore注释跳过该测试，并在其JavaDoc中添加一个@todo谜题。

    其次，您的拼图应尽可能接近触发存根的代码元素。假设您有一个单元测试，其中包含三个测试方法。由于该类尚未实现，所有方法都失败了。最好的方法是忽略它们，并创建三个（！）拼图。每个拼图都应解释您对该类的期望以及应该如何实现。

    第三，尽可能详细地描述。你的谜题将很快成为其他人的任务定义。因此，请清楚地解释你期望下一个人实现什么，如何实现，使用哪些文档等等。应该提供足够的信息，以便下一个被分配到谜题的人能够在没有额外输入的情况下实现你所需的类！

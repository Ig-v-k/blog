    The Hidden Dangers of Method Overloading

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    方法重载是许多编程语言中的常见特性，允许一个类拥有两个或更多具有相同名称但参数不同的方法。根据微软的说法，方法重载是“提高可重用库的可用性、生产力和可读性的最重要技术之一。”我不同意这种观点。在我看来，方法重载可能导致代码不易读，并引发更多的错误，因为在相同名称下维护两个或更多的实现会产生隐藏的语义，这必然导致误解和功能缺陷。

    让我们从一个Java示例开始。假设您希望在购物车中添加产品，可以使用产品ID或Product对象。如果只提供了ID，您希望购物车访问产品目录，查找相应的Product对象并添加它。这就是方法重载可以证明有益的地方（add()方法通过两个不同的签名和实现进行了两次定义）。

    class Cart {
  private final List<Product> products;
  void add(int id) {
    this.add(new Catalog().findById(id));
  }
  void add(Product p) {
    this.products.add(p);
  }
}
var c = new Cart();
c.add(new Product("book"));
c.add(42);

    这种方法确实很方便，至少有两个原因。首先，add(int)方法处理了从int到Product的转换，这样就不需要在其他地方重复—他们只需将产品的ID传递给该方法，让它完成工作—从而消除了代码重复。其次，由于“搜索目录”的功能没有超出Cart类的范围，简化了周围的代码。看起来，确实提高了代码的可用性和可读性。

    然而，所解决的问题（代码重复和复杂性）比引入的问题要少。虽然add(Product)方法的语义是明显的，但add(int)方法的运作对于用户来说并不清楚。也许它在目录中搜索？也许它从现有购物车中选择第n个产品并重新添加到购物车的末尾？或者它可能在用户之前下过的订单中搜索并提取第n个产品？当我们检查该方法的签名时，我们根本不知道。

    为了理解add(int)方法的功能，我们必须参考其Javadoc块，但这可能不够准确。此外，通常情况下，Javadoc块中的文档可能与方法内部的代码不同步。简而言之，使用add(int)方法的客户端不可避免地对其内部工作方式做出假设。如果客户端幸运的话，1)这个假设将是准确的，2)对方法进行的任何后续更改都不会使这个假设失效。

    似乎方法重载是问题的根源：在一个名称下维护多个实现不可避免地引入了一些隐含的语义。然后，当有东西被隐藏时，它会引发假设，进而导致误解，并最终导致错误。

    我相信，一个更好的替代方案看起来是这样的：

    class ProductInCatalog implements Product {
  private final Product p;
  ProductInCatalog(int id) {
    this.p = new Catalog().findById(id);
  }
  // all "Product" interface's methods
}
class Cart {
  private final List<Product> products;
  void add(Product p) {
    this.products.add(p);
  }
}
var c = new Cart();
c.add(new Product("book"));
c.add(new ProductInCatalog(42));

    ProductInCatalog 的构造函数并不是完全无代码的，尽管理想情况下应该是这样的，然而，对于我们当前的讨论来说，这一点并不那么重要。ProductInCatalog 类是目录中找到的 Product 的一个抽象。这个类被 Cart 类的客户端使用。客户端在充分了解并明确意图的情况下，将 42（产品的 ID）转换为 ProductInCatalog 的实例。这个设计不再隐藏任何元素。没有需要做出的假设，也没有代码必须满足的条件。

    我们是否仍然保留了方法重载提供的优势呢？确实如此。没有代码重复，并且代码的复杂性得到了降低。此外，使用“Cart”的代码可读性得到了显著提高。总结起来，我建议避免使用方法重载，即使这无疑会导致代码库中类的数量增加。然而，这是一个完全不同的讨论话题。

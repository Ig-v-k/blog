    Checked vs. Unchecked Exceptions: The Debate Is Not Over

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我们真的需要检查异常吗？这场辩论已经结束了，对吗？对我来说并非如此。尽管大多数面向对象的编程语言都没有检查异常，而且大多数程序员认为检查异常是Java的错误，但我相信相反的观点——非检查异常才是错误。此外，我认为多个异常类型也是一个坏主意。

    让我先解释一下我对面向对象编程中异常的理解。然后我会将我的理解与“传统”方法进行比较，并讨论它们之间的差异。所以，首先是我的理解。

    假设有一个方法，将一些二进制数据保存到文件中：

    public void save(File file, byte[] data)
  throws Exception {
  // save data to the file
}

    当一切顺利时，该方法只是保存数据并返回控制权。当出现问题时，它会抛出异常，我们必须对此进行处理：

    try {
  save(file, data);
} catch (Exception ex) {
  System.out.println("Sorry, we can't save right now.");
}

    当一个方法说它throws一个异常时，我理解这个方法是不安全的。它有时可能会失败，而我有责任要么1) 处理这个失败，要么2) 声明自己也是不安全的。

    我知道每个方法都是根据单一职责原则设计的。这对我来说是一个保证，如果方法save()失败，那么整个保存操作将无法完成。如果我需要知道此失败的原因，我将解除异常的链接，遍历包含在ex中的异常链和堆栈跟踪。

    我从不使用异常来控制流程，这意味着我从不恢复发生异常的情况。当发生异常时，我让它传递到应用程序的最高级别。有时候，我会重新抛出它，以便在链中添加更多的语义信息。这就是为什么对于save()抛出的异常的原因对我来说并不重要。我只知道方法失败了。这对我来说已经足够了。一直如此。

    出于同样的原因，我不需要区分不同的异常类型。我只不需要那种层次结构。对我来说Exception就足够了。同样地，那是因为我不使用异常来控制流程。

    这就是我对异常的理解。

    根据这种范例，我会说我们必须：

    • 永远使用已检查异常。

    • 永远不要抛出/使用未经检查的异常。

    • 只使用 Exception，不要使用任何子类型。

    • 在throws块中始终声明一个异常类型。

    • 永远不要捕获后不重新抛出；点击这里了解更多相关信息。

    这种范例与我在这个主题上找到的许多其他文章有所不同。让我们进行比较和讨论。

    Runtime vs. API Exceptions

    Oracle says 有些异常应该是 API 的一部分（受检异常），而有些异常是运行时异常，不应该是 API 的一部分（非受检异常）。它们将在 JavaDoc 中记录，但不会在方法签名中体现。

    我不理解这里的逻辑，我相信Java的设计者们也不理解。为什么有些异常很重要，而其他一些则不重要？为什么有些异常在方法签名的throws块中应该有一个适当的API位置，而其他一些则没有？标准是什么？

    我这里有一个答案。通过引入已检查和未检查的异常，Java开发人员试图解决方法过于复杂和混乱的问题。当一个方法过于庞大并且同时做了太多事情（违反了单一职责原则）时，让我们保留一些异常“隐藏”（也称为未检查异常）肯定更好。但这并不是一个真正的解决方案。这只是一个临时的补丁，带来了更多的伤害而不是好处—方法的规模和复杂性不断增加。

    未经检查的异常在Java设计中是一个错误，而不是已检查的异常。

    隐藏一个方法可能在某个点上失败的事实是一个错误。这正是未检查异常所做的。

    相反，我们应该让这个事实显而易见。当一个方法做了太多事情时，就会有太多的失败点，方法的作者会意识到有些地方出了问题——一个方法不应该在这么多情况下抛出异常。这将导致重构。未经检查的异常的存在会造成一团糟。顺便说一下，在Ruby、C#、Python、PHP等语言中根本不存在已检查的异常。这意味着这些语言的创造者对面向对象编程的理解甚至比Java的作者还要少。

    Checked Exceptions Are Too Noisy

    另一个常见反对受检异常的论点是它们使我们的代码变得冗长。我们必须在每个地方都加上try/catch，而不能专注于主要逻辑。Bozhidar Bozhanov甚至建议了一个解决这个冗长问题的技术方案。

    再次，我不理解这个逻辑。如果我想在save()方法失败时做某些操作，我会捕获异常并处理该情况。如果我不想这样做，我只需要声明我的方法也throws异常，并不关注异常处理。问题出在哪里？冗余代码是从哪里来的？

    我这里也有个答案。它来自于未检查异常的存在。我们简单地不能总是忽略失败，因为我们使用的接口不允许我们这样做。就是这样。例如，Runnable类，它在多线程编程中被广泛使用，它的run()方法不应该抛出任何异常。这就是为什么我们总是不得不在方法内部捕获所有异常并将已检查异常重新抛出为未检查异常。

    如果所有Java接口中的所有方法都声明为“安全”（不会抛出任何异常）或“不安全”（会抛出异常），那么一切都会变得合乎逻辑和清晰明了。如果你想保持“安全”，那就要对失败处理负责。否则，选择“不安全”，让用户自己担心安全问题。

    没有噪音，非常干净的代码，以及明显的逻辑。

    Inappropriately Exposed Implementation Details

    一些人认为，将一个已检查的异常放入方法签名的throws中，而不是在此处捕获并重新抛出一个新的类型，会鼓励我们在方法签名中有太多不相关的异常类型。例如，我们的方法save()可能声明它可能会抛出OutOfMemoryException，即使它似乎与内存分配无关。但它确实分配了一些内存，对吗？因此，在文件保存操作期间可能会发生内存溢出。

    再一次，我不明白这个论点的逻辑。如果所有的异常都被检查，并且我们没有多种异常类型，那么我们只需在任何地方抛出“异常”，就好了。我们为什么需要在首位关注异常类型呢？如果我们不使用异常来控制流程，我们就不会这样做。

    如果我们真的想要使我们的应用程序具有内存溢出防护能力，我们将引入一些内存管理器，它将具有类似于bigEnough()方法的东西，它将告诉我们我们的堆是否足够大以进行下一次操作。在这种情况下使用异常是面向对象编程中异常管理的完全不恰当的方法。

    Recoverable Exceptions

    在《Effective Java》一书中，Joshua Bloch提到要“对可恢复的情况使用受检异常，对于编程错误则使用运行时异常。”他的意思类似于这样：

    try {
  save(file, data);
} catch (Exception ex) {
  // We can't save the file, but it's OK
  // Let's move on and do something else
}

    这与一个著名的反模式不要使用异常进行流程控制有什么不同呢？乔舒亚，恕我直言，你是错误的。在面向对象编程中，并不存在可恢复的条件。异常表示从一个方法到另一个方法的调用链被打断，是时候沿着链向上停止了。但我们在异常后永远不会再返回。

    App#run()
  Data#update()
    Data#write()
      File#save() <-- Boom, there's a failure here, so we go up

    我们可以再次启动这个链，但在“抛出”之后不会返回。换句话说，在“捕获”块中我们不会执行任何操作。我们只是报告问题并结束执行。我们永远不会“恢复！”

    All arguments against checked exceptions demonstrate nothing but a serious misunderstanding of object-oriented programming by their authors. The mistake in Java and in many other languages is the existence of unchecked exceptions, not checked ones.

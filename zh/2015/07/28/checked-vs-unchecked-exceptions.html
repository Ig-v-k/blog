    Checked vs. Unchecked Exceptions: The Debate Is Not Over

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我们真的需要检查异常吗？辩论结束了，对吗？对我来说不是。虽然大多数面向对象的语言都没有它们，大多数程序员认为检查异常是Java的错误，但我相信相反的观点——不检查异常才是错误。此外，我认为多个异常类型也是一个坏主意。

    首先让我解释一下我对面向对象编程中异常的理解。然后我会将我的理解与“传统”方法进行比较，并讨论其中的差异。所以，首先是我的理解。

    假设有一个方法，将一些二进制数据保存到文件中：

    public void save(File file, byte[] data)
  throws Exception {
  // save data to the file
}

    当一切顺利时，该方法仅保存数据并返回控制权。当发生错误时，它会抛出Exception，我们必须对此进行处理。

    try {
  save(file, data);
} catch (Exception ex) {
  System.out.println("Sorry, we can't save right now.");
}

    当一个方法说它“throws”异常时，我理解这个方法是不安全的。它可能会偶尔失败，这是我的责任，要么1）处理这个失败，要么2）将自己声明为不安全。

    我知道每个方法都是根据”单一职责原则”（Single Responsibility Principle）设计的。这对我来说是一个保证，如果方法save()失败，那么整个保存操作将无法完成。如果我需要知道这个失败的原因，我将解开（un-chain）异常—遍历在ex中封装的链式异常和堆栈跟踪。

    我从不使用异常来进行流程控制，这意味着我从不恢复异常抛出的情况。当异常发生时，我让它上升到应用程序的最高级别。有时我会重新抛出它，以便在链中添加更多的语义信息。这就是为什么对于save()抛出的异常的原因对我来说无关紧要。我只知道方法失败了。这对我来说足够了。始终如此。

    出于同样的原因，我不需要区分不同的异常类型。我只不需要那种层次结构。对我来说，Exception就足够了。再次强调，那是因为我不使用异常来控制流程。

    这就是我理解异常的方式。

    根据这个范例，我会说我们必须：

    • 始终使用已检异常。

    • 永远不要抛出/使用未经检查的异常。

    • 只使用 Exception，不要使用任何子类型。

    • 在throws块中始终声明一个异常类型。

    • 永远不要捕获异常却不重新抛出；在这里阅读更多相关内容。

    这一范式与我在这个主题上找到的许多其他文章不同。让我们进行比较和讨论。

    Runtime vs. API Exceptions

    Oracle says 一些异常应该是API的一部分（已检查的异常），而一些异常是运行时异常，不应该是API的一部分（未检查的异常）。它们将在JavaDoc中进行文档化，但不会出现在方法签名中。

    我不理解这里的逻辑，我确信Java的设计者们也不理解。为什么有些异常很重要而另一些却不是？为什么有些异常在方法签名的throws块中需要一个适当的API位置，而其他的则不需要？标准是什么？

    我这里有一个答案。通过引入已检查和未检查的异常，Java开发人员试图解决方法过于复杂和混乱的问题。当一个方法过于庞大，在同一时间做太多事情（违反单一职责原则）时，让我们保持一些异常“隐藏”（也称为未检查异常）肯定是更好的。但这并不是一个真正的解决方案。这只是一个暂时的补丁，对我们所有人造成更多的伤害 - 方法的大小和复杂度不断增长。

    未经检查的异常是Java设计中的错误，而不是已检查的异常。

    隐藏一个方法可能在某一点上失败的事实是一个错误。这正是未检查异常所做的事情。

    相反，我们应该让这个事实变得可见。当一个方法做太多事情时，会有太多的失败点，并且方法的作者会意识到有些地方出错了——一个方法不应该在这么多情况下抛出异常。这将导致重构。未检查异常的存在会导致混乱。顺便说一下，在Ruby、C#、Python、PHP等语言中根本不存在已检查的异常。这意味着这些语言的创建者对面向对象编程的理解甚至比Java的作者还要少。

    Checked Exceptions Are Too Noisy

    另一个常见反对受检异常的论点是它们使我们的代码变得冗长。我们必须在各个地方放置try/catch，而无法集中精力于主要逻辑。Bozhidar Bozhanov甚至建议了解决这种冗余问题的技术方案。

    再说一遍，我不明白这个逻辑。如果我想在方法save()失败时做些什么，我会捕获异常并以某种方式处理这种情况。如果我不想那样做，我只需声明我的方法也会throws，并不关注异常处理。问题出在哪里？这种冗长是从哪里来的？

    我这里也有一个答案。它来源于未经检查的异常的存在。我们不能总是忽视失败，因为我们使用的接口不允许我们这样做。就是这样。例如，Runnable类，它被广泛用于多线程编程，它的run()方法不应该抛出任何异常。这就是为什么我们总是不得不在方法内部捕获所有异常并将检查异常重新抛出为未经检查的异常。

    如果所有Java接口中的所有方法都被声明为“安全”（不抛出任何异常）或者“不安全”（抛出异常），那么一切都会变得逻辑和清晰。如果你想保持“安全”，就要负责处理失败。否则，选择“不安全”，让你的用户去担心安全问题。

    没有噪音，非常整洁的代码和明显的逻辑。

    Inappropriately Exposed Implementation Details

    有人说，在方法签名中将可检查的异常放入throws而不是在此处捕获并重新抛出新类型的异常，会鼓励我们在方法签名中拥有太多与问题无关的异常类型。例如，我们的save()方法可能声明可能会抛出OutOfMemoryException，即使它似乎与内存分配无关。但是它确实分配了一些内存，对吧？因此，在文件保存操作期间可能会发生内存溢出。

    再一次，我不理解这个论点的逻辑。如果所有的异常都经过检查，并且我们没有多个异常类型，我们就会无论何时都抛出“Exception”，问题就解决了。为什么我们需要一开始就关注异常类型呢？如果我们不使用异常来控制流程，我们就不会这样做。

    如果我们真的想使我们的应用程序能够抵御内存溢出，我们将引入一些内存管理器，它将拥有类似于 bigEnough() 方法，该方法将告诉我们我们的堆是否足够大以进行下一步操作。在这种情况下使用异常是面向对象编程中完全不合适的异常处理方法。

    Recoverable Exceptions

    在《Effective Java》一书中，Joshua Bloch提到应该“对可恢复的情况使用检查异常，对程序错误使用运行时异常。” 他的意思是这样的：

    try {
  save(file, data);
} catch (Exception ex) {
  // We can't save the file, but it's OK
  // Let's move on and do something else
}

    这与一个名为“不要将异常用于流程控制”的著名反模式有什么不同呢？Joshua，恕我直言，你错了。在面向对象编程中，不存在可恢复的条件。异常表示从一个方法到另一个方法的调用链被打断，现在是时候沿着链向上停下来了。但是异常发生后我们不会再返回。

    App#run()
  Data#update()
    Data#write()
      File#save() <-- Boom, there's a failure here, so we go up

    我们可以再次开始这个链条，但是在throw后面我们不会回退。换句话说，在catch块中我们什么都不做。我们只报告问题并结束执行。我们永远不会”恢复！”

    All arguments against checked exceptions demonstrate nothing but a serious misunderstanding of object-oriented programming by their authors. The mistake in Java and in many other languages is the existence of unchecked exceptions, not checked ones.

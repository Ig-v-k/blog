    Public Static Literals ... Are Not a Solution for Data Duplication

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我在我的应用程序中的一个地方有一个 new String(array,"UTF-8")，并且在另一个地方也有完全相同的代码。实际上，可能有很多地方都有这个代码。每次都必须使用那个 "UTF-8" 常量来从字节数组创建一个 String，这样非常不方便。就像 Apache Commons 所做的那样，定义它一次并重复使用将非常方便；请参见CharEncoding.UTF_8（那里还有许多其他静态字面量）。这些人正在树立一个不好的榜样！public static “properties” 和实用类一样糟糕。

    以下是我所说的具体内容：

    package org.apache.commons.lang3;
public class CharEncoding {
  public static final String UTF_8 = "UTF-8";
  // some other methods and properties
}

    现在，当我需要从字节数组创建一个String时，我使用这个：

    import org.apache.commons.lang3.CharEncoding;
String text = new String(array, CharEncoding.UTF_8);

    假设我想将一个String转换为字节数组：

    import org.apache.commons.lang3.CharEncoding;
byte[] array = text.getBytes(CharEncoding.UTF_8);

    看起来很方便，对吧？这是Apache Commons设计者的想法（Java世界中最流行但实际上很糟糕的库之一）。我鼓励你有不同的想法。我不能告诉你停止使用Apache Commons，因为我们还没有更好的替代方案（但很快就会有！）。但在你自己的代码中，永远不要使用公共静态属性。即使这段代码对你来说看起来很方便，它是一个非常糟糕的设计。

    原因很类似于具有公共静态方法的实用类，它们是不可打破的硬编码依赖关系。一旦你使用CharEncoding.UTF_8，你的对象就开始依赖这个数据，它的用户（你的对象的用户）无法打破这个依赖关系。你可能会说，在"UTF-8"常量的情况下，这正是你的意图，以确保只使用Unicode。在这个特定的例子中，这可能是正确的，但从更全局的角度来看，就不一定了。

    在我们继续之前，让我向你展示我心目中的替代方案。下面是我建议的将字节数组转换为String的方法：

    String text = new UTF8String(array);

    这是伪代码，因为Java设计者将类String定义为final，我们不能真正地扩展它并创建UTF8String，但你可以理解这个概念。在现实世界中，它可能是这样的：

    String text = new UTF8String(array).toString();

    正如您所看到的，我们在UTF8String类的某个地方封装了”UTF-8”常量，而其用户对于这种”字节数组转字符串”的转换方式一无所知。

    通过引入UTF8String，我们解决了”UTF-8”字面量的重复问题。但是我们以一种适当的面向对象方式来解决它——我们将功能封装在一个类中，让每个人都可以实例化其对象并使用它们。我们解决的是功能重复的问题，而不仅仅是数据重复。

    将数据放置在一个共享的位置（CharEncoding.UTF_8）并不能真正解决重复问题；实际上它会让问题变得更糟，主要是因为它鼓励每个人使用相同的共享数据来复制功能。

    我在这里的观点是，每当您在应用程序中发现数据重复时，开始思考一下您正在重复的功能。您很容易找到一遍又一遍重复的代码。为这些代码创建一个新的类，并将数据作为私有属性（或私有静态属性）放置在其中。这是您改进设计并真正消除重复的方式。

    附注：您可以使用方法代替类，但不能使用静态字面量。

    附注：您也可以使用枚举，但只能以此方式。

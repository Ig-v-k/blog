    Public Static Literals ... Are Not a Solution for Data Duplication

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我在我的应用程序中的一个地方有一个new String(array,"UTF-8")，在另一个地方也完全相同的代码。实际上，我可能在很多地方都有它。每一次，我都必须使用那个"UTF-8"常量来从字节数组创建一个String。如果能够在某个地方定义它一次并重复使用，就像Apache Commons一样，那将非常方便；请参见CharEncoding.UTF_8（那里还有很多其他静态字面量）。这些家伙树立了一个不好的榜样！public static“属性”就像utility classes一样糟糕。

    这就是我所说的，具体来说：

    package org.apache.commons.lang3;
public class CharEncoding {
  public static final String UTF_8 = "UTF-8";
  // some other methods and properties
}

    现在，当我需要从字节数组创建一个String时，我使用这个：

    import org.apache.commons.lang3.CharEncoding;
String text = new String(array, CharEncoding.UTF_8);

    假设我想将一个String转换成字节数组：

    import org.apache.commons.lang3.CharEncoding;
byte[] array = text.getBytes(CharEncoding.UTF_8);

    看起来很方便，对吧？这就是Apache Commons的设计者们所认为的（Java世界中最流行但相当糟糕的库之一）。我鼓励你有不同的想法。我不能告诉你停止使用Apache Commons，因为我们还没有更好的替代品（至少目前还没有！）。但是在你自己的代码中，永远不要使用公共静态属性。即使这段代码对你来说看起来很方便，它是一个非常糟糕的设计。

    原因非常类似于具有公共静态方法的实用类 - 它们是不可打破的硬编码依赖关系。一旦使用了CharEncoding.UTF_8，您的对象就开始依赖于此数据，其用户（您的对象的用户）无法打破此依赖关系。您可能会说这是您的意图，在"UTF-8"常量的情况下-确保专门和独占地使用Unicode。在这个特定的例子中，这可能是正确的，但是从更全局的角度来看，情况就不同了。

    在我们继续之前，让我向您展示我心中的替代方案。这是我建议的将字节数组转换为String的方法：

    String text = new UTF8String(array);

    这是伪代码，因为Java设计者将类String定义为final，我们无法真正扩展它并创建UTF8String，但你可以理解这个想法。在现实世界中，它会像这样：

    String text = new UTF8String(array).toString();

    正如你所见，我们将”UTF-8”常量封装在UTF8String类的某个位置，而它的使用者不知道这个”字节数组转字符串”的转换是如何进行的。

    通过引入UTF8String，我们解决了“UTF-8”字面值重复的问题。但是我们以一种适当的面向对象的方式完成了这一点，将功能封装在一个类中，让每个人都可以实例化其对象并使用它们。我们解决了功能重复的问题，而不仅仅是数据重复。

    将数据放置在一个共享的地方（CharEncoding.UTF_8）并不能真正解决重复的问题；事实上，它使问题变得更糟，主要是因为它鼓励每个人使用相同的共享数据来复制功能。

    我在这里的观点是，每当你发现你的应用程序中有一些数据重复时，就要开始思考你正在复制的功能。你很容易找到一遍又一遍重复的代码。为这段代码创建一个新类，并将数据放在那里，作为一个私有属性（或私有静态属性）。这样你就可以改进你的设计并真正消除重复。

    PS. 你可以使用一个方法而不是一个类，但不能使用一个静态文字常量。

    PPS. 你也可以使用enums，但只能这种方式。

    class Time
    def hours
    msec / (60 * 60 * 1000)
    end
    def days
    msec / (24 * 60 * 60 * 1000)
    end
    end

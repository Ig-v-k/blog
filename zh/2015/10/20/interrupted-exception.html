    What Do You Do With InterruptedException?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    InterruptedException is a permanent source of pain in Java, for junior developers especially. But it shouldn’t be. It’s a rather simple and easy-to-understand idea. Let me try to describe and simplify it.

    让我们从这段代码开始：

    while (true) {
  // Nothing
}

    它做什么？什么都不做，只是无休止地旋转CPU。我们能终止它吗？在Java中不能。它只会在整个JVM停止时停止，当您按下Ctrl-C时。在Java中，除非线程自行退出，否则没有办法终止线程。这就是我们必须记住的原则，其他一切都将显而易见。

    让我们将这个无限循环放入一个线程中：

    Thread loop = new Thread(
  new Runnable() {
    @Override
    public void run() {
      while (true) {
      }
    }
  }
);
loop.start();
// Now how do we stop it?

    那么，当我们需要停止一个线程时，该如何做呢？

    这是Java中的设计方式。每个线程中都有一个标记，我们可以从外部设置它。线程可能会偶尔检查该标记并停止执行。自愿地！以下是具体实现方式：

    Thread loop = new Thread(
  new Runnable() {
    @Override
    public void run() {
      while (true) {
        if (Thread.interrupted()) {
          break;
        }
        // Continue to do nothing
      }
    }
  }
);
loop.start();
loop.interrupt();

    这是唯一的方法来请求一个线程停止。在这个例子中使用了两种方法。当我调用loop.interrupt()时，在线程loop的某个地方会设置一个标志为true。当我调用interrupted()时，该标志将被返回并立即设置为false。是的，这就是该方法的设计。它检查标志，返回它，并将其设置为false。我知道这很丑陋。

    因此，如果在线程内部我从不调用Thread.interrupted()，并且当标志位为true时不退出，那么没有人能够停止我。实际上，我将会忽略他们对interrupt()的调用。他们会要求我停止，但我会无视他们。他们将无法中断我。

    因此，总结我们目前所学的内容，一个适当设计的线程会不时地检查该标志并优雅地停止。如果代码不检查该标志并且从不调用Thread.interrupted()，那么它接受这样一个事实，即迟早会被通过点击Ctrl-C来突然终止。

    到目前为止，听起来合乎逻辑吗？我希望是的。

    现在，JDK 中有一些方法可以为我们检查标志并在设置时抛出 InterruptedException。例如，这就是方法 Thread.sleep() 的设计方式（采用了非常原始的方法）：

    public static void sleep(long millis)
  throws InterruptedException {
  while (/* You still need to wait */) {
    if (Thread.interrupted()) {
      throw new InterruptedException();
    }
    // Keep waiting
  }
}

    为什么要这样做呢？为什么不能等待并且不检查标志呢？嗯，我相信这样做是有充分理由的。原因如下（如果我说错了，请纠正）：代码应该要么非常快速执行，要么能够随时中断，中间没有任何其他情况。

    如果您的代码运行速度很快，您就不需要检查中断标志，因为您不想处理任何中断。如果您的代码运行速度较慢，可能需要几秒钟才能执行完毕，请明确地处理中断。

    这就是为什么InterruptedException是一个checked异常。它的设计告诉你，如果你想要暂停几毫秒，要让你的代码具备中断准备。实际上它是这样的：

    try {
  Thread.sleep(100);
} catch (InterruptedException ex) {
  // Stop immediately and go home
}

    嗯，你可以让它浮动到一个更高的层级，由他们负责捕捉。关键是有人必须捕捉它并处理这个线程。理想情况下，只需停止它，因为这就是标记的意义。如果抛出InterruptedException，这意味着有人检查了标记，我们的线程必须尽快完成正在做的事情。

    该帖子的拥有者不想再等待了。我们必须尊重我们的拥有者的决定。

    因此，当你捕获到 InterruptedException 时，你必须尽一切努力将正在进行的工作封装起来并退出。

    现在，再次查看Thread.sleep()的代码：

    public static void sleep(long millis)
  throws InterruptedException {
  while (/* ... */) {
    if (Thread.interrupted()) {
      throw new InterruptedException();
    }
  }
}

    请记住，Thread.interrupted() 不仅返回标志位，而且将其设置为 false。因此，一旦抛出 InterruptedException，标志位将被重置。线程不再知道由所有者发送的中断请求。

    线程的所有者要求我们停止，Thread.sleep()检测到了该请求，将其移除并抛出InterruptedException异常。如果再次调用Thread.sleep()，它将不会知道任何关于该中断请求的信息，也不会抛出任何异常。

    看明白了吗？不要丢失那个InterruptedException非常重要。我们不能只是吞下它然后继续执行。那将是对整个Java多线程理念的严重违反。我们的所有者（线程的所有者）要求我们停止，而我们却置之不理。这是一个非常糟糕的主意。

    这是我们大多数人使用InterruptedException的方式。

    try {
  Thread.sleep(100);
} catch (InterruptedException ex) {
  throw new RuntimeException(ex);
}

    看起来很合理，但不能保证更高级别的实际上会停止一切并退出。他们可能只是在那里捕捉一个运行时异常，线程仍然存活。线程的所有者会感到失望。

    我们必须通知上级，我们刚刚捕获到一个中断请求。我们不能只是抛出运行时异常。这种行为太不负责任了。整个线程都收到了一个中断请求，我们只是将其吞下，并将其转换为RuntimeException。我们不能如此轻率地对待这样严重的情况。

    这是我们需要做的事情：

    try {
  Thread.sleep(100);
} catch (InterruptedException ex) {
  Thread.currentThread().interrupt(); // Here!
  throw new RuntimeException(ex);
}

    我们将标志位设置为true！

    现在，没人会责怪我们对一个有价值的标志持有不负责任的态度。我们发现它处于“true”状态，清除了它，将其重新设置为“true”，然后抛出了一个运行时异常。之后会发生什么，我们并不在意。

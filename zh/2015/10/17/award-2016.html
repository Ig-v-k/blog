    这是软件质量奖的第二年。奖金仍然是$4,096。规则有些改变。继续阅读。顺便说一句，现在是2015年。

    Rules:

    • 一个人只能提交一个项目。

    • 截止日期为2016年9月1日。

    • 我会检查提交历史来确认您是该项目的主要贡献者。

    • 我保留拒绝任何提交的权利，无需解释。

    • 所有提交的内容将会在此页面上发布（包括被拒绝的内容）。

    • 最佳项目将获得$4,096。

    • 最终决策将由我做出，并且不可协商（尽管我可能会邀请其他人帮助我做出正确的决策）。

    • 之前获得任何现金奖励的获胜者不能再次提交。

    每个项目必须满足以下条件：

    • 开源（在GitHub中）。

    • 至少10,000行代码。

    • 至少一岁。

    • 面向对象（这是我唯一理解的东西）。

    使用此标准来选择最佳项目。

    什么不重要:

    • 流行度。即使没有人使用您的产品，它仍然有资格获得此奖项。我不关心流行度，质量才是关键。

    • 编程语言。我相信任何语言，只要使用得当，都可以用来设计高质量的产品。

    • 热点和趋势。即使你的项目只是一个命令行参数解析器，它仍然有资格获奖。我不关心你的营销职位；质量才是最重要的。

    顺便说一下，如果您想赞助这个奖项并增加奖金，请给我发送电子邮件。

    2016年10月18日：这是我对那七个入围者的分析。我尽量关注每个项目（它们都相当不错）。

    pholser/junit-quickcheck（19K LoC，80K HoC）

    • 更改是无法真正追踪的，例如这个提交没有任何与 GitHub 问题的链接，为什么会进行这个更改呢？

    • GitHub 的发布在这里，但是我没有找到具体是如何进行的，发布流程在哪里？此外，它们没有任何文档记录，也没有发布说明。

    • 有一些实用类，例如Lists，Items，和Sequences。

    • 有一些-ER类，例如Shrinker和SampleSizer（其中有代码丰富的构造函数）。

    • “生成器”的设计并不是真正的面向对象的，我猜。它们都是过程的提供者，从面向对象编程的角度来看，并不是真正的“对象”。

    • Score: 5

    NullVoxPopuli/aeonvera（46K行代码，835K行注释）

    • “供应商”的资产位于GitHub存储库中，这是一种不好的做法（包括jQuery）。

    • 这是Ruby on Rails，它是MVC，不是真正的OOP。还有一个带有贫血模型的ORM（在models/目录中）。除此之外，”serializers”、”services”、”validators”等等——也不是真正的OOP。

    • 我没有找到任何 GitHub 发布版本。

    • 在存储库中没有官方的发布流程。我根本无法理解这个产品是如何投入生产的，流程没有自动化（或者脚本不在存储库中）。对于一个 Web 应用来说，这是一个严重的问题。

    • 除此之外，此应用程序肯定比许多其他类似的 RoR 网络应用程序更为简洁。做得好。

    • Score: 4

    SimonKagstrom/kcov（15K LoC，50K HoC）

    • 我没有找到任何静态分析，尽管有很多针对C/C++的开源工具。

    • 有些代码片段是非常复杂的方法，例如在ptrace.cc，elf-parser.cc，或html-writer.cc中。

    • 有一些-ER类，例如”writers”、”verifiers”和”parsers”，例如：AddressVerifier或DwarfParser。

    • 接口以 I 为前缀，这是面向对象编程中的一种不好的实践，参见 reporter.hh。

    • 有一些“utils”，我会用类来替换它们（其中大部分）：utils.hh。

    • Score: 4

    skinny-framework/skinny-framework（44K LoC，191K HoC）

    • 看起来存储库的所有者直接向主分支进行提交，没有任何问题/拉取请求，例如。

    • 我没有找到任何静态分析。

    • 虽然有许多单元测试，但我没有找到代码覆盖率控制。

    • 有一些实用类，例如StringUtil，DateTimeUtil。

    • 这是MVC，它是面向对象编程中的一种反模式。

    • 有一种ORM，它也是一种反模式。

    • 代码的复杂性有时非常高，例如这个文件：AssociationsFeature.scala

    • Score: 2

    paypal/squbs（28K LoC，163K HoC）

    • 静态分析在哪里？

    • 异常吞咽似乎是这里的普遍做法，例如：这里，这里，这里，等等。

    • 有一些实用类，例如 ConfigUtil。

    • 总体印象是复杂度相当高，例如这些文件非常难以理解：ServiceRegistry.scala 或 UnicomplexBoot.scala。

    • 分数：3（主要是由于复杂性）

    ben-manes/caffeine（47K代码行，191K注释行）

    • 一些文件非常长，例如 BoundedLocalCache.java（3300 行代码！）。除了几个超长的文件之外，还有很多 .java 文件超过 300 行代码；这太多了，导致复杂度很高。代码看起来干净，但非常复杂。

    • 有一些-ER类，例如Weigher可以很容易地改名为Weight。”loaders”和”writers”也是如此。

    • Score: 4

    coala/coala（15K 行代码，274K 字节代码）

    • 这个项目去年是个决赛选手。

    • 有一些 -ER 类，比如 “parsers”、”collectors”、”importers”。

    • 我仍然没有看到一致的代码格式，看到一些行前有这么多空格，而且没有任何逻辑（至少我看不出这个逻辑，例如这个例子）。

    • 一些代码是纯粹的全局代码，不包含任何类，就像在 Processing.py 和 Collectors.py 中一样（也许这是不可避免的，但仍然看起来很奇怪）。

    • 一些文件相当长，比如 Lynter.py 或 ConsoleInteraction.py。

    • Score: 5

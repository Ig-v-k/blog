    Three Things I Expect From a Software Architect

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    软件架构师是软件项目中的关键人物，我在几个月前的《软件架构师做什么？》一文中已经解释过。架构师对我们正在开发的产品的技术质量负有个人责任。无论团队多么优秀，技术有多复杂，需求有多混乱，项目赞助人有多混乱，我们只会责怪架构师，不会责怪其他人。当然，如果我们成功了，我们也会奖励架构师。作为项目经理，我对一个优秀的架构师有以下期望。

    在Zerocracy的所有项目中，我期望软件架构师每周定期提交报告几次。每份报告包括三个必要部分：1）范围状态，2）问题，以及3）风险。

    Scope Status

    我正在寻找的第一种也是最重要的信息是范畴状态，它应该以产品分解结构（PBS）格式呈现。无论产品有多复杂或多小，一个优秀的架构师都应该能够创建一个由四到八个项目组成的PBS。例如：

    1. MySQL persistence [done]
2. OAuth login [done]
3. Input parsing in XML [75%]
4. S3 data storage [none]
5. UI cross-platform testing [none]

    这是我期望从一位优秀的建筑师那里每隔几天收到的报告的大小。建筑师在这里的主要目标是确保没有遗漏任何内容。无论项目有多大，所有技术组成部分都必须适应这个PBS。

    建筑师个人负责不漏掉PBS中的信息，并尽可能准确。如果遗漏了某些内容或者报告延迟了，这将成为更换建筑师的充分理由。

    在这里，进度的百分比也很重要。尽管个别任务是根据“0/100完成原则”进行管理，建筑师必须汇编这些百分比并确保编制准确。同样，任何错误都是不可原谅的。

    Issues

    一位建筑师定期报告的第二个重要部分是开发团队面临的当前问题清单。问题是已经发生的事情，我们正在遭受其影响。以下是一些实际例子：

    1. MySQL is too slow for our performance requirements
2. Java 1.6 doesn't allow us to use library X
3. We don't have a replacement for a Ruby guy who left us
4. Integration tests are not predictable

    再次，清单必须包括四到八个项目（不多不少），而且架构师应该在清单中提及最关键的问题。

    Risks

    现在，让我们来谈谈风险。风险是指尚未发生但可能很快发生的事情，如果发生了，我们将陷入麻烦。架构师负责密切关注所有潜在风险，并定期向项目经理报告最重要的风险。以下是一个简短风险报告的示例：

    1. Deployment platform may not support Java 8 [3/8]
2. Library X may take more than the two weeks planned [7/3]
3. We may lose a good Ruby developer soon [5/6]
4. Integration tests may not be safe enough [7/2]
5. We may fail to find an open source library [3/8]

    一个项目经理可能需要关于每个风险的额外信息，但这是另外一个故事。最重要的是让项目经理了解列表的前几项。每个风险都有两个与之相关的数字：概率和影响，范围从0到9。在上面的列表中，第一个风险的概率是3，影响是8。这意味着架构师认为最有可能这不会发生，但如果发生了，我们将会陷入麻烦。

    请注意，每个风险描述中的关键词是可能。风险是指尚未发生的事情。这是风险和问题之间最大的区别。问题是已经发生的风险。

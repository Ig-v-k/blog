    Constructors Must Be Code-Free

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在构造函数中应该完成多少工作？在构造函数内部进行一些计算并封装结果似乎是合理的。这样，当对象方法需要结果时，我们可以立即获得它们。听起来是个好方法吗？不，不是。有一个原因使它变得不好：它阻止了对象的组合，并使其无法扩展。

    假设我们正在创建一个表示人名的接口：

    interface Name {
  String first();
}

    很简单，对吧？现在，让我们试着实现它：

    public final class EnglishName implements Name {
  private final String name;
  public EnglishName(final CharSequence text) {
    this.name = text.toString().split(" ", 2)[0];
  }
  @Override
  public String first() {
    return this.name;
  }
}

    这有什么问题？这样更快，对吧？它只会将名称拆分成多个部分并封装它们一次。然后，无论我们调用多少次first()方法，它都会返回相同的值，而不需要再次拆分。然而，这种思维是有缺陷的！让我向你展示正确的方法并进行解释：

    public final class EnglishName implements Name {
  private final CharSequence text;
  public EnglishName(final CharSequence txt) {
    this.text = txt;
  }
  @Override
  public String first() {
    return this.text.toString().split("", 2)[0];
  }
}

    这是正确的设计。我可以看到你在微笑，所以让我证明我的观点。

    在我开始证明之前，我想请你阅读这篇文章：可组合的修饰器与命令式实用方法。它解释了静态方法和可组合修饰器之间的区别。上面的第一个代码片段非常接近一个命令式实用方法，尽管看起来像一个对象。第二个例子是一个真正的对象。

    在第一个例子中，我们滥用了new运算符，并将其转化为一个静态方法，它在此处立即为我们进行所有计算。这就是命令式编程的要点。在命令式编程中，我们立即进行所有计算并返回完全准备好的结果。在声明式编程中，我们相反地尽可能地延迟计算。

    让我们尝试使用我们的EnglishName类：

    final Name name = new EnglishName(
  new NameInPostgreSQL(/*...*/)
);
if (/* something goes wrong */) {
  throw new IllegalStateException(
    String.format(
      "Hi, %s, we can't proceed with your application",
      name.first()
    )
  );
}

    在这个片段的第一行中，我们只是创建了一个对象的实例并将其标记为name。我们还没有去数据库中获取完整的姓名，将其拆分为各个部分，并将其封装在name中。我们只是想创建一个对象的实例。这样的解析行为对我们来说是一个副作用，在这种情况下，会减慢应用程序的速度。正如您所看到的，如果出现问题并且需要构造一个异常对象，我们可能只需要name.first()。

    我的观点是，在构造函数内部进行任何计算都是一种不好的做法，必须避免，因为它们是副作用，不是对象所有者请求的。

    那么，在重复使用name时的性能如何呢？如果我们创建一个EnglishName的实例，然后调用name.first()五次，我们将得到对String.split()方法的五次调用。

    为了解决这个问题，我们创建了另一个类，一个可组合的装饰器，它将帮助我们解决这个“重复使用”问题：

    public final class CachedName implements Name {
  private final Name origin;
  public CachedName(final Name name) {
    this.origin = name;
  }
  @Override
  @Cacheable(forever = true)
  public String first() {
    return this.origin.first();
  }
}

    我正在使用jcabi-aspects中的Cacheable注解，但您可以使用Java（或其他语言）中的任何其他可用的缓存工具，例如Guava Cache。

    public final class CachedName implements Name {
  private final Cache<Long, String> cache =
    CacheBuilder.newBuilder().build();
  private final Name origin;
  public CachedName(final Name name) {
    this.origin = name;
  }
  @Override
  public String first() {
    return this.cache.get(
      1L,
      new Callable<String>() {
        @Override
        public String call() {
          return CachedName.this.origin.first();
        }
      }
    );
  }
}

    但请不要将CachedName设为可变和延迟加载的——这是一种反模式，我在《对象应该是不可变的》中已经讨论过。

    现在我们的代码将如下所示：

    final Name name = new CachedName(
  new EnglishName(
    new NameInPostgreSQL(/*...*/)
  )
);

    这只是一个非常原始的例子，但我希望你能理解。

    在这个设计中，我们基本上将对象分为两个部分。第一个部分知道如何从英文名中获取名字。第二个部分知道如何将这个计算的结果缓存到内存中。现在作为这些类的用户，我可以决定如何使用它们。我将决定是否需要缓存。这就是对象组合的全部意义。

    再次强调，构造函数中唯一允许的语句是赋值语句。如果你需要在其中放置其他内容，那就开始考虑重构——你的类肯定需要重新设计。

    Constructors Must Be Code-Free

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    构造函数内应该完成多少工作？在构造函数内进行一些计算并封装结果似乎是合理的。这样，当对象方法需要结果时，我们就已经准备好了。听起来是个好方法吗？不，不是。出于一个原因，这是一个坏主意：它阻止了对象的组合并使其无法扩展。

    Let’s say we’re making an interface that would represent a name of a person:

    interface Name {
  String first();
}

    Pretty easy, right? Now, let’s try to implement it:

    public final class EnglishName implements Name {
  private final String name;
  public EnglishName(final CharSequence text) {
    this.name = text.toString().split(" ", 2)[0];
  }
  @Override
  public String first() {
    return this.name;
  }
}

    这有什么问题呢？这样做更快，对吗？它只将名字拆分成几部分，并将它们封装起来。然后，无论我们调用 first() 方法多少次，它都会返回相同的值，而不需要再次进行拆分。然而，这种想法是错误的！让我向你展示正确的方法并解释一下：

    public final class EnglishName implements Name {
  private final CharSequence text;
  public EnglishName(final CharSequence txt) {
    this.text = txt;
  }
  @Override
  public String first() {
    return this.text.toString().split("", 2)[0];
  }
}

    This is the right design. I can see you smiling, so let me prove my point.

    在我开始证明之前，请您先阅读这篇文章：可组合的装饰器 vs. 命令式实用方法。它解释了静态方法和可组合装饰器之间的区别。上面的第一个片段非常接近于命令式实用方法，尽管看起来像一个对象。第二个示例是一个真正的对象。

    在第一个例子中，我们滥用了 new 运算符，并将其转化为一个静态方法，该方法在此处立即进行所有计算。这就是所谓的命令式编程。在命令式编程中，我们立即进行所有计算并返回完全准备好的结果。而在声明式编程中，我们则试图尽可能推迟计算。

    Let’s try to use our EnglishName class:

    final Name name = new EnglishName(
  new NameInPostgreSQL(/*...*/)
);
if (/* something goes wrong */) {
  throw new IllegalStateException(
    String.format(
      "Hi, %s, we can't proceed with your application",
      name.first()
    )
  );
}

    在这个片段的第一行中，我们只是创建了一个对象的实例并将其标记为“name”。我们不想立即访问数据库并从中获取完整的名称，将其拆分为各个部分，并将它们封装在“name”中。我们只是想创建一个对象的实例。这样的解析行为对我们来说将是一个副作用，在这种情况下，会减慢应用程序的速度。正如您所见，如果出现问题并且我们需要构造一个异常对象，我们可能只需要“name.first()”。

    我的观点是，在构造函数中进行任何计算都是一种不好的做法，必须避免，因为它们是副作用，且不是对象所有者所请求的。

    你可能会问，在重复使用name时性能如何。如果我们创建一个EnglishName的实例，然后连续调用name.first()五次，那么将会有五次对String.split()方法的调用。

    To solve that, we create another class, a composable decorator, which will help us solve this “re-use” problem:

    public final class CachedName implements Name {
  private final Name origin;
  public CachedName(final Name name) {
    this.origin = name;
  }
  @Override
  @Cacheable(forever = true)
  public String first() {
    return this.origin.first();
  }
}

    我正在使用来自jcabi-aspects的Cacheable注解（http://aspects.jcabi.com/annotation-cacheable.html），但您可以使用Java（或其他语言）中可用的任何其他缓存工具，例如Guava Cache（https://code.google.com/p/guava-libraries/wiki/CachesExplained）。

    public final class CachedName implements Name {
  private final Cache<Long, String> cache =
    CacheBuilder.newBuilder().build();
  private final Name origin;
  public CachedName(final Name name) {
    this.origin = name;
  }
  @Override
  public String first() {
    return this.cache.get(
      1L,
      new Callable<String>() {
        @Override
        public String call() {
          return CachedName.this.origin.first();
        }
      }
    );
  }
}

    但请不要使CachedName可变且懒加载——这是一个反模式，在对象应该是不可变的中我已经讨论过。

    This is how our code will look now:

    final Name name = new CachedName(
  new EnglishName(
    new NameInPostgreSQL(/*...*/)
  )
);

    It’s a very primitive example, but I hope you get the idea.

    在这个设计中，我们基本上将对象分为两部分。第一部分知道如何从英文名中获取名字。第二部分知道如何将这个计算的结果缓存到内存中。现在作为这些类的用户，我可以决定如何使用它们。我将决定是否需要缓存。这就是对象组合的含义。

    请允许我重申，构造函数中唯一允许的语句是赋值语句。如果您需要在其中放置其他内容，请开始考虑重构——您的类肯定需要重新设计。

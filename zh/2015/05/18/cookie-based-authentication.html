    How Cookie-Based Authentication Works in the Takes Framework

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    当您在Facebook登录页面输入您的电子邮件和密码时，您就可以进入您的帐户。然后，无论您在网站上的任何地方去，您总是可以在页面右上角看到您的照片。Facebook会记住您的信息，不会再次反复要求您输入密码。这要归功于HTTP cookie，并被称为基于cookie的身份验证。尽管这种机制经常引起一些安全问题，但它非常流行和简单。以下是Takes如何在几行代码中实现这一功能的方式。

    首先，让我们看看它是如何工作的。此外，让我们看看我认为它应该如何工作。

    第一步：用户输入电子邮件和密码，然后点击“提交”。服务器接收到一个带有此信息的POST请求。

    POST / HTTP/1.1
Host: www.facebook.com
Content-Type: application/x-www-form-urlencoded

email=me@yegor256.com&password=itisasecret ```

服务器将提供的信息与其记录进行匹配并决定下一步操作。如果信息无效，服务器将返回相同的登录页面，要求您重新输入。如果信息有效，服务器将返回类似如下的内容：

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=me@yegor256.com

    
由于响应状态码是303，浏览器将转到`Location`标头中指定的页面，并打开站点的首页。这是它发送给服务器的内容：

    GET / HTTP/1.1 Host: www.facebook.com Cookie: user=me@yegor256.com

    
服务器从“Cookie”头部获取我的电子邮件，并且知道这是我！无需再次要求密码。服务器相信Cookie中的信息。就是这样。这就是基于Cookie的身份验证的全部内容。

## Wait ... What About Security?

对了，那安全性怎么样？如果服务器信任`Cookie`头部中带有用户电子邮件的任何浏览器请求，那么任何人都可以从其他地方发送我的电子邮件并访问我的账户。

防止这种情况发生的第一步是使用一个只有服务器知道的秘密加密密钥加密电子邮件。除了服务器本身之外，没有人能够以与服务器解密所需方式相同的方式加密它。响应将如下所示，以使用[XOR密码](https://en.wikipedia.org/wiki/XOR_cipher)和`bamboo`作为秘密密钥的加密示例。

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=b1ccafd92c568515100f5c4d104671003cfa39

    
虽然这不是最好的加密机制，但对于适当的加密，最好使用一些更强大的加密方法，比如[DES](https://en.wikipedia.org/wiki/Data_Encryption_Standard)。

这听起来都很好，但是如果有人劫持了服务器和浏览器之间的流量，并获得了正确加密的电子邮件 cookie，会怎么样呢？在这种情况下，即使不知道其内容，盗贼也可以使用相同的 cookie 进行身份验证。服务器会相信这些信息，并允许这个人进入我的账户。这种攻击方式被称为中间人攻击（MITM）。为了防止这种情况发生，我们应该使用 HTTPS，并告知浏览器该 cookie 是敏感的，绝不能在没有 SSL 加密的情况下返回给服务器。这是通过 `Set-Cookie` 头部中的一个额外标志来实现的。

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=me@yegor256.com; Secure

    
还有一种与基于cookie的身份验证相关的攻击类型，这种攻击是基于浏览器能够将与网页关联的所有cookie暴露给其内部执行的JavaScript的能力。攻击者可能会将一些恶意的JavaScript代码注入到页面中（不要问我怎么做...只有在整个HTML渲染出错时才会发生），这段代码将获得对cookie的访问权限。然后，代码将把cookie发送到其他地方，以便攻击者可以收集。这种攻击类型称为[跨站脚本攻击](http://en.wikipedia.org/wiki/Cross-site_scripting)（XSS）。为了防止这种攻击，`Set-Cookie`头部还有另一个名为`HttpOnly`的标志。

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=me@yegor256.com; Secure; HttpOnly

    
这个标志的存在将告诉浏览器，这个特定的cookie只能通过HTTP请求传送回服务器。JavaScript无法访问它。

## How It's Done in Takes

这是[Takes](http://www.takes.org)框架中如何设计基于cookie的身份验证机制。整个框架由*takes*组成，接收请求并产生响应（[此文章]()更详细地解释了该框架）。当请求到达时，我们应该在`Cookie`头中找到身份验证cookie并将其翻译为用户凭据。当响应发送出去时，我们应该在其中添加`Set-Cookie`头，其中包含加密的用户凭据。就这样，只有这两个步骤。

假设我们有一个账户页面，应该显示当前用户的余额：

    final class TkAccount implements Take { private final Balances balances; @Override public Response act(final Request request) { final Identity user = // get it from request return RsHTML( String.format( “<html>Your balance is %s</html>”, this.balances.retrieve(user) ) ); } }

    
在收到`request`后，我们应该检索用户的身份信息，该信息被编码在一个认证cookie中。为了使这个机制可重用，我们有一个[`TkAuth`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/TkAuth.html)装饰器，它包装了一个现有的*take*，解码传入的cookie，并在请求中添加一个新的`TkAuth`头，其中包含用户的身份信息。

    final Codec codec = new CcHex(new CcXOR(new CcPlain())); final Pass pass = new PsCookie(codec); new TkAuth(new TkAccount(), pass);

    
再次，当`TkAuth`接收到一个包含身份验证cookie的请求时，它会要求`pass`解码cookie并返回一个有效的[`Identity`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/Identity.html)或者`Identity.ANONYMOUS`。

然后，当响应返回到浏览器时，`TkAuth` 会要求 `pass` 将身份编码回字符串，并将 `Set-Cookie` 添加到响应中。

[`PsCookie`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/PsCookie.html) 使用一个[`Codec`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/codecs/Codec.html)实例来执行这些反向和正向编码操作。

当我们的 `TkAccount`  *take* 想要从请求中检索当前已验证的用户身份时，它可以使用 [`RqAuth`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/RqAuth.html)，这是 [`Request`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/Request.html) 的实用装饰器。

    final class TkAccount implements Take { @Override public Response act(final Request request) { final Identity user = new RqAuth(request).identity(); // other manipulations with the user } }

    
`RqAuth` 装饰器使用由 `PsCookie` 添加的头部信息，以验证用户并创建一个 `Identity` 对象。

## How Is It Composable?

这个机制确实非常可扩展和“可组合”。假设我们希望在集成测试期间跳过身份验证。下面是如何实现：

    new TkAuth( take, // original application “take” new PsChain( new PsFake(/* if running integration tests */), new PsCookie( new CcHex(new CcXOR(new CcPlain())) ) ) );

    
[`PsChain`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/PsChain.html) 实现 [`Pass`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/Pass.html) 并尝试通过逐个询问所有封装的密码来验证用户。链中的第一个是 [`PsFake`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/PsFake.html)。在其构造函数中使用单个布尔参数，它会决定是返回一个虚假身份还是什么都不返回。通过一个单一的布尔触发器，我们可以关闭应用中的整个身份验证机制。

假设您想通过Facebook OAuth对用户进行身份验证。以下是如何进行的：

    new TkAuth( take, // original application “take” new PsChain( new PsByFlag( new PsByFlag.Pair( PsFacebook.class.getSimpleName(), new PsFacebook( “… Facebook API key …”, “… Facebook API secret …” ) ) ), new PsCookie( new CcHex(new CcXOR(new CcPlain())) ) ) );

    
当用户在您的网站上点击登录链接时，浏览器将跳转到 `facebook.com`，在那里验证他/她的身份。然后，Facebook返回一个`302`重定向响应，其中`Location`头设置为我们在登录链接中提供的URL。链接必须包含类似于这样的内容：`?PsByFlag=PsFacebook`。这将告诉[`PsByFlag`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/PsByFlag.html)此请求用于验证用户身份。

`PsByFlag` will iterate through all encapsulated "pairs" and try to find the right one. [`PsFacebook`](http://static.javadoc.io/org.takes/takes/1.1/org/takes/facets/auth/social/PsFacebook.html) will be the first and the right one. It will connect to the Facebook API using the provided credentials and will retrieve all possible information about the user.

这是我们如何实现一个注销机制的方法：

    new TkAuth( take, // original application “take” new PsChain( new PsByFlag( new PsByFlag.Pair( PsFacebook.class.getSimpleName(), new PsFacebook( “… Facebook API key …”, “… Facebook API secret …” ) ), new PsByFlag.Pair( PsLogout.class.getSimpleName(), new PsLogout() ) ), new PsCookie( new CcHex(new CcXOR(new CcPlain())) ) ) ); ```

    现在，我们可以在网站上的任何链接中添加“?PsByFlag=PsLogout”，这样就可以将当前用户注销。

    您可以通过查看Rultor中的TkAppAuth类，了解所有这些是如何在实际应用中工作的。

    Software Quality Award, 2015

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我是软件开发中规则和纪律的忠实粉丝；例如，参见你是黑客还是设计师？。而且，我也是面向对象编程的忠实粉丝，追求其纯粹的形式；例如，参见一个好对象的七个美德。除此之外，我还是Zerocracy的联合创始人和首席执行官，这是一家软件开发公司，我通过它将我对纪律和清晰设计的崇尚付诸实践。

    我希望鼓励你们与我一同分享我的热情——不仅仅通过阅读这个博客，还通过以有纪律的方式创造真正的开源软件。这个奖项是为那些勇敢逆流而上，将质量看得高于一切的人而设的。

    发送给我你自己的项目以供审查并参加比赛。

    Rules:

    一个人可以提交最多三个项目。

    提交截止日期为 2015年9月1日。

    提交必须通过电子邮件发送至me@yegor256.com。我只需要您的GitHub登录名和存储库名称；我将检查提交历史记录以确保您是该项目的主要贡献者。

    我保留拒绝任何提交且无需解释的权利。

    所有投稿都将在此页面上发布（包括被拒绝的投稿）。

    结果将于10月15日在此页面和通过电子邮件公布。

    最佳项目将获得$4,096。

    最佳的8个项目将获得1年的开源许可证，可以免费使用JetBrains的任意产品（每个项目一份许可证）。

    最终决策将由我作出，并且不可商议（尽管我可能会邀请其他人帮助我做出正确的决策）。

    每个项目必须：

    开源（在GitHub）。

    至少5000行代码。

    至少一年以上。

    面向对象（这是我唯一理解的东西）。

    最好的项目将包含（了解更多）：

    严格且明确的设计原则。

    持续交付。

    更追踪变更。

    自我记录的源代码。

    代码格式的严格规则。

    什么是无关紧要的：

    流行度。即使没有人使用你的产品，它仍然有资格获得这个奖项。我不关心流行度；质量才是关键。

    编程语言。我相信，任何语言只要正确使用，都可以用来设计高质量的产品。

    热点和趋势。即使你的项目只是另一个命令行参数解析器，它仍然有资格获奖。我不关心你的营销职位；质量才是最重要的。

    顺便提一下，如果您想赞助这个奖项并增加奖金，请给我发邮件（mailto:me@yegor256.com）。

    10月5日：我收到了raphw/byte-buddy的作者发来的一封电子邮件，要求我重新考虑对这个项目的决定。我快速查看了一下为什么这个项目被过滤掉，并决定将其包括在入围名单中。顺便说一句，如果你们中有任何人认为自己的项目被错误地排除在外，请随时给我发送电子邮件（mailto:yegor256@gmail.com）。

    十月十一日：今天我分析了所有的12个项目。它们都是非常优秀的项目，因此为了找到最好的一个，我专注于它们的“罪恶”，而非优点。以下是我初步发现的情况。

    coala-analyzer/coala（14K Python LoC，160K HoC）

    None is used in many places (over 400 places I found), which technically is NULL, and it’s a serious anti-pattern

    有一些全局函数，例如get_language_tool_results和DictUtilities。这绝对是面向对象编程中的一个糟糕的主意。

    Constants 类是一个糟糕的想法。

    在运行时检查对象类型是一个不好的主意，例如ClangCountVectorCreator.py。

    cindex.py有什么问题？它有近3200行代码，太多了。

    静态分析不是构建/发布流程中的强制步骤。这就是为什么我认为代码格式不一致，有时相当丑陋。例如，pylint报告了数百个问题。（更新：使用了scrutinizer，但我仍然相信在本地使用pylint会严重提高代码质量）

    有些方法有文档，而其他方法没有。我不明白其中的逻辑。如果所有方法都有文档就太好了。而且，并不是所有类都有文档。

    Score: 5

    checkstyle/checkstyle（83K Java LoC，553K HoC）

    有很多以ER结尾的类，例如SeverityLevelCounter，Filter，和AbstractLoader，它们是反模式。

    有很多实用类，它们在面向对象编程中绝对是一件糟糕的事情。它们甚至被分组到一个名为utils的特殊包中，这是一个非常糟糕的想法。

    设置器和获取器无处不在，连同不可变类一起，这真的不是面向对象编程的一部分，例如DetectorOptions。

    NULL is actively used, in many places—it’s a serious anti-pattern

    我找到了五个 .java 文件，每个文件中都有超过1000行代码，例如 ParseTreeBuilder.java 中有2500+行代码。

    有不同贡献者直接提交到主分支，并且其中一些提交没有与任何工单相关联。很难理解为什么会有这些提交。例如，看看这个：7c50922。是否有讨论参与？是谁做出的决定？完全不清楚。

    Releases are not documented at all.

    发布版本没有任何文档记录。

    发布过程没有自动化。至少我没有在仓库中找到任何发布脚本。

    Score: 3

    citiususc/hipster （5K Java LoC，64K HoC）

    获取器和设置器在许多地方都会使用，例如在DepthFirstSearch和Transition中。除此之外，几乎所有的类都是可变的，并且被用作算法需求的传输容器。我认为这不是面向对象编程应该被使用的方式。

    有公共的静态方法，甚至有实用类，例如这个具有有趣名字的F方法。

    NULL is used actively, especially in iterators—it’s a bad idea

    JavaDoc文档不一致，有些方法有文档，而有些没有。

    并非所有的提交都与工单相关联，例如，请查看此链接：8cfa5de。

    更改直接提交到master分支，不使用拉取请求。

    我没有找到自动发布的程序。我找到了一个用于定期快照部署到Bintray的程序，但是发布呢？它们是手动完成的吗？

    没有静态分析，所以有时候代码看起来很乱。

    单元测试的数量相当少。除此之外，我没有在任何地方找到真正的代码覆盖报告。

    Score: 4

    gulpjs/gulp（700行JavaScript代码）

    这个项目对于比赛来说太小了，只有700行代码。被取消资格了。

    Score: 0

    kaitoy/pcap4j (42K 行代码, 122K 字符)

    有一个 util 包含一些实用类，这是一种不好的做法。

    NULL is used in mutable objects, for example in AbstractPcapAddress; it’s a bad idea

    太多的 static 方法和变量。它们无处不在。甚至有一个名为 pcap4j-packetfactory-static 的模块，充满了具有静态方法的“类”。

    JavaDoc文档不一致，有时候甚至不完整，请查看此链接示例。

    只有几个问题和六个拉取请求。提交与问题没有关联。几乎没有任何改变的可追溯性。

    发布程序未自动化，releases 未记录。

    没有静态分析，这就是为什么代码有时看起来很混乱。

    Score: 3

    raphw/byte-buddy（84K 行代码，503K 字符数）

    我找到了超过20个.java文件，每个文件有超过1000行的代码。TypePool.java甚至有6200行代码！

    有许多public static 方法和属性。我意识到这可能是在Java中处理问题域的唯一方式，但还是…

    instanceof is used very often, and it’s a bad practice in OOP. Again, I understand that problem domain may require it sometimes, but still…

    大多数提交直接放到主分支上，不通过拉取请求或票据，这就是为什么它们的可追踪性被破坏了。

    发布流程不是自动化的（我没有找到脚本）。

    Score: 5

    subchen/snack-string（1K 行，2K 字符）

    这个项目相对较小，正好符合竞争要求，并由一位开发人员完成。除此之外，我没有看到任何问题。代码看起来是面向对象的，所有更改都可以追溯到问题和拉取请求，发布流程是自动化的，静态分析是强制性的，发布有文档记录。赞！

    Score: 10

    有几个实用类，例如Streams。

    发布过程没有记录和自动化。我没有找到它。而且，releases也没有任何文档记录。

    静态分析缺失。

    xvik/guice-persist-orient（17K LoC，54K HoC）

    有 ORM，所以有 getters和setters，例如在 VersionedEntity 中。

    依赖注入被积极使用，我相信这在面向对象编程中通常是一个不好的想法。但在这个项目中，我理解它是问题域所要求的。无论如何…

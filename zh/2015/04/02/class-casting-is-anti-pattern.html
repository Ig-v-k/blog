    Class Casting Is a Discriminating Anti-Pattern

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    类型转换是一种非常有用的技术，当没有时间或愿望来仔细思考和设计对象时。类型转换（或类转换）可以根据它们所属的类或实现的接口，以不同的方式与提供的对象一起工作。类转换帮助我们对可怜的对象进行歧视，并根据它们的种族、性别和宗教对它们进行隔离。这样做是否是一种良好的实践呢？

    这是类型转换的一个非常典型的例子（例如，Google Guava里面充满了它，比如Iterables.size()）。

    class Foo {
  int sizeOf(Iterable items) {
    int size = 0;
    if (items instanceof Collection) {
      size = Collection.class.cast(items).size();
    } else {
      for (Object item : items) {
        ++size;
      }
    }
    return size;
  }
}

    这个sizeOf()方法计算可迭代对象的大小。然而，它足够智能，以理解如果items也是Collection的实例，就无需实际迭代它们。将它们转换为Collection然后调用size()方法会更快。看起来很合理，但这种方法有什么问题呢？我看到两个实际问题。

    首先，sizeOf()和Collection之间存在隐藏的耦合。这种耦合对于sizeOf()的客户端是不可见的。他们不知道sizeOf()方法依赖于Collection接口。如果明天我们决定更改它，sizeOf()将无法工作。而且我们会感到非常惊讶，因为它的签名并没有提到这个依赖关系。这显然不会发生在Collection上，因为它是Java SDK的一部分，但对于自定义类，这可能会发生并且确实会发生。

    第二个问题是 sizeOf() 方法的不可避免的日益复杂。它所需要处理的特殊类型越多，它就会变得越复杂。这种 if/then 分叉是不可避免的，因为它必须检查所有可能的类型并给予它们特殊处理。这种复杂性是违反了 单一职责原则 的结果。该方法不仅计算 Iterable 的大小，还执行类型转换和基于该转换进行分叉。

    替代方案是什么？有几种选择，但最明显的是方法重载（在类似Ruby或PHP这样的半面向对象语言中不可用）。

    class Foo {
  int sizeOf(Iterable items) {
    int size = 0;
    for (Object item : items) {
      ++size;
    }
    return size;
  }
  int sizeOf(Collection items) {
    return items.size();
  }
}

    那不是更优雅吗？

    从哲学的角度来看，类型转换是对进入方法的对象的一种歧视。该对象符合方法签名提供的合同。它实现了Iterable接口，这是一个合同，并且它期望与进入同一方法的所有其他对象一样受到平等对待。但是该方法通过它们的类型来区分对象。该方法基本上是在询问对象的…种族。黑色的对象向右走，而白色的对象向左走。这就是instanceof在做的事情，也是歧视的本质。

    通过使用instanceof，该方法将传入的对象按照它们所属的特定组进行分组。在这种情况下，有两个组：集合和其他所有对象。如果你属于集合，你会得到特殊处理。即使你遵守了Iterable约定，我们仍然会特殊对待一些对象，因为它们属于一个被称为Collection的”精英”组。

    你可以说Collection只是一个对象可能遵守的另一个合同。这是真的，但在这种情况下，遵守该合同的人应该通过另一个门进入。你宣布sizeOf()接受所有遵守Iterable合同的人。我是一个对象，我按照合同的要求去执行。我进入这个方法，并期望和其他进入相同方法的人一样得到同等对待。但是，显然，在方法内部，我意识到一些对象有一些特权。这不是歧视吗？

    总结来说，我认为instanceof和类强制转换是反模式和代码异味。一旦你发现需要使用它们，就开始考虑重构。

    Redundant Variables Are Pure Evil

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    冗余变量是一种仅存在于解释其值的变量。我坚信这样的变量不仅仅是纯粹的噪音，而且是恶意的，对代码的可读性产生非常负面的影响。当我们引入冗余变量时，我们的目的是使代码更清晰、更易读。然而事实上，我们使其更冗长，更难理解。无一例外，任何只使用一次的变量都是冗余的，必须用一个值替代。

    在这里，变量fileName是冗余的。

    String fileName = "test.txt";
print("Length is " + new File(fileName).length());

    这段代码必须看起来不同。

    print("Length is " + new File("test.txt").length());

    这个例子非常原始，但我相信你已经看过许多这些冗余变量了。我们使用它们来”解释”代码—它不再只是一个字符串字面值"test.txt"，而是一个fileName。代码看起来更容易理解了，对吗？但实际上并非如此。

    首先，让我们深入了解代码的”可读性”是什么。我认为这个质量可以通过我需要理解所看到的代码所花费的时间（以秒为单位）来衡量。时间越长，可读性越低。理想情况下，我希望能在几秒钟内理解任何一段代码。如果我不能做到，那就是它的作者的失败。

    记住，如果我不理解你的代码，那是你的错。

    代码长度的增加会降低可读性。因此，在阅读代码时，我需要记住的变量名越多，消化代码并对其目的和影响做出结论的时间就越长。我认为，在不考虑放弃工作的情况下，我最多只能舒适地记住四个变量。

    新变量会使代码变得更长，因为需要额外的行来声明它们。而且，它们会使代码变得更复杂，因为读者必须记住更多的名称。

    因此，当你想引入一个新的变量来解释你的代码在做什么时，请停下来思考。你的代码在第一次写的时候就太复杂和冗长了！重构它，使用新的对象或方法而不是变量。通过将代码的一部分移入新的类或私有方法中，使代码变得更短。

    此外，我认为在完美设计的方法中，除了方法参数，你不需要任何变量。

    More variables with longer self-explainable names make code... #elegantobjects

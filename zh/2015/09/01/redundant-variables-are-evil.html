    Redundant Variables Are Pure Evil

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    冗余变量是专门用来“解释”其值的变量。我坚信这样的变量不仅仅是纯噪音，而且还是“邪恶”的，对代码的可读性有非常负面的影响。当我们引入一个冗余变量时，我们的目的是使代码更清晰、更易读。然而，实际上我们只会使代码更冗长、更难理解。毫无例外，只使用一次的变量是多余的，必须用一个值来替代。

    在这里，变量fileName是多余的。

    String fileName = "test.txt";
print("Length is " + new File(fileName).length());

    这段代码必须看起来不同。

    print("Length is " + new File("test.txt").length());

    这个例子非常原始，但我相信你已经看到过这些多余的变量很多次了。我们使用它们来“解释”代码—它不再只是一个字符串字面量"test.txt"，而是一个fileName。代码看起来更容易理解，对吧？实际上并不是这样。

    让我们首先了解一下代码的“可读性”。我认为这种质量可以通过我理解所查看的代码所需的时间来衡量。时间越长，可读性越低。理想情况下，我希望能够在几秒钟内理解任何一段代码。如果我不能做到这一点，那就是作者的失败。

    [请记住]（），如果我不理解你，那是你的错。

    代码的长度越长，阅读性就越差。因此，我在阅读代码时需要记住的变量名越多，就需要花更长的时间来理解代码并得出关于其目的和影响的结论。我认为最多只能记住四个变量名，而不必考虑辞职。

    新的变量使得代码变得更长，因为它们需要额外的行来声明。而且它们使得代码变得更复杂，因为读者需要记住更多的名称。

    因此，当您想要引入一个新变量来解释您的代码正在做什么时，请停下来思考一下。您的代码本来就太复杂和冗长了！使用新对象或方法对其进行重构，而不是使用变量。通过将代码的一部分移入新的类或私有方法中，使您的代码更简洁。

    此外，我认为在完美设计的方法中，除了方法参数之外，您不需要任何变量。

    More variables with longer self-explainable names make code... #elegantobjects

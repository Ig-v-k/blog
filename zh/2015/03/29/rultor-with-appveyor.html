    How AppVeyor Helps Me to Validate Pull Requests Before Rultor Merges Them

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    AppVeyor 是一个很棒的云持续集成服务，用于构建Windows项目。Rultor 是一个DevOps助手，利用Docker容器自动化发布、合并和部署操作。这些文章详细介绍了Rultor的工作原理和用途：Rultor.com，一个合并机器人和主分支必须为只读。

    问题是Rultor在Docker容器中运行所有脚本，而Docker无法构建Windows项目。唯一而且最佳的逻辑解决方案是在运行其他所有脚本之前触发AppVeyor。如果AppVeyor给出绿灯，我们会继续使用我们通常在Docker中的脚本。否则，整个构建将失败。下面我将解释这个自动化是如何在Takes framework中配置的。

    首先，我从我的AppVeyor账户获取了一个令牌（在写作时它在这里）。我创建了一个名为curl-appveyor.cfg的文本文件，其中包含了这个内容（里面并不是我的真实令牌，只是一个示例）：

    --silent
--header "Authorization: Bearer 1hdmsfbs7xccb9x6g1y4"
--header "Content-Type: application/json"
--header "Accept: application/json"

    然后，我使用rultor命令行工具对此文件进行了加密。

    $ rultor encrypt -p yegor256/takes curl-appveyor.cfg

    我创建的文件名为curl-appveyor.cfg.asc。我提交并推送到yegor256/takes GitHub存储库中。

    $ git add curl-appveyor.cfg.asc
$ git commit -am 'CURL config for Appveyor'
$ git push origin master

    然后，我在 Docker 脚本中配置了 AppVeyor 的“ping”。这是我在 .rultor.yml 文件中所做的。

    decrypt:
  curl-appveyor.cfg: "repo/curl-appveyor.cfg.asc"
merge:
  script: |-
    ver=$(curl -K ../curl-appveyor.cfg \
      --data "{accountName: 'yegor256',
        projectSlug: 'takes',
        pullRequestId: '${pull_id}'}" \
      https://ci.appveyor.com/api/builds | jq -r '.version')
    while true; do
      status=$(curl -K ../curl-appveyor.cfg \
        https://ci.appveyor.com/api/projects/yegor256/takes/build/${ver} \
        | jq -r '.build.status')
      if [ "${status}" == "success" ]; then break; fi
      if [ "${status}" == "failed" ]; then
        echo "see https://ci.appveyor.com/project/yegor256/takes/build/${ver}"
        exit 1
      fi
      echo "waiting for AppVeyor build ${ver}: ${status}"
      sleep 5s
    done
    mvn clean install

    这里没有魔法，非常简单。首先，我使用AppVeyor REST API的/api/builds端点开始一个新的构建。${pull_id}是一个环境变量，来自Rultor，它包含当前pull请求的整数编号。

    我使用jq来解析AppVeyor的JSON输出。

    一旦构建开始，我获取其唯一的version并开始循环检查其状态。我等待success或failed。其他任何状态都意味着构建仍在进行中，我应该继续循环。

    你可以在这个示例pull请求中看到它是如何工作的：yegor256/takes#93。

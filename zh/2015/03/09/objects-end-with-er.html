    Don't Create Objects That End With -ER

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    经理。控制器。助手。处理器。作者。读者。转换器。验证器。路由器。调度器。观察者。监听器。排序器。编码器。解码器。这是耻辱殿堂的类名。你在你的代码中见过它们吗？在你使用的开源库中见过它们吗？在模式书中见过它们吗？它们都是错误的。它们有什么共同之处？它们都以”-er”结尾。那么哪里有问题？它们不是真正的类，它们实例化的对象也不是真正的对象。相反，它们是一组假装是类的过程的集合。

    Peter Coad 曾说过：要质疑任何以 “-er” 结尾的类名。关于这个主题有一些不错的文章，包括你的编码规范正在伤害你 (作者：Carlo Pescio)，我曾经听到的最好的编程建议之一 (作者：Travis Griggs)，以及对象命名 - 不要在对象名中使用 ER (作者：Ben Hall)。反对使用 “-er” 后缀的主要论点是，“当你需要一个管理器时，通常意味着被管理的只是普通的数据结构，而真正工作的是这个管理器的智能过程。”

    我完全同意，但是我想再加上几句话。

    我已经在《一个好对象的七个美德》中提到过，一个好的对象名称不是一个职位头衔，但我没有解释为什么我这么认为。除此之外，在《实用类与函数式编程无关》中，我试图解释声明式和命令式编程范式之间的区别。现在是时候把这两个部分结合起来了。

    假设我是一个对象，你是我的客户。你给我一桶苹果，并要求我根据大小对它们进行排序。如果我生活在命令式编程的世界中，我会立即将它们排序好，然后我们将再也不会互动。我会按照你的要求完成工作，甚至不会思考为什么你需要它们排序。我只是一个不太关心你真正意图的排序器。

    List<Apple> sorted = new Sorter().sort(apples);
Apple biggest = sorted.get(0);

    正如你在这里看到的，真正的意图是找到桶里最大的苹果。

    这不是你期望的一个能够帮助你处理一桶苹果的好商业伙伴。

    相反，如果我生活在声明式编程的世界中，我会告诉你：“将它们视为已排序；接下来你想做什么？。”你会告诉我你现在需要最大的苹果。然后我会说：“没问题，给你。”为了返回最大的苹果，我不会对它们进行排序。我只会一个一个地遍历它们，并选择最大的。这个操作比先排序然后选择列表中的第一个要快得多。

    换句话说，我会默默地不按照你的指示行事，而是会尝试用自己的方式做事。我会成为你更聪明的合作伙伴，而不是那个命令式的排序器。我会成为一个真正的对象，表现得像一个已排序的苹果列表，而不是一个用于排序的过程。

    List<Apple> sorted = new Sorted(apples);
Apple biggest = sorted.get(0);

    See the difference?

    请特别注意sorter和sorted名称之间的区别。

    让我们回到类名。当你在类名后面添加”-er”后缀时，你立即将它变成了一个愚蠢的命令执行者，用来执行你的意愿。你不允许它思考和即兴发挥。你期望它完全按照你的要求来做事——排序、管理、控制、打印、写入、组合、连接等等。

    一个对象是一个活生生的有机体，它不希望被告诉该做什么。它希望成为其他对象的平等伙伴，根据其合约（在Java和C#中称为接口，Swift中称为协议）展示行为。

    从哲学角度来看，”-er” 后缀是对贫穷对象的一种不尊重的表现。

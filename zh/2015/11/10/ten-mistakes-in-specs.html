    10 Typical Mistakes in Specs

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    有一本名为《软件需求》的好书，由Karl Wiegers撰写，讲述了关于软件需求的内容。在我看来，每个软件工程师都应该阅读这本书。我不需要重复书中的内容，但我们在规格说明中一直在犯一些非常简单而又典型的错误。我一遍又一遍地在我们的文件中看到这些错误，这就是为什么我决定总结它们的原因。所以，以下是其中最重要和最典型的十个错误，从一个程序员阅读规格说明文件的角度来看。

    著名标准IEEE 830-1998的第4.3章指出，一个良好的规范应该是正确的，无歧义的，完整的，一致的，有优先级的，可验证的，可修改的和可追踪的。总共有八个特点。然后，该标准用相当简单的英语逐个解释这些特点。但是，我们是否有时间阅读那些枯燥的标准呢？那些是给大学教授和认证机构的。我们是从业者，天哪！…等等，我在开玩笑。

    无论项目有多小，我们有多么实际，总会有一份文件来解释需要完成的工作，它可以被称为“软件需求规范”，或者“规范”，或者只是“规范”。当然，有很多创造空间，但我们是工程师，而不是艺术家。我们必须遵循规则和标准，主要是因为它们使我们的沟通更容易。

    现在，我要说重点了。我通常看到的规格文件基本上违反了之前提到的所有八个原则。以下是它们具体是如何违反的概要。顺便说一句，所有的例子都来自真实的商业软件项目中的真实文件。

    No Glossary or a Messy One

    How about this:

    UUID is set incrementally to make sure there
are no two users with the same account number.

    UUID和账号之间有什么区别？它们是同一回事吗？看起来是的，对吗？或者它们是不同的… 知道UUID代表什么将会很好。是”唯一用户ID”还是”统一用户身份描述符”呢？我一点都不知道。我迷失了，我想找到这段文字的作者并对他或她做些坏事…

    我已经写过了，最糟糕的技术规范没有词汇表。根据我的经验，这是所有需求文档中最大的问题。这不是散文！也不是情书！这是技术文档。我们不能为了好玩而玩弄文字。我们不应该仅仅为了表达自己而使用产品规范。我们写作是为了被理解，而不是给读者留下深刻印象。而且规则也和图表一样：如果我无法理解你，那是你的问题。

    这是在适当修改后的文本中的样子：

    UUID is user unique ID, a positive 4-bytes integer.
UUID is set incrementally to make sure there
are no two users with the same UUID.

    Better now?

    因此，第一个也是最大的问题是滥用术语和仅靠文字，而没有在词汇表中进行预定义。

    Questions, Discussions, Suggestions, Opinions

    我最近在产品规格中看到了这个。

    I believe that multiple versions of the API
must be supported. What options do we have? I'd
suggest we go with versioned URLs. Feel free to
post your thoughts here.

    是的，这段文字在需求文档中完全存在。首先，作者表达了他对这个主题的个人意见。然后，作者问我有哪些可能的选择。接着，他建议我考虑一些事情，然后邀请我进行交谈。

    令人印象深刻，对吧？显然，作者具有非常有创造力的个性。但是我们应该尽可能让这个人远离项目文档。这不是需求文档所欣赏的内容。嗯，我们欣赏创造力，但是这四件事是严格禁止的：问题、讨论、建议和意见。

    规格中不能有任何问题。这些问题是给谁的？给我，一个程序员吗？我应该实现软件还是回答你的问题？我对与你一起头脑风暴不感兴趣。我期望你作为需求作者告诉我需要做什么。在撰写文件之前找到所有答案。那就是你的工作付费的地方。如果你没有答案，在那里写上类似TBD（待确定）的东西。但不要问问题。那很烦人。

    需求文档不是一个讨论板。作为规范的读者，我希望清楚地看到需要做什么，没有任何”也许”或”我们可以以不同的方式来做”。当然，你需要讨论这些问题，但请在记录之前进行讨论。请在其他地方进行讨论，比如Skype、Slack或电子邮件。如果你真的想在文档中讨论，可以使用带有版本跟踪功能的Google Docs或Word。但是讨论结束后，请从文档中删除其历史记录。它的存在只会让我这个程序员感到困惑。

    也不需要将需求格式化为“建议”。只需说出需要完成的工作以及软件的工作方式，不必担心错误。通常，人们在害怕直言不讳时会采取建议的方式。他们不会说“应用程序必须在Android 3.x及更高版本上运行”，而是说“我建议将应用程序与Android 3.x及更高版本兼容”。看到区别了吗？在第二个句子中，作者试图回避个人责任。他没有说“确切的是Android 3.x”；他只是提出建议。不要胆小；直言不讳地说出来。如果你犯了错误，我们会纠正你。

    当然，我们不希望看到意见。这不是给朋友的信，而是属于项目的正式文件。几个月或几周后，你可能会离开项目，其他人将与你的文档一起工作。规范是项目发起人和项目团队之间的合同。文档作者的观点在这里没有任何影响。不要写下”似乎Java会更快”并建议”我们应该使用它”，而是应该写”Java更快，所以我们必须使用它”。显然，你这样写是因为你这样认为。但一旦写下，我们不在乎它来自于谁以及你对这个问题的看法。这些信息只会让我们更加困惑，所以省略掉。只写事实，不要加入意见。

    请别误会，我并不反对创造力。程序员并非机器人，默默地按照文件中的指示实施。但是混乱的文件与创造力无关。如果你想让我创造，那就明确创造力的限度，并允许我在其中进行实验；例如：

    Multiple versions of the API must be supported. How exactly
that is done doesn't really matter.

    这就是你邀请我发挥创造力的方式。我意识到产品的用户对API中的版本机制没有真正的理由或期望。我可以自由地做任何我能做的事情。太好了，我会按照我的方式去做。

    但是再次强调一下：规范不是一个讨论区。

    Mixing Functional and Quality Requirements

    这是它的外观：

    User must be able to scroll down through
the list of images in the profile smoothly and fast.

    这是我看到的几乎每个规范中都会出现的典型错误。在这里，我们混合了一个功能要求（“滚动图像”）和一个非功能要求（“滚动流畅快速”）。为什么这样做不好呢？嗯，没有具体的原因，但它显示了缺乏纪律。

    这样的需求很难进行验证或测试，很难追踪，也很难实现。作为一个程序员，我不知道什么更重要：滚动还是确保滚动速度快。

    另外，修改这样的陈述很困难。如果明天我们增加另一个功能要求——例如滚动好友列表——我们希望这个滚动也是平滑快速的。然后几天后，我们又希望说“快速”表示反应时间少于10毫秒。我们将不得不在两个地方重复这些信息。看到我们的文档最终可能变得多么混乱吗？

    因此，我强烈建议您始终将功能需求和非功能需求分开进行文档记录。

    Mixing Requirements and Supplementary Docs

    这与之前的问题类似，可能会是这样的：

    User can download a PDF report that includes a full
list of transactions. Each transaction has ID,
date, description, account, and full amount. The report
also contains a summary and a link to the user account.

    很明显，这段文字描述了两件事情。第一是用户可以下载PDF报告。第二是报告的样式应该是什么样的。第一件事是一个功能要求，而第二件事必须在一个补充文件（或附录）中进行描述。

    一般来说，功能要求必须非常简短：“用户下载”，“用户保存”，“客户端请求和接收”等等。如果你的文本变得过长，那就有问题了。尝试将其中的一部分移到补充文档中。

    Un-measurable Quality Requirements

    这就是我所说的。

    Credit card numbers must be encrypted.
The app should launch in less than 2 seconds.
Each web page must open in less than 500 milliseconds.
User interface must be responsive.

    我可以通过打开我在过去几年中见过的许多项目的需求规格来找到更多的例子。它们看起来都一样。问题总是相同的：很难定义一个真正可测试和可衡量的非功能性需求。

    是的，这很困难。主要是因为有很多因素。以这个例子为例：“应用程序必须在2秒内启动。”在什么设备上？用户配置文件中有多少数据？“启动”是什么意思；是否包括配置文件加载时间？如果存在启动问题怎么办？这些都计算在内吗？有很多类似的问题。

    如果我们回答所有这些问题，需求文本将填满整整一页。没有人希望这样，但是拥有不可衡量的需求更是一种更大的问题。

    再说一遍，这并不容易，但却是必要的。努力确保所有质量要求都完整且没有歧义。

    Implementation Instructions

    这个例子说明了一个非常常见的问题：

    User authenticates via Facebook login button
and we store username, avatar, and email in the
database.

    这就是微观管理，而这是需求分析师绝不能对程序员做的事情。你不应该告诉我如何实现你想要的功能。你想让用户能够通过Facebook登录？就这么说。你真的在乎是通过按钮点击还是其他方式实现吗？你真的关心我在数据库中存储什么吗？如果我使用文件而不是数据库，这对你来说重要吗？

    我不这么认为。只有在非常罕见的情况下才会有关系。大部分时间，这只是微观管理。

    规范应该只要求对业务真正重要的事项。其他的事情就由我们程序员来决定。我们决定使用哪个数据库，按钮放在哪里，以及在数据库中存储哪些信息。

    如果你真的关心这个问题，因为有一些更高级别的限制，请直接说出来。但是，请注意，这不是给我们程序员的实施指导，而是作为这样的非功能性要求。

    Login page must look like this (screenshot attached).
We must store user email locally for future needs.

    重点是我对需求没有任何意见，但我强烈反对实施指导。

    Lack of Actor Perspective

    文本可能看起来像：

    PDF report is generated when required. It is
possible to download a report or save it
in the account.

    这里的问题是没有“参与者”参与其中。这个功能或多或少是清楚的，但是不清楚是谁在做所有这些。用户在哪里？这只是一个发生在某处的故事。这并不是程序员实现它所需要的真正内容。

    最好的解释功能的方法是通过用户故事。而一个好的用户故事总是有，猜猜看…一个用户。它总是以”用户…“开头，后面跟着一个动词。用户下载、用户保存、用户点击、打印、删除、格式化等。

    用户不一定是人类。可能是系统、RESTful API客户端、数据库，或者其他任何东西。但是始终是某个实体。”可以下载…“不是一个用户故事。这是可能对谁来说呢？

    Noise

    How about this:

    Our primary concern is performance and an attractive
user interface.

    这是噪音。作为本文的读者，我既不是投资者也不是用户，我是一名程序员。我不关心你在这个项目中的“主要关注点”是什么。我的工作是实现产品，以使其与规格相匹配。如果性能是你的主要关注点，请为我创建可衡量和可测试的需求。我会确保产品满足这些需求。如果你无法创建需求，请不要用这些无关紧要的信息垃圾邮件我。

    我不想分享你的担忧、信仰或意图。那是你的事情。你受雇于负责将所有这些准确、明确地转化为可测试和可衡量的要求。如果你做不到这一点，那是你的问题和你的责任。不要试图让这成为我的问题。

    很多时候…等等。非常非常多时候。不对。几乎总是。再错一次。总是！没错，规范文档总是充斥着杂音。有些文档少一点，有些多一点。我相信这是懒惰和不专业的文档作者的症状。大多数情况下，只是懒惰。

    他们不想去思考和翻译他们的关切、想法、思绪、意图和目标成为功能性和非功能性需求。他们只是将这些放入文件中，然后希望程序员们能够找到正确的解决方案。优秀的程序员应该能够弄清楚什么是良好的性能，对吧？我们只需告诉他们性能对我们来说很重要，他们就会找到解决办法的。

    不！不要那样做。正确地完成你的工作，让程序员完成他们的工作。

    我们作为程序员，不应接受这样的文档。我们应该直接拒绝它们，并要求需求作者重新编写并清除冗余内容。如果产品规格中有很多冗余内容，我建议甚至不要开始进行开发工作。

    Will Work, Needs to Work, Must Work

    这是又一个非常典型的错误：

    The API will support JSON and XML. Both formats
must fully support all data items. XML needs to
be validated by XSD schema.

    看起来很混乱吧？有三种不同的观点，没有一种适合规范文档。规范必须描述一个产品，就好像它已经存在了一样。规范必须听起来像是一本手册、教程或参考资料。这段文字必须重新写成这样：

    The API supports JSON and XML. Both formats
fully support all data items. XML is validated
by XSD schema.

    看得出区别了吗？所有的“必须”，“需要”和“将会”这些词只是给文件增加了疑虑。对于本规范的读者来说，“API将会支持”听起来像是“将来的某个时间，也许在下一个版本，它才会支持”。这不是作者的初衷，对吧？不应该存在疑虑、双关或者可能性。API就是支持的。就这样。

    I may have forgotten something important, but these issues are so obvious and so annoying … I’m going to use this post as a simple guide for our system analysts. Feel free to share your experience with requirements documents below in the comments.

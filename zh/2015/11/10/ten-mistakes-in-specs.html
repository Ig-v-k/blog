    10 Typical Mistakes in Specs

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    有一本很棒的书叫做《软件需求》（Software Requirements），由卡尔·威格斯（Karl Wiegers）撰写，讲述了关于软件需求的知识。在我看来，每个软件工程师都应该阅读这本书。我不需要重复书中的内容，但我们在规范中总是犯下一些非常简单且典型的错误。我在我们的文档中一次又一次地看到这些错误，这就是为什么我决定总结它们的原因。因此，在一个程序员读规范文件的角度来看，以下是十个最关键且典型的错误。

    著名标准《IEEE 830-1998》的第4.3章节中提到，一个良好的规范应该是“正确的”、“明确的”、“完整的”、“一致的”、“有等级的”、“可验证的”、“可修改的”和“可追溯的”。总共有八个特性。然后，该标准用相当简单的英语逐一解释了它们。但是我们有时间去阅读那些枯燥的标准吗？那些标准是给大学教授和认证机构的。我们是实践者，天哪！…等等，我在开玩笑。

    无论项目有多小，我们有多实际，总会有一份文件来解释需要完成的工作，它可能被称为“软件需求规范”，或者“规范”，或者只是“规范”。当然，有很大的创造空间，但我们是“工程师”，而不是艺术家。我们必须遵循规则和标准，主要是因为它们使我们的沟通更容易。

    现在，我要说重点了。我通常看到的规范几乎违背了之前提到的所有八个原则。下面是它们具体是如何违背的摘要。顺便说一下，所有的例子都来自真实的商业软件项目中的真实文件。

    No Glossary or a Messy One

    How about this:

    UUID is set incrementally to make sure there
are no two users with the same account number.

    UUID和账号之间有什么区别？它们是同一回事吗？看起来是的，对吗？或者它们可能是不同的…了解UUID代表什么将会很好。它是”唯一用户ID”还是”统一用户身份描述符”？我不知道。我迷失了，我想找到这段文字的作者并对他…或她采取一些不好的行动。

    我已经写过了，最糟糕的技术规范没有术语表。根据我的经验，这是所有需求文档中最大的问题。这不是散文！这不是情书！这是技术文档。我们不能为了好玩而玩弄词汇。我们不应该仅仅为了表达自己而使用产品规范。我们写作是为了被理解，而不是给读者留下印象。和图表一样，原则在这里也是一样的：如果我不理解你，那是你的错。

    以下是经过适当改写后的文本：

    UUID is user unique ID, a positive 4-bytes integer.
UUID is set incrementally to make sure there
are no two users with the same UUID.

    Better now?

    因此，第一个也是最大的问题是滥用术语和单纯的文字，而没有在词汇表中事先定义它们。

    Questions, Discussions, Suggestions, Opinions

    我最近在产品规格中看到过这个。

    I believe that multiple versions of the API
must be supported. What options do we have? I'd
suggest we go with versioned URLs. Feel free to
post your thoughts here.

    是的，这段文字在需求文档中原封不动地存在。首先，作者表达了他对主题的个人观点。然后，作者问我有哪些可能的选择。然后，他建议我考虑一些事情，之后，他邀请我进行谈话。

    令人印象深刻，不是吗？显然，作者有着非常有创造力的个性。但我们应该尽量让这个人远离项目文档。这不是需求文档所欣赏的。好吧，我们欣赏创造力，但这四个事物是严格禁止的：问题、讨论、建议和观点。

    规格说明中不能有任何“问题”。这些问题是面向谁的？我，一个程序员？我应该实施软件还是回答你的问题？我对与你一起进行头脑风暴不感兴趣。我期望你，一个需求作者，告诉我需要做什么。在撰写文档之前找到所有答案。这就是你的工作所付出的报酬。如果你没有答案，可以放一些“待定”（to be determined）之类的东西。但不要提问。那很烦人。

    需求文档不是一个“讨论”板。作为规格的读者，我希望看到需要做什么，而不是任何“可能”或“我们可以以不同的方式做”。当然，你需要讨论这些问题，但请在记录之前进行讨论。在其他地方进行讨论，比如Skype、Slack或电子邮件。如果你真的想在文档中进行讨论，使用带有版本跟踪的Google Docs或Word。但讨论结束后，从文档中删除其历史记录。它的存在只会让我这个程序员感到困惑。

    也没有必要将需求格式化为“建议”。只需明确说明需要做什么以及软件如何工作，不必担心出错。通常，人们在害怕直接说出来时会采用建议。例如，不要说“应用程序必须在Android 3.x及更高版本上运行”，而是说“我建议使应用程序与Android 3.x及更高版本兼容”。看到区别了吗？在第二个句子中，作者试图避免个人责任。他没有说“确切的Android 3.x”；他只是提出建议。不要胆小，直截了当地说。如果你犯了错误，我们会纠正你的。

    当然，“观点”也不受欢迎。这不是写给朋友的信；这是属于项目的正式文件。几个月或几周后，你可能会离开项目，其他人将会与你的文档一起工作。规格是项目发起人和项目团队之间的合同。文档作者的观点在这里没有任何影响。不要写“似乎Java会更快”并建议“我们应该使用它”，而是说“Java更快，所以我们必须使用它”。显然，你写这个是因为你这样认为。但一旦写在那里，我们不在乎它来自谁以及你对这个问题的看法。这些信息只会让我们更困惑，所以省略掉。只有事实，没有观点。

    别误会，我并不反对创造力。程序员不是机器人，默默地按照文档所说的实施。但一个混乱的文档与创造力无关。如果你希望我创造，要定义创造力的界限，让我在其中进行实验；例如：

    Multiple versions of the API must be supported. How exactly
that is done doesn't really matter.

    这就是你邀请我发挥创造力的方式。我意识到产品的用户对API中的版本控制机制没有任何理由或期望。我可以自由发挥。太好了，我会按照自己的方式来做。

    但是再次强调一点，规范不是一个讨论板。

    Mixing Functional and Quality Requirements

    这是它的外观：

    User must be able to scroll down through
the list of images in the profile smoothly and fast.

    我在几乎每一个规范中都见过这样的典型错误。在这里，我们混合了一个功能性要求（“滚动图像”）和一个非功能性要求（“滚动流畅且快速”）。为什么这样做不好呢？嗯，没有具体的原因，但它表现出了缺乏纪律性。

    这样的要求很难验证或测试，很难追踪，也很难实现。作为一名程序员，我不知道什么更重要：滚动还是确保滚动速度快。

    此外，修改这样的陈述也很困难。如果明天我们添加另一个功能性要求——例如滚动朋友列表——我们将希望要求这个滚动也是流畅且快速的。然后，几天后，我们将希望说“快速”意味着反应时间小于10毫秒。然后，我们将不得不在两个地方重复这些信息。看到我们的文档最终可能变得多么混乱吗？

    因此，我强烈建议您始终将功能性和非功能性要求分开记录。

    Mixing Requirements and Supplementary Docs

    这与先前的问题相似，可能会是这样的：

    User can download a PDF report that includes a full
list of transactions. Each transaction has ID,
date, description, account, and full amount. The report
also contains a summary and a link to the user account.

    很明显，这段文字描述了两件事情。第一是用户可以下载PDF报告。第二是这份报告应该是什么样子的。第一件事情是一个功能要求，而第二件必须在一个附加文件（或附录）中进行描述。

    通常来说，功能要求应该非常简短：“用户下载”，“用户保存”，“客户请求和接收”等等。如果你的文字变得更长，那就有些问题了。尝试将其中的一部分移动到一个附加文件中。

    Un-measurable Quality Requirements

    这就是我在谈论的内容：

    Credit card numbers must be encrypted.
The app should launch in less than 2 seconds.
Each web page must open in less than 500 milliseconds.
User interface must be responsive.

    我可以通过打开我在过去几年见过的许多项目的需求规范来找到许多更多的例子。它们看起来都一样。问题总是一样的：定义一个真正可测试和可衡量的非功能性需求非常困难。

    是的，很难。主要是因为有很多因素。以这一行为例：“应用程序必须在2秒内启动。”在什么设备上？用户配置文件中有多少数据？“启动”是什么意思；是否包括配置文件加载时间？如果存在启动问题呢？它们算吗？有很多类似的问题。

    如果我们回答了所有这些问题，需求文本将填满整个页面。没有人希望那样，但是拥有无法衡量的需求更加严重。

    再次强调，这并不容易，但是却是必要的。努力确保所有质量需求都是完整的且没有歧义。

    Implementation Instructions

    这个例子说明了一个非常常见的陷阱：

    User authenticates via Facebook login button
and we store username, avatar, and email in the
database.

    这是微观管理，而需求分析师绝不应该这样对待程序员。你不应该告诉我如何实现你所需的功能。你想让用户能够通过Facebook登录？直接说出来。你真的在乎是通过按钮点击还是其他方式来实现吗？你真的在乎我将什么存储在数据库中吗？如果我使用文件而不是数据库，这对你来说重要吗？

    我不这么认为。只有在非常罕见的情况下，这才会有关系。大多数时候，这只是微观管理。

    规范应该只要求与业务真正相关的内容。其他一切由我们程序员来决定。我们决定使用什么数据库，按钮将放在何处，以及将在数据库中存储哪些信息。

    如果你真的关心这个，因为有某些更高级的限制，那就直接说出来。但再次强调，不要作为对我们程序员的实施指令，而是作为像这样的非功能性需求：

    Login page must look like this (screenshot attached).
We must store user email locally for future needs.

    问题是，我并不反对需求，但我强烈反对实施指示。

    Lack of Actor Perspective

    正文可能如下所示：

    PDF report is generated when required. It is
possible to download a report or save it
in the account.

    这里的问题在于没有“参与者”。这个功能或多或少是清晰的，但是不清楚是谁在做所有这些事情。用户在哪里？这只是一个某个地方发生某些事情的故事。这并不是程序员实现所需要的。

    解释功能的最好方法是通过用户故事。一个好的用户故事总是有一个用户。它总是以“用户…”开头，后面跟着一个动词。用户下载，用户保存，用户点击，打印，删除，格式化等。

    用户不一定是人。它可以是一个系统，一个RESTful API客户端，一个数据库，任何东西。但总是某个人。”可以下载…“不是一个用户故事。是可以由谁来实现呢？

    Noise

    How about this:

    Our primary concern is performance and an attractive
user interface.

    这是噪音。作为这份文件的读者，我既不是投资者也不是用户。我是一个程序员。我不关心你在这个项目中的“主要关注点”是什么。我的工作就是按照规格实现产品。如果性能是你的主要关注点，给我制定可衡量和可测试的需求。我会确保产品满足这些需求。如果你无法提出需求，请不要用这些无关紧要的信息来骚扰我。

    我不想分享你的关注点、信念或意图。那是你的事情。你的工作是将这一切准确无误地翻译成可测试和可衡量的需求。如果你无法做到这一点，那是你的问题，是你的错。不要试图让这成为我的问题。

    很多时候…等等。很多很多时候。不对。几乎总是。错了，总是！没错，规格文档总是充斥着噪音。其中一些噪音稍微少一些，有些则更多。我认为这是懒惰和不专业的文档作者的症状。在大多数情况下，只是懒惰。

    他们不想去思考并将他们的关注点、想法、思维、意图和目标转化为功能性和非功能性需求。他们只是把它们放进文档中，希望程序员们能够找到正确的解决方案。优秀的程序员应该知道什么是良好的性能，对吧？我们只需告诉他们我们关注性能，他们会想出办法的。

    不！不要这样做。做好你的工作，让程序员们做好他们的工作。

    作为程序员，我们永远不应接受这样的文档。我们应该直接拒绝它们，并要求需求作者重新修改和清除噪音。我建议如果规格中有很多噪音，甚至不要开始工作在这个产品上。

    Will Work, Needs to Work, Must Work

    这是又一个非常典型的错误：

    The API will support JSON and XML. Both formats
must fully support all data items. XML needs to
be validated by XSD schema.

    看起来很混乱吧？有三个不同的观点，而且没有一个适合作为规范文档。规范必须描述一个产品，就好像它已经存在一样。规范必须像手册、教程或参考资料一样。这段文字必须重新写成这样：

    The API supports JSON and XML. Both formats
fully support all data items. XML is validated
by XSD schema.

    看到区别了吗？所有的“必须”，“需要”和“将要”这些词只是给文档增加了怀疑。对于这个规范的读者来说，“API将要支持”听起来像是“将来的某个时候，也许在下一个版本中，它将会支持”。这不是作者的本意，对吧？不应该有怀疑，没有双关，没有也许。API就是支持。

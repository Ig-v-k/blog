    Stop Comparing JSON and XML

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    JSON还是XML？哪个更好？哪个更快？我应该在下一个项目中使用哪个？停止吧！这些东西是不可比较的。这就像比较一辆自行车和一辆AMG S65。说真的，哪个更好？它们都能把你从家带到办公室，对吧？在某些情况下，自行车可能做得更好。但这是否意味着它们可以相互比较？同样的道理也适用于JSON和XML。它们是非常不同的东西，各有适用的领域。

    这是一个简单的JSON数据片段的样子（140个字符）：

    {
  "id": 123,
  "title": "Object Thinking",
  "author": "David West",
  "published": {
    "by": "Microsoft Press",
    "year": 2004
  }
}

    一个类似的文件在XML中的样子如下（167个字符）：

    <?xml version="1.0"?>
<book id="123">
  <title>Object Thinking</title>
  <author>David West</author>
  <published>
    <by>Microsoft Press</by>
    <year>2004</year>
  </published>
</book>

    看起来很容易比较，对吧？第一个例子稍微短一些，更容易理解，因为它不那么“神秘”，而且在JavaScript中也可以完全解析。就这样吧，让我们使用JSON并宣告XML的死亡！谁需要这个15年历史的笨重XML呢？

    好的，我需要它，而且我喜欢它。让我解释一下为什么。

    并且请别误会我的意思；我并不反对JSON。一点也不。它是一种很好的数据格式。但它只是一种数据格式。我们暂时使用它将数据从A点传输到B点。确实，它比XML更短且更易读。就是这样。

    XML不是一个数据格式；它是一种语言。一种非常强大的语言。让我向您展示它的能力。让我基本上解释一下为什么我喜欢它。我强烈推荐您阅读Elliotte Rusty Harold和W. Scott Means的XML in a Nutshell, Third Edition。

    我相信XML具有四个特点，这些特点使其与JSON或其他简单的数据格式（例如YAML）有着明显的区别。

    • XPath。要从上面的文档中获取出版年份等数据，我只需要发送一个XPath查询：/book/published/year/text()。然而，必须有一个能理解我的请求并返回2004的XPath处理器。这其中的美妙之处在于XPath 2.0是一个非常强大的查询引擎，具有自己的函数、谓词、轴等。你可以在你的XPath请求中放入任何逻辑，而不需要在Java中编写任何遍历逻辑，例如，你可以问：“David West在2004年出版了多少本书？”并得到一个答案，只需通过XPath。JSON与此相比根本无法媲美。

    • 属性和命名空间。您可以像上面使用id属性一样，将元数据附加到您的数据上。数据保留在元素内部，就像书籍作者的姓名一样，而元数据（关于数据的数据）可以和应该放在属性中。这在组织和结构化信息方面非常有帮助。除此之外，元素和属性都可以标记为属于特定的命名空间。这是在多个应用程序同时使用同一XML文档时非常有用的技术。

    • XML Schema. 在一个地方创建一个XML文档，在另一个地方对其进行多次修改，然后将其转移到另一个地方时，您希望确保其结构不会被任何这些操作破坏。其中一个可能使用<year>来存储出版日期，而另一个可能使用带有ISO-8601的<date>。为了避免结构上的混乱，创建一个补充文档，称为XML Schema，并将其与主文档一起发送。每个想要使用主文档的人都会首先使用提供的架构验证其正确性。这是一种在生产中进行的集成测试。RelaxNG是一种类似但更简单的机制；如果您认为XML Schema过于复杂，请尝试使用它。

    • XSL。您可以在没有任何Java/Ruby等代码的情况下对您的XML文档进行修改。只需创建一个XSL转换文档并将其”应用”到您的原始XML上。作为输出，您将获得一个新的XML。XSL语言（顺便说一句，它是纯函数的）是为分层数据操作而设计的。它比Java或任何其他面向对象/过程化方法更适合这个任务。您可以将XML文档转换为任何内容，包括纯文本和HTML。有些人抱怨XSL的复杂性，但请试一试。您不需要全部掌握它，因为它的核心功能非常直观。

    这不是一个完整的列表，但这四个特性对我来说非常重要。它们使我的文档具备了“自给自足”的能力。它可以自我验证（XML Schema），它知道如何修改自己（XSL），并且为我提供了非常方便的访问其内部任何内容的方式（XPath）。

    有很多围绕XML开发的语言、标准和应用程序，包括XForms、SVG、MathML、RDF、OWL、WSDL等等。但是在主流项目中，你可能不太可能使用它们，因为它们相对来说比较“小众”。

    JSON并非设计用于具备这些功能，尽管其中一些功能如今正在JSON世界中尝试找到它们的位置，包括用于查询的JSONPath，用于转换的一些工具，以及用于验证的json-schema。但与XML提供的功能相比，它们只是弱化的模仿品，我不认为它们有任何未来。或者这样说吧：我希望它们能早晚消失。它们只会将一个好的、简单的格式变得笨拙起来。

    因此，总结起来，JSON是一种没有额外功能的简单数据格式。它最适合的用例是AJAX。在其他所有情况下，我强烈建议您使用XML。

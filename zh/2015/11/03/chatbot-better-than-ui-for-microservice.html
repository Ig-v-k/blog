    A Chatbot Is Better Than a UI for a Microservice

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    一个聊天机器人（或称为闲聊机器人，正如维基百科所说）是一种以聊天格式与您对话的软件。我们在一些（微）服务中使用聊天机器人，它们完全替代用户界面。我认为这种方法没有任何创新，但在过去一年左右已被证明非常有效。这是这篇文章的动力所在。这是Rultor聊天机器人是如何为我们工作的以及它的好处。

    让我先举个例子。看一下 jcabi/jcabi-http#115 的 GitHub 提交。

    让我们看看这里发生了什么，然后我们会讨论它的内部设计。基本上，我在这里与一个聊天机器人交谈。聊天机器人的名字是@rultor（我去年写过它）。在1处，我正在要求聊天机器人发布jcabi-http库的新版本。在2处，聊天机器人回复，只是确认任务清楚并且正在处理。在3处，机器人说工作已完成，完成时间为九分钟。我们的对话结束了。就是这样。

    现在，这有什么特别之处呢？

    一件事：没有用户界面。嗯，没有传统的基于网页的HTML/CSS用户界面。没有登录、登出、个人资料、菜单，或者任何类似的东西。Rultor是一个没有网页界面的网络服务。与它通信的唯一方式是通过与其聊天机器人交流。

    这有什么好的呢？有几个原因。

    Service Is Not a Server

    这是一个传统网页系统的架构示意图：

    skinparam componentStyle uml2 User -> [Service] [Service] -> [Database]

    用户向服务提供指令并接收响应。这种通信是通过用户界面（UI）进行的——一组HTTP入口点，接收来自浏览器的请求并返回HTML + CSS响应。或者，如果用户在另一个服务上，则请求可能包含一些数据，而响应将是XML或JSON格式。你明白了；用户是客户端，而服务是服务器。

    就像在餐厅里一样——你说你想要什么，一个服务员去厨房，等在那里，几分钟后拿着意大利炭烤面回来。你是一个客户，那个可爱的女士就是服务员。

    在聊天机器人的情况下，情况就不同了。看看这个架构：

    skinparam componentStyle uml2 用户 -up-> [GitHub] [服务] -up-> [GitHub] [数据库] -up-> [服务]

    首先，用户通过GitHub提供的网络用户界面向GitHub发布请求。这对我们来说是一个通信中心。然后，服务通过其RESTful API连接到GitHub，并检查是否有任何新的请求。如果发现了新内容，服务会执行任务，准备好响应，并将其发布在那里。客户端将收到有关刚刚发布到工单的新响应的电子邮件通知。然后，客户端检查GitHub并找到响应。

    这是在餐厅里的场景：有一个板子上贴着便利贴。首先，你写下便利贴上的内容：“我想要意大利面碳烤，撒上帕尔玛干酪和新鲜黑胡椒”（天啊，我现在太饿了），然后将它钉在15号上。然后，你回到自己的桌子上。厨房里的一位厨师检查那块板子，找到了你的便利贴。他做了那份意大利面，加上了帕尔玛干酪、新鲜黑胡椒、一些罗勒叶和初榨橄榄油…是的，他做得很对…然后把它放在了板子旁边。你听到有人宣布15号订单已经准备好了。你去那里，取走食物，回到桌子上享用。

    问题是再也没有可爱的女士参与了。没有服务器。现在有两个参与与面板进行通信的方—你和厨房。厨房是我们的微服务，但它不再是一个服务器。

    这两个方面现在完全解耦了。他们彼此之间从不交流。他们都是通信中心的客户，这个通信中心可以是GitHub或者餐馆里的一个公告板。

    再次强调，微服务不再是一个服务器。相反，它是一个通信中心的客户端。而位置的转变为我们这些开发者带来了许多好处。

    No Need to Be Fast

    首先，我们不需要过多关注我们的用户界面的性能。嗯，我们根本不关心，因为我们没有用户界面。我们在乎GitHub的响应速度吗？并不是真的。当用户在GitHub上发布消息时，他们并不期望我们的聊天机器人在100毫秒内立即回答。（我相信这是任何经过适当设计的网络系统必须保证的。）

    我们在白板上留了个便条，然后我们假设厨房现在可能在忙别的事情。我们会等几秒钟甚至几分钟。另一方面，如果我给女服务员下命令，她在回复之前等了五秒钟，我会非常惊讶。如果她每个问题都这样做，我会开始自问她是否一切都好。

    我期望用户界面是即时的，但在聊天中，我不介意给机器人一些时间来回答。这是自然的。我们在与真人进行对话时，习惯了有延迟。他们需要一些时间来处理我们的信息、思考和输入回复。

    但是用户界面没有那种奢侈。它必须操作迅速，否则我会立刻感到沮丧。你也有同样的感受，对吗？

    No Need to Look Cute

    这种无服务器设计的另一个优点是不需要外观漂亮。没有网络界面、没有HTML、没有CSS，没有图形设计。也许并不是每个人都喜欢这样。大多数非专业用户可能仍然更喜欢与一个可爱的服务器交谈，而不是把一些纸条贴在墙上。但是，如果我们在处理专业的计算机工程师，他们并不那么挑剔。

    Rultor没有任何网页界面，它的用户根本不知道它的“外观”是什么样子的。它只是与您“交谈”。您所能看到的唯一东西就是它在GitHub上的头像。

    这样能节省大量用于设计的金钱和时间，如果你追求高质量的话，通常设计成本非常昂贵。如果你的网络服务看起来很普通，大部分用户会认为它的功能也很普通。许多好的想法仅仅因为其用户界面没有像人们习惯的那样给人留下深刻印象而夭折，这都多亏了所有那些 Pinterest 和 Instagram。

    一个外表好看的服务员会有更大的机会得到更多的小费，对吧？如果没有服务员，并且我们看不到厨师，我们只能通过食物的质量来评判他或她。

    一样。通过去除用户界面，我们可以专注于我们提供的服务的质量。我们不浪费时间和金钱来变得“友好”。我们将它们用于变得“有用”。

    Much Easier to Scale

    如果我们在那个板子上有太多的便利贴，我们只需要雇佣更多的厨师，甚至可能建造另一个厨房，问题就解决了。我们可以处理任意数量的顾客。只要这个板子足够强大，可以处理多个并行用户。

    GitHub是一个相当大的平台，拥有数十万用户和项目。如果我们有太多的请求进来，我们可以只需向Rultor添加更多的处理节点。记住，我们不再是一个服务器；我们是GitHub的客户端。我们决定何时连接到GitHub以及何时创建对提交请求的响应。

    创造一个可扩展的客户端比一个可扩展的服务器要容易得多，主要是因为没有人真正等待我们快速回应。我们收到的请求负载可以更轻松地管理，因为处理它们的决定是由我们自己做出的。

    Mistakes Are Not So Visible

    当你站在客户面前时，你的大部分错误是不可饶恕的，主要是因为它们非常明显。另一方面，当你在厨房做饭时，没有人能看到你的过失。只有当意大利面加了太多盐时，他们才会注意到。换句话说，他们将根据你的结果来评判你，而不是你是如何产生这些结果的。

    微服务也是同样的情况。当它作为服务器工作时，我们期望它能无缝运行，立即响应，并以结构化和组织良好的方式呈现所有内容。如果出现问题，就会显示在网页上。最好的情况是404错误，而最糟糕的情况是向用户呈现一些错误的信息。即使在微服务引擎内部，这个错误可能并不严重，但用户并不知道。用户会根据你的外观来评判你，即使是小错误也不会被忘记。

    然而，当你们都是信息板的客户时，你们互相看不到对方。用户与GitHub进行通信，而微服务与GitHub进行交互。错误不太明显。相信我，在Rultor公开使用的18个月里，我们遇到了许多这样的问题。我们曾经遇到过停机时间，严重的逻辑错误和数据损坏。但是这些问题很少在在线上变得可见。我们只是在服务器日志中看到了它们。用户并没有看到它们。嗯，大多数情况下是这样。

    Everything Is Traceable

    由于我们之间有一个沟通板，很容易看到我们讨论的完整历史，这非常直观。就像 Slack 的聊天历史一样。你可以看到我们从哪里开始，谁说了什么，以及得出了哪些结论。

    基本上，在Web用户界面中无法获得该可见性。嗯，你可能可以创建一个“操作历史”的特殊页面，但是谁会去查看呢？而且这些信息有多明显和简单呢？最重要的是，这些信息如何与用户界面相匹配呢？

    在日志中，您会看到“构建已启动”的记录，但是什么是构建，它是如何启动的？我如何重新启动它？使用哪些按钮和网络控件？这并不清楚。

    因此，一条按时间顺序排列的聊天记录是无与伦比的。

    Easy to Integrate With Other Services

    是的，请考虑这种方法的未来。如果有一个集中的留言板，用户可以与聊天机器人交谈，那为什么其他聊天机器人不能互相交谈呢？

    忘掉RESTful API吧。只是一个留言板，聊天机器人在其中发布请求并收集回复。它们完全解耦、可替换，而且可扩展性非常强。此外，它们的通信协议是可见且易追踪的。正如前面所解释的那样，它们还拥有许多其他优势。对于我们用户和程序员来说，监控和创建它们更加方便。

    也许完全摒弃RESTful API有些过激，但在一定程度上，我相信这种方法是可行的。

    我对这个想法没有深入研究，但是已经做了一些工作。我们有一个消息平台，可以让多个聊天机器人与用户进行通信。它叫做Netbout。它是一个非常原始的网络系统，具有独立的讨论区。简单来说，任何人都可以创建一个新的讨论，邀请几个朋友，并在那里发布消息。无论是用户还是聊天机器人都可以这样做。

    所以，当一个新候选人想要加入Zerocracy时，我们要求该候选人填写一个在线表格。当候选人点击“提交”按钮后，会开始一次新的讨论，第一个聊天机器人会决定谁应该对该人进行面试。决定是根据表格中列出的技能来做出的。聊天机器人会邀请我们的最佳程序员之一进行面试。面试结束后，另一个聊天机器人会向候选人解释下一步是什么，将其在我们的数据库中注册，并开始显示工作的进展情况。

    从用户的角度来看，它看起来像是与一些只懂得一些简单命令的人交谈。它非常直观且易于设计。

    我认为聊天机器人是与微服务进行交互的一种好方法。特别是当用户更或多或少有专业素养时。

    Seven Deadly Sins of a Software Project

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    可维护性是现代软件开发中最有价值的品质。可维护性基本上可以通过新开发者在开始进行重大更改之前学习软件所需的工作时间来衡量。时间越长，可维护性就越低。在某些项目中，这个时间要求几乎无限，这意味着它从根本上无法维护。我相信有七个根本而致命的罪恶使我们的软件无法维护。以下是它们。

    Anti-Patterns

    很不幸，我们使用的编程语言过于“灵活”。它们允许太多，并且禁止太少。例如，Java对于将整个应用程序放在一个名为“class”的单一实体中，并拥有几千个方法没有任何限制。从技术上讲，应用程序可以编译和运行。但这是一个众所周知的反模式，被称为上帝对象。

    因此，反模式是一种在技术上可以接受的设计方式，但普遍认为是错误的。每种语言都存在许多反模式。它们在产品中的存在就像活体中的肿瘤一样。一旦它开始生长，就很难停止。最终，整个机体死亡。最终，整个软件变得难以维护，必须重新编写。

    一旦你容忍一些反模式，最终你会得到更多的，而且这个“肿瘤”只会不断增长。

    这对于面向对象的语言（Java、C++、Ruby和Python）来说尤其如此，主要是因为它们从过程式语言（C、Fortran和COBOL）继承了很多东西。而且因为面向对象的开发者倾向于以过程式和命令式的方式思考。不幸的是。

    顺便提一下，除了现有的众所周知的反模式列表，我还认为以下几点也是相当糟糕的编码方法。

    我在这里唯一的实际建议是阅读和学习。也许这些书可以帮助你，或者我的书 “Elegant Objects“。始终对你的软件质量持怀疑态度，在它“只是正常工作”的时候也不要放松。就像癌症一样，越早诊断，生存机会就越大。

    Untraceable Changes

    当我查看提交历史时，我应该能够知道每次更改都改变了什么，谁做出了更改，以及更改的原因。此外，获取这三个答案所需的时间必须以秒为单位进行测量。在大多数项目中，这种情况并非如此。以下是一些建议：

    始终使用工单。无论项目大小或团队规模如何，即使只有你自己，都应为解决的每个问题创建工单（GitHub问题）。在工单中简要解释问题，并记录你的思考过程。将工单用作与问题相关的所有信息的临时存储。将可能在将来有任何意义的所有内容发布在工单上，这样其他人在试图理解那些“几个奇怪的提交”时就能明白。

    在提交中引用工单。毋庸置疑，每次提交都必须有一条消息。没有消息的提交是一种非常糟糕的做法，我甚至不会讨论为什么。但仅有一条消息是不够的。每条消息必须以你正在处理的工单编号开头。GitHub（我确信你正在使用它）将自动链接提交和工单，增加了变更的可追溯性。

    不要删除任何东西。Git允许我们进行“强制”推送，这会覆盖服务器上先前存在的整个分支。这只是一个例子，说明你可以破坏开发的历史记录。我也经常看到人们在GitHub讨论中删除他们的评论，以使工单看起来更“清洁”。这是错误的。永远不要删除任何东西；让你的历史与你一起保留，无论它现在看起来多么糟糕（或杂乱）。

    Ad Hoc Releases

    每个软件都必须在交付给最终用户之前进行打包。如果是一个Java库，它必须被打包成一个.jar文件并发布到某个仓库；如果是一个Web应用程序，它必须部署到某个平台等等。无论产品大小如何，总会有一个标准的流程来进行测试、打包和部署。

    一个理想的解决方案是将这个流程自动化，这样可以通过一个命令行执行一个单一的命令来完成。

    $ ./release.sh
...
DONE (took 98.7s)

    大多数项目都远未达到这一水平。它们的发布过程总是涉及一些神秘的操作，负责此事的人（也被称为DevOp）必须在各处点击按钮、登录某个地方、检查一些指标等等。这种即兴的发布过程仍然是整个软件工程行业中一种很典型的错误。

    我在这里只能给一个实际的建议：自动化。我在这方面使用rultor.com，但你可以使用任何你喜欢的工具。重要的是整个过程都是完全自动化的，并且可以从命令行执行。

    Volunteer Static Analysis

    静态分析使我们的代码看起来更好。通过使其看起来更好，我们不可避免地使其工作更好。但这只有在整个团队被强制(!)遵循静态分析器规则的情况下才会发生。我在《严格控制Java代码质量》一文中谈到了这一点。在Java项目中，我使用qulice.com，在Ruby项目中使用rubocop，但几乎每种语言都有许多类似的工具。

    你可以使用其中任何一种，但是一定要强制使用！在大多数使用静态分析的项目中，开发人员只是构建出漂亮的报告，并继续以之前的方式编写代码。这种“自愿”式的方法对项目没有任何好处。而且，它会产生质量上的幻觉。

    我想说的是，在部署流程中，静态分析必须成为强制步骤。如果违反任何静态分析规则，构建将无法通过。

    Unknown Test Coverage

    简而言之，测试覆盖率是指软件在单元测试或集成测试中被测试的程度。覆盖率越高，在测试运行期间执行的代码量就越大。显然，较高的覆盖率是一件好事。

    然而，许多项目开发者根本不知道他们的覆盖率。他们只是没有测量这个度量指标。他们可能有一些测试，但没有人知道它们对软件的深入程度，以及哪些部分根本没有经过测试。这种情况比低测试覆盖率更糟糕，低测试覆盖率至少可以被测量和向所有人报告。

    高覆盖率并不意味着高质量，这是显而易见的。但未知的覆盖率则是可维护性问题的明确指标。当新的开发人员加入项目时，他或她应该能够进行一些更改并查看覆盖率如何受到影响。理想情况下，测试覆盖率应该与静态分析一样进行检查，如果覆盖率低于预先定义的某个阈值（通常在80%左右），则构建应该失败。

    Nonstop Development

    我所说的无间断指的是没有里程碑和发布版本。无论你写的是什么类型的软件，你都必须频繁地发布和版本化它。一个没有明确发布历史的项目是一个难以维护的混乱。

    这主要是因为可维护性是通过阅读你的代码来理解你的方式。

    当我查看源代码以及其提交和发布历史时，我必须能够了解作者的意图，项目一年前的情况，现在的发展方向以及它的路线图等等。所有这些信息都必须包含在源代码中，并且更重要的是，在Git历史中也有记录。

    Git tags和GitHub的发布说明是两个强大的工具，它们为我提供了这些信息。充分利用它们。此外，不要忘记每个产品的二进制版本必须能够立即下载。即使项目目前正在开发3.4版本，我也必须能够下载版本0.1.3并立即进行测试。

    Undocumented Interfaces

    每个软件都有它所应该使用的接口。如果这是一个Ruby gem，我将使用其中的类和方法作为最终用户使用。如果这是一个网络应用程序，最终用户将会看到和控制一些网页来使用该应用程序。每个软件项目都有接口，并且必须进行仔细的文档记录。

    就像前面提到的一样，这也与可维护性有关。作为一个项目中的新程序员，我将从其接口开始学习。我应该能够理解它的功能并尝试自己使用它。

    我在这里谈论的是用户文档，而不是开发者文档。总的来说，我反对将文档放在软件内部。在这一点上，我完全赞同敏捷宣言——可工作的软件比全面的文档更重要。但这并不是指“外部”文档，它是供用户阅读的，而不是开发者。

    因此，必须清楚地记录软件与最终用户的交互。

    如果你的软件是一个库，那么它的最终用户是软件开发人员，他们将使用它，而不是为其做出贡献，只是将其作为一个“黑盒子”使用。

    These are the criteria being used to evaluate the open source projects entered in our award competition.

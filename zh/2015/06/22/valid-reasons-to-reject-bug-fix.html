    A Few Valid Reasons to Reject a Bug Fix

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    当某个东西不按预期工作时，就存在一个错误。一个错误修复基本上是一个补丁（一个拉取请求），用于现有代码库，旨在解决问题并确保”某个东西”按预期工作。很多时候，这样的补丁修复了一个问题，但却导致了很多其他问题。我认为有时候有必要拒绝一个错误修复，并要求作者重新做补丁，以保护项目免受更大的问题。根据我的经验，有几个合理的理由可以拒绝这样的修复。

    It Degrades Code Coverage

    这是一个非常常见的情况：在一个地方做出改变之后，单元测试在其他地方失败了。虽然修复了错误，但可能与之无关的单元测试开始报告错误。在压力下或仅仅因为懒惰，我们不修复它们，只是简单地删除测试或将其标记为暂时“跳过”。问题解决了，构建完成了，所以让我们合并补丁，收工吧，对吗？错了！

    尽管我尽可能支持省略步骤，但这是我不建议你省略的步骤。

    单元测试存在的目的就是在压力之下防止我们破坏产品。

    显然，有些情况下单元测试是错误的，我们必须删除它们。在这种情况下，不要忘记创建新的单元测试。

    也有些情况下，为了让系统尽快恢复在线，必须在几分钟内修复错误，而修复所有单元测试需要一个小时。这种情况表明产品的测试覆盖率存在严重问题。毫无疑问，我们必须进行修复，并要求我们的测试在某段时间内保持沉默。但在这种情况下，请确保在错误修复发布后，你的团队下一个任务是纠正那些被禁用的单元测试。我建议阅读Michael Feathers的《与遗留代码高效工作》（http://amzn.to/1SdcZ8M），该书涉及到了这个非常重要的主题。

    It Doesn’t Reproduce the Issue

    有时候，整个系统可能会因为一行代码中的一个小错误而崩溃。一个明显的错误修复方法是删除该错误，但如果我们关心项目的质量，这并不是一个好的项目所期望的。问题不在于错误拼写，而是在于缺乏在部署阶段捕获该错误的单元测试。

    真正的问题在于代码中这个特定部分缺乏测试代码覆盖率。通过删除错误拼写，我们并没有以任何方式帮助项目。而且，我们正在做一个不好的事情——我们正在“掩盖”真正的问题。

    因此，无论问题多么小或者表面上看起来无伤大雅，它的错误修复必须包含一个额外的测试，首先复现该错误。如果没有这样的测试，一个错误修复就是对项目资金的浪费。

    此外，如果没有一个单元测试复现这个问题，就无法保证我们的错误修复不会引入更多的错误。我甚至可以说，我们有越多的错误修复，熵就越高。而降低这种不确定性的唯一方法就是使用单元测试覆盖代码。没有测试，错误修复会给代码库带来更多的混乱。

    It Is Too Big

    修复错误不是功能；它们必须小而专注。程序员在修复错误时很容易陷入其中，并在修复时引入一些重构。结果是补丁变得相当庞大且难以理解。我不反对重构；对于一个项目来说，这是非常重要和积极的事情，但是请在修复错误并合并之后单独进行重构。

    修复错误时不要进行重构！

    创建一个新的单元测试，复现错误并提交。无论现有代码有多丑陋，都要在现有代码库中修复错误。通过创建新的错误来要求团队改善糟糕的代码基础。如果感兴趣，将这些错误分配给自己。或者也许其他人会对修复它们和重构代码感兴趣。但所有这些将在以后的新代码审查和新合并请求中进行。

    这不是因为懒惰和不愿意修复看起来不好的东西。这是纪律问题，比良好意图更重要。

    It Solves More Than One Issue

    始终一次只修复一个问题——就是这样简单。没有例外。当一个错误修复补丁包含修复多个问题的代码更改时，很难理解哪个问题被测试，哪个问题被重现，以及它们之间的关系。将几个错误修复合并到一个拉取请求中是非常不好的做法。

    无论修复多么简单，都要与其他问题分开。单独进行审查、测试和合并。这也将增加更改的可追溯性。始终容易理解是谁进行了修复，谁审查了代码，以及何时合并（和部署）。

    A Few Valid Reasons to Reject a Bug Fix

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    存在错误时，表示某件事情并不按预期工作。错误修复基本上是对现有代码库的补丁（一个拉取请求），旨在解决问题并确保“某件事情”按预期工作。很多时候，这样的补丁修复了一个问题，但却导致了许多其他问题。我认为，有时候有必要拒绝一个错误修复，并要求作者重新进行补丁，以保护项目免受更大的问题。根据我的经验，有几个有效的原因可以拒绝这样的修复。

    It Degrades Code Coverage

    这是一种非常常见的情况：在一个地方进行了更改后，单元测试在其他地方失败了。虽然修复了错误，但一些可能无关的单元测试开始报告失败。在压力下或者只是因为懒惰，我们不修复它们；我们只是移除这些测试或者将其标记为暂时的“跳过”。问题得到解决，构建成功，所以让我们合并这个补丁并结束这一天，对吧？错了！

    尽管我尽可能支持偷懒，但这个角是我不建议你削减的。

    单元测试的目的就是在我们承受压力时防止产品出现故障。

    显然，有时候单元测试是错误的，我们不得不删除它们。在这种情况下，请不要忘记创建新的测试。

    也有一些情况下，必须在几分钟内修复错误才能将系统恢复在线状态，而修复所有单元测试需要一个小时。这种情况强烈表明产品中的测试覆盖率存在严重问题。毫无疑问，我们必须修复错误并要求测试在一段时间内停止运行。但在这种情况下，请确保在错误修复发布后，团队下一个任务是纠正那些被禁用的单元测试。我建议阅读迈克尔·菲瑟斯的《与遗留代码高效工作》(Working Effectively With Legacy Code)，该书涉及了这个非常重要的主题。

    It Doesn’t Reproduce the Issue

    有时候整个系统可能因为一行代码中的小错误而崩溃。一个明显的错误修复方法是去掉这个错误，但如果我们关心项目的质量，一个好的项目并不希望我们只是做这样的修复。问题不在于错误本身，而是缺少单元测试，在部署阶段就可以发现这个错误。

    真正的问题是代码中这个特定部分缺乏测试代码覆盖率。通过修复拼写错误，我们并没有对项目有任何帮助。相反，我们正在做出一种不负责任的行为——我们在“掩盖”真正的问题。

    因此，无论问题有多么小或者是表面上的问题，它的错误修复都必须包含一个额外的测试，首先重现该错误。如果没有这样的测试，修复错误就是项目资金的浪费。

    此外，如果没有一个单元测试能够重现问题，我们无法保证我们的错误修复不会引入更多的错误。我甚至可以说，我们进行的错误修复越多，熵就越高。而唯一降低这种不确定性的方法就是使用单元测试覆盖代码。没有测试，错误修复会给代码库带来更多的混乱。

    It Is Too Big

    Bug修复不是特性；它们必须是小而专注的。程序员在修复错误时往往会过度并在修复过程中引入一些重构。结果是补丁变得相当庞大且难以理解。我并不反对重构；对于一个项目来说，它是非常重要且积极的事情，但请在修复和合并错误之后单独进行重构。

    在修复错误时不要进行重构！

    创建一个新的单元测试，复现这个 bug，并提交它。无论现有代码库多么丑陋，都要修复这个 bug。创建新的 bug，要求团队通过这个丑陋的代码库来改进情况。如果感兴趣，可以将这些 bug 分配给自己。也许其他人也会有兴趣修复它们并对代码进行重构。但所有这些都将在以后的其他拉取请求中完成，包括新的代码审查和新的合并。

    这不是关于懒惰和不愿修复看起来不好的事情。这是关于纪律，比良好意图更为重要。

    It Solves More Than One Issue

    一次只修复一个问题——就是这么简单。没有例外。当一个错误修复补丁包含修复多个问题的代码更改时，很难理解哪个问题被测试，哪个问题被复现，以及它们之间的关系。将几个错误修复合并到一个拉取请求中是一种非常糟糕的做法。

    无论修复有多简单，都要将其与其他内容分开。逐个进行审查、测试和合并。这样还可以增加变更的可追溯性。始终能够清楚地了解是谁进行了该修复，谁审查了代码，以及何时进行了合并（和部署）。

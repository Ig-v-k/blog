    Wikipedia's Definition of a Software Bug Is Wrong

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在本文撰写时，维基百科上的内容如下：

    软件错误是计算机程序或系统中的错误、缺陷、故障或错误，导致其产生不正确或意外的结果，或以意外的方式行为。

    我认为这是不完整的。这个定义完全排除了与“可维护性”和“可重用性”等方面相关的“非行为”缺陷。

    正如您所了解的那样，每个软件都有功能需求和非功能需求。功能需求告诉我们软件必须做什么，而非功能需求则记录软件如何做到这一点。例如，以下是一个功能需求：

    The user can generate a PDF report.

    如果我们的软件不能生成PDF报告，而是崩溃了，那就是一个功能性错误。如果它生成的是一个空白页面或纯文本文档，而不是PDF报告，那就是一个功能性错误。如果根本没有“生成PDF报告”的按钮，用户根本无法启动PDF生成过程，那就是一个功能性错误。

    这是一个非功能性需求的例子：

    PDF report generation must take less than 100ms.

    如果我们的软件生成了一个完全正确的PDF报告，但需要一分钟的时间，那就是一个非功能性的错误。

    到目前为止还不错，因为维基百科给出的错误定义完美地涵盖了它们两者——如果出现这些错误，将会导致我们的软件“产生错误或意外结果，或以意外的方式运行”。重点在于“产生”和“运行”这两个词。它们假设软件正在做某些事情，而我们正在观察它的行为。

    然而，并不仅仅如此。

    可维护性如何？我可能会有这种非功能性需求：

    The source code of the PDF generator must be
easy to maintain and extend for an average
Java programmer.

    这是一个相当模糊的要求，但你能理解的意思。

    可维护性和可重用性是任何现代软件程序非常关键的非功能组成部分，尤其考虑到市场上劳动力成本非常高。往往，确保软件的可维护性比速度更重要。如果它可维护但速度慢，我们可以找到新的程序员来改进代码。如果它速度快但不可维护，我们将无法对其进行任何操作，如果需要新增功能，我们将不得不从头重写它。阅读更多关于这个主题的内容，请参阅《你是黑客还是设计师？》。

    维基百科对软件错误的定义并未涵盖可维护性和可重用性缺陷。这导致了一个常见的困惑来源——不一致的代码风格并不是一个错误（请参见本帖下的讨论）。

    That is wrong.

    一致性的代码风格是一个软件缺陷，同样的，不完整的文档、缺乏文档、过于复杂的代码、缺乏代码风格指南等也都是软件缺陷。

    我会像这样重新编写维基百科上的软件错误定义段落：

    软件漏洞是计算机程序或系统中的错误、缺陷、失败或故障，导致其违反至少一个功能或非功能要求。

    这个定义在我看来更准确。

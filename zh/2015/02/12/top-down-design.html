    Code For the User, Not for Yourself

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    首先，无论使用何种方法论，我们都是为用户（也称为客户、项目赞助人、最终用户或客户端）编写软件。其次，无论使用何种方法论，我们都是逐步编写，逐个发布功能和错误修复。也许我在这里说的是一些非常明显的事情，但重要的是要记住，每个新版本首先应该满足用户的需求，而不是我们程序员的需求。换句话说，我们将一个大任务分解为较小的部分的方式应该以用户为目标，这就是为什么我们总是从上而下工作的原因。让我们通过一个实际的例子来看看我所说的意思。

    假设一个朋友与我签约，要求我开发一个非常类似于wc的命令行单词计数工具。他承诺支付我200美元作为报酬，并我答应将产品分两个阶段交付给他——先是alpha版本，再是beta版本。我答应在星期六发布alpha版本，星期日发布beta版本。他将在第一次发布后支付我100美元，剩余金额则在第二次发布后支付。

    我会用C语言编写，他将以现金支付。

    这个工具非常简单，我只用了几分钟就写好了。看一下吧：

    
# include <stdio.h>

# include <unistd.h>

int main() { char ch; int count = 0; while (1) { if (read(STDIN_FILENO, &ch, 1) <= 0) { break; } if (ch == ' ') { ++count; } } if (count > 0) { ++count; } printf("%d\n", count); return 0; } ```

但是让我们保持专业，不要忘记构建自动化和单元测试。这里是一个简单的`Makefile`，可以同时完成这两项任务。

    all: wc test wc: wc.c gcc -o wc wc.c test: wc echo ‘’ | ./wc | grep ‘0’ echo ‘Hello, world! How are you?’ | ./wc | grep ‘5’

    
现在我在命令行中运行`make`命令，并得到以下输出：

    $ make echo ‘’ | ./wc | grep ‘0’ 0 echo ‘Hello, world! How are you?’ | ./wc | grep ‘5’ 5 ```

    All clean!

    我准备拿到我的200美元。等等，协议是提供两个版本并分两次付款。让我们回过头来思考一下——我们如何将这个小工具分成两个部分呢？

    首先考虑，让我们先发布工具本身，然后再构建自动化和测试。这是一个好主意吗？我们能否在运行测试之前交付任何软件？如果我不将测试与其一起发布，我怎么知道它是否正常工作？如果我发布没有测试的任何东西，我的朋友会怎么看待我？这将是一个彻底的尴尬。

    好的，先发布Makefile然后再发布wc.c。但是，我朋友手里只有一些测试，没有产品，他会做什么呢？这个第一次发布将完全毫无意义，我也得不到我的100美元。

    现在我们要谈到这篇文章的重点了。我的意思是，每一个新的增量都必须给产品增加一些价值，这个价值是由顾客来感知的，而不是由我们程序员来评判。Makefile 绝对是一种有价值的工具，但对我的朋友来说它并没有任何价值。他不需要它，但我需要它。

    这是我要做的事情。我将发布一个由测试支持但完全是虚拟实现的工具框架。看一下：

    ```c

    include

    int main() { printf(“5\n”); return 0; } ```

    我会相应地修改 Makefile。我会禁用第一个测试以确保构建通过。

    我的工具有效吗？是的，有效。它能统计单词吗？对于某些输入是的。它对我的朋友有价值吗？显然是的！他可以从命令行运行它，并且可以将文件作为输入传递。不过，无论怎样，他总能得到统计结果为”5”。这有点糟糕，但这只是个Alpha版本。他并不指望它能完美运行。

    然而，它可行，有测试支持，并且已经适当打包。

    我刚刚所做的是一种自上而下的设计方法。首先，我创建了一些能为我的客户提供价值的东西。我确保它也满足了我的技术目标，比如适当的单元测试覆盖率和构建自动化。但对我来说，最重要的目标是确保我的朋友得到了某种东西……并付给了我。

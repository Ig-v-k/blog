    Code For the User, Not for Yourself

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    首先，无论采用何种方法论，我们都是为我们的用户（也称为客户、项目发起人、最终用户或客户）编写软件。其次，无论采用何种方法论，我们都是逐步编写代码，逐个发布功能和修复错误。也许我在这里说的是一些非常明显的事情，但重要的是要记住，每个新版本首先应满足用户的需求，而不是我们程序员的需求。换句话说，我们将一个大任务分解为较小的部分的方式应该是面向用户的，这就是为什么你总是从上至下工作。让我们通过一个实际的例子来理解我的意思。

    假设我受朋友委托创建一个与wc非常相似的命令行单词计数工具。他答应支付我200美元作为报酬，并且我答应分两个阶段交付产品——一个是alpha版本，另一个是beta版本。我承诺在周六发布alpha版本，并在周日发布beta版本。他将在第一次发布后支付我100美元，剩余的款项将在第二次发布后支付。

    我会用C语言编写，他会用现金付款。

    这个工具非常原始，我只花了几分钟写完。请看一下：

    
# include <stdio.h>

# include <unistd.h>

int main() { char ch; int count = 0; while (1) { if (read(STDIN_FILENO, &ch, 1) <= 0) { break; } if (ch == ' ') { ++count; } } if (count > 0) { ++count; } printf("%d\n", count); return 0; } ```

但让我们保持专业，不要忘记构建自动化和单元测试。这是一个简单的`Makefile`，可以同时完成这两个任务：

    all: wc test wc: wc.c gcc -o wc wc.c test: wc echo ‘’ | ./wc | grep ‘0’ echo ‘Hello, world! How are you?’ | ./wc | grep ‘5’

    
现在我在命令行中运行`make`命令，得到了以下输出：

    $ make echo ‘’ | ./wc | grep ‘0’ 0 echo ‘Hello, world! How are you?’ | ./wc | grep ‘5’ 5 ```

    All clean!

    我准备好拿到我的200美元了。等等，交易是要提供两个版本，并分两次付款。我们先退后一步思考一下——我们如何将这个小工具分成两个部分呢？

    第一思考，让我们首先发布工具本身，然后再构建自动化和测试。这是个好主意吗？我们能够在没有先运行测试的情况下交付软件吗？如果我不将测试与软件一起发布，我怎么能确定它能正常工作？我的朋友会对我没有测试发布任何东西有什么看法？这将是一种彻头彻尾的尴尬。

    好吧，让我们首先发布 Makefile，然后是 wc.c。但是，我朋友手上只有几个测试而没有产品，他会做什么呢？这个第一次发布将完全没有意义，我也拿不到我的100美元。

    现在我们要说到这篇文章的要点了。我的意思是，每个新的增量都必须为产品增加一些价值，这价值是由顾客来感知的，而不是我们程序员。Makefile 绝对是一个有价值的工具，但对我的朋友来说却没有任何价值。他不需要它，但我需要它。

    这是我要做的事情。我将发布一个工具的框架，由测试支持，但是实现完全是虚拟的。请看一下：

    ```c

    include

    int main() { printf(“5\n”); return 0; } ```

    而且我会相应地修改Makefile。我将禁用第一个测试以确保构建成功。

    我的工具有效吗？是的，有效。它可以计算单词吗？是的，对于某些输入有效。对我的朋友来说有价值吗？显然有！他可以从命令行运行它，并且可以将文件作为输入传递。尽管他总是得到计数为”5”的结果，但这有点令人失望，但这只是一个alpha版本。他并不指望它能完美运行。

    然而，它能够正常工作，有经过测试的支持，并且已经正确打包。

    我刚刚所做的是一种自上而下的设计方法。首先，我创建了一些能为我的客户提供价值的东西。我确保它还满足了我的技术目标，如适当的单元测试覆盖率和构建自动化。但对我来说最重要的目标是确保我的朋友收到了东西…并且付给了我。

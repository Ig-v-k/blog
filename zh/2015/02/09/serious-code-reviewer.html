    Four NOs of a Serious Code Reviewer

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    代码审查（又称同行审查）对于每个认真的软件开发团队来说，必须是一种强制性的实践。我希望对此没有争议。有些人会进行合并前的代码审查，以防止意外错误影响其主/开发分支。其他人会进行合并后的定期审查，以发现作者引入的错误和不一致之处。还有一些人既在合并前进行审查，又定期进行审查。代码审查非常类似于白盒测试技术，测试人员可以完全访问软件源代码以查找缺陷。无论哪种情况，代码审查都是提高质量和增强团队动力的重要工具。

    然而，做到这些并不是那么简单。我甚至会说，做错它们非常容易且舒适。我见过的大多数代码审查和审查人员都犯了相似的错误。这就是为什么我决定总结出我认为一个好的审查人员应该具备的四个基本原则。希望你觉得它们有用。

    No Fear

    一个严肃的代码审查者应该摒弃几种不同的恐惧。第一种也是最常见的就是怕冒犯代码的作者。”我最好闭上眼睛，假装我今天没看见她的错误，这样明天她就会忽略我的错误”—这种态度就是这种恐惧产生的。不用说，这是非生产性的，会降低代码质量和团队士气。我的建议是：直接、诚实、坦率。如果你不喜欢这段代码，就是不喜欢。你不应该在意你的意见会被代码的作者如何接受。

    如果你对同事有这样的感觉，说明管理模式存在问题。你害怕团队因为你“不是团队合作的人”而拒绝你，这个标签是由团队中最弱的成员贴上的，而不是项目赞助商。赞助商支付你来制作高质量的软件。赞助商不在乎你提高质量的意图冒犯了其他人多少，其他人对此并不在意。赞助商希望他的钱能产生可销售并能带来利润的可交付成果。赞助商并不为了你在办公室交朋友而付钱给你。

    下一种恐惧听起来像这样：“如果我拒绝这个代码，我会延迟发布”再次强调，这样的态度对整个项目来说是极其不利的。你会接受这个代码，并对其中你不喜欢的部分视而不见。代码将会进入下一个发布版本，我们将会更早地进行发货。你不会成为一个瓶颈，也没有人会说因为那个教条式的代码审查者，我们延迟了发布并且损失了一些现金。你会成为一个好的团队合作者，对吗？错！

    正如我之前提到的，专业的团队成员了解自己在项目中的个人角色，不会为他人掩护。如果拒绝糟糕的代码导致发布延迟，那是其作者的责任。你的专业责任是让这个错误变得明显。这就是你帮助团队学习和提高的方式。

    我认为明显的是，团队的教育和提升首先需要摒弃不合格的程序员并提拔优秀的。诚实无畏的代码审查者帮助团队学习和提高。

    还有一种担忧是这样表达的：”我可能错了，他们会嘲笑我” 更糟糕的是，他们可能会察觉到我的知识不足。他们可能会看到我不知道自己在做什么。最好还是保持安静，假装代码中没有错误。至少这样我就不会因为愚蠢的评论而尴尬自己。你知道如果保持沉默，看起来更聪明会更容易，对吗？错了！

    这个项目并不是为了让你看起来很好而付给你薪水。你得到薪水不是因为团队喜欢你，而是因为你每天都能交付可执行成果。作为专业人士，你的职责是为项目做出最好的决策，忽略所有人的意见，包括你的上司意见。就像“一个快乐的老板是一个虚假的目标”一样，我会说团队的尊重也是一个虚假的目标。不要试图去赢得尊重。相反，编写整洁的代码，尊重自然会随之而来。

    让我再强调一下：不要害怕因为对别人的代码发表不正确和愚蠢的评论而让自己尴尬。要对项目忠诚，而不是对周围人的期望。他们期望你聪明和机智，但项目期望你说出你对代码的真实想法。所以不要在意他们的意见；做正确的事情，说出你真正的想法。

    No Compromise

    好的，你勇敢地表达了对代码的想法，并且毫不犹豫地拒绝了它。你正在审查的分支不太好，并且你解释了原因。你要求作者在这里进行重构，在那里进行重写。接下来会发生什么呢？

    他（她）将试图与您达成协议。这是很自然的，几乎在我们团队的每个部门都有发生。代码的作者也是一位专业开发者，并且他也毫不畏惧。因此，他坚持认为他的实现方法是正确的，而您的想法是错误的。在这种情况下，一个专业的代码审查员应该做什么呢？

    最糟糕的事情（就像在任何冲突解决中）就是妥协。这比糟糕的代码更快地破坏了质量。妥协是一种冲突解决技巧，双方同意出于停止冲突而不得到他们想要的东西。换句话说，”为了停止争斗，让我们和平相处” 这是最糟糕的方法。

    不要选择一种糟糕的妥协，而是有三种专业的解决方案可以避免在代码问题上的争论。

    • “你是对的；我收回我的评论！” 这种情况可能发生，并且应该经常发生。你应该准备好承认自己的错误。你不喜欢这段代码，但它的作者向你解释了它的好处，而你接受了这个逻辑—不是因为你想停止与他争论，而是因为你真正理解了这个逻辑并接受了它。愿意说“我错了”是一个成熟和认真的开发者的第一个迹象。

    • “我永远不会接受这个，就这样！” 有些代码是值得这样做的，用这种方式解决冲突并没有错。对手可能会接受这个情况并重新编写一切。同时也能学到一些东西。

    • “让我们按照架构师的建议去做！” 在每个项目中，都有一位软件架构师做出最后的决策。倾听他的意见，并获得他的最终决定。邀请他参与讨论，并请他在冲突中选择一方。一旦他告诉你你错了，接受这个决定，并试着从中学习。

    因此，要么坚定地坚持自己的立场并为之奋斗，要么承认自己错了。无论哪种方式，都不要妥协！

    不要误会我，这不是指固执地坚持你的立场，不管情况有多糟糕。要灵活并且能够在场上学习。你的立场在谈判过程中可能会发生变化，但不要接受任何你不喜欢的事情。你可以在完全相信对手是正确的时候或者建筑师说服你时退出冲突。没有中间地带。

    No Bullshit

    再次，你毫不畏惧地表示应该以不同的方式设计一个方法。你的对手，代码的作者，回应说他不这么认为。你再次审视后决定坚持你的立场。你仍然认为自己是正确的，不打算妥协。现在怎么办？现在召唤一个架构师还为时过早，所以试着说服你的对手。

    在大多数情况下，说服就是教育。你知道他不知道的东西。这就是为什么他创造了你不喜欢的方法。你们中的一个人需要额外的教育。这是你成为同事的老师的机会。要成为一个有效的教师，你需要提供证据。你需要基于逻辑，并确保他理解并接受它。

    准备好展示链接、文章、书籍、报告、示例等。仅仅说“我知道这个，因为我已经写了15年的Java”是不够的。此外，这种论点只会让你的说服力减弱。

    如果你没有足够令人信服的证据，再次思考一下——也许你是错的。

    此外，请记住，你的工作是证明你正在审查的代码是有问题的。代码的作者不需要证明任何事情。他的代码从一开始就是很棒的。审查者的工作是展示为什么以及如何实际上并非如此。换句话说，你是原告，他是被告，而不是相反的情况。

    No Offense

    这是最后也是最困难的原则要遵循。无论代码有多糟糕，对手有多固执，你都必须保持职业素养。说实话，有时我发现这非常困难。在Zerocracy，我们在分布式团队中工作，每周雇佣一些新人。尽管我们有筛选标准，但其中一些人似乎很难相处。

    大约一年前，我遇到了一个有趣的情况。一个新人应该在一个已有的Java库中创建一个小的新功能（大约20到30行代码）。他在提交拉取请求之前给我（作为代码评审员）看了几百行代码。那些代码是完全糟糕的，显然不是他写的。我立即意识到他找到了代码并复制了过来。但是我能怎么办呢？我怎么能拒绝它，而不说他的态度对于一个专业开发人员来说是不可接受的呢？我不得不花一些时间客观地批评他的代码风格、设计等等。为了向他展示要修复这些问题，他应该删除那些垃圾代码，从头重新编写。在那个任务之后，我再也没有见过他。

    我的观点是，当你与专业人士打交道时，成为专业人士是很容易的。不幸的是，情况并非总是如此。但无论你面前的代码有多糟糕，都要保持耐心和有说服力。

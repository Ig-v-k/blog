    Four NOs of a Serious Code Reviewer

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    代码审查（又称同行审查）对于每个严肃的软件开发团队来说，都必须是一项强制性的实践。我希望对此没有争议。有些人在合并前进行代码审查，以防止意外错误影响他们的主分支/开发分支。其他人在合并后进行定期审查，以发现由作者引入的错误和不一致之处。还有一些人两者兼而有之，合并前进行审查，然后定期进行审查。代码审查与白盒测试技术非常相似，测试人员可以完全访问软件源代码来寻找缺陷。无论哪种情况，代码审查都是提高质量和增强团队动力的重要工具。

    然而，要正确进行代码审查并不那么简单。我甚至可以说，很容易做错而感到舒适。我看到的大多数代码审查和审查人员都犯了相似的错误。这就是为什么我决定总结我所认为的一个良好审查者的四个基本原则。希望你会觉得它们有帮助。

    No Fear

    一个认真的代码审查人员应该放弃几种不同的恐惧。第一个也是最常见的是害怕冒犯代码作者的恐惧。”我最好闭上眼睛，假装今天没看到她的错误，这样明天她就会忽略我的错误”—这就是这种恐惧产生的态度。不用说，这是适得其反的，会降低代码质量和团队士气。我的建议是：直接、诚实和坦率。如果你不喜欢这段代码，就是不喜欢。你不应该在意你的观点会被代码作者如何接受。

    如果你对同事有这样的感觉，说明管理模式有问题。你害怕被团队拒绝，被认为”不是团队合作者”，这是团队中最弱的成员贴在你身上的标签，而不是项目赞助人。赞助人付钱让你制作高质量的软件。赞助人不在乎你提高质量的意图会冒犯其他人，其他人在乎得更少。赞助人想要他的钱制作可卖给客户并带来利润的可交付成果。赞助人不付钱让你在办公室交朋友。

    下一种恐惧的表现是这样的：“如果我拒绝这段代码，我会延迟发布”同样，毋庸置疑，这种态度会对整个项目造成重大损害。你会接受这段代码，对其中不喜欢的部分视而不见。这段代码会进入下一次发布，我们会更快地发货。你不会成为瓶颈，也没人会说因为那个教条的代码审查人员，我们延迟了发布并损失了一些钱。你会成为一个好的团队合作者，对吗？错了！

    正如我之前提到的，专业的团队合作者了解他或她在项目中的个人角色，不为任何人掩盖错误。如果拒绝糟糕的代码延迟了发布，那是其作者的责任。你的职业责任是让这个错误显现出来。这就是你帮助团队学习和改进的方式。

    我认为很明显，团队的教育和改进首先需要摆脱糟糕的程序员并提拔优秀的程序员。诚实而无畏的代码审查人员帮助团队学习和改进。

    还有一种恐惧是这样表达的：“我可能错了，他们会嘲笑我”更糟糕的是，他们可能会发现我知识的不足。他们可能会看到我不知道自己在做什么。保持沉默，假装代码中没有错误可能会更好。至少那样我就不会因为愚蠢的评论而尴尬自己。你知道如果闭嘴就能显得聪明得多，对吗？错了！

    项目不是为了让你看起来好而付钱给你的。你领工资不是因为团队喜欢你，而是因为你每天产出可交付成果。你的职业责任是为项目做出最好的贡献，忽略每个人的意见，包括你的上司的意见。就像《一个快乐的老板是一个虚假的目标》一文所说，我想说团队的尊重是一个虚假的目标。不要试图赢得尊重。相反，编写干净的代码，尊重将自然而然地到来。

    让我重申一下：不要害怕因为对别人的代码发表不正确和愚蠢的评论而尴尬自己。忠于项目，而不是周围人的期望。他们期望你聪明而机智，但项目期望你说出你对代码的真实看法。所以不要理会他们的意见；做正确的事情，说出你真正的想法。

    No Compromise

    好的，你毫不畏惧地表达了对代码的看法，并且直接否决了它。你正在审查的分支不好，并且你解释了原因。你要求代码作者在这里进行重构，那里进行重写。接下来呢？

    他或她会试图与你达成协议。这是自然而然的，几乎在我们团队的每个分支中都会发生。代码的作者也是一位专业开发者，他也毫不畏惧。因此，他坚持认为他的实现方法是正确的，而你的想法是错误的。在这种情况下，专业的代码审查员应该怎么做？

    最糟糕的事情（就像在任何冲突解决中一样）是妥协。这比糟糕的代码更快地破坏了质量。妥协是一种冲突解决技巧，双方同意不为了结束冲突而放弃各自想要的东西。换句话说，“让我们和平相处，停止争斗”。这是最糟糕的方法。

    与其妥协，从代码争论中有三种专业的出路：

    • 「你是对的；我收回我的评论！」这种情况可能发生，而且应该经常发生。你应该准备好承认自己的错误。你虽然不喜欢这段代码，但它的作者向你解释了它的好处，你接受了这个逻辑——不是因为你想停止与他争吵，而是因为你真正理解并接受了这个逻辑。愿意说「我错了」是一个成熟和认真的开发者的第一个标志。

    • “我绝对不会接受这个，就是这样！” 有些代码值得这样处理，这样解决冲突并没有错。对手可能会接受这个情况并重新编写一切。这样还能学到一些东西。

    • “让我们按照架构师的建议行事！” 在每个项目中，都有一个软件架构师来做最终决策。向他请教意见并获取他的最终决定。邀请他参与讨论，并请他在冲突中选择一方。一旦他告诉你你的错误，接受这个决定并试着从中学习。

    因此，要么坚定立场并为之奋斗，要么承认自己错了。不管怎样，但不要妥协！请别误会，这并不是固执地坚持自己的观点，不管情况有多糟糕。要灵活变通，边谈判边学习。你的立场可能会且应该在谈判过程中发生变化，但不要接受任何你不喜欢的东西。你可以在完全确信对手正确或建筑师表示同意时退出冲突。没有中间地带。

    No Bullshit

    再次，你毫不畏惧地说一种方法应该有不同的设计。你的对手，代码的作者，回答说他不这么认为。你再次仔细看了一下，并决定坚持自己的立场。你仍然认为自己是对的，不会妥协。现在怎么办？现在找架构师还为时过早，所以试着说服你的对手。

    在大多数情况下，说服就是教导。你知道他不知道的东西。这就是他按照你不喜欢的方式创建那种方法的原因。你们中的一位需要额外的教育。这是你成为同事导师的机会。要成为一个有效的导师，你需要提供证据。你需要为你的逻辑提供依据，并确保他理解并接受。

    准备好展示链接、文章、书籍、报告、示例等。仅仅说“我知道这个，因为我已经写了15年的Java代码”是不够的。而且，这种类型的论证只会使你的说服力减弱。

    如果你没有足够的有说服力的证据，再次思考一下，也许你是错的。

    此外，请记住，你的工作是证明你正在审查的代码有问题。代码的作者不需要证明任何东西。他的代码默认就是很好的。审查人员的工作是展示为什么以及如何事实并非如此。换句话说，你是原告，他是被告。不是反过来的。

    No Offense

    这是最后也是最难遵守的原则。无论代码有多糟糕，对手有多固执，你都必须保持职业精神。说实话，有时我也觉得这很难。在Zerocracy，我们在分布式团队中工作，并每周雇用几个新人。尽管我们有筛选标准，但其中一些人似乎相当难对付。

    大约一年前，我遇到了一个有趣的情况，一个新人应该在现有的Java库中创建一个小的（20到30行代码）新功能。他在提交合并请求之后（我是代码审查人员），放入了几百行代码。那段代码是绝对的垃圾，显然不是他写的。我立即明白他是从某处找到并复制的。但我能怎么办呢？我怎么能拒绝它，而不说他的态度对一个专业开发者来说是不可接受的呢？为了让他明白要解决这一切，他应该删除垃圾代码并从头开始重新编写，我不得不花时间客观地指责他的代码风格、设计等方面。之后，我再也没见过他。

    我的观点是，当你和专业人士打交道时，保持职业精神很容易。不幸的是，情况并非总是如此。但无论你面前的代码有多糟糕，都要耐心和有说服力。

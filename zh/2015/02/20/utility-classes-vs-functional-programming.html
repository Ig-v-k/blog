    Utility Classes Have Nothing to Do With Functional Programming

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我最近被指责反对函数式编程，原因是我将实用类称为反模式。这完全是错误的！嗯，我确实认为它们是一个糟糕的反模式，但它们与函数式编程没有任何关系。我认为有两个基本原因。首先，函数式编程是声明式的，而实用类方法是命令式的。其次，函数式编程基于λ演算，其中函数可以赋值给变量。实用类方法在这个意义上不是函数。我一会儿会解码这些陈述。

    在Java中，基本上有两个有效的选择来替代这些被Guava、Apache Commons等积极推广的丑陋的实用类。第一个选择是使用传统类，第二个选择是Java 8 lambda。现在让我们看看为什么实用类与函数式编程相去甚远，并且这种误解是从哪里来的。

    这是一个典型的Java 1.0实用类的例子：Math。

    public class Math {
  public static double abs(double a);
  // a few dozens of other methods of the same style
}

    当你想计算浮点数的绝对值时，以下是如何使用它的方法：

    double x = Math.abs(3.1415926d);

    这有什么问题呢？我们需要一个函数，而我们从类 Math 中获取它。这个类里面有许多有用的函数，可以用于许多典型的数学运算，比如计算最大值、最小值、正弦、余弦等等。这是一个非常流行的概念；看看任何商业或开源产品就知道了。自从 Java 诞生以来，这些实用类就被广泛使用（这个 Math 类在 Java 的第一个版本中被引入）。嗯，从技术上讲并没有什么问题。代码可以正常工作。但它不是面向对象编程。相反，它是命令式和过程式的。我们在意吗？好吧，这取决于你自己的决定。让我们看看有什么区别。

    基本上有两种不同的方法：声明式和命令式。

    命令式编程侧重于以改变程序状态的语句来描述程序的操作方式。我们刚才看到了一个命令式编程的例子。这是另一个例子（这是纯粹的命令式/过程式编程，与面向对象编程无关）。

    public class MyMath {
  public double f(double a, double b) {
    double max = Math.max(a, b);
    double x = Math.abs(max);
    return x;
  }
}

    声明式编程关注于程序应该达到的目标，而不规定如何根据需要采取行动的顺序来实现。这是同样的代码在Lisp中的表达方式，Lisp是一种函数式编程语言：

    (defun f (a b) (abs (max a b)))

    有什么问题？只是语法上的差异吗？其实不然。

    有关命令式和声明式风格之间的区别有许多定义，但我将尝试给出自己的定义。在这个f函数/方法的场景中，基本上有三个角色相互作用：一个买家，一个结果的打包者，以及一个结果的消费者。假设我像这样调用这个函数：

    public void foo() {
  double x = this.calc(5, -7);
  System.out.println("max+abs equals to " + x);
}
private double calc(double a, double b) {
  double x = Math.f(a, b);
  return x;
}

    在这里，calc() 方法是一个买家，Math.f() 方法是结果的打包者，而 foo() 方法是消费者。无论使用哪种编程风格，这个过程中总是有这三个角色参与：买家、打包者和消费者。

    想象一下，你是一个买家，想为你的（女男）朋友购买一份礼物。第一种选择是去商店，支付50美元，让他们为你包装香水，然后送到朋友那里（并得到一个回吻）。这是一种命令式风格。

    第二个选择是去店里，支付50美元，然后获得一张礼品卡。然后您将此卡片交给朋友（并获得一吻作为回报）。当他或她决定将其兑换为香水时，他或她将会去店里取得。这是一种陈述式风格。

    See the difference?

    在第一种情况下，即命令式，你会迫使包装商（一个美容店）去找到香水的存货，将其包装好，并将其作为一种随时可用的产品呈现给你。而在第二种情况下，即陈述式，你只是从商店那里得到一个承诺，即当有必要时，员工将会找到香水的存货，将其包装好，并提供给那些需要的人。如果你的朋友从未使用过那张礼品卡去光顾该店，那么香水将会保持在库存中。

    此外，您的朋友可以将该礼品卡作为一个产品本身使用，而无需访问商店。他/她可以将其作为礼物送给其他人，或者仅仅将其与另一张卡或产品进行交换。这张礼品卡本身就变成了一个产品！

    所以区别在于消费者所得到的东西——要么是一个已经可以使用的产品（命令式），要么是一个可以之后兑换成真实产品的代金券（陈述式）。

    实用类，例如JDK中的Math或Apache Commons中的StringUtils，返回立即可用的产品，而Lisp和其他函数式语言中的函数则返回”凭证”。例如，在Lisp中调用max函数时，只有当您实际开始使用时，两个数字之间的实际最大值才会被计算出来。

    (let (x (max 1 5))
  (print "X equals to " x))

    直到这个print开始向屏幕输出字符之前，函数max不会被调用。这个x是在你尝试“购买”介于1和5之间最大值时返回给你的“凭证”。

    请注意，将一个Java静态函数嵌套到另一个函数中并不能使它们成为声明式函数。代码仍然是命令式的，因为它的执行立即产生结果。

    public class MyMath {
  public double f(double a, double b) {
    return Math.abs(Math.max(a, b));
  }
}

    “好的，”你可能会说，”我明白了，但是为什么声明式风格比命令式风格更好？有什么大不了的吗？”我就要说到这个了。让我先展示一下函数式编程中的函数和面向对象编程中的静态方法之间的区别。正如上面提到的，这是实用类和函数式编程之间的第二个重要区别。

    在任何函数式编程语言中，你可以这样做：

    (defun foo (x) (x 5))

    然后，稍后，您可以将其称为x：

    (defun bar (x) (+ x 1)) // defining function bar
(print (foo bar)) // passing bar as an argument to foo

    在函数式编程的术语中，Java中的静态方法并不是真正的“函数”。你不能像这样使用静态方法。你不能将静态方法作为参数传递给另一个方法。基本上，静态方法是一组按照唯一名称分组的过程或简单的Java语句。唯一访问它们的方法是调用过程并将所有必要的参数传递给它。该过程将计算某些内容并返回一个立即可用的结果。

    现在我们来到了最后一个问题，我能听到你在问：”好的，实用类并不是函数式编程，但它们看起来像函数式编程，它们工作得非常快，而且非常容易使用。为什么不使用它们呢？为什么要追求完美，当Java的20年历史证明实用类是每个Java开发者的主要工具呢？”

    除了我经常被指责为面向对象编程(OOP)的教条主义外，实际上还有几个非常实用的理由（顺便说一下，我确实是一个OOP教条主义者）：

    可测试性。对于实用类中的静态方法的调用是硬编码的依赖项，无法为测试目的而中断。如果您的类调用FileUtils.readFile()，我将无法在不使用真实磁盘文件的情况下对其进行测试。

    效率。由于其命令式特性，实用类比起其声明式替代方案要低效得多。它们在此刻立即进行所有计算，即使尚不必要，也会占用处理器资源。StringUtils.split()不是返回一个将字符串拆分成块的承诺，而是立即进行拆分。而且它会将其拆分成所有可能的块，即使只有第一个块被“购买者”需要。

    易读性。实用类往往非常庞大（试着阅读Apache Commons的StringUtils或FileUtils的源代码）。关注点分离的整个概念，使得面向对象编程如此美丽，却在实用类中缺失。它们只是将所有可能的过程放入一个庞大的.java文件中，当超过十几个静态方法时，这种做法变得非常难以维护。

    总之，让我重申一下：Utility类与函数式编程无关。它们只是一些静态方法的集合，它们是命令式的过程。尽量远离它们，无论你需要声明多少个小的实例，都使用坚实、内聚的对象。

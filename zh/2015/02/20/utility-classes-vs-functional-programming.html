    Utility Classes Have Nothing to Do With Functional Programming

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我最近被指责反对函数式编程，原因是我称实用类为反模式。这完全是错误的！嗯，我确实认为它们是可怕的反模式，但与函数式编程无关。我认为有两个基本原因。首先，函数式编程是声明式的，而实用类方法是命令式的。其次，函数式编程基于λ演算，函数可以赋值给变量。实用类方法在这个意义上不是函数。我会在一分钟内解释这些陈述。

    在Java中，基本上有两种有效的替代方案可以取代由Guava、Apache Commons和其他库积极推广的这些丑陋的工具类。第一种是使用传统的类，第二种是Java 8 lambda。现在让我们看看为什么工具类与函数式编程根本不搭边，以及这种误解是从哪里来的。

    这是一个典型的实用类的例子Math，来自Java 1.0。

    public class Math {
  public static double abs(double a);
  // a few dozens of other methods of the same style
}

    当你想计算浮点数的绝对值时，可以使用它的方法如下：

    double x = Math.abs(3.1415926d);

    这有什么问题？我们需要一个函数，并且我们从Math类中获取它。这个类里面有很多有用的函数，可以用于许多典型的数学运算，比如计算最大值、最小值、正弦、余弦等等。这是一个非常流行的概念；只要看看任何商业或开源产品就知道了。自从Java被发明以来，这些实用类就被广泛使用（这个Math类是在Java的第一个版本中引入的）。嗯，从技术上讲没有任何问题。代码会正常工作。但它并不是面向对象的编程。相反，它是命令式和过程式的。我们在乎吗？嗯，这取决于你自己决定。让我们看看区别是什么。

    基本上有两种不同的方法：声明性和命令性。

    命令式编程关注以改变程序状态的语句描述程序的运行方式。我们刚刚看到了一个命令式编程的例子。这是另一个例子（这是纯粹的命令式/过程式编程，与面向对象编程无关）。

    public class MyMath {
  public double f(double a, double b) {
    double max = Math.max(a, b);
    double x = Math.abs(max);
    return x;
  }
}

    声明式编程 关注于程序应该完成的内容，而不指定以一系列操作的顺序如何完成。下面是相同的代码在Lisp中的表现形式，Lisp是一种函数式编程语言：

    (defun f (a b) (abs (max a b)))

    有什么要注意的吗？只是语法上的差别吗？事实并非如此。

    有关命令式和声明式风格之间差异的很多定义，但我将尝试给出我的定义。在这个f函数/方法的场景中，基本上有三个角色进行交互：买家，结果的打包者和结果的消费者。假设我像这样调用这个函数：

    public void foo() {
  double x = this.calc(5, -7);
  System.out.println("max+abs equals to " + x);
}
private double calc(double a, double b) {
  double x = Math.f(a, b);
  return x;
}

    在这里，calc()方法是一个买家，Math.f()方法是结果的包装者，而foo()方法是一个消费者。无论使用哪种编程风格，这三个角色都会参与进程：买家、包装者和消费者。

    想象一下，你是一位买家，想要为你的（女朋友男朋友）购买一份礼物。第一种选择是去商店，支付50美元，让他们为你包装香水，然后将其送给朋友（并得到一个回吻）。这是一种命令式风格。

    第二个选择是前往商店，支付50美元，并获得一张礼品卡。然后您将向朋友展示这张卡（并得到一次亲吻作为回报）。当他或她决定将其兑换为香水时，他或她将会前往商店并获得它。这是一种陈述式的方式。

    See the difference?

    在第一种情况下，也就是命令式，你强制包装商（一家美容店）去找到那款香水的库存，给它包装好，并将其作为一个可以直接使用的产品呈现给你。而在第二种情景中，也就是陈述式，你只是从商店得到一种承诺，即在必要时，工作人员将会找到库存中的香水，对其进行包装，并提供给需要的人。如果你的朋友从未使用那张礼品卡去商店购物，那香水将会保持库存状态。

    此外，您的朋友可以将该礼品卡作为一个产品使用，而无需亲自访问商店。他或她可以将其作为礼物赠送给他人，或者只是将其兑换为另一张卡片或产品。这个礼品卡本身就成为了一个产品！

    所以不同之处在于消费者所获得的东西——要么是一个可以直接使用的产品（命令式），要么是一张可以在之后兑换成真实产品的兑换券（陈述式）。

    实用类，例如JDK中的Math或Apache Commons中的StringUtils，返回可立即使用的产品，而Lisp和其他函数式语言中的函数返回“凭证”。例如，如果在Lisp中调用max函数，实际上最大值只有在您开始使用它时才会计算出来。

    (let (x (max 1 5))
  (print "X equals to " x))

    直到这个 print 开始向屏幕输出字符，函数 max 不会被调用。这个 x 是当你尝试“购买” 1 和 5 之间的最大数时返回给你的“凭证”。

    请注意，将一个Java静态函数嵌套到另一个函数中并不能使其成为声明式函数。代码仍然是命令式的，因为其执行立即产生结果。

    public class MyMath {
  public double f(double a, double b) {
    return Math.abs(Math.max(a, b));
  }
}

    “好吧，”你可能会说，”我明白了，但为什么声明式风格比命令式风格更好？有什么了不起的呢？” 我来解释一下。让我先展示一下函数式编程中的函数和面向对象编程中的静态方法之间的区别。如上所述，这是实用类和函数式编程之间的第二个重大区别。

    在任何函数式编程语言中，您都可以这样做：

    (defun foo (x) (x 5))

    然后，稍后，你可以调用那个 x：

    (defun bar (x) (+ x 1)) // defining function bar
(print (foo bar)) // passing bar as an argument to foo

    Java中的静态方法在函数式编程的术语中并不是”函数”。你不能像这样使用静态方法。你不能将静态方法作为参数传递给另一个方法。基本上，静态方法是过程，或者简单地说，是在一个唯一名称下组合的Java语句。访问它们的唯一方式是调用一个过程并传递所有必要的参数给它。该过程将计算某些内容并返回一个立即可用的结果。

    现在我们来到了最后一个问题，我能听到你在问：“好的，实用类并不是函数式编程，但它们看起来像函数式编程，它们工作得很快，而且很容易使用。为什么不使用它们呢？为什么要追求完美，当 Java 历史的 20 年证明实用类是每个 Java 开发者的主要工具呢？”

    除了常常被指责为面向对象编程（OOP）的教条主义之外，还有一些非常实际的原因（顺便说一句，我是一个OOP的教条主义者）：

    • 可测试性。在实用类中调用静态方法是硬编码的依赖项，无法为测试目的而打破。如果你的类调用了FileUtils.readFile()，我将无法在没有使用真实磁盘文件的情况下对其进行测试。

    • 效率。实用类由于其命令式特性，比起声明式的替代方案效率要低得多。它们仅仅在此刻进行所有计算，即使此时还不是必要的，也会占用处理器资源。StringUtils.split()不是返回一个将字符串分解成块的承诺，而是立即进行分解。它会将其分解为所有可能的块，即使只有第一个块是”买方”所需的。

    • 易读性。实用类往往非常庞大（尝试阅读Apache Commons的StringUtils或FileUtils源代码）。实用类缺乏关注点分离的整体概念，这也是面向对象编程如此美妙之处。它们只是将所有可能的过程放入一个巨大的.java文件中，当超过十几个静态方法时，这个文件变得无法维护。

    总结一下，让我再强调一次：实用类与函数式编程无关。它们只是一些静态方法的集合，是命令式的过程。尽量远离它们，并且使用坚实、有凝聚力的对象，无论您需要声明多少个对象，无论它们有多小。

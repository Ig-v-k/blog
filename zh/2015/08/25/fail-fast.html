    Need Robust Software? Make It Fragile

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在任何软件项目中，目标是创建一个稳定的东西。我们不希望它在用户面前出现故障。我们也不希望我们的网站显示“内部应用程序错误”而不是网页。我们希望我们的软件能够正常工作，而不是失败。这是一个完全有效和合乎逻辑的愿望，但为了实现这一点，我们必须使我们的软件尽可能脆弱。这听起来可能与直觉相反，但事实就是如此。在开发过程中，你的应用程序越脆弱，它在生产环境中就越健壮。

    脆弱指的是“快速失败”（Fail Fast）哲学，与“安全失败”（Fail Safe）相反。我相信你知道两者之间的区别，但还是让我举一个例子来提醒你。这是Fail Safe:

    public int size(File file) {
  if (!file.exists()) {
    return 0;
  }
  return file.length();
}

    这个方法被用来计算并返回文件的大小。它首先检查文件是否存在。如果文件不存在，该方法返回零。实际上，文件是不存在的，所以没有大小。我们可以抱怨文件不存在，但是为什么要这样做呢？为什么要制造噪音呢？让我们保持安静并返回零。我们不会失败，因为我们试图保持应用程序的运行。这被称为“故障安全”。

    相反，这就是快速失败的样子：

    public int size(File file) {
  if (!file.exists()) {
    throw new IllegalArgumentException(
      "There is no such file; I can't get its length."
    );
  }
  return file.length();
}

    我们找不到文件？我们不会隐藏这个事实。我们会公开和可见地表明这个情况。我们会大声叫喊和哭泣。我们会抛出一个异常。我们希望应用程序崩溃、中断和失败，因为有人给了我们一个不存在的文件。我们会抱怨和抗议。这就是所谓的快速失败。

    如果我们在任何地方都遵循哪种哲学，会使我们的软件强大且具有容错能力？只有第二种——快速失败。

    为什么？因为失败越快、越容易修复。修复过程会更简单，也更容易察觉。快速失败是一种更好的可维护性方法。代码会更清晰。追踪失败变得更加容易。所有方法都准备好在发生任何微小问题时中断并抛出异常。

    在这个例子中，如果该方法返回零，无法确定文件是否存在且其大小实际上为零，还是文件名错误且未找到。故障安全方法隐藏问题，使代码难以维护，因此难以稳定。

    在开始阶段，我们会遇到许多崩溃和错误。但是所有的错误都是可见且易于理解的。我们会修复它们并用单元测试进行覆盖。每次修复都会使我们的软件更加”稳定”并且更好地经过测试覆盖。

    以故障安全方法设计的软件在初始阶段看起来更加稳定，但会迅速退化并最终变成难以维护的混乱状态。

    设计思路采用“快速失败”的软件，开始时会频繁崩溃，但每次修复都会提高其稳定性，最终变得非常稳定和强大。

    这就是为什么脆弱性是健壮性的关键成功因素。

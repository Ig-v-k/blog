    Need Robust Software? Make It Fragile

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在任何软件项目中，目标是创建一个稳定的东西。我们不希望它在用户面前出现故障。我们也不希望我们的网站显示“内部应用程序错误”，而不是网页。我们希望我们的软件能够正常工作，而不是失败。这是一个完全合理和逻辑上正确的愿望，但为了实现这一点，我们必须尽可能让我们的软件变得脆弱。这听起来可能有些违反直觉，但事实就是如此。你的应用程序在开发阶段越脆弱，它在生产环境中就越强大。

    当我提到脆弱时，我指的是“快速失败”（Fail Fast）的理念，这是与“安全失败”（Fail Safe）相反的。我相信你知道其中的区别，但还是让我通过一个例子来提醒你。这是一个安全失败的例子：

    public int size(File file) {
  if (!file.exists()) {
    return 0;
  }
  return file.length();
}

    这个方法被用来计算并返回文件大小。它首先检查文件是否存在。如果文件不存在，该方法返回零。的确，文件不存在，所以没有大小。我们可以抱怨文件不存在，但这有什么用呢？为什么要大惊小怪呢？让我们保持安静并返回零。我们之所以不会失败，是因为我们试图保持应用程序的运行。这就是所谓的故障安全。

    相反，故障快速（Fail Fast）是这样的：

    public int size(File file) {
  if (!file.exists()) {
    throw new IllegalArgumentException(
      "There is no such file; I can't get its length."
    );
  }
  return file.length();
}

    我们找不到一个文件？我们不会隐藏这个事实。我们会公开和可见地展示这种情况。我们会大声呼喊和哭泣。我们会抛出一个异常。我们希望应用崩溃、中断和失败，因为有人给了我们一个不存在的文件。我们会抱怨和抗议。这就是所谓的快速失败。

    如果我们在任何地方都遵循哪种哲学，将会使我们的软件健壮和能够抵御故障？只有第二种——快速失败。

    为什么？因为失败越快、越容易，修复就会越快。而且修复也会更简单且更可见。快速失败对于可维护性来说是一种更好的方法。代码变得更清晰。更容易追踪故障。所有方法都已准备好在出现最微小的问题时中断并抛出异常。

    在这个例子中，如果该方法返回零，很难确定文件是否存在以及其大小是否真的为零，还是文件名错误导致找不到。快速失败的方法会掩盖问题，使代码变得不易维护，这就是为什么很难稳定。

    在开始阶段，我们将会有许多崩溃和错误。但所有这些问题都将是可见和易于理解的。我们会修复它们并用单元测试覆盖它们。每个修复都会使我们的软件更加稳定，并且测试覆盖率更好。

    以快速失败的方法设计的软件在开始阶段会经常崩溃，但随着每个修复，它的稳定性会得到改善，最终变得非常稳定和健壮。

    这就是为什么脆弱性是健壮性的关键成功因素。

    OOP Alternative to Utility Classes

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    实用类（也称为辅助类）是一种“结构”，它仅具有静态方法且不封装状态。 StringUtils，IOUtils，FileUtils来自Apache Commons; Iterables和Iterators来自Guava，以及Files来自JDK7是实用类的完美示例。

    这个设计理念在Java世界（以及C＃，Ruby等）非常流行，因为实用类提供了在各个地方使用的常见功能。

    在这里，我们想要遵循DRY原则（不要重复自己），避免重复。因此，我们将常用的代码块放入实用类中，并在需要时重复使用它们：

    // This is a terrible design, don't reuse
public class NumberUtils {
  public static int max(int a, int b) {
    return a > b ? a : b;
  }
}

    确实，这是一种非常方便的技术！

    Utility Classes Are Evil

    然而，在面向对象的世界中，实用类被视为非常糟糕的（甚至有人可能说“可怕”的）做法。

    有很多关于这个主题的讨论，举几个例子：Nick Malik的《Are Helper Classes Evil?》（链接），Simon Hart的《Why helper, singletons and utility classes are mostly bad》（链接），Marshal Ward的《Avoiding Utility Classes》（链接），Dhaval Dalal的《Kill That Util Class!》（链接），Rob Bagby的《Helper Classes Are A Code Smell》（链接）。

    此外，StackExchange 上有一些关于实用类的问题：如果“实用类”是邪恶的，我应该将通用代码放在哪里？，实用类是邪恶的。

    他们所有论点的简要总结是，实用类不是真正的“对象”；因此，它们并不适合面向对象的世界。它们大多数是从“过程式”编程中继承而来，主要是因为我们当时习惯了功能分解的范式。

    假设您同意这些论点并希望停止使用实用类，我将通过示例展示如何用适当的对象来替代这些实用类。

    Procedural Example

    例如，假设您想要读取一个文本文件，将其分割成行，修剪每一行，然后将结果保存到另一个文件中。这可以使用Apache Commons的FileUtils来完成。

    void transform(File in, File out) {
  Collection<String> src = FileUtils.readLines(in, "UTF-8");
  Collection<String> dest = new ArrayList<>(src.size());
  for (String line : src) {
    dest.add(line.trim());
  }
  FileUtils.writeLines(out, dest, "UTF-8");
}

    上面的代码看起来很简洁，但这是过程式编程，而不是面向对象的编程。我们正在操作数据（字节和位），并明确指示计算机从哪里获取数据，然后在每一行代码中明确指示将数据放在哪里。我们正在定义执行的过程。

    Object-Oriented Alternative

    在面向对象的范式中，我们应该实例化和组合对象，从而让它们以自己希望的方式和时间来管理数据。我们不应该调用辅助的静态函数，而是应该创建能够展示我们所需行为的对象。

    public class Max implements Number {
  private final int a;
  private final int b;
  public Max(int x, int y) {
    this.a = x;
    this.b = y;
  }
  @Override
  public int intValue() {
    return this.a > this.b ? this.a : this.b;
  }
}

    这个程序性调用。

    int max = NumberUtils.max(10, 5);

    将成为面向对象的。

    int max = new Max(10, 5).intValue();

    Objects Instead of Data Structures

    这是我以面向对象的方式设计相同的文件转换功能的方式：

    void transform(File in, File out) {
  Collection<String> src = new Trimmed(
    new FileLines(new UnicodeFile(in))
  );
  Collection<String> dest = new FileLines(
    new UnicodeFile(out)
  );
  dest.addAll(src);
}

    FileLines implements Collection<String> and encapsulates all file reading and writing operations. An instance of FileLines behaves exactly as a collection of strings and hides all I/O operations. When we iterate it—a file is being read. When we addAll() to it—a file is being written.

    Trimmed also implements Collection<String> and encapsulates a collection of strings (Decorator pattern). Every time the next line is retrieved, it gets trimmed.

    参与代码片段的所有类都非常小：Trimmed，FileLines和UnicodeFile。它们各自负责自己的单一功能，因此完全遵循单一职责原则。

    在我们这边，作为库的用户，这可能不那么重要，但对于开发者来说，这是必要的。相比于在一个包含80多个方法和3000多行代码的实用类FileUtils中使用readLines()方法，开发、维护和单元测试FileLines类要容易得多。真的，看看它的源代码。

    面向对象的方法实现了延迟执行。只有在需要数据时，才会读取“in”文件。如果由于某种输入/输出错误而无法打开“out”，第一个文件甚至都不会被触及。只有在调用“addAll()”之后，整个过程才会开始。

    第二个片段中的所有行，除了最后一行之外，都将较小的对象实例化并组合成较大的对象。这种对象的组合对于CPU来说相当廉价，因为它不会造成任何数据转换。

    除此之外，显而易见的是，第二个脚本在 O(1) 空间中运行，而第一个脚本在 O(n) 中执行。这是我们在第一个脚本中以过程化方式处理数据的结果。

    在面向对象的世界中，没有数据；只有对象和它们的行为！

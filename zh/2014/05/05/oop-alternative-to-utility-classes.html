    OOP Alternative to Utility Classes

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    实用类（也称为辅助类）是一种只有静态方法且不封装状态的“结构”。Apache Commons 的 StringUtils、IOUtils、FileUtils；Guava 的 Iterables 和 Iterators；以及JDK7的 Files 都是实用类的完美示例。

    这种设计思想在Java世界（以及C＃、Ruby等）非常流行，因为实用类提供了在各处使用的常用功能。

    在这里，我们希望遵循DRY原则，避免重复。因此，我们将常见的代码块放入实用类中，并在需要时重复使用它们：

    // This is a terrible design, don't reuse
public class NumberUtils {
  public static int max(int a, int b) {
    return a > b ? a : b;
  }
}

    确实，这是一种非常方便的技术！

    Utility Classes Are Evil

    然而，在面向对象的世界中，实用类被认为是一种非常糟糕（甚至有人可能会说“可怕”）的做法。

    关于这个问题已经有很多讨论，列举几个如下：Nick Malik的《Are Helper Classes Evil?》（http://blogs.msdn.com/b/nickmalik/archive/2005/09/06/461404.aspx），Simon Hart的《Why helper, singletons and utility classes are mostly bad》（http://smart421.wordpress.com/2011/08/31/why-helper-singletons-and-utility-classes-are-mostly-bad-2/），Marshal Ward的《Avoiding Utility Classes》（https://github.com/marshallward/marshallward.org/blob/master/content/avoid_util_classes.rst），Dhaval Dalal的《Kill That Util Class!》（http://www.jroller.com/DhavalDalal/entry/kill_that_util_class），Rob Bagby的《Helper Classes Are A Code Smell》（http://www.robbagby.com/posts/helper-classes-are-a-code-smell/）。

    此外，StackExchange上也有一些关于实用类的问题：《If a “Utilities” class is evil, where do I put my generic code?》（http://stackoverflow.com/questions/3339929/if-a-utilities-class-is-evil-where-do-i-put-my-generic-code）、《Utility Classes are Evil》（http://stackoverflow.com/questions/3340032/utility-classes-are-evil）。

    对于所有这些观点的干货总结是，实用类不是对象；因此，它们不适合于面向对象的世界。它们大多是从过程式编程继承而来，主要是因为我们过去习惯了功能分解的范式。

    假设您同意这些观点并且想要停止使用实用类，我将通过示例展示如何用适当的对象取代这些创建物。

    Procedural Example

    假设你想要读取一个文本文件，将其拆分成行，修剪每一行，然后将结果保存到另一个文件中。这可以使用 Apache Commons 的 FileUtils 完成。

    void transform(File in, File out) {
  Collection<String> src = FileUtils.readLines(in, "UTF-8");
  Collection<String> dest = new ArrayList<>(src.size());
  for (String line : src) {
    dest.add(line.trim());
  }
  FileUtils.writeLines(out, dest, "UTF-8");
}

    上述代码看起来很简洁；然而，这是过程式编程，而不是面向对象的编程。我们正在操作数据（字节和位），并明确指示计算机从何处检索它们，然后将它们放置在每一行代码上。我们正在定义一种执行过程。

    Object-Oriented Alternative

    在面向对象的范式中，我们应该实例化和组合对象，从而让它们在何时以及如何它们想要的方式下管理数据。我们不应该调用辅助的静态函数，而是应该创建能够展示我们所寻求行为的对象。

    public class Max implements Number {
  private final int a;
  private final int b;
  public Max(int x, int y) {
    this.a = x;
    this.b = y;
  }
  @Override
  public int intValue() {
    return this.a > this.b ? this.a : this.b;
  }
}

    这个程序性的调用：

    int max = NumberUtils.max(10, 5);

    将会变成面向对象的：

    int max = new Max(10, 5).intValue();

    Objects Instead of Data Structures

    这是我如何以面向对象的方式设计与上述相同的文件转换功能：

    void transform(File in, File out) {
  Collection<String> src = new Trimmed(
    new FileLines(new UnicodeFile(in))
  );
  Collection<String> dest = new FileLines(
    new UnicodeFile(out)
  );
  dest.addAll(src);
}

    FileLines implements Collection<String> and encapsulates all file reading and writing operations. An instance of FileLines behaves exactly as a collection of strings and hides all I/O operations. When we iterate it—a file is being read. When we addAll() to it—a file is being written.

    Trimmed also implements Collection<String> and encapsulates a collection of strings (Decorator pattern). Every time the next line is retrieved, it gets trimmed.

    所有参与代码片段的类都非常小：Trimmed，FileLines和UnicodeFile。每个类都负责自己的单一功能，因此完全遵循单一职责原则。

    对于我们作为库的用户来说，这可能不那么重要，但对于开发者来说却是必须的。与在一个包含80多个方法和3000行代码的实用类FileUtils中使用readLines()方法相比，开发、维护和单元测试FileLines类要容易得多。不信的话，看看它的源代码吧。

    面向对象的方法支持惰性执行。只有在需要数据时，才会读取in文件。如果由于某种I/O错误无法打开out文件，那么第一个文件甚至都不会被触及。只有在调用addAll()后，整个过程才会开始。

    第二个代码片段中的所有行，除了最后一行，都会实例化和组合较小的对象以形成更大的对象。这种对象组合对CPU来说相当廉价，因为它不会导致任何数据转换。

    此外，显而易见的是，第二个脚本的空间复杂度为O(1)，而第一个脚本的执行时间为O(n)。这是我们在第一个脚本中处理数据时，采用过程化方法的结果。

    在面向对象的世界中，没有数据；只有对象和它们的行为！

    OOP Alternative to Utility Classes

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    一个实用类（也称为辅助类）是一个只有静态方法并且不封装状态的“结构”。Apache Commons中的StringUtils、IOUtils、FileUtils；Guava中的Iterables和Iterators；以及JDK7中的Files都是实用类的完美示例。

    这个设计理念在Java世界（以及C＃，Ruby等）非常流行，因为实用类提供了在各个地方使用的常见功能。

    这里，我们希望遵循DRY原则并避免重复。因此，我们将常见的代码块放入实用类中，并在需要时重复使用它们。

    // This is a terrible design, don't reuse
public class NumberUtils {
  public static int max(int a, int b) {
    return a > b ? a : b;
  }
}

    Indeed, this a very convenient technique!?

    Utility Classes Are Evil

    However, in an object-oriented world, utility classes are considered a very bad (some even may say “terrible”) practice.

    有很多关于这个主题的讨论；举几个例子：Nick Malik 的 Are Helper Classes Evil?，Simon Hart 的 Why helper, singletons and utility classes are mostly bad，Marshal Ward 的 Avoiding Utility Classes，Dhaval Dalal 的 Kill That Util Class!，Rob Bagby 的 Helper Classes Are A Code Smell。

    此外，关于实用类的问题在StackExchange上有一些讨论：如果“实用类”是邪恶的，那我应该把通用代码放在哪里？，实用类是邪恶的。

    他们所有论点的简要概括是，实用类不是真正的“对象”；因此，它们不适用于面向对象的世界。它们是从“过程式”编程中继承而来的，主要是因为我们过去习惯了函数分解的范式。

    假设您同意这些论点并希望停止使用实用类，我将通过示例展示如何用适当的对象替换这些实用类。

    Procedural Example

    比如说，你想读取一个文本文件，将其分割成行，去掉每一行的空白，并将结果保存在另一个文件中。这可以通过使用Apache Commons中的FileUtils来实现。

    void transform(File in, File out) {
  Collection<String> src = FileUtils.readLines(in, "UTF-8");
  Collection<String> dest = new ArrayList<>(src.size());
  for (String line : src) {
    dest.add(line.trim());
  }
  FileUtils.writeLines(out, dest, "UTF-8");
}

    上面的代码看起来很简洁；然而，这是过程式编程，而不是面向对象的编程。我们正在操作数据（字节和位），并明确指示计算机从哪里检索它们，然后在每行代码中指定将它们放在何处。我们正在定义执行的过程。

    Object-Oriented Alternative

    在面向对象的范式中，我们应该实例化和组合对象，让它们自行管理数据的时间和方式。我们应该创建能够展示我们所需行为的对象，而不是调用辅助的静态函数。

    public class Max implements Number {
  private final int a;
  private final int b;
  public Max(int x, int y) {
    this.a = x;
    this.b = y;
  }
  @Override
  public int intValue() {
    return this.a > this.b ? this.a : this.b;
  }
}

    This procedural call:

    int max = NumberUtils.max(10, 5);

    Will become object-oriented:

    int max = new Max(10, 5).intValue();

    Potato, potato? Not really; just read on…

    Objects Instead of Data Structures

    This is how I would design the same file-transforming functionality as above but in an object-oriented manner:

    void transform(File in, File out) {
  Collection<String> src = new Trimmed(
    new FileLines(new UnicodeFile(in))
  );
  Collection<String> dest = new FileLines(
    new UnicodeFile(out)
  );
  dest.addAll(src);
}

    FileLines implements Collection<String> and encapsulates all file reading and writing operations. An instance of FileLines behaves exactly as a collection of strings and hides all I/O operations. When we iterate it—a file is being read. When we addAll() to it—a file is being written.

    Trimmed also implements Collection<String> and encapsulates a collection of strings (Decorator pattern). Every time the next line is retrieved, it gets trimmed.

    参与代码段的所有类都相当小：Trimmed，FileLines和UnicodeFile。它们各自负责自己的单个特性，因此完美地遵循着单一职责原则。

    在我们这一边作为库的用户，这可能并不那么重要，但对于它们的开发者来说却是必不可少的。相比于在一个拥有80多个方法和3000行代码的实用程序类FileUtils中使用readLines()方法，开发、维护和单元测试FileLines类要容易得多。真的，看看它的源代码吧。

    一种面向对象的方法使得延迟执行成为可能。只有在需要数据时，才会读取“in”文件。如果由于某种I/O错误而无法打开“out”，则第一个文件甚至都不会被触及。整个过程只有在调用“addAll()”后才开始。

    第二个代码片段中的所有行，除了最后一行之外，都将小对象实例化并组合成更大的对象。由于它不会引起任何数据转换，所以这种对象组合对于CPU来说是相当廉价的。

    除此之外，很明显第二段脚本在O(1)空间中运行，而第一段脚本在O(n)中执行。这是我们对第一段脚本中数据的过程化处理的结果。

    In an object-oriented world, there is no data; there are only objects and their behavior!

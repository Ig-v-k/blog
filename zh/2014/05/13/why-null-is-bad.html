    Why NULL is Bad?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    一个在Java中使用NULL的简单示例：

    public Employee getByName(String name) {
  int id = database.find(name);
  if (id == 0) {
    return null;
  }
  return new Employee(id);
}

    这个方法有什么问题？

    它可能返回NULL而不是一个对象—这就是问题所在。在面向对象的范式中，NULL是一种糟糕的做法，应该尽量避免使用。已经有很多关于这个问题的观点发表过，包括Tony Hoare的Null References, The Billion Dollar Mistake演讲和David West的整本Object Thinking书籍。

    在这里，我将试图总结所有的论点，并展示如何避免使用NULL并用适当的面向对象构造替代它的例子。

    基本上，有两种可能的替代方案来代替NULL。

    第一种是Null Object设计模式（最好的方法是将其作为常量使用）：

    public Employee getByName(String name) {
  int id = database.find(name);
  if (id == 0) {
    return Employee.NOBODY;
  }
  return Employee(id);
}

    第二种可能的选择是在无法返回对象时通过抛出异常来快速失败。

    public Employee getByName(String name) {
  int id = database.find(name);
  if (id == 0) {
    throw new EmployeeNotFoundException(name);
  }
  return Employee(id);
}

    现在，让我们来看看反对NULL的论点。

    除了上述的Tony Hoare的演讲和David West的书，我在撰写本文之前阅读了以下出版物：Robert Martin的《Clean Code》，Steve McConnell的《Code Complete》，John Sonmez的《Say “No” to “Null”》，以及Stack Overflow上的《返回null是糟糕的设计吗？》讨论。

    Ad-hoc Error Handling

    每当您将对象作为输入时，必须检查它是否为NULL或有效的对象引用。如果您忘记检查，可能会在运行时出现NullPointerException（NPE），从而中断执行。因此，您的逻辑变得混乱，需要进行多次检查和if/then/else条件分支。

    // this is a terrible design, don't reuse
Employee employee = dept.getByName("Jeffrey");
if (employee == null) {
  System.out.println("can't find an employee");
  System.exit(-1);
} else {
  employee.transferTo(dept2);
}

    这是异常情况在C和其他命令式过程语言中应该如何处理的方式。面向对象编程引入了异常处理主要是为了摆脱这些临时错误处理块。在面向对象编程中，我们让异常一直上升直到它们达到应用程序范围的错误处理程序，这样我们的代码变得更清晰和更简短。

    dept.getByName("Jeffrey").transferTo(dept2);

    将NULL引用视为过程式编程的一种继承，并使用1）Null对象或2）异常来替代。

    Ambiguous Semantic

    为了明确传达其含义，函数getByName()必须被命名为getByNameOrNullIfNotFound()。每个返回对象或NULL的函数都应该遵循相同的命名规则。否则，代码读者将无法避免歧义。因此，为了保持语义的明确性，应该给函数取一个更长的名称。

    为了消除这种歧义，总是返回一个真实的对象、一个空对象或抛出一个异常。

    有人可能会辩称，有时我们必须返回NULL，出于性能考虑。例如，在Java中，接口Map的方法get()在映射中没有这样的项时返回NULL。

    Employee employee = employees.get("Jeffrey");
if (employee == null) {
  throw new EmployeeNotFoundException();
}
return employee;

    这段代码只搜索一次地图，这是由于在Map中使用了NULL。如果我们重构Map，使其方法get()在未找到内容时抛出异常，我们的代码将如下所示：

    if (!employees.containsKey("Jeffrey")) { // first search
  throw new EmployeeNotFoundException();
}
return employees.get("Jeffrey"); // second search

    显然，这种方法比第一个方法慢两倍。该怎么办？

    Map 接口（无意冒犯其作者）存在一个设计缺陷。它的 get() 方法应该返回一个 Iterator，这样我们的代码看起来会像这样：

    Iterator found = Map.search("Jeffrey");
if (!found.hasNext()) {
  throw new EmployeeNotFoundException();
}
return found.next();

    顺便说一下，这正是 C++ STL map::find() 方法的设计方式。

    Computer Thinking vs. Object Thinking

    if (employee == null) 这个语句对于了解 Java 中对象是指向数据结构的指针，并且 NULL 是指向空值的指针（在 Intel x86 处理器中为 0x00000000）的人来说是可以理解的。

    然而，如果你从对象的角度开始思考，这个语句就没有多少意义了。以下是我们的代码从对象的角度来看的样子:

    - Hello, is it a software department?
- Yes.
- Let me talk to your employee "Jeffrey" please.
- Hold the line please...
- Hello.
- Are you NULL?

    这次对话中的最后一个问题听起来很奇怪，对吧？

    相反，如果他们在我们要求与杰弗里通话后挂断电话，那对我们来说会造成问题（异常）。在那时，我们会尝试再次拨打电话或通知主管我们无法联系到杰弗里并完成更大的交易。

    另外，他们也可能让我们与另一个人通话，这个人不是杰弗里，但可以帮助我们解答大部分问题，或者拒绝帮助我们如果我们需要”杰弗里特定”的东西（空对象）。

    Slow Failing

    不同于“快速失败”，上述代码试图慢慢死去，并在其途中杀死其他人。它不会让每个人都知道出了问题，也不会立即启动异常处理，而是将此故障隐藏起来，不让客户端看到。

    这个论点与上面讨论的“即席错误处理”非常接近。

    将代码尽可能地设置为脆弱，让它在必要时崩溃。

    使您的方法对其操作的数据要求非常严格。如果提供的数据不足或与方法的主要使用场景不匹配，让它们通过抛出异常来表达不满。

    否则，返回一个空对象，该对象公开一些常见行为，并在所有其他调用上抛出异常：

    public Employee getByName(String name) {
  int id = database.find(name);
  Employee employee;
  if (id == 0) {
    employee = new Employee() {
      @Override
      public String name() {
        return "anonymous";
      }
      @Override
      public void transferTo(Department dept) {
        throw new AnonymousEmployeeException(
          "I can't be transferred, I'm anonymous"
        );
      }
    };
  } else {
    employee = Employee(id);
  }
  return employee;
}

    Mutable and Incomplete Objects

    总的来说，强烈建议在设计对象时考虑不可变性。这意味着一个对象在实例化时获取所有必要的信息，并在整个生命周期中不改变其状态。

    很多时候，NULL值被用于延迟加载，使对象变得不完整和可变。例如：

    public class Department {
  private Employee found = null;
  public synchronized Employee manager() {
    if (this.found == null) {
      this.found = new Employee("Jeffrey");
    }
    return this.found;
  }
}

    这项技术虽然被广泛使用，但在面向对象编程中却是一个反模式。主要是因为它使得一个“员工”对象对计算平台的性能问题负责，而这是一个“员工”对象不应该知道的事情。

    一个对象不应该管理状态和暴露与业务相关的行为，而是要负责缓存自己的结果，这就是懒加载的含义。

    缓存不是一个员工在办公室里做的事情，对吗？

    解决方案是？不要像上面的例子那样以如此原始的方式使用懒加载。相反，将这个缓存问题转移到应用程序的另一层。

    例如，在Java中，你可以使用面向方面编程的方面。例如，jcabi-aspects有一个@Cacheable注解，用于缓存方法返回的值：

    import com.jcabi.aspects.Cacheable;
public class Department {
  @Cacheable(forever = true)
  public Employee manager() {
    return new Employee("Jacky Brown");
  }
}

    我希望这个分析足够有说服力，让你停止对代码进行NULL处理。

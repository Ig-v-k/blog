    Why NULL is Bad?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Java 中对 NULL 的简单使用示例：

    public Employee getByName(String name) {
  int id = database.find(name);
  if (id == 0) {
    return null;
  }
  return new Employee(id);
}

    这个方法有什么问题？

    它可能返回NULL而不是一个对象—这就是问题所在。在面向对象的范式中，NULL是一种可怕的做法，应该尽一切可能避免使用。已经有一些关于这个问题的观点已经发表，包括Tony Hoare的Null References, The Billion Dollar Mistake演讲和David West的整本Object Thinking书。

    在这里，我将尝试总结所有的论点，并展示如何避免使用NULL并用适当的面向对象构造替代它的例子。

    基本上，有两个可能的选择来替代 NULL。

    第一个是Null Object设计模式（最好的方法是将其定义为常量）：

    public Employee getByName(String name) {
  int id = database.find(name);
  if (id == 0) {
    return Employee.NOBODY;
  }
  return Employee(id);
}

    第二种可能的选择是在无法返回对象时通过抛出异常来快速失败。

    public Employee getByName(String name) {
  int id = database.find(name);
  if (id == 0) {
    throw new EmployeeNotFoundException(name);
  }
  return Employee(id);
}

    现在，让我们来看看对NULL的反对观点。

    除了上面提到的Tony Hoare的演讲和David West的书，我在撰写这篇文章之前还阅读了以下出版物：Clean Code（罗伯特·马丁著）、Code Complete（史蒂夫·麦康奈尔著）、Say “No” to “Null”（约翰·桑梅兹著）、Is returning null bad design?（Stack Overflow的讨论）。

    Ad-hoc Error Handling

    每当您将对象作为输入时，您必须检查它是否为NULL或有效的对象引用。如果忘记检查，NullPointerException（NPE）可能会在运行时中断执行。因此，您的逻辑变得混乱，需要进行多次检查和if/then/else分支。

    // this is a terrible design, don't reuse
Employee employee = dept.getByName("Jeffrey");
if (employee == null) {
  System.out.println("can't find an employee");
  System.exit(-1);
} else {
  employee.transferTo(dept2);
}

    这是在C和其他命令式过程式语言中应该处理异常情况的方式。面向对象编程引入了异常处理，主要是为了摆脱这些临时错误处理块。在面向对象编程中，我们让异常向上传递，直到它们达到应用程序范围的错误处理程序，这样我们的代码会变得更加清晰和简洁。

    dept.getByName("Jeffrey").transferTo(dept2);

    考虑将NULL引用视为过程式编程的继承，并使用1）空对象或2）异常来代替。

    Ambiguous Semantic

    为了明确传达其含义，函数getByName()必须被命名为getByNameOrNullIfNotFound()。对于每个返回对象或NULL的函数，都应该采用相同的方式。否则，代码读者将不可避免地遇到歧义。因此，为了保持语义明确，应该给函数起更长的名称。

    为了消除这种歧义，始终返回一个真实对象、一个null对象或抛出一个异常。

    有人可能争论说为了性能考虑，我们有时需要返回NULL。例如，在Java中，接口Map的方法get()在地图中没有这样的项时返回NULL。

    Employee employee = employees.get("Jeffrey");
if (employee == null) {
  throw new EmployeeNotFoundException();
}
return employee;

    这段代码只搜索一次地图，这是因为在Map中使用了NULL。如果我们重构Map，使其方法get()在找不到内容时抛出异常，我们的代码将如下所示：

    if (!employees.containsKey("Jeffrey")) { // first search
  throw new EmployeeNotFoundException();
}
return employees.get("Jeffrey"); // second search

    显然，这种方法比第一种方法慢了两倍。怎么办？

    Map 接口（无冒犯其作者）存在一个设计缺陷。它的方法 get() 应该返回一个 Iterator，这样我们的代码会像这样：

    Iterator found = Map.search("Jeffrey");
if (!found.hasNext()) {
  throw new EmployeeNotFoundException();
}
return found.next();

    顺便说一下，这正是C++的STL map::find() 方法的设计方式。

    Computer Thinking vs. Object Thinking

    对于了解Java中对象是指向数据结构的指针，并且NULL是指向无效地址（在Intel x86处理器中为0x00000000）的人来说，语句if (employee == null)是可以理解的。

    然而，如果您开始以对象的角度思考，这个陈述就变得没有太多意义了。以下是我们从对象的角度看待代码的方式：

    - Hello, is it a software department?
- Yes.
- Let me talk to your employee "Jeffrey" please.
- Hold the line please...
- Hello.
- Are you NULL?

    这次对话中的最后一个问题听起来很奇怪，是吧？

    相反，如果他们在我们要求与杰弗里通话后挂断电话，这对我们来说会造成问题（异常情况）。在那时，我们会尝试再次拨打电话或通知我们的主管，我们无法与杰弗里取得联系并完成一笔更大的交易。

    或者，他们可能让我们与另一个人交谈，这个人不是杰弗里，但可以帮助我们解答大部分问题，或者如果我们需要与”杰弗里”有关的信息（空对象），他们可能拒绝帮助。

    Slow Failing

    不同于“快速失败”，上述代码试图慢慢消亡，并在途中杀死其他人。它不会让每个人都知道出现问题，并立即开始异常处理，而是将这个失败隐藏起来，不让客户端知晓。

    这个论点与上面讨论的”临时错误处理”接近。

    尽可能使代码易碎化是一种良好的实践，让它在必要时出现错误。

    让你的方法对其操作的数据要求非常严格。如果提供的数据不足或与方法的主要使用场景不符合，让它们通过抛出异常来抱怨。

    否则，返回一个空对象，该对象公开了一些常见的行为，并在所有其他调用上抛出异常：

    public Employee getByName(String name) {
  int id = database.find(name);
  Employee employee;
  if (id == 0) {
    employee = new Employee() {
      @Override
      public String name() {
        return "anonymous";
      }
      @Override
      public void transferTo(Department dept) {
        throw new AnonymousEmployeeException(
          "I can't be transferred, I'm anonymous"
        );
      }
    };
  } else {
    employee = Employee(id);
  }
  return employee;
}

    Say, you are designing a method findUserByName(), which has to find a user in the database. What would you return if nothing is found? #elegantobjects

    Mutable and Incomplete Objects

    通常情况下，强烈建议在设计对象时考虑不可变性。这意味着一个对象在实例化过程中获取所有必要的信息，并在其整个生命周期中不会改变其状态。

    非常常见的，NULL 值被用于延迟加载，使对象不完整且可变。例如：

    public class Department {
  private Employee found = null;
  public synchronized Employee manager() {
    if (this.found == null) {
      this.found = new Employee("Jeffrey");
    }
    return this.found;
  }
}

    这项技术虽然被广泛使用，但在面向对象编程中却是一种反模式。主要是因为它使得一个对象承担起了计算平台的性能问题，而这是一个“雇员”对象不应该意识到的事情。

    不是管理状态和公开其与业务相关的行为，而是对象必须负责缓存自己的结果——这就是惰性加载的作用。

    缓存是不是员工在办公室中做的事情？

    解决方法？不要像上面的示例那样简单地使用懒加载。相反，将这个缓存问题移到应用程序的另一层。

    例如，在Java中，您可以使用面向方面编程的方面。例如，jcabi-aspects具有@Cacheable注释，该注释会缓存方法返回的值。

    import com.jcabi.aspects.Cacheable;
public class Department {
  @Cacheable(forever = true)
  public Employee manager() {
    return new Employee("Jacky Brown");
  }
}

    我希望这个分析足够有说服力，让你停止将代码设为 NULL。

    How an Immutable Object Can Have State and Behavior?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    我经常听到针对不可变对象的这个论点：“是的，当状态不变时它们很有用。但是，在我们的情况下，我们处理的是频繁变化的对象。每次我们只需要改变标题时，我们根本无法承担创建一个新的document的代价。” 这就是我不同意的地方：如果你需要频繁更改标题，那么对象的标题并不是一个状态，而是一个文档的行为。即使标题经常更改，一个好的对象也可以并且必须是不可变的。让我解释一下。

    Identity, State, and Behavior

    基本上，每个对象都有三个元素：身份、状态和行为。身份是区分我们的document与其他对象的特征，状态是文档对自身了解的内容（也称为“封装的知识”），而行为是文档能够根据我们的请求为我们提供的功能。例如，这是一个可变的文档：

    class Document {
  private int id;
  private String title;
  Document(int id) {
    this.id = id;
  }
  public String getTitle() {
    return this.title;
  }
  public String setTitle(String text) {
    this.title = text;
  }
  @Override
  public String toString() {
    return String.format("doc #%d about '%s'", this.id, this.text);
  }
}

    让我们尝试使用这个可变对象:

    Document first = new Document(50);
first.setTitle("How to grill a sandwich");
Document second = new Document(50);
second.setTitle("How to grill a sandwich");
if (first.equals(second)) { // FALSE
  System.out.println(
    String.format("%s is equal to %s", first, second)
  );
}

    在这里，我们创建了两个对象，然后修改了它们的封装状态。显然，first.equals(second)会返回false，因为这两个对象具有不同的身份，尽管它们封装了相同的状态。

    方法 toString() 公开了文档的行为—文档可以将自身转换为字符串。

    为了修改文档的标题，我们只需再次调用其 setTitle() 方法。

    first.setTitle("How to cook pasta");

    简单来说，我们可以多次重复使用对象，修改其内部状态。这样做既快捷又方便，是不是？快捷，是的。方便，其实并不是。请继续阅读。

    Immutable Objects Have No Identity

    正如我之前提到的那样，不可变性是一个良好对象的优点之一，也是非常重要的。一个良好的对象是不可变的，而良好的软件只包含不可变的对象。不可变对象与可变对象的主要区别在于不可变对象没有身份，其状态永远不会改变。以下是相同文档的一个不可变变体：

    @Immutable
class Document {
  private final int id;
  private final String title;
  Document(int id, String text) {
    this.id = id;
    this.title = text;
  }
  public String title() {
    return this.title;
  }
  public Document title(String text) {
    return new Document(this.id, text);
  }
  @Override
  public boolean equals(Object doc) {
    return doc instanceof Document
      && Document.class.cast(doc).id == this.id
      && Document.class.cast(doc).title.equals(this.title);
  }
  @Override
  public String toString() {
    return String.format(
      "doc #%d about '%s'", this.id, this.text
    );
  }
}

    这个文档是不可变的，它的状态（id和title）就是它的身份。让我们看看如何使用这个不可变类（顺便说一下，我正在使用jcabi-aspects中的@Immutable注解）。

    Document first = new Document(50, "How to grill a sandwich");
Document second = new Document(50, "How to grill a sandwich");
if (first.equals(second)) { // TRUE
  System.out.println(
    String.format("%s is equal to %s", first, second)
  );
}

    我们无法再修改文档了。当我们需要更改标题时，我们必须创建一个新的文档：

    Document first = new Document(50, "How to grill a sandwich");
first = first.title("How to cook pasta");

    每次我们想要修改它封装的状态时，我们都必须同时修改它的身份，因为没有身份。状态就是身份。看一下上面equals()方法的代码—它通过比较文档的ID和标题来进行比较。现在，文档的ID+标题就是它的身份！

    What About Frequent Changes?

    现在我要谈谈我们一开始提出的问题：性能和便利性如何呢？我们不想每次修改文档标题都要更改整个文档。如果文档足够大，那将是一项巨大的负担。此外，如果一个不可变对象封装了其他不可变对象，那么即使只修改其中一个字符串，我们也必须更改整个层次结构。

    答案很简单。文档的标题不应该是其状态的一部分，而应该是其行为。例如，考虑以下情况：

    @Immutable
class Document {
  private final int id;
  Document(int id) {
    this.id = id;
  }
  public String title() {
    // read title from storage
  }
  public void title(String text) {
    // save text to storage
  }
  @Override
  public boolean equals(Object doc) {
    return doc instanceof Document
      && Document.class.cast(doc).id == this.id;
  }
  @Override
  public String toString() {
    return String.format("doc #%d about '%s'", this.id, this.title());
  }
}

    从概念上讲，这个文档充当了一个真实文档的代理，该真实文档的标题存储在某个地方，例如一个文件中。这就是一个好的对象应该做的事情——充当一个真实实体的代理。该文档提供了两个功能：读取标题和保存标题。以下是其接口的样子：

    @Immutable
interface Document {
  String title();
  void title(String text);
}

    title() reads the title of the document and returns it as a String, and title(String) saves it back into the document. Imagine a real paper document with a title. You ask an object to read that title from the paper or to erase an existing one and write new text over it. This paper is a “copy” utilized in these methods.

    现在我们可以对不可变的文档进行频繁更改，但文档仍保持不变。它不会停止成为不可变的，因为它的状态(id)没有改变。尽管我们改变了标题，它仍然是同一份文档，因为标题不是文档的状态。它是现实世界中文档之外的东西。文档只是我们和那个”东西”之间的代理。阅读和写入标题是文档的行为，而不是它的状态。

    Mutable Memory

    我们仍未回答的唯一问题是“copy”是什么，如果我们需要将文档标题保留在内存中会发生什么？

    让我们从“对象思维”的角度来看待它。我们有一个document对象，它应该代表着面向对象世界中的一个真实实体。如果这个实体是一个文件，我们可以很容易地实现title()方法。如果这个实体是一个Amazon S3对象，我们也可以轻松地实现标题读写方法，保持对象的不可变性。如果这个实体是一个HTTP页面，我们在实现标题读写时也没有问题，保持对象的不可变性。只要真实世界中存在一个文档并且具有自己的身份，我们就没有问题。我们的标题读写方法将与这个真实世界的文档进行通信，并提取或更新其标题。

    当这样的实体在现实世界中不存在时，问题就会出现。在这种情况下，我们需要创建一个可变对象属性，称为 title，通过 title() 进行读取，并通过 title(String) 进行修改。但是，对象是不可变的，所以根据定义，我们无法在其中拥有可变属性！那么我们该怎么办呢？

    Think.

    我们的对象怎么可能不代表一个真实的实体呢？记住，“真实世界”是指对象所处的生活环境周围的一切。有可能一个对象既不代表任何人，又能自行操作吗？不可能。每个对象都是一个真实世界实体的代表。那么，如果我们想要在对象内保留 title，又没有任何文件或HTTP页面与该对象相关，它代表的是谁呢？

    它代表着计算机内存。

    不可变文档 #50 的标题“如何烤三明治”存储在内存中，占用23个字节的空间。文档应该知道这些字节存储在哪里，并且能够读取并替换它们为其他内容。这23个字节是对象所代表的现实世界实体。这些字节与对象的状态无关。它们是一个可变的现实世界实体，类似于文件、HTTP页面或Amazon S3对象。

    不幸的是，Java（以及许多其他现代语言）不允许直接访问计算机内存。这是我们设计类的方式，如果可以直接访问的话。

    @Immutable
class Document {
  private final int id;
  private final Memory memory;
  Document(int id) {
    this.id = id;
    this.memory = new Memory();
  }
  public String title() {
    return new String(this.memory.read());
  }
  public void title(String text) {
    this.memory.write(text.getBytes());
  }
}

    Memory类将由JDK本地实现，而所有其他类都将是不可变的。Memory类将直接访问内存堆，并负责操作操作系统级别的malloc和free。拥有这样一个类将使我们能够使所有Java类都是不可变的，包括StringBuffer、ByteArrayOutputStream等。

    Memory类会明确强调软件程序中对象的任务，即成为一个数据动画师。对象不是保存数据的容器；它是在为数据添加动画效果。数据存在于某个地方，它是无生命力的、静止的、不动的等等。当对象处于活动状态时，数据是死的。对象的角色是让某个数据变得有生命力，为其添加动画效果，而不是成为一份数据。为了访问这份死数据，对象需要一些相关知识。对象可能需要一个数据库唯一键、一个HTTP地址、一个文件名，或者一个内存地址来找到数据并为其添加动画效果。但对象永远不应该将自己视为数据。

    What Is the Practical Solution?

    很不幸的是，在Java、Ruby、JavaScript、Python、PHP等许多高级语言中，我们没有这样一种能够代表内存的类。这似乎表明语言设计者没有理解活动对象与死数据的区别，这令人遗憾。我们被迫使用相同的语言结构——对象变量和属性——来混合数据和对象状态。也许将来我们会在Java和其他语言中拥有Memory类，但在那之前，我们有几个选择。

    使用C++。在C++和类似的低级语言中，可以直接访问内存并处理内存中的数据，就像处理文件或HTTP数据一样。在C++中，我们可以创建Memory类并按照上面解释的方式使用它。

    使用数组。在Java中，数组是一种具有独特特性的数据结构—它可以在声明为final时进行修改。您可以在不可变对象内部使用字节数组作为可变数据结构。这是一种替代解决方案，从概念上类似于Memory类，但更加原始。

    避免使用内存数据。尽量避免使用内存数据。在某些领域，这很容易做到；例如，Web应用程序、文件处理、I/O适配器等。然而，在其他领域，这要说起来比做起来更容易。例如，在游戏、数据处理算法和GUI中，大多数对象主要是通过在内存中操作数据来进行动画处理，因为内存是它们唯一的资源。在这种情况下，如果没有Memory类，你将得到可变对象 :( 没有解决办法。

    总结一下，不要忘记，对象是数据的动画生成器。它使用其封装的知识来获取数据。无论数据存储在哪里——文件中、HTTP中还是内存中——在概念上与对象状态非常不同，尽管它们看起来可能非常相似。

    一个好的对象是可变数据的不可变动画师。尽管它是不可变的，而数据是可变的，但在对象的生存环境中，它是活的而数据是死的。

    Rultor + Travis

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Rultor是一个编码团队助手。Travis是一个托管的持续集成系统。在本文中，我将展示我们的开源项目如何同时使用它们来实现无缝的持续交付。

    我将展示几个实际场景。

    Scenario #1: Merge Pull Request

    jcabi-mysql-maven-plugin 是一个用于 MySQL 集成测试的 Maven 插件。@ChristianRedl 提交了一个新功能的拉取请求#35。我审核了该请求并要求 Rultor 将其合并到 master 分支中。

    如你所见，Rultor执行了实际的合并操作。我通过将他的GitHub账号添加到项目合作者列表中，给予了他对项目的访问权限。

    在对Rultor进行“继续进行”的操作之前，我先检查了由Travis报告的预构建状态。

    Travis在pull请求中发现了一次新的提交，并立即（无需我方干预）在该分支上触发了一次构建。构建未失败，这就是为什么Travis给了我一个绿色标志。我看了看标志和代码。由于代码中的所有问题都被pull请求的作者修正，并且Travis没有抱怨，所以我向Rultor发出了“继续”的指令。

    Scenario #2: Continuous Integration

    尽管前一步骤保证了master分支始终干净且稳定，但我们利用Travis进行持续集成。每个提交到master的变动都会在Travis中触发一次新的构建。构建的结果会改变Travis中项目的状态，要么是“失败”，要么是“通过”。

    jcabi-aspects 是一个AOP AspectJ切面的集合。我们配置了Travis来连续构建它。这是它生成的徽章（左边的那个）：

    再次强调，即使只读主分支是一种有效的保护措施，可以防止构建失败，但它并不能保证主分支在任何时刻都是稳定的。例如，有时由于日历、环境、依赖关系、网络连接质量等方面的变化，单元测试会出现偶发性失败。

    理想情况下，单元测试应该要么失败要么通过，因为它们是与环境无关的。然而，实际情况是，单元测试远非理想。

    这就是为什么将只读主服务器与Rultor和Travis的持续集成结合起来，可以为我们提供更高的稳定性。

    Scenario #3: Release to RubyGems

    jekyll-github-deploy 是一个 Ruby gem，用于自动将 Jekyll 网站部署到 GitHub Pages。@leucos 提交了一个包含新功能的 pull request #4。该请求已成功合并到 master 分支。

    然后，我告诉Rultor应该将master分支发布到RubyGems，并设置一个新版本为1.5。

    Rultor执行了一个简单的脚本，其在.rultor.yml中进行了预配置。

    release:
  script: |
    ./test.sh
    rm -rf *.gem
    sed -i "s/2.0-SNAPSHOT/${tag}/g" jgd.gemspec
    gem build jgd.gemspec
    chmod 0600 ../rubygems.yml
    gem push *.gem --config-file ../rubygems.yml

    脚本是参数化的，正如您所见。有一个参数通过Rultor传递给脚本：${tag}。当我向Rultor提交一个命令时，我在GitHub问题中提供了这个参数。

    脚本测试宝石是否正常工作（集成测试），并在之后进行清理。

    $ ./test.sh
$ rm -rf *.gem

    然后，它会将jgd.gemspec中的版本更改为${tag}中提供的版本（它是一个环境变量），并构建一个新的.gem文件。

    $ sed -i "s/2.0-SNAPSHOT/${tag}/g" jgd.gemspec
$ gem build jgd.gemspec

    最后，它使用来自 ../rubygems.yml 的登录凭据将新构建的 .gem 推送到 RubyGems。这个文件是由 Rultor 在脚本启动之前创建的（这个机制将在下面讨论）。

    $ chmod 0600 ../rubygems.yml
$ gem push *.gem --config-file ../rubygems.yml

    如果一切顺利并且RubyGems确认成功部署，Rultor将向GitHub报告。这正是在pull request #4中发生的情况。

    Scenario #4: Deploy to CloudBees

    s3auth.com 是一个用于亚马逊 S3 存储桶的基本 HTTP 验证网关。它是一个 Java Web 应用程序。在它的拉取请求 #195 中，由 @carlosmiranda 修复了一个资源泄漏问题，并且该拉取请求由 Rultor 合并。

    然后，@davvd 指示 Rultor 将 master 分支部署到生产环境。Rultor 创建了一个新的 Docker 容器，并在其中运行了 mvn clean deploy。

    Maven将应用程序部署到CloudBees：

    整个过程花费了21分钟，正如您在Rultor生成的报告中所看到的。

    有一个值得提及的重要技巧。部署到生产环境总是意味着使用安全凭据，例如登录名、密码、SSH密钥等。

    在这个特定的例子中，Maven CloudBees 插件 需要 API 密钥、密钥和网络应用程序名称。这三个参数是保密的，不能以“开源”的方式揭示。

    所以，有一个机制可以通过其.rultor.yml文件（请注意前几行）配置Rultor。

    assets:
  settings.xml: "yegor256/home#assets/s3auth/settings.xml"
  pubring.gpg: "yegor256/home#assets/pubring.gpg"
  secring.gpg: "yegor256/home#assets/secring.gpg"

    这些YAML条目告知Rultor它必须从yegor256/home私有的GitHub仓库中获取assets/s3auth/settings.xml文件，并将其放入Docker容器的工作目录中，在开始Maven构建之前。

    这个 settings.xml 文件包含了CloudBees插件部署应用所需的秘密数据。

    如何一键部署到CloudBees会更好地解释这个过程。

    You Can Do The Same

    无论是Rultor还是Travis，都是免费的托管产品，只要你的项目是开源的，并且托管在GitHub上。

    Rultor+Travis的其他良好应用示例可在以下GitHub问题中看到：jcabi/jcabi-http#47，jcabi/jcabi-http#48

    PS. 你也可以在 Windows 平台上使用 AppVeyor 进行类似的操作：AppVeyor 在 Rultor 合并 Pull Requests 之前帮助我进行验证。

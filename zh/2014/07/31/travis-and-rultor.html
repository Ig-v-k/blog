    Rultor + Travis

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Rultor 是一个编码团队助手。Travis 是一个托管的持续集成系统。在这篇文章中，我将展示我们的开源项目如何同时使用它们来实现无缝持续交付。

    我将展示一些实际场景。

    Scenario #1: Merge Pull Request

    jcabi-mysql-maven-plugin 是一个用于MySQL集成测试的Maven插件。@ChristianRedl 提交了一个新功能的拉取请求#35。我审查了这个请求并要求Rultor将其合并到master分支中。

    正如您所看到的，Rultor进行了实际的合并操作。我通过将他的GitHub账户添加到项目协作者列表中，为他提供了项目访问权限。

    在给Rultor确认之前，我检查了Travis报告的预构建状态。

    Travis在拉取请求中发现了一个新的提交，并立即（在我没有进行任何交互的情况下）在该分支上触发了一次构建。构建没有失败，所以Travis给了我一个绿色的标志。我看了那个标志和代码。由于代码中的所有问题都已经被拉取请求的作者修复，并且Travis没有抱怨——所以我向Rultor发出了“前进”的指令。

    Scenario #2: Continuous Integration

    即使之前的步骤保证了master分支始终是干净且稳定的，我们仍然使用Travis进行持续集成。每次提交到master都会触发Travis中的新构建。构建的结果会改变Travis中项目的状态，要么是“失败”，要么是“通过”。

    jcabi-aspects 是一个AOP AspectJ方面的集合。我们配置了Travis来持续构建它。这是它生成的徽章（左边的）。

    再次强调，尽管只读主分支可以有效防止构建失败，但并不能保证主分支在任何时刻都是稳定的。例如，有时由于日历、环境、依赖关系、网络连接质量等原因，单元测试会偶尔失败。

    理想情况下，单元测试应该是环境无关的，要么失败要么成功。然而，在现实中，单元测试远非理想。

    这就是为什么将只读主服务器与Rultor和Travis的持续集成结合在一起可以提供更高的稳定性。

    Scenario #3: Release to RubyGems

    jekyll-github-deploy 是一个Ruby宝石（gem），可自动将Jekyll网站部署到GitHub Pages。@leucos 提交了一个拉取请求#4，添加了一个新功能。该请求已成功合并到master分支。

    然后，我指示Rultor将master分支发布到RubyGems，并设置一个新版本为1.5。

    Rultor执行了一个简单的脚本，在其.rultor.yml中预先配置（见这里）。

    release:
  script: |
    ./test.sh
    rm -rf *.gem
    sed -i "s/2.0-SNAPSHOT/${tag}/g" jgd.gemspec
    gem build jgd.gemspec
    chmod 0600 ../rubygems.yml
    gem push *.gem --config-file ../rubygems.yml

    这个脚本是有参数的，就像你看到的一样。有一个参数是由Rultor传递给脚本的：${tag}。这个参数是我在GitHub问题中提供的，当我向Rultor提交命令时。

    脚本测试宝石是否正常工作（集成测试），并在之后进行清理操作。

    $ ./test.sh
$ rm -rf *.gem

    然后，它会将jgd.gemspec中的版本更改为${tag}中提供的版本（它是一个环境变量），并构建一个新的.gem文件。

    $ sed -i "s/2.0-SNAPSHOT/${tag}/g" jgd.gemspec
$ gem build jgd.gemspec

    最后，它使用来自../rubygems.yml的登录凭据将新构建的.gem推送到RubyGems。这个文件是由Rultor在脚本启动之前创建的（这个机制在下面讨论）。

    $ chmod 0600 ../rubygems.yml
$ gem push *.gem --config-file ../rubygems.yml

    如果一切正常且RubyGems确认成功部署，Rultor会向GitHub提交报告。这正是在pull request #4中发生的情况。

    Scenario #4: Deploy to CloudBees

    s3auth.com 是一个用于亚马逊S3存储桶的基本HTTP身份验证网关。它是一个Java Web应用程序。在其拉取请求#195中，@carlosmiranda修复了一个资源泄漏问题，并由Rultor合并了该拉取请求。

    然后，@davvd 指示 Rultor 将 master 分支部署到生产环境。Rultor 创建了一个新的 Docker 容器，并在其中运行了 mvn clean deploy。

    Maven将应用程序部署到CloudBees：

    整个流程花了21分钟，就像你在Rultor生成的报告中看到的那样。

    有一个重要的技巧值得一提。部署到生产环境始终意味着使用安全凭证，比如登录、密码、SSH密钥等等。

    在这个特定的例子中，Maven CloudBees插件需要API密钥、密钥和Web应用程序名称。这三个参数被保密，不能以“开源”的方式公开。

    所以，有一个机制通过其.rultor.yml文件来配置Rultor（请注意前几行）。

    assets:
  settings.xml: "yegor256/home#assets/s3auth/settings.xml"
  pubring.gpg: "yegor256/home#assets/pubring.gpg"
  secring.gpg: "yegor256/home#assets/secring.gpg"

    这些YAML条目告诉Rultor，它必须从yegor256/home私有(!) GitHub存储库中获取assets/s3auth/settings.xml文件，并将其放置在Docker容器的工作目录中，在启动Maven构建之前。

    这个 settings.xml 文件包含了 CloudBees 插件部署应用程序所需的秘密数据。

    如何一键部署到CloudBees，更好地解释了这个过程。

    You Can Do The Same

    无论是Rultor还是Travis，只要您的项目是开源的并且托管在GitHub上，它们都是免费的托管产品。

    Rultor+Travis使用的其他好例子可以在这些GitHub问题中看到：jcabi/jcabi-http#47，jcabi/jcabi-http#48

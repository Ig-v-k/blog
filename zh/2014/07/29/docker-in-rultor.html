    Every Build in Its Own Docker Container

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Docker 是一个命令行工具，可以在虚拟Linux中运行shell命令，采用独立的文件系统。每次构建项目时，我们希望它们能在自己的Docker容器中运行。以这个Maven项目为例：

    $ sudo docker run -i -t ubuntu mvn clean test

    这个命令将启动一个新的Ubuntu系统，并在其中执行mvn clean test。我们的虚拟助手Rultor.com在我们部署、打包、测试和合并时会做完全相同的操作。

    Why Docker?

    它给我们带来了哪些好处？那么，在有很多其它虚拟化技术（例如LXC）的情况下，为什么选择Docker呢？

    嗯，有一些非常重要的好处：

    • 图像存储库 (hub.docker.com)

    • Versioning

    • “Application-centric” can be translated to Chinese as “以应用为中心”.

    让我们详细讨论它们。

    Image Repository

    Docker通过其公共仓库hub.docker.com实现图像共享。这意味着我在为我的应用程序准备工作环境后，将其制作成一个镜像并推送到仓库。

    假设我希望在一个预先安装了graphviz软件包的容器中执行我的Maven构建（以便启用dot命令行工具）。首先，我会启动一个纯净的Ubuntu容器，并在其中安装graphviz：

    $ sudo docker run -i -t ubuntu /bin/bash
root@215d2696e8ad:/# sudo apt-get install -y graphviz
Reading package lists... Done
Building dependency tree
Reading state information... Done
The following extra packages will be installed:
...
root@215d2696e8ad:/# exit
$ sudo docker ps -a
CONTAINER ID        IMAGE               COMMAND             CREATED              STATUS                     PORTS               NAMES
215d2696e8ad        ubuntu:14.04        /bin/bash           About a minute ago   Exited (0) 3 seconds ago                       high_mccarthy

    我有一个在几秒钟前停止的容器。容器的ID是215d2696e8ad。现在，我想要将其用于Rultor.com上的所有进一步的测试。我必须从中创建一个镜像。

    $ sudo docker commit 215d2696e8ad yegor256/beta
c5ad7718fc0e20fe4bf2c8a9bfade4db8617a25366ca5b64be2e1e8aa0de6e52

    我刚刚提交了一个新的镜像 yegor256/beta。这个镜像现在可以被重复使用。我可以从这个镜像创建一个新的容器，里面会安装有 graphviz！

    现在是时候在 Docker hub 上分享我的镜像，以便让它对 Rultor 可用。

    $ sudo docker push yegor256/beta
The push refers to a repository [yegor256/beta] (len: 1)

Pushing repository yegor256/beta (1 tags)

6c37f792ddac: Image already pushed, skipping
e54ca5efa2e9: Image already pushed, skipping
c5ad7718fc0e: Image successfully pushed

    最后一步是配置Rultor在所有构建中使用此镜像。为此，我将编辑我的GitHub存储库根目录中的.rultor.yml文件。

    docker:
  image: yegor256/beta

    就是这样。从现在开始，Rultor将在每个构建（合并、发布、部署等）中使用我自定义的预安装了graphviz的Docker镜像。

    此外，如果我想向镜像添加其他内容，这很容易做到。比如说，我想在构建镜像中安装Ruby。我从镜像中启动一个容器并进行安装（请注意，我现在不再从ubuntu镜像启动容器，而是从yegor256/beta启动）。

    $ sudo docker run -i -t yegor256/beta /bin/bash

    因此，这个Docker Hub对于Rultor和类似的系统来说是一个非常方便的功能。

    Versioning

    正如您在上面的示例中所看到的，对Docker镜像的每个更改都有自己的版本（哈希值），并且可以追踪更改。还可以回滚到任何特定的更改。

    Rultor本身不使用此功能，但Rultor用户能够以更高的精度控制其构建配置。

    Application-Centric

    Docker，与LXC或Vagrant不同，是以应用为中心的。这意味着当我们启动一个容器时，我们启动了一个应用程序。使用其他虚拟化技术时，当您获得一个虚拟机时，您获得了一个完全功能的Unix环境，您可以通过SSH登录并做任何您想做的事情。

    Docker使事情变得更简单。它不会为您提供对容器的SSH访问，而是在其中运行应用程序并向您显示其输出。这正是我们在Rultor中所需要的。我们需要运行一个自动化构建（例如Maven或Bundler），查看其输出并获得其退出代码。如果代码不是零，我们将构建失败并向用户报告。

    这是我们运行Maven构建的方式：

    $ sudo docker run --rm -i -t yegor256/rultor mvn clean test
[INFO] ------------------------------------------------------------------------
[INFO] Building jcabi-github 0.13
[INFO] ------------------------------------------------------------------------
[INFO]
[INFO] --- maven-clean-plugin:2.5:clean (default-clean) @ jcabi-github ---
[INFO]
...

    正如您所见，Maven会立即启动。我们不需要担心容器的内部细节，只需要在其中启动一个应用程序。

    此外，由于使用了--rm选项，在Maven执行结束后，容器会立即被销毁。

    这就是所谓的以应用为中心。

    我们对Docker的总体印象非常积极。

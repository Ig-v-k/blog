    Every Build in Its Own Docker Container

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Docker 是一个命令行工具，可以在虚拟的 Linux 环境中运行一个 shell 命令，并使用独立的文件系统。每次构建项目时，我们希望它们能够在自己的 Docker 容器中运行。以这个 Maven 项目为例：

    $ sudo docker run -i -t ubuntu mvn clean test

    这个命令将启动一个新的Ubuntu系统，并在其中执行mvn clean test。我们的虚拟助手Rultor.com在部署、打包、测试和合并时，就会执行完全相同的操作。

    Image Repository

    Docker通过其公共仓库 hub.docker.com 实现图像共享。这意味着，在我为应用程序准备好一个工作环境后，我将其制作成一个镜像并将其推送到该仓库。

    假设我想要在一个预装了 graphviz 包（以便启用 dot 命令行工具）的容器中执行我的 Maven 构建。首先，我会启动一个普通的 Ubuntu 容器，并在其中安装 graphviz。

    $ sudo docker run -i -t ubuntu /bin/bash
root@215d2696e8ad:/# sudo apt-get install -y graphviz
Reading package lists... Done
Building dependency tree
Reading state information... Done
The following extra packages will be installed:
...
root@215d2696e8ad:/# exit
$ sudo docker ps -a
CONTAINER ID        IMAGE               COMMAND             CREATED              STATUS                     PORTS               NAMES
215d2696e8ad        ubuntu:14.04        /bin/bash           About a minute ago   Exited (0) 3 seconds ago                       high_mccarthy

    我有一个几秒钟前停止的容器。容器的ID是215d2696e8ad。现在，我想在Rultor.com上将其重用于所有后续测试。我必须从它创建一个镜像：

    $ sudo docker commit 215d2696e8ad yegor256/beta
c5ad7718fc0e20fe4bf2c8a9bfade4db8617a25366ca5b64be2e1e8aa0de6e52

    我刚刚将我的新提交制作到了一个新的图像yegor256/beta。这个图像现在可以被重复使用。我可以从这个图像创建一个新的容器，里面会安装有graphviz！

    现在是时候在Docker Hub上分享我的镜像，以便让它在Rultor上可用了。

    $ sudo docker push yegor256/beta
The push refers to a repository [yegor256/beta] (len: 1)
Sending image list
Pushing repository yegor256/beta (1 tags)
511136ea3c5a: Image already pushed, skipping
d7ac5e4f1812: Image already pushed, skipping
2f4b4d6a4a06: Image already pushed, skipping
83ff768040a0: Image already pushed, skipping
6c37f792ddac: Image already pushed, skipping
e54ca5efa2e9: Image already pushed, skipping
c5ad7718fc0e: Image successfully pushed
Pushing tag for rev [c5ad7718fc0e] on {https://registry-1.docker.io/v1/repositories/yegor256/beta/tags/latest}

    最后一步是配置Rultor在所有构建中使用这个镜像。为了做到这一点，我将在我的GitHub存储库的根目录中编辑.rultor.yml。

    docker:
  image: yegor256/beta

    就是这样。从现在开始，Rultor将在每次构建（合并，发布，部署等）中使用我预先安装了graphviz的自定义Docker镜像。

    此外，如果我想要在图像中添加其他内容，这也很容易实现。比如说，我想要将Ruby安装到我的构建图像中。我会从图像中启动一个容器并进行安装（请注意，我之前是从ubuntu图像开始容器，而现在我是从yegor256/beta开始容器）。

    $ sudo docker run -i -t yegor256/beta /bin/bash
root@7e0fbd9806c9:/# sudo apt-get install -y ruby
Reading package lists... Done
Building dependency tree
Reading state information... Done
The following extra packages will be installed:
...
root@7e0fbd9806c9:/# exit
$ sudo docker ps -a
CONTAINER ID        IMAGE                  COMMAND             CREATED             STATUS                     PORTS               NAMES
7e0fbd9806c9        yegor256/beta:latest   /bin/bash           28 seconds ago      Exited (0) 2 seconds ago                       pensive_pare
215d2696e8ad        ubuntu:14.04           /bin/bash           10 minutes ago      Exited (0) 8 minutes ago                       high_mccarthy

    你现在可以看到我有两个容器。第一个是我正在使用的，其中包含了Ruby。第二个是我之前使用的，其中包含了graphviz。

    现在我需要再次提交并推送：

    $ sudo docker commit 7e0fbd9806c9 yegor256/beta

$ sudo docker push yegor256/beta
...

    因此，Docker Hub是Rultor和类似系统的非常方便的功能。

    Versioning

    正如你在上面的例子中所看到的，对Docker镜像的每一次更改都有自己的版本（哈希值），并且可以追踪这些更改。还可以回滚到任何特定的更改。

    Rultor本身不使用此功能，但Rultor用户能够以更高的精确度控制他们的构建配置。

    Application-Centric

    Docker与LXC或Vagrant不同，它是以应用为中心的。这意味着当我们启动一个容器时，我们启动的是一个应用程序。而使用其他虚拟化技术时，当你获得一个虚拟机时，你获得的是一个完全功能的Unix环境，你可以通过SSH登录并做任何你想做的事情。

    Docker使事情变得更简单。它不会给你容器的SSH访问权限，但会在容器内运行一个应用程序并显示其输出。这正是我们在Rultor中所需要的。我们需要运行自动构建（例如Maven或Bundler），查看其输出并获取其退出代码。如果代码不为零，我们将构建失败并向用户报告。

    这是我们运行Maven构建的方式：

    $ sudo docker run --rm -i -t yegor256/rultor mvn clean test
[INFO] ------------------------------------------------------------------------
[INFO] Building jcabi-github 0.13
[INFO] ------------------------------------------------------------------------
[INFO]
[INFO] --- maven-clean-plugin:2.5:clean (default-clean) @ jcabi-github ---
[INFO]
...

    正如您所见，Maven立即启动。我们不需要担心容器的内部。我们只需在其中启动一个应用程序。

    此外，由于--rm选项的存在，容器会在Maven执行完成后立即销毁。

    这就是应用为中心的意思。

    我们对Docker的整体印象非常积极。

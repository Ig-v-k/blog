    GitHub Guidelines

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    本手册介绍了在GitHub上托管的XDSD项目中使用的工作流程。您从被分配一个GitHub问题开始。接下来，您将收到一条来自项目经理的消息，其中包含问题编号、标题、描述以及预计耗时（通常为30分钟）。

    如果您不同意预算分配，请毫不犹豫地要求增加预算。一旦您对预算感到满意并理解工作范围，请在回复工单时表示并开始工作。请注意，您超出预算分配的时间不会得到支付。

    1. Fork

    即使你是开发团队的一员，也没有在 GitHub 上对仓库进行写入访问权限。因此，要贡献变更，你应该将仓库叉出到你自己的 GitHub 账户（创建一个私有副本），进行所需的更改，然后使用”拉取请求”提交给审查。

    在您提交拉取请求审查后，仓库所有者通过将您的更改合并到主仓库中来批准您的更改。这是我们保护主要开发流程免受意外损坏的方式。

    本文介绍了如何fork一个仓库：fork-a-repo

    这篇文章解释了如何在您的计算机上下载和安装GitHub：set-up-git

    最后，不要忘记将您的私有SSH密钥添加到GitHub：生成SSH密钥。

    2. Branch

    当你将我们的仓库fork到你的账户中后，将其克隆到你的电脑上，然后检出master分支。例如：

    git clone git@github.com:yegor256/xembly.git
git checkout master

    现在，是时候分支了（123 是你要处理的 GitHub 问题的编号，也是分支的名称）。

    git checkout -b 123

    按照惯例，我们使用与您正在处理的分支和问题相同的名称。

    3. Changes

    所有与任务相关的问题应在GitHub的issue中进行讨论。对于GitHub的issue，我们不使用电子邮件、Skype、电话或会议。所有问题都应直接在GitHub的issue中提问。

    如果有任何不清楚的地方或需要帮助，请随时提交新问题。接到无法实施的任务是非常常见的。不要惊慌。当你刚加入一个项目，还没有足够的信息时，通常会发生这种情况。如果发生这种情况，请不要试图自己解决问题或困难。

    这种情况的经验法则是：“如果有什么不清楚的地方，那是我们的错，而不是你的错。”因此，如果你不理解项目设计，那是项目设计师的错误。

    提交一个错误报告，请求解释一个设计概念。您将获得此报告的报酬，并且您在回复中收到的信息将与所有其他开发人员共享。

    阅读这篇文章：Bugs Are Welcome.

    不要期望有人会帮助你。你唯一的帮助来源就是源代码本身。如果代码没有解释你所需要了解的一切——那是一个错误，必须报告。

    4. Commit and Push

    使用文本编辑器或集成开发环境进行必要的更改。立即在进行更改时提交更改是一个好的实践。在提交更改之前不要积累过多的更改。

    git commit -am '#123: the description of the changes'
git push origin 123

    如果您对工作范围有疑问，请在GitHub问题中发布，并等待回答。如果您认为现有代码需要改进，请毫不犹豫地提交新的GitHub问题。

    不要试图在一个分支中解决所有问题；让其他程序员来处理它们。

    5. Pull Request

    在GitHub上创建一个拉取请求，使用以下文章中的流程：using-pull-requests

    将其发布在原问题中，并等待反馈。

    6. Code Review

    过一段时间后，您提交的拉取请求将由项目团队的某个人进行审查。在许多情况下，您可能会收到一些负面评论，并且您需要解决与这些评论有关的所有问题。在您的更改满足审查者的要求之前，您的拉取请求不会合并到“主分支”。

    请耐心倾听审阅人的意见，不过不要认为审阅人总是对的。如果你认为你的修改是正确的，请坚持让其他人审阅。

    在对分支进行必要的更改之后，不要忘记通过发布一条针对评审人员的消息来通知他/她在问题中。在每一轮评审-修复循环中都要做同样的事情：请评审人员再次查看你的代码。如果没有这样的通知，你的拉取请求可能会在沉默中停留更长时间。

    7. Merge

    当审核者认为一切都很顺利时，他将通知我们的自动合并机器人。然后，自动合并机器人将选择您的拉取请求并尝试将其合并到“主”分支。由于各种原因，这个操作经常失败。如果合并失败，无论原因如何，您有责任确保您的分支成功合并。

    如果由于与您的任务无关的测试失败而无法合并一个分支，请不要尝试自行修复。相反，请将问题报告为一个新的错误，并等待其解决。

    记住，在你的分支合并之前，你不会获得报酬。

    8. Payment

    一旦你的更改已合并，返回GitHub问题页并要求作者关闭它。一旦项目经理关闭该问题，你将在几个小时内收到你的付款。

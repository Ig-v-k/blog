    GitHub Guidelines

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    本手册将解释在GitHub上托管的XDSD项目的工作流程。当你被分配一个GitHub问题时，你将开始工作。接下来，你将收到一条来自项目经理的消息，其中包含问题编号、标题、描述以及预计工作时间（通常为30分钟）。

    如果您不同意预算分配，请毫不犹豫地要求增加。一旦您对预算感到满意，并理解工作范围，请在回复工单时明确表达，并开始工作。请注意，超过预算时间分配的时间将不会得到报酬。

    1. Fork

    即使你是开发团队的一员，你也没有对 GitHub 上的代码库进行写操作的权限。因此，为了做出贡献，你应该将代码库分叉到你自己的 GitHub 账户（创建一个私有副本），进行所需的更改，然后通过“pull request”提交给其他人进行审查。

    在您提交拉取请求审查后，仓库所有者通过将更改合并到主要仓库中来批准您的更改。这是我们保护主要开发流程免受意外损坏的方式。

    本文介绍了如何分叉（fork）一个存储库：分叉一个存储库

    这个链接解释了如何在您的电脑上下载和安装GitHub：设置Git

    最后，不要忘记将您的私有SSH密钥添加到GitHub：生成SSH密钥

    2. Branch

    一旦你将我们的存储库分叉到你的账户中，将其克隆到你的计算机上，然后检出master分支。例如：

    git clone git@github.com:yegor256/xembly.git
git checkout master

    现在是时候创建分支了（123 是你将要处理的 GitHub 问题的编号，也是分支的名称）。

    git checkout -b 123

    按照惯例，我们使用与您正在处理的分支和问题相同的名称。

    3. Changes

    所有与任务相关的问题都应该在GitHub问题中讨论。对于GitHub问题，我们不使用电子邮件，Skype，电话或会议。所有问题都应该直接在GitHub问题中提出。

    如果有什么不清楚的地方或需要帮助，请毫不犹豫地提交新问题。接到一项可能无法实施的任务是非常常见的。不要惊慌。通常情况下，这种情况发生在你刚加入一个项目，还没有足够的信息时。如果发生这种情况，请不要试图自己解决问题或困难。

    这类情况的经验法则是：“如果有什么不清楚的地方，那是我们的错，而不是你的错。”因此，如果你不理解项目设计，那是项目设计师的责任。

    提交一个错误报告，请求解释一个设计概念。您将会为此报告获得报酬，并且您在回复中收到的信息将与所有其他开发人员共享。

    阅读这篇文章：Bugs Are Welcome。

    不要期待任何人来帮助你。你唯一的帮助来源就是源代码本身。如果代码没有解释你需要知道的一切，那就是一个bug，必须要报告。

    4. Commit and Push

    使用文本编辑器或集成开发环境进行任何必要的更改。在你做出更改后立即提交更改是一个良好的实践。不要在提交更改之前累积大量的更改。

    git commit -am '#123: the description of the changes'
git push origin 123

    如果您对工作范围有疑问，请在 GitHub 问题中发布并等待答复。如果您认为现有代码需要改进，请毫不犹豫地在 GitHub 上提交新的问题。

    不要试图在一个分支上修复所有问题；让其他程序员来处理它们。

    5. Pull Request

    在GitHub中使用以下文章中的过程创建拉取请求：使用拉取请求

    将其编号放在原问题中并等待反馈。

    6. Code Review

    过一段时间后，你的拉取请求将由项目团队中的某人进行审查。在许多情况下，你可能会收到一些负面评论，你必须修复与之相关的所有问题。除非你的更改符合审阅者的要求，否则你的拉取请求将不会合并到“主分支(master branch)”。

    对审核人要有耐心，并认真倾听他的意见。然而，不要认为你的审核人总是正确的。如果你认为自己的改动是合理的，坚持让其他人来审核它们。

    在您对分支进行所需的更改后，请不要忘记通过发布一个针对他/她的消息通知审查者在问题上。在每一轮的审查-修复循环中也要做同样的事情：请审查者再次查看您的代码。如果没有这样的通知，您的拉取请求可能会在沉默中停留更长时间。

    7. Merge

    当审核人员觉得一切都很好时，他会通知我们的自动合并机器人。自动合并机器人会选择您的拉取请求并尝试将其合并到master分支中。出于各种原因，这个操作经常失败。如果合并失败，无论原因如何，您有责任确保您的分支成功合并。

    如果由于与您的任务无关的测试失败而无法合并分支，请不要尝试自行修复。相反，请将问题报告为新的错误并等待其解决。

    请注意，在您的分支被合并之前，您是没有薪水的。

    8. Payment

    一旦您的更改被合并，返回GitHub问题页面并要求作者关闭它。一旦项目经理关闭了该问题，您将在几个小时内收到您的付款。

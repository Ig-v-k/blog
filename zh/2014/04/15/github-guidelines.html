    GitHub Guidelines

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    本手册解释了在GitHub上托管的XDSD项目中使用的工作流程。您会在GitHub问题分配给您时开始。接下来，您将收到项目经理的消息，其中包含问题编号、标题、描述以及预计工时预算（通常为30分钟）。

    如果您不同意预算分配，请毫不犹豫地要求增加预算。一旦您对预算感到满意并理解工作范围，请在回复工单时说明，并开始工作。请注意，超出预算分配的时间不会得到报酬。

    1. Fork

    即使您是开发团队的一员，但您没有在GitHub上对存储库进行写入访问权限。因此，要贡献更改，您应将存储库fork到自己的GitHub帐户（创建一个私人副本），进行所需的更改，然后使用“拉取请求”将其提交进行审查。

    在您提交拉取请求的审查后，存储库所有者通过将更改合并到主存储库中来批准您的更改。这是我们如何保护主要开发流免受意外损坏的方法。

    本文说明如何fork存储库：fork-a-repo

    这篇文章说明如何在计算机上下载和安装GitHub：set-up-git

    最后，不要忘记将您的私人SSH密钥添加到GitHub中：generating-ssh-keys

    2. Branch

    一旦您将我们的存储库分叉到您的账户中，请将其克隆到您的计算机上，然后检出“master”分支。例如：

    git clone git@github.com:yegor256/xembly.git
git checkout master

    现在，是时候创建分支了（123 是你要处理的 GitHub 问题的编号，也是分支的名称）。

    git checkout -b 123

    按照惯例，我们使用与您正在处理的分支和问题相同的名称。

    3. Changes

    所有与任务相关的问题都应在GitHub的issue中讨论。对于GitHub的issue，我们不使用电子邮件、Skype、电话或会议。所有问题应直接在GitHub的issue中提问。

    如果有什么不清楚或需要帮助的，请毫不犹豫地提交新的issue。接到一个你可能无法实现的任务是很常见的。不要惊慌。这通常发生在你刚加入一个项目时，还没有足够的信息。如果这种情况发生，不要试图自己解决问题或issue。

    对于这种情况的一般准则是：“如果有什么不清楚的，那是我们的错，而不是你的错。”因此，如果你不理解项目设计，那是项目设计者的错。

    提交一个错误报告，请求对设计概念进行解释。你会得到报酬，并且你收到的回复信息将与其他开发人员共享。

    阅读这篇文章：Bugs Are Welcome。

    不要期望任何人帮助你。你唯一的帮助来源是源代码本身。如果代码没有解释你需要知道的一切——那是一个bug，必须报告。

    4. Commit and Push

    使用文本编辑器或IDE进行必要的更改。在您进行更改时立即提交更改是一种良好的实践。不要在提交更改之前积累大量的更改。

    git commit -am '#123: the description of the changes'
git push origin 123

    如果您对工作范围有疑问，请在 GitHub 问题中发布并等待答复。如果您认为现有代码需要改进，请毫不犹豫地提交新的 GitHub 问题。

    不要试图在一个分支中修复所有问题；让其他程序员来处理它们。

    5. Pull Request

    使用以下文章中的流程在GitHub上创建一个pull请求：使用pull请求

    将其编号发布在原始问题中，并等待反馈。

    6. Code Review

    过一段时间，你的拉取请求将由项目团队的某人进行审核。在许多情况下，你可能会收到一些负面评论，你需要修复与这些评论有关的所有问题。在你的更改满足评审者之前，你的拉取请求不会合并到主分支。

    请耐心对待评审者，并仔细听取他的意见。然而，不要认为你的评审者总是对的。如果你认为你的更改是有效的，请坚持要求其他人重新审查。

    在你对分支进行所需的更改之后，不要忘记通过发布一条给他/她的留言来通知评审者。在每一轮的审查-修复循环中都要这样做：请评审者再次查看你的代码。如果没有这样的通知，你的拉取请求可能会在静默中停留更长时间。

    7. Merge

    当一切对审核者而言都看起来很好时，他会通知我们的自动合并机器人。然后，自动合并机器人会选择您的拉取请求，并尝试将其合并到“master”分支中。出于各种原因，这个操作经常失败。如果合并失败，不论原因如何，您有责任确保您的分支成功合并。

    如果由于与您的任务无关的测试失败而无法合并分支，请不要尝试自行修复。相反，请将问题报告为新的错误，并等待解决。

    请记住，在您的分支合并之前，您是没有被支付的。

    8. Payment

    当你的更改被合并后，回到 GitHub 的问题页面并要求作者关闭它。一旦该问题被项目经理关闭，你将在几小时内收到付款。

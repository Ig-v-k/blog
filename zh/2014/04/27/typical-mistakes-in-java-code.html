    Typical Mistakes in Java Code

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    这个页面包含我在与我一起工作的人的Java代码中看到的大部分典型错误。静态分析（我们使用qulice）由于明显的原因不能捕捉所有错误，这就是为什么我决定在这里列出它们的原因。

    如果你想看到其他内容添加在这里，请告诉我，我会乐意满足。

    所列出的所有错误都与面向对象编程一般和Java特别相关。

    Class Names

    你的类应该是现实生活实体的抽象，不应包含”验证器”、”控制器”、”管理器”等。如果你的类名以”-er”结尾，那么它是一个糟糕的设计。顺便说一下，这是我对好对象的七个特征。此外，这篇文章详细解释了这个想法：不要创建以-ER结尾的对象。

    当然，实用类是反模式，比如StringUtils、FileUtils和IOUtils等Apache的类库。上述都是糟糕设计的完美例子。阅读这篇后续文章：面向对象编程中替代实用类的方法。

    当然，永远不要为了区分接口和类而添加后缀或前缀。例如，以下所有命名都是错误的：IRecord、IfaceEmployee或RecordInterface。通常，接口的名称是真实实体的名称，而类的名称应该解释其实现细节。如果对于实现没有特别的说明，可以将其命名为Default、Simple或类似的名称。例如：

    class SimpleUser implements User {};
class DefaultRecord implements Record {};
class Suffixed implements Name {};
class Validated implements Content {};

    Method Names

    方法可以返回某些东西或返回void。如果一个方法返回某些东西，那么它的名称应该解释它返回了什么，例如（永远不要使用get前缀）：

    boolean isValid(String name);
String content();
int ageOf(File file);

    如果函数返回 void，那么它的名称应该解释它的作用。例如：

    void save(File file);
void process(Work work);
void append(File file, String line);

    你可以在《优雅的对象》这本书的第2.4节中了解更多关于这个想法的内容。刚刚提到的规则只有一个例外—JUnit的测试方法。下面会对它们进行解释。

    Test Method Names

    在JUnit测试中，方法名称应该以英文句子的形式创建，不包含空格。通过示例更容易解释。

    /**

* HttpRequest can return its content in Unicode.

* @throws Exception If test fails */ @Test public void returnsItsContentInUnicode() throws Exception { } ```







重要的是在你的 Javadoc 的第一句话中用被测试类的名称开头，接着是“可以”（或“不可以”）。因此，你的第一句话应该总是类似于“某人*可以*做某事”。

方法名会完全相同，但没有主语。如果我在方法名的开头加上主语，我应该得到一个完整的英文句子，就像上面的例子：“`HttpRequest` 以 Unicode 返回其内容。”

注意测试方法不以“可以”开头。只有 Javadoc 注释使用“可以”。

将测试方法声明为抛出 `Exception` 是一个好的习惯。

## Variable Names

避免使用复合变量名，比如`timeOfDay`，`firstItem`或`httpRequest`。我指的是类变量和方法内的变量。变量名应足够长，以避免在其可见范围内产生歧义，但如果可能的话不应太长。变量名应为单数或复数形式的名词，或者是适当的缩写。有关此内容的更多信息，请参阅此文章：《复合名称是代码异味》。例如：

    List names; void sendThroughProxy(File file, Protocol proto); private File content; public HttpRequest request;

    
有时候，如果构造函数将传入的数据保存在实例化对象中，可能会发生构造函数参数与类内属性之间的冲突。在这种情况下，我建议通过去除元音字母来创建缩写（参见[USPS缩写街道名称的方式](http://pe.usps.gov/text/pub28/28apc_002.htm)）。

Another example:

    public class Message { private String recipient; public Message(String rcpt) { this.recipient = rcpt; } }

    
在许多情况下，变量名称的最佳提示可以通过阅读其类名来确定。只需用小写字母编写即可，这样就可以了。

    File file; User user; Branch branch;

    


然而，对于原始类型（如`Integer number`或`String string`）*绝不*要这样做。

当存在具有不同特征的多个变量时，您还可以使用形容词。例如：

    String contact(String left, String right);

    
## Constructors

没有例外，只应该有一个构造函数来存储数据到对象变量中。其他的构造函数应该用不同的参数调用这个构造函数。例如：

    public class Server { private String address; public Server(String uri) { this.address = uri; } public Server(URI uri) { this(uri.toString()); } }

    
更多相关信息请参阅《只能有一个主构造函数》。

## One-time Variables

尽量避免使用一次性变量。所谓"一次性"指的是只使用一次的变量。就像这个例子中一样。

    String name = “data.txt”; return new File(name);

    
上述变量仅使用一次，代码应进行重构为：

    return new File(“data.txt”);

    
有时候，在非常罕见的情况下——主要是因为更好的格式设置——可能会使用一次性变量。然而，尽量避免这种情况以减少成本。

## Exceptions



毋庸置疑，你**绝不应该**吞没异常，而是应该让它们尽可能地上升。私有方法应该始终让已检查的异常传播出去。

永远不要使用异常来控制流程。例如，以下代码是错误的：

    int size; try { size = this.fileSize(); } catch (IOException ex) { size = 0; }

    
说真的，如果那个 `IOException` 提示“磁盘已满”，你还会假设文件大小为零然后继续吗？

## Indentation

对于缩进，主要规则是括号要么在一行结束，要么在同一行关闭（相反的规则适用于闭合括号）。例如，以下示例是不正确的，因为第一个括号没有在同一行关闭，并且在它后面还有符号。第二个括号也有问题，因为在它前面有符号，并且没有在同一行打开。

    final File file = new File(directory, “file.txt”);

    
正确的缩进应该如下所示：

    StringUtils.join( Arrays.asList( “first line”, “second line”, StringUtils.join( Arrays.asList(“a”, “b”) ) ), “separator” );

    
缩进的第二个重要规则是尽量将内容放在一行上，但字符数不得超过80个。上面的示例是无效的，因为它可以被压缩。

    StringUtils.join( Arrays.asList( “first line”, “second line”, StringUtils.join(Arrays.asList(“a”, “b”)) ), “separator” );

    
## Redundant Constants

类常量应该在你想要在类方法之间共享信息，并且这些信息是你的类的特征时使用。不要将常量用作字符串或数值字面量的替代品，这是一种非常糟糕的做法，会导致代码污染。常量（与面向对象编程中的任何对象一样）应该在现实世界中具有一定的意义。这些常量在现实世界中有什么意义：

    class Document { private static final String D_LETTER = “D”; // bad practice private static final String EXTENSION = “.doc”; // good practice }

    


另一个典型错误是在单元测试中使用常量来避免测试方法中的重复字符串/数字字面量。不要这样做！每个测试方法都应该使用自己的一组输入值。

在每个新的测试方法中使用新的文本和数字。它们是独立的。那么，为什么它们必须共享相同的输入常量呢？

## Test Data Coupling

这是一个测试方法中的数据耦合的示例。

    User user = new User(“Jeff”); // maybe some other code here MatcherAssert.assertThat(user.name(), Matchers.equalTo(“Jeff”)); ```

    在最后一行，我们将"Jeff"与第一行的相同字符串文字进行了结合。如果几个月后有人想要更改第三行的值，他/她必须额外花费时间找到同一方法中其他地方使用的"Jeff"。

    为了避免这种数据耦合，您应该引入一个变量。更多相关信息请参见：《有关单元测试搭建的几个想法》。

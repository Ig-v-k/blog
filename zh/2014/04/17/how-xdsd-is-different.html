    How XDSD Is Different

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    eXtremely Distributed Software Development（简称XDSD）是一种与传统软件开发团队工作方式显著不同的方法。大多数XDSD方法与传统方法有很大区别（但却很关键），以至于许多新手感到困惑。本文将帮助您加入一个以XDSD原则管理的项目后快速入门，无论您是开发人员还是项目赞助人。

    We Pay Only For Closed Tasks

    与许多其他项目不同的是，在XDSD中，我们只支付完成的任务和商定的时间预算。让我通过一个例子来解释。假设你是一个Ruby程序员，你得到一个新任务，需要你修复一个损坏的单元测试。这个任务的时间预算为30分钟，这在大多数情况下都是如此。有时候，任务的时间预算可能是15分钟或1小时。

    在我们的例子中，我们同意每小时50美元的合同费率。对于这个破损的测试，您将获得25美元的报酬，即按每小时50美元计费的30分钟任务。

    无论你实际修复测试所花费的时间有多长，都无关紧要。你在项目上真正花费的时间可能是五分钟或五个小时。尽管如此，你只会获得30分钟的工作报酬。如果你在5分钟内修复了破损的测试，你将获得25美元。即使完成这个任务花费了你一个小时，甚至一个月，你仍然只能获得25美元。

    此外，如果您未能修复单元测试并完全关闭任务，您将不会收到任何有关此作业的报酬。

    您可以在以下文章中查看有关此原则的更多详细信息：无责任原则或完成定义。

    如上所述，这是XDSD与其他方法之间最重要的区别之一。许多人在看到这个原则实施时感到困惑，有些人因此而离开我们的项目。他们习惯于每月底得到报酬，不管他们实际交付了多少工作量。在XDSD中，我们认为这种做法非常不公平。我们认为那些交付更多成果的人应该得到更多现金。相反地，那些没有交付成果的人应该得到较少的报酬。

    We Deliver Unfinished Components

    由于我们大部分的任务都是半小时的规模，我们鼓励开发人员交付未完成的组件。在下面的文章中了解更多关于这个概念的内容：拼图驱动开发（Puzzle Driven Development）。

    No Informal Communications

    与您可能合作过的其他项目或团队不同，XDSD不使用任何非正式的沟通渠道。需要澄清的是，我们从不使用电子邮件，不在Skype上聊天，也不进行任何会议或电话。此外，XDSD没有维护邮件列表。我们唯一的沟通方式是使用一个票务跟踪系统（在大多数项目中是GitHub Issues）。

    此外，我们不鼓励开发者之间关于个体任务范围的横向沟通。当分配任务时，你唯一的联系人（也是你唯一的客户）就是任务的作者。你可以通过工单与作者沟通以澄清任务需求。

    当任务的要求清晰明确，而且你完全理解它们时，将结果交给作者并等待他关闭任务。作者关闭任务后，项目经理会支付给你报酬。

    我们对这个原则非常严格——不允许非正式的沟通。然而，这并不意味着我们对你的意见和建设性的批评不感兴趣。相反，我们鼓励每个人提交他们的建议和漏洞。顺便说一句，我们会为漏洞支付报酬（有关漏洞报告和支付的详细信息，请参阅下一节）。

    由于我们没有正式的沟通方式，项目团队成员不需要在特定的时间工作。相反，团队成员可以根据自己所在的时区在方便的时间工作，包括工作日和周末。

    We Pay For Bugs

    与许多其他软件团队不同，XDSD 欢迎我们所有项目中的错误报告。因此，我们要求公开提出错误，并期望团队成员报告它们。请查看以下文章，了解 XDSD 的完整错误报告细节：欢迎报告错误（Bugs Are Welcome）。

    我们期望与项目有关的每个人都报告每个发现的错误。此外，我们鼓励团队成员提出建议。在XDSD中，我们为每个正确报告的错误支付团队成员的报酬。

    XDSD之所以为报告的漏洞付款，是因为我们相信发现的漏洞越多，最终产品的质量就越高。一些新开发者对于接收到“你必须在A类中找到10个漏洞”的任务感到惊讶。通常，自然的反应是问“如果没有漏洞怎么办？”然而，我们相信任何软件产品都可能存在无限数量的漏洞；只是需要花费时间和努力去发现它们而已。

    Only Pull Request

    我们从不授予团队成员对master分支的访问权限—无论你在项目上工作多长时间。因此，你必须始终通过拉取请求提交你的更改（我们的大多数项目都在GitHub上完成）。

    我们执行这个政策并不是因为我们不信任我们的开发人员，而是因为我们不信任任何人。当然，这只是一个玩笑。阅读这篇文章：主分支必须是只读的。

    No Compromises About Code Quality

    在将任何更改合并到master分支之前，我们会使用单元测试和静态分析器对整个代码库进行检查。单元测试是现代软件开发中非常常见的组件，你应该对此不感到惊讶。然而，静态分析的严格性常常会让XDSD的新手感到沮丧，我们理解这一点。我们对源代码的质量和统一性比大多数竞争的软件开发团队更加关注。

    更重要的是，我们绝不会妥协。如果您的拉取请求违反了静态分析器的任何一个规则，都将被拒绝。而且，不管这个违规看起来多么微小或无意，都没有关系。这个合并过程是完全自动化的，无法绕过。

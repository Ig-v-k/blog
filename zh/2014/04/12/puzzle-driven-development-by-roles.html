    PDD by Roles

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    在这篇文章中，我将尝试以谜题驱动开发（PDD）的精神来为您介绍一个项目。在此过程中，我将试图传达各个项目成员的典型观点。

    基本上，在任何软件团队中都有几个关键角色：

    • 项目经理—分配任务并在完成后支付报酬

    • 系统分析师—记录产品负责人的想法

    • 架构师—定义系统组件如何相互交互

    • 设计师—负责实现最复杂的组件

    • 程序员—实现所有组件

    • 测试人员—发现并报告错误

    除了项目经理以外，每个人以两种方式影响项目：他们修复项目，同时也会破坏项目。让我用一个简单的例子来解释这个问题。

    Fix and Break

    假设为了简单起见，一个项目是我为一个亲密的朋友编写的一个简单软件工具。我创建了第一个草稿版本0.0.1并交给了他。对我来说，这个项目已经完成了。我完成了工作，希望再也不用回来了。

    然而，项目的现实情况却大相径庭。仅仅几个小时后，我接到朋友的电话，说他在这个工具中发现了几个错误。他要求我修复它们。现在，我可以看到这个项目并没有完成。事实上，它有几个错误，这意味着还有几个任务需要完成。

    我打算修复这个项目，通过移除错误。我实现了软件的一个新版本，命名为0.0.2并且将它发送给我的朋友。再次强调，我相信我的项目已经完成。它已经修复好了，应该被关闭。

    这种情况一再重复，直到我的朋友不再给我打电话。换句话说，直到他不再破坏我的项目。

    很明显，我的朋友越是破坏我的项目，最终交付的软件质量就越高。0.0.1版本只是一个非常初步的版本，尽管我当时发布时认为它是最终版本。几个月后，经过修复了数百个错误之后，3.5.17版本将更加成熟和稳定。

    这是这种“修复和破坏”方法的结果。

    该图显示了项目中时间和混乱之间的关系。我的朋友向我报告的错误破坏了项目，增加了它的不稳定性（或者简单地说，增加了它的混乱程度）。我发布的新版本解决了这些错误，并修复了项目。你的 GitHub 提交动态应该与此图类似，例如：

    当项目开始时，它的混乱程度相对较低，然后开始增长。混乱程度达到最高点后开始下降。

    Project Manager

    项目经理的工作是尽力修复项目。他必须利用赞助商的时间和资金来消除所有的错误和不一致，并将项目恢复到“修复”状态。

    当我说“错误”时，我不仅指软件错误，还包括：

    • 不明确或模糊的需求

    • 尚未实施的功能

    • 功能性和非功能性错误

    • 测试覆盖不足

    • 未解决的 @todo 标记

    • 缺乏风险分析

    • etc.

    项目经理给我分配任务，他希望通过完成这些任务来修复和稳定项目，将其恢复到一个没有缺陷的状态。

    作为一个软件团队的成员，我的工作是帮助他进行必要的修复，同时尽力去破坏这个项目！就像我朋友的例子一样，他通过向我报告bug不断破坏这个项目。这就是他如何帮助我们两个人提高产品的最终质量。

    我应该做相同的事情，并且在我工作在某个功能上时，总是尽力报告新的错误。我应该同时修复和破坏。

    现在让我们更仔细地看一下项目角色。

    System Analyst

    产品负责人提交一个非正式的功能请求，通常以“如果有…就好了”开始。我是一个系统分析师，我的工作是将负责人的英文翻译成软件需求规格说明书中的正式规格，以便程序员和我理解。实现该功能不是我的责任。

    我的任务在于当SRS的新版本被变更控制委员会签署后完成。我是产品负责人的口译员，将他们的语言翻译成SRS文档所需的正式语言。我的唯一客户是产品负责人。一旦她关闭功能请求，我就会得到报酬。

    除了来自产品所有者的功能请求之外，我经常收到关于SRS质量的投诉。对于一些团队成员来说，这个文档可能不够清晰。因此，我的工作是解决清晰度问题并修复SRS。这些团队成员也是我的客户。当他们关闭他们的错误报告时，我就会得到报酬。

    在这两种情况下（功能请求或错误修复），只要我有足够的时间，我可以立即对SRS进行更改。然而，并不总是可能的。我可以提交一个错误并等待解决；但是，我不想让我的客户等待。

    这就是谜题驱动开发对我有帮助的地方。与其提交错误报告，我会在SRS文档中添加“TBD”谜题。这些谜题是通常非常严格的正式需求的非正式替代品。它们满足我的客户，因为它们是用简单的英语编写的，技术人员可以理解。

    因此，当我没有时间时，我不会等待。在我无法创建一个适当和正式的需求描述或者不知道该写什么的地方，我会使用TBD-s来修改SRS。

    Architect

    现在，我是架构师，我的任务是实现一个在SRS中正式规定的需求。项目经理期望我提供一个可工作的功能，而我只有在架构清晰且类已经设计和实现之后才能交付。

    作为一名建筑师，我的责任是将所有组件组装在一起，并确保它们的适配。在大多数情况下，我不是自己创建这些组件，而是告诉每个人应该如何创建它们。我的工作流程如下：

    “[SRS] -> [UML] [UML] -> [Source code]” translated into Chinese without translating technical terms and proper nouns: “[SRS] -> [UML] [UML] -> [源代码]”

    我从SRS接收需求，生成UML图表，并向设计师解释如何根据我的图表创建源代码。我并不关心源代码的实现方式。我更关心组件的交互以及整体架构如何满足功能和非功能要求。

    当系统分析师在SRS中将其状态更改为“已实施”，我的任务将被关闭并支付。系统分析师是我的唯一客户。我必须将我的解决方案销售给他。项目经理将在系统分析师将功能要求的状态从“指定”更改为“已实施”时关闭我的任务并支付我。

    这项任务听起来很庞大，而且我只有半个小时的时间。显然，以谜题驱动的开发应该能帮助我。我将创建很多工单和谜题。例如：

    • SRS无法准确解释需求。

    • 非功能性需求不清晰

    • UML图表不够清晰。

    • 组件尚未实现

    • 构建不是自动化的

    • 持续集成未配置

    • 代码质量无法控制。

    • 性能测试不是自动化的

    当我所有的难题都被解决后，我可以回到我的主要任务并完成功能实现。显然，这可能需要很长时间 - 几天甚至几周。

    但是，主要任务的时间成本不到一个小时。这么辛苦工作的意义何在呢？其实很简单；我会通过所有报告的错误来赚取我的小时数。从这个小的半小时任务中，我将产生许多工单，而每一个都会给我额外的现金。

    Designer and Programmer

    设计师和程序员之间唯一真正的区别在于他们各自任务的复杂性和每小时收费标准。设计师通常进行更复杂和更高层次的实施，而程序员负责实施所有低层细节。

    我是一个程序员，我的任务是实现一个类或方法，或者修复一些功能上的错误。在大多数情况下，我只有半个小时的时间。而且，大部分任务都比这需要更多的时间。

    拼图驱动开发帮助我将任务分解为更小的子任务。我总是从单元测试开始。在单元测试中，我尝试复现一个错误或者模拟一个功能。当我的测试失败时，我会提交它并确定剩余的时间。如果我还有时间使其通过，我会进行修改，提交更改并向项目经理报告。

    如果我没有时间实施修复，我会标记那些没有@todo标记的代码片段，提交它们并向项目经理报告我已经完成了。

    就像你所看到的，我正在修复代码的同时也破坏它。我正在使用我的新单元测试来修复它，但是用@todo谜题来破坏它。

    这是我如何帮助提高项目整体质量的方式 - 同时修复和破坏。

    Tester

    我是一名测试员，我的主要动机是发现错误。这可能与你之前听到的相矛盾；但在XDSD中，我们计划在项目的每个阶段发现一定数量的错误。

    作为一个测试人员，我从项目经理那里接收任务。这些任务通常类似于“审查功能X并找出其中的10个错误”。项目经理需要找到一定数量的错误才能修复项目。从他的角度来看，当找到200个错误时，项目就被修复了。这就是为什么他要求我找到更多错误。

    因此，为了响应请求，我在整体工作中为了”修复”部分找出了一些错误。与此同时，我也可以自行发现缺陷并报告它们。这是我任务中的”破坏”部分。

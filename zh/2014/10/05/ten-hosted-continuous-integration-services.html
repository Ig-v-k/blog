    Best Hosted Continuous Integration Services for a Private Repository

    Best Four

    Shippable 在配置方面很容易，因为它可以直接理解 .travis.yml。界面易于导航，因为它根本没有“设置”页面（或者我没有找到）。所有的配置都通过存储库中的 shippable.yml 文件完成。该服务看起来稳定且健壮，目前没有任何抱怨。关于它们特别酷的是它们允许在 Docker 容器中构建。

    Travis 是我迄今见过的最好的平台。主要是因为它是最受欢迎的。与GitHub完美集成，并且有完善的文档。一个重要的缺点是每月129美元的价格。”用这笔钱，你可以获得一个专用的EC2实例，并在那里安装Jenkins”——我的一些客户这样说。我强烈不同意，因为Jenkins需要24x7的管理，这比129美元要贵得多，但往往很难解释清楚。

    Wercker 是来自阿姆斯特丹的欧洲产品，目前仍处于测试版阶段，因此对所有项目免费。这个平台看起来非常有前途。对于私有仓库，它仍然是免费的，并且有投资支持。他们还有一个有趣的构建”盒子”的概念，可以像Docker容器一样进行预配置。在过去几个月中，它的稳定性相当好，目前还没有任何投诉。

    AppVeyor 是唯一一个可以运行 Windows 构建的工具。尽管我主要使用 Java 和 Ruby 这两种预期具备平台独立性的语言，但它们常常表现出相反的情况。当你的构建在 Linux 上成功时，几乎无法保证它能在 Windows 或 Mac 上通过。我计划在每个项目中都使用 AppVeyor，并与其他 CI 服务结合使用。以下是我如何将 Maven 构建与 AppVeyor 集成。

    Others

    SemaphoreApp 配置简单且易于使用。它给人以轻量级系统的印象，这是我非常欣赏的。不足之处在于他们没有预安装任何Maven，而是使用一个旧版本的Maven，但这可以通过一个简短的自定义脚本轻松解决，该脚本会下载并解压最新版本的Maven。另一个不足之处是它们无法通过文件（例如.travis.yml）进行配置—您必须通过用户界面完成一切操作。它们还支持构建之间的缓存。

    Snap-CI 是 ThoughtWorks 的产品，是 Go 的作者，一个开源的持续集成服务器。它看起来比其他的复杂一些，它可以让你定义“阶段”并将它们组合成流水线。我还不确定这些机制如何对我们大多数工作在小型和中型项目中的帮助，但它们看起来很“酷”。另外，每个构建只能使用2Gb的内存，这是一个非常不幸的限制—我的一些Java项目因此而失败。除此之外，他们不提供对构建服务器的完全访问权限，例如，我们无法修改/etc中的任何东西—这对我们来说是一个阻碍。

    Codeship 运行良好，但他们的网页界面看起来有点过时。除此之外，他们承诺与拉取请求一起工作，但我没有成功配置它们。尽管它们构建了拉取请求，但它们不会在GitHub中通知我们。也许我会找到方法，目前还不清楚。

    CircleCI 我仍然不知道为什么我的构建失败了。真的很难配置和理解发生了什么。正在努力弄清楚…

    SolanoLabs 看起来相当不成熟且难以配置。他们甚至不支持在添加新仓库时自动配置GitHub钩子。然而，他们的销售人员却非常积极地向我发送垃圾邮件。

    Hosted-CI 只适用于iOS/OSX。他们不提供任何免费的服务，即使是针对开源项目。我还没有机会测试他们。

    CloudBees 基本上是一个托管的Jenkins。我不是很喜欢Jenkins，所以无法推荐这个平台。

    DeployBot居然不允许我通过GitHub登录？它们似乎更注重”部署”，而不仅仅是持续集成。

    Vexor 看起来不错，并且提供了一种独特的计费模式——按构建次数收费，而不是按月收费。我肯定会推荐你去试试。不过，我无法让它正常工作…

    GreenHouseCI 是一个用于移动应用（iOS、Android等）的CI平台。看起来很有趣，只是我没有一个完整的移动应用来测试它。

    BTW, if you don’t like the idea of keeping continuous integration in cloud, consider these on-premise software packages (in order or preference): Jenkins, TeamCity, Go, Strider, BuildBot.

    请记住，无论您的持续集成服务有多好和昂贵，除非您将主分支设为只读，否则它将无法帮助您。

    1 This means that the platform can build your repo in Linux environment. Almost all of them do that by default, unless you configure them otherwise.
    2 Some of them can build on Windows platform.
    3 MacOS support means that an Objective-C/Swift product can be built there.
    4 I mean GitHub pull request support here. Some of them can be integrated with GitHub and will build pull requests before they are merged. Build status will be visible in GitHub. A pretty convenient feature.
    5 Logs compression is a critical feature, at least for me. Most of my logs are from Maven and without col -b they look too long and unreadable.
    6 Looks like Docker containers must be supported by all of them, but unfortunately it’s not the case. Ideally, all builds should run in containers.

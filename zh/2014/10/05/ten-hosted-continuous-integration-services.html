    Best Four

    Shippable 非常容易配置，因为它能够直接理解 .travis.yml。用户界面易于导航，因为它根本没有 “设置” 页面（或者我没有找到）。所有的配置都通过存储库中的 shippable.yml 文件进行。该服务看起来稳定且健壮，迄今为止没有任何投诉。他们特别酷的地方是允许在 Docker 容器中构建。

    Travis 是我迄今为止见过最好的平台。主要原因是它是最受欢迎的。它与GitHub完美集成，并且有适当的文档。一个重要的缺点是每月129美元的价格。”用这笔钱你可以获得一台专用的EC2实例并在那里安装Jenkins” —— 我的一些客户说。我强烈不同意，因为Jenkins将需要24x7的管理，这比129美元的费用要高得多，但很难解释清楚。

    Wercker是一款来自阿姆斯特丹的欧洲产品，目前仍处于测试阶段，因此对所有项目均免费。这个平台看起来非常有前景。私有代码库仍然免费，并且得到了投资的支持。他们还有一个有趣的构建“盒子”概念，可以类似于Docker容器进行预配置。过去几个月来，它的稳定性非常好，目前还没有任何投诉。

    AppVeyor 是唯一一个运行 Windows 构建的服务。尽管我主要使用 Java 和 Ruby，它们被认为是跨平台的，但它们往往表现出相反的情况。当你的构建在 Linux 上成功时，几乎没有保证它能在 Windows 或 Mac 上通过。我计划在每个项目中使用 AppVeyor，并与其他 CI 服务结合使用。下面是我如何将 Maven 构建 与 AppVeyor 集成。

    Others

    SemaphoreApp 容易配置和使用。它给人一种轻量级系统的印象，这一点我非常欣赏。不足之处是，他们没有预装任何 Maven，但这可以通过一个简短的自定义脚本来解决，该脚本下载并解压最新的 Maven 版本。另一个不足之处是他们不能通过一个文件（如 .travis.yml）进行配置—您需要通过用户界面完成所有操作。他们还支持构建之间的缓存。

    Snap-CI 是 ThoughtWorks 的产品，它是 Go 的创作者，一个开源的持续集成服务器。它看起来比其他的要复杂一些，允许您定义“阶段”并将它们组合成流水线。我还不确定这些机制如何对我们主要在小型和中型项目上工作的项目有所帮助，但它们看起来很“酷”。另外，每次构建的内存限制为2GB，这是非常不幸的限制——我的一些Java项目因此而失败。除此之外，他们不提供对构建服务器的完全访问权限，例如我们无法修改/etc中的任何内容——这对我们来说是一个阻碍。

    Codeship 运行良好，但是他们的网页界面看起来有点过时。除此之外，他们承诺可以与拉取请求一起工作，但是我没有成功配置它们。他们根本不会通知我们在 GitHub 上的拉取请求，尽管他们会构建这些请求。也许我会找到一种方法，目前还不清楚。

    CircleCI 我还不知道为什么我的构建失败。真的很难配置和理解发生了什么。正在尝试弄清楚…

    SolanoLabs 看起来相对不成熟且难以配置。当添加新的仓库时，他们甚至不支持自动配置 GitHub 钩子。然而，他们的销售人员却非常积极地给我发送垃圾邮件。

    Hosted-CI 仅适用于iOS/OSX。他们不提供任何免费的服务，即使是针对开源项目也是如此。我还没有机会测试他们。

    CloudBees 基本上是一个托管的 Jenkins。我真的不喜欢 Jenkins，所以不能推荐这个平台。

    DeployBot 连我都不能通过 GitHub 登录，哎？他们似乎更注重“部署”，而不仅仅是持续集成。

    Vexor 看起来很不错，并且提供了一种独特的计费模式——按构建次数收费，而不是按月计费。我绝对建议你试一试。不过，我无法让它正常工作…

    GreenHouseCI 是一个面向移动应用（iOS、Android等）的持续集成平台。似乎很有意思，只是我没有一个大规模的移动应用来测试它。

    BTW, if you don’t like the idea of keeping continuous integration in cloud, consider these on-premise software packages (in order or preference): Jenkins, TeamCity, Go, Strider, BuildBot.

    请记住，无论您的持续集成服务有多好、多么昂贵，除非您将主分支设置为只读，否则它都无法帮助您。

    1 This means that the platform can build your repo in Linux environment. Almost all of them do that by default, unless you configure them otherwise.
    2 Some of them can build on Windows platform.
    3 MacOS support means that an Objective-C/Swift product can be built there.
    4 I mean GitHub pull request support here. Some of them can be integrated with GitHub and will build pull requests before they are merged. Build status will be visible in GitHub. A pretty convenient feature.
    5 Logs compression is a critical feature, at least for me. Most of my logs are from Maven and without col -b they look too long and unreadable.
    6 Looks like Docker containers must be supported by all of them, but unfortunately it’s not the case. Ideally, all builds should run in containers.

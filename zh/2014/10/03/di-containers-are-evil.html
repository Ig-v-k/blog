    Dependency Injection Containers are Code Polluters

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    虽然依赖注入（也称为“DI”）是面向对象编程中自然的对象组合技术（在马丁·福勒引入这个术语之前就已经存在了很长时间），但是Spring IoC、Google Guice、Java EE6 CDI、Dagger和其他依赖注入框架将其转变成了一种反模式。

    我不打算讨论对”setter injections”（例如在Spring IoC中）和”field injections”（例如在PicoContainer中）的明显反对意见。这些机制违背了面向对象编程的基本原则，鼓励我们创建不完整的、可变的对象，在应用程序执行过程中填充数据。请记住：理想的对象必须是不可变的，并且不应包含setter。

    相反，让我们来谈谈“构造函数注入”（比如在Google Guice中）以及它与依赖注入容器的使用。我将试图展示为什么我认为这些容器至少是多余的。

    What is Dependency Injection?

    这就是依赖注入（与普通的对象组合没有太大区别）的含义。

    public class Budget {
  private final DB db;
  public Budget(DB data) {
    this.db = data;
  }
  public long total() {
    return this.db.cell(
      "SELECT SUM(cost) FROM ledger"
    );
  }
}

    对象 data 被称为 “依赖项”。

    Budget不知道它正在使用哪种数据库。它只需要从数据库中获取一个单元格的能力，使用任意的SQL查询，通过cell()方法。例如，我们可以通过实现DB接口的PostgreSQL来实例化一个Budget。

    public class App {
  public static void main(String... args) {
    Budget budget = new Budget(
      new Postgres("jdbc:postgresql:5740/main")
    );
    System.out.println("Total is: " + budget.total());
  }
}

    换句话说，我们在一个新对象budget中“注入”了依赖。

    与这种“依赖注入”方法相对的是让Budget决定它想与哪个数据库一起工作的方法。

    public class Budget {
  private final DB db =
    new Postgres("jdbc:postgresql:5740/main");
  // class methods
}

    这很脏，导致了1）代码重复，2）无法重用，3）无法测试等等。无需讨论为什么。这是明显的。

    因此，通过构造函数进行依赖注入是一种令人惊奇的技术。嗯，实际上并不仅仅是一种技术。更像是Java和其他面向对象语言的一个特性。几乎任何对象都希望封装一些知识（也就是所谓的“状态”）。这就是构造函数的作用。

    What is a DI Container?

    到目前为止一切都好，但接下来是黑暗面——依赖注入容器。以下是它的工作方式（我们以Google Guice为例）：

    import javax.inject.Inject;
public class Budget {
  private final DB db;
  @Inject
  public Budget(DB data) {
    this.db = data;
  }
  // same methods as above
}

    请注意：构造函数使用了 @Inject 注解。

    然后，在应用程序启动时，我们需要在某个地方配置一个容器。

    Injector injector = Guice.createInjector(
  new AbstractModule() {
    @Override
    public void configure() {
      this.bind(DB.class).toInstance(
        new Postgres("jdbc:postgresql:5740/main")
      );
    }
  }
);

    有些框架甚至允许我们在XML文件中配置注入器。

    从现在开始，我们不允许像之前那样使用new操作符来实例化Budget。相反，我们应该使用刚刚创建的注入器。

    public class App {
  public static void main(String... args) {
    Injection injector = // as we just did in the previous snippet
    Budget budget = injector.getInstance(Budget.class);
    System.out.println("Total is: " + budget.total());
  }
}

    该注入机制会自动发现，要实例化 Budget 对象，就必须为其构造函数提供参数。它将使用我们在注入器中实例化的 Postgres 类的实例。

    这是使用Guice的正确和推荐方式。然而，还有一些更糟糕的模式是可能的，但不推荐使用。例如，您可以将您的注入器作为单例并在Budget类中直接使用它。然而，即使是DI容器制造商也认为这些机制是错误的。让我们忽略它们，专注于推荐的情况。

    Setter injection

    我们创建的容器只是在代码库中添加了更多行，甚至更多文件（如果我们使用XML）。它并没有添加任何东西，除了额外的复杂性。如果我们有一个问题：“什么数据库用作预算的参数？”我们应该始终记住这一点。

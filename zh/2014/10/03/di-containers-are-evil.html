    Dependency Injection Containers are Code Polluters

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    尽管依赖注入（也称为“DI”）是面向对象编程中一种自然的对象组合技术（在 Martin Fowler 引入该术语之前就有人使用了），但是 Spring IoC、Google Guice、Java EE6 CDI、Dagger 和其他 DI 框架将其转化为一种反模式。

    我不打算讨论对”setter injections”（比如在Spring IoC中）和”field injections”（比如在PicoContainer中）的明显争论。这些机制简单地违反了面向对象编程的基本原则，并鼓励我们创建不完整的、可变的对象，在应用程序执行过程中被填充数据。请记住：理想的对象必须是不可变的，不能包含setter方法。

    取而代之，让我们谈谈“构造函数注入”（就像Google Guice中的一样）及其与依赖注入容器的使用。我将尝试展示为什么我认为这些容器至少是多余的。

    What is Dependency Injection?

    这就是所谓的依赖注入（与普通的对象组合并无太大不同）。

    public class Budget {
  private final DB db;
  public Budget(DB data) {
    this.db = data;
  }
  public long total() {
    return this.db.cell(
      "SELECT SUM(cost) FROM ledger"
    );
  }
}

    对象 data 被称为“依赖项”。

    Budget 不知道它正在使用哪种类型的数据库。它只需要从数据库中获取一个单元格的能力，可以使用任意的 SQL 查询，通过 cell() 方法。例如，我们可以使用 PostgreSQL 实现的 DB 接口来实例化一个 Budget。

    public class App {
  public static void main(String... args) {
    Budget budget = new Budget(
      new Postgres("jdbc:postgresql:5740/main")
    );
    System.out.println("Total is: " + budget.total());
  }
}

    换句话说，我们正在向一个新对象 budget 中”注入”一个依赖项。

    另一种替代“依赖注入”方法的方式是让Budget自行决定要与哪个数据库配合工作：

    public class Budget {
  private final DB db =
    new Postgres("jdbc:postgresql:5740/main");
  // class methods
}

    这很脏，导致 1）代码重复，2）无法重用和3）无法测试等等。不需要讨论为什么。这是很明显的。

    因此，通过构造函数进行依赖注入是一种很棒的技术。嗯，实际上并不仅仅是一种技术。更像是Java和所有其他面向对象语言的一个特性。几乎任何对象都希望封装一些知识（也就是所谓的“状态”）。这就是构造函数的作用。

    What is a DI Container?

    到目前为止一切都很顺利，但现在出现了一个阴暗面——依赖注入容器。以下是它的工作原理（我们以Google Guice为例）：

    import javax.inject.Inject;
public class Budget {
  private final DB db;
  @Inject
  public Budget(DB data) {
    this.db = data;
  }
  // same methods as above
}

    请注意：构造函数被标注为 @Inject。

    然后，在应用程序启动时，我们应该在某个地方配置一个容器。

    Injector injector = Guice.createInjector(
  new AbstractModule() {
    @Override
    public void configure() {
      this.bind(DB.class).toInstance(
        new Postgres("jdbc:postgresql:5740/main")
      );
    }
  }
);

    有些框架甚至允许我们在XML文件中配置注入器。

    从现在开始，我们不允许像以前那样使用new运算符来实例化Budget。相反，我们应该使用刚刚创建的注入器。

    public class App {
  public static void main(String... args) {
    Injection injector = // as we just did in the previous snippet
    Budget budget = injector.getInstance(Budget.class);
    System.out.println("Total is: " + budget.total());
  }
}

    注入器会自动发现，为了实例化Budget，它必须为其构造函数提供一个参数。它将使用我们在注入器中实例化的Postgres类的实例。

    这是使用Guice的正确和推荐方式。然而，还有一些更不推荐的模式是可能的。例如，您可以将您的注入器设置为单例，并在Budget类内部使用它。然而，即使是DI容器的制造商也认为这些机制是错误的。让我们忽略它们，专注于推荐的情况。

    What Is This For?

    让我重申和总结一下依赖注入容器的错误使用场景:

    • Field injection

    • Setter injection

    • 将注入器作为依赖项传递。

    • 使注入器成为全局单例

    如果我们把它们都放在一边不管，我们剩下的只有上面解释的构造函数注入。那么这对我们有什么帮助呢？为什么我们需要它？为什么不能在应用程序的主类中使用简单的 new 关键字？

    我们创建的容器只是在代码库中添加了更多行，甚至可能添加更多文件，如果我们使用XML。它除了增加复杂性之外没有任何其他作用。如果我们有这样的问题：”预算的参数使用了什么数据库？”，我们应该始终记住这一点。

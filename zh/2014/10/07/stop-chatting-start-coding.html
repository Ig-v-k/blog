    Stop Chatting, Start Coding

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    eXtremely Distributed Software Development（XDSD）的第一原则是“每个人都为经过验证的可交付成果获得报酬”。这意味着，为了获得报酬，每个程序员都必须编写代码，将其提交到代码库，通过代码审查，并确保代码合并到目标分支中。只有这样，他的成果才会被认可并支付报酬。

    对我大多数客户来说，这听起来已经很极端了。他们习惯了按小时或按月付费的传统模式。然而，他们立即意识到XDSD的好处，因为对他们来说，这种方法意味着项目资金不会被浪费在无法产生结果的活动上。

    But that’s not all.

    这个原则还意味着除了明确分配给他/她的任务外，没有人会因为任何事情而被付费。因此，当一个程序员对当前的设计、规范、配置等方面有问题时，没有人会有兴趣回答。为什么呢？因为这个问题是没有付费的。在XDSD中，通过Skype、Slack、HipChat或电子邮件回答问题是不受欢迎的。这个项目根本不会为这种活动付费。这就是为什么我们的程序员没有人这样做的原因。

    关于这个理念的更多内容请点击这里：It’s Not a School!

    在XDSD项目中，我们绝对不会使用任何（我是认真的！）非正式的沟通渠道。我们不进行会议或电话会议。我们绝不会通过Skype或电话讨论任何技术问题。

    那么，我们如何解决问题和共享信息呢？

    我们使用任务跟踪系统来处理这个。当开发人员有问题时，他会将其提交为一个新的“工单”。然后，项目经理会接手并将其分配给另一个能够回答该问题的开发人员。然后，答案会通过跟踪系统或直接进入源代码。

    “问题票据”在其作者对答案感到满意时关闭。当票据关闭时，回答问题的人将得到报酬。

    使用这个模型，我们显著改善了项目沟通，使其更加清晰透明。我们还节省了大量的项目资金，因为团队成员所花费的每个小时都可以追溯到他所编写的代码行。

    你可以通过实际案例看到这是如何发生的，例如，在这个工单中（该项目是开源的，这就是为什么所有的交流都是公开的）：jcabi/jcabi-github#731。一位Java开发者在他的Git仓库中遇到了问题。显然他做错了什么，无法自己解决问题。他通过提交一个新的错误报告向项目寻求帮助。他因为这个错误报告而得到了报酬。然后，另一个团队成员被指派去帮助他。通过一系列的建议和指导，问题最终得到了解决，并且他也因为解决方案而得到了报酬。总共，该项目花费了45分钟，问题得到了解决。

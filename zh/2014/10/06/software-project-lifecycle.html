    Project Lifecycle in Zerocracy

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    除了作为一个亲力亲为的程序员外，我还是Zerocracy的联合创始人和首席技术官，这是一家专门从事定制软件开发的公司。在我们参与的所有项目中，我扮演着技术和管理方面的领导角色。

    我为那些对雇佣我和/或我的团队感兴趣的人写了这篇文章。当您选择与我们合作时，本文将展示从第一天到项目结束的情况。

    您将会看到下面我们的软件开发方法与许多其他团队所使用的方法有很大的不同。我个人非常注重代码质量以及连接我们团队的内部流程的质量。

    在我与Zerocracy中的每个项目中，有四个阶段：

    • 思考。在这里，我们试图理解：这个产品将解决什么问题？我们还在调查产品的边界—谁将与软件一起工作（参与者）以及他们将如何使用它（用户故事）。可交付成果：规范。持续时间：从2天到3周不等。参与者：产品所有者、分析师、架构师、项目经理。

    • 构建。在这里，软件架构师正在创建概念验证（也称为MVP或原型或骨架）。这是一个由一个人完成的工作，几乎没有与其他人的任何互动。架构师根据规范在非常有限的时间内构建产品。结果可能会有多个错误和未完成的部分，但它将实现主要的用户故事。架构师还配置了持续集成和交付流程。交付成果：可工作的软件。持续时间：2-5 天。参与者：架构师。

    • 修复。在这个阶段，我们正在向骨架中添加所有的内容。这个阶段需要花费大部分的时间和预算，涉及到许多参与者。在某些项目中，我们邀请多达50人同时工作。由于我们将所有的不一致都视为错误，所以这个阶段主要是为了发现、报告和修复错误，以稳定产品并准备好市场推出。我们每天都会增量和发布软件，最好是给用户冠军。可交付成果：通过拉取请求进行错误修复。持续时间：从几周到几个月。参与者：程序员（们），设计师（们），测试人员（们），代码审查人员（们），架构师，项目经理。

    • 使用。在这个最后阶段，我们将产品推向最终用户，并收集他们的反馈意见（无论是积极还是消极的）。他们向我们报告的一切都被记录为错误。然后，我们对这些错误进行分类和修复。这个阶段可能需要几年时间，但从不涉及新功能的主动实施。可交付成果：通过拉取请求进行错误修复。持续时间：几个月。参与者：程序员，代码审查员，项目经理。

    最大的（即最长且最昂贵的）阶段当然是Fixing（修复）。这通常需要大部分时间（超过70%）。然而，最重要且风险最大的阶段是第一个阶段——Thinking（思考）。在思考阶段犯的错误将比后面犯的错误代价更大。

    Thinking

    思考是第一阶段，也是最重要的阶段。

    首先，我们为项目取一个名字并创建一个GitHub存储库。我们尽量将所有的项目（包括开源和商业项目）都放在GitHub上。这主要是因为该平台非常受欢迎、功能强大，而且价格也非常便宜（$7/mo 可以享受5个私有项目的套餐）。我们还会在GitHub的问题跟踪器中进行所有的沟通。

    然后，我们创建一个简单的半页 SRS 文档（软件需求规格说明书）。通常这是直接在源代码中完成的，但有时会在 GitHub 的 README.md 文件中完成。重要的是，这个文档应该受版本控制。在项目过程中，我们会对其进行非常频繁的修改。README.md 应该简要地识别系统的主要 “参与者” 并定义产品范围。

    尽管这只是半页纸，但创建这个初始SRS文件是整个项目中最重要且最昂贵的任务。我们非常重视这一步骤。通常，这份文件由我们的系统分析师之一与项目发起人进行直接沟通而编写。我们不能在这一步犯错误。

    然后，我们邀请了几位新的系统分析师加入项目。这些人负责将我们最初的 README 转化为一个更完整和详细的规范。他们首先通过提问的方式开始工作，将每个问题作为 GitHub 问题逐一提交。每个问题都是针对产品负责人提出的。系统分析师根据他/她的答案修改 README 文档。有时我们使用 Requs。

    在思考阶段的最后，我们估计项目的规模，以代码行为单位。通过使用这个代码行指标，我们可以大致地估计预算。

    Building

    这是一项建筑师的单人工作。我们每个项目都有一位建筑师，他个人负责质量和技术决策。我们有几位出色的工程师担任这个角色。

    建设阶段相对来说非常简单。根据README，建筑师需要在几个工作日内实施解决方案。无论想法有多么宏大，规划开发有多么庞大，建筑师仍然需要在三天内创建（从零开始！）产品。

    除了构建软件本身之外，架构师还需要配置所有基本的DevOps流程，包括：1）自动化测试和质量控制，2）部署和发布流水线，3）制品库，4）持续集成服务，等等。

    这个阶段的结果是一个可部署到目标位置并供测试人员使用的可工作软件包。在这个阶段还会定义技术质量要求。

    关于建设阶段的更多信息，请参阅：开展软件项目的九个步骤

    Fixing

    现在是建立一个分布式程序员团队的时候了。首先，我们邀请那些在其他项目上工作过并已经证明了他们的质量的人。我们经常邀请新人，通过 Stack Overflow、GitHub、Upwork 和其他来源找到他们。一个普通项目的平均团队规模是15-25名程序员。

    在这个阶段，我们将任何不一致之处都视为错误。如果文档中有不清楚的地方，或者某些内容可以进行重构以提高可读性，或者某个函数可以改进以提升性能——对我们来说，这些都是错误。而且我们欢迎错误出现在我们的项目中。我们鼓励每个人尽可能报告更多的错误。这就是我们实现高质量的方式。

    这就是为什么这个阶段被称为修复阶段。我们正在报告错误并修复它们。成百上千的错误。有时甚至成千上万。产品就在我们眼前不断发展，因为每次错误修复后，我们都会重新部署整个产品到生产平台上。

    每个bug都会在其自己的GitHub工单和Git分支中进行报告、分类、讨论和修复。我们从不允许任何人直接提交到master分支——所有的更改都必须通过我们的质量控制，并由我们的合并机器人rultor.com合并到master分支。

    还值得一提的是，与产品负责人和程序员之间的所有沟通都只通过GitHub问题进行。我们从不使用任何聊天、Skype、电子邮件或会议软件。我们只通过GitHub的工单和评论进行沟通。

    Using

    这是最后阶段，可能需要相当长的时间。到目前为止，产品已经准备就绪，并已发布到市场上。但我们仍然会接收到产品所有者的错误报告和功能请求，我们仍然会通过与修复阶段相同的流程来解决它们。

    我们尽量保持这个阶段尽可能安静，尽量减少报告和修复的错误数量。多亏了我们在之前的阶段进行了积极主动的错误发现和修复工作，通常在使用阶段我们很少遇到问题。

    大型功能请求呢？在这个阶段，我们通常会尝试将它们转化为新的项目，并单独开发，从头开始思考。

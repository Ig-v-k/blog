    Java Method Logging with AOP and Annotations

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    有时候，我想要通过slf4j和log4j记录每个方法的执行，看看它接收了什么参数，返回了什么以及每个执行所花费的时间。这是我如何使用AspectJ，jcabi-aspects和Java 6注解来实现的。

    public class Foo {
  @Loggable
  public int power(int x, int p) {
    return Math.pow(x, p);
  }
}

    这是我在log4j输出中看到的内容：

    [INFO] com.example.Foo #power(2, 10): 1024 in 12μs
[INFO] com.example.Foo #power(3, 3): 27 in 4μs

    不错，是吗？现在，让我们来看看它是如何工作的。

    Annotation with Runtime Retention

    注解是在Java 6中引入的一种技术。它是一种元编程工具，不会改变代码的工作方式，但会给某些元素（方法、类或变量）打上标记。换句话说，注解只是附加在代码上可以被看到和读取的标记。一些注解仅在编译时可见，编译后的.class文件中不存在。其他注解在编译后仍然可见，并可以在运行时访问。

    例如，@Override属于第一种类型（其保留类型为SOURCE），而来自JUnit的@Test属于第二种类型（保留类型为RUNTIME）。@Loggable——我在上面的脚本中使用的注解——是第二种类型的注解，来自jcabi-aspects。它在编译后的.class文件中保留了字节码。

    再次强调，重要的是要理解，即使方法power()被注解并编译，它目前还没有发送任何内容到slf4j。它只是包含一个标记，表示“请记录我的执行”。

    Aspect Oriented Programming (AOP)

    AOP 是一种有用的技术，它能够在源代码中添加可执行的代码块，而不需要显式地对其进行更改。在我们的示例中，我们不想在类内部记录方法的执行过程。相反，我们希望其他类拦截对方法 power() 的每次调用，测量其执行时间，并将此信息发送给 slf4j。

    我们希望该拦截器能够理解我们的 @Loggable 注解，并记录对该特定方法 power() 的每次调用。当然，将来如果我们在其他方法中放置相同的注解，应该使用同一个拦截器。

    这种情况完全符合 AOP 最初的目的，即避免在多个类中重新实现某些常见行为。

    日志记录是我们主要功能的一个附加功能，我们不想在代码中添加多个日志记录指令。相反，我们希望日志记录在幕后进行。

    在 AOP 方面，我们的解决方案可以解释为创建一个切面，在特定的连接点上交叉代码，并应用一个环绕建议来实现所需的功能。

    AspectJ

    让我们看看这些神奇的词是什么意思。但是，首先，让我们看看jcabi-aspects如何使用AspectJ来实现它们（这是一个简化的示例，完整的代码可以在MethodLogger.java中找到）。

    @Aspect
public class MethodLogger {
  @Around("execution(* *(..)) && @annotation(Loggable)")
  public Object around(ProceedingJoinPoint point) {
    long start = System.currentTimeMillis();
    Object result = point.proceed();
    Logger.info(
      "#%s(%s): %s in %[msec]s",
      MethodSignature.class.cast(point.getSignature()).getMethod().getName(),
      point.getArgs(),
      result,
      System.currentTimeMillis() - start
    );
    return result;
  }
}

    这是一个内部有一个around()的方面。该方面使用@Aspect进行注解，而advice使用@Around进行注解。正如上面所讨论的，这些注解只是.class文件中的标记，除了向那些对运行时有兴趣的人提供一些元信息之外，它们并不起作用。

    注解@Around有一个参数，在本例中表示如果满足以下条件，则应将advice应用于一个方法：

    1. 它的可见性修饰符是 * （public、protected 或 private）。

    2. 它的名字是*（任意名称）;

    3. 它的参数是 ..（任何参数）; and

    4. 它被注释为 @Loggable。

    当要拦截一个带注解的方法的调用时，方法around()在执行实际方法之前执行。当要拦截对方法power()的调用时，方法around()会接收一个类ProceedingJoinPoint的实例，并且必须返回一个对象，该对象将作为方法power()的结果使用。

    为了调用原始方法power()，通知必须调用连接点对象的proceed()方法。

    我们编译这个切面，并将其与我们的主文件Foo.class一起放在类路径中。到目前为止，一切顺利，但我们还需要进行最后一步，以便将我们的切面放入行动中—我们应该应用我们的通知。

    Binary Aspect Weaving

    方面编织是应用建议的过程名称。方面编织器通过向方面注入调用来修改原始代码。AspectJ正是这样做的。我们给它两个二进制Java类Foo.class和MethodLogger.class；它返回三个—修改后的Foo.class，Foo$AjcClosure1.class和未修改的MethodLogger.class。

    为了理解哪些建议应该应用于哪些方法，AspectJ编织器使用.class文件中的注释。此外，它使用反射来浏览类路径上的所有类。它分析满足@Around注释条件的方法。当然，它找到了我们的power()方法。

    所以，有两个步骤。首先，我们使用javac编译我们的.java文件并获得两个文件。然后，AspectJ编织/修改它们并创建自己的额外类。我们的Foo类在编织后看起来像这样：

    public class Foo {
  private final MethodLogger logger;
  @Loggable
  public int power(int x, int p) {
    // Prepares the "point" object
    return this.logger.around(point);
  }
  private int power_aroundBody(int x, int p) {
    return Math.pow(x, p);
  }
}

    AspectJ编织器将我们原本的功能移动到一个新的方法power_aroundBody()，并将所有power()调用重定向到切面类MethodLogger。

    现在，在Foo类中不再只有一个方法power()，而是有四个类一起工作。从现在开始，每次调用power()时，背后都会发生以下情况：

    正如您所看到的，切面编织过程将类和切面连接在一起，通过连接点在它们之间传递调用。没有编织，类和切面都只是带有附加注释的已编译Java二进制文件。

    jcabi-aspects

    jcabi-aspects 是一个JAR库，其中包含Loggable注解和MethodLogger切面（顺便说一句，还有很多其他切面和注解）。您无需编写自己的方法日志切面。只需将一些依赖项添加到类路径中，并为切面织入配置jcabi-maven-plugin（在Maven Central中获取它们的最新版本）。

    <project>
  <dependencies>
    <dependency>
      <groupId>com.jcabi</groupId>
      <artifactId>jcabi-aspects</artifactId>
    </dependency>
    <dependency>
      <groupId>org.aspectj</groupId>
      <artifactId>aspectjrt</artifactId>
    </dependency>
  </dependencies>
  <build>
    <plugins>
      <plugin>
        <groupId>com.jcabi</groupId>
        <artifactId>jcabi-maven-plugin</artifactId>
        <executions>
          <execution>
            <goals>
              <goal>ajc</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
    </plugins>
  </build>
</project>

    由于这种编织过程需要大量的配置工作，我创建了一个方便的Maven插件，其中包含一个ajc目标，可以完成整个方面编织工作。您可以直接使用AspectJ，但我建议您使用jcabi-maven-plugin。

    XML+XSLT in a Browser

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    分离数据和其呈现方式 是一个很好的概念。以HTML和CSS为例。HTML应该只包含纯粹的数据，而CSS应该用于格式化这些数据，使其对人类可读。多年前，这可能是HTML/CSS的初衷，但实际上并不是这样。主要是因为CSS的功能不够强大。

    我们仍然需要使用HTML标签来格式化我们的数据，而CSS可以在定位和装饰方面提供一些帮助。

    另一方面，具有XSLT的XML完美地实现了将数据与呈现分离的理念。XML文档，像HTML一样，应该只包含数据，不包含任何关于定位或格式化的信息。XSL样式表定位和装饰数据。XSL是一种更强大的语言。这就是为什么可以在XML中避免任何格式化的原因。

    最新版本的Chrome、Safari、Firefox和IE都支持此机制。当浏览器从服务器检索XML文档，并且该文档带有与之关联的XSL样式表时，浏览器会即时将XML转换为HTML。

    Working Example

    让我们回顾一下这种工作方式的简单Java Web应用程序。它使用Takes Framework使这种机制成为可能。在下一篇文章中，我将解释ReXSL的工作原理。不过，现在让我们专注于使用XML传递原始数据并使用XSL样式表进行格式化的概念。

    打开http://www.stateful.co——这是一个由Atomic Counters at Stateful.co文章解释的有状态的Web原语集合。

    在Chrome或Safari中打开它。当你这样做时，你应该看到一个正常的网页，上面有一个标志、一些文本、一些链接、页脚等等。现在检查它的源代码（我假设你知道如何做这个）。

    这大致是你将会看到的（我假设你懂 XML，如果不懂，请立即开始学习）：

    <?xml-stylesheet type='text/xsl' href='/xsl/index.xsl'?>
<page date="2014-06-15T15:30:49.521Z" ip="10.168.29.135">
  <menu>home</menu>
  <documentation>.. some text here ..</documentation>
  <version>
    <name>1.4</name>
    <revision>5c7b5af</revision>
    <date>2014-05-29 07:58</date>
  </version>
  <links>
    <link href="..." rel="rexsl:google" type="text/xml"/>
    <link href="..." rel="rexsl:github" type="text/xml"/>
    <link href="..." rel="rexsl:facebook" type="text/xml"/>
  </links>
  <millis>70</millis>
</page>

    正如您所见，它是一个具有属性、元素和数据的合法XML文档。它完全没有关于如何向最终用户展示其元素的信息。实际上，这个文档更适合机器解析，而不是由人类阅读。

    该文档包含对请求者而言非常重要的数据。如何呈现这些数据或者是否呈现数据完全取决于请求者。

    它的第二行关联了文档与由浏览器单独加载的XSL样式表/xsl/index.xsl。

    <?xml-stylesheet type='text/xsl' href='/xsl/index.xsl'?>

    在Chrome中打开开发者工具，您会发现在页面加载完成后，浏览器会加载XSL样式表，然后加载所有其他资源，包括几个CSS样式表、jQuery和一个SVG徽标。

    index.xsl includes layout.xsl, that’s why it is loaded right after.

    让我们来考虑一个index.xsl的例子（实际上它要复杂得多，请查看layout.xsl）。例如：

    <xsl:stylesheet version="2.0"
  xmlns:xsl="http://www.w3.org/1999/XSL/Transform"
  xmlns="http://www.w3.org/1999/xhtml">
  <xsl:template match="page">
    <html>
      <body>
        <p>
          Current version of the application is
          <xsl:value-of select="version/name"/>
        </p>
      </body>
    </html>
  </xsl:template>
</xsl:stylesheet>

    我认为将此XSL样式表应用于我们的XML文档后，HTML页面的外观将会很明显。

    对我来说，这个XSL看起来干净简单易懂。然而，我经常听到人们说XSLT是一门难以理解的编程语言。我并不觉得它很难理解。当然，我没有使用它的所有功能。但是，对于简单的页面渲染，我只需要了解一些简单的命令和XML转换的原理即可。

    Why Not a Templating Engine?

    现在，为什么这种方法比广泛使用的Java模板引擎更好，包括JSP，JSF，Velocity，FreeMarker，Tiles等等？

    嗯，我看到了很多原因。但是，最重要的原因是：

    1. Web UI和API是同一个页面。不需要为RESTful API开发单独的页面——被计算机访问的Web用户界面就是一个API。根据我的经验，这样做可以大大避免代码重复的问题。

    2. XSL本身可以进行测试，无需服务器。为了测试我们的网站在某些数据下的显示效果，我们只需创建一个带有必要测试数据的新XML文档，将其与XSL关联，并在浏览器中打开。我们还可以修改XML并在浏览器中刷新页面。这使得HTML/CSS设计师的工作更加容易，独立于程序员。

    3. XSL是一种强大的函数式语言。与其他大部分看起来像是权宜之计的模板引擎相比，XSL是一个完整且设计良好的环境。一旦你熟悉了其语法和编程概念，编写XSL本身就是一种乐趣。你不是将指令注入到HTML文档中（像在JSP和其他语言中那样），而是编程将数据转化为展示——这是一种不同的思维方式，给人更好的感受。

    4. XML输出是完全可测试的。MVC中的一个控制器生成一个包含所有XSL样式表所需数据的XML文档，可以通过简单的XPath表达式轻松地在一个单元测试中进行测试。注入数据到模板引擎的控制器的测试是一个更复杂的操作，有时甚至是不可能的。我还在使用PHP和Ruby进行编写。它们都面临着完全相同的问题，尽管它们的模板引擎由于语言的解释性质更加强大。

    Is It Fully Supported?

    如果所有浏览器都支持XML+XSL渲染，那将是非常好的。然而，事实并非如此。只有最新版本的现代浏览器支持XSL。请查看由Julian Reschke完成的此比较。除此之外，XSLT 2.0根本就没有支持。

    然而，还有一种解决方法。我们可以通过浏览器发出请求时的User-Agent HTTP头来了解是哪个浏览器，然后在服务器端将XML转换为HTML。因此，对于支持XSL的现代浏览器，我们将提供XML，而对于其他所有浏览器，则提供HTML。

    这正是ReXSL框架的工作方式。在Internet Explorer中打开http://www.stateful.co你将看到一个HTML文档，而不是像Chrome一样是一个XML文档。

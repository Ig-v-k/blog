    Strict Control of Java Code Quality

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    有许多工具可以控制Java代码的质量，包括Checkstyle, PMD, FindBugs, Cobertura等等。它们通常用于分析质量并生成一些漂亮的报告。这些报告通常由持续集成服务器（如Jenkins）发布。

    Qulice更进一步。它聚合了几个质量检查器，将它们配置为最严格的模式，并在任何一个检查器失败时中断构建过程。

    认真的。Checkstyle 中有超过 130 个检查项，PMD 中有超过 400 条规则，FindBugs 中有超过 400 个错误。它们全部都应该说：“是的，我们喜欢你的代码。”否则，你的构建不应该通过。

    你觉得呢？每次代码有一个小问题就被拒绝，这样对你来说方便吗？对团队来说，强制开发人员如此关注代码质量，这样是否有益于生产力？

    First Reaction

    如果您加入我们的团队作为Java开发人员，您将在分支中开发您的代码，然后Rultor将把您的更改合并到master分支中。不过，在实际合并之前，Rultor将运行自动构建脚本，以确保您的分支不会破坏它。

    作为一种静态分析工具，Qulice只是自动化构建脚本中的一个步骤。实际上，它是一个Maven插件，我们使用Maven 3x来自动化Java构建。因此，如果您的更改违反了Qulice的任何规则，您的整个分支都会被拒绝。

    On Second Thought

    我的经验告诉我，一个人越早适应Qulice的严格质量控制，他/她就能越快学习和成长；他/她就是越好的程序员；他/她就能与我们和我们的项目一起走得更远。

    What Do Projects Get From Qulice?

    让我们以一个简单的规则为例。这是一段Java代码，Qulice会对其进行投诉（因为Checkstyle的DesignForExtension规则）：

    public class Employee {
  public String name() {
    return "Jeff";
  }
}

    这段代码有什么问题？name() 方法不是最终方法，可以被扩展 Employee 类的子类重写。从设计角度来看，这是错误的，因为子类允许破坏超类，覆盖其方法。

    什么是正确的设计？就是这个：

    public class Employee {
  public final String name() {
    return "Jeff";
  }
}

    现在，该方法是最终的，不能被子类覆盖。这是一种更安全的设计（根据Checkstyle的说法，我也同意）。

    所以，假设我们将此规则应用于项目中的所有类。这对项目有什么好处呢？它可以向其成员（程序员）承诺更高质量的工作，与其他没有此限制的项目相比，主要是因为：

    • 设计的可预测性—我不需要滚动整个类来确保它没有可能被意外覆盖的方法。我可以确定在这个项目中不会发生这种情况。换句话说，我知道可以期待什么。

    • 更少的隐藏技巧—设计的可预测性更高，使错误和技巧更容易被发现。源代码的标准化使其统一。这意味着阅读和发现问题更容易。

    • 行业标准—决定使用这种设计的是Checkstyle，而不是项目架构师。对于我作为项目开发者来说，这意味着我正在遵循行业标准。这使得项目（及其领导者）更值得尊敬。

    • 学习—我敢打赌，阅读这篇文章的大多数人都不知道上面解释的设计规则。仅仅通过阅读这篇文章，你已经学到了新的东西。想象一下，当你的代码符合Qulice的所有900条规则（Checkstyle + PMD + FindBugs）后，你可以学到多少东西。

    关于学习的观点将我们带到了最后，也是最重要的一个思考。

    What Do I Get from Qulice?

    作为一个程序员，我希望你已经意识到在一个把质量标准提高到Qulice所要求的项目中工作会带给你什么。是的，你会学到很多关于编写高质量Java代码的新知识。

    除此之外，我实际上会说，每写一行新代码，你都在获得免费的课程。而这位老师是一款软件，由数百名Java开发人员编写，已经存在了十年之久。Qulice只是将这些软件工具整合在一起。说实话，真正的质量检查和规则的作者是那些开发人员。

    那么，我该如何告诉那些抱怨质量规则过于严格的人呢？我会说这样的话：“你想要学习和进步，还是只想拿钱就算了呢？”

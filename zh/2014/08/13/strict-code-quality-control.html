    Strict Control of Java Code Quality

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    有许多控制Java代码质量的工具，包括Checkstyle、PMD、FindBugs、Cobertura等等。它们通常用于分析质量并生成一些漂亮的报告。很多时候，这些报告会被持续集成服务器（如Jenkins）发布。

    Qulice采取了更进一步的措施。它聚合了几个质量检查工具，将它们配置为最严格的模式，并在任何一个工具失败时中断您的构建过程。

    严肃地说，Checkstyle 中有超过130个检查规则，PMD 中有超过400个规则，FindBugs 中有超过400个错误。所有这些规则和错误都应该表示：“是的，我们喜欢你的代码。”否则，你的构建将无法通过。

    你觉得呢？每次代码中只有一个不符合900个检查之一，都被拒绝，这样方便吗？对团队的效率有帮助吗？强制开发人员如此关注代码质量是否有生产性？

    What Do Projects Get From Qulice?

    让我们以一个简单的规则为例。这是一段Java代码，Qulice会对其进行投诉（由于Checkstyle的DesignForExtension规则）。

    public class Employee {
  public String name() {
    return "Jeff";
  }
}

    这段代码有什么问题？name() 方法不是最终方法，可以被继承 Employee 的类重写。从设计角度来看，这是错误的，因为子类可以打破父类，覆盖其方法。

    什么是正确的设计？就是这个：

    public class Employee {
  public final String name() {
    return "Jeff";
  }
}

    现在，这个方法是最终的，不可以被子类重写。这是一个更安全的设计（根据Checkstyle和我的观点）。

    所以，假设我们将此规则应用于项目中的所有类。这给项目带来了什么好处？与没有此限制的其他项目相比，它可以向成员（程序员）保证更高质量的工作，主要原因有：

    • 设计的可预测性—我不需要滚动整个类来确保它没有可以意外被覆盖的方法。我确信这在这个项目中不会发生。换句话说，我知道可以期望什么。

    • 少一些隐藏的技巧—设计的可预测性更高，更容易发现错误和技巧。源代码的标准化使之统一。这意味着更容易阅读和发现问题。

    • 行业标准—使用该设计的决定由Checkstyle决定，而不是项目架构师。对我作为项目开发人员来说，这意味着我正在遵循行业标准。这使得项目（以及其领导者）更受尊重。

    • 学习——我敢打赌，阅读本篇文章的大部分人都不知道上述解释的设计规则。仅仅通过阅读这篇文章，你已经学到了新的知识。想象一下，当你的代码符合Qulice（Checkstyle + PMD + FindBugs）的900个规则时，你将学到多少知识。

    关于学习的观点引导我们来到最后，也是最重要的讨论内容。

    What Do I Get from Qulice?

    作为一个程序员，我希望你已经意识到在一个像Qulice要求的项目中工作会带给你什么。是的，你会学到很多关于编写高质量Java代码的新知识。

    除此之外，我要说的是，每当你编写一行新代码时，你实际上都在获得免费的课程。而这位老师是一款软件，由数百名Java开发者编写了十年之久。Qulice只是将这些软件工具整合在一起。实际上，真正负责质量检查和规则的是开发者们。

    那么，我对那些抱怨质量规则过于严格的人该怎么说呢？我会这样说：“你是想学习和进步，还是只想赚钱然后逃避责任呢？”

    Hits-of-Code Instead of SLoC

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    代码行数（也称为SLoC）是一个声誉糟糕的度量标准。试着谷歌一下你自己，你会发现大量的文章都在抨击它对软件开发过程的适用性和破坏性。主要的论点是我们不能通过编写的代码行数来衡量编程的进展。最著名的引述可能归功于比尔·盖茨。

    用代码行数来衡量编程进展，就像用重量来衡量飞机建造进展一样。

    基本上，这意味着飞机的某些部分在同样的时间内要比其他部分更加轻便，但需要更多的努力（例如像中央计算机这样的部件）。我们不应该测量飞机的重量，而是应该测量投入其中的努力…不管怎样。所以，这就是想法。我们可以测量程序员触碰代码行的次数。不是计算代码行的数量，而是计算它们实际修改的次数—我们可以从Git（或任何其他版本控制系统）获取这些信息。你触碰飞机的那个部分越多—你在上面花费的努力就越多，对吧？

    我称之为代码行数 (HoC)，并创建了一个小工具来帮助我们仅用一行计算出这个数字。它是一个Ruby gem，安装它并运行：

    $ gem install hoc
$ hoc
54687

    54687这个数字是你代码库中的代码行数总和。这个数字背后的原则很简单——每当在Git提交中修改、创建或删除一行代码时，计数器就会递增。

    这个度量指标比 LoC 更好的主要原因是它更符合实际投入到代码库中的工作量。以下是原因。

    It Always Increments

    HoC指标始终上升。今天它不可能比昨天低，就像努力一样，它总是递增的。代码行数并非如此。今天你可能有一个庞大的代码库，但在重构之后，它会变得更小。代码行数减少了。这是否意味着你的效率降低了？绝对不是，但对于非程序员来说，代码行数指标就是这样的。例如，项目经理可能会认为，由于代码库的大小在过去一个月内保持不变，团队没有在工作。

    HoC不具有这种违反直觉的效果。相反，HoC会随着您的每次提交而增长。您在代码库上工作的时间越长，HoC就会越大。您的产品的绝对大小无关紧要，重要的是您投入了多少努力。这就是为什么HoC非常直观，并且可以用作软件开发进度的衡量标准。

    看一下这个18个月的图表；它同时显示了两个指标。我使用了与rultor相同的Java代码库，这是一个DevOps助手。正如你在图表中所看到的，代码库在几个月前经历了一次重大重构。我认为很明显，这个图表上的哪个指标更能告诉我们有关产品投入的努力。

    It Is Objective

    对于 House of Code（HoC）来说，代码库的绝对规模大小并不重要，而只有你相对于它的贡献有多大。

    假设你有30万行代码，其中95%是从第三方库中复制粘贴过来的（顺便说一句，这是一种非常常见且糟糕的做法——将第三方代码放在自己的代码库中）。代码行数会很多，但实际的自定义代码部分相对较少。因此，代码行数指标会误导人——它总是显示30万，只是在此基础上略有增加或减少。每个人都会觉得团队正在处理一个30万行的代码库。

    另一方面，HoC 将始终考虑实际被修改的代码部分。HoC 的价值将与实际与代码库一起工作的程序员的努力客观相关。

    It Exposes Complexity of Lines

    LoC通常因其对代码复杂性的中立性而受到批评。自动生成的ORM类或复杂的排序算法在代码行数上可能相同，但是前者只需几秒钟编写，而后者可能需要数周甚至数月。这就是为什么代码行数通常被认为是一个不准确的指标。

    Hits-of-Code 考虑了复杂性，因为你使用那个排序算法的时间越长，对其代码行的修改也就越多。嗯，如果你经常使用 Git，并频繁提交你的更改，那么这个说法就是正确的——这是告诉 Git 你的工作进展的方式。

    Conclusion

    最后，看看我们团队在过去几年里完成的这个开放项目列表。每个项目都有两个指标：代码行数和代码点击数。有趣的是，相对较小的项目却有非常大的（超过一百万）点击数。这立刻让我想起我们在其中投入了多少时间以及它们的年龄。

    我在这个分析中使用了HoC指标：每行代码支付多少钱？那篇文章比较了一个每个HoC支付$3.98的传统项目和一个由Zerocracy管理的开源项目，它支付了¢13。

    我的结论是，这个代码行数指标可以作为软件开发项目中的进度跟踪工具。此外，它还可以用于团队规模、项目预算、开发进度等方面的估计。显然，代码行数指标不能是唯一的指标，但与其他指标结合使用可以在估算、规划和跟踪方面提供很大帮助。

    Getters/Setters. Evil. Period.

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    有一个长期的争论，始于2003年Allen Holub在他的著名文章《为什么getter和setter方法是邪恶的》中，讨论的是getter/setter是否是一种反模式，应该避免使用，还是它们在面向对象编程中是我们不可避免的需求。我将尽量在这个讨论中加入我的观点。

    以下文本的要点是：getter和setter是一种糟糕的做法，使用这种做法的人不能被原谅。再次强调，我并不是说在可能的情况下应该避免使用get/set。不。我是说你的代码中永远不应该有它们。

    足够傲慢以吸引你的注意力？你已经使用那个get/set模式15年了，而且你是一位受人尊敬的Java架构师？你不想听一个陌生人胡说八道？好吧，我理解你的感受。当我偶然发现David West的《对象思考》（http://amzn.to/266oJr4）时，我几乎有同样的感觉，这本书是我迄今为止读过的关于面向对象编程最好的书。所以，请冷静下来并尽量理解，我会尽力解释。

    Existing Arguments

    在面向对象的世界中，有一些关于”访问器”（另一个称呼为getter）的争论。我认为所有这些争论都不够有力。让我们简要地回顾一下它们。

    讲述，而不是询问 Allen Holub说道：“不要询问你需要完成工作所需的信息；而是询问包含有该信息的对象来为你完成工作。”

    违反了封装原则 一个对象可以被其他对象撕裂开来，因为它们能够通过设置器将任何新数据注入其中。该对象无法足够安全地封装自己的状态，因为任何人都可以修改它。

    暴露的实现细节 如果我们可以从另一个对象中获取一个对象，那么我们对于第一个对象的实现细节依赖过多。如果明天它改变了，比如结果的类型，我们也必须修改我们的代码。

    所有这些理由都是合理的，但它们忽略了主要问题。

    Fundamental Disbelief

    大多数程序员认为对象是带有方法的数据结构。我引用了Bozhidar Bozhanov的文章《Getters and Setters Are Not Evil》（链接：http://java.dzone.com/articles/getters-and-setters-are-not）。

    但是大多数人为其生成getter和setter的对象都是简单的数据持有者。

    这种误解是一个巨大的误解的后果！对象不是“简单的数据持有者”。对象不是带有附加方法的数据结构。这个“数据持有者”的概念来自过程化语言，特别是C和COBOL。我再说一遍：对象不是一组数据元素和操纵它们的函数。对象不是数据实体。

    What is it then?

    A Ball and A Dog

    在真正的面向对象编程中，对象就像你和我一样是有生命的存在。它们是有生命的生物，拥有自己的行为、属性和生命周期。

    一个活体有可能有一个setter吗？你能把一个球“设定”给一只狗吗？实际上不行。但下面这款软件却正是在做这件事情：

    Dog dog = new Dog();
dog.setBall(new Ball());

    这听起来怎么样？

    你能从狗身上取出一个球吗？嗯，如果她吃下了它并且你要进行手术的话，你可能可以。在那种情况下，是的，我们可以从狗身上”取出”一个球。这就是我所说的。

    Dog dog = new Dog();
Ball ball = dog.getBall();

    又或者更荒谬的例子：

    Dog dog = new Dog();
dog.setWeight("23kg");

    你能想象这个交易在现实世界中吗？

    它看起来是否与你每天所写的内容相似？如果是，那么你就是一个过程式程序员。承认吧。关于这一点，大卫·韦斯特在他的书的第30页上有这样说：

    成功的过程式开发人员转变为成功的面向对象开发人员的第一步是进行脑切除手术。

    你需要动脑手术吗？嗯，我肯定需要，并在阅读韦斯特的《对象思维》时接受了这样的手术。

    Object Thinking

    开始像一个对象一样思考，你会立即重命名那些方法。这可能是你会得到的：

    Dog dog = new Dog();
dog.take(new Ball());
Ball ball = dog.give();

    现在，我们把这只狗当作一只真实的动物来对待，它可以从我们这里接过球，并且在我们要求的时候还给我们。值得一提的是，狗不能还回NULL。狗根本不知道NULL是什么。对象思维立即从你的代码中消除了NULL引用。

    除此之外，对象思维将导致对象的不变性，就像在”狗的重量”的例子中一样。您应该这样重写它：

    Dog dog = new Dog("23kg");
int weight = dog.weight();

    狗是一个不可变的生物，不允许任何外部人改变她的体重、大小、名字等等。她可以根据要求告诉她的体重或名字。公有方法展示对象的某些“内部”请求没有问题。但这些方法不是“getters”，并且不应该以“get”为前缀。我们并不是从狗“获取”任何东西。我们不是获取她的名字。我们是在询问她告诉我们她的名字。看到区别了吗？

    我们这里并不是在谈论语义学。我们是在区分过程式编程思维和面向对象编程思维。在过程式编程中，我们处理数据，操作它们，在必要时获取、设置和删除。我们掌控一切，数据只是被动的组成部分。对我们来说，狗不是什么重要的东西——它只是一个“数据持有者”。它没有自己的生活。我们可以自由地从中获取任何必要的信息并将任何数据设置进去。这就是C、COBOL、Pascal和许多其他过程式语言的工作方式。

    相反，在一个真正的面向对象的世界中，我们将对象视为活生生的有生命的有自己的出生日期和死亡时刻的有自己的身份和习惯的生物。我们可以要求一只狗给我们一些数据（例如她的体重），她可能会把这些信息返回给我们。但我们总是记住这只狗是一个主动的组件。她决定我们请求后会发生什么。

    这就是为什么，在对象中以 set 或 get 开头的任何方法在概念上都是不正确的。这与破坏封装无关，正如许多人所争论的那样。而是关于你是否像一个对象思考，或者你是否仍然在用Java语法写COBOL。

    PS。是的，您可能会问，JavaBeans、JPA、JAXB和许多其他依赖于get/set符号的Java API呢？Ruby的内建特性简化了访问器的创建，那又如何呢？嗯，这些都是我们的不幸。与真正理解和欣赏真实对象的美丽世界相比，留在原始的COBOL过程世界中要容易得多。

    PPS. 忘了说，是的，通过setter进行依赖注入也是一个糟糕的反模式。关于它，将在接下来的一篇文章中进行讨论。

    PPPS. 这是我建议使用的替代方案：printers.

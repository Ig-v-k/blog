    Built-in Fake Objects

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    虽然模拟对象是进行单元测试的完美工具，但通过模拟框架进行模拟可能会让你的单元测试变得难以维护。多亏了它们，我们经常听到”模拟是不好的”和”模拟是邪恶的”。

    这种复杂性的根本原因是我们的对象太大了。它们有许多方法，而这些方法会返回其他对象，这些对象也有方法。当我们将这样一个对象的模拟版本作为参数传递时，我们应该确保它的所有方法都返回有效的对象。

    这导致了不可避免的复杂性，使得单元测试几乎不可能维护。

    Object Hierarchy

    以 jcabi-dynamo 中的 Region 接口为例（本文中的所有代码片段都经过简化以便言简意赅）。

    public interface Region {
  Table table(String name);
}

    它的 table() 方法返回一个 Table 接口的实例，该接口有自己的方法：

    public interface Table {
  Frame frame();
  Item put(Attributes attrs);
  Region region();
}

    由 frame() 方法返回的接口 Frame 也有它自己的方法。以此类推。为了创建一个正确模拟的 Region 接口实例，通常需要创建十几个其他的模拟对象。使用 Mockito，代码如下所示：

    public void testMe() {
  // many more lines here...
  Frame frame = Mockito.mock(Frame.class);
  Mockito.doReturn(...).when(frame).iterator();
  Table table = Mockito.mock(Table.class);
  Mockito.doReturn(frame).when(table).frame();
  Region region = Mockito.mock(Region.class);
  Mockito.doReturn(table).when(region).table(Mockito.anyString());
}

    而所有这些都只是在实际测试之前的脚手架。

    Sample Use Case

    假设你正在开发一个使用jcabi-dynamo来管理DynamoDB中数据的项目。你的类可能类似于这样：

    public class Employee {
  private final String name;
  private final Region region;
  public Employee(String empl, Region dynamo) {
    this.name = empl;
    this.region = dynamo;
  }
  public Integer salary() {
    return Integer.parseInt(
      this.region
        .table("employees")
        .frame()
        .where("name", this.name)
        .iterator()
        .next()
        .get("salary")
        .getN()
    );
  }
}

    你可以想象使用Mockito来对这个类进行单元测试会有多么困难。首先，我们需要模拟Region接口。然后，我们需要模拟一个Table接口，并确保它被table()方法返回。然后，我们需要模拟一个Frame接口，等等。

    单元测试将比课程本身要长得多。除此之外，它的真正目的是测试对员工工资的检索，对读者来说并不明显。

    此外，当我们需要测试一个类似类的类似方法时，我们将需要从头开始重新模拟。再次，这将需要编写多行代码，这些代码看起来与我们已经编写的代码非常相似。

    Fake Classes

    解决方案是创建假类，并将它们与真实类一起发布。这就是jcabi-dynamo的做法。只需查看其JavaDoc。其中有一个名为com.jcabi.dynamo.mock的包，只包含适用于单元测试的假类。

    尽管它们的唯一目的是优化单元测试，但我们将其与生产代码一同打包在同一个JAR包中进行发布。

    当使用一个虚假的类MkRegion时，测试将会是这样的。

    public class EmployeeTest {
  public void canFetchSalaryFromDynamoDb() {
    Region region = new MkRegion(
      new H2Data().with(
        "employees", new String[] {"name"},
        new String[] {"salary"}
      )
    );
    region.table("employees").put(
      new Attributes()
        .with("name", "Jeff")
        .with("salary", new AttributeValue().withN(50000))
    );
    Employee emp = new Employee("Jeff", region);
    assertThat(emp.salary(), equalTo(50000));
  }
}

    这个测试对我来说很明显。首先，我们创建了一个虚假的DynamoDB地区，它在H2Data存储（内存中的H2数据库）之上运行。这个存储将准备好一个包含哈希键name和一个salary属性的单个employees表。

    然后，我们将一条记录放入表中，其中包含一个哈希值为Jeff和一个工资为50000。

    最后，我们创建了一个Employee类的实例，并检查它如何从DynamoDB中获取工资。

    我目前在几乎每个我正在使用的开源库中都在做同样的事情。我正在创建一组虚假类，以简化库内部和用户的测试。

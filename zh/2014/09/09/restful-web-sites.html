    RESTful API and a Web Site in the Same URL

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    看看GitHub的RESTful API，例如。要获取有关存储库的信息，您应该向api.github.com/repos/yegor256/rultor发出GET请求。作为响应，您将获得一个JSON文档，其中包含有关yegor256/rultor存储库的所有详细信息。试试看，这个URL不需要任何身份验证。

    要在一个漂亮的HTML+CSS页面中打开相同的存储库，您应该使用不同的URL：github.com/yegor256/rultor。URL不同，服务器端肯定不同，但数据的性质完全相同。唯一变化的是表示层。

    在第一个案例中，我们得到JSON；在第二个案例中，得到HTML。

    将它们结合起来怎么样？将它们使用相同的 URL 和相同的服务器端处理机制怎么样？将整个渲染任务转移到客户端（浏览器）上，让服务器仅处理数据怎么样？

    XSLT是可以帮助我们实现这一目标的技术。在《在浏览器中使用XML+XSLT》一文中，我简要解释了它在浏览器中的工作原理。简而言之，服务器返回一个带有一些数据和指向XSL样式表的链接的XML。在浏览器中执行的样式表将XML转换为HTML。XSL语言和其他渲染引擎（如JSP、JSF、Tiles等）一样强大，事实上，它更加强大。

    使用这种方法，我们从服务器上直接移除整个渲染层（MVC模式中的“View”），并将其转移到浏览器中。

    如果我们能够实现，Web服务器将仅公开一个RESTful API，并且每个响应页面将附带一个XSL样式表。我们将获得什么？我们将在帖子的结尾讨论。现在，让我们看看我们将面临哪些问题：

    1. JSON没有渲染层。JSON没有像XSLT这样的东西。因此，我们将不得不忘记JSON，只使用XML。对我来说，这听起来完全没问题。其他人不喜欢XML，更喜欢只使用JSON。我从来不理解他们。

    2. XSLT 2.0 不被所有浏览器支持。即使 XSLT 1.0 也只被部分浏览器支持。例如，Internet Explorer 8 根本不支持 XSLT。

    3. 浏览器仅支持GET和POST HTTP方法，而传统的RESTful API还使用了至少PUT和DELETE方法。

    第一个问题实际上并不是问题。这只是一个品味问题（以及教育水平问题）。而最后两个问题则更为严重。让我们来讨论一下它们。

    XSL Transformation on the Server

    XSLT在一些浏览器中不受支持。我们如何解决这个问题？

    我认为最好的方法是在每个请求中解析User-Agent HTTP头，并猜测该浏览器的特定版本是否支持XSLT。这并不难做，因为这些兼容性信息是公开的。

    如果浏览器不支持XSLT，我们可以在服务器端进行转换。我们已经拥有由服务器生成的包含数据的XML，以及已经附加到其上的XSL。我们只需要将后者应用于前者，得到一个HTML页面。然后，我们将HTML返回给浏览器。

    除此之外，我们还可以关注Accept头部。如果它被设置为application/xml或text/xml，无论User-Agent所说的是什么，我们都会返回XML。这基本上意味着有些API客户端与我们交互，而不是一个浏览器。而且这个客户端对HTML不感兴趣，只对纯粹的XML格式数据感兴趣。

    POST Instead of PUT

    对此没有变通方法。浏览器对于PUT或DELETE一无所知。所以，在我们的RESTful API中，我们也应该忘记它们。我们应该只使用两种方法来设计我们的API：GET和POST。这可行吗？是的。为什么不呢？虽然不如使用所有六种方法那样华丽（有些API也使用OPTIONS和HEAD），但它能够正常工作。

    What Do We Gain?

    好的，这里是问题——为什么我们需要这个？大多数人现在的工作方式有什么问题吗？我们为什么不能将网站与 API 分开？如果我们将它们结合起来，我们会获得什么好处？

    自2011年以来，我一直在所有的网络应用程序中将它们结合在一起使用。我目前体验到的最大优势是避免代码重复。

    很明显，在服务器中我们不会重复使用控制器（在MVC的情况下）。我们只有一层控制器，它们同时控制API和网站（因为它们现在是同一个东西）。

    避免代码重复是一个非常重要的成就。此外，我认为这是任何软件项目最重要的目标。

    Deployment Script vs. Rultor

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    当我解释Rultor如何自动化部署/发布流程时，经常听到类似的话：

    但我已经有一个脚本可以自动部署所有东西了。

    这种回应非常常见，所以我决定在一篇文章中总结我对自动化Rultor部署/发布流程的三个主要论点：1）隔离的Docker容器，2）日志的可见性和3）凭证的安全性。

    阅读有关它们的信息，并查看Rultor在您现有的部署脚本之上提供的功能。

    在我们开始讨论之前，让我强调一下，Rultor是一个对你的自定义脚本非常有用的接口。当你决定使用Rultor自动化部署时，你不需要丢弃任何现有的脚本。你只需要教会Rultor如何调用它们。

    Isolated Docker Containers

    你一旦开始从Rultor调用部署脚本，你将获得的第一个优势是Docker的使用。我相信你知道Docker是什么，但对于那些不知道的人来说，它是一个管理虚拟Linux“机器”的工具。它是一个命令行脚本，当你需要在新的虚拟机（也称为“容器”）中运行一些脚本时，你会调用它。Docker几乎立即启动容器并运行你的脚本。Docker的美妙之处在于每个容器都是一个完全隔离的Linux环境，拥有自己的文件系统、内存、进程等。

    当您告诉Rultor运行部署脚本时，它会启动一个新的Docker容器并在其中运行您的脚本。但是您可能会问，这给我带来了什么好处呢？

    主要的好处是在你的脚本执行完毕后容器会被销毁。这意味着你可以在容器内进行所有的预配置，而不用担心与主要工作平台发生冲突。让我举个例子。

    我正在MacBook上开发，我在上面安装和删除我需要用于开发的软件包。同时，我有一个项目，为了部署它，需要PHP 5.3，MySQL 5.6，Phing，PHPUnit，PHPCS和xdebug。每个MacOS版本都需要特定的配置才能使这些应用程序运行起来，这是一项耗时的工作。

    我可以更换笔记本电脑，也可以更换 MacOS 版本，但项目保持不变。它仍然需要相同的一套软件包才能成功运行其部署脚本。而且该项目已经不再进行主动开发。我现在主要使用 Java 工作，因此日常工作中并不需要这些软件包。但是，当我需要对那个 PHP 项目进行小修复并部署时，我必须安装所有所需的 PHP 软件包并进行配置。只有在此之后，我才能部署这个小修复。

    至少可以说它很烦人。

    Docker给予我将所有这些自动化在一起的能力。我的现有部署脚本将获得一个前导部分，它将在一个干净的Ubuntu容器中安装和配置所有必需的与PHP相关的软件包。这个前导部分将在每次运行我的部署脚本时，在一个Docker容器内部执行。例如，它可能看起来像这样：

    在开始使用 Rultor 之前，我的部署脚本是这样的：

    
# !/bin/bash

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

只有两行。第一行是运行所有单元测试。第二行是将文件传输到生产服务器的FTP部署。非常简单。但是，这个脚本只能在安装了PHP 5.3、MySQL、Phing、xdebug、PHPCS和PHPUnit的情况下才能正常工作。再次强调，每次升级我的MacOS或更换笔记本电脑时，安装和配置它们都是一项繁重的工作。

毋庸置疑，如果/当有人加入项目并尝试运行我的脚本，他/她将不得不再次进行这个预安装工作。

所以，这是一个新的脚本，我现在正在使用它。每次都在一个新的Docker容器中执行：

```bash

# !/bin/bash

# First, we install all prerequisites

sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

请运行以下命令进行安装：sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

# And now the same script I had before

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

显然，在我的 MacBook 上运行这个脚本（不使用虚拟化）会引发很多麻烦。嗯，我这里甚至都没有 `apt-get`。

因此，Rultor给您带来的第一个好处是将您的部署脚本隔离在自己的虚拟环境中。我们主要得益于Docker。

## Visibility of Logs

传统上，我们将部署脚本保存在某个 `~/deploy` 目录中，并使用一组神奇的参数来运行它们。在一个小项目中，你自己来完成这个过程，而这个目录在你自己的笔记本电脑上。在一个更大的项目中，有一个“部署”服务器，它有一个带有一组脚本的神奇目录，只有少数可信任的高级开发人员能够执行这些脚本。我已经见过这种设置很多次了。

最大的问题在于追溯性。几乎不可能找出是谁部署了什么，以及为什么某个特定的部署失败了。高级部署专家只需SSH到服务器上，使用神奇的脚本和参数运行。日志通常会丢失，问题追踪非常困难，甚至不可能。

Rultor 提供了不同寻常的功能。使用 Rultor，不再需要通过 SSH 访问部署脚本。所有的脚本都保存在 `.rultor.yml` 配置文件中，您可以通过在问题跟踪系统（例如 GitHub、JIRA 或 Trac）中发布消息来启动它们。Rultor 运行脚本并将完整的日志发布到您的工单中。日志将永久保存在您的项目中。您可以随时回到您正在处理的工单，并检查部署失败的原因以及实际执行的指令是什么。

例如，查看此 GitHub 问题，在那里我正在部署 Rultor 的新版本，并且失败了几次：[yegor256/rultor#563](https://github.com/yegor256/rultor/issues/563)。我所有的失败尝试都有记录。我可以随时回顾并进行调查。对于一个大项目来说，这些信息至关重要。

因此，与独立部署脚本相比，Rultor的第二个好处是每个操作的可见性。

## Security of Credentials

当你有一个自定义脚本存储在你的笔记本电脑或者那个秘密的团队部署服务器上时，你的生产凭证就离得很近了。没有其他方法。如果你的软件与数据库交互，它必须知道登录凭证（用户名、密码、数据库名、端口号等）。嗯，在最糟糕的情况下，有些人会将这些信息直接硬编码到源代码中。我们甚至不打算讨论这种情况，因为它太糟糕了。

但是假設你將資料庫憑證與原始碼分開。你會有一個像 `db.properties` 或 `db.ini` 的檔案，它會在部署之前附加到應用程式上。你也可以將該檔案直接保存在生產伺服器上，這樣更好，但並非總是可能的，特別是在 PaaS 部署的情況下，例如。

在将工件部署到存储库时存在类似的问题。假设你经常部署到RubyGems.org。你的`~/.gem/credentials`文件中包含了你的秘密API密钥。

所以，很常见的情况是，你的部署脚本会有一些包含敏感和安全信息的文件。而这些文件中的信息是以明文、开放的格式存在的。没有加密，没有保护措施。只是以纯文本形式存储了用户名、密码、代码和令牌等信息。

为什么这样不好呢？对于一个只有一台笔记本电脑的开发者来说，这听起来并不是一个问题。尽管如此，我不喜欢在机场的某个地方丢失一台笔记本电脑，而其中的所有凭据都是开放的，并且随时可以被使用。你可能会争辩说，有一些磁盘保护工具，比如MacOS的FileVault或Windows的BestCrypt。是的，也许。

但是让我们看看当我们有一个团队的开发者一起工作并共享那些带有凭证的部署脚本和文件时会发生什么。一旦你将部署脚本的访问权限给团队的新成员，你就必须分享所有的敏感数据。这是无法回避的。为了使用这些脚本，他/她必须能够打开带有凭证的文件。

这是一个问题，如果您关心您的数据的安全。

Rultor通过在部署脚本使用敏感数据之前即时提供GPG解密来解决此问题。在`.rultor.yml`配置文件中，您只需这样说：

    decrypt: db.ini: “repo/db.ini.asc” deploy: script: ftp put db.ini production ```

    然后，您可以使用Rultor GPG密钥对db.ini进行加密，并无畏地将db.ini.asc提交到存储库中。除了Rultor服务器在运行部署脚本之前，没有人能够打开和阅读该文件。

    因此，Rultor相对于独立部署脚本的第三个优点是对敏感数据的适当安全性保护。

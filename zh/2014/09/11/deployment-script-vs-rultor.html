    Deployment Script vs. Rultor

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    当我解释Rultor是如何自动化部署/发布流程的时候，经常会听到类似的话：

    但我已经有一个自动部署一切的脚本了。

    这个回应非常常见，所以我决定在一篇文章中总结我的三个主要论点，即自动化的Rultor部署/发布流程：1）隔离的Docker容器，2）日志的可见性和3）凭据的安全性。

    阅读有关它们的内容，并查看 Rultor 在您现有的部署脚本之上提供的功能。

    在我们开始讨论之前，让我强调一下，Rultor是一个对你的自定义脚本非常有用的接口。当你决定使用Rultor自动化部署时，你并不会丢弃你已有的任何脚本。你只需要教会Rultor如何调用它们即可。

    Isolated Docker Containers

    开始从Rultor调用部署脚本的第一个优点是可以使用Docker。我相信你知道什么是Docker，但对于那些不知道的人来说，它是一个虚拟Linux“机器”的管理器。它是一个命令行脚本，当你需要在新的虚拟机（也称为“容器”）中运行一些脚本时，你可以调用它。Docker几乎立即启动容器并运行你的脚本。Docker的优点是每个容器都是一个完全隔离的Linux环境，拥有自己的文件系统、内存、进程等。

    当您告诉Rultor运行您的部署脚本时，它会启动一个新的Docker容器并在其中运行您的脚本。但是，您可能会问，这给我带来了什么好处呢？

    主要好处是，容器在脚本完成后立即被销毁。这意味着您可以在容器内进行所有的预配置，而不必担心与主要工作平台的冲突。让我举个例子。

    我在MacBook上开发，我会安装和卸载我需要的开发包。同时，我有一个项目需要部署，该项目需要PHP 5.3、MySQL 5.6、Phing、PHPUnit、PHPCS和xdebug的支持。每个MacOS版本都需要特定的配置才能使这些应用程序正常运行，这是一项耗时的工作。

    我可以更换笔记本电脑，也可以更换 MacOS 版本，但项目保持不变。它仍然需要相同的软件包集合才能成功运行部署脚本。而且该项目不再进行主动开发。我现在的日常工作主要使用 Java，所以我根本不需要这些软件包。但是，当我需要对那个 PHP 项目进行小修复并部署时，我必须安装所有所需的 PHP 软件包并进行配置。只有在这之后，我才能部署这个小修复。

    这真是令人恼火，至少可以这么说。

    Docker使我能够将所有这些自动化在一起。我的现有部署脚本将获得一个前文，其中将在一个干净的Ubuntu容器中安装和配置所有必要的与PHP相关的软件包。这个前文将在每次运行我的部署脚本时，在一个Docker容器内部执行。例如，它可能会像这样：

    在我开始使用Rultor之前，我的部署脚本是这样的：

    
# !/bin/bash

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

只有两行。第一行是运行单元测试。第二行是将文件通过FTP部署到生产服务器。非常简单。但是，这个脚本只能在安装了PHP 5.3、MySQL、Phing、xdebug、PHPCS和PHPUnit的情况下工作。再次强调，每次升级MacOS或更换笔记本电脑时，安装和配置它们都是一项繁琐的工作。

毋庸置疑，如果/当有人加入项目并尝试运行我的脚本，他/她将不得不再次进行这个预安装工作。

所以，这是一个新的脚本，我现在正在使用它。每次都在一个新的Docker容器中执行：

```bash

# !/bin/bash

# First, we install all prerequisites

sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

# And now the same script I had before

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

显然，在我的MacBook上运行这个脚本（无虚拟化）会引起很多麻烦。嗯，我这里甚至没有`apt-get`。

因此，Rultor给您带来的第一个好处是将您的部署脚本隔离在自己的虚拟环境中。这主要得益于Docker。

## Visibility of Logs

传统上，我们将部署脚本保存在某个 `~/deploy` 目录中，并使用一组魔法参数来运行它们。在一个小项目中，你会自己执行这个操作，并且该目录位于你自己的笔记本电脑上。在一个较大的项目中，有一台"部署"服务器，它有一个带有一组脚本的魔法目录，只有少数信任的高级开发人员可以执行这些脚本。我已经多次见到这种设置。

这里最大的问题是可追溯性。几乎不可能找出是谁部署了什么以及为什么某个特定的部署失败了。高级部署专家仅仅通过SSH登录服务器，并使用神奇的参数运行那些神奇的脚本。日志通常丢失，问题追踪非常困难或者根本不可能。

Rultor 提供了一些不同的功能。使用 Rultor，不再需要通过 SSH 访问部署脚本。所有脚本都保存在 `.rultor.yml` 配置文件中，并且你可以通过在问题跟踪系统（例如 GitHub、JIRA 或 Trac）中发布消息来启动它们。Rultor 运行脚本，并将完整的日志发布到你的工单中。日志将永久保存在你的项目中。你可以随时回到你正在处理的工单，并检查为什么部署失败以及实际执行了哪些指令。

例如，请查看此 GitHub 问题，我在部署 Rultor 的新版本时失败了几次：[yegor256/rultor#563](https://github.com/yegor256/rultor/issues/563)。我所有的失败尝试都被记录下来。我可以随时回顾它们并进行调查。对于一个大项目来说，这些信息是至关重要的。

因此，与独立部署脚本相比，Rultor的第二个好处是每个操作的可见性。

## Security of Credentials

当您在笔记本电脑或秘密团队部署服务器上放置自定义脚本时，您的生产凭据就靠近它。没有其他方法。如果您的软件与数据库一起使用，它必须知道登录凭据（用户名、密码、数据库名称、端口号等）。嗯，在最糟糕的情况下，有些人会将这些信息硬编码到源代码中。我们甚至不会讨论这种情况，因为它太糟糕了。

但是假设您将数据库凭据与源代码分开。您将会有类似于 `db.properties` 或 `db.ini` 的文件，它将在部署前附加到应用程序上。您还可以将该文件直接保存在生产服务器上，这样更好，但并非总是可能的，尤其是在 PaaS 部署中。

部署工件到存储库也存在类似的问题。假设，您经常部署到RubyGems.org。您的`~/.gem/credentials`将包含您的秘密API密钥。

所以，很多时候，您的部署脚本会附带一些包含敏感和安全信息的文件。而这些文件以明文、开放的格式呈现这些信息。没有加密，没有保护。只是以纯文本形式呈现的用户名、密码、代码和令牌。

为什么这很糟糕呢？对于一个拥有一台笔记本电脑的单个开发人员来说，这似乎不是一个问题。尽管如此，我不喜欢在某个机场遗失一台笔记本电脑，并且所有凭据都是开放且随时可用的。你可能会说，有一些磁盘保护工具，如MacOS的FileVault或Windows的BestCrypt。是的，也许。

但是让我们来看看当我们有一个开发团队一起工作并共享这些附带凭证的部署脚本和文件时会发生什么。一旦你给团队的新成员访问你的部署脚本的权限，你就必须共享所有敏感数据。这是无法避免的。为了使用这些脚本，他/她必须能够打开带有凭证的文件。

这是一个问题，如果你关心你的数据的安全性。

Rultor通过在部署脚本使用敏感数据之前提供即时的GPG解密来解决这个问题。在`.rultor.yml`配置文件中，你只需要这样说：

    decrypt: db.ini: “repo/db.ini.asc” deploy: script: ftp put db.ini production ```

    然后，使用Rultor GPG密钥对您的db.ini进行加密，然后毫不畏惧地提交db.ini.asc到代码库中。除了Rultor服务器在运行部署脚本之前，没有人能够打开和阅读该文件。

    因此，与独立部署脚本相比，Rultor的第三个好处是对敏感数据的适当安全性保护。

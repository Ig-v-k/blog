    Deployment Script vs. Rultor

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    当我解释Rultor如何自动化部署/发布流程时，经常会听到类似的反应：

    但是我已经有一个能够自动部署所有东西的脚本了。

    这种回应非常常见，所以我决定在一篇文章中总结我对自动化 Rultor 部署/发布流程的三个主要论点：1）隔离的 Docker 容器，2）日志可见性和3）凭据安全性。

    了解它们并查看Rultor在您现有的部署脚本上提供的功能。

    在我们开始争论之前，让我强调一下Rultor是你自定义脚本的一个有用的界面。当你决定使用Rultor进行自动化部署时，你不会丢弃任何现有的脚本。你只需要教会Rultor如何调用它们。

    Isolated Docker Containers

    一旦你开始从Rultor调用部署脚本，你会获得的第一个优势是使用Docker。我相信你知道什么是Docker，但是对于那些不知道的人来说，它是一个虚拟Linux“机器”的管理器。它是一个命令行脚本，当你需要在一个新的虚拟机（也称为“容器”）中运行一些脚本时，你可以调用它。Docker几乎立即启动容器并运行你的脚本。Docker的美妙之处在于每个容器都是一个完全隔离的Linux环境，具有自己的文件系统、内存、进程等。

    当您告诉Rultor运行您的部署脚本时，它会启动一个新的Docker容器并在其中运行您的脚本。但是，您可能会问，这给我带来了什么好处？

    主要的好处是在你的脚本完成后立即销毁容器。这意味着你可以在容器内进行所有的预配置，而不用担心与你的主要工作平台发生冲突。让我举个例子。

    我在 MacBook 上进行开发，安装和移除我需要用于开发的软件包。同时，我有一个项目需要部署，为了能够成功部署，需要 PHP 5.3、MySQL 5.6、Phing、PHPUnit、PHPCS 和 xdebug。每个 MacOS 版本都需要进行特定的配置才能让这些应用程序正常运行，这是一个费时的任务。

    我可以更换笔记本电脑，也可以更换MacOS版本，但项目保持不变。为了成功运行部署脚本，仍然需要相同的软件包集。而且该项目已不再进行积极开发。现在，我在日常工作中不再需要这些软件包，因为我现在更多地使用Java。但是，当我需要对那个PHP项目进行微小修复并部署时，我必须安装所有所需的PHP软件包并进行配置。只有在这之后，我才能部署这个微小的修复。

    这真是令人讨厌，至少可以这么说。

    Docker使我能够将所有这些自动化在一起。我的现有部署脚本将会获得一个前导部分，在一个干净的Ubuntu容器中安装和配置所有必要的与PHP相关的软件包。这个前导部分将在每次运行我的部署脚本时，在一个Docker容器内部执行。例如，它可能看起来像这样：

    在我开始使用Rultor之前，我的部署脚本是这样的：

    
# !/bin/bash

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

只有两行。第一行是运行所有单元测试。第二行是将文件传输到生产服务器的FTP部署。非常简单。但是，只有在安装了PHP 5.3、MySQL、Phing、xdebug、PHPCS和PHPUnit之后，此脚本才能运行。再次强调，每次升级我的MacOS或更换笔记本电脑时，安装和配置它们都是很费时的工作。

不用说，如果/当有人加入项目并尝试运行我的脚本，他/她将不得不再次进行这个预安装工作。

所以，这是一个新的脚本，我现在正在使用它。每次都在一个新的Docker容器中执行：

```bash

# !/bin/bash

# First, we install all prerequisites

sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

请运行以下命令安装：sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

# And now the same script I had before

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

显然，在我的 MacBook 上运行这个脚本（无需虚拟化）会带来很多麻烦。嗯，这里甚至没有 `apt-get`。

因此，Rultor给你带来的第一个好处是将你的部署脚本隔离在自己的虚拟环境中。我们主要得益于Docker。

## Visibility of Logs

传统上，我们将部署脚本保存在某个 `~/deploy` 目录中，并使用一组神奇的参数来运行它们。在小型项目中，您自己完成此操作，并且该目录位于您自己的笔记本电脑上。在较大的项目中，有一个“部署”服务器，该服务器有一个具有一组只能由少数可信任的高级开发人员执行的脚本的神奇目录。我已经多次见到这种设置。

这里面最大的问题是可追溯性。几乎不可能弄清楚是谁部署了什么，以及为什么某个特定的部署失败了。高级部署专家只需通过SSH登录服务器，并运行那些带有神奇参数的神奇脚本。日志通常丢失，问题跟踪非常困难或不可能。

Rultor 提供了与众不同的功能。使用 Rultor，不再需要通过 SSH 访问部署脚本。所有脚本都保存在 `.rultor.yml` 配置文件中，您可以通过在问题跟踪系统（例如 GitHub、JIRA 或 Trac）中发布消息来启动它们。Rultor 运行脚本并将完整日志发布到您的工单中。日志将永久保存在您的项目中。您可以随时回到您正在处理的工单并检查部署失败的原因以及实际执行的指令。

例如，查看此GitHub问题，我在部署Rultor的新版本时失败了几次：[yegor256/rultor#563](https://github.com/yegor256/rultor/issues/563)。所有我的失败尝试都有记录。我可以随时回顾并进行调查。对于一个大项目来说，这些信息至关重要。

因此，与独立部署脚本相比，Rultor的第二个好处是可以看到每个操作的可见性。

## Security of Credentials

当您的自定义脚本放置在您的笔记本电脑或秘密团队部署服务器上时，您的生产凭据就与之紧密相连。没有其他办法。如果您的软件与数据库一起工作，它就必须知道登录凭据（用户名、密码、数据库名、端口号等）。在最糟糕的情况下，有些人直接在源代码中硬编码这些信息。我们甚至不讨论这种情况，因为它非常糟糕。

但是假设你将数据库凭据与源代码分离开来。你会拥有类似 `db.properties` 或 `db.ini` 的文件，在部署前会与应用程序一起附加。你还可以直接将该文件保存在生产服务器上，这样更好，但并非总是可能的，尤其是在PaaS部署中。

部署艺术品到存储库也存在类似的问题。假设你经常部署到RubyGems.org。你的`~/.gem/credentials`文件将包含你的秘密API密钥。

所以，很多时候，您的部署脚本会伴随一些包含敏感和安全信息的文件。而且这些文件将这些信息以明文、开放的格式呈现。没有加密，没有保护。只是以纯文本形式保存用户名、密码、代码和令牌。

为什么这样不好？对于一个使用单一笔记本电脑的开发者来说，这似乎不是个问题。尽管如此，我不喜欢在机场丢失一台笔记本电脑，其中所有的凭证都是打开和准备好被使用的。你可能会说有一些磁盘保护工具，比如MacOS的FileVault或Windows的BestCrypt。是的，也许。

但让我们看看当我们有一个开发团队一起工作并共享那些具有凭据的部署脚本和文件时会发生什么。一旦你给团队的新成员访问你的部署脚本的权限，你就不得不共享所有敏感数据。这就是无法避免的。为了使用这些脚本，他/她必须能够打开带有凭据的文件。

这是个问题，如果你关心数据的安全性。

Rultor通过在部署脚本使用之前即时提供敏感数据的GPG解密来解决这个问题。在`.rultor.yml`配置文件中，你只需这样说：

    decrypt: db.ini: “repo/db.ini.asc” deploy: script: ftp put db.ini production ```

    然后，你使用Rultor GPG密钥加密你的db.ini文件，并毫不畏惧地提交db.ini.asc到代码库。除了Rultor服务器在运行部署脚本之前，没有人能够打开和阅读该文件。

    因此，与独立部署脚本相比，Rultor的第三个好处是对敏感数据的适当安全性。

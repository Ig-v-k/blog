    Logging in Unit Tests, a Bad Practice

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    日志记录是调试的不可避免的一部分。至少在现代高级编程语言和架构中是如此。在三十年前的汇编语言中并不是这样，但现在是这样的。有时我们会追踪变量，但很少。更常见的是将它们打印到控制台。此外，我们不仅使用println或其他用于控制台打印的方法将它们打印出来；相反，我们会将消息发送到一个日志记录框架，该框架处理控制台或其他日志记录目标，比如文件。这种框架的优点在于，我们在调试完成后不需要删除日志记录，只需配置框架在生产环境中抑制所有调试级别的消息即可。一些日志记录可能会在单元测试中发生。我们是把它们留在那里，还是删除它们呢？

    这里有一个例子（这是一个对我们目前正在开发的静态分析器Polystat的真实单元测试CalcTest.java进行简化的版本）：

    import com.jcabi.log.Logger;
import com.jcabi.xml.XML;
import org.hamcrest.MatcherAssert;
import org.hamcrest.Matchers;
import org.junit.jupiter.api.Test;
public final class FooTest {
  @Test
  public void buildsSimpleXml() {
    final XML xml = new Foo().build();
    Logger.debug(this, "This is the XML:\n%s", xml.toString());
    MatcherAssert.assertThat(
      xml,
      Matchers.notNullValue()
    );
  }
}

    这是Java，我正在使用JUnit5 + Hamcrest，以及我的自定义日志框架jcabi-log，它是Slf4j的装饰器，使用Log4j打印到控制台。

    这里发生了什么？有一个名为Foo的类，其中有一个名为build()的方法，它生成一个XML文档（我正在使用jcabi-xml库，它是JDK DOM的装饰器）。然后，单元测试会将XML文档的内容打印到控制台，并进行一个非常愚蠢的断言：该文档不为NULL。这是愚蠢的，因为如果它为NULL，日志语句在.toString()调用时就会失败。

    我是这段代码的开发者，所以我知道发生了什么：我懒得编写一个正确的断言，它可以查看XML文档并确保其中有正确的元素。我只是将它打印到控制台，目视确认其有效性，然后就结束了。如果我有更多时间，我会编写一个更好的单元测试（我刚刚在Polystat测试中进行了改进）：

    import com.jcabi.matchers.XhtmlMatchers;
import org.hamcrest.MatcherAssert;
import org.junit.jupiter.api.Test;
public final class FooTest {
  @Test
  public void buildsSimpleXml() {
    MatcherAssert.assertThat(
      XhtmlMatchers.xhtml(new Foo().build()),
      XhtmlMatchers.hasXPath("//foo")
    );
  }
}

    现在，XML文档已经构建并测试其内部是否存在//foo XPath。只有在断言失败的情况下，文档的内容才会打印到控制台上。如果XML具有所需的XPath，就不会有控制台输出，这意味着对于未来的开发人员来说没有噪音。

    此外，现在这是一个单语句测试，这本身是一个好的实践。

    回顾我的测试和日志经验，我认为在单元测试中记录日志是一个不好的主意。有时候是不可避免的，因为我们懒惰或者简单地没有足够的时间，但仍然是不好的。记录日志帮助我们通过视觉确认输出的正确性，但它将这个知识从项目中带走。那些以后将与测试一起工作的人将不知道我们在那里看到了什么。他们将看到控制台上的输出，但不会理解它是否仍然符合我写下时的期望。

    我想说，在单元测试中的每一行记录日志都是其作者的一条信息：“我对现在看到的数据有一些了解，但我太懒惰告诉你，你只需要相信它看起来很好。”

    我建议我们不要在我们的代码中留下这样的信息。

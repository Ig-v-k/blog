    Objectionary: Dictionary and Factory for EO Objects

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    自从Kernighan和Ritchie的时代，我们在库中共享二进制代码。你需要在C++中使用printf()打印一些文本吗？你可以获得包含700多个其他函数的libc库。你需要复制一个Java流吗？你可以获得包含copy()和其他140多个方法和类的Apache Commons IO。在我所了解的所有语言中，如Ruby、Python、JavaScript和PHP，情况都是一样的：你需要一个对象、一个类、一个函数或一个方法——你必须将整个库添加到你的构建中。是否更优雅地处理各个对象会更好呢？

    这个想法不是新的，也不是我的。我从David West的书《Object Thinking》中得到了这个想法，他在书中建议创建一个“Objectionary”（第306页），这是一个“字典和对象工厂的组合”，具有以下特性：

    • 对象的总数少于2000个；

    • 每个对象都是一个自治的可执行实体；

    • 每个对象都有一个唯一的ID和一个唯一的“地址”。

    • 物体只不过是物体的集合。

    • 对象需要特定硬件的虚拟机进行执行。

    十七年后（该书于2004年出版），我们在我们的新编程语言EO之上实现了这个想法。该语言故意比Java或C++简单得多。您可以在此处阅读它的正式描述。

    要将EO程序转换为可执行实体并发布到Objectionary，假设JVM用作目标平台（带有🌵标记的步骤由我们的eo-maven-plugin实现），必须经过以下强制步骤：

    • Assemble🌵:

    • 解析🌵：.eo ➜ .xmir

    • Optimize🌵: .xmir ➜ 更好的 .xmir

    • 发现🌵：找到所有外国别名

    • Pull🌵: download foreign .eo objects

    拉取🌵: 下载 外部的 .eo 对象

    • 解决🌵：下载并解压.jar构件

    • 位置🌵：将.class文件移动到target/classes/目录下。

    • 标记🌵：将在.jar中找到的.eo源文件标记为外部的

    • ↑ 如果仍有一些 .eo 文件未解析，请返回到 Parse

    • 转译🌵：.xmir ➜ .java

    • Assemble🌵：与上面相同，但用于测试。

    • 编译: .java ➜ .class

    • 测试：运行所有单元测试

    • Unplace🌵：删除.class文件的工具

    • Unspile🌵：删除自动生成的.java文件

    • 复制🌵：将.eo文件复制到.jar内的EO-SOURCES/目录中。

    • 部署：打包 .jar 文件并将其放入 Maven Central。

    • 推送：发送一个拉取请求到yegor256/objectionary。

    • 合并：我们测试并合并拉取请求

    这是一个迭代的过程，它一遍又一遍地循环，直到所有需要的.eo对象都被解析并且它们的原子作为.class文件存在。然后，所有的.xmir文件被转译为.java，然后编译为.class二进制文件。然后，测试、打包并部署到Maven中央仓库。然后，通过拉取请求合并到Objectionary的master分支。

    算法的第一部分可以使用我们的Maven插件自动化完成，只需将.eo源文件放在src/main/eo/中，并将其添加到pom.xml文件中：

    <project>
  <build>
    <plugins>
      <plugin>
        <groupId>org.eolang</groupId>
        <artifactId>eo-maven-plugin</artifactId>
        <version></version>
        <executions>
          <execution>
            <goals>
              <goal>register</goal>
              <goal>assemble</goal>
              <goal>transpile</goal>
              <goal>copy</goal>
              <goal>unplace</goal>
              <goal>unspile</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
    </plugins>
    [...]
  </build>
  [...]
</project>

    register 目标会扫描 src/main/eo/ 目录，找到所有 .eo 源文件，并将它们在 target/eo-foreigns.csv 中进行”注册”。接下来，assemble 目标会调用以下目标：parse、optimize、discover、pull 和 resolve。当这些目标进行解析、优化、拉取等操作时，它们都会使用 CSV 目录。

    当它们全部完成后，assemble 会检查目录：是否还有任何 .eo 文件需要解析？如果有，将再次开始另一个循环，从解析开始。当所有 .eo 文件都被解析后，目标 transpile 将被执行，它将 .xmir 文件转换为 .java 并放置在 target/generated-sources 中。其余的工作由标准的 maven-compiler-plugin 完成。

    让我们详细讨论每个步骤。

    Parse 🌵

    假设，这是位于 src/main/eo/hello.eo 的 .eo 源代码。

    +alias org.eolang.io.stdout
[] > hello
  "Jeff" > user
  stdout > @
    "Hello, %s!"
    user

    它将被解析为XMIR（XML中间表示）：

    <program>
  <o name="hello" line="1">
    <o name="user" data="string" line="2">Jeff</o>
    <o name="@" base="stdout" line="3">
      <o data="string" line="4">Hello, %s!</o>
      <o base="user" line="5"/>
    </o>
  </o>
</program>

    如果你想知道这个XML是什么意思，请阅读这个文档：其中有一个关于XMIR的部分。

    Optimize 🌵

    在这一步骤中，由解析器生成的XMIR经过许多XSL转换，有时会获得额外的元素和属性。我们的示例XMIR可能会获得一个新属性@ref，将对象user的引用指向对象定义所在的行。

    <program>
  <o name="hello" line="1">
    <o name="user" data="string" line="2">Jeff</o>
    <o name="@" base="stdout" line="3">
      <o data="string" line="4">Hello, %s!</o>
      <o base="user" line="5" ref="2"/>
    </o>
  </o>
</program>

    一些XSL转换可能会检查语法或语义错误，并在发现错误时添加一个新的元素<errors/>。因此，如果解析过程中没有发现任何语法错误，所有其他错误将在XMIR文档内可见，例如像这样：

    <program>
  <errors>
    <error line=>The program has no package</error>
  </errors>
  <o name="hello" line="1">
    <o name="user" data="string" line="2">Jeff</o>
    <o name="@" base="stdout" line="3">
      <o data="string" line="4">Hello, %s!</o>
      <o base="user" line="5" ref="2"/>
    </o>
  </o>
</program>

    顺便说一下，这不是一个真实的错误，我只是编造出来的。

    Discover 🌵

    在这一步中，我们找出哪些对象是”foreign”的。在我们的例子中，对象user不是foreign，因为它在我们面前的代码中已经定义了，而对象stdout在这里没有定义，所以是一个foreign对象。

    通过查看所有的.xmir文件，我们可以轻松判断哪个对象是foreign，只需查看它们的名称即可。一旦我们看到对org.eolang.io.stdout的引用，我们就会检查目录中是否存在包含所有.eo源代码的文件org/eolang/io/stdout.eo。如果该文件不存在，我们将将对象名称放入CSV目录并宣称它是foreign。

    Pull

    在这里，我们只是试图找到Objectionary中所有外部对象的源代码.eo文件，通过查看其GitHub存储库。例如，这是我们可以找到stdout.eo的地方。我们在那里找到它们并将其拉到本地磁盘上。

    请注意，我们拉取的是源代码，而不是二进制文件或已编译的XMIR文档，而是以.eo格式的源代码。

    Resolve 🌵

    这是在拉取之后stdout.eo可能看起来的样子。

    +package org.eolang.io
+rt jvm org.eolang:eo-runtime:0.10.2
[text] > stdout /bool

    该对象是一个原子。这意味着即使我们拥有它的源代码，但没有一段特定于平台的二进制代码，它就不完整。原子是由运行时平台实现的对象，在执行EO程序的地方（也称为FFI机制）。以+rt（运行时）开头的行解释了如何获取运行时代码。jvm部分是运行时的名称。

    我们访问Maven Central，在那里找到了org.eolang:eo-runtime:0.10.2这个构件，并对其进行解压缩（毕竟它是一个带有.class文件的压缩归档文件）。

    顺便说一下，一个程序可以包含多个+rt元指令，例如：

    +package org.eolang.io
+rt jvm org.eolang:eo-runtime:0.10.2
+rt ruby eo-core:0.5.8
+rt python eo-basics:0.0.3
[text] > stdout /bool

    在这里，三个运行时平台将知道如何获取stdout原子的缺失代码：EO➝Java将转到Maven Central获取JAR构件，EO➝Ruby将转到RubyGems尝试通过名称为eo-core和版本为0.5.8的gem进行查找，而EO➝Python将转到PyPi尝试查找版本为0.0.3的eo-basics包。

    Place 🌵

    接下来，我们将在解压的JAR中找到的所有.class文件放置在target/classes目录中。我们这样做是为了帮助Maven编译器插件在类路径中找到它们。

    Mark 🌵

    在每个到达的JAR文件中，我们可以找到.eo源文件。它们是在构建JAR文件时，该文件在类路径中具有的程序。我们也将其视为外部对象，并添加到CSV目录中。

    Transpile 🌵

    当目录中注册的所有外部对象都被下载、编译和优化后，我们就准备好开始源到源编译。我们不直接将XMIR编译成字节码，而是将其转译为.java文件，让Java编译器来生成字节码。

    我们相信将代码转译为Java与将代码编译为字节码相比，有一些优点：

    • 输出代码更易于阅读和调试。

    • 现有编译器的优化能力被重复利用。

    • 请将以下Markdown段落从英文翻译成中文，不要翻译技术术语和专有名词：

    转译器的复杂性低于编译器。

    • 输出代码的可移植性更高。

    我们已经有两个EO➝Java的转译器：一个正式版和由HSE大学制作的一个。我们还有由Innopolis大学的学生制作的EO➝Python实验性转译器。很可能，在您阅读本文时，会有更多的转译器可用。

    尽管我们相信转译，但仍然有可能创建EO➝Bytecode、EO➝LLVM或EO➝x86编译器。欢迎您尝试！

    Compile

    在这一步骤中，标准的Maven编译器插件会在target/generated-sources目录中找到自动生成的.java文件，并将其转换为.class文件。

    Unplace 🌵

    在这里，我们删除了从依赖中解压出来的所有.class文件。这是必要的，以避免将它们打包到最终的JAR文件中。

    我们之所以进行放置和取消放置，是因为Maven编译插件不允许在运行时扩展类路径。如果可能的话，我们只需从Maven中央仓库下载依赖项并将其添加到类路径中，而无需解压、放置和取消放置。

    Unspile 🌵

    在这里，我们从 target/classes/ 目录中删除所有从 .eo 自动生成的 .class 文件。我们不想发布可以从 .eo 源代码生成的二进制文件。我们只想发布原始的 .java 文件中的原子。

    Copy 🌵

    在这一步中，我们从src/main/eo/中获取所有的.eo源文件，并将它们复制到target/classes/EO-SOURCES/目录中。之后，它们将与.class文件一起打包成一个.jar文件，并部署到Maven Central。在复制过程中，我们将运行时版本中的0.0.0替换为当前部署版本。请查看文件stdout.eo，在其源代码库中。

    +package org.eolang.io
+rt jvm org.eolang:eo-runtime:0.0.0
[text] > stdout /bool

    在+rt行的版本为0.0.0。当源代码被复制到JAR文件中时，这段文字会被替换掉。

    将源代码与二进制文件一起发布的动机如下。当原子二进制文件从Java编译为字节码时，它们会与转换后的源代码一起保存。它们一起编译。此外，单元测试还依赖于原子源代码和自动生成/转换的源代码。我们希望JAR文件的未来用户知道在编译进行时我们使用了哪些源代码，也许让他们能够重现它，或者至少知道获取到的二进制文件的环境。

    从更实际的角度来看，我们需要在JAR文件中包含这些源代码，以便让Mark步骤理解哪些对象值得与解析的原子一起提取出来。

    Deploy

    在这里，我们将target/classes/中的所有内容打包成一个JAR档案，并将其部署到Maven Central。

    我建议还将源代码部署到GitHub Pages上，让用户可以在Web上查看。而且，当我们向Objectionary提交pull request时，这也将会很有帮助。请查看我在我的一个EO库中的这个.rultor.yml脚本，它将.eo源代码正确地部署到GitHub Pages，并替换其中的0.0.0版本标记。

    Push

    当部署完成并且Maven Central更新其CDN服务器后，是时候向yegor256/objectionary提交一个拉取请求了。对象的.eo源代码放在objects/中，它们的单元测试放在tests/中。基本上，我们只需要将src/main/eo/和src/test/eo复制过去就可以了。但是，停一下…有一个重要的细节。在源代码中，就像之前说的那样，我们将+rt版本设置为0.0.0。在这里，当我们复制到Objectionary时，版本必须设置为真实的数字。

    Merge

    当拉取请求到达时，yegor256/objectionary 存储库中预配置的 GitHub Action 将所有 .eo 源代码转换为所有已知平台，并运行所有单元测试。如果一切正常，我们会审查拉取请求，并决定所建议的对象是否与 Objectionary 中已存在的其他对象相符。

    一旦拉取请求合并后，这些对象将成为 EO 的所有对象的集中式字典的一部分。请查看 此拉取请求，其中一个新对象被提交到 Objectionary，其原子已部署到 Maven Central。

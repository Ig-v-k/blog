    Objectionary: Dictionary and Factory for EO Objects

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    自从Kernighan和Ritchie的时代以来，我们在库中共享二进制代码。在C++中需要用printf()打印一些文本吗？你可以获得内含700+其他函数的libc库。需要复制一个Java流吗？你可以获得内含copy()和140+其他方法和类的Apache Commons IO库。在我所了解的所有语言中，如Ruby、Python、JavaScript、PHP，都存在相同的情况：你需要一个对象、一个类、一个函数或一个方法，就必须将整个库添加到你的构建中。是否更优雅的做法是处理单个对象呢？

    这个想法并不新鲜，也不是我自己的。我是从David West的书Object Thinking中得到的，他建议创建一个Objectionary（第306页），这是一个“字典和对象工厂的结合体”，具有以下特点：

    • 物体的总数少于2000个。

    • 每个对象都是一个独立的可执行实体；

    • 每个对象都有一个唯一的ID和一个唯一的“地址”。

    • 对象只不过是对象的集合。

    • 对象执行需要硬件特定的虚拟机。

    十七年后（该书于2004年出版），我们在我们的新编程语言EO上实现了这个想法。该语言故意比Java或C++简单得多。您可以在此处阅读它的更或多或少正式的描述。

    将EO程序转换为可执行实体并发布到Objectionary，需要按照以下强制步骤进行，假设JVM被用作目标平台（由我们的eo-maven-plugin实现的步骤标有🌵）：

    • Assemble🌵:

    • Parse🌵: .eo ➜ .xmir

    解析🌵: .eo ➜ .xmir

    • 优化🌵：.xmir ➜ 更好的 .xmir

    • “Discover🌵：找到所有外文别名”

    • 拉取🌵：下载 外部的 .eo 对象

    • 解析🌵：下载并解压.jar文件。

    • Place🌵: 将.class文件移动到target/classes/

    • Mark🌵：将在.jar中找到的.eo源代码标记为外部文件

    • 如果仍有一些 .eo 文件没有解析，请返回到 Parse。

    • Transpile🌵: .xmir ➜ .java

    转译🌵：.xmir ➜ .java

    • Assemble🌵：与上述相同，但用于测试。

    • 编译：.java ➜ .class

    • 测试：运行所有单元测试

    • Unplace🌵: 删除 .class 文件的构件

    • “Unspile🌵: remove auto-generated .java files” can be translated to Chinese as: “Unspile🌵：移除自动生成的.java文件”

    • 复制🌵：将.eo文件复制到.jar内的EO-SOURCES/目录中。

    • 部署: 打包 .jar 文件并将其放入 Maven Central。

    • 推送：将拉取请求发送到 yegor256/objectionary。

    • 合并：我们测试并合并拉取请求

    这是一个迭代过程，它一遍又一遍地循环，直到所有所需的 .eo 对象都被解析，并且它们的原子出现为 .class 文件。然后，所有的 .xmir 文件被转译为 .java，然后编译为 .class 二进制文件。然后，进行测试，打包，并部署到 Maven Central。然后，通过拉取请求合并到 Objectionary 的 master 分支。

    算法的第一部分可以使用我们的Maven插件自动化完成，只需将.eo源文件放置在src/main/eo/目录中，并将以下内容添加到pom.xml文件中：

    <project>
  <build>
    <plugins>
      <plugin>
        <groupId>org.eolang</groupId>
        <artifactId>eo-maven-plugin</artifactId>
        <version></version>
        <executions>
          <execution>
            <goals>
              <goal>register</goal>
              <goal>assemble</goal>
              <goal>transpile</goal>
              <goal>copy</goal>
              <goal>unplace</goal>
              <goal>unspile</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
    </plugins>
    [...]
  </build>
  [...]
</project>

    register目标将扫描src/main/eo/目录，找到所有的.eo源文件，并将它们“注册”到target/eo-foreigns.csv这个特殊的CSV目录中。接下来，assemble目标将调用以下目标：parse、optimize、discover、pull和resolve。所有这些目标在解析、优化、拉取等过程中都使用CSV目录。

    当所有文件都完成时，assemble 检查目录：是否还有 .eo 文件需要解析？如果需要，将开始另一个循环，再次进行解析。当所有 .eo 文件都被解析后，将执行目标 transpile，将 .xmir 文件转换为 .java 并将它们放置在 target/generated-sources 中。剩下的工作由标准的 maven-compiler-plugin 完成。

    让我们详细讨论每个步骤。

    Parse 🌵

    假设，这是 .eo 源代码位于 src/main/eo/hello.eo：

    +alias org.eolang.io.stdout

[] > hello "Jeff" > user stdout > @ "Hello, %s!" user ```

它将被解析为XMIR（XML中间表示）：
    Jeff Hello, %s! 
    
如果你想知道这个XML是什么意思，请阅读[此文档](https://arxiv.org/abs/2111.13384)：其中有关于XMIR的部分。

## Optimize 🌵

在这个步骤中，由解析器生成的XMIR通过许多XSL转换进行处理，有时会添加额外的元素和属性。我们的示例XMIR可能会得到一个新的属性`@ref`，该属性指向对象`user`在定义该对象的行。
    Jeff Hello, %s! 
    
一些XSL转换可能会检查语法或语义错误，并在发现问题时添加一个新的元素`<errors/>`。因此，如果解析过程中没有发现任何语法错误，所有其他错误将在XMIR文档中可见，例如，像这样：
    <error line=>The program has no package</error> Jeff Hello, %s!

    ```

    顺便说一句，这不是一个真正的错误，我只是编造出来的。

    Discover 🌵

    在这一步中，我们找出哪些对象是“foreign”（外部的）。在我们的例子中，user对象不是外部的，因为它在我们面前的代码中被定义了，而stdout对象在这里没有被定义，所以它是一个外部对象。

    浏览所有的.xmir文件时，只需查看它们的名称，我们就可以轻松判断哪个对象是外部对象。一旦我们看到对org.eolang.io.stdout的引用，我们就检查在包含所有.eo源文件的目录中是否存在org/eolang/io/stdout.eo文件。如果该文件不存在，我们将对象名称放入CSV目录，并声明它为外部对象。

    Pull

    在这里，我们只是尝试找到Objectionary中所有外部对象的源代码.eo文件，方法是查看其GitHub存储库。例如，我们可以在这里找到stdout.eo。我们在那里找到它们并将其拉到本地磁盘上。

    请注意，我们拉取的是源代码。不是二进制文件或已编译的XMIR文档，而是以.eo格式的源代码。

    Resolve 🌵

    这是stdout.eo在pull之后的样子。

    ```xml +package org.eolang.io +rt jvm org.eolang:eo-runtime:0.10.2

    [text] > stdout /bool ```

    这个对象是一个atom。这意味着即使我们有它的源代码，没有一段特定于平台的二进制代码，它就不完整。atom是由运行时平台实现的对象，在EO程序执行时使用（也称为FFI机制）。以+rt（运行时）开头的行解释了如何获取运行时代码。jvm部分是运行时的名称。

    我们访问Maven Central，在那里找到了artifact org.eolang:eo-runtime:0.10.2，然后解压它（毕竟它是一个带有.class文件的zip归档文件）。

    顺便说一下，一个程序可能包含一些 +rt 元指令，例如：

    ```text +package org.eolang.io +rt jvm org.eolang:eo-runtime:0.10.2 +rt ruby eo-core:0.5.8 +rt python eo-basics:0.0.3

    [text] > stdout /bool ```

    在这里，三个运行时平台将会知道从哪里获取缺失的stdout原子的代码：EO➝Java将前往Maven Central获取JAR构件，EO➝Ruby将前往RubyGems尝试按名称eo-core和版本0.5.8查找宝石（gem），而EO➝Python将前往PyPi尝试查找版本为0.0.3的eo-basics包。

    Place 🌵

    接下来，我们将解压的JAR中找到的所有.class文件放置在target/classes目录中。我们这样做是为了帮助Maven编译插件在类路径中找到它们。

    Mark 🌵

    在每个到达的JAR文件中，我们可以找到.eo源代码。它们是构建此JAR文件时在类路径中存在的程序。我们也将其视为外部对象，并添加到CSV目录中。

    Transpile 🌵

    当目录中注册的所有外部对象都被下载、编译和优化后，我们就准备开始转译了。我们不直接将XMIR编译成字节码，而是将其转译为.java文件，然后让Java编译器来生成字节码。

    我们认为将代码转译为Java相比编译为字节码有一些好处：

    • 输出的代码更易于阅读和调试。

    • 现有编译器的优化能力被重复利用。

    • 跨语言编译器的复杂度低于编译器。

    • 输出代码的可移植性更高。

    我们已经有两个EO➝Java的转换器：一个官方版本和由HSE大学制作的另一个版本。我们还有由Innopolis大学的学生制作的EO➝Python实验性转换器。很可能，当您阅读本文时，会有更多的转换器可用。

    即使我们相信转译，仍然有可能创建EO➝Bytecode、EO➝LLVM或EO➝x86编译器。欢迎您尝试！

    Compile

    在这个步骤中，标准的Maven编译器插件会在target/generated-sources目录中找到自动生成的.java文件，并将它们转换为.class文件。

    Unplace 🌵

    在这里，我们删除了从依赖项中解压出来的所有 .class 文件。这是必要的，以避免将它们打包到最终的 JAR 文件中。

    我们之所以进行放置和取消放置，只是因为 Maven 编译器插件 不允许 我们在运行时扩展类路径。如果可能的话，我们只需从 Maven 中央仓库下载依赖项并将其添加到类路径，而无需解压、放置和取消放置。

    Unspile 🌵

    在这里，我们从target/classes/目录中删除所有由.eo自动生成的.class文件。我们不想发布可以从.eo源文件生成的二进制文件。我们只想发布原始的.java文件。

    Copy 🌵

    在这一步中，我们从src/main/eo/目录中提取所有的.eo源文件，并将它们复制到target/classes/EO-SOURCES/目录中。稍后，它们将与.class文件一起打包成一个.jar，并部署到Maven Central。在复制过程中，我们将运行时版本中的0.0.0替换为当前部署的版本。请查看文件stdout.eo，它位于源代码仓库中。

    ```text +package org.eolang.io +rt jvm org.eolang:eo-runtime:0.0.0

    [text] > stdout /bool ```

    位于 +rt 行的版本为 0.0.0。当源代码被复制到 JAR 文件时，此文本将被替换。

    将源代码与二进制文件一起发布的动机如下。当Atom二进制文件从Java编译为字节码时，它们与转译后的源代码一起存放。它们是一同编译的。此外，单元测试也依赖于Atom源代码和自动生成/转译的源代码。我们希望未来的JAR用户知道在编译过程中我们所使用的源代码，也许让他们能够重现该过程，或者至少知道他们获取的二进制文件所处的环境。

    从更实际的角度来看，我们需要将这些源文件放在JAR中，以便让“Mark”步骤了解哪些对象值得与已解析的原子一起提取。

    Deploy

    在这里，我们将target/classes/中的所有内容打包到一个JAR归档文件中，并将其部署到Maven中央仓库。

    我建议也将源代码部署到GitHub Pages上，以便让用户在网上查看。此外，当我们向Objectionary发起拉取请求时，这也将非常有帮助。请查看我其中一个EO库中的.rultor.yml脚本，它将.eo源代码正确地部署到GitHub Pages，并替换其中的0.0.0版本标记。

    Push

    当部署完成并且Maven Central更新了其CDN服务器后，是时候向yegor256/objectionary提交一个拉取请求了。对象的.eo源码放在objects/中，它们的单元测试放在tests/中。基本上，我们只需要将src/main/eo/和src/test/eo复制到那里。但是，停一下…有一个重要的细节。在源码中，正如前面所说，我们将+rt版本设置为0.0.0。在这里，当我们复制到Objectionary时，版本必须设置为实际数字。

    Merge

    当拉取请求到达时，yegor256/objectionary 仓库中预配置的 GitHub Action 将所有的 .eo 源代码转译为已知平台上的所有代码，并运行所有单元测试。如果一切正常，我们会审查拉取请求，并决定所建议的对象是否与 Objectionary 中已有的对象相匹配。

    一旦合并了拉取请求，这些对象将成为 EO 所有对象的集中字典的一部分。看一下此拉取请求，在将其原子部署到 Maven Central 后，一个新对象被提交到 Objectionary。

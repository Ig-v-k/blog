    Objectionary: Dictionary and Factory for EO Objects

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    自从Kernighan和Ritchie的时代以来，我们在库中共享二进制代码。你需要在C++中使用printf()打印一些文本吗？你可以得到libc库，里面还有其他700多个函数。你需要复制一个Java流吗？你可以得到Apache Commons IO，里面有copy()和其他140多个方法和类。在我所了解的所有语言中，比如Ruby、Python、JavaScript、PHP，都会出现同样的情况：你需要一个对象、一个类、一个函数或者一个方法，就必须将整个库添加到你的构建中。不过，如果能处理单个对象会更加优雅，不是吗？

    这个想法并不新鲜，也不是我的原创。我从Object Thinking这本书中得到的，作者是David West，他建议创建一个Objectionary（第306页），它是一个”字典和对象工厂的结合体”，具有以下特点：

    对象的总数少于2000个；

    每个对象都是一个自主可执行的实体；

    每个对象都有一个唯一的ID和一个唯一的“地址”；

    对象只不过是对象的集合。

    对象需要硬件特定的虚拟机进行执行。

    十七年后（该书于2004年出版），我们在我们的新编程语言EO上实现了这个想法。这种语言故意比Java或C++简单得多。您可以在这里阅读其相对正式的描述。

    将一个EO程序转变为可执行实体并发布到Objectionary需要经过以下强制步骤，假设JVM被用作目标平台（被🌵标记的步骤由我们的eo-maven-plugin实现）：

    Assemble🌵:

    解析🌵: .eo ➜ .xmir

    优化🌵: .xmir ➜ 更好的.xmir

    发现🌵：找到所有的外语别名

    拉取🌵: 下载 外部的 .eo 对象

    解决方法🌵：下载并解压.jar文件。

    地点🌵：将.class文件移动到target/classes/

    Mark🌵：将在 .jar 中发现的 .eo 源代码标记为外部的

    ↑ Go back to Parse if some .eo files are still not parsed

    转译🌵：.xmir ➜ .java

    Assemble🌵：与上述相同，但用于测试。

    编译：.java ➜ .class

    测试：运行所有单元测试

    Unplace🌵: 移除.class文件的工件

    Unspile🌵: 移除自动生成的.java文件

    复制🌵：将 .eo 文件复制到 .jar 内的 EO-SOURCES/ 文件夹中。

    部署：打包 .jar 文件，并将其放入Maven中央仓库。

    Push: 向yegor256/objectionary发送一个pull请求

    合并：我们测试并合并拉取请求

    这是一个迭代的过程，一遍又一遍地循环，直到所有需要解析的 .eo 对象都被解析并且它们的原子存在于 .class 文件中。然后，所有的 .xmir 文件会被转译为 .java，然后编译为 .class 二进制文件。然后，进行测试、打包，并部署到 Maven 中央仓库。然后，通过拉取请求合并到 Objectionary 的 master 分支中。

    算法的第一部分可以通过我们的Maven插件来自动化处理，只需将.eo源文件放置在src/main/eo/目录中，并将其添加到pom.xml文件中。

    <project>
  <build>
    <plugins>
      <plugin>
        <groupId>org.eolang</groupId>
        <artifactId>eo-maven-plugin</artifactId>
        <version></version>
        <executions>
          <execution>
            <goals>
              <goal>register</goal>
              <goal>assemble</goal>
              <goal>transpile</goal>
              <goal>copy</goal>
              <goal>unplace</goal>
              <goal>unspile</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
    </plugins>
    [...]
  </build>
  [...]
</project>

    register目标将扫描src/main/eo/目录，找到所有的.eo源文件，并将它们在target/eo-foreigns.csv中进行”注册”。接下来，assemble目标将调用以下目标：parse，optimize，discover，pull和resolve。所有这些目标在解析、优化、拉取等操作时都会使用CSV目录。

    当它们全部完成时，assemble 检查目录：是否还有需要解析的 .eo 文件？如果有，将再次开始一个循环并进行解析。当所有的 .eo 文件都被解析后，将执行目标 transpile，将 .xmir 文件转换为 .java 并放置在 target/generated-sources 中。其余的工作由标准的 maven-compiler-plugin 完成。

    让我们详细讨论每个步骤。

    Parse 🌵

    假设，这是 .eo 源代码位于 src/main/eo/hello.eo。

    +alias org.eolang.io.stdout

[] > hello "Jeff" > user stdout > @ "Hello, %s!" user ```

它将被解析为此XMIR（XML中间表示）：
    Jeff Hello, %s! 
    
如果你想知道这个XML是什么意思，阅读[这篇文档](https://arxiv.org/abs/2111.13384)：里面有关于XMIR的部分。

## Optimize 🌵

在这一步骤中，由解析器生成的XMIR经过许多XSL转换，有时会获得额外的元素和属性。我们的示例XMIR可能会获得一个新的属性`@ref`，将引用指向定义对象`user`的行。
    Jeff Hello, %s! 
    
一些XSL转换可能会检查语法或语义错误，并在发现错误时添加一个新元素`<errors/>`。因此，如果解析过程没有发现任何语法错误，所有其他错误将在XMIR文档中可见，例如，像这样：
    <error line=>The program has no package</error> Jeff Hello, %s!

    ```

    顺便说一下，这并不是一个真正的错误，我只是编造出来的。

    Discover 🌵

    在这一步中，我们找出哪些对象是“外部的”。在我们的例子中，对象user不是外部对象，因为它在我们面前的代码中被定义了，而对象stdout在这里没有被定义，所以它是一个外部对象。

    浏览所有的.xmir文件，我们可以通过查看它们的名称轻松判断哪个对象是外部的。一旦我们看到对org.eolang.io.stdout的引用，我们就会检查包含所有.eo源代码文件的目录中是否存在org/eolang/io/stdout.eo文件。如果文件不存在，我们将该对象名称放入CSV目录中，并宣称它是外部的。

    Pull

    在这里，我们只是尝试找到Objectionary中所有外部对象的源代码.eo文件，通过查看其GitHub存储库。例如，这就是我们会找到stdout.eo的地方。我们在那里找到它们并拉到本地磁盘上。

    请注意，我们拉取的是源代码。不是二进制文件或编译后的XMIR文档，而是以.eo格式的源代码。

    Resolve 🌵

    这是 stdout.eo 在拉取之后可能会看起来像的样子。

    ```xml +package org.eolang.io +rt jvm org.eolang:eo-runtime:0.10.2

    [text] > stdout /bool ```

    该对象是一个原子（atom）。这意味着尽管我们有它的源代码，但没有特定于平台的二进制代码它是不完整的。原子是由运行时平台实现的对象，EO程序在该平台上执行（也称为FFI机制）。以+rt（运行时）开头的行解释了如何获取运行时代码。jvm部分是运行时的名称。

    我们前往Maven Central，在那里找到了artifact org.eolang:eo-runtime:0.10.2，然后解压它（毕竟它是一个带有.class文件的zip归档文件）。

    顺便说一下，一个程序可能包含多个+rt元指令，例如：

    ```text +package org.eolang.io +rt jvm org.eolang:eo-runtime:0.10.2 +rt ruby eo-core:0.5.8 +rt python eo-basics:0.0.3

    [text] > stdout /bool ```

    在这里，三个运行时平台将知道如何获取stdout原子的丢失代码：EO➝Java将去Maven Central获取JAR构件，EO➝Ruby将去RubyGems尝试通过名称为eo-core和版本为0.5.8的gem进行查找，而EO➝Python将去PyPi尝试查找版本为0.0.3的eo-basics包。

    Place 🌵

    接下来我们将在解压的JAR文件中找到的所有.class文件放入target/classes目录中。我们这样做是为了帮助Maven编译插件在类路径中找到它们。

    Mark 🌵

    每个到达的JAR文件中，我们可以找到.eo源文件。它们是在构建JAR文件时在类路径中存在的程序。我们也将它们视为外部对象，并添加到CSV目录中。

    Transpile 🌵

    当目录中注册的所有外部对象被下载、编译和优化后，我们就可以开始转译。我们将XMIR转译成.java，然后让Java编译器生成字节码。

    我们认为将代码转译为Java与编译为字节码相比，有几个好处。

    输出代码更易于阅读和调试。

    现有编译器的优化能力被重复利用。

    请将以下的Markdown段落从英文翻译成中文，不要翻译技术术语和专有名词：

    一个转译器的复杂度比编译器低。

    输出代码的可移植性更高。

    我们已经有两个EO➝Java的转译器：一个标准的和由HSE University制作的另一个。我们还有由Innopolis University的学生制作的EO➝Python实验性转译器。很可能在您阅读此文章时，将有更多的转译器可用。

    尽管我们相信转译，但仍然可以创建 EO➝Bytecode、EO➝LLVM 或 EO➝x86 编译器。欢迎您尝试！

    Compile

    在这个步骤中，标准的Maven编译插件会在target/generated-sources目录中找到自动生成的.java文件，并将其转换为.class文件。

    Unplace 🌵

    在这里，我们删除了从依赖项中解压出来的所有.class文件。这是必要的，以避免将它们打包到最终的JAR文件中。

    我们之所以进行放置和取消放置，只是因为Maven Compiler插件不允许我们在运行时扩展类路径。如果可能的话，我们只需从Maven中央仓库下载依赖并将其添加到类路径中，而无需解压、放置和取消放置。

    Unspile 🌵

    在这里，我们从target/classes/目录中删除所有.class文件，这些文件是从.eo自动生成的。我们不希望发货二进制文件，这些文件可以从.eo源文件生成。我们只希望发货原始的.java文件。

    Copy 🌵

    在这一步中，我们从src/main/eo/目录中获取所有的.eo源文件，并将它们复制到target/classes/EO-SOURCES/目录中。稍后，它们将与.class文件一起打包成一个.jar文件，并部署到Maven Central。在复制过程中，我们将运行时版本中的0.0.0替换为当前部署的版本。请查看文件stdout.eo，在其源代码存储库中：

    ```text +package org.eolang.io +rt jvm org.eolang:eo-runtime:0.0.0

    [text] > stdout /bool ```

    在+rt行的版本是0.0.0。当源代码复制到JAR文件中时，此文本会被替换。

    将源代码与二进制文件一起发布的动机如下。当Atom二进制文件从Java编译为字节码时，它们与转译后的源代码保持在一起。它们一起编译。此外，单元测试还依赖于Atom源代码和自动生成/转译的源代码。我们希望将来使用JAR文件的用户知道编译过程中我们使用了哪些源代码，以便他们可能重现它，或者至少知道他们获取的二进制文件的背景信息。

    从一个更实际的角度来看，我们需要在JAR中包含这些资源，以便让Mark步骤理解哪些对象值得与已解析的原子一起提取。

    Deploy

    在这里，我们将所有内容打包到target/classes/中，并将其部署到Maven Central。

    我建议也将源代码部署到GitHub页面上，以便用户可以在Web上查看它们。此外，当我们向Objectionary提交拉取请求时，这将非常有帮助。在我的一个EO库中，检查这个.rultor.yml脚本，它会将.eo源代码正确地部署到GitHub页面上，并替换其中的0.0.0版本标记。

    Push

    当部署完成并且Maven Central更新其CDN服务器后，就可以提交一个Pull Request到yegor256/objectionary。对象的.eo源代码放在objects/目录中，它们的单元测试放在tests/目录中。基本上，我们只需将src/main/eo/和src/test/eo复制到那里。但是，请注意一个重要的细节。正如之前所说，在源代码中，我们将+rt版本设置为0.0.0。在这里，当我们复制到Objectionary时，版本必须设置为实际的数字。

    Merge

    当拉取请求到达时，存储在yegor256/objectionary仓库中的一个预配置的GitHub Action将所有的.eo源代码转换为所有已知平台，并运行所有的单元测试。如果一切都很干净，我们会审核拉取请求，并决定所建议的对象是否与Objectionary中已有的其他对象相匹配。

    一旦合并了拉取请求，这些对象将成为 EO 的所有对象的集中字典的一部分。看看这个拉取请求，在将其原子部署到 Maven Central 后，一个新的对象被提交到 Objectionary。

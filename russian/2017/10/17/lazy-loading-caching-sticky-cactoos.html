    Lazy Loading and Caching via Sticky Cactoos Primitives

    The following text is a translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Вы, конечно же, знаете, что такое ленивая загрузка, не так ли? И, безусловно, вы знаете о кэшировании. Насколько мне известно, в Java нет элегантного способа реализовать ни одно из них. Вот что я выяснил самостоятельно с помощью примитивов Cactoos.

    Допустим, нам нужен объект, который будет шифровать некоторый текст. Говоря в более объектно-ориентированном стиле, он будет инкапсулировать текст и превращаться в его зашифрованную форму. Вот как мы его будем использовать (давайте сначала создадим тесты).

    interface Encrypted {
  String asString() throws IOException;
}
Encrypted enc = new EncryptedX("Hello, world!");
System.out.println(enc.asString());

    Теперь давайте реализуем это очень примитивным способом, с использованием одного основного конструктора. Механизм шифрования просто добавляет +1 к каждому байту во входящих данных и предполагает, что шифрование не сломает ничего (очень глупое предположение, но для примера это будет работать).

    class Encrypted1 implements Encrypted {
  private final String text;
  Encrypted1(String txt) {
    this.text = txt;
  }
  @Override
  public String asString() {
    final byte[] in = this.text.getBytes();
    final byte[] out = new byte[in.length];
    for (int i = 0; i < in.length; ++i) {
      out[i] = (byte) (in[i] + 1);
    }
    return new String(out);
  }
}

    Все выглядит правильно? Я проверил это и оно работает. Если вводить "Hello, world!", то вывод будет "Ifmmp-!xpsme\""

    Далее, предположим, что мы хотим, чтобы наш класс принимал InputStream, а также String. Мы хотим вызывать его, например, следующим образом:

    Encrypted enc = new Encrypted2(
  new FileInputStream("/tmp/hello.txt")
);
System.out.println(enc.toString());

    Here is the most obvious implementation, with two primary constructors (again, the implementation is primitive, but works):

    class Encrypted2 implements Encrypted {
  private final String text;
  Encrypted2(InputStream input) throws IOException {
    ByteArrayOutputStream baos =
      new ByteArrayOutputStream();
    while (true) {
      int one = input.read();
      if (one < 0) {
        break;
      }
      baos.write(one);
    }
    this.text = new String(baos.toByteArray());
  }
  Encrypted2(String txt) {
    this.text = txt;
  }
  // asString() is exactly the same as in Encrypted1
}

    Технически это работает, но чтение потока происходит прямо внутри конструктора, что является плохой практикой. Основные конструкторы не должны делать ничего, кроме присваивания атрибутов, в то время как вторичные могут только создавать новые объекты.

    Let’s try to refactor and introduce lazy loading:

    class Encrypted3 implements Encrypted {
  private String text;
  private final InputStream input;
  Encrypted3(InputStream stream) {
    this.text = null;
    this.input = stream;
  }
  Encrypted3(String txt) {
    this.text = txt;
    this.input = null;
  }
  @Override
  public String asString() throws IOException {
    if (this.text == null) {
      ByteArrayOutputStream baos =
        new ByteArrayOutputStream();
      while (true) {
        int one = input.read();
        if (one < 0) {
          break;
        }
        baos.write(one);
      }
      this.text = new String(baos.toByteArray());
    }
    final byte[] in = this.text.getBytes();
    final byte[] out = new byte[in.length];
    for (int i = 0; i < in.length; ++i) {
      out[i] = (byte) (in[i] + 1);
    }
    return new String(out);
  }
}

    Works great, but looks ugly. The ugliest part is these two lines of course:

    this.text = null;
this.input = null;

    Они делают объект изменяемым и используют NULL. Это некрасиво, поверьте мне. К сожалению, ленивая загрузка и NULL-ссылки всегда идут вместе в классических примерах. Однако есть более хороший способ реализации. Давайте перепишем наш класс, на этот раз используя Scalar из Cactoos.

    class Encrypted4 implements Encrypted {
  private final IoCheckedScalar<String> text;
  Encrypted4(InputStream stream) {
    this(
      () -> {
        ByteArrayOutputStream baos =
          new ByteArrayOutputStream();
        while (true) {
          int one = stream.read();
          if (one < 0) {
            break;
          }
          baos.write(one);
        }
        return new String(baos.toByteArray());
      }
    );
  }
  Encrypted4(String txt) {
    this(() -> txt);
  }
  Encrypted4(Scalar<String> source) {
    this.text = new IoCheckedScalar<>(source);
  }
  @Override
  public String asString() throws IOException {
    final byte[] in = this.text.value().getBytes();
    final byte[] out = new byte[in.length];
    for (int i = 0; i < in.length; ++i) {
      out[i] = (byte) (in[i] + 1);
    }
    return new String(out);
  }

    Теперь выглядит намного лучше. Прежде всего, есть только один основной конструктор и два дополнительных. Во-вторых, объект неизменяемый. В-третьих, здесь все еще есть много места для улучшений: мы можем добавить еще конструкторов, которые будут принимать другие источники данных, например File или массив байтов.

    Вкратце, атрибут, который должен загружаться “ленивым” способом, представлен внутри объекта в виде “функции” (лямбда-выражения в Java 8). Пока мы не обращаемся к этому атрибуту, он не загружается. Когда нам нужно работать с ним, функция выполняется и у нас есть результат.

    Однако у этого кода есть одна проблема. Он будет считывать входной поток каждый раз, когда мы вызываем asString(), что, очевидно, не сработает, поскольку только первый раз поток будет содержать данные. При каждом последующем вызове поток будет просто пустым. Таким образом, нам нужно убедиться, что this.text.value() выполняет инкапсулированное Scalar только один раз. Все последующие вызовы должны возвращать ранее вычисленное значение. Поэтому нам нужно его кэшировать. Вот как:

    class Encrypted5 implements Encrypted {
  private final IoCheckedScalar<String> text;
  // same as above in Encrypted4
  Encrypted5(Scalar<String> source) {
    this.text = new IoCheckedScalar<>(
      new StickyScalar<>(source)
    );
  }
  // same as above in Encrypted4

    StickyScalar будет гарантировать, что только первый вызов его метода value() пройдет к инкапсулированному Scalar. Все остальные вызовы будут получать результат первого вызова.

    Последняя проблема, которую нужно решить, связана с конкурентностью. Код, который у нас есть выше, не является потокобезопасным. Если я создам экземпляр Encrypted5 и передам его двум потокам, которые одновременно вызовут asString(), результат будет непредсказуемым, просто потому, что StickyScalar не является потокобезопасным. Однако, есть другой примитив, который поможет нам - SyncScalar:

    class Encrypted5 implements Encrypted {
  private final IoCheckedScalar<String> text;
  // same as above in Encrypted4
  Encrypted5(Scalar<String> source) {
    this.text = new IoCheckedScalar<>(
      new SyncScalar<>(
        new StickyScalar<>(source)
      )
    );
  }
  // same as above in Encrypted4

    Now we’re safe and the design is elegant. It includes lazy loading and caching.

    I’m using this approach in many projects now and it seems convenient, clear, and object-oriented.

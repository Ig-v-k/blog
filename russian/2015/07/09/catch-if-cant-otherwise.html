    Catch Me If You ... Can't Do Otherwise

    The following text is a translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Я не знаю, является ли это анти-паттерном или просто обычной и очень популярной ошибкой, но я вижу это повсюду и просто обязан написать об этом. Я говорю о перехвате исключений без повторного выбрасывания. Я говорю о чем-то подобном этому Java-коду:

    try {
  stream.write(data);
} catch (IOException ex) {
  ex.printStackTrace();
}

    Pay attention: I don’t have anything against this code:

    try {
  stream.write('X');
} catch (IOException ex) {
  throw new IllegalStateException(ex);
}

    This is called exception chaining and is a perfectly valid construct.

    Так в чем проблема поймать исключение и зарегистрировать его? Давайте сначала посмотрим на широкую картину. Речь идет об объектно-ориентированном программировании - это означает, что мы имеем дело с объектами. Вот как будет выглядеть объект (точнее, его класс):

    финальный класс Wire { private final OutputStream stream; Wire(final OutputStream stm) { this.stream = stm; } public void send(final int data) { try { this.stream.write(x); } catch (IOException ex) { ex.printStackTrace(); } } }

    Here is how I’m using this class:

    new Wire(stream).send(1);

    Выглядит хорошо, верно? Мне не нужно беспокоиться о том IOException, когда я вызываю send(1). Он будет обработан внутренне, и если возникнет, будет записан стек вызовов. Но это совершенно неправильный способ мышления, и он унаследован от языков без исключений, таких как C.

    Исключения были разработаны для упрощения нашего дизайна путем перемещения всего кода обработки ошибок из основной логики. Более того, мы не просто перемещаем его, но также концентрируем его в одном месте - в методе main(), точке входа всего приложения.

    Основная цель исключения заключается в сборе максимального количества информации об ошибке и передаче ее на самый высокий уровень, где пользователь способен что-то сделать с этой информацией. Цепочка исключений дополнительно помогает, позволяя нам расширять эту информацию по мере поднятия ошибки выше. По сути, мы помещаем нашу пузырьковую оболочку (исключение) в каждый раз больший пузырь, когда мы его перехватываем и повторно выбрасываем. Когда оно достигает поверхности, имеется множество пузырьков, каждый из которых остается внутри другого, как русская кукла. Исходное исключение является наименьшим пузырьком.

    Когда вы перехватываете исключение, не перебрасывая его, вы, по сути, лопаете пузырь. Все, что находится внутри него, включая исходное исключение и все остальные пузыри с информацией внутри них, оказываются в ваших руках. Вы не позволяете мне их увидеть. Вы как-то используете их, но я не знаю, как. Вы делаете что-то за кулисами, скрывая потенциально важную информацию.

    Если ты это скрываешь от меня, я не могу обещать своему пользователю, что буду честным с ним и открыто сообщать о проблеме при ее возникновении. Я просто не могу больше доверять твоему методу send(), и мой пользователь не будет мне доверять.

    By catching exceptions without re-throwing them, you’re basically breaking the chain of trust between objects.

    My suggestion is to catch exceptions as seldom as possible, and every time you catch them, re-throw.

    К сожалению, дизайн Java противоречит этому принципу во многих местах. Например, в Java есть проверяемые и непроверяемые исключения, хотя по моему мнению должны быть только проверяемые (те, которые нужно перехватывать или объявлять как throwable). Кроме того, Java позволяет объявлять несколько типов исключений как throwable в одном методе - еще одна ошибка; следует придерживаться объявления только одного типа. Также в иерархии есть обобщенный класс Exception, который также является ошибкой по моему мнению. Кроме того, некоторые встроенные классы не позволяют выбрасывать проверяемые исключения, например, Runnable.run(). В Java существует много других проблем с исключениями.

    But try to keep this principle in mind and your code will be cleaner: catch only if you have no other choice.

    P.S. Here is how the class should look:

    финальный класс Wire { private final OutputStream stream; Wire(final OutputStream stm) { this.stream = stm; } public void send(final int data) throws IOException { this.stream.write(x); } }

    How Cookie-Based Authentication Works in the Takes Framework

    The following text is a translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Когда вы вводите свою электронную почту и пароль на странице входа в Facebook, вы попадаете в свою учетную запись. Затем, независимо от того, куда вы переходите на сайте, вы всегда видите свое фото в правом верхнем углу страницы. Facebook запоминает вас и не запрашивает пароль снова и снова благодаря HTTP-кукисам, и это называется аутентификацией на основе кукисов. Несмотря на то, что этот механизм часто вызывает некоторые проблемы с безопасностью, он очень популярен и прост. Вот как Takes делает это возможным в нескольких строках кода.

    First, let’s see how it works. Moreover, let’s see how I believe it should work.

    Шаг первый: пользователь вводит электронную почту и пароль, а затем нажимает “отправить”. Сервер получает POST-запрос с этой информацией внутри:

    POST / HTTP/1.1
Host: www.facebook.com
Content-Type: application/x-www-form-urlencoded

email=me@yegor256.com&password=itisasecret ```

Сервер сопоставляет предоставленную информацию с записями и принимает решение о дальнейших действиях. Если информация недействительна, он возвращает ту же страницу входа и просит вас ввести все данные заново. Если информация действительна, сервер возвращает что-то вроде этого:

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=me@yegor256.com

    
Поскольку код состояния ответа равен 303, браузер переходит на страницу, указанную в заголовке Location, и открывает главную страницу сайта. Вот что он отправляет на сервер:

    GET / HTTP/1.1 Host: www.facebook.com Cookie: user=me@yegor256.com

    
Сервер получает мою электронную почту из заголовка Cookie и понимает, что это снова я! Нет необходимости запрашивать пароль еще раз. Сервер доверяет информации из cookie. Вот и все. Именно это и есть аутентификация на основе cookie.

## Wait ... What About Security?

Правильно, а что насчет безопасности? Если сервер доверяет любому запросу браузера с пользовательским электронным адресом в заголовке Cookie, кто угодно сможет отправлять мои электронные письма с другого места и получить доступ к моему аккаунту.

Первый шаг для предотвращения этого - зашифровать электронную почту с помощью секретного ключа шифрования, известного только серверу. Никто, кроме самого сервера, не сможет зашифровать ее таким же образом, как сервер должен расшифровать ее. Ответ будет выглядеть так, используя в качестве примера шифрование с помощью XOR-шифра с бамбуком в качестве секретного ключа.

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=b1ccafd92c568515100f5c4d104671003cfa39

    
This is not the best encryption mechanism, though; for proper encryption, it's better to use something stronger like DES.

Все это звучит хорошо, но что, если кто-то перехватывает трафик между сервером и браузером и получает правильно зашифрованный email-файл cookie? В этом случае вор смог бы использовать тот же самый файл cookie для аутентификации даже не зная его содержания. Сервер доверил бы информации и позволил бы человеку войти в мой аккаунт. Такой тип атаки называется "человек посередине" (MITM). Чтобы предотвратить это, мы должны использовать HTTPS и сообщить браузеру, что файл cookie является конфиденциальным и никогда не должен отправляться на сервер без SSL-шифрования. Это делается с помощью дополнительного флага в заголовке Set-Cookie.

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=me@yegor256.com; Secure

    
Есть еще один тип атаки, связанный с аутентификацией на основе куки, основанный на возможности браузера раскрыть все куки, связанные с веб-страницей, для выполнения внутри нее JavaScript. Злоумышленник может внедрить некоторый вредоносный код JavaScript на страницу (не спрашивайте меня, как ... это произойдет только если весь HTML-рендеринг выполнен неправильно), и этот код получит доступ к куке. Затем код отправит куку куда-то еще, чтобы злоумышленник мог ее собрать. Этот тип атаки называется межсайтовый скриптинг (XSS). Чтобы предотвратить это, существует еще один флаг для заголовка Set-Cookie, называемый HttpOnly:

    HTTP/1.1 303 See Other Location: www.facebook.com Set-Cookie: user=me@yegor256.com; Secure; HttpOnly ```

    Присутствие этого флага скажет браузеру, что данный файл cookie может быть передан обратно на сервер только через HTTP запросы. JavaScript не будет иметь к нему доступа.

    How It’s Done in Takes

    Вот как механизм аутентификации на основе файлов cookie спроектирован в фреймворке Takes. Весь фреймворк состоит из takes, которые получают запросы и генерируют ответы (эта статья подробнее объясняет фреймворк). Когда поступает запрос, мы должны найти файл cookie аутентификации в заголовке Cookie и перевести его в учетные данные пользователя. Когда ответ уходит, мы должны добавить к нему заголовок Set-Cookie с зашифрованными учетными данными пользователя. Вот и все, всего два шага.

    Let’s say we have an account page that is supposed to show the current user’s balance:

    окончательный класс TkAccount реализует Take { private final Balances balances; @Override public Response act(final Request request) { final Identity user = // получить его из запроса return RsHTML( String.format( “<html>Ваш баланс составляет %s</html>”, this.balances.retrieve(user) ) ); } }

    Сразу после поступления запроса, мы должны извлечь идентификатор пользователя, закодированный в аутентификационном файле cookie. Чтобы сделать этот механизм повторно используемым, у нас есть декоратор TkAuth, который оборачивает существующий код, декодирует входящий файл cookie и добавляет новый заголовок TkAuth к запросу с информацией об идентификации пользователя.

    final Codec codec = new CcHex(new CcXOR(new CcPlain())); final Pass pass = new PsCookie(codec); new TkAuth(new TkAccount(), pass);

    финальный Кодек кодек = new CcHex(new CcXOR(new CcPlain())); финальный Пароль пароль = new PsCookie(кодек); new TkAuth(new TkAccount(), пароль);

    Опять же, когда TkAuth получает запрос с аутентификационным куки внутри, он запрашивает пароль для декодирования куки и возвращает либо правильное Идентификационное значение, либо Идентификацию.ANONYMOUS.

    Затем, когда ответ возвращается в браузер, TkAuth запрашивает пароль для кодирования идентификатора обратно в строку и добавляет Set-Cookie в ответ.

    PsCookie uses an instance of Codec in order to do these backward and forward encoding operations.

    Когда наш TkAccount хочет получить идентификатор текущего аутентифицированного пользователя из запроса, он может использовать RqAuth, декоратор утилиты Request.

    окончательный класс TkAccount, реализующий интерфейс Take { @Override public Response act(final Request request) { final Identity user = new RqAuth(request).identity(); // другие манипуляции с пользователем } }

    The RqAuth decorator uses the header, added by PsCookie, in order to authenticate the user and create an Identity object.

    How Is It Composable?

    Этот механизм действительно очень гибкий и “компонуемый”. Предположим, мы хотим пропустить аутентификацию во время интеграционного тестирования. Вот как:

    новый TkAuth( take, // оригинальное приложение “take” new PsChain( new PsFake(/* если запущены интеграционные тесты */), new PsCookie( new CcHex(new CcXOR(new CcPlain())) ) ) );

    PsChain реализует Pass и пытается аутентифицировать пользователя, запрашивая все инкапсулированные пропуски один за другим. Первый в цепочке - PsFake. Используя только один логический аргумент в своем конструкторе, он принимает решение, возвращать фальшивую идентификацию или ничего не возвращать. С помощью одного логического триггера мы можем выключить весь механизм аутентификации в приложении.

    Let’s say you want to authenticate users through Facebook OAuth. Here is how:

    новый TkAuth( take, // оригинальное приложение “take” new PsChain( new PsByFlag( new PsByFlag.Pair( PsFacebook.class.getSimpleName(), new PsFacebook( “… ключ Facebook API …”, “… секрет Facebook API …” ) ) ), new PsCookie( new CcHex(new CcXOR(new CcPlain())) ) ) );

    Когда пользователь нажимает на ссылку для входа на вашем сайте, браузер переходит на facebook.com, где происходит проверка его личности. Затем Facebook возвращает ответ на перенаправление 302 с заголовком Location, установленным на URL, который мы предоставляем в ссылке для входа. Ссылка должна содержать что-то вроде этого: ?PsByFlag=PsFacebook. Это сообщит PsByFlag, что этот запрос аутентифицирует пользователя.

    PsByFlag будет перебирать все инкапсулированные “пары” и попытается найти нужную. PsFacebook будет первым и правильным. Он подключится к API Facebook, используя предоставленные учетные данные, и получит всю возможную информацию о пользователе.

    Here is how we can implement a logout mechanism:

    новый TkAuth( take, // оригинальное приложение “take” new PsChain( new PsByFlag( new PsByFlag.Pair( PsFacebook.class.getSimpleName(), new PsFacebook( “… Facebook API key …”, “… Facebook API secret …” ) ), new PsByFlag.Pair( PsLogout.class.getSimpleName(), new PsLogout() ) ), new PsCookie( new CcHex(new CcXOR(new CcPlain())) ) ) );

    Now, we can add ?PsByFlag=PsLogout to any link on the site and it will log the current user out.

    You can see how all this works in a real application by checking out the TkAppAuth class in Rultor.

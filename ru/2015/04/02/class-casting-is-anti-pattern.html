    Class Casting Is a Discriminating Anti-Pattern

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Приведение типов - очень полезная техника, когда нет времени или желания тщательно продумывать и проектировать объекты. Приведение типов (или приведение классов) помогает нам работать с предоставленными объектами по-разному, основываясь на классе, к которому они принадлежат, или на интерфейсе, который они реализуют. Приведение классов помогает нам дискриминировать бедных объектов и разделять их по их расе, полу и религии. Может ли это быть хорошей практикой?

    Это очень типичный пример приведения типов (Google Guava полон таких примеров, например Iterables.size()).

    class Foo {
  int sizeOf(Iterable items) {
    int size = 0;
    if (items instanceof Collection) {
      size = Collection.class.cast(items).size();
    } else {
      for (Object item : items) {
        ++size;
      }
    }
    return size;
  }
}

    Этот метод sizeOf() вычисляет размер итерируемого объекта. Однако, он достаточно умён, чтобы понять, что если items также являются экземплярами Collection, нет необходимости фактически проходить по ним. Было бы гораздо быстрее привести их к типу Collection и затем вызвать метод size(). Звучит логично, но в этом подходе есть две практические проблемы.

    Сначала есть скрытое связывание sizeOf() и Collection. Это связывание не видно клиентам sizeOf(). Они не знают, что метод sizeOf() зависит от интерфейса Collection. Если завтра мы решим его изменить, sizeOf() не будет работать. И мы будем очень удивлены, потому что его сигнатура ничего не говорит об этой зависимости. Это, конечно же, не произойдет с Collection, так как она является частью Java SDK, но с пользовательскими классами это может и произойдет.

    Вторая проблема - неизбежное растущее увеличение сложности метода sizeOf(). Чем больше специальных типов ему приходится обрабатывать по-разному, тем сложнее он станет. Это if/then ветвление неизбежно, так как он должен проверять все возможные типы и давать им особое обращение. Такая сложность является результатом нарушения принципа единственной ответственности. Метод вычисляет не только размер Iterable, но и выполняет приведение типов и ветвление на основе этого приведения.

    Что является альтернативой? Есть несколько вариантов, но самым очевидным является перегрузка методов (недоступная в полноценных ООП-языках, таких как Ruby или PHP):

    class Foo {
  int sizeOf(Iterable items) {
    int size = 0;
    for (Object item : items) {
      ++size;
    }
    return size;
  }
  int sizeOf(Collection items) {
    return items.size();
  }
}

    Это не более изящно?

    Философски говоря, приведение типов - это дискриминация по отношению к объекту, поступающему в метод. Объект соответствует контракту, предоставленному сигнатурой метода. Он реализует интерфейс Iterable, который является контрактом, и ожидает равного обращения со всеми остальными объектами, поступающими в тот же метод. Но метод дискриминирует объекты по их типам. Метод фактически спрашивает объект о его… расе. Черные объекты идут вправо, а белые объекты - влево. Вот что делает этот instanceof, и в этом суть дискриминации.

    Используя instanceof, метод отделяет входящие объекты по определенным группам, к которым они принадлежат. В данном случае есть две группы: коллекции и все остальные. Если вы являетесь коллекцией, вам предоставляется особое обращение. Несмотря на то, что вы соответствуете контракту Iterable, мы все равно особо обрабатываем некоторые объекты, потому что они принадлежат к “элитной” группе под названием Collection.

    Вы можете сказать, что Collection - это просто еще один контракт, с которым объект может согласиться. Это верно, но в этом случае должна быть еще одна дверь, через которую должны пройти те, кто работает по этому контракту. Вы объявили, что sizeOf() принимает всех, кто работает по контракту Iterable. Я - объект и делаю то, что говорит контракт. Я вхожу в метод и ожидаю равного обращения со всеми остальными, кто входит в этот же метод. Но, по-видимому, когда я оказываюсь внутри метода, я понимаю, что некоторые объекты имеют некоторые особые привилегии. Разве это не дискриминация?

    В заключение, я бы считал использование instanceof и приведения типов классов анти-паттернами и запахами кода. Как только вы видите необходимость в их использовании, начинайте думать о рефакторинге.

    Temporal Coupling Between Method Calls

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Временная связность возникает между последовательными вызовами методов, когда они должны быть выполнены в определенном порядке. Это неизбежно в императивном программировании, но мы можем снизить негативное влияние, просто преобразовав эти статические процедуры в функции. Взгляните на этот пример.

    Here is the code:

    class Foo {
  public List<String> names() {
    List<String> list = new LinkedList();
    Foo.append(list, "Jeff");
    Foo.append(list, "Walter");
    return list;
  }
  private static void append(
    List<String> list, String item) {
    list.add(item.toLowerCase());
  }
}

    Что вы об этом думаете? Я считаю, что понятно, что делает names() — создает список имен. Чтобы избежать дублирования, есть дополнительная процедура append(), которая приводит элемент к нижнему регистру и добавляет его в список.

    Это плохой дизайн.

    Это процедурный дизайн, и между строками в методе names() есть временная связь.

    Давайте сначала покажу вам лучший (хотя и не лучший!) дизайн, а затем попытаюсь объяснить его преимущества:

    class Foo {
  public List<String> names() {
    return Foo.with(
      Foo.with(
        new LinkedList(),
        "Jeff"
      ),
      "Walter"
    );
  }
  private static List<String> with(
    List<String> list, String item) {
    list.add(item.toLowerCase());
    return list;
  }
}

    Идеальный дизайн для метода with() создаст новый экземпляр List, заполнит его с помощью addAll(list), затем добавит в него add(item) и, наконец, вернет. Это было бы абсолютно неизменяемым, но медленным.

    Итак, что не так с этим:

    List<String> list = new LinkedList();
Foo.append(list, "Jeff");
Foo.append(list, "Walter");
return list;

    Выглядит абсолютно чисто, не так ли? Создайте список, добавьте в него два элемента и верните его. Да, это чисто — пока что. Потому что мы помним, что делает append(). Через несколько месяцев мы вернемся к этому коду, и он будет выглядеть так:

    List<String> list = new LinkedList();
// 10 more lines here
Foo.append(list, "Jeff");
Foo.append(list, "Walter");
// 10 more lines here
return list;

    Теперь ли это ясно, что append() фактически добавляет "Jeff" в list? Что произойдет, если я удаляю эту строку? Будет ли это влиять на результат, возвращаемый в последней строке? Я не знаю. Мне нужно проверить тело метода append(), чтобы убедиться.

    А что насчет сначала возвращения list, а затем вызова append()? Вот что может сделать “рефакторинг” в нашем коде:

    List<String> list = new LinkedList();
if (/* something */) {
  return list;
}
// 10 more lines here
Foo.append(list, "Walter");
Foo.append(list, "Jeff");
// 10 more lines here
return list;

    Прежде всего, мы возвращаем list слишком рано, когда он еще не готов. Но кто-нибудь сказал мне, что эти два вызова append() должны происходить до return list? Во-вторых, мы поменяли порядок вызовов append(). Опять же, кто-нибудь сказал мне, что важно вызывать их в этом определенном порядке?

    Никто. Нигде. Это называется временной связностью.

    Наши строки связаны вместе. Они должны оставаться в этом определенном порядке, но знание об этом порядке скрыто. Легко нарушить порядок, и наш компилятор не сможет нас поймать.

    В отличие от этого, в этом дизайне нет никакого “порядка”:

    return Foo.with(
  Foo.with(
    new LinkedList(),
    "Jeff"
  ),
  "Walter"
);

    Он просто возвращает список, который строится с помощью нескольких вызовов метода with(). Это одна строка, а не четыре.

    Как уже обсуждалось ранее, идеальный метод в ООП должен содержать только один оператор, а именно оператор return.

    То же самое верно и для валидации. Например, этот код плох:

    list.add("Jeff");
Foo.checkIfListStillHasSpace(list);
list.add("Walter");

    В то время как этот гораздо лучше:

    list.add("Jeff");
Foo.withEnoughSpace(list).add("Walter");

    See the difference?

    И, конечно же, идеальным подходом было бы использование составных декораторов вместо этих уродливых статических методов. Но если по какой-то причине это не возможно, просто не делайте эти статические методы похожими на процедуры. Убедитесь, что они всегда возвращают результаты, которые становятся аргументами для последующих вызовов.

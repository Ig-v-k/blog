    Temporal Coupling Between Method Calls

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Временная связь возникает между последовательными вызовами методов, когда они должны происходить в определенном порядке. Это неизбежно в императивном программировании, но мы можем уменьшить отрицательный эффект, просто превратив эти статические процедуры в функции. Взгляните на этот пример.

    Here is the code:

    class Foo {
  public List<String> names() {
    List<String> list = new LinkedList();
    Foo.append(list, "Jeff");
    Foo.append(list, "Walter");
    return list;
  }
  private static void append(
    List<String> list, String item) {
    list.add(item.toLowerCase());
  }
}

    Что вы об этом думаете? Я считаю, что понятно, что делает names() - создает список имен. Чтобы избежать дублирования, есть дополнительная процедура append(), которая преобразует элемент в нижний регистр и добавляет его в список.

    Это плохой дизайн.

    Это процедурный дизайн, и между строками в методе names() существует временная связь.

    Сначала позвольте мне показать вам лучший (хотя и не самый лучший!) дизайн, а затем я постараюсь объяснить его преимущества.

    class Foo {
  public List<String> names() {
    return Foo.with(
      Foo.with(
        new LinkedList(),
        "Jeff"
      ),
      "Walter"
    );
  }
  private static List<String> with(
    List<String> list, String item) {
    list.add(item.toLowerCase());
    return list;
  }
}

    Идеальное решение для метода with() создало бы новый экземпляр List, заполнило бы его с помощью addAll(list), затем add(item) и, наконец, вернуло бы его. Это было бы полностью неизменяемым, но медленным.

    Итак, что не так с этим:

    List<String> list = new LinkedList();
Foo.append(list, "Jeff");
Foo.append(list, "Walter");
return list;

    Оно выглядит совершенно чистым, не так ли? Создайте список, добавьте в него два элемента и верните его. Да, он чистый - пока что. Потому что мы помним, что делает append(). Через несколько месяцев, когда мы вернемся к этому коду, он будет выглядеть так:

    List<String> list = new LinkedList();
// 10 more lines here
Foo.append(list, "Jeff");
Foo.append(list, "Walter");
// 10 more lines here
return list;

    Теперь ли это так ясно, что append() фактически добавляет "Jeff" в list? Что произойдет, если я удалю эту строку? Будет ли это влиять на результат, возвращаемый в последней строке? Я не знаю. Мне нужно проверить тело метода append(), чтобы убедиться.

    Как насчёт сначала возвращать list, а затем вызывать append()? Это то, что возможно сделает “рефакторинг” нашего кода:

    List<String> list = new LinkedList();
if (/* something */) {
  return list;
}
// 10 more lines here
Foo.append(list, "Walter");
Foo.append(list, "Jeff");
// 10 more lines here
return list;

    Прежде всего, мы возвращаем list слишком рано, когда он еще не готов. Но кто-то мне сказал, что эти два вызова append() должны произойти перед return list? Во-вторых, мы изменили порядок вызовов append(). И снова, кто-то мне сказал, что важно вызывать их в этом конкретном порядке?

    Никто. Нигде. Это называется временной связью.

    Наши строки связаны между собой. Они должны оставаться в этом определенном порядке, но знание о порядке скрыто. Легко нарушить порядок, и наш компилятор не сможет нас поймать.

    Наоборот, этот дизайн не имеет какого-либо “порядка”:

    return Foo.with(
  Foo.with(
    new LinkedList(),
    "Jeff"
  ),
  "Walter"
);

    Он просто возвращает список, который строится несколькими вызовами метода with(). Это одна строка вместо четырех.

    Как уже обсуждалось ранее, идеальный метод в ООП должен иметь только одно выражение, а именно return.

    То же самое относится и к проверке. Например, этот код плох:

    list.add("Jeff");
Foo.checkIfListStillHasSpace(list);
list.add("Walter");

    В то время как этот гораздо лучше:

    list.add("Jeff");
Foo.withEnoughSpace(list).add("Walter");

    See the difference?

    И, конечно же, идеальным подходом было бы использовать составные декораторы вместо этих ужасных статических методов. Но если по какой-то причине это невозможно, просто не делайте эти статические методы похожими на процедуры. Убедитесь, что они всегда возвращают результаты, которые становятся аргументами для последующих вызовов.

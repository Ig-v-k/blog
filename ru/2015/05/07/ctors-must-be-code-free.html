    Constructors Must Be Code-Free

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    На сколько работы должно выполняться внутри конструктора? Кажется разумным провести некоторые вычисления внутри конструктора и затем инкапсулировать результаты. Таким образом, когда результаты потребуются методам объекта, у нас уже будут готовы. Звучит как хороший подход? Нет, это плохая идея по одной причине: она мешает композиции объектов и делает их нерасширяемыми.

    Let’s say we’re making an interface that would represent a name of a person:

    interface Name {
  String first();
}

    Pretty easy, right? Now, let’s try to implement it:

    public final class EnglishName implements Name {
  private final String name;
  public EnglishName(final CharSequence text) {
    this.name = text.toString().split(" ", 2)[0];
  }
  @Override
  public String first() {
    return this.name;
  }
}

    В чем проблема с этим? Это же быстрее, верно? Оно разделяет имя на части только один раз и инкапсулирует их. Затем, несмотря на то, сколько раз мы вызываем метод first(), он будет возвращать одно и то же значение и не будет нуждаться в повторном разделении. Однако, это неправильное мышление! Позвольте мне показать вам правильный путь и объяснить:

    public final class EnglishName implements Name {
  private final CharSequence text;
  public EnglishName(final CharSequence txt) {
    this.text = txt;
  }
  @Override
  public String first() {
    return this.text.toString().split("", 2)[0];
  }
}

    This is the right design. I can see you smiling, so let me prove my point.

    Прежде чем начать доказательство, позвольте мне попросить вас прочитать эту статью: Composable Decorators vs. Imperative Utility Methods. В ней объясняется разница между статическим методом и составными декораторами. Первый фрагмент, приведенный выше, очень похож на императивный утилитарный метод, хотя и выглядит как объект. Второй пример является настоящим объектом.

    В первом примере мы злоупотребляем оператором new и превращаем его в статический метод, который выполняет все вычисления прямо здесь и сейчас. В этом заключается суть императивного программирования. В императивном программировании мы выполняем все вычисления немедленно и возвращаем полностью готовые результаты. В декларативном программировании, напротив, мы пытаемся откладывать вычисления настолько долго, насколько это возможно.

    Let’s try to use our EnglishName class:

    final Name name = new EnglishName(
  new NameInPostgreSQL(/*...*/)
);
if (/* something goes wrong */) {
  throw new IllegalStateException(
    String.format(
      "Hi, %s, we can't proceed with your application",
      name.first()
    )
  );
}

    На первой строке этого отрывка мы просто создаем экземпляр объекта и называем его name. Мы пока не хотим обращаться к базе данных и извлекать полное имя оттуда, разделять его на части и инкапсулировать их внутри name. Нам просто нужно создать экземпляр объекта. Такое поведение разбора будет для нас побочным эффектом и, в данном случае, замедлит приложение. Как видите, нам может понадобиться только name.first(), если что-то идет не так и нам нужно создать объект исключения.

    Моя точка зрения заключается в том, что проводить любые вычисления внутри конструктора - это плохая практика, которую следует избегать, поскольку они являются побочными эффектами и не требуются владельцем объекта.

    Как насчет производительности при повторном использовании name, вы можете спросить. Если мы создадим экземпляр EnglishName и затем вызовем метод name.first() пять раз, мы получим пять вызовов метода String.split().

    To solve that, we create another class, a composable decorator, which will help us solve this “re-use” problem:

    public final class CachedName implements Name {
  private final Name origin;
  public CachedName(final Name name) {
    this.origin = name;
  }
  @Override
  @Cacheable(forever = true)
  public String first() {
    return this.origin.first();
  }
}

    Я использую аннотацию Cacheable из библиотеки jcabi-aspects, но вы также можете использовать любые другие инструменты кэширования, доступные в Java (или других языках), такие как Guava Cache.

    public final class CachedName implements Name {
  private final Cache<Long, String> cache =
    CacheBuilder.newBuilder().build();
  private final Name origin;
  public CachedName(final Name name) {
    this.origin = name;
  }
  @Override
  public String first() {
    return this.cache.get(
      1L,
      new Callable<String>() {
        @Override
        public String call() {
          return CachedName.this.origin.first();
        }
      }
    );
  }
}

    Пожалуйста, не делайте CachedName изменяемым и лениво загружаемым - это антипаттерн, о котором я уже говорил в Objects Should Be Immutable.

    This is how our code will look now:

    final Name name = new CachedName(
  new EnglishName(
    new NameInPostgreSQL(/*...*/)
  )
);

    It’s a very primitive example, but I hope you get the idea.

    В этом дизайне мы в основном разделяем объект на две части. Первая знает, как получить имя из английского имени. Вторая знает, как кэшировать результаты этого вычисления в памяти. И теперь это мое решение, как пользователь этих классов, как именно использовать их. Я решу, нужно ли мне кэширование или нет. В этом и состоит объектная композиция.

    Позвольте мне повторить, что единственное разрешенное выражение внутри конструктора - это присваивание. Если вам нужно поместить что-то еще туда, начните думать о рефакторинге - вашему классу определенно требуется переработка.

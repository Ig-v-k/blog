    The Hidden Dangers of Method Overloading

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Перегрузка методов является общей особенностью многих языков программирования, позволяющей классу иметь два или более метода с одним и тем же именем, но с разными параметрами. Согласно Microsoft, перегрузка методов является “одной из самых важных техник для улучшения удобства использования, производительности и читабельности повторно используемых библиотек.” Я не согласен. По моему мнению, перегрузка методов может привести к менее читабельному коду и большему количеству ошибок, потому что поддержка двух или более реализаций под одним и тем же именем приводит к скрытым семантикам, которые неизбежно приводят к недопониманиям и функциональным дефектам.

    Давайте начнем с примера на Java. Предположим, вы хотите разрешить добавление продукта в корзину покупок, имея при этом либо идентификатор продукта, либо объект Product. Если предоставлен только идентификатор, вы хотели бы, чтобы корзина покупок получила доступ к каталогу продуктов, нашла соответствующий объект Product и добавила его. В этом случае перегрузка методов может оказаться полезной (метод add() определен дважды с двумя разными сигнатурами и реализациями):

    class Cart {
  private final List<Product> products;
  void add(int id) {
    this.add(new Catalog().findById(id));
  }
  void add(Product p) {
    this.products.add(p);
  }
}
var c = new Cart();
c.add(new Product("book"));
c.add(42);

    Этот подход действительно удобен по нескольким причинам. Во-первых, метод add(int) обрабатывает преобразование из int в Product, что не требуется повторять в других местах—просто передайте идентификатор продукта в этот метод и позвольте ему сделать всю работу—тем самым устраняется дублирование кода. Во-вторых, так как функциональность “поиска в каталоге” не выходит за пределы класса Cart, это упрощает окружающий код. Кажется, что действительно улучшаются удобство использования и читаемость кода.

    Однако решаемые проблемы (дублирование кода и сложность) являются менее значимыми, чем проблемы, вносимые в процессе. В то время как семантика метода add(Product) очевидна, работа метода add(int) не ясна для его пользователей. Возможно, он осуществляет поиск в каталоге? Может быть, он выбирает n-й продукт из уже существующей корзины и добавляет его в конец корзины? Или, возможно, он осуществляет поиск в ранее размещенных пользователем заказах и извлекает n-й продукт оттуда? Мы просто не знаем, когда рассматриваем сигнатуру метода.

    Для того чтобы понять, что делает метод add(int), мы должны обратиться к его блоку Javadoc, который может быть недостаточно точным. Кроме того, как это часто бывает, документация в блоке Javadoc может быть несоответствующей коду внутри метода. Проще говоря, клиент, использующий метод add(int), неизбежно делает предположение о его внутренней работе. Если клиенту повезет, 1) предположение будет верным и 2) любые последующие изменения, внесенные в метод, не нарушат это предположение.

    Кажется, что перегрузка метода является корнем проблемы: поддержка нескольких реализаций под одним именем неизбежно вводит некоторые скрытые семантики в большинстве из них. Затем, когда что-то затуманивается, это вызывает предположения, которые в свою очередь приводят к недоразумениям и, в конечном итоге, к ошибкам.

    Я считаю, что лучшая альтернатива выглядит так:

    class ProductInCatalog implements Product {
  private final Product p;
  ProductInCatalog(int id) {
    this.p = new Catalog().findById(id);
  }
  // all "Product" interface's methods
}
class Cart {
  private final List<Product> products;
  void add(Product p) {
    this.products.add(p);
  }
}
var c = new Cart();
c.add(new Product("book"));
c.add(new ProductInCatalog(42));

    Конструктор ProductInCatalog не является свободным от кода, каким он должен был быть, однако это не так важно для нашего текущего обсуждения. Класс ProductInCatalog служит абстракцией Product, найденного в каталоге. Этот класс используется клиентом класса Cart. Клиент, полностью осознавая и с явным намерением, преобразует 42 (ID продукта) в экземпляр ProductInCatalog. Этот дизайн больше не скрывает никаких элементов. Не требуется делать никаких предположений и никаких условий, которые код должен выполнять.

    Мы до сих пор сохраняем преимущества, предоставляемые перегрузкой методов? Действительно, да. Нет дублирования кода, и сложность кода снижается. Кроме того, читаемость кода с использованием Cart значительно улучшается. В заключение, я рекомендую избегать перегрузки методов, хотя это несомненно приведет к большему количеству классов в кодовой базе. Однако, это уже совершенно другая дискуссия.

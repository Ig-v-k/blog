    DynamoDB + Rake + Maven + Rack::Test

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    В SixNines.io, одном из моих Ruby-приложений для веба, я использую DynamoDB, облачную базу данных NoSQL от AWS. Она работает отлично, но проблема в том, что создание интеграционного теста для проверки работы моего кода с “реальным” сервером и таблицами DynamoDB не так просто. Позвольте показать вам, как эта проблема была решена. Код является открытым и вы можете найти его в репозитории yegor256/sixnines на GitHub.

    How to bootstrap DynamoDB Local

    Сначала вам нужно использовать DynamoDB Local, инструмент командной строки, созданный AWS именно для целей тестирования. Вам нужно запустить его перед интеграционными тестами и остановить после.

    Чтобы упростить задачу, я предлагаю вам использовать jcabi-dynamodb-maven-plugin, плагин Maven, который я создал несколько лет назад. Вам потребуется добавить pom.xml в ваш репозиторий и запустить/остановить Maven из Rakefile, так же, как я делаю здесь:

    task :dynamo do
  FileUtils.rm_rf('dynamodb-local/target')
  pid = Process.spawn('mvn', 'install', chdir: 'dynamodb-local')
  at_exit do
    `kill -TERM #{pid}`
  end
  begin
    Dynamo.new.aws.describe_table(table_name: 'sn-endpoints')
  rescue Exception => e
    sleep(5)
    retry
  end
end

    Сначала я удаляю dynamodb-local/target - это каталог, в котором Maven хранит свои временные файлы, чтобы убедиться, что мы всегда начинаем с чистого листа.

    Затем я запускаю mvn install, используя Process.spawn, как фоновый процесс с pid в качестве его идентификатора процесса (это не будет работать в Windows, только в Linux/Mac). Затем я немедленно регистрирую at_exit хук Ruby, который будет выполнен, если Ruby завершается по любой причине. Я уверен, почему я должен это сделать - чтобы избежать запущенных фоновых процессов после завершения или прерывания Rake.

    Обратите внимание, я использую kill -TERM вместо kill -KILL, чтобы дать Maven шанс завершить все, правильно завершить работу DynamoDB Local, закрыть его TCP-порт и выйти.

    How to check that it’s running

    Далее я проверяю статус sn-endpoints, одной из таблиц в DynamoDB Local. Если сервер работает, она должна быть доступна. Она будет создана с помощью jcabi-dynamodb-maven-plugin в соответствии с sn-endpoints.json, его конфигурацией в формате JSON.

    Скорее всего, таблица не будет готова сразу, так как требуется время для запуска Maven, запуска сервера и создания таблицы. Поэтому, если происходит исключение, я его перехватываю, жду 5 секунд и пытаюсь снова. Я продолжаю пытаться несколько раз, пока сервер не будет готов. В конечном итоге он будет готов. На моем MacBook это занимает примерно 12-15 секунд, что означает 2-3 попытки/исключения.

    How to connect to DynamoDB Local

    Моим классам нужно знать, как подключаться к серверу во время интеграционных тестов. В продакшене им нужно подключаться к AWS, а при тестировании им нужно знать о только что запущенном экземпляре DynamoDB Local. Вот что есть у меня в классе Dynamo, который отвечает за само подключение к DynamoDB. Его решение о том, куда подключаться, основано на переменной окружения RACK_ENV, которая установлена на "test" в test__helper.rb, который подключается через rake/testtask перед всеми другими тестами, благодаря двойному подчеркиванию в его имени.

    Если переменная среды установлена на "test", Dynamo берет параметры подключения из YAML-файла dynamodb-local/target/dynamo.yml, созданного maven-resources-plugin:copy-resources. TCP-порт локальной базы данных DynamoDB будет здесь, а также ключ аутентификации и секрет DynamoDB.

    How to run the integration tests

    Это самая простая часть. Я просто использую свои объекты так, как предполагается в производстве, и они подключаются к локальной базе данных DynamoDB вместо AWS.

    Я использую Rack::Test для тестирования всего приложения посредством набора HTTP-запросов. Например, здесь я пытаюсь отобразить страницу учетной записи пользователя Джеффа. Ожидается, что код HTTP-ответа будет 200.

    require 'test/unit'
require 'rack/test'
class AppTest < Test::Unit::TestCase
  include Rack::Test::Methods
  def app
    Sinatra::Application
  end
  def test_user_account
    header('Cookie', 'sixnines=jeff')
    get('/a')
    assert_equal(200, last_response.status)
  end

    Теперь вы можете запустить весь тест из командной строки. Вы можете посмотреть, как Rultor выполняет его при выпуске новой версии: полный журнал. Также посмотрите, как это работает в Travis. В двух словах:

    • Вы вызываете rake в командной строке, и Rake запускается;

    • Rake пытается выполнить задачу default, которая зависит от задачи test;

    • Rake пытается запустить test, которая зависит от dynamo;

    • Rake внутри задачи test запускает mvn install в фоновом режиме с этим pom.xml;

    • Maven распаковывает пакет установки локальной версии DynamoDB;

    • Maven сохраняет произвольный TCP-порт и сохраняет его значение в ${dynamo.port};

    • Мавен сохраняет ${dynamo.port} и информацию о ключе/секретной паре здесь.

    • Maven starts DynamoDB Local, связывая его с зарезервированным TCP-портом;

    • Рейк ожидает доступности DynamoDB Local на зарезервированном порту;

    • Rake импортирует все тестовые классы, начиная с test__helper.rb;

    • Переменная среды RACK_ENV устанавливается на значение "test".

    • Rack::Test пытается отправить веб-страницу;

    • Dynamo загружает YAML-конфигурацию из dynamo.yml и подключается к DynamoDB Local.

    • Rake stops;

    • Руби завершает Мэйвен и останавливает DynamoDB Local.

    That’s it.

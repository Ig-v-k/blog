    Single Statement Unit Tests

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Множество статей и книг уже было написано о шаблонах и анти-шаблонах модульного тестирования. Я хочу добавить еще одну рекомендацию, которая, я считаю, может помочь нам сделать наши тесты, и наш код в производстве, более объектно-ориентированными. Вот она: метод тестирования должен содержать только одно утверждение (assert).

    Посмотрите на этот тестовый метод из RandomStreamTest из OpenJDK 8, созданный Брайаном Гоэтцом.

    @Test
public void testIntStream() {
  final long seed = System.currentTimeMillis();
  final Random r1 = new Random(seed);
  final int[] a = new int[SIZE];
  for (int i=0; i < SIZE; i++) {
    a[i] = r1.nextInt();
  }
  final Random r2 = new Random(seed);
  final int[] b = r2.ints().limit(SIZE).toArray();
  assertEquals(a, b);
}

    В этом методе есть две части: алгоритм и проверка. Алгоритм подготавливает два массива целых чисел, а проверка сравнивает их и выбрасывает исключение AssertionError, если они не равны.

    Я говорю, что первая часть, алгоритм, - это то, что мы должны попытаться избежать. Единственное, что нам нужно, это утверждение. Вот как я бы переработал этот метод тестирования:

    @Test
public void testIntStream() {
  final long seed = System.currentTimeMillis();
  assertEquals(
    new ArrayFromRandom(
      new Random(seed)
    ).toArray(SIZE),
    new Random(seed).ints().limit(SIZE).toArray()
  );
}
private static class ArrayFromRandom {
  private final Random random;
  ArrayFromRandom(Random r) {
    this.random = r;
  }
  int[] toArray(int s) {
    final int[] a = new int[s];
    for (int i=0; i < s; i++) {
      a[i] = this.random.nextInt();
    }
    return a;
  }
}

    Если бы у Java были имена, этот код выглядел бы еще более изящно:

    @Test
public void testIntStream() {
  assertEquals(
    new ArrayFromRandom(
      new Random(System.currentTimeMillis() as seed)
    ).toArray(SIZE),
    new Random(seed).ints().limit(SIZE).toArray()
  );
}

    Как видите, в этом методе есть только один “утверждение”: assertEquals().

    Hamcrest со своим assertThat() и его коллекцией базовых матчеров - это идеальный инструмент для того, чтобы сделать наши тестовые методы с одним оператором еще более связанными и удобочитаемыми.

    Если мы согласимся следовать этому принципу, то есть ряд практических преимуществ.

    • Повторное использование. Классы, которые нам придется создать для проверки утверждений теста, будут повторно используемыми в других методах и случаях тестирования. Точно так же, как в приведенном выше примере, ArrayFromRandom может быть использован где-то еще. Аналогично, матчеры Hamcrest могут и будут составлять библиотеку повторно используемых компонентов теста.

    • Краткость. Поскольку будет довольно сложно создать длинный тестовый метод, когда в нем есть только один assert, вы и ваши коллеги-программисты неизбежно будете писать более короткий и более читаемый код.

    • Читаемость. С помощью единственного assert всегда будет очевидно, каково назначение тестового метода. Он будет начинаться с объявления намерения, в то время как все остальные детали на более низком уровне будут сдвинуты вправо.

    • Непреобразуемость. Будет почти невозможно иметь сеттеры в производственном коде, если тестовые методы не предусматривают место для алгоритмического кода. Неизбежно вы создадите неизменяемые объекты, чтобы сделать их проверяемыми с помощью единственного assert.

    Наибольшая польза, которую мы получаем, когда этот принцип применяется к нашим тестам, заключается в том, что они становятся декларативными и объектно-ориентированными, а не алгоритмическими, императивными и процедурными.

    Streams vs. Decorators

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Streams API был введен в Java 8 вместе с лямбда-выражениями, всего лишь несколько лет назад. Я, как дисциплинированный поклонник Java, попробовал использовать эту новую возможность в нескольких своих проектах, например здесь и здесь. Мне это не очень понравилось, и я вернулся к хорошим старым декораторам. Более того, я создал Cactoos, библиотеку декораторов, чтобы заменить Guava, которая не так хороша во многих местах.

    Вот примитивный пример. Предположим, у нас есть набор измерений, поступающих из некоторого источника данных, все они являются числами от нуля до единицы:

    Iterable<Double> probes;

    Теперь нам нужно показать только первые 10 из них, игнорируя нули и единицы, и изменить их масштаб на (0..100). Звучит как простая задача, верно? Существует три способа сделать это: процедурный, объектно-ориентированный и способ с использованием Java 8. Давайте начнем с процедурного способа:

    int pos = 0;
for (Double probe : probes) {
  if (probe == 0.0d || probe == 1.0d) {
    continue;
  }
  if (++pos > 10) {
    break;
  }
  System.out.printf(
    "Probe #%d: %f", pos, probe * 100.0d
  );
}

    Почему это процедурный способ? Потому что он императивный. Почему он императивный? Потому что он процедурный. Шутка, конечно.

    Это необходимо, потому что мы указываем компьютеру, какие данные поместить куда и как их перебрать. Мы не объявляем результат, а строим его императивно. Это работает, но не масштабируется. Мы не можем взять часть этого алгоритма и применить его к другому случаю использования. Мы не можем легко изменить его, например, чтобы использовать числа из двух источников вместо одного и т. д. Это процедурно. Достаточно сказано. Не делайте это так.

    Теперь, Java 8 предоставляет нам Streams API, который предполагает функциональный способ сделать то же самое. Давайте попробуем использовать его.

    Сначала нам нужно создать экземпляр Stream, который Iterable не позволяет получить напрямую. Затем мы используем API потока для выполнения задачи.

    StreamSupport.stream(probes.spliterator(), false)
  .filter(p -> p == 0.0d || p == 1.0d)
  .limit(10L)
  .forEach(
    probe -> System.out.printf(
      "Probe #%d: %f", 0, probe * 100.0d
    )
  );

    Это будет работать, но для всех зондов будет выводиться Probe #0, потому что forEach() не работает с индексами. В Stream интерфейсе Java 8 (и Java 9 тоже) нет функции forEachWithIndex(). Вот обходной путь с использованием атомарного счетчика:

    AtomicInteger index = new AtomicInteger();
StreamSupport.stream(probes.spliterator(), false)
  .filter(probe -> probe != 0.0d && probe != 1.0d)
  .limit(10L)
  .forEach(
    probe -> System.out.printf(
      "Probe #%d: %f",
      index.getAndIncrement(),
      probe * 100.0d
    )
  );

    “Что с этим не так?” - можете спросить вы. Во-первых, посмотрите, как легко мы попали в беду, когда не нашли правильный метод в интерфейсе Stream. Мы сразу же отказались от “потоковой” парадигмы и вернулись к доброму старому процедурному глобальному переменному (счетчику). Во-вторых, мы не видим, что происходит внутри методов filter(), limit() и forEach(). Как именно они работают? В документации говорится, что этот подход является “декларативным” и каждый метод в интерфейсе Stream возвращает экземпляр некоторого класса. Какие это классы? Мы не имеем ни малейшего представления, просто глядя на этот код.

    Эти две проблемы связаны между собой. Самая большая проблема с этим потоковым API - это сам интерфейс Stream - он огромный. На момент написания здесь есть 43 метода. Сорок три, в одном интерфейсе! Это противоречит каждому принципу объектно-ориентированного программирования, начиная с SOLID и заканчивая более серьезными.

    Каким образом можно реализовать тот же алгоритм в объектно-ориентированном стиле? Вот как я бы сделал это с помощью Cactoos, который представляет собой просто набор простых классов на Java:

    new And(
  new Mapped<Double, Scalar<Boolean>>(
    new Limited<Double>(
      new Filtered<Double>(
        probes,
        probe -> probe != 0.0d && probe != 1.0d
      ),
      10
    ),
    probe -> () -> {
      System.out.printf(
        "Probe #%d: %f", 0, probe * 100.0d
      );
      return true;
    }
  )
).value();

    Давайте посмотрим, что здесь происходит. Сначала Filtered декорирует нашу итерируемую переменную probes для удаления определенных элементов из нее. Обратите внимание, что Filtered реализует Iterable. Затем Limited, также являющийся Iterable, выбирает только первые десять элементов. Затем Mapped преобразует каждую пробу в экземпляр Scalar<Boolean>, который выполняет печать строки.

    Наконец, экземпляр And проходит по списку «скаляров» и запрашивает у каждого из них возврат значения boolean. Он печатает строку и возвращает true. Поскольку это true, And делает следующую попытку с следующим скаляром. В конечном итоге, его метод value() возвращает true.

    Но подождите, здесь нет индексов. Давайте добавим их. Для этого мы просто используем другой класс, называемый AndWithIndex.

    new AndWithIndex(
  new Mapped<Double, Func<Integer, Boolean>>(
    new Limited<Double>(
      new Filtered<Double>(
        probes,
        probe -> probe != 0.0d && probe != 1.0d
      ),
      10
    ),
    probe -> index -> {
      System.out.printf(
        "Probe #%d: %f", index, probe * 100.0d
      );
      return true;
    }
  )
).value();

    Вместо Scalar<Boolean> мы теперь отображаем наши зонды на Func<Integer, Boolean> чтобы они могли принимать индекс.

    Красота такого подхода заключается в том, что все классы и интерфейсы являются небольшими, и поэтому они очень композиционны. Чтобы сделать итерируемый набор ограниченным, мы декорируем его с помощью Limited; чтобы сделать фильтрованным, мы декорируем его с помощью Filtered; чтобы сделать что-то еще, мы создаем новый декоратор и используем его. Мы не ограничены одним единственным интерфейсом, как Stream.

    В конечном итоге, декораторы - это объектно-ориентированный инструмент для изменения поведения коллекций, тогда как потоки - это нечто другое, для чего я даже не могу найти название.

    P.S. Кстати, таким же образом алгоритм можно реализовать с помощью Iterables из Guava.

    Iterable<Double> ready = Iterables.limit(
  Iterables.filter(
    probes,
    probe -> probe != 0.0d && probe != 1.0d
  ),
  10
);
int pos = 0;
for (Double probe : probes) {
  System.out.printf(
    "Probe #%d: %f", pos++, probe * 100.0d
  );
}

    Это странное сочетание объектно-ориентированного и функционального стилей.

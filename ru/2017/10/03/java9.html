    Java 9: The Good, The Bad, and Private Interface Methods

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Java 9 был выпущен несколько недель назад. Проверьте заметки о выпуске, они содержат много интересных функций. Однако, я думаю, что не все так хорошо, как описывают это Оракл и сторонники Java на первый взгляд. Я вижу три тенденции в мире Java, которые можно назвать хорошим, плохим и ужасным, соответственно. Давайте начнем с хорошей.

    The Platform

    Первый тренд - очевидное улучшение платформы, которая компилирует Java, упаковывает JAR-файлы и запускает байт-код. Она определенно становится лучше с каждым новым релизом Java. Вот список улучшений, которые сделала Java 9, и безусловно они очень полезны.

    • JSR 376: Модульная система, известная как Jigsaw

    • “JEP 222: jshell” should be translated to Russian as follows: “JEP 222: jshell”

    • JEP 238: Многоверсионные JAR-файлы

    • JEP 282: jlink -> JEP 282: jlink

    • “JEP 158: Unified logging” would be translated to Russian as follows: “JEP 158: Унифицированное ведение журнала”.

    Платформа явно становится более зрелой. Это хорошая тенденция.

    The JDK

    Вторая тенденция, которую я наблюдаю с Java 6, показывает, что JDK, который в основном представляет собой коллекцию классов и интерфейсов, разработанных и поддерживаемых Oracle, становится всё больше с каждым новым релизом. В Java 9 они добавили и расширили, помимо прочего, следующее:

    • JEP 221, 224 225, 261: Обработка Javadoc (расширенная)

    • “JEP 268: XML Catalogs (new)” - “JEP 268: XML-каталоги (новые)”

    • “JEP 262: TIFF image I/O (новое)”

    • “JEP 251: мультирезольюционные изображения (новые)”

    • “JEP 110: HTTP 2.0 клиент (новый)”

    • JEP 236: Парсер для Nashorn (расширенный)

    Конечно, некоторые функции должны быть реализованы в самой JDK, такие как поддержка Unicode (JEP 267), платформо-специфичные функции Desktop (JEP 272), подсказки Spin-Wait (JEP 285), компактные строки (JEP 254) и API процессов (JEP 102). Их реализация зависит от основной платформы и должна быть предоставлена вместе с JVM.

    Но что делает клиент HTTP 2.0 в JDK вместе с JAX-RS, JPA, JAX-WS, JDBC и многими другими вещами, которые, по моему мнению, должны быть максимально удалены от Oracle? Они не являются платформо-специфичными и могут быть намного лучше разработаны открытым сообществом в виде независимых пакетов. Собирать их под одной общей зонтичной маркой - это ошибка, на мой взгляд.

    Я считаю, что крупные корпорации только убивают рынок программного обеспечения, вместо того чтобы делать его лучше, из-за финансовых и политических мотивов, к которым они подвергают его. Точно так же происходит и с JDK. Благодаря монополии Oracle, он лишен гибкости и динамичности в развитии. Другими словами, мы застряли с тем, что Oracle и его большие друзья считают правильным.

    Таким образом, увеличение размера JDK - плохая тенденция. Вместо этого я считаю, что Oracle только выиграет, сделав его меньше, делегируя всё, что не связано с платформой, сообществу открытого программного обеспечения, поддерживая программистов каким-либо образом и поощряя открытые и эффективные процессы стандартизации на рынке.

    The Language

    Java был разработан Джеймсом Гослингом в Sun Microsystems в 1995 году как объектно-ориентированный язык. Было много опасений относительно этого утверждения об объектно-ориентированности, и я также не уверен, что Java более объектно-ориентированна, чем процедурна. Тем не менее, официально она является объектно-ориентированной.

    В Java было унаследовано множество процедурных особенностей от C/C++, начиная с ее первой версии, включая статические методы, NULL, реализационное наследование и т.д. Это не был идеальный объектно-ориентированный язык и, как я понимаю, он не собирался им становиться. Основная идея заключалась в создании чего-то, что можно было бы написать один раз и запустить везде. Однако язык был также большим достижением, не только JVM. Он был простым и привлекательным.

    Java 5 сделал серьезный шаг вперед в 2004 году и улучшил язык, добавив обобщения, цикл for-each, varargs и статический импорт. Однако были введены аннотации и перечисления, которые помогли языку отклониться от объектной парадигмы к чему-то совершенно другому и процедурному.

    Java 7 добавил try-with-resource в 2011 году, что было хорошим шагом, соответствующим парадигме ООП.

    Java 8 добавил лямбда-выражения в 2014 году, что стало замечательной функцией, но совершенно нерелевантной для ООП. Лямбда и API потоков превратили Java в смесь объектной, процедурной и функциональной парадигм. Методы по умолчанию также были добавлены в интерфейсы, что превратило типы в библиотеки кода. Типы в библиотеки! Если спросите меня, это еще хуже, чем наследование реализации.

    Теперь Java 9 внесла следующее “улучшение” в интерфейсы, позволяя им иметь приватные методы. Приватные статические методы в типах! Вы можете в это поверить? Какой будет следующий шаг? Атрибуты, вероятно, в Java 10.

    Теперь давайте рассмотрим, что было сделано с некоторыми основными классами в JDK, чтобы понять, в каком направлении движется язык. Возьмем только два примера.

    Методы-фабрики для коллекций (JEP 269). Вместо введения новых конструкторов и позволения нам делать это:

    List<Integer> list = new ArrayList<>(1, 2, 3);

    …in Java 9 they created more static factory methods and made us do this:

    List<Integer> list = List.of(1, 2, 3);

    “Меньше конструкторов, больше статических методов!” - кажется, это философия тех, кто представил этот JEP. Лишним говорить, что это полностью противоречит самой сути объектно-ориентированного программирования. Объекты должны создаваться конструкторами, а не статическими методами, независимо от того, что говорит Джошуа Блох. Статические методы делают момент использования оператора new невидимым для нас, и поэтому код становится гораздо менее поддерживаемым - мы просто не знаем точно, какой класс инстанциируется и какие реальные аргументы его конструктора.

    Кстати, с Cactoos вы можете сделать это правильным образом:

    List<Integer> list = new ListOf(1, 2, 3);

    This is OOP.

    Новые методы в InputStream. К уже избыточному классу InputStream было добавлено три новых метода: transferTo(), readNBytes() и readAllBytes(). Теперь, когда мы хотим скопировать поток ввода в поток вывода, мы должны использовать следующий подход:

    input.transferTo(output);

    Это одна из самых типичных ошибок, которую делают молодые программисты ООП: они делают свои интерфейсы большими. Просто потому что им нужно больше функциональности. Я полагаю, что принцип разделения интерфейса является частью знаменитого SOLID и существует уже много лет. Что с вами не так, Oracle? Какой будет следующий шаг? В Java 10 у нас также будут saveToFile() и printToConsole()? Как насчет emailToAFriend()?

    Вот как вы можете сделать то же самое с помощью утилитарного класса IOUtils из commons-io:

    IOUtils.copy(input, output);

    Это не идеально, но лучше. Самым объектно-ориентированным способом является использование объектов, а не утилитарных классов и статических методов. Так это работает в Cactoos:

    new LengthOf(new TeeInput(input, output)).length();

    This is OOP.

    SRP is a Hoax

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Принцип единственной ответственности, согласно “Чистому коду” Роберта Мартина, означает, что “класс должен иметь только одну причину для изменений”. Давайте попробуем разобраться в этом довольно неясном утверждении и посмотрим, как оно помогает нам проектировать лучшее объектно-ориентированное программное обеспечение. Если оно помогает.

    Я однажды упомянул SRP в своем посте о SOLID, говоря, что это действительно не помогает программистам понять хорошо известное понятие “высокая связность”, которое было введено Ларри Константином еще в 1974 году. Теперь давайте рассмотрим это на примере и проанализируем, как мы можем улучшить класс с учетом SRP и станет ли он более объектно-ориентированным.

    Давайте попробуем класс AwsOcket из jcabi-s3 (я упростил код):

    class AwsOcket {
  boolean exists() { /* ... */ }
  void read(final OutputStream output) { /* ... */ }
  void write(final InputStream input) { /* ... */ }
}

    Поправьте меня, если я не прав, но согласно SRP этот класс отвечает за слишком много функций: 1) проверка наличия объекта в AWS S3, 2) чтение его содержимого и 3) изменение его содержимого. Верно? Это плохой дизайн и его нужно изменить.

    Чтобы изменить это и сделать его ответственным только за одну вещь, мы должны ввести метод получения, который будет возвращать клиента AWS, а затем создать три новых класса: ExistenceChecker, ContentReader и ContentWriter. Они будут выполнять проверку, чтение и запись соответственно. Теперь, чтобы прочитать содержимое и вывести его на консоль, я в настоящее время делаю следующее:

    if (ocket.exists()) {
  ocket.read(System.out);
}

    Завтра, если я проведу рефакторинг класса, я буду делать следующее:

    if (new ExistenceChecker(ocket.aws()).exists()) {
  new ContentReader(ocket.aws()).read(System.out);
}

    Помимо того факта, что эти проверяющие, читатели и писатели на самом деле не являются классами, а всего лишь обладателями процедур, использование этого ocket превращается в кошмар. Мы уже не можем знать, что произойдет с ним, когда мы передадим его куда-то. Мы не можем, например, гарантировать, что содержимое, которое идет от него, будет расшифровано или декодировано на лету. Мы просто не можем его украсить. Он больше не является объектом, а обладателем клиента AWS, который используется некоторыми другими классами где-то.

    Да, теперь она отвечает только за одну вещь: инкапсулирует ссылку на клиента AWS. Это идеальный класс с точки зрения SRP. Но это уже не объект.

    То же самое произойдет с любым классом, если вы примените принцип SRP в полной мере: он станет хранилищем данных или других объектов с набором методов установки и получения значений для них. Возможно, с одним дополнительным методом помимо них.

    Моя точка зрения заключается в том, что SRP - неправильная идея.

    Создание небольших и связанных классов - хорошая идея, но считать их ответственными “за одну вещь” является вводящим в заблуждение упрощением концепции “высокой связности”. Это только делает их глупыми носителями чего-то другого, вместо того чтобы быть инкапсуляторами и декораторами более мелких сущностей для создания более крупных.

    В нашей борьбе за эту фальшивую идею SRP мы утрачиваем гораздо более важный принцип, который на самом деле касается истинного объектно-ориентированного программирования и мышления: инкапсуляции. Гораздо менее важно, сколько задач выполняет объект, чем насколько тщательно он защищает сущности, которые инкапсулирует. Монструозный объект с ста методами - это гораздо меньшая проблема, чем DTO с пятью парами геттеров и сеттеров! Потому что DTO разбрасывает проблему по всему коду, где мы даже не можем ее найти, в то время как монструозный объект всегда перед нами, и мы всегда можем его рефакторить на более мелкие части.

    Инкапсуляция идет в первую очередь, размер идет следующим, если вообще.

    Traits and Mixins Are Not OOP

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Сразу же скажу, что функции, о которых мы будем говорить здесь, — это настоящий яд, внедренный в объектно-ориентированное программирование теми, кому отчаянно необходима лоботомия, так же как предложил Дэвид Уэст в своей книге Object Thinking. У этих функций разные названия, но самые распространенные — трейты и миксины. Я серьезно не понимаю, как мы всё еще можем называть программирование объектно-ориентированным, когда у него есть эти функции.

    Сначала вкратце рассмотрим, как они работают. Давайте используем модули Ruby в качестве примера реализации. Предположим, что у нас есть класс Book:

    class Book
  def initialize(title)
    @title = title
  end
end

    Теперь мы хотим, чтобы класс Book использовал статический метод (процедуру), который делает что-то полезное. Мы можем либо определить его в утилитном классе, и позволить Book вызывать его.

    class TextUtils
  def self.caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  def print
    puts "My title is #{TextUtils.caps(@title)}"
  end
end

    Or we may make it even more “convenient” and extend our module in order to access its methods directly:

    module TextModule
  def caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  extend TextModule
  def print
    puts "My title is #{caps(@title)}"
  end
end

    Похоже, что это хорошо, если вы не понимаете разницу между объектно-ориентированным программированием и статическими методами. Кроме того, если мы на минуту забудем о чистоте ООП, то этот подход на самом деле выглядит менее читаемым для меня, хотя и имеет меньше символов; трудно понять, откуда берется метод caps(), когда он вызывается просто как #{caps(@title)}, а не как #{TextUtils.caps(@title)}. Не так ли?

    Миксины начинают лучше выполнять свою роль, когда мы их include. Мы можем комбинировать их, чтобы создать поведение класса, которое нам нужно. Давайте создадим два миксина. Первый будет называться PlainMixin и будет выводить название книги так, как оно есть, а второй будет называться CapsMixin и будет делать заглавными то, что уже было выведено.

    module CapsMixin
  def to_s
    super.to_s.split.map(&:capitalize).join(' ')
  end
end
module PlainMixin
  def to_s
    @title
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    Вызов Book без включенного миксина напечатает его заголовок так, как он есть. Как только мы добавим оператор include, поведение to_s будет переопределено, и метод print будет давать другой результат. Мы можем объединять миксины, чтобы получить необходимую функциональность. Например, мы можем добавить еще один миксин, который будет сокращать заголовок до 16 символов.

    module AbbrMixin
  def to_s
    super.to_s.gsub(/^(.{16,}?).*$/m,'\1...')
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include AbbrMixin, CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    Я уверен, вы уже понимаете, что у них обоих есть доступ к приватному атрибуту @title класса Book. Фактически, у них полный доступ ко всему в классе. Они буквально являются “кусками кода”, которые мы внедряем в класс, чтобы сделать его более мощным и сложным. Что не так с таким подходом?

    Это та же проблема, что и с annotations, DTOs, getters и utility classes - они разбивают объекты на части и помещают куски функциональности туда, где объекты не видят их.

    В случае использования миксинов, функциональность находится в модулях Ruby, которые делают предположения о внутренней структуре Book и дополнительно предполагают, что программист все еще будет понимать, что находится в Book после изменения внутренней структуры. Такие предположения полностью нарушают саму идею инкапсуляции.

    Такая плотная связь между миксинами и приватной структурой объекта приводит только к не поддерживаемому и сложному для понимания коду.

    The very obvious alternatives to mixins are composable decorators. Take a look at the example given in the article:

    Text text = new AllCapsText(
  new TrimmedText(
    new PrintableText(
      new TextInFile(new File("/tmp/a.txt"))
    )
  )
);

    Doesn’t it look very similar to what we were doing above with Ruby mixins?

    Однако, в отличие от миксинов, декораторы сохраняют объекты компактными и связными, добавляя дополнительную функциональность поверх них. Миксины делают наоборот - они делают объекты более сложными и, благодаря этому, менее читаемыми и поддерживаемыми.

    Я искренне верю, что они просто яд. Тот, кто их изобрел, далеко не понимал философию объектно-ориентированного дизайна.

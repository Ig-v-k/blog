    Traits and Mixins Are Not OOP

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Позвольте мне сказать сразу, что особенности, о которых мы будем говорить здесь, чистая яд, принесенная в объектно-ориентированное программирование теми, кто отчаянно нуждался в лоботомии, как предложил Дэвид Уэст в своей книге Object Thinking. У этих особенностей есть разные названия, но наиболее распространены traits и mixins. Я серьезно не могу понять, как мы все еще можем называть программирование объектно-ориентированным, когда в нем есть эти особенности.

    Сначала рассмотрим их работу вкратце. Давайте в качестве примера возьмем модули Ruby. Предположим, у нас есть класс Book:

    class Book
  def initialize(title)
    @title = title
  end
end

    Теперь мы хотим, чтобы класс Book использовал статический метод (процедуру), выполняющую полезные действия. Мы можем определить его в утилитарном классе и позволить Book вызывать его.

    class TextUtils
  def self.caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  def print
    puts "My title is #{TextUtils.caps(@title)}"
  end
end

    Или мы можем сделать его еще более “удобным” и расширить наш модуль, чтобы иметь прямой доступ к его методам.

    module TextModule
  def caps(text)
    text.split.map(&:capitalize).join(' ')
  end
end
class Book
  extend TextModule
  def print
    puts "My title is #{caps(@title)}"
  end
end

    Кажется, это неплохо - если вы не понимаете разницу между объектно-ориентированным программированием и статическими методами. Более того, если мы забудем о чистоте ООП на минуту, этот подход, на самом деле, кажется менее читаемым для меня, даже если в нем меньше символов; трудно понять, откуда берется метод caps(), когда он вызывается просто как #{caps(@title)}, а не как #{TextUtils.caps(@title)}. А вы не так считаете?

    Миксины начинают лучше выполнять свою роль, когда мы их включаем. Мы можем комбинировать их, чтобы создавать поведение класса, которое нам нужно. Давайте создадим два миксина. Первый будет называться PlainMixin и будет выводить название книги так, как оно есть, а второй будет называться CapsMixin и будет делать заглавными то, что уже было выведено.

    module CapsMixin
  def to_s
    super.to_s.split.map(&:capitalize).join(' ')
  end
end
module PlainMixin
  def to_s
    @title
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    Вызов Book без включенного миксина будет печатать его заголовок так, как он есть. После добавления оператора include, поведение to_s будет переопределено, и метод print будет давать другой результат. Мы можем объединять миксины, чтобы получить требуемую функциональность. Например, мы можем добавить еще один, который сократит заголовок до 16 символов.

    module AbbrMixin
  def to_s
    super.to_s.gsub(/^(.{16,}?).*$/m,'\1...')
  end
end
class Book
  def initialize(title)
    @title = title
  end
  include AbbrMixin, CapsMixin, PlainMixin
  def print
    puts "My title is #{self}"
  end
end

    Я уверен, вы уже понимаете, что обе они имеют доступ к приватному атрибуту @title класса Book. Фактически, они имеют полный доступ ко всему в классе. Они буквально являются “кусочками кода”, которые мы внедряем в класс, чтобы сделать его более мощным и сложным. Что не так с этим подходом?

    Это та же проблема, что и с аннотациями, DTO, геттерами и утилитарными классами - они разделяют объекты и помещают фрагменты функциональности в места, которые объекты не видят.

    В случае с mixin’ами, функциональность находится в модулях Ruby, которые делают предположения о внутренней структуре Book и дополнительно предполагают, что программист будет все еще понимать, что находится в Book после изменения внутренней структуры. Такие предположения полностью нарушают основную идею инкапсуляции.

    Такая тесная связь между миксинами и приватной структурой объектов ведет только к неподдерживаемому и трудночитаемому коду.

    Очевидными альтернативами к миксинам являются композиционные декораторы. Взгляните на приведенный в статье пример:

    Text text = new AllCapsText(
  new TrimmedText(
    new PrintableText(
      new TextInFile(new File("/tmp/a.txt"))
    )
  )
);

    Не похоже ли это очень похоже на то, что мы делали выше с помощью смешивания Ruby?

    Однако, в отличие от миксинов, декораторы делают объекты компактными и последовательными, добавляя дополнительную функциональность поверх них. Миксины, наоборот, делают объекты более сложными и, благодаря этому, менее читаемыми и поддерживаемыми.

    Я искренне считаю, что они просто яд. Тот, кто их изобрел, далеко не понимал философию объектно-ориентированного дизайна.

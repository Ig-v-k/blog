    Constructors or Static Factory Methods?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Я считаю, что Джошуа Блох сказал это первым в своей очень хорошей книге Effective Java: статические фабричные методы предпочтительнее для создания объектов по сравнению с конструкторами. Я не согласен. Не только потому, что я считаю статические методы чистым злом, но в основном потому, что в этом конкретном случае они притворяются хорошими и заставляют нас думать, что мы должны их любить.

    Давайте проанализируем логику и посмотрим, почему она неверна с точки зрения объектно-ориентированного подхода.

    Это класс с одним основным и двумя вторичными конструкторами:

    class Color {
  private final int hex;
  Color(String rgb) {
    this(Integer.parseInt(rgb, 16));
  }
  Color(int red, int green, int blue) {
    this(red << 16 + green << 8 + blue);
  }
  Color(int h) {
    this.hex = h;
  }
}

    Это аналогичный класс с тремя статическими фабричными методами:

    class Color {
  private final int hex;
  static Color makeFromRGB(String rgb) {
    return new Color(Integer.parseInt(rgb, 16));
  }
  static Color makeFromPalette(int red, int green, int blue) {
    return new Color(red << 16 + green << 8 + blue);
  }
  static Color makeFromHex(int h) {
    return new Color(h);
  }
  private Color(int h) {
    this.hex = h;
  }
}

    Какой из них вам нравится больше?

    Согласно Джошуа Блоху, использование статических фабричных методов вместо конструкторов имеет три основных преимущества (на самом деле их четыре, но четвертое не применимо к Java более).

    • They have names.

    • They can cache.

    • They can subtype.

    Я считаю, что все три имеют смысл … если дизайн неправильный. Они являются хорошими оправданиями для обходных путей. Рассмотрим их по очереди.

    They Have Names

    Вот как вы создаете объект цвета красного томата с помощью конструктора:

    Color tomato = new Color(255, 99, 71);

    Вот как это делается с помощью статического фабричного метода:

    Color tomato = Color.makeFromPalette(255, 99, 71);

    Кажется, что makeFromPalette() семантически богаче, чем просто new Color(), верно? Ну да. Кто знает, что означают эти три числа, если мы просто передаем их конструктору. Но слово “палитра” помогает нам сразу разобраться во всем.

    True.

    Однако правильным решением будет использование полиморфизма и инкапсуляции для разбиения проблемы на несколько семантически богатых классов.

    interface Color {
}
class HexColor implements Color {
  private final int hex;
  HexColor(int h) {
    this.hex = h;
  }
}
class RGBColor implements Color {
  private final Color origin;
  RGBColor(int red, int green, int blue) {
    this.origin = new HexColor(
      red << 16 + green << 8 + blue
    );
  }
}

    Теперь мы используем правильный конструктор правильного класса:

    Color tomato = new RGBColor(255, 99, 71);

    See, Joshua?

    They Can Cache

    Допустим, мне нужен красный цвет помидора в нескольких местах в приложении:

    Color tomato = new Color(255, 99, 71);
// ... sometime later
Color red = new Color(255, 99, 71);

    Будут созданы два объекта, что явно неэффективно, так как они идентичны. Было бы лучше сохранить первый экземпляр где-то в памяти и возвращать его при поступлении второго вызова. Статические фабричные методы позволяют решить эту очень проблему.

    Color tomato = Color.makeFromPalette(255, 99, 71);
// ... sometime later
Color red = Color.makeFromPalette(255, 99, 71);

    Затем где-то внутри Color мы храним приватную статическую Map со всеми уже созданными объектами.

    class Color {
  private static final Map<Integer, Color> CACHE =
    new HashMap<>();
  private final int hex;
  static Color makeFromPalette(int red, int green, int blue) {
    final int hex = red << 16 + green << 8 + blue;
    return Color.CACHE.computeIfAbsent(
      hex, h -> new Color(h)
    );
  }
  private Color(int h) {
    return new Color(h);
  }
}

    Это очень эффективно с точки зрения производительности. С небольшим объектом, таким как наш Color, проблема может не быть так очевидной, но когда объекты становятся больше, их создание и сборка мусора могут занимать много времени.

    True.

    Однако есть объектно-ориентированный способ решить эту проблему. Мы просто вводим новый класс Palette, который становится хранилищем цветов.

    class Palette {
  private final Map<Integer, Color> colors =
    new HashMap<>();
  Color take(int red, int green, int blue) {
    final int hex = red << 16 + green << 8 + blue;
    return this.computerIfAbsent(
      hex, h -> new Color(h)
    );
  }
}

    Теперь мы создаем экземпляр Palette один раз и просим его возвращать нам цвет каждый раз, когда нам это нужно:

    Color tomato = palette.take(255, 99, 71);
// Later we will get the same instance:
Color red = palette.take(255, 99, 71);

    Видишь, Джошуа, здесь нет статических методов, нет статических атрибутов.

    They Can Subtype

    Допустим, у нашего класса Color есть метод lighter(), который должен перевести цвет в следующий доступный светлый цвет.

    class Color {
  protected final int hex;
  Color(int h) {
    this.hex = h;
  }
  public Color lighter() {
    return new Color(hex + 0x111);
  }
}

    Однако иногда более предпочтительно выбирать следующий более светлый цвет из набора доступных цветов Pantone.

    class PantoneColor extends Color {
  private final PantoneName pantone;
  PantoneColor(String name) {
    this(new PantoneName(name));
  }
  PantoneColor(PantoneName name) {
    this.pantone = name;
  }
  @Override
  public Color lighter() {
    return new PantoneColor(this.pantone.up());
  }
}

    Затем мы создаем статический фабричный метод, который будет определять, какая реализация Color наиболее подходит для нас:

    class Color {
  private final String code;
  static Color make(int h) {
    if (h == 0xBF1932) {
      return new PantoneColor("19-1664 TPX");
    }
    return new RGBColor(h);
  }
}

    Если требуется настоящий красный цвет, мы возвращаем экземпляр PantoneColor. Во всех остальных случаях это просто стандартный RGBColor. Решение принимается статическим фабричным методом. Вот как мы его будем вызывать:

    Color color = Color.make(0xBF1932);

    Невозможно выполнить аналогичное “разветвление” с помощью конструктора, поскольку он может возвращать только класс, в котором он объявлен. Статический метод имеет всю необходимую свободу для возвращения любого подтипа Color.

    True.

    Однако в объектно-ориентированном мире мы можем и должны делать все по-другому. Во-первых, мы сделаем Color интерфейсом:

    interface Color {
  Color lighter();
}

    Затем мы перенесем этот процесс принятия решений в отдельный класс Colors, так же, как мы сделали в предыдущем примере.

    class Colors {
  Color make(int h) {
    if (h == 0xBF1932) {
      return new PantoneColor("19-1664-TPX");
    }
    return new RGBColor(h);
  }
}

    И мы будем использовать экземпляр класса Colors вместо статического фабричного метода внутри Color.

    colors.make(0xBF1932);

    Однако это все еще не совсем объектно-ориентированное мышление, потому что мы отнимаем принятие решения у объекта, к которому оно относится. Будь то статический фабричный метод make() или новый класс Colors - не имеет значения, как именно - мы разделяем наши объекты на две части. Первая часть - сам объект, а вторая - алгоритм принятия решений, который находится где-то еще.

    Гораздо более объектно-ориентированным решением было бы поместить логику в объект класса PantoneColor, который бы декорировал исходный RGBColor.

    class PantoneColor {
  private final Color origin;
  PantoneColor(Color color) {
    this.origin = color;
  }
  @Override
  public Color lighter() {
    final Color next;
    if (this.origin.hex() == 0xBF1932) {
      next = new RGBColor(0xD12631);
    } else {
      next = this.origin.lighter();
    }
    return new PantoneColor(next);
  }
)

    Затем мы создаем экземпляр RGBColor и украшаем его PantoneColor.

    Color red = new PantoneColor(
  new RGBColor(0xBF1932)
);

    Мы просим «красный» вернуть более светлый цвет, и он возвращает тот, который принадлежит палитре Pantone, а не тот, который просто светлее по координатам RGB.

    Color lighter = red.lighter(); // 0xD12631

    Конечно, этот пример довольно примитивен и требует дальнейшего усовершенствования, если мы действительно хотим, чтобы он был применим ко всем цветам Pantone, но я надеюсь, что вы понимаете идею. Логика должна оставаться внутри класса, не где-то снаружи, не в статических фабричных методах или даже в каком-то другом дополнительном классе. Речь идет о логике, которая принадлежит именно этому классу, конечно. Если это касается управления экземплярами класса, то могут быть контейнеры и хранилища, как в предыдущем примере выше.

    В заключение, я настоятельно рекомендую никогда не использовать статические методы, особенно когда они собираются заменить конструкторы объектов. Создание объекта через его конструктор - самый “священный” момент в любом объектно-ориентированном программном обеспечении, не упускайте красоты этого момента.

    Five Features to Make Java Even Better

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Я наткнулся на эту предложенную идею от Брайана Гоетца о классах данных в Java и сразу понял, что у меня тоже есть несколько идей о том, как сделать Java лучше в качестве языка. У меня, на самом деле, их много, но вот краткий список пяти самых важных.

    Глобальные переменные. В Java есть Одиночки, которые, как мы все знаем, являются ничем иным, как глобальными переменными. Не было бы замечательно включить глобальные переменные в Java и избавиться от Одиночек. PHP, JavaScript, Ruby и многие другие языки имеют их, почему Java нет? Посмотрите на этот код:

    class User {
  private static User INSTANCE;
  private User() {}
  public static User getInstance() {
    synchronized (User.INSTANCE) {
      if (User.INSTANCE == null) {
        User.INSTANCE = new User();
      }
    }
    return User.INSTANCE;
  }
  public String getName() {
    // return user's name
  }
}

    Затем, чтобы получить к нему доступ, нам нужно использовать:

    String name = User.getInstance().getName();

    Это Singleton. Заметьте, насколько многословен он? Мы можем просто заменить его глобальной переменной (global - это ключевое слово, которое я предлагаю использовать).

    global User user;

    And then:

    user.getName();

    Гораздо меньше кода нужно написать, и намного проще его читать!

    Global Functions and Namespaces

    Чтобы сгруппировать статические методы вместе, мы создаем утилитарные классы, где нам нужно определить приватные конструкторы, чтобы предотвратить их создание. Кроме того, нам нужно помнить, в каком конкретном утилитарном классе находится статический метод. Это просто дополнительные хлопоты. Я предлагаю добавить глобальные функции в Java и необязательные “пространства имен” для их группировки. Взгляните на этот утилитарный класс:

    class TextUtils {
  private TextUtils() {}
  public static String trim(String text) {
    if (text == null) {
      return "";
    }
    return text.trim();
  }
}

    Теперь посмотрите на эту глобальную функцию с пространством имён:

    namespace TextUtils {
  String trim(String text) {
    if (text == null) {
      return "";
    }
    return text.trim();
  }
}

    Моя точка зрения заключается в том, что поскольку мы уже используем классы как сборники функций, давайте сделаем это более удобным. В некоторых приложениях нам даже не понадобятся пространства имен, только глобальные функции, как в C и C++.

    Full Access to Private Attributes and Methods

    Для того чтобы получить доступ к приватному атрибуту или методу объекта извне, нам нужно использовать API отражения. Это не особенно сложно, но требует написания нескольких строк кода, которые не так легко прочитать и понять.

    class Point {
  private int x;
  private int y;
}
Point point = new Point();
Field field = point.getClass().getDeclaredField("x");
field.setAccessible(true);
int x = (int) field.get(point);

    Я предлагаю разрешить любому объекту получить доступ к любым атрибутам и методам другого объекта.

    Point point = new Point();
int x = point.x;

    Конечно, если они являются приватными, компилятор выдаст предупреждение. На этапе компиляции вы просто игнорируете предупреждение и продолжаете работу. Если вам действительно важна инкапсуляция, обратите внимание на предупреждение и сделайте что-то иное. Но в большинстве случаев программисты игнорируют его, так как они все равно с радостью используют Reflection API.

    NULL by Default

    Было бы удобно позволить нам вызывать конструкторы и методы с неполным набором аргументов. Аргументы, которые мы не предоставляем, будут по умолчанию установлены в null. Кроме того, когда метод должен что-то вернуть, но нет оператора return, Java должна возвращать null. Это почти точно также работает в PHP, Ruby и многих других языках. Я считаю, что это было бы удобной функцией и для разработчиков Java.

    Мы не будем нуждаться в определении такого большого количества методов, когда некоторые аргументы являются опциональными. Перегрузка методов очень громоздка и сложна для понимания. Вместо этого у нас должен быть один метод с длинным списком аргументов. Некоторые из них будут предоставлены вызывающей стороной, а другие будут установлены в null. Метод будет решать, что делать, например:

    void save(File file, String encoding) {
 if (encoding == null) {
   encoding = "UTF-8";
 }
}

    Затем мы просто вызываем либо save(f), либо save(f, "UTF-16"). Метод поймет, что мы имеем в виду. Мы также можем сделать это даже более удобным, как это сделано в Ruby, предоставляя аргументы метода по именам:

    save(file: f, encoding: "UTF-16");

    Также, когда нет ничего для возврата, метод должен по умолчанию возвращать null. Запись return null - это просто излишнее использование строки кода и на самом деле не улучшает читаемость. Посмотрите:

    String load(File file) {
 if (file.exists()) {
   return read_the_content();
 }
}

    Из этого кода очевидно, что если файл существует, метод загружает и возвращает его содержимое. Если нет, он возвращает null, что будет хорошим индикатором для вызывающей стороны, что что-то не так и содержимое файла недоступно.

    Getters and Setters

    Я думаю, что очевидно, что нам нужна эта функция: каждый частный атрибут должен автоматически иметь установщик и получатель. Нет необходимости создавать их, Java предоставит их из коробки, так же как Kotlin и Ruby. В чем смысл иметь атрибут, если нет методов получения и установки, верно?

    С помощью этой новой функции нам больше не понадобится помощь Lombok или IntelliJ IDEA.

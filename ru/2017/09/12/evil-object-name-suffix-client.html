    Yet Another Evil Suffix For Object Names: Client

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Некоторое время назад мы говорили о суффиксах “-ER” в названиях объектов и классов. Мы согласились, что они являются злом и должны быть избеганы, если мы хотим, чтобы наш код был по-настоящему объектно-ориентированным, а наши объекты были объектами, а не сборниками процедур. Теперь я готов представить новый зловещий суффикс: Client.

    Позвольте мне привести пример. Вот как может выглядеть объект с таким суффиксом (это псевдокодовая версия AmazonS3Client из AWS Java SDK):

    class AmazonS3Client {
  createBucket(String name);
  deleteBucket(String name);
  doesBucketExist(String name);
  getBucketAcl(String name)
  getBucketPolicy(String name);
  listBuckets();
  // 160+ more methods here
}

    Все “клиенты” выглядят похожими: они инкапсулируют URL-адрес назначения с некоторыми учетными данными и предоставляют некоторое количество методов, которые передают данные в/из “сервера”. Несмотря на то, что такой дизайн выглядит как правильный объект, на самом деле он не следует истинному духу объектно-ориентированного программирования. Вот почему он не настолько поддерживаем, каким должен быть, по двум причинам:

    Уровень его охвата слишком широкий. Поскольку клиент является абстракцией сервера, он неизбежно должен представлять весь функционал сервера. Когда функционал довольно ограничен, проблем нет. Возьмем HttpClient из Apache HttpComponents в качестве примера. Однако, когда сервер более сложный, размер клиента также увеличивается. В AmazonS3Client на момент написания этого текста имеется более 160 (!) методов, в то время как всего несколько десятков было всего несколько лет сотен версий назад.

    Он сфокусирован на данных. Сама идея клиент-серверного взаимодействия связана с передачей данных. Возьмем в качестве примера HTTP RESTful API сервиса AWS S3. На стороне AWS существуют объекты: ведра, объекты, версии, политики контроля доступа и т.д., и сервер преобразует их в данные JSON/XML. Затем данные приходят к нам, и клиент на нашей стороне работает с JSON или XML. Они остаются данными для нас и никогда не становятся ведрами, объектами или версиями.

    Последствия зависят от ситуации, но наиболее вероятными являются следующие:

    Процедурный код. Поскольку клиент возвращает данные, код, который работает с этими данными, скорее всего будет процедурным. Взгляните на результаты методов AWS SDK, они все выглядят как объекты, но на самом деле это просто структуры данных: S3Object, ObjectMetadata, BucketPolicy, PutObjectResult и т.д. Все они являются Data Transfer Objects только с методами получения и установки.

    Дублированный код. Если мы действительно решим оставаться объектно-ориентированными, нам придется преобразовать данные, которые возвращает клиент, в объекты. Скорее всего, это приведет к дублированию кода в нескольких проектах. У меня тоже была такая же ситуация, когда я начал работать с S3 SDK. Очень скоро я понял, что для избежания дублирования лучше создать библиотеку, которая выполняет задачу преобразования данных S3 SDK в объекты: jcabi-s3.

    Проблемы с тестированием. Поскольку клиент в большинстве случаев представляет собой достаточно большой класс/интерфейс, мокирование его в модульных тестах или создание его заменителей/подделок для тестирования является довольно сложной задачей.

    Проблемы со статикой. Классы клиентов, даже если их методы не являются статическими, очень похожи на утилитарные классы, которые хорошо известны своей анти-OOP-природой. Проблемы, с которыми мы сталкиваемся при использовании утилитарных классов, практически те же самые, что и с “клиентскими” классами.

    Проблемы расширяемости. Само собой разумеется, почти невозможно “украсить” объект клиента, когда у него уже есть 160+ методов и он продолжает расти. Единственный возможный способ добавления новой функциональности - создание новых методов. В результате мы получаем монструозный класс, который невозможно повторно использовать без модификации.

    Что предлагается в качестве альтернативы?

    Правильным решением было бы заменить “клиенты” на объекты, представляющие сущности серверной стороны, а не весь сервер. Например, с помощью SDK S3 это могут быть Bucket, Object, Version, Policy и т. д. Каждый из них предоставляет функциональность реальных ведер, объектов и версий, которую может предоставить AWS S3.

    Конечно, нам понадобится объект высокого уровня, который каким-то образом представляет весь API/сервер, но он должен быть маленьким. Например, в примере S3 SDK его можно назвать “Регион”, что означает весь регион AWS с ведрами. Затем мы можем получить ведро из него и нам больше не понадобится регион. Затем, чтобы перечислить объекты в ведре, мы просим ведро сделать это за нас. Нет необходимости каждый раз общаться со всем “объектом сервера”, хотя технически такая связь, конечно, происходит.

    В общем, проблема заключается не столько в суффиксе имени, сколько в самой идее представления всего сервера на стороне клиента вместо его сущностей. Такая абстракция 1) слишком громоздка и 2) очень зависит от данных.

    Кстати, ознакомьтесь с некоторыми библиотеками JCabi (Java) для примеров объектно-ориентированных клиентов без объектов “клиент”: jcabi-github, jcabi-dynamo, jcabi-s3 или jcabi-simpledb.

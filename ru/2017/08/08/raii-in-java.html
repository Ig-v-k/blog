    RAII in Java

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Resource Acquisition Is Initialization (RAII) - это идея проектирования, введенная в C++ Бьярне Страуструпом, для обеспечения безопасности исключений при управлении ресурсами. Благодаря сборщику мусора Java не имеет эту функцию, но мы можем реализовать что-то подобное с помощью try-with-resources.

    Проблему, которую решает RAII, можно понять, взглянув на следующий код (я уверен, что вы знаете, что такое Semaphore и как оно работает в Java):

    class Foo {
  private Semaphore sem = new Semaphore(5);
  void print(int x) throws Exception {
    this.sem.acquire();
    if (x > 1000) {
      throw new Exception("Too large!");
    }
    System.out.printf("x = %d", x);
    this.sem.release();
  }
}

    Код довольно примитивный и не делает ничего полезного, но вы, скорее всего, понимаете идею: метод print(), если вызывается из нескольких параллельных потоков, позволяет печатать только пять из них параллельно. Иногда он не позволяет некоторым из них печатать и генерирует исключение, если x больше чем 1000.

    Проблема с этим кодом заключается в утечке ресурсов. Каждый вызов print() с x больше чем 1000 будет занимать одно разрешение из семафора и не возвращать его. В пяти вызовах с исключениями семафор будет пустым, и все остальные потоки не будут ничего печатать.

    Какое решение? Вот оно:

    class Foo {
  private Semaphore sem = new Semaphore(5);
  void print(int x) throws Exception {
    this.sem.acquire();
    if (x > 1000) {
      this.sem.release();
      throw new Exception("Too large!");
    }
    System.out.printf("x = %d", x);
    this.sem.release();
  }
}

    Мы должны освободить разрешение перед тем, как вызвать исключение.

    Однако есть еще одна проблема, которая возникает: дублирование кода. Мы освобождаем разрешение в двух местах. Если мы добавим больше инструкций throw, нам также придется добавить больше вызовов sem.release().

    Очень элегантное решение было введено в C++ и называется RAII. Вот как это бы выглядело на Java:

    class Permit {
  private Semaphore sem;
  Permit(Semaphore s) {
    this.sem = s;
    this.sem.acquire();
  }
  @Override
  public void finalize() {
    this.sem.release();
  }
}
class Foo {
  private Semaphore sem = new Semaphore(5);
  void print(int x) throws Exception {
    new Permit(this.sem);
    if (x > 1000) {
      throw new Exception("Too large!");
    }
    System.out.printf("x = %d", x);
  }
}

    Посмотрите, как прекрасен код внутри метода Foo.print(). Мы просто создаем экземпляр класса Permit и он сразу получает новое разрешение на семафоре. Затем мы выходим из метода print(), либо из-за исключения, либо обычным путем, и метод Permit.finalize() освобождает разрешение.

    Элегантно, не так ли? Да, это так, но это не будет работать в Java.

    Это не будет работать, потому что, в отличие от C++, Java не уничтожает объекты, когда их область видимости закрывается. Объект класса Permit не будет уничтожен, когда мы выйдем из метода print(). Он будет уничтожен в конечном итоге, но мы не знаем когда именно. Скорее всего, он будет уничтожен после того, как все разрешения в семафоре будут получены и мы заблокируемся.

    В Java также есть решение. Оно не настолько элегантно, как в C++, но оно работает. Вот оно:

    class Permit implements Closeable {
  private Semaphore sem;
  Permit(Semaphore s) {
    this.sem = s;
  }
  @Override
  public void close() {
    this.sem.release();
  }
  public Permit acquire() {
    this.sem.acquire();
    return this;
  }
}
class Foo {
  private Semaphore sem = new Semaphore(5);
  void print(int x) throws Exception {
    try (Permit p = new Permit(this.sem).acquire()) {
      if (x > 1000) {
        throw new Exception("Too large!");
      }
      System.out.printf("x = %d", x);
    }
  }
}

    Обратите внимание на блок try и на интерфейс Closeable, который теперь реализует класс Permit. Объект p будет “закрыт” при выходе из блока try. Он может выйти либо в конце, либо с помощью операторов return или throw. В любом случае будет вызван метод Permit.close(): вот как работает try-with-resources в Java.

    Я добавил метод acquire() и переместил sem.acquire() из конструктора Permit, потому что я считаю, что конструкторы должны быть без кода.

    В заключение, RAII - отличный подход к проектированию, когда вы работаете с ресурсами, которые могут утечь. Несмотря на то, что Java не имеет его изначально, мы можем реализовать его с помощью try-with-resources и Closeable.

    Synchronized Decorators to Replace Thread-Safe Classes

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Вы знаете, что такое потокобезопасность, верно? Если нет, приведен ниже простой пример. Все классы должны быть потокобезопасными, верно? Не совсем. Некоторые из них должны быть потокобезопасными? Снова неправильно. Я думаю, что ни один из них не должен быть потокобезопасным, в то время как все они должны предоставлять синхронизированные декораторы.

    Давайте начнем с примера (который, кстати, изменяемый):

    class Position {
  private int number = 0;
  @Override
  public void increment() {
    int before = this.number;
    int after = before + 1;
    this.number = after;
  }
}

    Что вы думаете - это thread-safe? Этот термин относится к тому, будет ли объект этого класса работать без ошибок при использовании несколькими потоками одновременно. Допустим, у нас есть два потока, работающих с одним и тем же объектом, position, и вызывающих его метод increment() в один и тот же момент времени.

    Мы ожидаем, что целое число number будет равно 2, когда оба потока завершатся, потому что каждый из них увеличит его один раз, верно? Однако, скорее всего, это не произойдет.

    Посмотрим, что произойдет. В обоих потоках переменная before будет равна 0 при их запуске. Затем переменная after будет установлена в 1. Затем оба потока установят this.number = 1, и в результате в переменной number у нас будет неожиданное значение 1, а не ожидаемое значение 2. Понятна проблема? Классы с таким недостатком в своем дизайне не являются потокобезопасными.

    Самым простым и очевидным решением будет сделать наш метод synchronized. Это гарантирует, что независимо от количества потоков, вызывающих его одновременно, они все будут выполняться последовательно, а не параллельно: один поток за другим. Конечно, это займет больше времени, но предотвратит возникновение этой ошибки.

    class Position {
  private int number = 0;
  @Override
  public synchronized void increment() {
    int before = this.number;
    int after = before + 1;
    this.number = after;
  }
}

    Класс, который гарантирует, что он не сломается, независимо от количества потоков, работающих с ним, называется thread-safe.

    Теперь вопрос: нужно ли делать все классы потокобезопасными или только некоторые из них? Кажется, лучше иметь все классы без ошибок, верно? Почему бы кому-то понадобился объект, который может сломаться в какой-то момент? Ну, не совсем так. Помните, здесь есть проблема производительности; у нас не всегда есть несколько потоков, и мы всегда хотим, чтобы наши объекты работали максимально быстро. Механизм синхронизации между потоками безусловно замедлит нас.

    Я считаю, что правильным подходом будет иметь два класса. Первый класс не является потокобезопасным, в то время как второй класс - это декоратор с синхронизацией, который будет выглядеть так:

    class SyncPosition implements Position {
  private final Position origin;
  SyncPosition(Position pos) {
    this.origin = pos;
  }
  @Override
  public synchronized void increment() {
    this.origin.increment();
  }
}

    Теперь, когда нам нужно, чтобы наш объект position был потокобезопасным, мы добавляем к нему декоратор SyncPosition.

    Position position = new SyncPosition(
  new SimplePosition()
);

    Когда нам нужна обычная простая позиция без какой-либо защиты от потоков, мы делаем следующее:

    Position position = new SimplePosition();

    Создание функциональности класса, которая одновременно обширна и безопасна для работы с потоками, нарушает, по моему мнению, известный принцип единственной ответственности.

    Кстати, эта проблема очень близка к проблеме защитного программирования и валидаторов.

    How Much Cohesion Is Enough?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Какой из вариантов лучше: books.del(42) или books.book(42).del()? Я использую оба и редко могу определить, какой из них лучше. Первый вариант короче, в то время как второй более объектно-ориентированный. Первый вариант сложнее расширить, в то время как второй более многословный и требует больше строк кода, что увеличивает вероятность ошибок. Какой из них вы предпочитаете?

    Конечно, подойдет любой вариант, но вопрос в том, какой дизайн более объектно-ориентированный? Кажется, это зависит от размера объекта books. Если он небольшой, нет необходимости сначала получать книгу, мы можем просто удалить ее прямо там:

    interface Books {
  void create(String title);
  void delete(int id);
  void rename(int id, String title);
}

    Однако, если он(а) больше, лучше сначала взять “book”.

    interface Book {
  void delete(); // Here!
  void rename(String title);
}
interface Books {
  void create(String title);
  int total();
  Iterable<Book> all();
  Book all(String query);
  Book book(int id); // Here!
}

    Есть ли явная жесткая граница между ними? Есть ли строгое правило или, возможно, неоднозначный вопрос “Достаточно ли этот класс уже большим?” всегда должен решаться путем голосования?

    Давайте попробуем дать два крайних ответа: 1) никогда не достаточно большим, и 2) всегда достаточно большим.

    • Если класс никогда не готов к извлечению его частей и превращению их в новые объекты, у нас получится очень большой класс с множеством атрибутов, методов и длинными списками аргументов в почти каждом из них.

    • Если класс всегда достаточно хорош для извлечения, мы будем иметь много маленьких классов, методы с почти отсутствующими аргументами и … лучшую объектно-ориентированную архитектуру.

    Общий знаменатель - связность.

    Высокосвязные классы включают атрибуты и методы, которые связаны между собой, в то время как низкосвязные классы включают все, что разработчики решили добавить, даже если некоторые элементы на самом деле не связаны. Первый вариант даст нам класс с очень низкой связностью, тогда как второй вариант приведет к большому количеству маленьких классов с высокой связностью.

    Таким образом, второй вариант лучше? Да, это так. Меньшие классы, более высокая связность, … но больше возможностей потерять фокус и разместить функциональность во многих местах. “Все методы в одном объекте” - это более популярный дизайн, даже если он менее связан, именно потому что его легче создать: просто поместите все в одно место и закончите. Позже, конечно, появятся проблемы с поддержкой.

    Главное заключается в том, что в этом случае нет четкого различия между правильным и неправильным дизайном. Мы просто должны сделать все возможное, чтобы сохранить классы с высокой связностью, уменьшая количество методов в каждом из них. Если есть всего несколько методов, нет необходимости извлекать Book, но как только количество методов становится больше, Book становится отличным кандидатом для новой сущности, которую нужно определить.

    Сколько методов нормально?

    Никто не знает, но я бы предложил вам задать вопрос о связности вашего класса, как только вы увидите более семи методов или более четырех атрибутов. Также начните думать о рефакторинге, когда любой из ваших методов (кроме конструкторов) принимает более двух аргументов.

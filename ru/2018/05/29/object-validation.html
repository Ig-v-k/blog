    Object Validation: to Defer or Not?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Я ранее сказал, что конструкторы должны быть свободны от кода и ничего не делать, кроме инициализации атрибутов. С тех пор самым часто задаваемым вопросом стал: А что насчет проверки аргументов? Если они “неверные”, то в чем смысл создавать объект в “недопустимом” состоянии? Такой объект в конечном итоге приведет к ошибке в неожиданный момент. Разве не лучше сразу же выбросить исключение при создании объекта? Чтобы сразу же завершить выполнение, так сказать? Вот что я думаю.

    Давайте начнем с этого кода на Ruby:

    class Users {
  def initialize(file)
    @file = file
  end
  def names
    File.readlines(@file).reject(&:empty?)
  end
}

    Мы можем использовать его для чтения списка пользователей из файла.

    Users.new('all-users.txt').names

    Существует несколько способов злоупотребления этим классом:

    • Передайте nil конструктору вместо имени файла;

    • Передайте что-то другое, что не является String;

    • Передайте файл, который не существует;

    • Передайте каталог вместо файла.

    Видите ли вы разницу между этими четырьмя ошибками, которые мы можем сделать? Давайте посмотрим, как наш класс может защититься от каждой из них:

    class Users {
  def initialize(file)
    raise "File name can't be nil" if file.nil?
    raise 'Name must be a String' unless file.is_a?(String)
    @file = file
  end
  def names
    raise "#{@file} is absent" unless File.exist?(@file)
    raise "#{@file} is not a file" unless File.file?(@file)
    File.readlines(@file).reject(&:empty?)
  end
}

    Первые две возможные ошибки были отфильтрованы в конструкторе, в то время как другие две - позже, в методе. Почему я сделал это таким образом? Почему бы не поместить все из них в конструктор?

    Поскольку первые два компрометируют состояние объекта, в то время как последние два - его поведение во время выполнения. Вы помните, что объект является представителем набора других объектов, которые он инкапсулирует, называемых атрибутами. Объект класса Users не может представлять nil или число. Он может представлять только файл с именем типа String. С другой стороны, то, что содержится в этом файле и является ли он действительно файлом, не делает состояние недействительным. Это только вызывает проблемы для поведения.

    Несмотря на то, что разница может показаться незначительной, она очевидна. Существуют две фазы взаимодействия с инкапсулированным объектом: подключение и разговор.

    Сначала мы инкапсулируем file и хотим быть уверены, что это действительно файл. Мы еще не общаемся с ним, не хотим, чтобы он работал для нас, мы просто хотим убедиться, что это действительно объект, с которым мы сможем общаться в ближайшем будущем. Если это nil или float, у нас возникнут проблемы в будущем, наверняка. Вот почему мы вызываем исключение из конструктора.

    Затем наступает вторая фаза - общение, где мы передаем управление объекту и ожидаем, что он будет вести себя правильно. На этой фазе у нас может быть иные процедуры проверки, чтобы убедиться, что наше взаимодействие пройдет гладко. Следует отметить, что эти проверки очень ситуативны. Мы можем вызывать names() несколько раз, и каждый раз столкнуться с различной ситуацией с файлом на диске. Вначале его может и не существовать, а через несколько секунд он будет готов и доступен для чтения.

    Идеальным образом, язык программирования должен предоставлять инструменты для первого типа проверок, например с помощью строгой типизации. В Java, например, нам не нужно было бы проверять тип file, компилятор поймал бы эту ошибку ранее. В Kotlin мы могли бы избавиться от проверки NULL благодаря их возможности Null Safety. Ruby менее мощный, чем эти языки, поэтому нам приходится проверять “вручную”.

    Итак, чтобы подытожить, проверка в конструкторах - не плохая идея, при условии, что проверки не затрагивают объекты, а только подтверждают, что они достаточно хороши для последующей работы.

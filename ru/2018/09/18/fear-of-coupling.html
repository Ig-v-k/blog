    Fear of Decoupling

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Объекты общаются друг с другом через свои методы. В основных языках программирования, таких как Java или C#, объект может иметь уникальный набор методов вместе с некоторыми методами, которые ему приходится иметь из-за реализации определенных типов, также известных как интерфейсы. Мой опыт общения с многими программистами говорит мне, что большинство из нас довольно пугаются объектов, которые реализуют слишком много методов интерфейсов. Мы не хотим иметь с ними дело, так как они полиморфны и, из-за этого, ненадежны. Это справедливый страх. Давайте попробуем проанализировать, откуда он берется.

    Как обычно, давайте начнем с простого примера на Java. Вот сумма денег, которую я собираюсь отправить пользователю через, скажем, API PayPal:

    interface Money {
  double cents();
}

    Вот теперь я, метод, который отправляет деньги:

    void send(Money m) {
  double c = m.cents();
  // Send them over via the API...
}

    Эти два фрагмента кода, как мы говорим, слабо связаны. Метод send() не имеет представления о том, какой класс предоставлен и как именно реализован метод cents(). Может быть, это простой объект-константа в один доллар.

    class OneDollar implements Money {
  @Override
  double cents() {
    return 100.0d;
  }
}

    Или, может быть, это намного более сложная сущность, которая сначала устанавливает сетевое соединение, чтобы получить текущий курс обмена доллара США на евро, обновить базу данных, а затем возвращать результат некоторого вычисления.

    class EmployeeHourlyRate implements Money {
  @Override
  double cents() {
    // Fetch the exchange rate;
    // Update the database;
    // Calculate the hourly rate;
    // Return the value.
  }
}

    Метод send() не знает, что именно передается в качестве его первого аргумента. Все, что он может сделать, это надеяться, что метод cents() выполнит свою работу правильно. Что, если это не произойдет?

    Если я разработчик метода send() и полностью готов взять на себя ответственность за ошибки, которые может вызвать мой метод, я хочу знать, кто мои соавторы. И я хочу быть абсолютно уверенным в их работе. Не просто работе, а работе точно так, как я ожидаю. В идеале я бы хотел написать их сам. Идеально было бы гарантировать, что никто не трогает их после того, как я их реализую. Ты понимаешь сарказм, верно?

    Это может звучать как шутка, но я слышал это утверждение много раз. Говорят, что “лучше быть полностью уверенным, что две части взаимодействуют друг с другом, вместо того, чтобы полагаться на чертову полиморфность и тратить часы на отладку чего-то, что я не написал”. И они правы, знаете ли. Полиморфизм - когда кажущийся примитивный объект типа Money делает все, что ему вздумается, включая HTTP-запросы и SQL-запросы UPDATE - не добавляет надежности всему приложению, не так ли?

    No, it doesn’t.

    Очевидно, полиморфизм значительно облегчает жизнь разработчиков этого типа Money и его “предков”, поскольку им не нужно особо задумываться о пользователях. Им важно только вернуть double, когда вызывается cents(). Они не должны беспокоиться о скорости, возможных исключениях, использовании памяти и многих других вещах, поскольку интерфейс этого не требует. Он просто говорит им вернуть double и закончить дело. Пусть другие заботятся обо всем остальном. Легко, верно? Но, как вы можете сказать, это детский и эгоистичный способ мышления!

    Yes, it is.

    However…

    Вы, безусловно, слышали о принципе “Fail Fast”, который вкратце гласит, что для создания надежного и стабильного приложения необходимо сделать его компоненты настолько хрупкими и уязвимыми, насколько это возможно в ответ на любую возможную исключительную ситуацию. Они должны ломаться при любой возможности и позволять пользователям справляться с ошибками. С такой философией ни один объект не будет предполагать что-либо хорошее о своих коллегах и всегда будет стремиться эскалировать проблемы на более высокие уровни, что в конечном итоге затронет конечного пользователя, который сообщит об этом команде. Команда исправит все ошибки, и весь продукт стабилизируется.

    Если философия противоположна и каждый объект пытается решать проблемы на своем индивидуальном микроуровне, большинство исключительных ситуаций никогда не будут видны пользователям, тестировщикам, архитекторам и программистам, которые должны заниматься ими и находить для них решения. Благодаря этому “тщательному” подходу индивидуальных объектов, стабильность и надежность всего приложения будут страдать.

    Мы можем применить ту же логику к “страху перед свободной связью”.

    Когда мы беспокоимся о том, как работает Money.cents() и хотим контролировать его поведение, мы делаем себе и всему проекту большую неудобство. В итоге мы подрываем стабильность продукта, вместо того чтобы сделать его более устойчивым. Некоторые даже хотят запретить полиморфизм, объявив метод send() таким образом:

    void send(EmployeeHourlyRate m) {
  // Now I know that it's not some abstract Money,
  // but a very specific class EmployeeHourlyRate, which
  // was implemented by Bobby, a good friend of mine.
}

    Здесь мы ограничиваем количество ошибок, которые могут быть в нашем коде, так как мы знакомы с Бобби, видели его код, знаем, как он работает и каких исключений ожидать. Мы в безопасности. Да, мы в безопасности. По крайней мере, пока. Но стратегически говоря, не позволяя нашему программному обеспечению делать все возможные ошибки и выбрасывать все возможные исключения во всех необычных ситуациях, мы серьезно ограничиваем его способность к правильному тестированию, и поэтому оно нестабильно.

    Как я уже упоминал ранее, единственный способ повысить качество программного обеспечения - это находить и исправлять его ошибки. Чем больше ошибок мы исправляем, тем меньше остается скрытых и еще не исправленных. Боязнь ошибок и наше намерение предотвращать их лишь наносят нам ущерб.

    Вместо этого, мы должны позволить всем, а не только Бобби, реализовывать Money и передавать эти реализации в send(). Да, некоторые из них могут вызывать проблемы и даже приводить к видимым для пользователя ошибкам. Но если наше руководство правильно понимает концепцию качества программного обеспечения, они не будут винить нас за ошибки. Вместо этого, они будут поддерживать нас в поиске как можно большего числа ошибок, воспроизведении их с помощью автоматических тестов, исправлении и повторном развертывании.

    Таким образом, страх открепления - это ничто иное, как Fail Safe.

    Don't Parse, Use Parsing Objects

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Традиционным способом интеграции объектно-ориентированного бэк-энда с внешней системой является использование объектов передачи данных (data transfer objects), которые сериализуются в JSON перед отправкой и десериализуются при возвращении. Этот подход популярен, но неправильный. Часть сериализации должна быть заменена принтерами, о которых я рассказывал ранее. Вот мое предложение по десериализации, которая должна выполняться - угадайте, - объектами.

    Предположим, есть точка входа бэк-энда, которая должна зарегистрировать новую книгу в библиотеке, приходящую в формате JSON:

    {
  "title": "Object Thinking",
  "isbn: "0735619654",
  "author: "David West"
}

    Также есть объект класса Library, который ожидает, что ему будет передан объект типа Book в его метод register().

    class Library {
  public void register(Book book) {
    // Create a new record in the database
  }
}

    Скажите также, тип Book имеет простой метод isbn():

    interface Book {
  String isbn();
}

    Теперь вот точка входа HTTP (я использую Takes и Cactoos), которая принимает POST-запрос multipart/form-data и регистрирует книгу в библиотеке:

    public class TkUpload implements Take {
  private final Library library;
  @Override
  public Response act(Request req) {
    String body = new RqPrint(
      new RqMtSmart(new RqMtBase(req)).single("book")
    ).printBody();
    JsonObject json = Json.createReader(
      new InputStreamOf(body)
    ).readObject();
    Book book = new BookDTO();
    book.setIsbn(json.getString("isbn"));
    library.register(book);
  }
}

    Что не так с этим? Ну, несколько вещей.

    Во-первых, он не является повторно используемым. Если нам потребуется что-то подобное в другом месте, мы должны будем написать эту обработку HTTP и разбор JSON снова.

    Во-вторых, обработка ошибок и проверка также не являются повторно используемыми. Если мы добавим это к методу выше, мы должны будем скопировать это везде. Конечно, DTO может это инкапсулировать, но обычно это не то, для чего используются DTO.

    В-третьих, код выше является достаточно процедурным и имеет много временной связности.

    Лучшим решением было бы скрыть этот разбор внутри нового класса JsonBook:

    class JsonBook implements Book {
  private final String json;
  JsonBook(String body) {
    this.json = body;
  }
  @Override
  public String isbn() {
    return Json.createReader(
      new InputStreamOf(body)
    ).readObject().getString("isbn");
  }
}

    Затем RESTful точка входа будет выглядеть следующим образом:

    public class TkUpload implements Take {
  private final Library library;
  @Override
  public Response act(Request req) {
    library.register(
      new JsonBook(
        new RqPrint(
          new RqMtSmart(
            new RqMtBase(req)
          ).single("book")
        ).printBody()
      )
    );
  }
}

    Не правда ли, это более изящно?

    Вот несколько примеров из моих проектов: RqUser из zerocracy/farm и RqUser из yegor256/jare.

    Как видно из приведенных выше примеров, иногда мы не можем использовать implements, потому что некоторые примитивы в Java не являются интерфейсами, а являются final классами: String - “идеальный” пример. Вот почему мне приходится делать так:

    class RqUser implements Scalar<String> {
  @Override
  public String value() {
    // Parsing happens here and returns String
  }
}

    Но помимо этого, эти примеры отлично демонстрируют принцип “разбора объектов”, предложенный выше.

    Typical Mistakes in Java Code

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    На этой странице содержатся наиболее типичные ошибки, которые я замечаю в Java-коде людей, работающих со мной. Статический анализ (мы используем qulice) не может выявить все ошибки по очевидным причинам, и поэтому я решил перечислить их все здесь.

    Дайте мне знать, если вы хотите увидеть что-то еще добавленное здесь, и я с радостью выполню вашу просьбу.

    Все перечисленные ошибки связаны с объектно-ориентированным программированием в общем и с Java в частности.

    Class Names

    Ваш класс должен быть абстракцией реальной сущности без “валидаторов”, “контроллеров”, “менеджеров” и т. д. Если имя вашего класса заканчивается на “-er” - это плохой дизайн. Кстати, вот мои семь добродетелей хорошего объекта. Кроме того, в этом посте более подробно объясняется эта идея: Не создавайте объекты, оканчивающиеся на -ER.

    И, конечно же, утилитарные классы являются антипаттернами, такими как StringUtils, FileUtils и IOUtils из Apache. Вышеперечисленные примеры являются идеальными примерами плохого дизайна. Прочитайте этот последующий пост: ООП-альтернатива утилитарным классам.

    Конечно же, никогда не добавляйте суффиксы или префиксы для различия между интерфейсами и классами. Например, все эти имена являются ужасно неправильными: IRecord, IfaceEmployee или RecordInterface. Обычно имя интерфейса является именем реальной сущности, в то время как имя класса должно объяснять детали его реализации. Если нет ничего конкретного для сказания о реализации, назовите его Default, Simple или что-то подобное. Например:

    class SimpleUser implements User {};
class DefaultRecord implements Record {};
class Suffixed implements Name {};
class Validated implements Content {};

    Method Names

    Методы могут либо возвращать значение, либо возвращать void. Если метод возвращает значение, то его имя должно объяснять что он возвращает, например (никогда не используйте префикс get):

    boolean isValid(String name);
String content();
int ageOf(File file);

    Если он возвращает void, то его имя должно объяснять что он делает. Например:

    void save(File file);
void process(Work work);
void append(File file, String line);

    Вы можете узнать больше об этой идее в книге Elegant Objects, разделе 2.4. Есть только одно исключение из только что упомянутого правила - методы тестирования для JUnit. Они объясняются ниже.

    Test Method Names

    Имена методов в тестах JUnit должны быть созданы как английские предложения без пробелов. Легче объяснить на примере:

    /**

* HttpRequest can return its content in Unicode.

* @throws Exception If test fails */ @Test public void returnsItsContentInUnicode() throws Exception { } ```







Важно начинать первое предложение вашей Javadoc с названия класса, который вы тестируете, с последующим `может` (или `не может`). Таким образом, ваше первое предложение должно всегда быть похожим на "кто-то *может* что-то делать".

Название метода будет точно таким же, но без подлежащего. Если я добавлю подлежащее в начало названия метода, я должен получить полное английское предложение, как в примере выше: "`HttpRequest` возвращает свое содержимое в формате Unicode."

Обратите внимание, что тестовый метод не начинается с `может`. Только комментарии Javadoc используют 'может'.

Хорошей практикой является всегда объявлять тестовые методы с возможным выбрасыванием `исключения`.

## Variable Names

Избегайте составных имен переменных, таких как `timeOfDay`, `firstItem` или `httpRequest`. Я имею в виду и классовые переменные, и переменные внутри методов. Имя переменной должно быть достаточно длинным, чтобы избежать неоднозначности в ее области видимости, но при этом не слишком длинным, если это возможно. Имя должно быть существительным в единственном или множественном числе, или соответствующим сокращением. Больше об этом можно узнать в этой публикации: Составное имя - это запах кода. Например:

    List names; void sendThroughProxy(File file, Protocol proto); private File content; public HttpRequest request;

    
Иногда может возникать конфликт между параметрами конструктора и свойствами класса, если конструктор сохраняет входные данные в созданном объекте. В таком случае я рекомендую создавать аббревиатуры, удаляя гласные (см. как [USPS сокращает названия улиц](http://pe.usps.gov/text/pub28/28apc_002.htm)).

Another example:

    public class Message { private String recipient; public Message(String rcpt) { this.recipient = rcpt; } }

    
Во многих случаях лучшую подсказку для имени переменной можно получить, прочитав название ее класса. Просто напишите его с маленькой буквы, и все должно быть хорошо:

    File file; User user; Branch branch;

    


Однако *никогда* не делайте так для примитивных типов, таких как `Integer number` или `String string`.

Вы также можете использовать прилагательное, когда есть несколько переменных с разными характеристиками. Например:

    String contact(String left, String right);

    
## Constructors

Без исключений, должен быть только *один* конструктор, который сохраняет данные в переменные объекта. Все остальные конструкторы должны вызывать этот с разными аргументами. Например:

    public class Server { private String address; public Server(String uri) { this.address = uri; } public Server(URI uri) { this(uri.toString()); } }

    
Больше информации об этом можно найти в статье "Может быть только один основной конструктор".

## One-time Variables

Избегайте использования одноразовых переменных вообще во всех случаях. Под "одноразовыми" я подразумеваю переменные, которые используются только один раз. Как в этом примере:

    String name = “data.txt”; return new File(name);

    
Эта переменная выше используется только один раз, и код должен быть отрефакторен следующим образом:

    return new File(“data.txt”);

    
Иногда, очень редко --- в основном из-за лучшего форматирования --- могут использоваться одноразовые переменные. Тем не менее, старайтесь избегать таких ситуаций во всех случаях.

## Exceptions



Само собой разумеется, что **никогда** не следует проглатывать исключения, а лучше поднимать их как можно выше. В частных методах всегда следует пропускать проверяемые исключения наружу.

Никогда не используйте исключения для управления потоком выполнения. Например, следующий код является ошибочным:

    int size; try { size = this.fileSize(); } catch (IOException ex) { size = 0; }

    
Серьезно, а что, если эта `IOException` говорит "диск заполнен?" Вы все равно будете предполагать, что размер файла равен нулю и продолжите действовать?

## Indentation

Для отступов главное правило состоит в том, что скобка должна либо заканчивать строку, либо быть закрытой на той же строке (обратное правило применяется к закрывающей скобке). Например, следующий пример неправильный, потому что первая скобка не закрыта на той же строке и после неё есть символы. Вторая скобка также нарушает правило, потому что перед ней есть символы и она не открыта на той же строке.

    final File file = new File(directory, “file.txt”);

    
Правильный отступ должен выглядеть так:

    StringUtils.join( Arrays.asList( “first line”, “second line”, StringUtils.join( Arrays.asList(“a”, “b”) ) ), “separator” );

    
Второе важное правило отступов гласит, что вы должны помещать как можно больше информации в одну строку - в пределах 80 символов. Приведенный выше пример недействителен, так как он может быть сокращен.

    StringUtils.join( Arrays.asList( “first line”, “second line”, StringUtils.join(Arrays.asList(“a”, “b”)) ), “separator” );

    
## Redundant Constants

Классовые константы следует использовать, когда вы хотите обмениваться информацией между методами класса, и эта информация является характеристикой вашего класса. Не используйте константы в качестве замены строковых или числовых литералов - это очень плохая практика, которая приводит к загрязнению кода. Константы (как и любой объект в ООП) должны иметь смысл в реальном мире. Какое значение эти константы имеют в реальном мире:

    class Document { private static final String D_LETTER = “D”; // bad practice private static final String EXTENSION = “.doc”; // good practice }

    


Еще одна типичная ошибка - использование констант в модульных тестах для избегания дублирования строк/числовых литералов в методах тестирования. Не делайте так! Каждый метод тестирования должен работать с собственными наборами входных значений.

Используйте новые тексты и числа в каждом новом методе тестирования. Они независимы. Так почему им нужно общее набор констант входных данных?

## Test Data Coupling

Это пример [data coupling](http://en.wikipedia.org/wiki/Coupling_%28computer_programming%29) в тестовом методе:

    User user = new User(“Jeff”); // maybe some other code here MatcherAssert.assertThat(user.name(), Matchers.equalTo(“Jeff”)); ```

    На последней строке мы связываем "Jeff" с той же строковой литералом из первой строки. Если через несколько месяцев кто-то захочет изменить значение на третьей строке, ему/ей придется затратить дополнительное время, чтобы найти, где еще используется "Jeff" в том же методе.

    Чтобы избежать такой связи данных, следует ввести переменную. Подробнее об этом здесь: Несколько мыслей о каркасе модульного тестирования.

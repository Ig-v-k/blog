    GitHub Guidelines

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Это руководство объясняет рабочий процесс при работе с проектом XDSD, размещенным на GitHub. Вы начинаете, когда вам назначается задача на GitHub. Затем вы получите сообщение от менеджера проекта, содержащее номер, заголовок, описание задачи и ее бюджет в часах (обычно 30 минут).

    Если вы не согласны с распределением бюджета, не стесняйтесь попросить его увеличить. Как только вы будете удовлетворены бюджетом и поймете объем работы, сообщите об этом в ответе на задачу и начните работать. Имейте в виду, что вы не будете оплачиваться за время, затраченное сверх установленного бюджета.

    1. Fork

    Несмотря на то, что вы являетесь частью команды разработчиков, у вас нет прав на запись в репозиторий на GitHub. Следовательно, для внесения изменений вы должны сделать форк репозитория на свою учетную запись GitHub (создать его частную копию), внести необходимые изменения, а затем отправить их на рассмотрение с помощью “pull request”.

    После отправки запроса на рассмотрение pull request владелец репозитория утверждает ваши изменения путем их объединения с основным репозиторием. Таким образом мы защищаем основной поток разработки от случайного повреждения.

    В этой статье объясняется, как сделать форк репозитория: fork-a-repo

    А вот здесь объясняется, как скачать и установить GitHub на ваш компьютер: set-up-git

    Наконец, не забудьте добавить свой приватный SSH-ключ в GitHub: generating-ssh-keys

    2. Branch

    После того, как вы сделали форк нашего репозитория в свой аккаунт, склонируйте его на свой компьютер и переключитесь на ветку master. Например:

    git clone git@github.com:yegor256/xembly.git
git checkout master

    Теперь пришло время создать ветку (123 - номер GitHub задачи, над которой вы будете работать, и название ветки):

    git checkout -b 123

    По общепринятому соглашению мы используем одинаковые названия для ветки и задачи, над которыми вы работаете.

    3. Changes

    Все вопросы, связанные с задачами, должны обсуждаться в GitHub issue. Для GitHub issue не используются электронная почта, Skype, телефонные звонки или встречи. Все вопросы должны задаваться непосредственно в GitHub issue.

    Не стесняйтесь создавать новые issue, если что-то не ясно или вам нужна помощь. Очень часто бывает так, что вам может быть сложно выполнить задачу. Не паникуйте. Это обычно случается, когда вы только присоединились к проекту и пока не имеете достаточно информации. Если это произойдет, не пытайтесь разобраться с проблемой или вопросом самостоятельно.

    Правило при таких ситуациях следующее: “Если что-то не ясно, это наша вина, а не ваша.” Поэтому, если вы не понимаете дизайн проекта, это вина дизайнера проекта.

    Отправьте отчет об ошибке, запросив объяснение концепции дизайна. Вы будете оплачены за этот отчет, и информация, которую вы получите в ответе, будет распространяться между всеми другими разработчиками.

    Прочитайте эту статью: Bugs Are Welcome.

    Не ожидайте, что кто-то поможет вам. Вашим единственным источником помощи является сам исходный код. Если код не объясняет все, что вам нужно знать — это ошибка, которую нужно сообщить.

    4. Commit and Push

    Внесите все необходимые изменения с помощью текстового редактора или IDE. Хорошей практикой является фиксация изменений сразу после их внесения. Не накапливайте большое количество изменений перед фиксацией.

    git commit -am '#123: the description of the changes'
git push origin 123

    Если у вас возникли вопросы о характере работы, опубликуйте их в проблеме на GitHub и дождитесь ответа. Если вы считаете, что существующий код нуждается в улучшениях, не стесняйтесь создать новую проблему на GitHub.

    Не пытайтесь исправить все проблемы в одной ветке; позвольте другим программистам позаботиться о них.

    5. Pull Request

    Создайте pull request в GitHub, используя процесс, описанный в следующей статье: использование pull-запросов

    Опубликуйте его номер в оригинальной проблеме и дождитесь обратной связи.

    6. Code Review

    Через некоторое время ваш запрос на включение изменений будет рассмотрен представителем команды проекта. Во многих случаях вы можете получить несколько негативных комментариев, и вам придется исправить все проблемы, связанные с ними. Ваш запрос не будет включен в основную ветку (master branch), пока ваши изменения не удовлетворят рецензента.

    Будьте терпеливы с рецензентом и внимательно слушайте его. Однако не считайте, что ваш рецензент всегда прав. Если вы считаете, что ваши изменения валидны, настаивайте на том, чтобы кто-то другой их проверил.

    После внесения необходимых изменений в ветку не забудьте уведомить рецензента в задаче, разместив сообщение, адресованное ему/ей. На каждом этапе цикла рецензирования и исправлений делайте то же самое: просите рецензента еще раз взглянуть на ваш код. Без таких уведомлений ваш запрос на включение изменений может оставаться незамеченным на долгое время.

    7. Merge

    Когда все выглядит хорошо для рецензента, он сообщит нашему автоматизированному боту о слиянии. Затем автоматизированный бот выберет ваш запрос на слияние и попытается объединить его с веткой master. По разным причинам такая операция часто не удается. Если слияние не удалось, независимо от причины, вашей обязанностью является убедиться, что ваша ветка успешно объединена.

    Если вы не можете объединить ветку из-за ошибок в тестах, не связанных с вашей задачей, не пытайтесь исправлять их самостоятельно. Вместо этого сообщите о проблеме как о новой ошибке и дождитесь ее решения.

    Помните, пока ваша ветка не объединена, вы не получаете оплату.

    8. Payment

    После слияния ваших изменений вернитесь к задаче в GitHub и попросите автора закрыть ее. Как только задача будет закрыта менеджером проекта, вы получите оплату в течение нескольких часов.

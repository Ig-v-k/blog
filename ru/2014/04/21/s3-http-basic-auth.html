    Basic HTTP Auth for S3 Buckets

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Amazon S3 - простое и очень полезное хранилище двоичных объектов (также известных как “файлы”). Чтобы использовать его, вы создаете там “ведро” с уникальным именем и загружаете свои объекты.

    После этого, AWS гарантирует, что ваш объект будет доступен для скачивания через их RESTful API.

    Несколько лет назад AWS представила функцию S3, называемую размещение статического веб-сайта.

    С помощью размещения статического веб-сайта вы просто включаете функцию, и все объекты в вашем ведре становятся доступными через общедоступный HTTP. Это потрясающая возможность для размещения статического контента, такого как изображения, файлы JavaScript, видео и аудио контент.

    При использовании размещения необходимо изменить запись CNAME в вашем DNS так, чтобы она указывала на www.example.com.aws.amazon.com. После изменения записи DNS ваш статический веб-сайт будет доступен по адресу www.example.com, как обычно.

    Однако, при использовании Amazon S3 невозможно защитить ваш веб-сайт, потому что содержимое является исключительно статическим. Это означает, что у вас не может быть страницы входа на фронтенде. С этой услугой вы можете либо сделать свои объекты полностью общедоступными, чтобы их мог видеть любой в Интернете, либо назначить права доступа к ним, но только для пользователей, подключенных через RESTful API.

    Однако мой случай использования услуги был немного сложнее. Я хотел разместить свой статический контент как объекты S3. Однако я хотел сделать это так, чтобы только несколько человек имели доступ к контенту с использованием своих веб-браузеров.

    HTTP Basic Authentication

    Протокол HTTP предлагает удобную функцию “базовой аутентификации”, которая не требует дополнительных страниц сайта.

    Когда HTTP-запрос поступает на сервер, он не передает содержимое, а отвечает с помощью статусного кода 401. Этот ответ буквально означает: “Я не знаю, кто вы такой, пожалуйста, аутентифицируйтесь”.

    Браузер отображает свой собственный экран входа и запрашивает имя пользователя и пароль. После ввода учетных данных они объединяются, кодируются с помощью Base64 и добавляются в следующий запрос в заголовке Authorization HTTP.

    Теперь браузер пытается сделать еще одну попытку получить ту же самую веб-страницу. Но на этот раз HTTP-запрос содержит заголовок:

    Authorization: Basic am9lOnNlY3JldA==

    Вышеуказанное является всего лишь примером. В данном примере закодированная часть Base64 означает joe:secret, где joe - это имя пользователя, а secret - введенный им пароль.

    На этот раз сервер имеет информацию об аутентификации и может принять решение, является ли этот пользователь аутентифицированным (его пароль совпадает с записями сервера) и авторизованным (у него есть разрешение на доступ к запрашиваемой веб-странице).

    s3auth.com

    Поскольку Amazon не предоставляет такую ​​возможность, я решил создать простой веб-сервис, s3auth.com, который находится перед моими ведрами Amazon S3 и реализует механизм аутентификации и авторизации через HTTP.

    Вместо того, чтобы делать мои объекты общедоступными, я делаю их приватными и указываю в записи CNAME адрес relay.s3auth.com. Затем HTTP-запросы от веб-браузеров приходят на мой сервер, подключаются к Amazon S3, извлекают мои объекты и доставляют их обратно в виде HTTP-ответов.

    Сервер реализует аутентификацию и авторизацию с использованием специального файла .htpasswd в корне моего ведра. Формат файла .htpasswd идентичен формату, используемому сервером Apache HTTP — один пользователь на строку. Каждая строка содержит имя пользователя и хэш его пароля.

    Implementation

    Я сделал эту программу с открытым исходным кодом в основном, чтобы гарантировать моим пользователям, что сервер не хранит их личные данные где-либо, а действует только как сервис-прокси. В результате, исходный код программы находится на GitHub.

    В целях конфиденциальности и удобства я использую только OAuth2 для учетных записей пользователей. Это означает, что я не знаю, кто мои пользователи. У меня нет их имен или электронных адресов, только их номера учетных записей в Facebook, Google Plus или GitHub. Конечно, я могу найти их имена, используя эти номера, но эта информация все равно публична.

    Сервер реализован на Java6. Для его хостинга я использую один сервер Amazon EC2 m1.small под управлением Ubuntu. В настоящее время сервер кажется работающим надлежащим образом и стабильным.

    Extra Features

    Помимо аутентификации и авторизации, сервер s3auth.com может отображать списки страниц - так же, как сервер Apache HTTP. Если у вас есть коллекция объектов в вашем ведре, но отсутствует файл index.html, Amazon S3 выдает результат “страница не найдена”. Напротив, мой сервер отображает список объектов в ведре, когда отсутствует index.html, и позволяет перемещаться вверх или вниз по папке.

    Когда в вашем ведре включена функция версионирования, вы можете просматривать все версии любого объекта в браузере. Для этого просто добавьте ?all-versions в конец URL-адреса для отображения списка. Затем щелкните версию, чтобы сервер s3auth.com получил и отобразил ее.

    Traction

    Я создал эту службу в основном для себя, но, похоже, я не единственный с такими проблемами, описанными выше. В данный момент s3auth.com размещает более 300 доменов и передает более 10 Мб данных каждый час.

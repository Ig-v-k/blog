    Every Build in Its Own Docker Container

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Docker - это инструмент командной строки, который может выполнять оболочку команд в виртуальной Linux внутри изолированной файловой системы. Каждый раз, когда мы собираем наши проекты, мы хотим, чтобы они запускались в собственных контейнерах Docker. Возьмем в качестве примера этот проект Maven.

    $ sudo docker run -i -t ubuntu mvn clean test

    Эта команда запустит новую систему Ubuntu и выполнит mvn clean test внутри нее. Rultor.com, наш виртуальный ассистент, именно это делает с нашими сборками, когда мы разворачиваем, упаковываем, тестируем и объединяем их.

    Why Docker?

    Какие преимущества он нам дает? И почему Docker, когда есть много других технологий виртуализации, таких как LXC, например?

    Ну, есть несколько очень важных преимуществ:

    • Репозиторий изображений (hub.docker.com)

    • Versioning

    • “Application-centric” - “Ориентированный на приложение”

    Давайте обсудим их подробнее.

    Image Repository

    Docker позволяет обмениваться образами через свой общедоступный репозиторий на hub.docker.com. Это означает, что после подготовки рабочей среды для моего приложения, я создаю из нее образ и загружаю его на хаб.

    Допустим, я хочу, чтобы мой сборка Maven выполнялась в контейнере с предустановленным пакетом graphviz (чтобы включить инструмент командной строки dot). Сначала я бы запустил обычный контейнер Ubuntu и установил в нем graphviz.

    $ sudo docker run -i -t ubuntu /bin/bash
root@215d2696e8ad:/# sudo apt-get install -y graphviz
Reading package lists... Done
Building dependency tree
Reading state information... Done
The following extra packages will be installed:
...
root@215d2696e8ad:/# exit
$ sudo docker ps -a
CONTAINER ID        IMAGE               COMMAND             CREATED              STATUS                     PORTS               NAMES
215d2696e8ad        ubuntu:14.04        /bin/bash           About a minute ago   Exited (0) 3 seconds ago                       high_mccarthy

    У меня есть контейнер, который остановился несколько секунд назад. ID контейнера - 215d2696e8ad. Теперь я хочу сделать его многоразовым для всех последующих тестов на Rultor.com. Мне нужно создать из него образ.

    $ sudo docker commit 215d2696e8ad yegor256/beta
c5ad7718fc0e20fe4bf2c8a9bfade4db8617a25366ca5b64be2e1e8aa0de6e52

    Я только что сделал новый коммит к новому образу yegor256/beta. Этот образ можно использовать прямо сейчас. Я могу создать новый контейнер из этого образа, и внутри него уже будет установлен graphviz!

    Теперь пришло время поделиться моим образом на Docker Hub, чтобы он стал доступным для Rultor.

    $ sudo docker push yegor256/beta
The push refers to a repository [yegor256/beta] (len: 1)
Sending image list
Pushing repository yegor256/beta (1 tags)
511136ea3c5a: Image already pushed, skipping
d7ac5e4f1812: Image already pushed, skipping
2f4b4d6a4a06: Image already pushed, skipping
83ff768040a0: Image already pushed, skipping
6c37f792ddac: Image already pushed, skipping
e54ca5efa2e9: Image already pushed, skipping
c5ad7718fc0e: Image successfully pushed
Pushing tag for rev [c5ad7718fc0e] on {https://registry-1.docker.io/v1/repositories/yegor256/beta/tags/latest}

    Последний шаг - настроить Rultor для использования этого образа во всех сборках. Для этого я отредактирую файл .rultor.yml в корневой директории моего репозитория на GitHub.

    docker:
  image: yegor256/beta

    Вот и всё. С этого момента Rultor будет использовать моё настраиваемое Docker-образ с предустановленным graphviz в каждой сборке (слияние, релиз, развертывание и т. д.).

    Более того, если и когда я захочу добавить что-то ещё в образ, это легко сделать. Скажем, я хочу установить Ruby в свой образ сборки. Я запускаю контейнер из образа и устанавливаю его (обратите внимание, что я запускаю контейнер не из образа ubuntu, как делал раньше, а из yegor256/beta).

    $ sudo docker run -i -t yegor256/beta /bin/bash
root@7e0fbd9806c9:/# sudo apt-get install -y ruby
Reading package lists... Done
Building dependency tree
Reading state information... Done
The following extra packages will be installed:
...
root@7e0fbd9806c9:/# exit
$ sudo docker ps -a
CONTAINER ID        IMAGE                  COMMAND             CREATED             STATUS                     PORTS               NAMES
7e0fbd9806c9        yegor256/beta:latest   /bin/bash           28 seconds ago      Exited (0) 2 seconds ago                       pensive_pare
215d2696e8ad        ubuntu:14.04           /bin/bash           10 minutes ago      Exited (0) 8 minutes ago                       high_mccarthy

    Теперь вы можете видеть, что у меня есть два контейнера. Первый из них - тот, который я использую прямо сейчас; в нем содержится Ruby. Второй - тот, который я использовал раньше, и в нем содержится graphviz.

    Теперь мне нужно снова сделать коммит и отправить изменения:

    $ sudo docker commit 7e0fbd9806c9 yegor256/beta
6cbfb7a6b18a2182f42171f6bb5aef67c4819b5c2795edffa6a63ba78aaada2d
$ sudo docker push yegor256/beta
...

    Таким образом, этот Docker хаб является очень удобной функцией для Rultor и подобных систем.

    Versioning

    Как вы видели в приведенном выше примере, каждое изменение в образе Docker имеет свою собственную версию (хеш), и возможно отслеживать изменения. Также возможно откатиться к определенному изменению.

    Rultor сам не использует эту функциональность, но пользователи Rultor могут управлять своими конфигурациями сборки с гораздо большей точностью.

    Application-Centric

    Docker, в отличие от LXC или Vagrant, например, ориентирован на приложения. Это означает, что когда мы запускаем контейнер - мы запускаем приложение. В других технологиях виртуализации, когда вы получаете виртуальную машину - вы получаете полностью функциональную среду Unix, где вы можете войти через SSH и делать все, что хотите.

    Docker делает вещи проще. Он не предоставляет вам доступ к контейнеру через SSH, но запускает приложение внутри и показывает вам его вывод. Это именно то, что нам нужно в Rultor. Нам нужно запускать автоматическую сборку (например, Maven или Bundler), видеть ее вывод и получать код ее завершения. Если код не равен нулю, мы считаем сборку неудачной и сообщаем пользователю.

    Вот как мы запускаем сборку Maven:

    $ sudo docker run --rm -i -t yegor256/rultor mvn clean test
[INFO] ------------------------------------------------------------------------
[INFO] Building jcabi-github 0.13
[INFO] ------------------------------------------------------------------------
[INFO]
[INFO] --- maven-clean-plugin:2.5:clean (default-clean) @ jcabi-github ---
[INFO]
...

    Как видите, Maven запускается немедленно. Мы не беспокоимся о внутренних механизмах контейнера. Мы просто запускаем приложение внутри него.

    Более того, благодаря опции --rm, контейнер уничтожается сразу после завершения выполнения Maven.

    Вот в чем суть подхода, ориентированного на приложение.

    В целом, наше впечатление о Docker очень положительное.

    How to Release to Maven Central, in One Click

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Когда я выпускаю новую версию jcabi-aspects, Java-библиотеки с открытым исходным кодом, на Maven Central, это занимает 30 секунд моего времени. Может быть, даже меньше. Недавно я выпустил версию 0.17.2. Вы можете посмотреть, как это произошло, в GitHub issue #80:

    Как видите, я дал команду Rultor и она выпустила новую версию в центральный репозиторий Maven. Я ничего больше не делал.

    Теперь посмотрим, как вы можете сделать то же самое. Как вы можете настроить свой проект так, чтобы выпуск его новой версии в Maven Central занял всего несколько секунд вашего времени.

    Кстати, я предполагаю, что вы разместили свой проект на GitHub. Если нет, весь этот учебник не будет работать. Если вы до сих пор не на GitHub, я настоятельно рекомендую перейти туда.

    Prepare Your POM

    Убедитесь, что ваш pom.xml содержит все необходимые элементы, описанные в Central Sync Requirements, требуемые Sonatype. Мы развернем на Sonatype, и они синхронизируют все артефакты JAR (и не только) с Maven Central.

    Register a Project With Sonatype

    Создайте учетную запись в Sonatype JIRA и создайте запрос, чтобы утвердить ваш groupId. Этот OSSRH Guide подробно объясняет этот шаг.

    Create and Distribute a GPG Key

    Создайте GPG-ключ и распространите его, как указано в статье Работа с PGP-подписями.

    После выполнения этого шага у вас должно быть два файла: pubring.gpg и secring.gpg.

    Create settings.xml

    Создайте settings.xml, рядом с двумя файлами .gpg, созданными на предыдущем этапе.

    <settings>
  <profiles>
    <profile>
      <id>foo</id> 
      <properties>
        <gpg.homedir>/home/r</gpg.homedir>
        <gpg.keyname>9A105525</gpg.keyname>
        <gpg.passphrase>my-secret</gpg.passphrase>
      </properties>
    </profile>
  </profiles>
  <servers>
    <server>
      <id>oss.sonatype.org</id>
      <username></username>
      <password></password>
    </server>
  </servers>
</settings>

    В этом примере 9A105525 - это ID вашего открытого ключа, а my-secret - это парольная фраза, которую вы использовали при генерации ключей.

    Encrypt Security Assets

    Теперь зашифруйте эти три файла с помощью rultor remote.

    $ gem install rultor
$ rultor encrypt -p me/test pubring.gpg
$ rultor encrypt -p me/test secring.gpg
$ rultor encrypt -p me/test settings.xml

    Вместо me/test вы должны использовать название вашего проекта на GitHub.

    Вы получите три новых файла: pubring.gpg.asc, secring.gpg.asc и settings.xml.asc. Добавьте их в корневую директорию вашего проекта, закоммитьте и запушьте. Файлы содержат вашу секретную информацию, но только сервер Rultor может их расшифровать.

    Add Sonatype Repositories

    Я рекомендовал бы использовать jcabi-parent в качестве родительского pom-файла для вашего проекта. Это избавит от необходимости выполнять множество дополнительных шагов. Если вы уже используете jcabi-parent, пропустите этот шаг.

    Однако, если вы не используете jcabi-parent, вы должны добавить эти два репозитория в свой pom.xml:

    <project>
  [...]
  <distributionManagement>
    <repository>
      <id>oss.sonatype.org</id>
      <url>https://oss.sonatype.org/service/local/staging/deploy/maven2/</url>
    </repository>
    <snapshotRepository>
      <id>oss.sonatype.org</id>
      <url>https://oss.sonatype.org/content/repositories/snapshots</url>
    </snapshotRepository>
  </distributionManagement>
</project>

    Configure GPG Plugin

    Опять же, я бы рекомендовал использовать jcabi-parent, который автоматически настраивает этот плагин. Если вы его уже используете, пропустите этот шаг.

    В противном случае добавьте этот плагин в ваш pom.xml:

    <project>
  [..]
  <build>
    [..]
    <plugins>
      [..]
      <plugin>
        <artifactId>maven-gpg-plugin</artifactId>
        <version>1.5</version>
        <executions>
          <execution>
            <id>sign-artifacts</id>
            <phase>verify</phase>
            <goals>
              <goal>sign</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
    </plugins>
  </build>
</project>

    Configure Versions Plugin

    Еще раз рекомендую использовать http://parent.jcabi.com. Он настраивает все необходимые плагины напрямую из коробки. Если вы уже используете его, пропустите этот шаг.

    В противном случае, добавьте этот плагин в ваш pom.xml:

    <project>
  [..]
  <build>
    [..]
    <plugins>
      [..]
      <plugin>
        <groupId>org.codehaus.mojo</groupId>
        <artifactId>versions-maven-plugin</artifactId>
        <version>2.1</version>
        <configuration>
          <generateBackupPoms>false</generateBackupPoms>
        </configuration>
      </plugin>
    </plugins>
  </build>
</project>

    Configure Sonatype Plugin

    Да, ты прав, http://parent.jcabi.com тоже поможет тебе здесь. Если ты его используешь, пропусти этот шаг тоже.

    В противном случае, добавьте эти четыре плагина в ваш файл pom.xml:

    <project>
  [..]
  <build>
    [..]
    <plugins>
      [..]
      <plugin>
        <artifactId>maven-deploy-plugin</artifactId>
        <configuration>
          <skip>true</skip>
        </configuration>
      </plugin>
      <plugin>
        <artifactId>maven-source-plugin</artifactId>
        <executions>
          <execution>
            <id>package-sources</id>
            <goals>
              <goal>jar</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
      <plugin>
        <artifactId>maven-javadoc-plugin</artifactId>
        <executions>
          <execution>
            <id>package-javadoc</id>
            <phase>package</phase>
            <goals>
              <goal>jar</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
      <plugin>
        <groupId>org.sonatype.plugins</groupId>
        <artifactId>nexus-staging-maven-plugin</artifactId>
        <version>1.6</version>
        <extensions>true</extensions>
        <configuration>
          <serverId>oss.sonatype.org</serverId>
          <nexusUrl>https://oss.sonatype.org/</nexusUrl>
          <description>${project.version}</description>
        </configuration>
        <executions>
          <execution>
            <id>deploy-to-sonatype</id>
            <phase>deploy</phase>
            <goals>
              <goal>deploy</goal>
              <goal>release</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
    </plugins>
  </build>
</project>

    Create Rultor Configuration

    Создайте файл .rultor.yml в корневом каталоге вашего проекта (ссылка на страницу справки подробно объясняет этот формат).

    decrypt:
  settings.xml: "repo/settings.xml.asc"
  pubring.gpg: "repo/pubring.gpg.asc"
  secring.gpg: "repo/secring.gpg.asc"
release:
  script: |
    mvn versions:set "-DnewVersion=${tag}"
    git commit -am "${tag}"
    mvn clean deploy -Pjcabi --settings /home/r/settings.xml

    Вы можете сравнить свой файл с живой конфигурацией Rultor для jcabi-aspects.

    Run It!

    Теперь пришло время узнать, как все это работает. Создайте новое обращение в трекере задач GitHub и разместите в нем нечто подобное (подробнее о командах Rultor).

    @rultor release, tag is `0.1`

    Вы получите ответ в течение нескольких секунд. Остальное будет сделано Rultor’ом.

    Enjoy!

    Кстати, если что-то не работает так, как я объяснил, не стесняйтесь отправить запрос в систему учета проблем Rultor. Я постараюсь вам помочь.

    Да, забыл упомянуть, Rultor также выполняет две важные задачи. Во-первых, он создает релиз GitHub с правильным описанием. Во-вторых, он публикует твит о релизе, который вы можете ретвитнуть, чтобы объявить своим подписчикам. Оба функционала очень удобны для меня. Например:

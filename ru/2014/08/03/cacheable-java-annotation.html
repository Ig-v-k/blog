    Cache Java Method Results

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Скажем, у вас есть метод, который требует времени для выполнения, и вы хотите, чтобы его результат был кэширован. Есть множество решений, включая Apache Commons JCS, Ehcache, JSR 107, Guava Caching и многие другие.

    jcabi-aspects предлагает очень простую реализацию на основе аспектов AOP и аннотаций Java6.

    import com.jcabi.aspects.Cacheable;
public class Page {
  @Cacheable(lifetime = 5, unit = TimeUnit.MINUTES)
  String load() {
    return new URL("http://google.com").getContent().toString();
  }
}

    Результат метода load() будет кэшироваться в памяти в течение пяти минут.

    How It Works?

    Этот пост о AOP, AspectJ и логировании методов объясняет, как работает “aspect weaving” (я настоятельно рекомендую вам сначала его прочитать).

    Здесь я объясню, как работает кэширование.

    Подход очень прямолинейный. Есть статическая хэш-карта с ключами “координаты метода” и значениями - их результаты. Координаты метода состоят из объекта, владельца метода и имени метода с типами параметров.

    В приведенном выше примере сразу после завершения метода load() карта получает новую запись (упрощенный пример, конечно).

    key: [page, "load()"]
value: "<html>...</html>"

    Каждый последующий вызов load() будет перехвачен аспектом из jcabi-aspects и немедленно обработан с помощью значения из кэш-мапы. Метод не будет получать никакого управления до конца своего срока жизни, который в данном примере составляет пять минут.

    What About Cache Flushing?

    Иногда необходимо иметь возможность очистки кэша до его окончания срока службы. Вот практический пример:

    import com.jcabi.aspects.Cacheable;
public class Employees {
  @Cacheable(lifetime = 1, unit = TimeUnit.HOURS)
  int size() {
    // calculate their amount in MySQL
  }
  @Cacheable.FlushBefore
  void add(Employee employee) {
    // add a new one to MySQL
  }
}

    Очевидно, что количество сотрудников в базе данных будет отличаться после выполнения метода add(), и результат size() должен быть недействительным в кэше. Эта операция недействительности называется “flushing”, и ее вызывает аннотация @Cacheable.FlushBefore.

    На самом деле, каждый вызов add() недействителен для всех кэшированных методов в этом классе, а не только для size().

    Также существует @Cacheable.FlushAfter. Разница заключается в том, что FlushBefore гарантирует, что кэш уже недействителен, когда начинается метод add(). FlushAfter делает кэш недействительным после завершения метода add(). Эта небольшая разница иногда имеет большое значение.

    Cache Java Method Results

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Скажем, у вас есть метод, который требует времени для выполнения, и вы хотите, чтобы его результат был кэширован. Существует много решений, включая Apache Commons JCS, Ehcache, JSR 107, Guava Caching и многие другие.

    jcabi-aspects offers a very simple one, based on AOP aspects and Java6 annotations:

    import com.jcabi.aspects.Cacheable;
public class Page {
  @Cacheable(lifetime = 5, unit = TimeUnit.MINUTES)
  String load() {
    return new URL("http://google.com").getContent().toString();
  }
}

    The result of load() method will be cached in memory for five minutes.

    How It Works?

    В этом посте об AOP, AspectJ и логировании методов объясняется, как работает “сплетение аспектов” (я настоятельно рекомендую вам сначала его прочитать).

    Here I’ll explain how caching works.

    Подход очень простой. Есть статическая хэш-карта, где ключи представлены в виде “координат метода”, а значения - их результатами. Координаты метода состоят из объекта, владельца метода и имени метода с типами параметров.

    In the example above, right after the method load() finishes, the map gets a new entry (simplified example, of course):

    key: [page, "load()"]
value: "<html>...</html>"

    Каждый последующий вызов load() будет перехвачен аспектом из jcabi-aspects и сразу же разрешен с помощью значения из кэш-карты. Метод не будет получать никакого контроля до конца его жизненного цикла, который в данном примере составляет пять минут.

    What About Cache Flushing?

    Sometimes it’s necessary to have the ability to flush cache before the end of its lifetime. Here is a practical example:

    import com.jcabi.aspects.Cacheable;
public class Employees {
  @Cacheable(lifetime = 1, unit = TimeUnit.HOURS)
  int size() {
    // calculate their amount in MySQL
  }
  @Cacheable.FlushBefore
  void add(Employee employee) {
    // add a new one to MySQL
  }
}

    Очевидно, что количество сотрудников в базе данных будет отличаться после выполнения метода add(), и результат метода size() должен быть недействительным в кэше. Эта операция недействительности называется “flushing”, и ее вызывает аннотация @Cacheable.FlushBefore.

    Actually, every call to add() invalidates all cached methods in this class, not only size().

    Также есть @Cacheable.FlushAfter. Разница заключается в том, что FlushBefore гарантирует, что кэш уже будет недействителен, когда метод add() начинает работу. FlushAfter делает кэш недействительным после того, как метод add() завершается. Эта небольшая разница иногда имеет большое значение.

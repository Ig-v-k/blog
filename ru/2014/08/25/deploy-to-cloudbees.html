    How to Deploy to CloudBees, in One Click

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Когда я разворачиваю новую версию stateful.co, Java веб-приложение, на CloudBees, это занимает 30 секунд моего времени. Может быть, даже меньше. Недавно я развернул версию 1.6.5. Вы можете увидеть, как все произошло, в GitHub issue #6.

    Как видите, я дал команду Rultor и он упаковал, протестировал и развернул новую версию в CloudBees. Я ничего больше не делал.

    Теперь давайте посмотрим, как вы можете сделать то же самое. Как вы можете настроить свой проект так, чтобы развертывание его новой версии на CloudBees занимало всего несколько секунд вашего времени.

    Since CloudBees is shutting down its PaaS service by the end of December, 2014, this article will have no sense after that.

    Configure the CloudBees Maven Plugin

    Добавьте этот профиль в свой pom.xml:

    <project>
  [..]
  <profiles>
    <profile>
      <id>cloudbees</id>
      <activation>
        <property><name>bees.appId</name></property>
      </activation>
      <pluginRepositories>
        <pluginRepository>
          <id>cloudbees-public-release</id>
          <url>
            http://repository-cloudbees.forge.cloudbees.com/public-release
          </url>
        </pluginRepository>
      </pluginRepositories>
      <build>
        <pluginManagement>
          <plugins>
            <plugin>
              <artifactId>maven-deploy-plugin</artifactId>
              <configuration>
                  <skip>true</skip>
              </configuration>
            </plugin>
          </plugins>
        </pluginManagement>
        <plugins>
          <plugin>
            <groupId>com.cloudbees</groupId>
            <artifactId>bees-maven-plugin</artifactId>
            <version>1.3.2</version>
            <configuration>
              <appid>${bees.id}</appid>
              <apikey>${bees.key}</apikey>
              <secret>${bees.secret}</secret>
            </configuration>
            <executions>
              <execution>
                <id>deploy-to-production</id>
                <phase>deploy</phase>
                <goals>
                  <goal>deploy</goal>
                </goals>
              </execution>
            </executions>
          </plugin>
        </plugins>
      </build>
    </profile>
  </profiles>
</project>

    Этот плагин не находится в Maven Central (к сожалению). Поэтому нам нужно указать <pluginRepository>.

    Обратите внимание на то, что мы также отключаем maven-deploy-plugin, так как он попытается разместить ваш пакет WAR в хранилище из раздела <distributionManagement>. Мы хотим избежать этого.

    Профиль активируется только тогда, когда определено свойство bees.id. Это не произойдет во время обычной разработки и тестирования, но произойдет во время цикла развертывания, инициированного Rultor, поскольку мы определим это свойство в settings.xml (обсуждается ниже).

    Secure Access to CloudBees

    Создайте учетную запись в CloudBees и зарегистрируйте свое веб-приложение там. CloudBees бесплатен, пока вам не потребуется слишком большая вычислительная мощность. Я считаю, что веб-приложения должны быть легкими по определению, поэтому бесплатный уровень CloudBees - идеальный выбор.

    Создайте файл settings.xml (но не загружайте его в репозиторий!).

    <settings>
  <profiles>
    <profile>
      <id>cloudbees</id>
      <properties>
        <bees.id>stateful/web</bees.id>
        <bees.key></bees.key>
        <bees.secret></bees.secret>
      </properties>
    </profile>
  </profiles>
</settings>

    Зашифруйте этот файл с помощью удаленного сервера Rultor:

    $ gem install rultor
$ rultor encrypt -p me/test settings.xml

    Вместо me/test используйте название вашего проекта на GitHub.

    Вы должны получить файл settings.xml.asc; добавьте его в корневую директорию вашего проекта, сделайте коммит и отправьте. Этот файл содержит ваши учетные данные CloudBees, но в зашифрованном формате. Никто не может прочитать его, кроме сервера Rultor.

    Configure Versions Plugin

    Я рекомендую использовать jcabi-parent. Он настраивает необходимый плагин “из коробки”. Если вы уже используете его, пропустите этот шаг.

    В противном случае, добавьте этот плагин в ваш pom.xml:

    <project>
  [..]
  <build>
    [..]
    <plugins>
      [..]
      <plugin>
        <groupId>org.codehaus.mojo</groupId>
        <artifactId>versions-maven-plugin</artifactId>
        <version>2.1</version>
        <configuration>
          <generateBackupPoms>false</generateBackupPoms>
        </configuration>
      </plugin>
    </plugins>
  </build>
</project>

    Configure Rultor

    Создайте файл .rultor.yml в корневом каталоге вашего проекта (этот ссылочная страница подробно объясняет этот формат).

    decrypt:
  settings.xml: "repo/settings.xml.asc"
release:
  script: |
    mvn versions:set "-DnewVersion=${tag}"
    git commit -am "${tag}"
    mvn clean deploy --settings /home/r/settings.xml

    Вы можете сравнить свой файл с живой конфигурацией Rultor stateful.co.

    Run It!

    Теперь пришло время узнать, как все это работает. Создайте новый тикет в трекере проблем GitHub и вставьте в него что-то подобное (подробнее читайте о командах Rultor):

    @rultor release, tag is `0.1`

    Вы получите ответ в течение нескольких секунд. Остальное выполнит Rultor.

    Enjoy!

    Кстати, если что-то не работает так, как я объяснил, не стесняйтесь отправить запрос в трекер проблем Rultor. Я постараюсь вам помочь.

    Также, аналогичная конфигурация может быть выполнена для [Heroku] (используя [jcabi-heroku-maven-plugin] (http://heroku.jcabi.com)) и для AWS Elastic Beanstalk (используя [jcabi-beanstalk-maven-plugin] (http://beanstalk.jcabi.com)). Возможно, я также посвящу им отдельные посты.

    XML+XSLT in a Browser

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Разделение данных и их представления - отличная концепция. Возьмем, к примеру, HTML и CSS. HTML предполагается содержать чистые данные, а CSS предполагается форматировать эти данные для их удобного чтения человеком. Много лет назад, это, вероятно, и было намерением HTML/CSS, но на самом деле это не работает так. Главным образом, потому что CSS недостаточно мощный.

    Мы все еще должны форматировать наши данные с помощью HTML-тегов, в то время как CSS может помочь немного с позиционированием и оформлением.

    С другой стороны, XML с XSLT идеально реализует идею разделения данных и представления. XML-документы, подобно HTML, предполагается содержать только данные, без информации о расположении или форматировании. XSL-таблицы стилей располагают и оформляют данные. XSL является более мощным языком. Вот почему в XML можно избежать любого форматирования.

    Последние версии Chrome, Safari, Firefox и IE поддерживают этот механизм. Когда браузер получает XML-документ с сервера, и у документа есть ассоциированный с ним XSL-стилевой файл, браузер трансформирует XML в HTML на лету.

    Working Example

    Давайте рассмотрим простое веб-приложение на Java, которое работает следующим образом. Оно использует Фреймворк Takes, который делает этот механизм возможным. В следующем сообщении я объясню, как работает ReXSL. А пока давайте сосредоточимся на идее предоставления чистых данных в формате XML и их форматировании с помощью таблицы стилей XSL.

    Откройте http://www.stateful.co - это сборник состояний веб-примитивов, описанных в статье “Атомарные счетчики на Stateful.co”.

    Откройте его в Chrome или Safari. Когда вы это сделаете, вы должны увидеть обычную веб-страницу с логотипом, некоторым текстом, некоторыми ссылками, нижним колонтитулом и т. д. Теперь проверьте его исходные коды (я предполагаю, что вы знаете, как это делать).

    Это примерно то, что вы увидите (предполагается, что вы понимаете XML, если нет, начинайте изучать его немедленно):

    <?xml-stylesheet type='text/xsl' href='/xsl/index.xsl'?>
<page date="2014-06-15T15:30:49.521Z" ip="10.168.29.135">
  <menu>home</menu>
  <documentation>.. some text here ..</documentation>
  <version>
    <name>1.4</name>
    <revision>5c7b5af</revision>
    <date>2014-05-29 07:58</date>
  </version>
  <links>
    <link href="..." rel="rexsl:google" type="text/xml"/>
    <link href="..." rel="rexsl:github" type="text/xml"/>
    <link href="..." rel="rexsl:facebook" type="text/xml"/>
  </links>
  <millis>70</millis>
</page>

    Как видите, это правильный XML-документ с атрибутами, элементами и данными. Он не содержит никакой информации о том, как его элементы должны быть представлены конечному пользователю. Фактически, этот документ более подходит для машинного анализа, а не для чтения человеком.

    Документ содержит данные, которые важны для его заказчика. Заказчик сам выбирает, как отображать эти данные или не отображать их вообще.

    Вторая строка связывает документ с таблицей стилей XSL /xsl/index.xsl, которая загружается браузером отдельно.

    <?xml-stylesheet type='text/xsl' href='/xsl/index.xsl'?>

    Откройте инструменты разработчика в Chrome, и вы увидите, что сразу после загрузки страницы браузер загружает таблицу стилей XSL, а затем все остальные ресурсы, включая несколько таблиц стилей CSS, jQuery и SVG-логотип.

    index.xsl includes layout.xsl, that’s why it is loaded right after.

    Давайте рассмотрим пример index.xsl (в реальности он намного сложнее, проверьте layout.xsl. Например:

    <xsl:stylesheet version="2.0"
  xmlns:xsl="http://www.w3.org/1999/XSL/Transform"
  xmlns="http://www.w3.org/1999/xhtml">
  <xsl:template match="page">
    <html>
      <body>
        <p>
          Current version of the application is
          <xsl:value-of select="version/name"/>
        </p>
      </body>
    </html>
  </xsl:template>
</xsl:stylesheet>

    Я думаю, что очевидно, как будет выглядеть HTML-страница после применения этого XSL-стиля к нашему XML-документу.

    Для меня этот XSL выглядит чистым и понятным. Однако я часто слышу от людей, что XSLT - это язык программирования, сложный для понимания. Я совсем не нахожу его сложным. Конечно, я не использую все его возможности. Но для простой генерации страницы мне достаточно знать несколько простых команд и принцип преобразования XML.

    Why Not a Templating Engine?

    Теперь, почему этот подход лучше, чем все эти широко используемые шаблонизаторы Java, включая JSP, JSF, Velocity, FreeMarker, Tiles, и т.д.?

    Ну, я вижу несколько причин. Но самые важные -

    1. Web UI и API - это одни и те же страницы. Нет необходимости разрабатывать отдельные страницы для RESTful API - пользовательский интерфейс веб-приложения, доступный с помощью компьютера, является API. По моему опыту, это позволяет избежать массового дублирования кода.

    2. XSL можно протестировать самостоятельно без сервера. Чтобы протестировать, как будет выглядеть наш веб-сайт с определенными данными, мы просто создаем новый XML-документ с необходимыми тестовыми данными, связываем его с XSL и открываем его в браузере. Мы также можем изменять XML и обновлять страницу в браузере. Это значительно упрощает работу дизайнера HTML/CSS и делает ее независимой от программистов.

    3. XSL - это мощный функциональный язык. По сравнению со всеми остальными движками шаблонов, которые больше похожи на обходные пути, XSL является полной и хорошо разработанной средой. Написание XSL (после того, как вы привыкнете к его синтаксису и концепциям программирования) само по себе приносит удовольствие. Вы не внедряете инструкции в HTML-документ (как в JSP и всех остальных). Вместо этого вы программирование преобразования данных в отображение - это другой подход и намного лучшее ощущение.

    4. XML-вывод идеально поддается тестированию. Контроллер в архитектуре MVC, который генерирует XML-документ с данными, необходимыми для таблицы стилей XSL, может быть легко протестирован в одном модульном тесте с использованием простых выражений XPath. Тестирование контроллера, который внедряет данные в систему шаблонов, является гораздо более сложной операцией — иногда даже невозможной. Я также работаю с PHP и Ruby. У них точно такие же проблемы — даже несмотря на то, что их системы шаблонов гораздо более мощные благодаря интерпретируемому характеру языков.

    Is It Fully Supported?

    Все было бы замечательно, если бы все браузеры поддерживали отображение XML+XSL. Однако это далеко от истины. Только последние версии современных браузеров поддерживают XSL. Проверьте это сравнение, сделанное Джулианом Решке. Кроме того, XSLT 2.0 вообще не поддерживается.

    Есть, однако, обходной путь. Мы можем определить браузер, отправляющий запрос (через заголовок HTTP User-Agent) и преобразовать XML в HTML на стороне сервера. Таким образом, для современных браузеров, поддерживающих XSL, мы будем доставлять XML, а для всех остальных - HTML.

    Вот как работает ReXSL framework. Откройте http://www.stateful.co в Internet Explorer и вы увидите HTML-документ, а не XML-документ, как это происходит в Chrome.

    Java Method Logging with AOP and Annotations

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Иногда я хочу регистрировать (с помощью slf4j и log4j) каждое выполнение метода, видя передаваемые ему аргументы, возвращаемые значения и время, затраченное на каждое выполнение. Вот как я это делаю с помощью AspectJ, jcabi-aspects и аннотаций Java 6.

    public class Foo {
  @Loggable
  public int power(int x, int p) {
    return Math.pow(x, p);
  }
}

    Вот что я вижу в выводе log4j:

    [INFO] com.example.Foo #power(2, 10): 1024 in 12μs
[INFO] com.example.Foo #power(3, 3): 27 in 4μs

    Хорошо, не так ли? Теперь давайте посмотрим, как это работает.

    Annotation with Runtime Retention

    Аннотации - это техника, введенная в Java 6. Это инструмент метапрограммирования, который не изменяет способ работы кода, но ставит отметки на определенные элементы (методы, классы или переменные). Другими словами, аннотации - это просто маркеры, прикрепленные к коду, которые можно увидеть и прочитать. Некоторые аннотации предназначены лишь для просмотра во время компиляции - они не существуют в файле .class после компиляции. Другие остаются видимыми после компиляции и могут быть доступны во время выполнения.

    Например, @Override относится к первому типу (его тип сохранения - SOURCE), в то время как @Test из JUnit относится ко второму типу (тип сохранения - RUNTIME). @Loggable - который я использую в приведенном выше скрипте - это аннотация второго типа, из jcabi-aspects. Она сохраняется в байт-коде в файле .class после компиляции.

    Снова важно понимать, что даже если метод power() аннотирован и скомпилирован, до сих пор он ничего не отправляет в slf4j. Он просто содержит маркер, который говорит “пожалуйста, записывайте моё выполнение”.

    Aspect Oriented Programming (AOP)

    AOP - это полезная техника, которая позволяет добавлять исполняемые блоки в исходный код без явного изменения его. В нашем примере мы не хотим вести журнал выполнения метода внутри класса. Вместо этого мы хотим, чтобы другой класс перехватывал каждый вызов метода power(), измерял время его выполнения и отправлял эту информацию в slf4j.

    Мы хотим, чтобы этот перехватчик понимал нашу аннотацию @Loggable и регистрировал каждый вызов конкретного метода power(). И, конечно же, тот же самый перехватчик должен использоваться для других методов, где мы будем размещать ту же аннотацию в будущем.

    Этот случай идеально подходит для первоначального намерения AOP - избежать повторной реализации некоторого общего поведения в нескольких классах.

    Логирование является дополнительной функцией нашей основной функциональности, и мы не хотим загромождать наш код несколькими инструкциями логирования. Вместо этого, мы хотим, чтобы логирование происходило в фоновом режиме.

    В терминах AOP наше решение можно объяснить как создание аспекта, который пересекает код в определенных точках присоединения и применяет окружающую совету, реализующую необходимую функциональность.

    AspectJ

    Давайте посмотрим, что означают эти волшебные слова. Но сначала давайте посмотрим, как jcabi-aspects реализует их с использованием AspectJ (это упрощенный пример, полный код вы можете найти в MethodLogger.java).

    @Aspect
public class MethodLogger {
  @Around("execution(* *(..)) && @annotation(Loggable)")
  public Object around(ProceedingJoinPoint point) {
    long start = System.currentTimeMillis();
    Object result = point.proceed();
    Logger.info(
      "#%s(%s): %s in %[msec]s",
      MethodSignature.class.cast(point.getSignature()).getMethod().getName(),
      point.getArgs(),
      result,
      System.currentTimeMillis() - start
    );
    return result;
  }
}

    Это аспект с единственным around advice around(), расположенным внутри. Аспект аннотирован @Aspect, а advice - @Around. Как уже обсуждалось выше, эти аннотации являются просто маркерами в файлах .class. Они ничего не делают, за исключением предоставления некоторой мета-информации тем, кто интересуется временем выполнения.

    Аннотация @Around имеет один параметр, который в данном случае указывает, что совет должен быть применен к методу, если:

    1. его модификатор видимости - * (public, protected или private);

    2. его имя * (любое имя);

    3. его аргументы - .. (любые аргументы); и

    4. он аннотирован с помощью @Loggable

    Когда вызов аннотированного метода должен быть перехвачен, метод around() выполняется перед выполнением фактического метода. Когда вызов метода power() должен быть перехвачен, метод around() получает экземпляр класса ProceedingJoinPoint и должен вернуть объект, который будет использован в качестве результата метода power().

    Для вызова исходного метода power(), совет должен вызывать proceed() объекта точки присоединения.

    Мы компилируем этот аспект и делаем его доступным в classpath вместе с нашим основным файлом Foo.class. До сих пор все хорошо, но нам нужно сделать последний шаг, чтобы применить наш совет.

    Binary Aspect Weaving

    Аспектное плетение - это процесс применения советов. Аспектный плетельщик изменяет исходный код, внедряя вызовы аспектов. Именно это делает AspectJ. Мы передаем ему два двоичных файла Java - Foo.class и MethodLogger.class; в ответ получаем три файла - измененный Foo.class, Foo$AjcClosure1.class и неизменный MethodLogger.class.

    Для понимания, к каким методам следует применять какие советы, AspectJ-ткач использует аннотации из файлов .class. Он также использует отражение для просмотра всех классов на пути к классу. Он анализирует, какие методы удовлетворяют условиям аннотации @Around. Конечно же, он находит наш метод power().

    Итак, есть два шага. Сначала мы компилируем наши .java файлы, используя javac, и получаем два файла. Затем AspectJ переплетает/изменяет их и создает свой собственный дополнительный класс. Наш класс Foo выглядит примерно так после переплетения:

    public class Foo {
  private final MethodLogger logger;
  @Loggable
  public int power(int x, int p) {
    // Prepares the "point" object
    return this.logger.around(point);
  }
  private int power_aroundBody(int x, int p) {
    return Math.pow(x, p);
  }
}

    Аспектный ткач AspectJ перемещает нашу исходную функциональность в новый метод power_aroundBody(), и перенаправляет все вызовы power() на аспектный класс MethodLogger.

    Вместо одного метода power() в классе Foo теперь у нас есть четыре класса, работающих вместе. Отныне это происходит за кулисами при каждом вызове power():

    Как видите, процесс компоновки аспектов связывает классы и аспекты, передавая вызовы между ними через точки соединения. Без компоновки, как классы, так и аспекты представляют собой просто скомпилированные исполняемые файлы Java с прикрепленными аннотациями.

    jcabi-aspects

    jcabi-aspects - это JAR-библиотека, которая содержит аннотацию Loggable и аспект MethodLogger (кстати, есть еще много других аспектов и аннотаций). Вам не нужно писать свой собственный аспект для журналирования методов. Просто добавьте несколько зависимостей в свой classpath и настройте jcabi-maven-plugin для аспектного плетения (получите их последние версии в Maven Central).

    <project>
  <dependencies>
    <dependency>
      <groupId>com.jcabi</groupId>
      <artifactId>jcabi-aspects</artifactId>
    </dependency>
    <dependency>
      <groupId>org.aspectj</groupId>
      <artifactId>aspectjrt</artifactId>
    </dependency>
  </dependencies>
  <build>
    <plugins>
      <plugin>
        <groupId>com.jcabi</groupId>
        <artifactId>jcabi-maven-plugin</artifactId>
        <executions>
          <execution>
            <goals>
              <goal>ajc</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
    </plugins>
  </build>
</project>

    Поскольку этот процесс плетения требует значительного усилия для настройки, я создал удобный плагин Maven с целью ajc, который выполняет всю работу по плетению аспектов. Вы можете использовать AspectJ напрямую, но я рекомендую использовать jcabi-maven-plugin.

    Вот и все. Теперь вы можете использовать аннотацию @com.jcabi.aspects.Loggable и ваши методы будут записываться через slf4j.

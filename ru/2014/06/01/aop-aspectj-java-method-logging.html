    Java Method Logging with AOP and Annotations

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Иногда мне хочется вести журнал (через slf4j и log4j) каждого выполнения метода, видеть, какие аргументы он получает, что возвращает и сколько времени занимает каждое выполнение. Вот как я это делаю с помощью AspectJ, jcabi-aspects и аннотаций Java 6:

    public class Foo {
  @Loggable
  public int power(int x, int p) {
    return Math.pow(x, p);
  }
}

    Это то, что я вижу в выводе log4j:

    [INFO] com.example.Foo #power(2, 10): 1024 in 12μs
[INFO] com.example.Foo #power(3, 3): 27 in 4μs

    Приятно, не так ли? Теперь давайте посмотрим, как это работает.

    Annotation with Runtime Retention

    Аннотации - это техника, введенная в Java 6. Это инструмент метапрограммирования, который не изменяет работу кода, но дает метки определенным элементам (методам, классам или переменным). Другими словами, аннотации - это просто маркеры, прикрепленные к коду, которые можно увидеть и прочитать. Некоторые аннотации предназначены только для просмотра во время компиляции - они не существуют в файлах .class после компиляции. Другие остаются видимыми после компиляции и могут быть использованы во время выполнения.

    Например, @Override относится к первому типу (его тип удержания SOURCE), в то время как @Test из JUnit относится ко второму типу (тип удержания RUNTIME). @Loggable - это аннотация второго типа, которую я использую в приведенном выше скрипте, из jcabi-aspects. Она остается в байт-коде в файле .class после компиляции.

    Опять же, важно понимать, что даже если метод power() аннотирован и скомпилирован, он пока не отправляет ничего в slf4j. Он просто содержит маркер, который говорит: “пожалуйста, записывайте мое выполнение”.

    Aspect Oriented Programming (AOP)

    AOP - это полезная техника, которая позволяет добавлять исполняемые блоки в исходный код без явного его изменения. В нашем примере мы не хотим регистрировать выполнение метода внутри класса. Вместо этого мы хотим, чтобы другой класс перехватывал каждый вызов метода power(), измерял время его выполнения и отправлял эту информацию в slf4j.

    Мы хотим, чтобы этот перехватчик понимал нашу аннотацию @Loggable и регистрировал каждый вызов этого конкретного метода power(). И, конечно же, этот же перехватчик должен использоваться для других методов, в которых мы будем размещать эту же аннотацию в будущем.

    Этот случай идеально соответствует первоначальному намерению AOP - избежать повторной реализации некоторого общего поведения в нескольких классах.

    Регистрация является дополнительной функцией нашей основной функциональности, и мы не хотим загромождать наш код несколькими инструкциями по регистрации. Вместо этого мы хотим, чтобы регистрация происходила в фоновом режиме.

    С точки зрения AOP, наше решение можно объяснить как создание аспекта, который пересекает код в определенных точках объединения и применяет совет вокруг, реализующий желаемую функциональность.

    AspectJ

    Давайте посмотрим, что означают эти волшебные слова. Но сначала давайте посмотрим, как jcabi-aspects их реализует с использованием AspectJ (это упрощенный пример, полный код вы можете найти в MethodLogger.java).

    @Aspect
public class MethodLogger {
  @Around("execution(* *(..)) && @annotation(Loggable)")
  public Object around(ProceedingJoinPoint point) {
    long start = System.currentTimeMillis();
    Object result = point.proceed();
    Logger.info(
      "#%s(%s): %s in %[msec]s",
      MethodSignature.class.cast(point.getSignature()).getMethod().getName(),
      point.getArgs(),
      result,
      System.currentTimeMillis() - start
    );
    return result;
  }
}

    Это аспект с единственным around advice around() внутри. Аспект аннотирован с помощью @Aspect, а совет аннотирован с помощью @Around. Как уже обсуждалось выше, эти аннотации являются только маркерами в файлах .class. Они ничего не делают, кроме как предоставляют некоторую мета-информацию тем, кто интересуется временем выполнения.

    Аннотация @Around имеет один параметр, который - в данном случае - говорит, что совет должен быть применен к методу, если:

    1. его модификатор видимости - * (public, protected или private);

    2. его имя - * (любое имя);

    3. его аргументы - .. (любые аргументы); и

    4. это помечено с помощью @Loggable

    Когда вызывается аннотированный метод, метод around() выполняется перед выполнением самого метода. Когда вызывается метод power(), метод around() получает экземпляр класса ProceedingJoinPoint и должен вернуть объект, который будет использоваться в результате метода power().

    Для вызова исходного метода power(), совету необходимо вызвать proceed() объекта точки соединения.

    Мы компилируем этот аспект и делаем его доступным в classpath вместе с нашим основным файлом Foo.class. Пока все хорошо, но нам нужно сделать последний шаг, чтобы применить наш совет в действие — мы должны применить наш совет.

    Binary Aspect Weaving

    Аспектное переплетение - это процесс применения советов. Аспектный переплетчик изменяет исходный код, внедряя вызовы аспектов. AspectJ делает именно это. Мы передаем ему два бинарных Java-класса Foo.class и MethodLogger.class; в ответ получаем три - измененный Foo.class, Foo$AjcClosure1.class и неизменный MethodLogger.class.

    Для определения советов, которые должны быть применены к определенным методам, аспектный переплетчик AspectJ использует аннотации из файлов .class. Он также использует отражение, чтобы просматривать все классы в пути поиска классов. Он анализирует, какие методы удовлетворяют условиям аннотации @Around. Конечно же, он находит наш метод power().

    Итак, есть два шага. Сначала мы компилируем наши файлы .java с помощью javac и получаем два файла. Затем AspectJ переплетает/изменяет их и создает свой собственный дополнительный класс. Наш класс Foo выглядит примерно так после переплетения:

    public class Foo {
  private final MethodLogger logger;
  @Loggable
  public int power(int x, int p) {
    // Prepares the "point" object
    return this.logger.around(point);
  }
  private int power_aroundBody(int x, int p) {
    return Math.pow(x, p);
  }
}

    Аспектный средство AspectJ перемещает нашу исходную функциональность в новый метод power_aroundBody() и перенаправляет все вызовы power() к аспектному классу MethodLogger.

    Вместо одного метода power() в классе Foo, у нас теперь четыре класса, работающих вместе. Отныне, это происходит за кадром при каждом вызове power():

    Как видите, процесс аспектного внедрения соединяет вместе классы и аспекты, передавая вызовы между ними через точки соединения. Без внедрения, как классы, так и аспекты являются только скомпилированными бинарными файлами Java с прикрепленными аннотациями.

    jcabi-aspects

    jcabi-aspects - это JAR-библиотека, которая содержит аннотацию Loggable и аспект MethodLogger (кстати, есть еще много других аспектов и аннотаций). Вам не нужно писать свой собственный аспект для логирования методов. Просто добавьте несколько зависимостей в ваш classpath и настройте jcabi-maven-plugin для аспектного вязания (получите их последние версии в Maven Central).

    <project>
  <dependencies>
    <dependency>
      <groupId>com.jcabi</groupId>
      <artifactId>jcabi-aspects</artifactId>
    </dependency>
    <dependency>
      <groupId>org.aspectj</groupId>
      <artifactId>aspectjrt</artifactId>
    </dependency>
  </dependencies>
  <build>
    <plugins>
      <plugin>
        <groupId>com.jcabi</groupId>
        <artifactId>jcabi-maven-plugin</artifactId>
        <executions>
          <execution>
            <goals>
              <goal>ajc</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
    </plugins>
  </build>
</project>

    Поскольку этот процесс плетения требует значительных усилий для настройки, я создал удобный плагин Maven с целью ajc, который выполняет всю работу по плетению аспектов. Вы можете использовать AspectJ напрямую, но я рекомендую использовать jcabi-maven-plugin.

    Вот и все. Теперь вы можете использовать аннотацию @com.jcabi.aspects.Loggable, и ваши методы будут записываться через slf4j.

    Если что-то не работает, как описано, не стесняйтесь сообщить о проблеме в GitHub issue.

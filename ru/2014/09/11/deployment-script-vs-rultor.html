    Deployment Script vs. Rultor

    Когда я объясняю, как Rultor автоматизирует процессы развертывания/релиза, очень часто слышу что-то вроде:

    Но у меня уже есть скрипт, который разворачивает все автоматически.

    Этот ответ очень распространен, поэтому я решил суммировать мои три основных аргумента в пользу автоматизированного процесса развертывания/выпуска Rultor в одной статье: 1) изолированные контейнеры Docker, 2) видимость логов и 3) безопасность учетных данных.

    Прочитайте о них и посмотрите, что дополнительного предлагает Rultor помимо вашего существующего скрипта(ов) развертывания.

    Прежде чем начать обсуждение, позвольте мне подчеркнуть, что Rultor - это полезный интерфейс для ваших собственных скриптов. Когда вы решаете автоматизировать развертывание с помощью Rultor, вы не отказываетесь от своих существующих скриптов. Вы просто научите Rultor их вызывать.

    Isolated Docker Containers

    Первое преимущество, которое вы получаете, начиная вызывать ваши скрипты развертывания из Rultor, это использование Docker. Я уверен, вы знаете, что такое Docker, но для тех, кто не знает - это менеджер виртуальных “машин” Linux. Это командный сценарий, который вы вызываете, когда вам нужно запустить какой-либо скрипт в новой виртуальной машине (также известной как “контейнер”). Docker запускает контейнер почти мгновенно и выполняет ваш скрипт. Преимущество Docker заключается в том, что каждый контейнер является полностью изолированной Linux-средой со своей собственной файловой системой, памятью, процессами и т.д.

    Когда вы говорите Rultor’у запустить ваш сценарий развертывания, он создает новый контейнер Docker и запускает ваш сценарий там. Но какая от этого выгода, вы спрашиваете?

    Основное преимущество заключается в том, что контейнер уничтожается сразу после завершения вашего скрипта. Это означает, что вы можете выполнять все предварительные настройки внутри контейнера без какой-либо опасности конфликта с основной рабочей платформой. Позвольте мне привести пример.

    Я разрабатываю на MacBook, где я устанавливаю и удаляю пакеты, необходимые мне для разработки. В то же время у меня есть проект, который для развертывания требует PHP 5.3, MySQL 5.6, Phing, PHPUnit, PHPCS и xdebug. Каждая версия MacOS должна быть настроена специально для запуска и работы этих приложений, и это занимает много времени.

    Я могу сменить ноутбуки и версии MacOS, но проект остается тем же самым. Для успешного запуска сценария развертывания по-прежнему требуется один и тот же набор пакетов. Кроме того, проект больше не находится в активной разработке. Мне просто не нужны эти пакеты для моей повседневной работы, так как я сейчас больше работаю с Java. Однако, когда мне нужно внести незначительное исправление в этот проект на PHP и развернуть его, мне приходится устанавливать все необходимые пакеты PHP и настраивать их. Только после этого я могу развернуть это небольшое исправление.

    Это, мягко говоря, раздражает.

    Docker дает мне возможность автоматизировать все это вместе. Мой существующий сценарий развертывания будет содержать преамбулу, которая установит и настроит все необходимые пакеты, связанные с PHP, в чистом контейнере Ubuntu. Эта преамбула будет выполняться при каждом запуске моего сценария развертывания, внутри контейнера Docker. Например, она может выглядеть так:

    Мой скрипт развертывания выглядел так до того, как я начал использовать Rultor:

    
# !/bin/bash

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

Всего две строки. Первая строка - полный набор модульных тестов. Вторая строка - развертывание через FTP на сервере продукции. Очень просто. Но этот скрипт будет работать только если установлены PHP 5.3, MySQL, Phing, xdebug, PHPCS и PHPUnit. Опять же, это много работы, чтобы установить и настроить их каждый раз, когда я обновляю MacOS или меняю ноутбук.

Нужно отметить, что если/когда кто-то присоединяется к проекту и пытается запустить мои скрипты, ему/ей придется выполнить эту предварительную установку снова.

Итак, вот новый скрипт, который я использую сейчас. Он выполняется внутри нового контейнера Docker каждый раз:

```bash

# !/bin/bash

# First, we install all prerequisites

sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

sudo apt-get установить -у php5 php5-mysql mysql sudo apt-get установить php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

# And now the same script I had before

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

Очевидно, запуск этого скрипта на моем MacBook (без виртуализации) вызовет много проблем. К тому же, у меня здесь даже нет `apt-get`.

Таким образом, первое преимущество, которое дает вам Rultor, - это изоляция вашего сценария развертывания в собственной виртуальной среде. Большую часть этого мы обязаны Docker.

## Visibility of Logs

Традиционно мы храним скрипты развертывания в каталоге `~/deploy` и запускаем их с помощью набора волшебных параметров. В небольшом проекте вы делаете это самостоятельно, а этот каталог находится на вашем собственном ноутбуке. В крупном проекте есть "сервер развертывания", на котором находится этот волшебный каталог со скриптами, которые могут быть выполнены только несколькими доверенными старшими разработчиками. Я часто видел такую настройку.

Самая большая проблема здесь - это отслеживаемость. Почти невозможно узнать, кто развернул что и почему произошла ошибка в конкретном развертывании. Старшие специалисты по развертыванию просто подключаются к серверу по SSH и запускают эти волшебные скрипты с волшебными параметрами. Журналы обычно теряются, и отслеживание проблемы становится очень сложным или невозможным.

Rultor предлагает нечто другое. С Rultor больше нет доступа по SSH к скриптам развёртывания. Все скрипты хранятся в файле конфигурации `.rultor.yml`, и вы запускаете их, отправляя сообщения в вашу систему отслеживания задач (например, GitHub, JIRA или Trac). Rultor запускает скрипт и публикует его полный журнал прямо в вашей задаче. Журнал остаётся с вашим проектом навсегда. Вы всегда можете вернуться к задаче, с которой работали, и проверить, почему развёртывание не удалось и какие инструкции были фактически выполнены.

Например, посмотрите на эту проблему на GitHub, где я развертывал новую версию самого Rultor и несколько раз не смог это сделать: [yegor256/rultor#563](https://github.com/yegor256/rultor/issues/563). Все мои неудачные попытки протоколируются. Я всегда могу вернуться к ним и провести расследование. Для большого проекта эта информация является жизненно важной.

Таким образом, второе преимущество Rultor по сравнению со скриптом автономного развертывания заключается в возможности просмотра каждой отдельной операции.

## Security of Credentials

Когда у вас есть настраиваемый сценарий на вашем ноутбуке или на том секретном сервере развертывания команды, ваши производственные учетные данные остаются рядом с ним. Просто нет другого способа. Если ваше программное обеспечение работает с базой данных, оно должно знать учетные данные для входа (имя пользователя, пароль, имя БД, номер порта и т. д.). Что ж, в худшем случае некоторые люди просто зашивают эту информацию прямо в исходный код. Мы даже не будем обсуждать этот случай, так плохо это.

Но предположим, что вы отделяете учетные данные БД от исходного кода. У вас будет что-то вроде файла `db.properties` или `db.ini`, который будет прикреплен к приложению непосредственно перед развертыванием. Вы также можете хранить этот файл непосредственно на сервере продакшена, что еще лучше, но не всегда возможно, особенно при развертывании в PaaS, например.

Аналогичная проблема существует с размещением артефактов в репозитории. Допустим, вы регулярно размещаете их на RubyGems.org. В файле `~/.gem/credentials` хранится ваш секретный ключ API.

Таким образом, очень часто ваши сценарии развертывания сопровождаются некоторыми файлами с чувствительной и защищенной информацией. И эти файлы содержат эту информацию в простом, открытом формате. Нет никакого шифрования, никакой защиты. Просто имена пользователей, пароли, коды и токены в виде обычного текста.

Почему это плохо? Ну, для одного разработчика с одним ноутбуком это не кажется проблемой. Хотя мне не нравится идея потерять ноутбук где-то в аэропорту со всеми открытыми и готовыми к использованию учетными данными. Вы можете возразить, что существуют инструменты для защиты диска, такие как FileVault для MacOS или BestCrypt для Windows. Да, возможно.

Но давайте посмотрим, что происходит, когда у нас есть команда разработчиков, работающих вместе и обменивающихся этими сценариями развертывания **и** файлами с учетными данными. Как только вы предоставляете доступ к вашим сценариям развертывания новому члену команды, вам приходится делиться всеми этими конфиденциальными данными. Просто нельзя обойти этот момент. Чтобы использовать сценарии, ему/ей нужно иметь возможность открывать файлы с учетными данными.

Это проблема, если вам важна безопасность ваших данных.

Rultor решает эту проблему, предлагая мгновенное расшифрование GPG ваших конфиденциальных данных, прямо перед тем, как они используются вашими скриптами развертывания. В конфигурационном файле `.rultor.yml` вы просто говорите:

    decrypt: db.ini: “repo/db.ini.asc” deploy: script: ftp put db.ini production ```

    Затем вы шифруете свой файл db.ini с помощью ключа GPG Rultor и смело коммитите файл db.ini.asc в репозиторий. Никто не сможет открыть и прочитать этот файл, за исключением самого сервера Rultor, непосредственно перед выполнением скрипта развертывания.

    Таким образом, третьим преимуществом Rultor по сравнению с отдельным сценарием развертывания является надлежащая безопасность конфиденциальных данных.

    Deployment Script vs. Rultor

    Когда я объясняю, как Rultor автоматизирует процессы развертывания/выпуска, очень часто я слышу что-то вроде:

    Но у меня уже есть скрипт, который автоматически развертывает все.

    Этот ответ очень распространенный, поэтому я решил суммировать мои три основных аргумента в пользу автоматизированных процессов развертывания/релиза Rultor в одной статье: 1) изолированные контейнеры Docker, 2) видимость журналов и 3) безопасность учетных данных.

    Прочитайте о них и посмотрите, что дает вам Rultor сверху вашего существующего сценария развертывания.

    Перед тем, как мы приступим к аргументам, позвольте мне подчеркнуть, что Rultor - это полезный интерфейс для ваших пользовательских сценариев. Когда вы решаете автоматизировать развертывание с помощью Rultor, вы не избавляетесь от своих существующих сценариев. Вы просто учитесь Rultor’у, как их вызывать.

    Isolated Docker Containers

    Первое преимущество, которое вы получаете после того, как начинаете вызывать свои скрипты развертывания из Rultor, - это использование Docker. Я уверен, что вы знаете, что такое Docker, но для тех, кто не знает - это менеджер виртуальных “машин” Linux. Это командный сценарий, который вы вызываете, когда вам нужно запустить какой-либо скрипт в новой виртуальной машине (так называемом “контейнере”). Docker запускает контейнер практически мгновенно и выполняет ваш скрипт. Преимущество Docker заключается в том, что каждый контейнер представляет собой полностью изолированную среду Linux с собственной файловой системой, памятью, процессами и т. д.

    Когда вы говорите Rultor’у выполнить ваш скрипт развёртывания, он запускает новый контейнер Docker и выполняет ваш скрипт внутри него. Но какая польза от этого, вы спрашиваете?

    Основное преимущество заключается в том, что контейнер уничтожается сразу после завершения вашего скрипта. Это означает, что вы можете выполнять все предварительные настройки внутри контейнера без каких-либо опасений конфликта с вашей основной рабочей платформой. Позвольте мне привести пример.

    Я разрабатываю на MacBook, где я устанавливаю и удаляю пакеты, которые мне нужны для разработки. В то же время у меня есть проект, который для развертывания требует PHP 5.3, MySQL 5.6, Phing, PHPUnit, PHPCS и xdebug. Каждая версия MacOS должна быть настроена специально, чтобы запустить эти приложения, и это занимает много времени.

    Я могу менять ноутбуки и версии MacOS, но проект остается тем же самым. Для успешного выполнения сценария развертывания все еще требуется один и тот же набор пакетов. И проект больше не находится в активной разработке. Мне просто не нужны эти пакеты для моей повседневной работы, так как я сейчас больше работаю с Java. Но, когда мне нужно внести небольшое исправление в этот проект на PHP и развернуть его, мне приходится устанавливать все необходимые пакеты PHP и настраивать их. Только после этого я могу развернуть это небольшое исправление.

    Это, мягко говоря, раздражает.

    Docker дает мне возможность автоматизировать все вместе. Мой существующий скрипт развертывания будет получать вступление, которое установит и настроит все необходимые пакеты, связанные с PHP, в чистом контейнере Ubuntu. Это вступление будет выполняться при каждом запуске моего скрипта развертывания, внутри контейнера Docker. Например, это может выглядеть так:

    Мой скрипт развертывания выглядел так до того, как я начал использовать Rultor:

    
# !/bin/bash

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

Всего две строки. Первая строка - полное выполнение модульных тестов. Вторая строка - развертывание по FTP на сервер производства. Очень просто. Но этот скрипт будет работать только если установлены PHP 5.3, MySQL, Phing, xdebug, PHPCS и PHPUnit. Опять же, это много работы, чтобы установить и настроить их каждый раз при обновлении MacOS или смене ноутбука.

Само собой разумеется, что если/когда кто-то присоединится к проекту и попытается запустить мои скрипты, он/она снова должны будут выполнить эту предварительную установку.

Итак, вот новый скрипт, который я использую сейчас. Он выполняется внутри нового Docker-контейнера каждый раз:

```bash

# !/bin/bash

# First, we install all prerequisites

sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

# And now the same script I had before

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

Очевидно, запуск этого скрипта на моем MacBook (без виртуализации) вызвал бы много проблем. Вообще, у меня здесь даже нет `apt-get`.

Таким образом, первое преимущество, которое дает вам Rultor, заключается в изоляции вашего сценария развертывания в собственной виртуальной среде. Это в основном достигается благодаря Docker.

## Visibility of Logs

Традиционно мы храним скрипты развертывания в какой-то директории `~/deploy` и запускаем их с помощью магического набора параметров. В небольшом проекте вы делаете это самостоятельно, и эта директория находится на вашем собственном ноутбуке. В большом проекте есть "сервер развертывания", который имеет эту магическую директорию со множеством скриптов, которые могут выполняться только несколькими доверенными старшими разработчиками. Я видел такую ​​настройку много раз.

Самая большая проблема здесь - это отслеживаемость. Почти невозможно узнать, кто развернул что и почему произошла ошибка при конкретном развертывании. Старшие гуру развертывания просто подключаются по SSH к серверу и запускают волшебные скрипты с волшебными параметрами. Журналы обычно теряются, и отслеживание проблем становится очень сложным или невозможным.

Rultor предлагает нечто отличное. С помощью Rultor больше нет доступа по SSH к скриптам развертывания. Все скрипты остаются в файле конфигурации `.rultor.yml`, и вы запускаете их, размещая сообщения в системе отслеживания проблем (например, GitHub, JIRA или Trac). Rultor запускает скрипт и публикует его полный журнал прямо в вашем тикете. Журнал остается с вашим проектом навсегда. Вы всегда можете вернуться к тикету, с которым работали, и проверить, почему развертывание не удалось и какие инструкции были фактически выполнены.

Например, посмотрите на эту проблему в GitHub, где я развертывал новую версию самого Rultor и несколько раз не смог выполнить это: [yegor256/rultor#563](https://github.com/yegor256/rultor/issues/563). Все мои неудачные попытки зарегистрированы. Я всегда могу вернуться к ним и исследовать. Для большого проекта эта информация является важной.

Таким образом, второе преимущество Rultor по сравнению с автономным сценарием развертывания - это видимость каждой отдельной операции.

## Security of Credentials

Когда у вас есть пользовательский скрипт на вашем ноутбуке или на том секретном сервере развертывания команды, ваши учетные данные для продакшна находятся рядом с ним. Просто нет никакого другого пути. Если ваше программное обеспечение работает с базой данных, оно должно знать учетные данные для входа (имя пользователя, пароль, имя БД, номер порта и т. д.). В худшем случае некоторые люди просто жестко закодируют эту информацию прямо в исходный код. Мы даже не собираемся обсуждать этот случай, потому что он настолько плох.

Но предположим, вы отдельно храните учетные данные для базы данных от исходного кода. У вас будет что-то вроде файла `db.properties` или `db.ini`, который будет прикреплен к приложению непосредственно перед развертыванием. Вы также можете хранить этот файл непосредственно на производственном сервере, что даже лучше, но не всегда возможно, особенно при использовании PaaS-развертывания, например.

Подобная проблема существует с развертыванием артефактов в репозитории. Допустим, вы регулярно разворачиваете на RubyGems.org. Ваш `~/.gem/credentials` будет содержать ваш секретный ключ API.

Таким образом, очень часто ваши сценарии развертывания сопровождаются некоторыми файлами с конфиденциальной и защищенной информацией. И эти файлы содержат эту информацию в обычном, открытом формате. Без шифрования, без защиты. Просто имена пользователей, пароли, коды и токены в виде обычного текста.

Почему это плохо? Что ж, для одного разработчика с одним ноутбуком это не кажется проблемой. Хотя я не люблю идею потерять ноутбук где-то в аэропорту с открытыми и готовыми к использованию учетными данными. Можно возразить, что есть инструменты защиты дисков, такие как FileVault для MacOS или BestCrypt для Windows. Да, возможно.

Но давайте посмотрим, что происходит, когда у нас есть команда разработчиков, работающих вместе и обменивающихся этими скриптами **и** файлами с учетными данными. Как только вы предоставляете доступ к своим скриптам для развертывания новому участнику команды, вам приходится делиться всей этой конфиденциальной информацией. Просто нельзя этого избежать. Чтобы использовать эти скрипты, он/она должен/должна иметь возможность открывать файлы с учетными данными.

Это проблема, если вам важна безопасность ваших данных.

Рultor решает эту проблему, предлагая мгновенное расшифрование GPG ваших конфиденциальных данных, прямо перед их использованием в ваших сценариях развертывания. В конфигурационном файле `.rultor.yml` вы просто говорите:

    decrypt: db.ini: “repo/db.ini.asc” deploy: script: ftp put db.ini production ```

    Затем вы шифруете свой db.ini с помощью Rultor GPG-ключа и смело фиксируете db.ini.asc в репозитории. Никто не сможет открыть и прочитать этот файл, кроме самого сервера Rultor, непосредственно перед запуском скрипта развертывания.

    Таким образом, третьим преимуществом Rultor перед автономным сценарием развертывания является надлежащая безопасность конфиденциальных данных.

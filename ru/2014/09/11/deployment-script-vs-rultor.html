    Deployment Script vs. Rultor

    Когда я объясняю, как Rultor автоматизирует процессы развертывания/выпуска, очень часто слышу что-то вроде:

    Но у меня уже есть скрипт, который автоматически разворачивает все.

    Этот ответ очень распространен, поэтому я решил суммировать мои три основных аргумента в пользу автоматизированного процесса развертывания/выпуска Rultor в одной статье: 1) изолированные контейнеры Docker, 2) видимость журналов и 3) безопасность учетных данных.

    Прочтите о них и посмотрите, что дает вам Rultor в дополнение к вашим существующим скриптам развертывания.

    Прежде чем начать аргументацию, позвольте мне подчеркнуть, что Rultor является полезным интерфейсом для ваших пользовательских скриптов. Когда вы решаете автоматизировать развертывание с помощью Rultor, вы не отказываетесь от своих существующих скриптов. Вы просто обучаете Rultor’у, как их вызывать.

    Isolated Docker Containers

    Первое преимущество, которое вы получаете, начиная вызывать ваши сценарии развертывания из Rultor, это использование Docker. Я уверен, что вы знаете, что такое Docker, но для тех, кто не знает, - это менеджер виртуальных Linux “машин”. Это командный скрипт, который вы вызываете, когда вам нужно запустить какой-то сценарий в новой виртуальной машине (так называемом “контейнере”). Docker запускает контейнер практически мгновенно и выполняет ваш сценарий. Преимущество Docker заключается в том, что каждый контейнер представляет собой полностью изолированную Linux-среду с собственной файловой системой, памятью, процессами и т. д.

    Когда вы указываете Rultor выполнить ваш скрипт развертывания, он запускает новый контейнер Docker и выполняет ваш скрипт в нем. Но какая польза от этого, вы спрашиваете?

    Основное преимущество заключается в том, что контейнер уничтожается сразу после завершения вашего сценария. Это означает, что вы можете выполнить все предварительные настройки внутри контейнера, не беспокоясь о конфликтах с вашей основной рабочей платформой. Позвольте мне привести пример.

    Я разрабатываю на MacBook, где устанавливаю и удаляю пакеты, необходимые для разработки. В то же время у меня есть проект, который для развертывания требует PHP 5.3, MySQL 5.6, Phing, PHPUnit, PHPCS и xdebug. Каждая версия MacOS требует конкретной настройки для запуска и работы этих приложений, и это занимает много времени.

    Я могу сменить ноутбук и версии MacOS, но проект остаётся тем же самым. Для успешного запуска скрипта развёртывания все ещё требуется тот же набор пакетов. Кроме того, проект больше не находится в активной разработке. Мне просто не нужны эти пакеты для моей повседневной работы, так как я сейчас больше работаю с Java. Но если мне нужно внести небольшое исправление в этот проект на PHP и задеплоить его, мне приходится устанавливать все необходимые PHP-пакеты и настраивать их. Только после этого я могу задеплоить это небольшое исправление.

    Это раздражает, мягко говоря.

    Docker дает мне возможность автоматизировать все это вместе. Мой существующий скрипт развертывания получит предисловие, которое установит и настроит все необходимые пакеты, связанные с PHP, в чистом контейнере Ubuntu. Это предисловие будет выполняться при каждом запуске моего скрипта развертывания, внутри Docker контейнера. Например, это может выглядеть так:

    Мой сценарий развёртывания выглядел так до того, как я начал использовать Rultor:

    
# !/bin/bash

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

Только две строки. Первая - полный запуск модульных тестов. Вторая - развертывание по FTP на сервере продукции. Очень просто. Но этот скрипт будет работать только если установлены PHP 5.3, MySQL, Phing, xdebug, PHPCS и PHPUnit. Опять же, это много работы, чтобы установить и настроить их каждый раз, когда я обновляю MacOS или меняю ноутбук.

Само собой разумеется, что если/когда кто-то присоединяется к проекту и пытается запустить мои скрипты, ему/ей придется повторно выполнить эту предустановочную работу.

Итак, вот новый скрипт, который я сейчас использую. Он выполняется внутри нового контейнера Docker каждый раз:

```bash

# !/bin/bash

# First, we install all prerequisites

sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

# And now the same script I had before

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

Очевидно, запуск этого скрипта на моем MacBook (без виртуализации) вызвал бы много проблем. Ну, у меня здесь даже нет `apt-get`.

Таким образом, первое преимущество, которое дает вам Rultor, - это изоляция вашего сценария развертывания в собственной виртуальной среде. Большая часть этого достигается благодаря Docker.

## Visibility of Logs

Традиционно, мы храним скрипты развертывания в директории `~/deploy` и запускаем их с помощью набора волшебных параметров. В небольшом проекте вы делаете это самостоятельно, и эта директория находится на вашем собственном ноутбуке. В более крупном проекте есть "сервер развертывания", на котором находится волшебная директория с набором скриптов, которые могут выполняться только несколькими доверенными старшими разработчиками. Я видел такую ​​настройку много раз.

Самая большая проблема здесь - это трассируемость. Почти невозможно выяснить, кто и зачем развернул тот или иной проект, и почему конкретное развертывание не удалось. Старшие гуру развертывания просто подключаются по SSH к серверу и запускают эти волшебные сценарии с волшебными параметрами. Часто логи теряются, и отслеживание проблемы становится очень трудным или невозможным.

Rultor предлагает нечто иное. С Rultor больше нет доступа по SSH к скриптам развёртывания. Все скрипты остаются в файле конфигурации `.rultor.yml`, и вы запускаете их, отправляя сообщения в вашу систему отслеживания задач (например, GitHub, JIRA или Trac). Rultor запускает скрипт и публикует его полный журнал прямо в вашем тикете. Журнал остаётся с вашим проектом навсегда. Вы всегда можете вернуться к тикету, с которым работали, и проверить, почему развёртывание не удалось и какие инструкции были фактически выполнены.

Например, посмотрите на эту проблему в GitHub, где я развертывал новую версию самого Rultor и несколько раз неудачно: [yegor256/rultor#563](https://github.com/yegor256/rultor/issues/563). Все мои неудачные попытки протоколируются. Я всегда могу вернуться к ним и исследовать. Для большого проекта эта информация важна.

Таким образом, второе преимущество Rultor по сравнению со скриптом отдельного развертывания - это видимость каждой отдельной операции.

## Security of Credentials

Когда у вас есть пользовательский скрипт на вашем ноутбуке или на том секретном сервере развертывания команды, ваши учетные данные для продакшна остаются рядом с ним. Просто нет другого способа. Если ваше программное обеспечение работает с базой данных, оно должно знать учетные данные для входа (имя пользователя, пароль, имя БД, номер порта и т. д.). В худшем случае некоторые люди просто жестко закодируют эту информацию прямо в исходный код. Мы даже не будем обсуждать этот случай, потому что он настолько плох.

Но предположим, вы отдельно храните учетные данные базы данных от исходного кода. У вас будет что-то вроде файла `db.properties` или `db.ini`, который будет прикреплен к приложению непосредственно перед развертыванием. Вы также можете хранить этот файл непосредственно на сервере продакшн, что еще лучше, но не всегда возможно, особенно с развертыванием на платформе как услуга, например.

Подобная проблема существует с размещением артефактов в репозиториях. Допустим, вы регулярно размещаете их на RubyGems.org. В файле `~/.gem/credentials` будут содержаться ваши секретные API-ключи.

Часто скрипты развертывания сопровождаются файлами, содержащими чувствительную и защищенную информацию. И эти файлы содержат эту информацию в простом, открытом формате. Без шифрования, без защиты. Просто имена пользователей, пароли, коды и токены в виде обычного текста.

Почему это плохо? Ну, для одного разработчика с одним ноутбуком это не звучит как проблема. Хотя я не люблю идею потерять ноутбук где-то в аэропорту, когда все учетные данные открыты и готовы к использованию. Вы можете возразить, что существуют инструменты для защиты диска, такие как FileVault для MacOS или BestCrypt для Windows. Да, возможно.

Но давайте посмотрим, что происходит, когда у нас есть команда разработчиков, работающих вместе и обменивающихся этими скриптами развертывания **и** файлами со своими учетными данными. Как только вы даете доступ к скриптам развертывания новому участнику команды, вам приходится делиться всей этой чувствительной информацией. Просто нет другого пути. Для использования скриптов ему/ей необходимо иметь возможность открывать файлы со своими учетными данными.

Это проблема, если вам важна безопасность ваших данных.

Рултор решает эту проблему, предлагая мгновенное расшифрование ваших конфиденциальных данных с помощью GPG прямо перед их использованием в ваших скриптах развертывания. В конфигурационном файле `.rultor.yml` вы просто указываете:

    decrypt: db.ini: “repo/db.ini.asc” deploy: script: ftp put db.ini production ```

    Затем вы шифруете свой db.ini с помощью ключа GPG Rultor, и смело коммитите db.ini.asc в репозиторий. Никто не сможет открыть и прочитать этот файл, кроме самого сервера Rultor, непосредственно перед запуском сценария развертывания.

    Таким образом, третьим преимуществом Rultor по сравнению с отдельным сценарием развёртывания является надлежащая безопасность конфиденциальных данных.

    Deployment Script vs. Rultor

    Когда я объясняю, как Rultor автоматизирует процессы развертывания/выпуска, очень часто я слышу что-то вроде:

    Но у меня уже есть скрипт, который автоматически разворачивает все.

    Этот ответ очень распространен, поэтому я решил суммировать свои три основных аргумента в пользу автоматизированных процессов развертывания/релиза в Rultor в одной статье: 1) изолированные контейнеры Docker, 2) видимость логов и 3) безопасность учетных данных.

    Прочтите о них и посмотрите, что Rultor предоставляет вам в дополнение к вашим существующим скриптам развертывания.

    Прежде чем мы начнем с аргументами, позвольте мне подчеркнуть, что Rultor - полезный интерфейс для ваших пользовательских скриптов. Когда вы решите автоматизировать развертывание с помощью Rultor, вы не выбрасываете свои существующие скрипты. Вы просто научите Rultor их вызывать.

    Isolated Docker Containers

    Первое преимущество, которое вы получаете, как только начинаете вызывать свои скрипты развёртывания из Rultor, это использование Docker. Я уверен, что вы знаете, что такое Docker, но для тех, кто не знает — это менеджер виртуальных “машин” Linux. Это командная строка, которую вы вызываете, когда вам нужно выполнить некоторый скрипт в новой виртуальной машине (так называемом “контейнере”). Docker запускает контейнер практически мгновенно и выполняет ваш скрипт. Красота Docker в том, что каждый контейнер - это полностью изолированная среда Linux со своей собственной файловой системой, памятью, процессами и т.д.

    Когда вы говорите Rultorу запустить ваш скрипт развертывания, он создает новый контейнер Docker и запускает ваш скрипт внутри него. Но какая польза от этого, вы спрашиваете?

    Основное преимущество заключается в том, что контейнер уничтожается сразу после завершения вашего скрипта. Это означает, что вы можете выполнять все предварительные настройки внутри контейнера без каких-либо опасений конфликта с вашей основной рабочей платформой. Позвольте мне привести пример.

    Я разрабатываю на MacBook, где устанавливаю и удаляю пакеты, необходимые для разработки. В то же время у меня есть проект, который для развертывания требует PHP 5.3, MySQL 5.6, Phing, PHPUnit, PHPCS и xdebug. Каждая версия MacOS требует отдельной настройки для запуска этих приложений, и это занимает много времени.

    Я могу менять ноутбуки и версии MacOS, но проект остается неизменным. Для успешного запуска его скрипта развертывания все равно требуется один и тот же набор пакетов. И проект больше не находится в активной разработке. Мне просто не нужны эти пакеты для моей повседневной работы, так как я сейчас больше работаю с Java. Но, когда мне нужно внести небольшое исправление в этот проект на PHP и развернуть его, мне приходится устанавливать все необходимые пакеты PHP и настраивать их. Только после этого я могу развернуть это небольшое исправление.

    Это раздражает, мягко говоря.

    Docker дает мне возможность автоматизировать все это вместе. Мой существующий сценарий развертывания будет иметь преамбулу, которая установит и настроит все необходимые пакеты, связанные с PHP, в чистом контейнере Ubuntu. Эта преамбула будет выполняться при каждом запуске моего сценария развертывания, внутри контейнера Docker. Например, она может выглядеть так:

    Мой скрипт развёртывания выглядел так, прежде чем я начал использовать Rultor:

    
# !/bin/bash

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

Всего две строки. Первая - это полный запуск модульных тестов. Вторая - развертывание по FTP на сервере продакшена. Очень просто. Но этот скрипт будет работать только если установлены PHP 5.3, MySQL, Phing, xdebug, PHPCS и PHPUnit. Опять же, это много работы, чтобы установить и настроить их каждый раз, когда я обновляю MacOS или меняю ноутбук.

Само собой разумеется, что если/когда кто-то присоединится к проекту и попытается запустить мои скрипты, ему/ей придется выполнить эту предварительную установку заново.

Итак, вот новый скрипт, который я использую сейчас. Он выполняется внутри нового контейнера Docker каждый раз:

```bash

# !/bin/bash

# First, we install all prerequisites

sudo apt-get install -y php5 php5-mysql mysql sudo apt-get install php-pear sudo pear channel-discover pear.phpunit.de sudo pear install phpunit/PHPUnit sudo pear install PHP_CodeSniffer sudo pecl install xdebug sudo pear channel-discover pear.phing.info sudo pear install phing/phing

# And now the same script I had before

phing test git ftp push --user ".." \ --passwd ".." \ --syncroot php/src \ ftp://ftp.example.com/ ```

Очевидно, запуск этого скрипта на моем MacBook (без виртуализации) вызвал бы много проблем. Ведь у меня даже нет `apt-get` здесь.

Таким образом, первое преимущество, которое дает вам Rultor, заключается в изоляции вашего сценария развертывания в собственной виртуальной среде. Это в значительной степени достигается благодаря Docker.

## Visibility of Logs

Традиционно, мы храним скрипты развертывания в директории `~/deploy` и запускаем их с помощью волшебного набора параметров. В небольшом проекте вы делаете это самостоятельно, и эта директория находится на вашем собственном ноутбуке. В большом проекте есть "сервер развертывания", который имеет эту волшебную директорию со набором скриптов, которые могут быть выполнены только несколькими доверенными старшими разработчиками. Я видел такую ​​настройку много раз.

Самая большая проблема здесь - это трассируемость. Почти невозможно определить, кто развернул что и почему произошла ошибка в конкретном развертывании. Старшие гуру развертывания просто подключаются через SSH к серверу и запускают эти волшебные скрипты с волшебными параметрами. Журналы обычно теряются, и отслеживание проблемы становится очень сложным или невозможным.

Rultor предлагает нечто другое. С Rultor больше нет доступа через SSH к скриптам развертывания. Все скрипты остаются в файле конфигурации `.rultor.yml`, и вы запускаете их, отправляя сообщения в вашу систему отслеживания проблем (например, GitHub, JIRA или Trac). Rultor выполняет скрипт и публикует его полный журнал прямо в вашей задаче. Журнал остается с вашим проектом навсегда. Вы всегда можете вернуться к задаче, с которой работали, и проверить, почему развертывание не удалось и какие инструкции были фактически выполнены.

Например, посмотрите этот вопрос на GitHub, где я развертывал новую версию самого Rultor, и несколько раз неудачно: [yegor256/rultor#563](https://github.com/yegor256/rultor/issues/563). Все мои неудачные попытки были зарегистрированы. Я всегда могу вернуться к ним и исследовать. Для большого проекта эта информация является важной.

Таким образом, второе преимущество Rultor по сравнению со скриптом независимой развертки - видимость каждой отдельной операции.

## Security of Credentials

Когда у вас есть пользовательский скрипт на вашем ноутбуке или на секретном сервере развертывания команды, ваши производственные учетные данные находятся рядом с ним. Просто нет другого способа. Если ваше программное обеспечение работает с базой данных, оно должно знать данные для входа (имя пользователя, пароль, имя БД, номер порта и т. д.). Что ж, в худшем случае некоторые люди просто жестко закодируют эту информацию прямо в исходный код. Мы даже не собираемся обсуждать этот случай, так плохо это.

Но предположим, что вы отделите учетные данные для базы данных от исходного кода. У вас будет что-то вроде файла `db.properties` или `db.ini`, который будет прикреплен к приложению непосредственно перед развертыванием. Вы также можете хранить этот файл непосредственно на сервере продакшн, что является еще лучшим вариантом, хотя и не всегда возможным, особенно при использовании PaaS-развертывания, например.

Подобная проблема возникает при размещении артефактов в репозитории. Скажем, вы регулярно размещаете их на RubyGems.org. В файле `~/.gem/credentials` будет содержаться ваш секретный API-ключ.

Таким образом, очень часто ваши скрипты развертывания сопровождаются некоторыми файлами с чувствительной и защищенной информацией. И эти файлы содержат эту информацию в обычном открытом формате. Никакого шифрования, никакой защиты. Просто имена пользователей, пароли, коды и токены в виде обычного текста.

Почему это плохо? Ну, для одного разработчика с одним ноутбуком это не кажется проблемой. Но я не люблю идею потери ноутбука где-то в аэропорту, когда все учетные данные открыты и готовы к использованию. Вы можете возразить, что есть инструменты защиты дисков, такие как FileVault для MacOS или BestCrypt для Windows. Да, возможно.

Но давайте посмотрим, что происходит, когда у нас есть команда разработчиков, работающих вместе и обменивающихся этими сценариями развертывания **и** файлами с учетными данными. Как только вы предоставляете доступ к вашим сценариям развертывания новому участнику команды, вам нужно передавать всю эту конфиденциальную информацию. Просто нет другого пути. Чтобы использовать сценарии, он/она должен быть в состоянии открывать файлы с учетными данными.

Это проблема, если вам важна безопасность ваших данных.

Rultor решает эту проблему, предлагая мгновенное расшифрование GPG ваших конфиденциальных данных, прямо перед их использованием в ваших сценариях развертывания. В конфигурационном файле `.rultor.yml` вы просто говорите:

    decrypt: db.ini: “repo/db.ini.asc” deploy: script: ftp put db.ini production ```

    Затем вы шифруете ваш db.ini с помощью Rultor GPG ключа и смело коммитите db.ini.asc в репозиторий. Никто не сможет открыть и прочитать этот файл, кроме самого сервера Rultor, непосредственно перед запуском сценария развертывания.

    Таким образом, третьим преимуществом Rultor по сравнению с отдельным сценарием развертывания является надлежащая безопасность конфиденциальных данных.

    OOP Alternative to Utility Classes

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Утилитарный класс (или класс-помощник) - это “структура”, которая имеет только статические методы и не инкапсулирует состояние. StringUtils, IOUtils, FileUtils из Apache Commons; Iterables и Iterators из Guava, и Files из JDK7 являются идеальными примерами утилитарных классов.

    Эта концепция дизайна очень популярна в мире Java (а также в C#, Ruby и т.д.), так как утилитарные классы предоставляют общий функционал, который используется повсеместно.

    Здесь мы хотим следовать принципу DRY и избегать дублирования. Поэтому мы помещаем общие блоки кода в утилитарные классы и повторно используем их при необходимости.

    // This is a terrible design, don't reuse
public class NumberUtils {
  public static int max(int a, int b) {
    return a > b ? a : b;
  }
}

    Действительно, это очень удобная техника!?

    Utility Classes Are Evil

    Однако, в объектно-ориентированном мире утилитарные классы считаются очень плохой (некоторые даже скажут “ужасной”) практикой.

    Было много обсуждений этой темы, чтобы назвать несколько: Are Helper Classes Evil? Ник Малик, Why helper, singletons and utility classes are mostly bad Саймон Харт, Avoiding Utility Classes Маршал Уорд, Kill That Util Class! Дхавал Далал, Helper Classes Are A Code Smell Роб Багби.

    Кроме того, на StackExchange есть несколько вопросов о вспомогательных классах: Если класс “Утилиты” плох, где я должен разместить свой общий код?, Вспомогательные классы - это зло.

    Сухое изложение всех их аргументов заключается в том, что утилитарные классы не являются настоящими объектами; поэтому они не вписываются в объектно-ориентированный мир. Они были унаследованы от процедурного программирования, в основном потому, что мы привыкли к парадигме функционального разложения на тот момент.

    Предполагая, что вы согласны с аргументами и хотите перестать использовать утилитарные классы, я покажу на примере, как эти сущности могут быть заменены правильными объектами.

    Procedural Example

    Скажем, вы хотите прочитать текстовый файл, разделить его на строки, обрезать каждую строку, а затем сохранить результаты в другой файл. Это можно сделать с помощью FileUtils из Apache Commons.

    void transform(File in, File out) {
  Collection<String> src = FileUtils.readLines(in, "UTF-8");
  Collection<String> dest = new ArrayList<>(src.size());
  for (String line : src) {
    dest.add(line.trim());
  }
  FileUtils.writeLines(out, dest, "UTF-8");
}

    Выглядит, что код выше выглядит аккуратно; однако это процедурное программирование, а не объектно-ориентированное. Мы манипулируем данными (байтами и битами) и явно указываем компьютеру, откуда их извлекать, а затем куда помещать на каждой отдельной строке кода. Мы определяем процедуру выполнения.

    Object-Oriented Alternative

    В объектно-ориентированной парадигме мы должны создавать и объединять объекты, позволяя им управлять данными так, как они считают нужным. Вместо вызова дополнительных статических функций, мы должны создавать объекты, способные предоставлять нужное нам поведение.

    public class Max implements Number {
  private final int a;
  private final int b;
  public Max(int x, int y) {
    this.a = x;
    this.b = y;
  }
  @Override
  public int intValue() {
    return this.a > this.b ? this.a : this.b;
  }
}

    Этот процедурный вызов:

    int max = NumberUtils.max(10, 5);

    Станет объектно-ориентированным:

    int max = new Max(10, 5).intValue();

    Картошка, картошка? Не совсем; просто продолжайте чтение…

    Objects Instead of Data Structures

    Вот как я бы разработал ту же функциональность преобразования файлов, но с использованием объектно-ориентированного подхода:

    void transform(File in, File out) {
  Collection<String> src = new Trimmed(
    new FileLines(new UnicodeFile(in))
  );
  Collection<String> dest = new FileLines(
    new UnicodeFile(out)
  );
  dest.addAll(src);
}

    FileLines implements Collection<String> and encapsulates all file reading and writing operations. An instance of FileLines behaves exactly as a collection of strings and hides all I/O operations. When we iterate it—a file is being read. When we addAll() to it—a file is being written.

    Trimmed also implements Collection<String> and encapsulates a collection of strings (Decorator pattern). Every time the next line is retrieved, it gets trimmed.

    Все классы, участвующие в этом кодовом фрагменте, довольно маленькие: Trimmed, FileLines и UnicodeFile. Каждый из них отвечает за свою собственную одну функцию и отлично придерживается принципа единственной ответственности.

    С нашей стороны, как пользователя библиотеки, это может быть не так важно, но для их разработчиков это обязательно. Гораздо проще разрабатывать, поддерживать и тестировать класс FileLines, чем использовать метод readLines() в утилитарном классе FileUtils с более чем 80 методами и 3000 строками кода. Посмотрите его исходный код, серьезно.

    Объектно-ориентированный подход позволяет отложенное выполнение. Файл in не считывается, пока его данные не понадобятся. Если возникает ошибка ввода-вывода при открытии файла out, первый файл вообще не будет затронут. Весь процесс начинается только после вызова addAll().

    Все строки во втором отрывке, кроме последней, создают и объединяют маленькие объекты в большие. Компоновка объектов в данном случае дешева для процессора, так как она не вызывает никаких преобразований данных.

    Кроме того, очевидно, что второй скрипт выполняется в пространстве O(1), в то время как первый скрипт работает в O(n). Это является последствием нашего процедурного подхода к данным в первом скрипте.

    В мире объектно-ориентированного программирования нет данных; есть только объекты и их поведение!

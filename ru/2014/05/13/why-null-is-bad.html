    Why NULL is Bad?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Простой пример использования NULL в Java:

    public Employee getByName(String name) {
  int id = database.find(name);
  if (id == 0) {
    return null;
  }
  return new Employee(id);
}

    Что не так с этим методом?

    Он может вернуть NULL вместо объекта — в этом и заключается проблема. NULL является ужасной практикой в объектно-ориентированной парадигме и должно быть избегано во всех случаях. Об этом уже было опубликовано несколько мнений, включая презентацию Null References, The Billion Dollar Mistake Тони Хоара и целую книгу Object Thinking Дэвида Уэста.

    Здесь я постараюсь подвести итоги всех аргументов и показать примеры того, как использование NULL можно избежать и заменить на соответствующие объектно-ориентированные конструкции.

    В основном, существует два возможных варианта вместо NULL.

    Первый из них - паттерн проектирования Null Object (лучший способ - сделать его константой):

    public Employee getByName(String name) {
  int id = database.find(name);
  if (id == 0) {
    return Employee.NOBODY;
  }
  return Employee(id);
}

    Второй возможный вариант - быстрое завершение работы путем генерации исключения при невозможности вернуть объект.

    public Employee getByName(String name) {
  int id = database.find(name);
  if (id == 0) {
    throw new EmployeeNotFoundException(name);
  }
  return Employee(id);
}

    Теперь давайте рассмотрим аргументы против NULL.

    Помимо презентации Тони Хоара и книги Дэвида Уэста, упомянутых выше, я прочитал следующие публикации перед написанием этого поста: Чистый код Роберта Мартина, Полное кодирование Стива Макконнелла, Скажите “нет” “null” Джона Сонмеза, Возврат null - плохой дизайн? дискуссия на Stack Overflow.

    Ad-hoc Error Handling

    Каждый раз, когда вы получаете объект в качестве входных данных, вы должны проверить, является ли он NULL или допустимой ссылкой на объект. Если вы забудете проверить, NullPointerException (NPE) может привести к остановке выполнения программы во время выполнения. Таким образом, ваша логика становится загрязненной множественными проверками и ветвлениями if/then/else.

    // this is a terrible design, don't reuse
Employee employee = dept.getByName("Jeffrey");
if (employee == null) {
  System.out.println("can't find an employee");
  System.exit(-1);
} else {
  employee.transferTo(dept2);
}

    Вот как исключительные ситуации должны обрабатываться в C и других императивных процедурных языках. ООП в первую очередь представил обработку исключений для избавления от этих блоков обработки ошибок на ходу. В ООП мы позволяем исключениям подниматься вверх, пока они не достигнут обработчика ошибок, применяемого на уровне приложения, и наш код становится намного более чистым и короче.

    dept.getByName("Jeffrey").transferTo(dept2);

    Рассмотрите NULL ссылки как наследие процедурного программирования и используйте 1) Null объекты или 2) Исключения вместо них.

    Ambiguous Semantic

    Для явного выражения своего значения функция getByName() должна быть названа getByNameOrNullIfNotFound(). То же самое должно происходить с каждой функцией, которая возвращает объект или NULL. В противном случае, неизбежна неоднозначность для читателя кода. Таким образом, чтобы сохранить семантическую однозначность, следует давать функциям более длинные названия.

    Чтобы избавиться от этой неоднозначности, всегда возвращайте реальный объект, пустой объект или вызывайте исключение.

    Некоторые могут утверждать, что иногда нам приходится возвращать NULL в целях повышения производительности. Например, метод get() интерфейса Map в Java возвращает NULL, когда в карте отсутствует такой элемент.

    Employee employee = employees.get("Jeffrey");
if (employee == null) {
  throw new EmployeeNotFoundException();
}
return employee;

    Этот код ищет значение в карте только один раз из-за использования NULL в Map. Если бы мы переработали Map таким образом, чтобы его метод get() выбрасывал исключение, если ничего не найдено, наш код будет выглядеть так:

    if (!employees.containsKey("Jeffrey")) { // first search
  throw new EmployeeNotFoundException();
}
return employees.get("Jeffrey"); // second search

    Очевидно, что этот метод вдвое медленнее первого. Что делать?

    Интерфейс Map (без обид для его авторов) имеет ошибку в дизайне. Его метод get() должен был возвращать Iterator, чтобы наш код выглядел так:

    Iterator found = Map.search("Jeffrey");
if (!found.hasNext()) {
  throw new EmployeeNotFoundException();
}
return found.next();

    Кстати, именно таким образом разработан метод map::find() в C++ STL.

    Computer Thinking vs. Object Thinking

    Утверждение if (employee == null) понимается тем, кто знает, что объект в Java является указателем на структуру данных, а NULL - указатель на ничто (0x00000000 в процессорах Intel x86).

    Однако, если начать думать в терминах объектов, эта утверждение перестает быть столь же понятной. Так выглядит наш код с точки зрения объекта:

    - Hello, is it a software department?
- Yes.
- Let me talk to your employee "Jeffrey" please.
- Hold the line please...
- Hello.
- Are you NULL?

    Последний вопрос в этом разговоре звучит странно, не так ли?

    Вместо этого, если они положат трубку после нашей просьбы поговорить с Джеффри, это вызывает проблему для нас (исключение). В этот момент мы пытаемся позвонить снова или сообщить нашему руководителю, что не можем связаться с Джеффри и завершить более крупную сделку.

    Возможно, они могут позволить нам поговорить с другим человеком, который не является Джеффри, но может помочь с большинством наших вопросов или отказаться помочь, если нам нужна информация, связанная исключительно с Джеффри (Null Object).

    Slow Failing

    Вместо быстрого сбоя, указанный выше код пытается умирать медленно, убивая других на своем пути. Вместо того, чтобы дать всем знать, что что-то пошло не так и что обработка исключения должна начаться немедленно, он скрывает эту ошибку от своего клиента.

    Этот аргумент близок к обсуждаемому ранее “ад-хок обработке ошибок”.

    Это хорошая практика сделать ваш код максимально хрупким, позволяя ему сломаться, когда это необходимо.

    Сделайте ваши методы чрезвычайно требовательными к данным, с которыми они работают. Позвольте им выразить свое неудовлетворение, выбрасывая исключения, если предоставленные данные недостаточны или просто не соответствуют основному сценарию использования метода.

    В противном случае возвращается объект Null, который показывает некоторое общее поведение и генерирует исключения при всех остальных вызовах.

    public Employee getByName(String name) {
  int id = database.find(name);
  Employee employee;
  if (id == 0) {
    employee = new Employee() {
      @Override
      public String name() {
        return "anonymous";
      }
      @Override
      public void transferTo(Department dept) {
        throw new AnonymousEmployeeException(
          "I can't be transferred, I'm anonymous"
        );
      }
    };
  } else {
    employee = Employee(id);
  }
  return employee;
}

    Say, you are designing a method findUserByName(), which has to find a user in the database. What would you return if nothing is found? #elegantobjects

    Mutable and Incomplete Objects

    В общем, настоятельно рекомендуется проектировать объекты с учетом неизменяемости. Это означает, что объект получает все необходимые знания при создании и никогда не изменяет свое состояние на протяжении всего жизненного цикла.

    Очень часто значения NULL используются при ленивой загрузке, чтобы сделать объекты неполными и изменяемыми. Например:

    public class Department {
  private Employee found = null;
  public synchronized Employee manager() {
    if (this.found == null) {
      this.found = new Employee("Jeffrey");
    }
    return this.found;
  }
}

    Эта технология, хоть и широко используется, является анти-паттерном в ООП. В основном, потому что она делает объект ответственным за проблемы производительности вычислительной платформы, о чем объект “Сотрудник” не должен знать.

    Вместо управления состоянием и представления связанного с бизнесом поведения, объект должен заботиться о кэшировании собственных результатов - в этом суть ленивой загрузки.

    Кэширование - это не то, что сотрудник делает в офисе, верно?

    Решение? Не используйте ленивую загрузку таким примитивным образом, как в приведенном выше примере. Вместо этого переместите эту проблему кэширования на другой уровень вашего приложения.

    Например, в Java вы можете использовать аспекты программирования с использованием аспектов. Например, jcabi-aspects имеет аннотацию @Cacheable, которая кэширует значение, возвращаемое методом.

    import com.jcabi.aspects.Cacheable;
public class Department {
  @Cacheable(forever = true)
  public Employee manager() {
    return new Employee("Jacky Brown");
  }
}

    Я надеюсь, что этот анализ был достаточно убедительным, чтобы вы прекратили использовать NULL в своем коде.

    Synchronization Between Nodes

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Когда два или более программных модуля обращаются к одному и тому же ресурсу, они должны быть синхронизированы. Это означает, что только один модуль одновременно может работать с ресурсом. Без такой синхронизации возникают столкновения и конфликты. Это особенно актуально, когда речь идет о “ресурсах”, которые не поддерживают атомарные транзакции.

    Для решения этой проблемы и предотвращения конфликтов, нам необходимо внести еще один элемент в ситуацию. Все программные модули перед доступом к ресурсу должны захватить блокировку с централизованного сервера. После завершения манипуляций с ресурсом, модуль должен освободить блокировку. Пока блокировка захвачена одним модулем, другие модули не смогут ее захватить. Подход очень прост и хорошо известен. Однако, я не нашел ни одной облачной службы, которая предоставляла бы такую услугу блокировки и разблокировки через RESTful API. Поэтому я решил создать одну — stateful.co.

    Вот практический пример. У меня есть веб-приложение на Java, которое размещено на Heroku. Там работают три сервера (также называемые “диносами”), на которых запущено одно и то же приложение с расширением .war. Почему именно три? Потому что веб-трафик достаточно активный, и одного сервера недостаточно мощностей. Поэтому мне нужно иметь их три. Все они запускают одинаковые приложения.

    Каждое веб-приложение работает с таблицей в Amazon DynamoDB. Оно обновляет таблицу, добавляет в нее новые элементы, иногда удаляет некоторые элементы и выбирает их. Пока все идет хорошо, но конфликты неизбежны. Вот пример типичного сценария взаимодействия между веб-приложением и DynamoDB (я использую jcabi-dynamo).

    Table table = region.table("posts");
Item item = table.frame()
  .where("name", "Jeff")
  .iterator().next();
String salary = item.get("salary");
item.put("salary", this.recalculate(salary));

    Логика здесь очевидна. Сначала я получаю элемент из таблицы posts, затем считываю его salary, и затем изменяю его в соответствии с моим алгоритмом перерасчета. Проблема в том, что другой модуль может начать делать то же самое, пока я выполняю перерасчет. Он будет считывать то же самое начальное значение из таблицы и начнет точно такой же перерасчет. Затем он сохранит новое значение, и я тоже сохраню. В результате зарплата Джеффа будет изменена только один раз, в то время как пользователи ожидают двойного изменения, так как двое из них инициировали две транзакции с двумя разными веб-приложениями.

    Правильным подходом здесь является сначала «заблокировать» таблицу DynamoDB, даже до чтения заработной платы. Затем выполнять модификации и в конечном итоге разблокировать ее. Вот как stateful.co помогает мне. Все, что мне нужно сделать, это создать новую именованную блокировку в веб-панели stateful.co, получить свои ключи аутентификации и изменить мой Java-код.

    Sttc sttc = new RtSttc(
  new URN("urn:github:526301"), // my GitHub ID
  "9FF3-4320-73FB-EEAC" // my secret key!
);
Locks locks = sttc.locks();
Lock lock = locks.get("posts-table-lock");
Table table = region.table("posts");
Item item = table.frame()
  .where("name", "Jeff")
  .iterator().next();
new Atomic(lock).call(
  new Callable<Void>() {
    @Override
    public void call() {
      String salary = item.get("salary");
      item.put("salary", this.recalculate(salary));
      return null;
    }
  }
);

    Как видите, я оборачиваю эту критическую транзакцию в Callable, которая будет выполняться в изоляции. Этот подход, очевидно, не гарантирует атомарности транзакции - если часть транзакции завершится неудачно, не будет автоматического отката и таблица DynamoDB останется в “неправильном” состоянии.

    Замки от stateful.co гарантируют изоляцию при использовании ресурсов, и вы можете использовать любой тип ресурсов, включая таблицы NoSQL, файлы, объекты S3, встроенное программное обеспечение и т.д.

    Я не должен забыть добавить эту зависимость в мой pom.xml.

    <dependency>
  <groupId>co.stateful</groupId>
  <artifactId>java-sdk</artifactId>
</dependency>

    Конечно, вы можете сделать то же самое; сервис абсолютно бесплатен. И вы можете использовать любые другие языки, а не только Java. Кстати, если заинтересованы, вы можете внести свой вклад с помощью собственного SDK на предпочитаемом вами языке; я добавлю его в коллекцию на GitHub.

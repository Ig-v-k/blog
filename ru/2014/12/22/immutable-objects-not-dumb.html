    Immutable Objects Are Not Dumb

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    После нескольких недавних публикаций о неизменяемости, включая “Объекты должны быть неизменяемыми” и “Как неизменяемый объект может иметь состояние и поведение?”, меня удивило количество комментариев, где говорилось, что я плохо понимаю идею. Большинство из этих комментариев утверждали, что неизменяемый объект всегда должен вести себя одинаково - в этом суть неизменяемости. Какая это неизменяемость, если метод возвращает разные результаты при каждом вызове? Так не ведут себя хорошо известные неизменяемые классы. Возьмем, например, String, BigInteger, Locale, URI, URL, Inet4Address, UUID или оберточные классы для примитивов, такие как Double и Integer. Другие комментарии возражали даже против самого определения неизменяемого объекта как представителя изменяемой реальной сущности. Как неизменяемый объект может представлять изменяемую сущность? Что за?

    Я очень удивлен. В этом посте будет разъяснено определение неизменяемого объекта. Сначала, вот быстрый ответ. Как неизменяемый объект может представлять изменяемую сущность? Рассмотрим неизменяемый класс File и его методы, например length() и delete(). Класс является неизменяемым, согласно документации Oracle, и его методы могут возвращать разные значения при каждом их вызове. Объект класса File, будучи абсолютно неизменяемым, представляет изменяемую сущность реального мира, файл на диске.

    В этом посте я сказал, что “объект является неизменяемым, если его состояние не может быть изменено после его создания.” Это определение не мое; оно взято из Java Concurrency in Practice by Goetz et al., раздел 3.4 (кстати, я настоятельно рекомендую его прочитать). Теперь посмотрите на этот класс (я использую jcabi-http для чтения и записи через HTTP):

    @Immutable
class Page {
  private final URI uri;
  Page(URI addr) {
    this.uri = addr;
  }
  public String load() {
    return new JdkRequest(this.uri)
      .fetch().body();
  }
  public void save(String content) {
    new JdkRequest(this.uri)
      .method("PUT")
      .body().set(content).back()
      .fetch();
  }
}

    Что такое “состояние” в этом классе? Правильно, this.uri является состоянием. Он уникально идентифицирует каждый объект этого класса и неизменяем. Таким образом, класс создает только неизменяемые объекты. И каждый объект представляет изменяемую сущность реального мира - веб-страницу с URI.

    В этой ситуации нет противоречия. Класс является полностью неизменяемым, в то время как веб-страница, которую он представляет, может изменяться.

    Почему большинство программистов, с которыми я общался, считают, что если основная сущность изменяема, то и объект тоже изменяемый? Я думаю, ответ простой - они считают, что объекты являются структурами данных с методами. Именно поэтому, с этой точки зрения, неизменяемый объект - это структура данных, которая никогда не меняется.

    Вот откуда происходит ошибка мышления - объект не является структурой данных. Он является живым организмом, представляющим сущность реального мира внутри живой среды объекта (компьютерной программы). Он инкапсулирует некоторые данные, которые помогают найти сущность в реальном мире. Инкапсулированные данные представляют собой координаты представляемой сущности. В случае String или URL координаты совпадают с самой сущностью, но это всего лишь случайное явление, а не общее правило.

    Неизменяемый объект не является структурой данных, которая не изменяется, несмотря на то, что String, BigInteger и URL выглядят так. Объект является неизменяемым только в том случае, если он не изменяет координаты реального объекта, который он представляет. В классе Page это означает, что объект класса, однажды созданный, никогда не изменит this.uri. Он всегда будет указывать на одну и ту же веб-страницу, независимо от всего.

    И объект не гарантирует ничего относительно поведения этой веб-страницы. Страница - динамическое существо реального мира, живущее своей собственной жизнью. Наш объект не может обещать ничего относительно страницы. Единственное, что он обещает, это то, что он всегда останется верным этой странице - он никогда не забудет или не изменит своих координат.

    Концептуально говоря, неизменяемость означает преданность, вот и всё.

    Singletons Must Die

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Я думаю, что говорить о том, что синглтон - это анти-паттерн, слишком очевидно, так как есть тонны статей об этом (синглтон как анти-паттерн). Однако чаще всего вопрос заключается в том, как определить глобальные вещи без синглтона; и ответ на это неочевиден для многих из нас. Есть несколько примеров: пул подключений к базе данных, репозиторий, карта конфигурации и т. д. Все они естественным образом кажутся “глобальными”; но что мы делаем с ними?

    Предполагаю, что вы уже знаете, что такое синглтон и почему это антипаттерн. Если нет, рекомендую вам прочитать эту тему на Stack Overflow: Что так плохо в синглтонах?

    Теперь, когда мы согласились, что это плохая сделка, что делать, если нам нужно, скажем, иметь доступ к пулу подключений к базе данных в разных местах в приложении? Просто нам нужно что-то вроде этого:

    class Database {
  public static Database INSTANCE = new Database();
  private Database() {
    // Create a connection pool
  }
  public java.sql.Connection connect() {
    // Get a new connection from the pool
    // and return it
  }
}

    Позднее, скажем, в методе JAX-RS REST, нам нужно извлечь что-то из базы данных.

    @Path("/")
class Index {
  @GET
  public String text() {
    java.sql.Connection connection =
      Database.INSTANCE.connect();
    return new JdbcSession(connection)
      .sql("SELECT text FROM table")
      .fetch(new SingleOutcome(String.class))
  }
}

    В случае, если вы не знакомы с JAX-RS, это простая архитектура MVC, и этот метод text() является “контроллером”. Кроме того, я использую JdbcSession, простую обертку для JDBC из jcabi-jdbc.

    Мы хотим, чтобы Database.INSTANCE был синглтоном, верно? Нам нужно, чтобы он был доступен глобально, чтобы любой MVC-контроллер мог иметь к нему прямой доступ. Поскольку мы все понимаем и согласны, что синглтон - это зло, на что мы его заменяем?

    Внедрение зависимостей - это ответ.

    Нам нужно сделать этот пул соединений с базой данных зависимостью контроллера и обеспечить его предоставление через конструктор. Однако в этом конкретном случае, для JAX-RS, мы не можем сделать это через конструктор из-за его некрасивой архитектуры. Но мы можем создать ServletContextListener, создать экземпляр Database в его методе contextInitialized() и добавить этот экземпляр в атрибут servletContext. Затем внутри контроллера мы получаем контекст сервлета, добавляя аннотацию javax.ws.rs.core.Context к сеттеру и используя getAttribute() на нем. Это абсолютно ужасно и процедурно, но лучше, чем синглтон.

    Правильное объектно-ориентированное проектирование будет передавать экземпляр Database всем объектам, которые могут нуждаться в нем, через их конструкторы.

    Тем не менее, что делать, если существует много зависимостей? Создать конструктор с 10 аргументами? Нет, не нужно. Если нашим объектам действительно требуется 10 зависимостей для выполнения работы, мы должны разделить их на более мелкие.

    Вот и все. Забудьте о синглтонах; никогда не используйте их. Превратите их в зависимости и передавайте их от объекта к объекту через оператор new.

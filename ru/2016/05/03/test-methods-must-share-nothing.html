    Test Methods Must Share Nothing

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Константы… Я уже писал об этом некоторое время назад, в основном говоря, что они плохая вещь, если они являются публичными. Они уменьшают дублирование, но вводят связывание. Намного лучше способ избавиться от дублирования - это создание новых классов или методов - традиционный метод ООП. Это кажется логичным, и в наших проектах я вижу все меньше и меньше публичных констант. В некоторых проектах их вообще нет. Но меня все еще беспокоит одно: модульные тесты. Большинство программистов кажется считают, что когда статический анализ говорит, что в одном файле слишком много похожих литералов, лучший способ избавиться от них - это использовать приватную статическую литералу. Это просто неправильно.

    Unit-тесты, естественно, дублируют много кода. Методы тестирования содержат похожий или практически идентичный функционал, и это почти неизбежно. Хорошо, мы можем использовать больше @Before и @BeforeClass функций, но иногда это просто невозможно. У нас может быть, скажем, 20 методов тестирования в одном файле FooTest.java. Подготовка всех объектов в одном “before” невозможна. Поэтому нам приходится снова и снова выполнять определенные действия в наших методах тестирования.

    Давайте взглянем на один из классов в нашем Takes Framework: VerboseListTest. Это модульный тест и у него есть проблема, о которой я пытаюсь вам рассказать. Обратите внимание на этот приватный литерал MSG. Он используется впервые в методе setUp() в качестве аргумента конструктора объекта, а затем в нескольких методах тестирования, чтобы проверить, как этот объект ведет себя. Позвольте упростить этот код:

    class FooTest {
  private static final String MSG = "something";
  @Before
  public final void setUp() throws Exception {
    this.foo = new Foo(FooTest.MSG);
  }
  @Test
  public void simplyWorks() throws IOException {
    assertThat(
      foo.doSomething(),
      containsString(FooTest.MSG)
    );
  }
  @Test
  public void simplyWorksAgain() throws IOException {
    assertThat(
      foo.doSomethingElse(),
      containsString(FooTest.MSG)
    );
  }
}

    Вот в чем суть происходящего в VerboseListTest, и это очень неправильно. Почему? Потому что это общая литеральная строка MSG создает неестественную связь между этими двумя тестовыми методами. Они ничего не имеют общего, потому что они тестируют разные поведения класса Foo. Но эта приватная константа объединяет их. Теперь они как-то связаны.

    Если и когда я хочу изменить один из методов тестирования, мне может потребоваться изменить и другой. Скажем, я хочу узнать, как ведет себя doSomethingElse(), если инкапсулированное сообщение является пустой строкой. Что мне делать? Я изменяю значение константы FooTest.MSG, которая используется другим методом тестирования. Это называется связыванием. И это плохо.

    Что мы делаем? Что ж, мы можем использовать эту строковую литералу "something" в обоих тестовых методах.

    class FooTest {
  @Test
  public void simplyWorks() throws IOException {
    assertThat(
      new Foo("something").doSomething(),
      containsString("something")
    );
  }
  @Test
  public void simplyWorksAgain() throws IOException {
    assertThat(
      new Foo("something").doSomethingElse(),
      containsString("something")
    );
  }
}

    Как видите, я избавился от метода setUp() и от приватной статической константы MSG. Что у нас есть теперь? Дублирование кода. Строка "something" появляется четыре раза в классе теста. Никакие статические анализаторы этого не потерпят. Более того, в классе VerboseListTest есть семь (!) тестовых методов, которые используют MSG. Таким образом, у нас будет 14 вхождений "something", верно? Да, вы правы, и скорее всего поэтому один из авторов этого тестового случая ввел константу, чтобы избавиться от дублирования. Кстати, @Happy-Neko сделал это в pull request #513, код был просмотрен @carlosmiranda, и я одобрил изменения. Таким образом, ошибку допустили три человека, включая меня самого.

    Итак, какой подход правильный, чтобы избежать дублирования кода и в то же время не вводить связность? Вот он:

    class FooTest {
  @Test
  public void simplyWorks() throws IOException {
    final String msg = "something";
    assertThat(
      new Foo(msg).doSomething(),
      containsString(msg)
    );
  }
  @Test
  public void simplyWorksAgain() throws IOException {
    final String msg = "something else";
    assertThat(
      new Foo(msg).doSomethingElse(),
      containsString(msg)
    );
  }
}

    Эти литералы должны быть разными. Это то, что говорит любой статический анализатор, когда он видит "что-то" во многих местах. Он задает нам вопрос: почему они одинаковые? Действительно ли так важно использовать "что-то" повсюду? Почему бы не использовать разные литералы? Конечно, мы можем. И мы должны.

    В конечном итоге каждому методу тестирования должен быть присвоен собственный набор данных и объектов. Они никогда не должны разделяться между методами тестирования. Методы тестирования всегда должны быть независимыми, не имеющими ничего общего.

    Имея это в виду, мы легко можем сделать вывод, что методы, такие как setUp() или любые общие переменные в классах тестов, являются злом. Их не следует использовать и они просто не должны существовать. Я думаю, что их изобретение в JUnit причинило много вреда Java-коду.

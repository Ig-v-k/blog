    If-Then-Else Is a Code Smell

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    В большинстве случаев (возможно, даже во всех), оператор if-then-else может и должен быть заменен декоратором или просто другим объектом. Я уже почти год планировал написать об этом, но только сегодня нашел реальный случай в своем коде, который идеально иллюстрирует проблему. Поэтому пришло время продемонстрировать и объяснить.

    Взгляните на класс DyTalk из yegor256/rultor и его метод modify(). Вкратце, он предотвращает сохранение любых данных в DynamoDB, если не было изменений в XML-документе. Это допустимый случай и его необходимо проверить, но способ его реализации просто неправильный. Вот как это работает (упрощенный пример):

    class DyTalk implements Talk {
  void modify(Collection<Directive> dirs) {
    if (!dirs.isEmpty()) {
      // Apply the modification
      // and save the new XML document
      // to the DynamoDB table.
    }
  }
}

    Что не так, вы спрашиваете? Эта функциональность разветвления if-then-else действительно не принадлежит этому объекту — вот что не так. Изменение XML-документа и сохранение его в базу данных является его функциональностью, в то время как не сохранение ничего, если набор инструкций для модификации пуст, не является (это очень похоже на защитное программирование). Вместо этого должен быть декоратор, который выглядел бы так:

    class QuickTalk implements Talk {
  private final Talk origin;
  void modify(Collection<Directive> dirs) {
    if (!dirs.isEmpty()) {
      this.origin.modify(dirs);
    }
  }
}

    Теперь, если и когда нам нужно, чтобы наш разговор был более умным в ситуациях, когда список директив пуст, мы украшаем его с помощью QuickTalk. Преимущества очевидны: класс DyTalk является более компактным и, следовательно, более связанным.

    Но вопрос гораздо шире, чем просто это. Можем ли мы сделать из этого правило? Можем ли мы сказать, что каждое и каждое разделение плохо и должно быть исключено из класса? А что насчет разделения, которое происходит внутри метода и не может быть преобразовано в декоратор?

    Я предлагаю следующее простое правило: Если возможно преобразовать ветвление if-then-else в декоратор, это должно быть сделано. Если это не сделано, то это недостаток в коде. Понятно?

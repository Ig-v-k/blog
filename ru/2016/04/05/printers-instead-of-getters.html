    Printers Instead of Getters

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Геттеры и сеттеры плохи. Необходимо обсуждать это, это решено. Вы не согласны? Обсудим позже. Пока давайте скажем, что мы хотим избавиться от геттеров. Основной вопрос - как это вообще возможно? Ведь нам нужно получить данные из объекта, верно? Неверно.

    Я предлагаю использовать “принтеры” вместо этого. Вместо доступа к данным через геттеры, объект будет иметь функциональность печати самого себя в какое-либо средство.

    Допустим, это наш класс:

    public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
}

    Нам нужно преобразовать его в формат XML. Более или менее традиционный способ сделать это - с помощью геттеров и JAXB:

    import javax.xml.bind.annotation.XmlElement;
import javax.xml.bind.annotation.XmlRootElement;
@XmlRootElement
public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
  @XmlElement
  public String getIsbn() {
    return this.isbn;
  }
  @XmlElement
  public String getTitle() {
    return this.title;
  }
}

    Это очень оскорбительный способ обращения с объектом. Мы фактически выставляем все, что находится внутри, на показ. Это был милый небольшой самодостаточный объект, и мы превратили его в мешок с данными, к которым кто угодно может получить доступ разными способами. Мы можем получить доступ к ним для чтения, конечно.

    Удобно иметь эти геттеры, можно сказать. Мы все привыкли к ним. Если мы хотим преобразовать его в JSON, они будут очень полезны. Если мы хотим использовать этот бедный объект как объект данных в JSP, геттеры помогут нам. В Java много примеров, где активно используются геттеры.

    Это не потому, что они так эффективны. Это потому, что мы так процедурно мыслим. Мы не доверяем нашим объектам. Мы доверяем только хранимым ими данным. Мы не хотим, чтобы объект Book генерировал XML. Мы хотим, чтобы он дал нам данные. Мы построим XML сами. Book слишком глуп, чтобы справиться с этой работой. Мы намного умнее!

    Я предлагаю перестать думать таким образом. Вместо этого давайте попробуем дать этому бедному Book шанс и снабдить его “принтером”:

    public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
  public String toXML() {
    return String.format(
      "<book><isbn>%s</isbn><title>%s</title></book>",
      this.isbn, this.title
    );
  }
}

    Это не самая лучшая реализация, но вы поняли идею. Объект больше не раскрывает свои внутренние данные. Мы не можем получить его ISBN и заголовок. Мы можем только попросить его распечатать себя в формате XML.

    Мы можем добавить дополнительный принтер, если требуется другой формат:

    public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
  public String toJSON() {
    return String.format(
      "{\"isbn\":\"%s\", \"title\":\"%s\"}",
      this.isbn, this.title
    );
  }
}

    Снова, не самая лучшая реализация, но вы видите, что я пытаюсь показать. Каждый раз, когда нам нужен новый формат, мы создаем новый принтер.

    Вы можете сказать, что объект будет довольно большим, если будет много форматов. Это верно, но большой объект - плохой дизайн с самого начала. Я бы сказал, что если у нас есть более одного принтера - это проблема.

    Итак, что делать, если нам нужны несколько форматов? Используйте “медиа”, куда эти принтеры смогут печатать. Допустим, у нас есть объект, который представляет запись в MySQL. Мы хотим, чтобы он мог быть распечатан в XML, HTML, JSON, некоторый двоичный формат и Бог знает что еще. Мы можем добавить столько принтеров, сколько нам нужно, но объект будет большим и некрасивым. Чтобы избежать этого, введем новый объект, который представляет медиа, куда данные будут печататься:

    public class Book {
  private final String isbn =
    "0735619654";
  private final String title =
    "Object Thinking";
  public Media print(Media media) {
    return media
      .with("isbn", this.isbn)
      .with("title", this.title);
  }
}

    Опять же, это очень примитивный дизайн неизменного класса Media, но вы поняли идею - media принимает данные. Теперь мы хотим вывести наш объект в формате JSON (этот дизайн не совсем идеален, так как JsonObjectBuilder не является неизменным, хотя и выглядит таким…).

    class JsonMedia implements Media {
  private final JsonObjectBuilder builder;
  JsonMedia() {
    this("book");
  }
  JsonMedia(String head) {
    this(Json.createObjectBuilder().add(head));
  }
  JsonMedia(JsonObjectBuilder bdr) {
    this.builder = bdr;
  }
  @Override
  public Media with(String name, String value) {
    return new JsonMedia(
      this.builder.add(name, value)
    );
  }
  public JsonObject json() {
    return this.builder.build();
  }
}

    Теперь мы создаем экземпляр JsonMedia и просим нашу книгу напечатать себя там:

    JsonMedia media = new JsonMedia("book");
media = book.print(media);
JsonObject json = media.json();

    Вот! Объект JSON готов, и книга не знает, что именно было только что напечатано. Нам нужно напечатать книгу в XML? Мы создаем XmlMedia, который будет печатать книгу в XML. Класс Book остается небольшим, в то время как сложность объектов “медиа” неограничена.

    Моя идея здесь проста - нет геттеров, только принтеры!

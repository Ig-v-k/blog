    Java Annotations Are a Big Mistake

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Аннотации были введены в Java 5, и мы все были взволнованы. Такой замечательный инструмент для сокращения кода! Больше нет файлов конфигурации Hibernate/Spring XML! Просто аннотации, прямо там, где нам нужны. Больше нет маркерных интерфейсов, только сохраняемые во время выполнения отражение-обнаруживаемые аннотации! Я тоже был возбужден. Более того, я создал несколько библиотек с открытым исходным кодом, которые активно используют аннотации. Возьмите, например, jcabi-aspects. Однако, я больше не в восторге. Более того, я считаю, что аннотации - это большая ошибка в Java-дизайне.

    Долгая история коротка, есть одна большая проблема с аннотациями - они побуждают нас реализовывать функциональность объекта вне самого объекта, что противоречит самому принципу инкапсуляции. Объект уже не является цельным, так как его поведение не определяется полностью собственными методами - некоторая функциональность остается в другом месте. Почему это плохо? Давайте рассмотрим несколько примеров.

    @Inject

    Предположим, что мы аннотируем свойство с помощью @Inject:

    import javax.inject.Inject;
public class Books {
  @Inject
  private final DB db;
  // some methods here, which use this.db
}

    Затем у нас есть инжектор, который знает, что нужно внедрить:

    Injector injector = Guice.createInjector(
  new AbstractModule() {
    @Override
    public void configure() {
      this.bind(DB.class).toInstance(
        new Postgres("jdbc:postgresql:5740/main")
      );
    }
  }
);

    Теперь мы создаем экземпляр класса Books через контейнер:

    Books books = injector.getInstance(Books.class);

    Класс Books не знает, как и кто будет внедрять экземпляр класса DB в него. Это произойдет за кулисами и вне его контроля. Внедрение будет осуществляться. Это может показаться удобным, но такое отношение причиняет много вреда всей кодовой базе. Управление потеряно (не инвертировано, а потеряно!). Объект больше не отвечает за то, что с ним происходит.

    Вместо этого, вот как это должно быть сделано:

    class Books {
  private final DB db;
  Books(final DB base) {
    this.db = base;
  }
  // some methods here, which use this.db
}

    В этой статье объясняется, почему контейнеры внедрения зависимостей - плохая идея: Контейнеры внедрения зависимостей загрязняют код. Аннотации в основном заставляют нас создавать контейнеры и использовать их. Мы перемещаем функциональность из наших объектов и помещаем ее в контейнеры или другие места. Ведь мы не хотим дублировать один и тот же код снова и снова, верно? Да, дублирование плохо, но разрыв объекта еще хуже. Гораздо хуже. То же самое верно для ORM (JPA/Hibernate), где активно используются аннотации. Проверьте этот пост, он объясняет, что не так с ORM: ORM - оскорбительный анти-паттерн. Сами по себе аннотации не являются ключевым фактором мотивации, но они помогают и подталкивают нас разрывать объекты и хранить их части в разных местах. Они являются контейнерами, сессиями, менеджерами, контроллерами и т. д.

    @XmlElement

    Вот как работает JAXB, когда вы хотите преобразовать свой POJO в XML. Сначала вы прикрепляете аннотацию @XmlElement к геттеру:

    import javax.xml.bind.annotation.XmlElement;
import javax.xml.bind.annotation.XmlRootElement;
@XmlRootElement
public class Book {
  private final String title;
  public Book(final String title) {
    this.title = title;
  }
  @XmlElement
  public String getTitle() {
    return this.title;
  }
}

    Затем вы создаете маршаллер и просите его преобразовать экземпляр класса Book в XML:

    final Book book = new Book("0132350882", "Clean Code");
final JAXBContext ctx = JAXBContext.newInstance(Book.class);
final Marshaller marshaller = ctx.createMarshaller();
marshaller.marshal(book, System.out);

    Кто создает XML? Не book. Кто-то другой, вне класса Book. Это очень неправильно. Вместо этого, вот как это должно было быть сделано. Сначала, класс, который не знает о XML:

    class DefaultBook implements Book {
  private final String title;
  DefaultBook(final String title) {
    this.title = title;
  }
  @Override
  public String getTitle() {
    return this.title;
  }
}

    Затем - декоратор, который выводит его в XML:

    class XmlBook implements Book{
  private final Book origin;
  XmlBook(final Book book) {
    this.origin = book;
  }
  @Override
  public String getTitle() {
    return this.origin.getTitle();
  }
  public String toXML() {
    return String.format(
      "<book><title>%s</title></book>",
      this.getTitle()
    );
  }
}

    Теперь, чтобы распечатать книгу в формате XML, мы делаем следующее:

    String xml = new XmlBook(
  new DefaultBook("Elegant Objects")
).toXML();

    Функциональность печати XML находится внутри XmlBook. Если вам не нравится идея декоратора, вы можете переместить метод toXML() в класс DefaultBook. Это не важно. Важно то, что функциональность всегда остается там, где она должна быть - внутри объекта. Только объект знает, как распечатать себя в XML. Никто другой!

    @RetryOnFailure

    Вот пример (из моей собственной библиотеки):

    import com.jcabi.aspects.RetryOnFailure;
class Foo {
  @RetryOnFailure
  public String load(URL url) {
    return url.openConnection().getContent();
  }
}

    После компиляции мы запускаем так называемый AOP-прядильщик, который технически преобразует наш код в нечто подобное этому:

    class Foo {
  public String load(URL url) {
    while (true) {
      try {
        return _Foo.load(url);
      } catch (Exception ex) {
        // ignore it
      }
    }
  }
  class _Foo {
    public String load(URL url) {
      return url.openConnection().getContent();
    }
  }
}

    Я упростил фактический алгоритм повторного вызова метода при ошибке, но я уверен, что вы понимаете идею. AspectJ, движок AOP, использует аннотацию @RetryOnFailure в качестве сигнала, информируя нас о том, что класс должен быть обернут в другой. Это происходит за кулисами. Мы не видим этот дополнительный класс, который реализует алгоритм повторных попыток. Но байт-код, созданный AspectJ-плетельщиком, содержит измененную версию класса Foo.

    Вот в чем именно проблема этого подхода - мы не видим и не контролируем создание этого дополнительного объекта. Композиция объектов, являющаяся самым важным процессом в проектировании объектов, скрыта где-то за кулисами. Можно сказать, что нам не нужно видеть ее, поскольку она является дополнительной. Я не согласен. Мы должны видеть, как наши объекты составляются. Нам может быть не важно, как они работают, но мы должны видеть весь процесс композиции.

    Гораздо лучше выглядел бы такой дизайн (вместо аннотаций):

    Foo foo = new FooThatRetries(new Foo());

    А затем реализация FooThatRetries:

    class FooThatRetries implements Foo {
  private final Foo origin;
  FooThatRetries(Foo foo) {
    this.origin = foo;
  }
  public String load(URL url) {
    return new Retry().eval(
      new Retry.Algorithm<String>() {
        @Override
        public String eval() {
          return FooThatRetries.this.load(url);
        }
      }
    );
  }
}

    И теперь реализация Retry:

    class Retry {
  public <T> T eval(Retry.Algorithm<T> algo) {
    while (true) {
      try {
        return algo.eval();
      } catch (Exception ex) {
        // ignore it
      }
    }
  }
  interface Algorithm<T> {
    T eval();
  }
}

    Код стал длиннее? Да. Стал ли он чище? Гораздо больше. Жалею, что два года назад я не понял этого, когда начал работать с jcabi-aspects.

    The bottom line is that annotations are bad. Don’t use them. What should be used instead? Object composition.

    Что может быть хуже аннотаций? Configurations. Например, XML-конфигурации. Механизмы XML-конфигурации в Spring являются идеальным примером ужасного дизайна. Я уже много раз говорил об этом. Позвольте мне повторить еще раз—Spring Framework является одним из самых плохих программных продуктов в мире Java. Если вы можете избежать его, вы сделаете себе большую услугу.

    В OOP не должно быть никаких “конфигураций”. Если наши объекты являются реальными объектами, то мы не можем их настраивать. Мы можем только создавать их экземпляры. И лучший метод создания экземпляров - оператор new. Этот оператор является ключевым инструментом для разработчика ООП. Забрать его у нас и предложить нам “механизмы конфигурации” - это непростительное преступление.

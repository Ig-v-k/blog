    Java Annotations Are a Big Mistake

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Аннотации были введены в Java 5, и мы все были в восторге. Такой великолепный инструмент для сокращения кода! Больше нет файлов конфигурации Hibernate/Spring XML! Просто аннотации прямо там, где нам нужны. Больше нет маркерных интерфейсов, только аннотация с сохранением времени выполнения, открываемая при использовании рефлексии! Я тоже был в восторге. Более того, я создал несколько библиотек с открытым исходным кодом, которые активно используют аннотации. Возьмите, к примеру, jcabi-aspects. Однако, я больше не в восторге. Более того, я считаю, что аннотации - большая ошибка в дизайне Java.

    Долгая история коротко говоря, существует одна большая проблема с аннотациями - они побуждают нас реализовывать функциональность объекта вне самого объекта, что противоречит самому принципу инкапсуляции. Объект уже не является цельным, поскольку его поведение не определяется полностью его собственными методами - некоторая функциональность остается где-то в другом месте. Почему это плохо? Давайте рассмотрим несколько примеров.

    @Inject

    Предположим, мы аннотируем свойство с помощью @Inject:

    import javax.inject.Inject;
public class Books {
  @Inject
  private final DB db;
  // some methods here, which use this.db
}

    Затем у нас есть инжектор, который знает, что нужно внедрить:

    Injector injector = Guice.createInjector(
  new AbstractModule() {
    @Override
    public void configure() {
      this.bind(DB.class).toInstance(
        new Postgres("jdbc:postgresql:5740/main")
      );
    }
  }
);

    Теперь мы создаём экземпляр класса Books через контейнер:

    Books books = injector.getInstance(Books.class);

    Класс Books не знает, как и кто будет внедрять экземпляр класса DB в него. Это произойдет за кулисами и вне его контроля. Инъекция сделает это. Это может показаться удобным, но такое отношение наносит большой ущерб всей кодовой базе. Утрачивается контроль (не инвертированный, а утраченный!). Объект больше не контролирует происходящее с ним. Он не может быть ответственным за то, что с ним происходит.

    Вместо этого вот как это должно быть сделано:

    class Books {
  private final DB db;
  Books(final DB base) {
    this.db = base;
  }
  // some methods here, which use this.db
}

    В этой статье объясняется, почему контейнеры внедрения зависимостей - неправильная идея. Контейнеры внедрения зависимостей загрязняют код. Аннотации, по сути, заставляют нас создавать контейнеры и использовать их. Мы перемещаем функциональность из наших объектов и помещаем ее в контейнеры или в другое место. Это происходит потому, что мы не хотим дублировать один и тот же код снова и снова, верно? Это верно, дублирование плохо, но разрыв объекта - еще хуже. Гораздо хуже. То же самое относится к ORM (JPA/Hibernate), где активно используются аннотации. Вот этот пост объясняет, в чем проблема с ORM: ORM - это оскорбительный анти-паттерн. Аннотации сами по себе не являются ключевым мотиватором, но они помогают и подталкивают нас разрывать объекты и хранить части в разных местах. Это контейнеры, сессии, менеджеры, контроллеры и т. д.

    @XmlElement

    Так работает JAXB, когда вы хотите преобразовать POJO в XML. Сначала вы прикрепляете аннотацию @XmlElement к геттеру:

    import javax.xml.bind.annotation.XmlElement;
import javax.xml.bind.annotation.XmlRootElement;
@XmlRootElement
public class Book {
  private final String title;
  public Book(final String title) {
    this.title = title;
  }
  @XmlElement
  public String getTitle() {
    return this.title;
  }
}

    Затем вы создаете маршалер и просите его преобразовать экземпляр класса Book в XML:

    final Book book = new Book("0132350882", "Clean Code");
final JAXBContext ctx = JAXBContext.newInstance(Book.class);
final Marshaller marshaller = ctx.createMarshaller();
marshaller.marshal(book, System.out);

    Кто создает XML? Не book. Кто-то другой, вне класса Book. Это очень неправильно. Вместо этого, вот как это должно было быть сделано. Сначала класс, который не имеет понятия об XML:

    class DefaultBook implements Book {
  private final String title;
  DefaultBook(final String title) {
    this.title = title;
  }
  @Override
  public String getTitle() {
    return this.title;
  }
}

    Затем декоратор, который выводит его в формате XML:

    class XmlBook implements Book{
  private final Book origin;
  XmlBook(final Book book) {
    this.origin = book;
  }
  @Override
  public String getTitle() {
    return this.origin.getTitle();
  }
  public String toXML() {
    return String.format(
      "<book><title>%s</title></book>",
      this.getTitle()
    );
  }
}

    Теперь, чтобы распечатать книгу в формате XML, мы выполняем следующие действия:

    String xml = new XmlBook(
  new DefaultBook("Elegant Objects")
).toXML();

    Функциональность печати XML находится внутри XmlBook. Если вам не нравится идея с декоратором, вы можете переместить метод toXML() в класс DefaultBook. Это не важно. Важно то, что функциональность всегда остается там, где она должна быть - внутри объекта. Только объект знает, как распечатать себя в XML. Никто другой!

    @RetryOnFailure

    Вот пример (из моей собственной библиотеки):

    import com.jcabi.aspects.RetryOnFailure;
class Foo {
  @RetryOnFailure
  public String load(URL url) {
    return url.openConnection().getContent();
  }
}

    После компиляции мы запускаем так называемый AOP-ткач, который технически превращает наш код в нечто подобное:

    class Foo {
  public String load(URL url) {
    while (true) {
      try {
        return _Foo.load(url);
      } catch (Exception ex) {
        // ignore it
      }
    }
  }
  class _Foo {
    public String load(URL url) {
      return url.openConnection().getContent();
    }
  }
}

    Я упростил фактический алгоритм повторного вызова метода при ошибке, но уверен, что вы понимаете идею. AspectJ, движок AOP, использует аннотацию @RetryOnFailure в качестве сигнала, информируя нас о том, что класс должен быть обернут в другой. Это происходит внутри. Мы не видим этот дополнительный класс, который реализует алгоритм повторного вызова. Но байт-код, созданный с помощью инструмента AspectJ, содержит измененную версию класса Foo.

    Вот что именно неправильно в таком подходе - мы не видим и не контролируем создание этого дополнительного объекта. Композиция объектов, являющаяся наиболее важным процессом в объектном проектировании, скрыта где-то за кулисами. Можно сказать, что нам не нужно видеть ее, так как это дополнительно. Я не согласен. Мы должны видеть, как наши объекты составляются. Нам может быть не важно, как они работают, но мы должны видеть весь процесс композиции.

    Гораздо лучше бы выглядела такая конструкция (вместо аннотаций):

    Foo foo = new FooThatRetries(new Foo());

    И затем, реализация FooThatRetries:

    class FooThatRetries implements Foo {
  private final Foo origin;
  FooThatRetries(Foo foo) {
    this.origin = foo;
  }
  public String load(URL url) {
    return new Retry().eval(
      new Retry.Algorithm<String>() {
        @Override
        public String eval() {
          return FooThatRetries.this.load(url);
        }
      }
    );
  }
}

    И теперь, реализация Retry:

    class Retry {
  public <T> T eval(Retry.Algorithm<T> algo) {
    while (true) {
      try {
        return algo.eval();
      } catch (Exception ex) {
        // ignore it
      }
    }
  }
  interface Algorithm<T> {
    T eval();
  }
}

    Код стал длиннее? Да. Стал ли он более чистым? Гораздо больше. Жалею, что два года назад я не понял это, когда начал работать с jcabi-aspects.

    The bottom line is that annotations are bad. Don’t use them. What should be used instead? Object composition.

    Что может быть хуже аннотаций? Конфигурации. Например, конфигурации в формате XML. Механизмы конфигурации Spring XML являются примером отвратительного дизайна. Я уже говорил об этом много раз. Позвольте мне повторить еще раз - Spring Framework является одним из самых плохих программных продуктов в мире Java. Если вы можете держаться подальше от него, сделаете себе большую услугу.

    В ООП не должно быть никаких “конфигураций”. Мы не можем настраивать наши объекты, если они являются реальными объектами. Мы можем только создать их экземпляры. И лучший метод создания экземпляров - это оператор new. Этот оператор является ключевым инструментом для разработчика ООП. Забрать его у нас и дать нам “механизмы конфигурации” - это непростительное преступление.

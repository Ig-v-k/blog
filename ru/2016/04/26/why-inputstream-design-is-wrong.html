    Why InputStream Design Is Wrong

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Речь не только о InputSteam, этот класс - хороший пример плохого дизайна. Я говорю о трех перегруженных методах read(). Я упомянул эту проблему в разделе 2.9 Elegant Objects. Вкратце, я настоятельно считаю, что интерфейсы должны быть “функционально бедными”. InputStream должен был быть интерфейсом с одним методом read(byte[]). Затем, если его авторы хотели дать нам дополнительную функциональность, они должны были создать дополнительные “умные” классы.

    Вот как это выглядит сейчас:

    abstract class InputStream {
  int read();
  int read(byte[] buffer, int offset, int length);
  int read(byte[] buffer);
}

    Что не так? Очень удобно иметь возможность чтения одного байта, массива байтов или даже массива байтов с прямым позиционированием в конкретное место в буфере!

    Однако у нас всё ещё не хватает нескольких методов: для чтения байтов и немедленного сохранения в файл, преобразования в текст с выбранной кодировкой, отправки по электронной почте и публикации в Twitter. Было бы здорово иметь такие возможности прямо в бедном InputStream. Надеюсь, команда Oracle Java уже работает над ними.

    Тем временем, давайте посмотрим, что именно не так с тем, что эти талантливые инженеры уже разработали для нас. Или, может быть, позвольте мне показать, как я бы разработал InputStream, и мы сравнимся:

    interface InputStream {
  int read(byte[] buffer, int offset, int length);
}

    Это мой дизайн. InputStream отвечает за чтение байтов из потока. Для этой функции существует только один метод. Это удобно для всех? Читает ли он и публикует на Twitter? Пока нет. Нужна ли нам эта функциональность? Конечно, да, но это не означает, что мы добавим ее в интерфейс. Вместо этого мы создадим дополнительный “умный” класс:

    interface InputStream {
  int read(byte[] buffer, int offset, int length);
  class Smart {
    private final InputStream origin;
    public Smart(InputStream stream) {
      this.origin = stream;
    }
    public int read() {
      final byte[] buffer = new byte[1];
      final int read = this.origin.read(buffer, 0, 1);
      final int result;
      if (read < 1) {
        result = -1;
      } else {
        result = buffer[0];
      }
      return result;
    }
  }
}

    Теперь мы хотим прочитать один байт из потока. Вот как:

    final InputStream input = new FileInputStream("/tmp/a.txt");
final byte b = new InputStream.Smart(input).read();

    Функциональность чтения одного байта находится за пределами InputStream, поскольку это не является его задачей. Потоку не нужно знать, как управлять данными после их чтения. Все, за что поток отвечает, - это чтение, а не разбор или дальнейшая обработка.

    Интерфейсы должны быть компактными.

    Очевидно, перегрузка методов в интерфейсах является признаком плохого кода. Интерфейс с более чем тремя методами является отличным кандидатом для рефакторинга. Если методы перегружают друг друга, это серьезная проблема.

    Интерфейсы должны быть маленькими!

    Вы можете сказать, что разработчики InputStream заботились о производительности, поэтому позволили нам реализовать метод read() в трех различных формах. Тогда я должен спросить снова, почему бы не создать метод для чтения и сразу же опубликовать его в Twitter? Это было бы невероятно быстро. Ведь именно это мы все хотим? Быстрое программное обеспечение, которое никому не хочется читать или поддерживать.

    interface Storage {
    int readInt();
    String readString();
    float readFloat();
    }

    Data Transfer Object Is a Shame

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    DTO, насколько я понимаю, является основой паттерна проектирования ORM, который я просто “обожаю”. Но давайте перейдем к сути: DTO - это просто позор, и человек, который его придумал, просто ошибается. Нет оправдания за то, что он сделал.

    Кстати, насколько мне известно, его имя было Мартин Фаулер. Возможно, он не был единственным изобретателем DTO, но он сделал его легальным и рекомендовал его использование. Со всем уважением, он просто ошибался.

    Основная идея объектно-ориентированного программирования (ООП) заключается в скрытии данных за объектами. У этой идеи есть название: инкапсуляция. В ООП данные не должны быть видимыми. Объекты должны иметь доступ только к данным, которые они инкапсулируют, и никогда не к данным, инкапсулированным другими объектами. Нет никаких споров по поводу этого принципа - это то, что делает ООП особенным.

    Однако, DTO полностью противоречит этому принципу.

    Давайте рассмотрим практический пример. Предположим, что это сервис, который получает JSON-документ из некоторого RESTful API и возвращает DTO, который мы затем можем сохранить в базе данных.

    Book book = api.loadBookById(123);
database.saveNewBook(book);

    Я предполагаю, что произойдет следующее внутри метода loadBookById():

    Book loadBookById(int id) {
  JsonObject json = /* Load it from RESTful API */
  Book book = new Book();
  book.setISBN(json.getString("isbn"));
  book.setTitle(json.getString("title"));
  book.setAuthor(json.getString("author"));
  return book;
}

    Я прав? Я уверен, что да. Это уже выглядит отвратительно для меня. В любом случае, продолжим. Вот что, скорее всего, произойдет в методе saveNewBook() (я использую чистый JDBC):

    void saveNewBook(Book book) {
  Statement stmt = connection.prepareStatement(
    "INSERT INTO book VALUES (?, ?, ?)"
  );
  stmt.setString(1, book.getISBN());
  stmt.setString(2, book.getTitle());
  stmt.setString(3, book.getAuthor());
  stmt.execute();
}

    Эта Book является классическим примером шаблона проектирования передачи данных (data transfer object). Все, что она делает, это передает данные между двумя частями кода, двумя процедурами. Объект book довольно глупый. Все, что он умеет делать, это… ничего. Он ничего не делает. Фактически, он вовсе не является объектом, а скорее пассивной и анемичной структурой данных.

    Какой дизайн является правильным? Их несколько. Например, мне нравится этот.

    Book book = api.bookById(123);
book.save(database);

    Это то, что происходит в bookById():

    Book bookById(int id) {
  return new JsonBook(
    /* RESTful API access point */
  );
}

    Вот что происходит в Book.save():

    void save(Database db) {
  JsonObject json = /* Load it from RESTful API */
  db.createBook(
    json.getString("isbn"),
    json.getString("title"),
    json.getString("author")
  );
}

    Что произойдет, если в JSON будет множество других параметров книги, которые не подойдут хорошо в качестве параметров для одного метода createBook()? А что насчет этого:

    void save(Database db) {
  db.create()
    .withISBN(json.getString("isbn"))
    .withTitle(json.getString("title"))
    .withAuthor(json.getString("author"))
    .deploy();
}

    Есть много других вариантов. Но главное, что данные никогда не выходят за пределы объекта book. Как только объект создан, данные не видны и не доступны никому еще. Мы можем только попросить наш объект сохранить сам себя или распечатать себя на некоторые носители, но мы никогда не получим какие-либо данные от него.

    Сама идея DTO неправильна, потому что она превращает объектно-ориентированный код в процедурный код. У нас есть процедуры, которые обрабатывают данные, а DTO - это всего лишь контейнер для этих данных. Не думайте так и не делайте это.

    PS. Существуют и другие названия DTO: business objects, domain objects (не в DDD), entity objects, JavaBeans.

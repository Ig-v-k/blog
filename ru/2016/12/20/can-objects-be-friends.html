    Can Objects Be Friends?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Как уже обсуждалось ранее, правильная инкапсуляция приводит к полному отсутствию “голых данных”. Однако, остаётся вопрос: как объекты могут взаимодействовать, если они не могут обмениваться данными? В конечном итоге, нам придётся раскрыть некоторые данные, чтобы позволить другим объектам использовать их, верно? Да, это верно. Однако у меня, кажется, есть решение, которое сохраняет инкапсуляцию, позволяя объектам взаимодействовать.

    Предположим, что это наш объект:

    class Temperature {
  private int t;
  public String toString() {
    return String.format("%d C", this.t);
  }
}

    Это представляет собой температуру. Единственное поведение, которое оно предоставляет, - это печать температуры в градусах Цельсия. Мы не хотим раскрывать t, потому что это приведет к проблеме “голых данных”. Мы хотим сохранить t в тайне, и это вполне логичное желание.

    Теперь мы хотим иметь возможность выводить температуру в Фаренгейтах. Самый очевидный подход - это добавить еще один метод toFahrenheitString(), или добавить логический флаг в объект, который изменит поведение метода toString(), верно? Любое из этих решений лучше, чем добавление метода getT(), но ни одно из них не идеально.

    Что, если мы создадим этот декоратор:

    class TempFahrenheit implements Temperature {
  private TempCelsius origin;
  public String toString() {
    return String.format(
      "%d F", this.origin.t * 1.8 + 32
    );
  }
}

    Это должно работать просто великолепно:

    Temperature t = new TempFahrenheit(
  new TempCelsius(35)
);

    Единственная проблема заключается в том, что она не компилируется в Java, потому что класс TempFahrenheit не имеет доступа к закрытому полю t класса TempCelsius. Если мы сделаем поле t публичным, то каждый сможет читать его напрямую, и у нас возникнет проблема “обнаженных данных” - серьезное нарушение инкапсуляции.

    Однако, если мы позволим доступ только к одному классу, все будет в порядке. Что-то вроде этого (не будет работать в Java; это всего лишь концепция):

    class TempCelsius {
  trust TempFahrenheit; // here!
  private int t;
  public String toString() {
    return String.format("%d C", this.t);
  }
}

    Поскольку ключевое слово trust размещается в классе, который обеспечивает доступ, у нас не будет проблемы с “голыми данными” - мы всегда будем точно знать, какие объекты обладают знаниями о t. Когда мы что-то меняем в t, мы точно знаем, где нужно обновить код.

    What do you think?

    P.S. После обсуждения этой идеи ниже в комментариях я начал думать, что нам совсем не нужно это ключевое слово trust. Вместо этого мы просто должны дать всем декораторам доступ ко всем приватным атрибутам объекта.

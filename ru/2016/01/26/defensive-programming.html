    Defensive Programming via Validating Decorators

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Вы проверяете входные параметры своих методов на допустимость? Я не проверяю. Раньше я делал это, но больше нет. Теперь я просто позволяю своим методам аварийно завершаться с нулевым указателем и другими исключениями, когда параметры недопустимы. Это может показаться нелогичным, но только в начале. Я предлагаю вам вместо этого использовать декораторы проверки.

    Давайте рассмотрим этот достаточно типичный пример на Java:

    class Report {
  void export(File file) {
    if (file == null) {
      throw new IllegalArgumentException(
        "File is NULL; can't export."
      );
    }
    if (file.exists()) {
      throw new IllegalArgumentException(
        "File already exists."
      );
    }
    // Export the report to the file
  }
}

    Довольно защищенно, не так ли? Если мы удалим эти проверки, код станет гораздо короче, но он будет выдавать довольно запутанные сообщения, если клиентом будет передано значение NULL. Более того, если файл уже существует, наш Report будет бесшумно перезаписывать его. Довольно опасно, не так ли?

    Да, мы должны защищать себя, и мы должны быть оборонительными.

    Но не таким образом, не путем нагромождения класса проверками, которые никак не связаны с его основной функциональностью. Вместо этого мы должны использовать декораторы для проверки. Вот как. Во-первых, должен быть интерфейс “Report”:

    interface Report {
  void export(File file);
}

    Затем класс, который реализует основные функции:

    class DefaultReport implements Report {
  @Override
  void export(File file) {
    // Export the report to the file
  }
}

    И, наконец, ряд декораторов, которые защитят нас:

    class NoWriteOverReport implements Report {
  private final Report origin;
  NoWriteOverReport(Report rep) {
    this.origin = rep;
  }
  @Override
  void export(File file) {
    if (file.exists()) {
      throw new IllegalArgumentException(
        "File already exists."
      );
    }
    this.origin.export(file);
  }
}

    Теперь клиенту предоставляется возможность создавать сложные объекты из декораторов, выполняющих свои специфические задачи. Основной объект будет выполнять отчетность, а декораторы будут проверять параметры.

    Report report = new NoNullReport(
  new NoWriteOverReport(
    new DefaultReport()
  )
);
report.export(file);

    Чего мы достигаем с помощью такого подхода? Прежде всего: меньших объектов. И меньшие объекты всегда означают более высокую поддерживаемость. Наш класс DefaultReport всегда будет оставаться небольшим, независимо от того, сколько проверок мы можем придумать в будущем. Чем больше вещей нам нужно проверить, тем больше декораторов для проверки мы создадим. Все они будут небольшими и связанными. И мы сможем объединять их в различных вариациях.

    Кроме того, такой подход делает наш код намного более повторно используемым, поскольку классы выполняют очень мало операций и по умолчанию не защищают себя. Хотя защита является важной функцией, мы будем использовать декораторы проверки. Но это не всегда будет так. Иногда проверка может быть слишком затратной с точки зрения времени и памяти, и мы можем захотеть работать непосредственно с объектами, которые не защищают себя.

    Я также решил больше не использовать Java Validation API по той же причине. Его аннотации делают классы более громоздкими и менее связанными. Вместо этого я использую декораторы для проверки.

    Strong Typing without Types

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    В 1974 году Лисков и Зиллес определили язык строго типизированным, как тот, в котором “всякий раз, когда объект передается из вызывающей функции в вызываемую функцию, его тип должен быть совместим с объявленным типом в вызываемой функции”. Строгая проверка типов, безусловно, снижает количество ошибок типов, что приводит к более высокому качеству. Однако вопрос состоит в следующем: действительно ли нам нужны типы для строгого обеспечения типизации?

    Например, это место, где мы ожидаем поступление экземпляра интерфейса Java Book:

    void print(Book b) {
  System.out.printf(
    "The ISBN is: %s%n", b.isbn()
  );
}

    Тип Book может выглядеть следующим образом:

    interface Book {
  String isbn();
}

    Если объект, который не реализует интерфейс “Book”, передается в метод “print()”, компилятор выдаст ошибку “несоответствие типов”. Программисту будет сложно совершить ошибку и передать объект типа, скажем, “Car”, в метод “print()”. Однако, это все равно будет возможно с помощью динамического приведения типов.

    Car car = new Car("Mercedes-Benz G63");
print(Book.class.cast(car)); // Here!

    Этот код будет компилироваться без проблем, но при выполнении будет сгенерировано исключение ClassCastException, поскольку невозможно будет привести тип Car к типу Book.

    Красота сильной типизации заключается в том, что она предотвращает ошибки. Однако это увеличивает сложность кода: сначала нужно создавать типы, затем объявлять их во всех функциях, требуется приведение типов, которое сложно отлаживать, и так далее. Приверженцы слабой типизации ссылка жалуются на это много и создают языки программирования, такие как Ruby, которые вообще не имеют типов, например:

    def print(b)
  puts(format("This is ISBN: %s", b.isbn))
end

    Здесь функция print() не ожидает, что b будет иметь какой-либо определенный тип. Все, что поступает, подходит. Позже, когда приходит время вызывать .isbn, время выполнения проверяет, имеет ли входящий b такой метод. Если да, то все работает хорошо, если нет, возникает ошибка времени выполнения NoMethodError.

    So far so good.

    Однако вот идея: а что, если мы объединим простоту и краткость динамической типизации с безопасностью строгой типизации, полностью избавившись от типов и позволив компилятору выводить информацию о типе из кода, работающего с объектами? Вот наш код снова:

    void print(Book b) {
  System.out.printf(
    "The ISBN is: %s%n", b.isbn()
  );
}

    Подумайте об этом: на момент компиляции уже ясно, что у b должен быть как минимум один метод isbn(). Нет необходимости заставлять программистов явно определять тип Book и упоминать в подписи метода print(), что только книги допустимы: эта информация легко может быть выведена из тела метода print()! Компилятор может просмотреть все операторы в методе print() и ясно понять, что именно будет сделано с объектом b. Этой информации должно быть достаточно, чтобы определить “тип” входящего объекта. Нет необходимости просить программиста делать это явно и тратить еще пять строк кода в новом файле для объявления типа Book. Компилятор может сделать это за нас.

    Конечно, для того чтобы это работало, мы должны запретить приведение типов любого вида, что невозможно в Java, C++, C# и других псевдо-объектно-ориентированных языках. Но это возможно в EO!

    WDYT?

    Fat vs. Skinny Design

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Кажется, иерархии типов/классов в ООП могут быть спроектированы двумя крайними способами: либо с полной инкапсуляцией данных в виду; либо с несколькими интерфейсами, позволяющими работать с непосредственными данными, а классам заниматься их разбором и преобразованием в более мелкие элементы данных. Возможно, вы удивитесь, но я предлагаю второй вариант, который выглядит более элегантным. Мне кажется, что мы не теряем объектную ориентацию, а наоборот, получаем гибкость, повторное использование, возможность тестирования и т. д.

    Взгляните на это (давайте назовем его толстым, и я объясню почему позже):

    interface Article {
  Head head();
}
interface Head {
  Author author();
  String title();
}
interface Author {
  String name();
  String email();
}

    Чтобы получить имя автора, мы делаем:

    // It is stored in PostgreSQL (that's why the Pg
// prefix) and retrieves everything using SQL
Article a = new PgArticle();
String name = a.head().author().name();

    Теперь давайте сравним его с альтернативным дизайном (который намного менее «толстый» по сравнению с предыдущим, я бы даже назвал его «худым»).

    interface Article {
  String head();
}
class TxtHead {
  private final Article article;
  String author();
  String title();
}
class TxtAuthor {
  private final Head head;
  String name();
  String email();
}

    Здесь, чтобы получить имя автора, мы должны извлечь заголовок в виде String, извлечь автора в виде String, а затем извлечь имя в виде String.

    Article a = new PgArticle();
String head = a.head();
String author = new TxtHead(head).author();
String name = new TxtAuthor(author).name();

    В первом дизайне было три интерфейса, в то время как во втором только один интерфейс и два класса. Я называю первый “толстым”, потому что он возвращает интерфейсы, которые уже реализуют функциональность, которую мы ищем, и нам не нужно покрывать их дополнительными декораторами или адаптерами. Его иерархия из трех интерфейсов достаточно богата, чтобы предоставить нам все, что нам нужно. Вот почему он толстый. Второй, с другой стороны, довольно худой, есть только один интерфейс, который возвращает нам простые текстовые данные, которые нам приходится разбирать самостоятельно. Нам нужно его подкрасить.

    Кажется, что худощавый дизайн лучше по нескольким причинам:

    Расширяемость. Тонкий дизайн определенно проще расширять. Для извлечения новой информации от автора нам просто нужно добавить новый метод в класс TxtAuthor. Нам не нужно перерабатывать всю иерархию интерфейсов и изменять все их реализации. Мы работаем с чистыми данными, которые затем обрабатываются в декораторах, адаптерах и других вспомогательных умных классах.

    Согласованность. Худой дизайн явно более согласован, поскольку все, что связано с управлением данных PostgreSQL, находится в одном классе SqlArticle. Напротив, толстый дизайн распределяет функциональность среди множества классов и, благодаря этому, делает всю группу классов более сложной для поддержки.

    Повторное использование. Класс TxtAuthor безусловно может быть использован в любом другом месте, где требуется анализ информации об авторе, в то время как класс PgAuthor подходит только для одного конкретного случая: получение и анализ данных, связанных с PostgreSQL.

    Тестируемость. Очевидно, что узкий дизайн намного проще тестировать, потому что подмена одного интерфейса является гораздо более простой задачей, чем подмена всей иерархии. Чтобы протестировать класс TxtAuthor, мы просто передаем некоторый поддельный текст его конструктору и проверяем, как он работает. Чтобы протестировать класс PgAuthor, нам потребуется сделать намного больше, включая запуск поддельного экземпляра сервера PostgreSQL.

    Все сказанное выше верно как для 1) извлечения данных из PostgreSQL, так и для 2) манипуляций с данными в PostgreSQL. Конечно, для манипуляций может потребоваться множество методов, существующих в SqlArticle, что сделает стройный дизайн некрасивым, и станет очевидным, что некоторые из этих методов должны быть перемещены в классы/интерфейсы более низкого уровня. Это лишь демонстрирует, что не всегда возможно создать стройный дизайн с помощью одного интерфейса, как в приведенном выше примере. Иногда нам просто нужно сделать его “полнее”.

    Однако существует одна серьезная проблема, связанная с узким дизайном: он позволяет сырым голым данным выходить из SqlArticle, что противоречит самой идее объектно-ориентированного программирования, как мы знаем. Действительно, если мы позволим TxtHead выполнять парсинг, мы можем потерять некоторый интересный контекст, связанный с PostgreSQL, который доступен только внутри SqlArticle. Мы не хотим, чтобы сложный разбор данных происходил далеко от места их происхождения. Мы хотим, чтобы все, что связано с данными, происходило там, где живут данные: внутри SqlArticle.

    Это обоснованное опасение, но передача информации, связанной с PostgreSQL (например, настройки подключения), из PgArticle в PgHead, а затем в PgAuthor, является еще более серьезным нарушением принципа инкапсуляции данных.

    В реальных ситуациях, конечно, невозможно представить исключительно узкие дизайны с единственным интерфейсом. Все они будут в некоторой степени объемными. Мое предложение, однако, заключается в том, чтобы стараться делать дизайны менее объемными, позволяя пользователям интерфейса наряжать их по своему усмотрению. Это предложение очень близко к тому, что я ранее сказал о умных классах, но на этот раз принцип более общий.

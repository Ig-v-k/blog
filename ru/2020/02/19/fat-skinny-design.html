    Fat vs. Skinny Design

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    Кажется, иерархии типов/классов в ООП могут быть разработаны двумя крайними способами: с полной инкапсуляцией данных в виду или с несколькими интерфейсами, делающими сырые данные видимыми и позволяющими классам работать с ними, разбирать и преобразовывать их в более мелкие элементы данных. Возможно, вас удивит, но я предлагаю второй вариант более элегантным. Кажется, что мы не теряем объектно-ориентированность, а, наоборот, получаем гибкость, повторное использование, тестирование и так далее.

    Взгляните на это (давайте назовем его толстым, и я объясню почему позже):

    interface Article {
  Head head();
}
interface Head {
  Author author();
  String title();
}
interface Author {
  String name();
  String email();
}

    Для получения имени автора мы делаем:

    // It is stored in PostgreSQL (that's why the Pg
// prefix) and retrieves everything using SQL
Article a = new PgArticle();
String name = a.head().author().name();

    Визуально этот дизайн может выглядеть так (в UML):

    Теперь давайте сравним его с альтернативным дизайном (который намного менее “толстый” по сравнению с предыдущим, я бы назвал его “стройным”):

    interface Article {
  String head();
}
class TxtHead {
  private final Article article;
  String author();
  String title();
}
class TxtAuthor {
  private final Head head;
  String name();
  String email();
}

    Здесь, чтобы получить имя автора, нам нужно извлечь заголовок как String, извлечь автора как String, а затем извлечь имя как String.

    Article a = new PgArticle();
String head = a.head();
String author = new TxtHead(head).author();
String name = new TxtAuthor(author).name();

    Визуально в UML это выглядит так:

    В первом дизайне было три интерфейса, в то время как во втором только один интерфейс и два класса. Я называю первый “толстым”, потому что он возвращает интерфейсы, которые уже реализуют функциональность, которую мы ищем, и нам не нужно покрывать их дополнительными декораторами или адаптерами. Его иерархия из трех интерфейсов достаточно богата, чтобы предоставить нам все необходимое. Вот почему он толстый. Второй, с другой стороны, довольно стройный, есть только один интерфейс, который возвращает нам простые текстовые данные, которые нам приходится анализировать самостоятельно. Нам нужно их “нарядить”.

    Кажется, что стройный дизайн лучше по ряду причин:

    • Расширяемость. Тонкий дизайн определенно легче расширить. Чтобы извлечь новую информацию от автора, нам просто нужно добавить новый метод в класс TxtAuthor. Нам не нужно перерабатывать всю иерархию интерфейсов и изменять все их реализации. Мы имеем дело с чистыми данными, которые управляются и разбираются позже в декораторах, адаптерах и других дополнительных умных классах.

    • Связанность. Худой дизайн определенно является более связанным, поскольку все, что связано с управлением данных PostgreSQL, находится в одном классе SqlArticle. Напротив, толстый дизайн распределяет функциональность между множеством классов и, благодаря этому, делает всю группу классов более сложной в поддержке.

    • Повторное использование. Класс TxtAuthor можно безусловно использовать в любом другом месте, где требуется разбор информации об авторе, в то время как класс PgAuthor подходит только для одного конкретного случая: получение и разбор данных, связанных с PostgreSQL.

    • Тестируемость. Очевидно, что тонкий дизайн намного проще тестировать, потому что подконтрольный интерфейс гораздо проще подделывать, чем всю иерархию. Чтобы протестировать класс TxtAuthor, мы просто передаем фиктивный текст в его конструктор и проверяем, как он работает. Чтобы протестировать класс PgAuthor, нам придется сделать намного больше, включая запуск фальшивого экземпляра сервера PostgreSQL.

    Все вышесказанное верно как для 1) извлечения данных из PostgreSQL, так и для 2) манипуляций с данными в PostgreSQL. Конечно, манипуляции могут требовать существования множества методов в SqlArticle, что сделает худое проектирование неэстетичным, и станет очевидным, что некоторые из этих методов должны быть перемещены в классы/интерфейсы более низкого уровня. Это только демонстрирует, что не всегда возможно создать худое проектирование с помощью одного интерфейса, как в приведенном выше примере. Иногда нам просто нужно сделать его более “толстым”.

    Однако, существует одна серьезная проблема, связанная с худым проектированием: это позволяет голым данным вылезти из SqlArticle, что противоречит самой идее объектно-ориентированного программирования, как мы знаем. Действительно, если мы позволим TxtHead выполнять разбор, мы можем потерять некоторый интересующий нас контекст, связанный с PostgreSQL, который доступен только внутри SqlArticle. Мы не хотим, чтобы сложный разбор данных происходил далеко от места их создания. Мы хотим, чтобы все, что связано с данными, происходило там, где живут данные: внутри SqlArticle.

    Это обоснованная проблема, но позволить информации, связанной с PostgreSQL (например, параметрами подключения), перемещаться из PgArticle в PgHead, а затем в PgAuthor, является еще более серьезным нарушением принципа инкапсуляции данных.

    В реальных ситуациях, конечно, невозможно представить себе чистые худые проектирования с одним интерфейсом. Все они будут в некоторой степени “толстыми”. Однако я предлагаю попытаться сделать проектирования менее “толстыми”, позволяя пользователям интерфейса наряжать их так, как им нравится. Это предложение очень близко к тому, что я сказал ранее о “умных” классах, но на этот раз принцип более широкий.
